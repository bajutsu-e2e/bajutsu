# Web UI ツアー：実機 Simulator を操作し、あらゆる証跡を収集する

これは **実機 Simulator で Bajutsu を評価する iOS 開発者**向けのデモです。ローカルの Web UI
（[`bajutsu serve`](../../docs/ja/cli.md)）を使って Simulator を起動し、showcase アプリに対して
シナリオを実行し、各実行が捕捉する証跡（スクリーンショット、動画、デバイスログ、通信、ビジュアル
リグレッションの差分、システムアラート処理）をブラウザで辿ります。**合否の判定に AI を介在させること
は一切ありません**。

まずセットアップ不要で同じ流れを見たい場合（Mac/Simulator 不要）は、
[60秒ツアー](../tour/README.ja.md) を実行してください: `uv run python demos/tour/tour.py`。

## 前提条件

- Xcode と起動できる Simulator のある macOS（`open -a Simulator`）。
- XCUITest ランナー。Web UI が必要時にビルドしますが、事前にビルドすることもできます:
  `make -C demos/showcase runner-build`。
- システムアラート部分のみ: `ANTHROPIC_API_KEY`（環境変数か gitignore された `.env`）。それ以外は
  完全に決定論的で、キーは不要です。

アプリを先にビルドする必要は**ありません**。`.app` が無い初回には、設定された `build` コマンドを Web UI が
実行し、選んだデバイスへインストールします。

## 起動

```bash
make -C demos webui                  # http://127.0.0.1:8765  （変更は PORT=8766）
```

これはリポジトリ直下の `make serve` をラップし、[`showcase.config.yaml`](showcase.config.yaml) を
バインドします。この設定が Web UI を showcase の各アプリと、その [`scenarios/`](scenarios) にある
シナリオに向けます。表示された URL を開いてください。

> Web UI は必ず `make serve`（素の `bajutsu serve` ではなく）経由で起動してください。ラッパーが
> バックエンドの依存を必要時にインストールするので、クローン直後でも
> `no available actuator among ['xcuitest']` で失敗しません。

## ツアー

UI には2つのタブがあります。**Replay**（シナリオを決定論的に実行し、レポートを見る）と **Record**
（自然言語の目標からエージェントでシナリオを著作する）です。以下のツアーは、特記なき限り Replay です。

### 1. シナリオを実行してレポートを読む

**Replay → Run** で、`showcase-swiftui` アプリを選び、一覧から Simulator を選び（停止中なら自動で起動
されます。「booted」はすでに起動中のものを指します）、`smoke.yaml` を選んで **Run**。ライブログがデバイス
を起動し、初回はアプリをビルド/インストールし、実行する様子が見えます。完了すると、対話的な**レポート**が
埋め込み表示されます。

レポートの **Result** パネルを開くと、各ステップに、操作対象の**スクリーンショット**と**アクセシビリティ
ツリー**が、一致した要素をハイライトした状態で表示されます。**Expectations** テーブルには、合否を決めた
機械的なアサーションが並びます。これが中心的な約束であり、ステップごとに監査できる決定論的な判定です。

### 2. 通信の記録（観測、そしてモック）

showcase は **BajutsuKit** をリンクしているので、アプリが行う各 HTTP(S) のやり取りを TLS の*後*で捕捉
します（プロキシも CA も不要）。通信を行うシナリオ（Stable カタログの `stable.refresh` や Log タブの
`log.submit`）を実行すると、レポートに **Network** タブが増え、各やり取り（メソッド、URL、ステータス、
所要時間、ヘッダ、ボディ）が表示されます。あるリクエストは意図的に秘匿ヘッダとボディフィールドを載せて
いるので、redaction がそれらをその場でマスクする様子も見えます。

次に [`network_mock.yaml`](scenarios/network_mock.yaml) を実行します。同じ画面ですが、`mocks:` ルールが
実エンドポイントの代わりに定型のレスポンスを返します。Network タブはそのやり取りを **mocked** と示します。
ライブ依存のない決定論的な通信です。

### 3. ビジュアルリグレッション（VRT）：UI からベースラインを承認する

[`visual.yaml`](scenarios/visual.yaml) を実行します。初回は**ベースラインが無い**ため `visual` の期待は
意図的に失敗し、レポートには捕捉したスクリーンショットと **「Approve as baseline」**ボタンが表示されます。
クリックすると、そのスクリーンショットが baselines フォルダへ昇格します。再実行すると、今度はそのベース
ラインと**ピクセル比較**して合格します。後で画面がずれた場合は、レポートが差分割合つきの**並列差分**
（swipe / onion / blend モード）を表示します。`threshold` がアンチエイリアスの揺れを許容し、`exclude` の
矩形が変動領域（時計、ステータスバー、フローティングのタブバー）をマスクします。UIKit と SwiftUI の双子は
同じ画面を描きますがピクセル単位では一致しないので、各々が自分のベースライン一式を持ちます。

> CLI での同じ承認手順: `make -C demos/showcase vrt` のあと `vrt-approve`。

### 4. システムアラート：SpringBoard のプロンプトを突破する

iOS バックエンドのアクセシビリティ照会は前面アプリしか見えないため、iOS のシステムプロンプト（権限ダイアログ）は
実行を静かにブロックします。**Dismiss alerts** を有効（既定。`ANTHROPIC_API_KEY` が必要）にして
[`permission.yaml`](scenarios/permission.yaml) を実行すると、Permissions タブが通知／位置情報のプロンプト
を出し、それがステップをブロックしたときにガードがスクリーンショットを撮り、Claude vision にどこが閉じる
ボタンかを尋ね、タップして、ステップを再試行します。レポートの手順テーブルには、アラートのラベルを伴う
**「system alert dismissed」**の行が出ます。ここが `run` で AI が補助する唯一の箇所であり、しかも
*デバイスを操作する*だけで、判定には決して関与しません。シナリオは `alertHandling: { instruction:
"tap Allow" }` で、閉じる代わりに許可することもできます。

### 5. 動画とデバイスログ

シナリオはどれもシナリオ全体の画面録画を記録するので、どの実行のレポートにも **Video** タブ（シークバー
付き）と **Log** タブ（検索できるデバイスログ）が、`os_signpost` 区間のアプリ **Trace** とともに付きます。
[`modals.yaml`](scenarios/modals.yaml) を実行すると、4種類のモーダルが開いて閉じる様子で動画がとりわけ
読みやすくなります。「X のたびに記録」をルールとして一度書けば、AI なしで毎回再現されます。設定としての
証跡です。

### 6. 自然言語から新しいシナリオを著作する（Record タブ）

**Record** タブに切り替えます。目標（「3番目の馬を開いて Favorite をオンにする」）を入力し、アプリと
エージェントを選んで **Record**。エージェントがライブ画面を読み、一手ずつ提案します（推論からアクションへ
ストリーミングされます）。実行されたステップは決定論的なシナリオとして書き出され、編集して **Save** し、
Replay タブから再生できます。`showcase-swiftui-noax` アプリを選ぶと、アプリが識別子を一切公開しないときに
エージェントが stability ladder を下っていく様子が見えます。showcase が示すために作られた、アクセシビリティ
の A/B です。これが Tier 1（AI 著作）から Tier 2（決定論的再生）への受け渡しであり、アーキテクチャ全体が
一つのウィンドウに収まっています。

## このデモが示すもの

- **信頼でき、監査できる一つの判定**：合否は機械的なアサーションだけで決まり、それを裏付ける
  スクリーンショット／要素ツリー／通信／動画が各ステップに添付されます。
- **証跡の幅広さ**：UI の状態、通信（観測とモック）、ビジュアルリグレッション、システムアラート、動画、
  ログを、その場限りのスクリプトではなく再利用可能なルールで捕捉します。
- **AI は役立つところに、判定には決して**：著作（Record）とアラート処理は*操作*を補助し、ゲートは決定論
  のままです。
