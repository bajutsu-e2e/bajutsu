# BE-0015 — Web UI の公開ホスティング

* 提案: [BE-0015](BE-0015-web-ui-public-hosting-ja.md)
* 状態: **提案**
* トラック: [提案](../README-ja.md#提案)
* トピック: オーサリング体験（record / GUI エディタ）

## はじめに

ローカルの `serve` を共有・公開サービスへ移行します。コントロールプレーン（Linux: FastAPI + Postgres + Redis + R2）と macOS ワーカープール（Orka）に分離し、認証・隔離・per-run Simulator を追加します。`subprocess.Popen` をジョブキューに置き換える中核部分のリファクタリングが必要です。

## 動機

ローカルの Web UI（`bajutsu serve`、[bajutsu/serve/](../../../bajutsu/serve/)）は Tier 1 のオーサリング向け便利機能です。`127.0.0.1` にバインドし、認証はなく、同一ホスト上で `bajutsu run` を subprocess 起動します。つまり単一マシン・単一ユーザのツールであり、公開ホスティングはここにある3つの制約を取り除くためのものです。

- **共有できない。** シナリオのオーサリングも失敗の調査も共同作業ですが、run ツリー・ライブログ・レポートはすべて1台のノート PC に留まります。同僚のレポートを開く、進行中のオーサリングを見る、結果をレビューする、いずれもファイルを手でコピーしないとできません。
- **オーサリングする全員に設定済みの Mac が要る。** run は iOS Simulator を駆動し、Simulator は macOS 上にしか存在しません。そのため今は、設定済みの macOS マシンを持つ人しかオーサリングも run もできません。共有サービスなら、ブラウザさえあればマネージドな Mac プールに対して誰でも実行できます。
- **UI がホストに縛られている。** リクエストごとに配信マシン上で subprocess を起動するため（[jobs.py](../../../bajutsu/serve/jobs.py) の `run_job`）、1ホストを超えてスケールできず、あるユーザの run を別のユーザから隔離することもできません。

公開ホスティングは、このノート PC ごとの便利機能を共有インフラへ変えます。ログイン済みユーザがプロジェクトとシナリオを選び、マネージドな Mac プールで実行し、ライブログを見て、できたレポートを URL で共有します。これはまさに*オーサリング体験*のトピックです。机ごとに macOS を要する前提を取り除き、run とレポートを一級の共有可能な成果物にします。

その際、prime directives（[CLAUDE.md](../../../CLAUDE.md)）を侵してはいけません。`run` は完全に決定論的なままで、合否判定の経路には大規模言語モデル（LLM: Large Language Model）を一切入れません。決定論的ゲートは使い捨ての継続的インテグレーション（CI: Continuous Integration）ランナー上に置いたままにします。ホステッドの Mac プールが担うのは対話的オーサリングだけで、回帰テストの物量は担いません。

## 詳細設計

具体的なサーバ・DB・ストレージ・デプロイのスタック（各レイヤで検討した製品の比較を含む）は [cloud-hosting.md](../../ja/cloud-hosting.md) にあります。本節では proposal として確定するアーキテクチャを示し、技術選定の詳細はそのページに委ねます。

### 分離トポロジ

Web UI は薄いランチャーです。`/api/run` は最終的に `idb` と `simctl` 経由で iOS Simulator を駆動し、Simulator は macOS 上にしか存在しません。したがって「Web UI をホスティングする」とは実質「ランナーをホスティングする」であり、ランナーには Mac が要ります。汎用の Linux PaaS（Platform as a Service）では run を実行できません。このため現状の単一プロセス設計には無い分離が必要になります。

- Linux 上の**コントロールプレーン**（安価で水平スケール）が、ステートフルなマルチユーザ部分（認証・run 履歴・ジョブキュー・レポート配信）を持ちます。
- **macOS ワーカープール**（高価・macOS 限定・隔離が必要）は、ジョブを lease し、クリーンな Simulator で実行し、ログを流し、成果物をアップロードするだけのステートレスなワーカーに絞り込みます。

### 中核のリファクタ: プロセス内 subprocess からジョブキューへ

今の `bajutsu serve` は CLI をプロセス内で実行します。`ServeState.popen`（既定は `subprocess.Popen`）が `bajutsu run`/`record` をバックグラウンドスレッドで起動し、`run_job` が結合出力を `Job.lines` にバッファして、ブラウザがおよそ1秒ごとにポーリングして読み出します（[jobs.py](../../../bajutsu/serve/jobs.py)）。コマンドを実行するのが配信プロセスではなくリモートワーカーになるため、ここが変わる中核です。

ホステッドでも `serve` が今組み立てているのと同じ `bajutsu run` の argv を実行しますが、その起動がブローカーを跨ぎます。

1. コントロールプレーンはリクエストを呼び出し元のプロジェクトに対して検証し、`run` レコードを書き、subprocess を起動する代わりに、それを表す**ジョブを enqueue** します（おおよそ `{run_id, project, scenario_ref, app, options, byo_key_ref}`）。
2. macOS ワーカーがそのジョブを **lease** し、そのプロジェクトのシナリオを（クライアント指定のパスではなく id で）取得し、コマンドを実行します。
3. ワーカーは実行しながらログ行と状態を run 単位のチャネルに **PUBLISH** します。コントロールプレーンはそのチャネルを購読し、ブラウザへ **server-sent events（SSE）** で中継します。これがポーリングループを置き換えます。
4. 完了時にワーカーは **`runs/<id>/` ツリーをオブジェクトストレージへアップロード**し、結果（exit code・run id・manifest 要約）をコントロールプレーンへ **POST** します。

こうしてプロセス内の `Job`/`ServeState` ライフサイクルは2つに分かれます。キューに積むジョブのペイロードがプロセス内 `Job` を置き換え、ライブログ配信は `Job.lines` のポーリングから publish/subscribe + SSE へ移ります。決定論的コア（`bajutsu run` とそのレポート）は終始不変で、変わるのはその起動方法と配管だけです。

### run ごとの Simulator 隔離

公開マルチテナントのサービスは run どうしを隔離しなければなりません。各 run は消去済みのクリーンな Simulator（`bajutsu run --erase`）上で用意し、これはローカル UI の高速な `--no-erase` 再利用ループを意図的に捨てることを意味します。アプリごとの差分は今と同じく config（`apps.<name>`）に置いたままなので、ランナーとドライバはアプリを跨いで不変です。

### セキュリティ要件（公開前に必須）

今の `serve` が安全なのは localhost 限定かつ単一ユーザだからにすぎず、公開は両方の前提を壊します。以下は任意ではなく前提条件です（詳細は [cloud-hosting.md](../../ja/cloud-hosting.md#セキュリティ強化公開前に必須)）。

- **全エンドポイントに認証**。org 単位のロールベースアクセス制御を付け、run 起動はユーザ/org 単位でレート制限します。
- **任意パスでのシナリオ実行を排除**。現状 `/api/run` は、リクエストの `scenario` を `scenarios_dir` 内かの検査なしに `bajutsu run` の argv へそのまま渡します（[helpers.py](../../../bajutsu/serve/helpers.py) の `run_command`）。ホステッドではシナリオをプロジェクト単位で保存し、ワーカーが id で取得します。`backend`/`udid` も自由文字列でなく許可リストで検証します。
- **org ごとの BYO（Bring Your Own）`ANTHROPIC_API_KEY`**。AI 経路（`record`・`run --dismiss-alerts`）のコストと悪用の面を、鍵を持つ org に限定します。
- **ワーカーのサンドボックス化**。シナリオは実質「デバイスを操作する非信頼コード」なので、各 run は使い捨ての Mac と Simulator 上で、egress 許可リスト付き・テナント間でシークレット共有なしで実行します。
- **署名付き・期限切れの成果物 URL**、CORS（オリジン間リソース共有）/CSRF（クロスサイトリクエストフォージェリ）対策、標準セキュリティヘッダ、誰が何をいつ実行したかの監査ログ。
- **org 単位のクォータと並行上限**。1 テナントが希少で高価な Mac プールを枯渇させないようにします。

### デプロイ（段階的）

完全な段階計画は [cloud-hosting.md](../../ja/cloud-hosting.md#デプロイ計画段階的) にあります。要点として、フェーズ1は MVP（Minimum Viable Product）で、コンテナ化したコントロールプレーンを Linux プラットフォームに、マネージドの Postgres と Redis、成果物をオブジェクトストレージに、GitHub ログイン、macOS ワーカー1ノードを置き、ログイン済みユーザが共有インフラ上でプロジェクトとシナリオを選び、実行し、ライブログを見て、レポートを end-to-end で閲覧できるようにします。フェーズ2はコントロールプレーンをスケールさせ、キュー深さに応じて Mac プールをオートスケールし、org 単位の並行クォータとフル可観測性を加えます。Linux 側は安価で弾力的ですが、Mac が費用を支配しきれいにゼロスケールしません。だからこそ決定論的ゲートは使い捨て CI に置いたままにし、ホステッドプールは対話的オーサリングだけを担います。

### `serve` からの移行（書き直しではなく段階的に）

以下の各ステップは独立に出荷・テスト可能で、決定論的コアは終始不変です。詳細は [cloud-hosting.md](../../ja/cloud-hosting.md#servepy-からの移行ゼロからの書き直しではなく段階的に)。

1. 純粋ヘルパ（`list_scenarios`・`list_runs`・`run_command`・`Job` モデル）を Web アプリへ移し、既存 HTML を最初のフロントとして流用します。
2. プロセス内の `run_job` を enqueue-to-broker に置換し、今組み立てているのと同じ `bajutsu run` argv を実行するワーカーのエントリポイントを追加します。
3. ポーリングを publish/subscribe 上の SSE に、ローカルのファイル配信を署名付きオブジェクトストレージ URL に置換します。
4. 認証とリレーショナルデータベース（org・project・run）を追加し、シナリオと app のソースをファイルシステムからプロジェクト単位ストレージへ移します。
5. macOS ワーカーを1台立て、上記セキュリティ項目を閉じてから、プールをスケールします。

## 検討した代替案

ここに挙げるのはアーキテクチャ水準の選択です。各レイヤの製品の代替（例: RQ か Celery か、Orka か EC2 Mac か、自前 GitHub OAuth かマネージドプロバイダか）は [cloud-hosting.md](../../ja/cloud-hosting.md#選定スタック推奨) に表でまとめています。

- **`serve` をローカル限定のまま据え置く（現状維持）。** 最も単純で、個人のオーサリングには今後も妥当な既定です。*本提案*の答えとしては不採用です。トピックが共有のオーサリング体験だからで、localhost のツールは run もレポートも共有できず、Mac を持たないオーサーには使えません。
- **ホステッド Mac 1台で分離しない（すべてを単一 macOS ホストで実行）。** リモート Mac 上でプロセス内 `subprocess.Popen` モデルを維持してキューのリファクタを避ける案です。不採用: 1ホストを超えてスケールできず、テナントを隔離できず、Linux に置くべき安価なステートフル作業（認証・履歴・レポート配信）に希少で高価な Mac 時間を費やしてしまいます。分離トポロジは、まさにその作業を Mac から外すために存在します。
- **ライブログを SSE でなく WebSocket で流す。** WebSocket は双方向で、これでも動きます。SSE を選ぶのは、ログ配信が一方向（ワーカー→ブラウザ）で、素の HTTP 上では SSE の方が単純だからです。双方向の run 制御が要るときのために WebSocket は温存します。
- **シナリオをファイルシステムに置き、`/api/run` でパスを渡す。** これは今の `serve` の挙動で、作業量は最小です。公開ホスティングでは不採用: クライアントが選んだパスを run の argv に渡すのは任意パス実行の穴なので、シナリオはプロジェクト単位で保存し id で取得しなければなりません（セキュリティ要件を参照）。
