## Introduction

Add a free-text search box to the roadmap status dashboard (the GitHub Pages page shipped by
[BE-0094](../BE-0094-roadmap-status-dashboard/BE-0094-roadmap-status-dashboard.md)), so a reader can
type a keyword and immediately narrow the cards to the matching items. Search matches an item's **id,
title, topic, and status** and composes with the existing status-chip filter. Like the rest of the
page it is a pure client-side enhancement over already-rendered markup — no new data is emitted and
no build-time or runtime dependency is added.

## Motivation

The dashboard already lets a reader filter by status (the four toggle chips) and collapse categories,
but the only way to find a *specific* item — "where's the one about tracking issues?", "did we ever
propose X?" — is the browser's Ctrl+F or scanning the cards by eye. As the roadmap grows past
~200 items across many topics, that becomes the slowest part of using the page: Ctrl+F highlights raw
text anywhere on the page (including the intro prose and collapsed sections) rather than *filtering to
the matching cards*, and it can't combine with the status filter. A first-class search box turns
"find the BE item I half-remember" into one keystroke-driven interaction, which is the dashboard's
main day-to-day job.

This is a documentation/contributor-workflow improvement only. It stays clear of every prime
directive: the dashboard is a static docs-site artifact, so there is no LLM anywhere near it, no
`run`/CI verdict path, no determinism surface, and nothing app-specific — the search is generic over
whatever items the metadata produces.

## Detailed design

The dashboard is generated by [`scripts/build_roadmap_dashboard.py`](../../scripts/build_roadmap_dashboard.py):
`render_html` emits one `.be-card` per item (each already carrying `data-status`, its id, title, and
origin), grouped into `.be-cat` sections whose `<h3>` is the Topic, and a `<script>` block
progressively enhances the status chips. The search box slots into exactly that structure.

The work is MECE across these units:

1. **Search input in the markup.** Render a labelled `<input type="search">` (with a placeholder and
   an `aria-label`) on its own row (`.be-search-row`) above the status chips, both inside the
   `.be-filters` container (the chips sit in a `.be-chips` row). It is inert without JavaScript, so
   the no-JS page is unchanged (progressive enhancement, matching the existing filter).

2. **Card-level searchable metadata.** So the matcher needn't scrape rendered text, give each
   `.be-card` a `data-topic` attribute (the id, title, and status are already present as
   `data-status` / rendered text). The four searchable fields — id, title, topic, status — are then
   readable directly off each card's attributes/text. Match case-insensitively on a substring; treat
   whitespace-separated words as AND terms (all terms must match somewhere across the fields).

3. **Filter logic that composes with status.** Extend the existing `apply()` in `_SCRIPT` so a card
   is visible only when it passes **both** the status chips **and** the current query (empty query =
   matches everything). Reuse the existing category/group hiding: a category left with no visible card
   hides, a group left with no visible category hides. When a query is active, matching categories
   **expand** (overriding the "all-status-on ⇒ collapse to overview" default) so results are visible
   without a click.

4. **Empty-state message.** When a non-empty query matches zero cards, show a small "No items match
   *query*" line instead of a blank page, cleared when the query is emptied or broadened.

5. **Styling.** Add the input's CSS to `_STYLE` (consistent with the chip styling, one line, no new
   assets), and ensure it reads well in both the light and dark MkDocs themes.

6. **Tests.** Extend [`tests/test_roadmap_dashboard.py`](../../tests/test_roadmap_dashboard.py) to
   assert the rendered page contains the search input, that every card carries `data-topic`, and that
   the filter script wires the input's `input` event. This is the machine-checkable outcome; it needs
   no browser and keeps the gate fast and LLM-free.

No change to what the page *contains* (the item set and its fields are unchanged) — only how a reader
narrows it. `docs/architecture.md` / `docs/ja/` need no update, as they describe the dashboard's
purpose, not its filter affordances; if a sentence there enumerates the filters, update it in the same
PR (BE-0113).

## Alternatives considered

- **Rely on the browser's Ctrl+F.** Zero code, but it highlights instead of filtering, spills into the
  intro prose and collapsed sections, and can't combine with the status chips — exactly the gaps this
  item closes.
- **Server-side / indexed search (e.g. MkDocs' built-in search plugin).** The site already has
  site-wide search, but it searches *pages*, not *cards within the dashboard*, and returns you to the
  top of the page rather than filtering the grid. A card-level filter is the right granularity and
  keeps the dashboard self-contained (one generated page, no plugin coupling).
- **Full-text search over each item's prose (Introduction, Motivation, …).** Rejected for this item:
  the dashboard renders only metadata, not the proposal bodies, so matching prose would force the
  generator to embed every item's full text into the page — a large, redundant payload for marginal
  gain. Scoping to id/title/topic/status keeps the page lean; prose search already lives in the repo
  (grep / GitHub search) and in the `/roadmap-filter` AI skill (BE-0162).
