## はじめに

ロードマップ status ダッシュボード（[BE-0094](../BE-0094-roadmap-status-dashboard/BE-0094-roadmap-status-dashboard-ja.md)
が公開した GitHub Pages のページ）にフリーテキストの検索ボックスを追加します。読者がキーワードを入力すると、
一致する項目のカードだけにその場で絞り込めるようにします。検索は各項目の **ID、タイトル、トピック、状態** に
対して一致を取り、既存の状態チップによる絞り込みと組み合わせて働きます。ページのほかの部分と同じく、これは
すでにレンダリング済みのマークアップに対する純粋なクライアントサイドの拡張です。新たに出力する情報はなく、
ビルド時にも実行時にも依存を追加しません。

## 動機

ダッシュボードはすでに状態での絞り込み（4 つのトグルチップ）とカテゴリの折りたたみを備えています。しかし
特定の項目を探す手立ては、ブラウザの Ctrl+F かカードの目視だけです。「トラッキング Issue の項目はどこだったか」
「X という提案はあったか」といった探し方に応える機能がありません。ロードマップが多数のトピックにわたって
200 項目を超えて育つと、ここがページを使ううえで最も遅い作業になります。Ctrl+F はページ上のどこにある文字でも
（導入部の文章や折りたたまれたセクションを含めて）ハイライトするだけで、一致するカードへの絞り込みにはならず、
状態による絞り込みとも組み合わせられません。第一級の検索ボックスがあれば、「うろ覚えの BE 項目を探す」作業が
キー入力ひとつの操作になります。これはダッシュボードの日々の主な役割そのものです。

これはドキュメントと開発ワークフローの改善にとどまります。プライムディレクティブのいずれにも触れません。
ダッシュボードは静的なドキュメントサイトの生成物なので、その近傍に大規模言語モデル（LLM）は存在せず、
`run` や継続的インテグレーション（CI）の合否判定の経路もなく、決定性に関わる面もなく、アプリ固有の要素も
ありません。検索はメタデータが生む項目に対して汎用に働きます。

## 詳細設計

ダッシュボードは [`scripts/build_roadmap_dashboard.py`](../../scripts/build_roadmap_dashboard.py) が生成します。
`render_html` は項目ごとに `.be-card` を 1 つ出力し（各カードはすでに `data-status`、ID、タイトル、origin を
持ちます）、`<h3>` がトピックである `.be-cat` セクションにまとめ、`<script>` ブロックが状態チップを段階的に
拡張します。検索ボックスはこの構造にそのまま収まります。

作業は次の単位に MECE（漏れなく重複なく）で分解できます。

1. **マークアップへの検索入力欄の追加**。ラベル付きの `<input type="search">`（プレースホルダーと `aria-label`
   を持つ）を、状態チップの上に、`.be-filters` コンテナ内の独立した行（`.be-search-row`）として描画します
   （チップは同じコンテナの `.be-chips` 行に入ります）。JavaScript がなくても無害に無効化されるので、JavaScript
   を切ったページの見え方は変わりません（既存の絞り込みと同じ段階的拡張です）。

2. **カード単位の検索可能なメタデータ**。マッチ処理がレンダリング済みの文字列を走査せずに済むよう、各 `.be-card`
   に `data-topic` 属性を与えます（ID、タイトル、状態はすでに `data-status` や描画済みテキストとして存在します）。
   検索対象の 4 フィールド（ID、タイトル、トピック、状態）は、各カードの属性とテキストから直接読み取れるように
   なります。マッチは大文字小文字を区別しない部分一致とし、空白区切りの語は AND 条件として扱います（すべての語が
   いずれかのフィールドに一致する必要があります）。

3. **状態絞り込みと組み合わさるフィルタ処理**。`_SCRIPT` の既存の `apply()` を拡張し、カードは状態チップと現在の
   クエリの**両方**を満たすときにだけ表示します（空のクエリはすべてに一致します）。カテゴリとグループの隠蔽は既存の
   処理を再利用します。表示カードが 1 つも残らないカテゴリは隠し、表示カテゴリが 1 つも残らないグループは隠します。
   クエリが有効なときは、一致するカテゴリを**展開**し（「全状態がオンならば概要へ折りたたむ」という既定を上書きして）、
   クリックしなくても結果が見えるようにします。

4. **該当なしの表示**。空でないクエリがどのカードにも一致しないときは、空白のページではなく「一致する項目が
   ありません（*クエリ*）」という小さな一文を示します。クエリを空にするか広げると消えます。

5. **スタイル**。入力欄の CSS を `_STYLE` に追加します（チップのスタイルと揃え、1 行で、新たなアセットは
   使いません）。MkDocs のライトテーマとダークテーマの双方で読みやすくします。

6. **テスト**。[`tests/test_roadmap_dashboard.py`](../../tests/test_roadmap_dashboard.py) を拡張し、レンダリング
   したページに検索入力欄が含まれること、すべてのカードが `data-topic` を持つこと、フィルタスクリプトが入力欄の
   `input` イベントを結線することを確認します。これが機械的に検証できる成果です。ブラウザを必要とせず、ゲートを
   高速かつ LLM のない状態に保ちます。

ページが**含む内容**は変わりません（項目の集合とそのフィールドはそのままです）。変わるのは読者の絞り込み方だけ
です。`docs/architecture.md` や `docs/ja/` の更新は要りません。これらはダッシュボードの目的を述べるもので、
絞り込みの手段を述べるものではないからです。もしそこに絞り込み手段を列挙した文があれば、同じ PR で更新します
（BE-0113）。

## 検討した代替案

- **ブラウザの Ctrl+F に任せる**。コードは不要ですが、絞り込みではなくハイライトであり、導入部の文章や折りたたまれた
  セクションにまで及び、状態チップとも組み合わせられません。まさにこの項目が埋める隙間です。
- **サーバーサイドまたは索引付きの検索（例：MkDocs 組み込みの検索プラグイン）**。サイトにはすでに全体検索が
  ありますが、これは**ページ**を検索するもので、**ダッシュボード内のカード**を検索するものではなく、絞り込みでは
  なくページ先頭へ戻します。カード単位のフィルタが適切な粒度であり、ダッシュボードを自己完結（生成された 1 ページ、
  プラグインへの結合なし）に保ちます。
- **各項目の本文（はじめに、動機など）に対する全文検索**。この項目では見送ります。ダッシュボードはメタデータのみを
  描画し、提案の本文は描画しません。本文に一致させるには、生成側がすべての項目の全文をページに埋め込む必要があり、
  わずかな利得のために大きく冗長なペイロードになります。ID・タイトル・トピック・状態に絞ればページは軽いままです。
  本文の検索はすでにリポジトリ内（grep や GitHub の検索）と `/roadmap-filter` の AI スキル（BE-0162）にあります。
