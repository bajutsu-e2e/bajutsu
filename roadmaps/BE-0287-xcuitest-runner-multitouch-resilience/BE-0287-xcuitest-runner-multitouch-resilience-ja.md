## はじめに

オンデバイスのエンドツーエンド（E2E）ジョブ `xcuitest (multi-touch)` は、二本指ジェスチャの実行中に
XCUITest runner が到達不能になると不安定に失敗します。ドライバは、実行の途中で runner がクラッシュした
状況と、操作がアプリに届かなかった状況とを区別できません。本項目は、実行途中に起きた runner クラッシュに
対して runner チャネルを頑健にすることを提案します。具体的には、runner が停止したことを検出し、明確で
決定論的な runner クラッシュの失敗として表面化させ、冪等性が許す場合には runner の復帰を待って処理を
続けます。取りこぼしたピンチや回転を、アサーションの不一致に化けさせません。変更範囲は xcuitest
バックエンドの内側に収まり、触れる面は runner、リトライの継ぎ目、決定論的な判定の3つだけです。

## 動機

あるプルリクエストの iOS E2E ワークフローを一度再実行したところ、この不安定さが直接現れました。
`xcuitest (multi-touch)` ジョブを3回試行して、1回が成功し、2回が別々の壊れ方で失敗しました。対象の
シナリオは [`gestures_multitouch.yaml`](../../demos/showcase/scenarios/gestures_multitouch.yaml) で、
showcase アプリの二本指ターゲットをピンチと回転で操作し、各ターゲットのミラーされたアクセシビリティ値が
`pinched` と `rotated` に変わることをアサートします。

2回の失敗は原因を共有しますが、現れ方が異なります。1回目は runner がそのままクラッシュしました。runner
がホストするループバックサーバが接続を受け付けなくなり、`GET /screenshot` 呼び出しが
`XcuitestChannelError: runner channel GET /screenshot failed: [Errno 61] Connection refused` を送出しました。
2回目は、runner が約30秒のあいだ `GET /health` に応答せず（`runner channel GET /health failed ...
Connection refused` のリトライが連続し）、復帰したときにはピンチがアプリに届いておらず、最終アサーションは
`expect: expected equals='pinched' but actual='idle'` を読み取りました。二本指の操作が runner を不安定に
し、その乱れがハードなチャネルエラーとして出るか、静かに取りこぼされたジェスチャとして出るかは、
タイミング次第です。

この不安定さは3つの意味で高くつきます。必須の集約チェック `E2E (iOS)` がランダムに失敗すると、ゲートへの
信頼が損なわれます。メンテナは本物の回帰と区別できないため、反射的な再実行か、無駄な調査のどちらかを
招きます。再実行のたびに、ジェスチャに到達する前にアプリをビルドして Simulator を起動する macOS ランナーの
従量課金枠を消費します。さらに悪いことに、`actual='idle'` というメッセージは調査者を誤った場所に
向かわせます。ピンチという操作が壊れているかのように読めますが、実際に起きたのは runner がジェスチャを
取りこぼしたことです。

既存のリトライの継ぎ目（[BE-0207](../BE-0207-xcuitest-channel-transient-retry/BE-0207-xcuitest-channel-transient-retry-ja.md)）は、
この状況を扱いませんし、もともと扱う設計でもありません。BE-0207 の継ぎ目は、1回のチャネル呼び出しの中の
一時的なブレを吸収します。対象となる要求を計3回まで試行し、試行の間は0.5秒の指数バックオフを取るので、予算全体は
およそ1.5秒で尽きます。30秒間いなくなった runner は、この予算を何倍も超えます。BE-0207 の継ぎ目は
書き込みに対して意図的に保守的でもあります。すでに届いた副作用のある要求は再送しません。届いたかも
しれないピンチを再送すると、二重に操作してしまうおそれがあるからです。したがって予算を大きくしても、
取りこぼしたジェスチャを覆い隠すことはできません。実行途中のクラッシュからの回復は、1秒未満のブレを
ならすのとは別の問題であり、専用の仕組みが必要です。

## 詳細設計

作業は独立した4つの単位に分かれます。Unit 1 と Unit 2 は runner 側の根本原因に対処し、実機での反復が
必要です。Unit 3 と Unit 4 は Python のドライバチャネルを強化し、模擬トランスポートに対して単体テスト
できます。各単位は依存順に並びますが、別々のプルリクエストとして着地できます。

**Unit 1 — 二本指操作下での runner クラッシュを特徴づける。** Simulator 上で失敗を再現し、runner 側の
診断情報を取得して、根本原因を推測ではなく事実として突き止めます。未解決の問いは、ピンチや回転の操作
そのものが XCUITest runner プロセスをクラッシュさせるのか、runner がホストするループバック HTTP サーバを
ハングさせるのか、その下の `idb_companion` が死ぬのか、のどれかです。再現した実行について runner の
クラッシュログとコンソール出力を集め、実際にどれが失敗するのかを記録します。この単位が生むのは診断で
あり、コードではありません。

**Unit 2 — 根本原因をあるべき場所で直す。** Unit 1 が runner 自身のジェスチャ処理（[`BajutsuKit/`](../../BajutsuKit)
の XCUITest 側）の欠陥を示したら、そこで直し、二本指の操作が runner を不安定にしないようにします。原因が
companion や Bajutsu が所有しない下位層にある場合は、この単位では上流の制約を記録し、Unit 3 に
引き継ぎます。Unit 3 が、上流の制約を致命的ではなく耐えられるものにします。どちらの道を取るかは Unit 1 の
診断が決めるので、2つの単位は順に書きます。

**Unit 3 — 実行途中のクラッシュを検出し、決定論的に表面化させる。** チャネルが1回の呼び出しのリトライ
予算を超えて到達不能なままのとき、ドライバは、実行を誤解を招くアサーションに漂わせるのではなく、2つの
結果のあいだを決定論的に選ばなければなりません。冪等な読み取り（`GET`、または runner に届かなかった
書き込み）については、既存の `await_ready` のヘルスポーリングを再利用して runner の復帰を待ち、処理を
続けます。読み取りの再発行は安全だからです。すでに届いた可能性のある、副作用を伴う書き込みについては、
黙って再送せず、クラッシュを名指しする専用の runner クラッシュ診断で失敗します。こうして実行は
`actual='idle'` ではなく、runner がジェスチャの途中で死んだという正直なメッセージで止まります。この区別は、
リトライの継ぎ目がすでに引いている `delivered` の分割そのものなので、この単位は新しい仕組みを作らず
既存の継ぎ目を拡張します。

**Unit 4 — すべての回復を記録に残す。** 決定論優先（[`CLAUDE.md`](../../CLAUDE.md) の
プライムディレクティブ 2）は、本物の失敗を成功にならすことを禁じるので、回復を黙って済ませるわけには
いきません。回復するたびに、リトライの継ぎ目が再送したブレを記録するのと同じくらいはっきりとクラッシュを
記録し、クラッシュして回復した実行が、一度もクラッシュしなかった実行と見分けられなくならないように
します。回復した実行は再現可能で監査可能なままとし、冪等に回復できないクラッシュは Unit 3 が挙げる
大きな失敗のままにします。届いた書き込みを再適用しないのは Unit 3 の役割であり、この単位が足すのは
可観測性の保証だけです。

## 検討した代替案

**ジョブ全体を CI で隔離するか自動再試行する。** `xcuitest (multi-touch)` を失敗許容にする、あるいは
ジョブ単位の再試行で包むと、クラッシュに触れないままチェックを緑にできます。採用しません。本物の runner の
欠陥を緑のチェックの裏に隠し、静かな再試行のたびに macOS の従量課金枠を消費し続け、
[BE-0281](../BE-0281-ios-on-device-actuation-coverage/BE-0281-ios-on-device-actuation-coverage-ja.md) が
積み上げている実機アクチュエーションの網羅を、増やすどころか崩すからです。

**リトライの継ぎ目の予算を増やす。** `_MAX_ATTEMPTS` やバックオフを上げて、1回の呼び出しで30秒の停止を
待ち切らせても、肝心の状況は救えません。届いたピンチは、二重操作のおそれなしには再送できないからです。
しかもあらゆる本物のチャネル失敗を30秒の遅さに落とします。リトライの継ぎ目は設計上、1秒未満のブレを
ならすものです。クラッシュまで覆うように引き伸ばすと、2つの別問題を混同します。

**各ジェスチャの後に固定の待機を入れる。** ピンチの後にアサートの前で待つのは、条件を無視した固定
`sleep` であり、プライムディレクティブ 2 が明確に禁じます。そのうえ効果もありません。シナリオの `expect`
ブロックは、すでに期限まで木を読み直す有界の条件待ちだからです。値が `idle` のままなのは、アサートが
早すぎるからではなく、ジェスチャが取りこぼされたからです。届かなかった操作は、いくら待っても回復
できません。
