## はじめに

シナリオ / アプリ一覧の表示、ワンクリック実行、実行ログのストリーミング、ブラウザ内レポート表示を提供する小規模なランチャです（stdlib のみ使用）。CI ゲートには含まれない Tier 1 の利便機能です。予定している GUI エディタ（可視編集と要素ピッカー）の基盤にもなります。

## 動機

CLI が契約ではありますが、シナリオを書いて再実行する手段としては必ずしも使いやすくありません。実行のたびに `run` / `record` の適切なフラグを思い出し、アプリと Simulator を選び、バイナリが古ければ先にビルドし、生成された `report.html` を手で開く必要があります。この手間は、ツールが安く回せるようにしたいループ（書く → 実行する → レポートを見る → 直す）をまさに妨げます。小さなランチャがこれを取り除きます。アプリとシナリオを選んでクリックすれば、出力がストリームされ、レポートが表示され、レポートの相対アセットリンクが解決するよう実行の証跡も配信されます。あくまで Tier 1 の利便機能であり、CI ゲートには含めません。そして予定する GUI エディタ（BE-0013）やより充実した record フローの置き場所にもなります。

## 詳細設計

実装は `bajutsu/serve/` を参照。起動は必ず `make serve` で行います（`bajutsu serve` を直接呼ばないこと）。このラッパが idb backend の依存（idb クライアントと `idb_companion`）をオンデマンドで導入するため、素の `serve` ではそれが飛ばされ、実行が `no available actuator` で失敗します。

サーバは stdlib のみで構成します。`127.0.0.1` にバインドした `ThreadingHTTPServer` で、network collector と同じ方式です。シングルページアプリのシェルと CSS/JS を `bajutsu/templates/serve.*` から 1 つの自己完結 HTML レスポンスにインライン化します（`/static` ルートは別途設けません）。CLI の上に 2 つのトップレベルタブを置きます。

* **Record** は自然言語のゴールからシナリオを生成します。`python -m bajutsu record …` を起動し、エージェントのターンごとの進捗をストリームし、結果を選択したアプリの設定済みシナリオディレクトリに（自動命名で、既存ファイルを上書きせずに）書き出します。
* **Replay** は `python -m bajutsu run …` でシナリオを実行し、完了時に生成された `report.html` を埋め込みます。**History** 一覧から過去の実行を再表示でき、**Approve** ボタンで撮影したスクリーンショットを `visual` ベースラインへ昇格できます（`POST /api/approve`）。

各リクエストはバックグラウンドスレッドで CLI を起動します（`run_job`）。選択した Simulator を並列起動し、バイナリが無ければアプリをオンデマンドでビルドし（config の `apps.<name>.build`）、コマンドを実行して結合出力を 1 行ずつ取り込み、そこから生成された `runs/<id>` を解析します。ジョブはキャンセル可能です。起動される子プロセスは serve プロセスの環境を継承し、`idb` クライアントが解決するよう venv の `bin` を `PATH` の先頭に付けます。アプリ差分は config に閉じます。アプリのドロップダウン、そのシナリオディレクトリ、ビルドコマンド、`appPath` はすべて `apps.<name>` 由来であり、サーバ自体は app-agnostic です。`--config` は任意で、`--root` に閉じたファイルブラウザから UI 上で開けます。

決定性は保たれます。ここで pass/fail を判定することは一切なく、UI は決定的な `run` をシェル経由で呼ぶだけで、ループに入る AI は `record`（Tier 1）とオプトインのアラートガードのみです。UI が設定する唯一のシークレット `ANTHROPIC_API_KEY` はそのセッション中だけメモリに保持し、ディスクには書かず、起動したジョブが継承します。

## 検討した代替案

* **重量級の Web フレームワーク（Flask / FastAPI）とビルドステップを伴うフロントエンド。** 不採用。localhost のオーサリング利便機能のために、依存の重さや別のアセットパイプラインを抱える理由はありません。stdlib の `ThreadingHTTPServer` とインライン化したテンプレートなら、まっさらなクローンからインストールでき、既存の network collector とも揃います。
* **ネイティブのデスクトップアプリやインプロセス GUI。** 不採用。ブラウザはすでに普遍的で、追加のツールキットが要らず、ランナーがすでに生成する自己完結の `report.html` をそのまま表示できます。レポート表示を作り直さず再利用します。
* **CLI を起動せず、インプロセスでデバイスを駆動する。** 不採用。同じ `run` / `record` コマンドをシェル経由で呼ぶことで CLI を唯一の契約に保ち、UI が CI の挙動から乖離しないようにできます。キャンセルと分離も自然に得られます。
