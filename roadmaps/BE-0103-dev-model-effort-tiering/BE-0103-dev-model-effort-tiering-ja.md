## はじめに

Claude Code のセッションが費やす**モデルと推論エフォート**を、**目の前のタスクの負荷**に合わせます。
これにより、このリポジトリを開発するセッションが、機械的な雑務に最上位のトークンを払い続けるのをやめられます。
本項目は製品機能ではなく、**コントリビューターの作業フロー**に関する項目です。対象は Claude Code で Bajutsu を
開発するときの進め方であって、Bajutsu 自身が Claude を呼び出す経路ではありません。

成果物は 2 つです。1 つ目は、[`docs/ai-development.md`](../../docs/ai-development.md)（日英両方）に置く
「タスク → モデル/エフォート」の規約です。作業のフェーズ（探索や雑務は軽く、実装や設計は重く）と、サブエージェントへ
委譲するときに適切な `model` を渡す方法の両方を扱います。2 つ目は、リポジトリ自身のスキル（`.claude/skills/*`）の
frontmatter に**既定の `model:` を埋め込む**ことです。こうすれば経済的な選択が自動で適用され、切り替えを覚えておく
必要がなくなります。あくまで指針として運用し（難しい個別ケースではセッションがいつでも上位モデルへ引き上げられます）、
決定的な `run` / CI ゲートには手を触れず、そこは引き続きモデルを呼びません。

## 動機

このリポジトリは、他に比べてもエージェント主導の度合いが高いのが特徴です。`CLAUDE.md`、各スキル（`ideation`、
`implement-be`、`japanese-tech-writing`）、サブエージェントへのファンアウトはいずれも日々の作業の一部で、多数の
並行セッションが動かします。したがってトークン消費は現実の反復コストであり、いまはその使い方が一律です。

1. **セッションに引き下げを促すものが何もありません。** ハーネスには必要なつまみがすでに揃っています。スキルや
   エージェントごとの `model:` frontmatter、Agent ツールの `model` 上書き、対話的な `/model` / `/fast`（エフォート）
   がそれです。ところがリポジトリ内のどのスキルも `model:` を宣言しておらず、どのフェーズにどの能力が要るかを述べた
   規約も文書化されていません。その結果、セッションは起動時のまま（多くは最上位モデルの高エフォート）で全作業を通します。

2. **タスクの負荷には大きな幅があります。** BE 項目の実装、軽くないリファクタリング、設計上の判断には、確かに高
   エフォートの高性能モデルが要ります。一方、ロードマップ index の再生成、ドキュメントのリンク修正、機械的なリネーム、
   翻訳の下書きにはそれは要りません。小さいモデルの低エフォートで、はるかに少ないトークンから同じ成果物が得られます。
   軽いタスクを最上位で動かすのは、そのまま無駄になります。

3. **失敗の出方が非対称で、しかも静かです。** 過剰なモデル割り当てはトークンを目に見えず浪費します（出力は問題なく
   見えます）。逆に不足は、悪い結果としてはっきり表面化します。そのため自然な流れは*常に最上位*へ傾き、これこそ取り除き
   たい無駄です。とはいえ、どこでも無闇に引き下げれば、難しいタスクで品質を損ないます。手当ては、その場限りの推測では
   なく、**明文化された対応関係**と**妥当な既定値の埋め込み**です。

これを埋めれば、「簡単な作業には安いモデルを」という、勤勉なセッションが覚えていれば実践する程度の口伝が、既定値が
自ら適用される明文の規約に変わります。

## 詳細設計

高度としてはコントリビューターの作業フローに属し、触れるのは `.claude/` のスキルのメタデータとドキュメントだけです。
作業は次の 4 つで MECE に分かれます。

### 1. タスク → 能力の対応表

繰り返し発生する開発タスクを**能力段階**へ対応づける唯一の表を定めます。軸は 2 つ、**モデル**（高性能 ↔ 安価）と
**推論エフォート**（高 ↔ 低/なし）です。この表を単一の出典として一度だけ文書化し、他の作業がそれを反映します。初期の
段階分けは次のとおりです。

- **重（高性能モデル、高エフォート）**：BE 項目の実装（`implement-be`）、軽くないリファクタリング、アーキテクチャや
  設計の判断、ゲート失敗のデバッグ。
- **中（中位モデル、中エフォート）**：ロードマップの発想と起草（`ideation`）、日本語の技術文書執筆と翻訳のレビュー
  （`japanese-tech-writing`）、PR レビュー。
- **軽（安価モデル、低エフォートまたはエフォートなし）**：ロードマップ index の再生成や promote、ドキュメントの整形と
  リンク修正、機械的なリネーム、ロックファイルや整形の雑務、人間や中段階のレビューに回す前の翻訳の下書き。

具体的なモデル id は、実装時点の Claude のラインアップに合わせて選びます。表はタスクごとの*段階*と、段階からモデル id
への対応を固定するので、新しいモデルへ差し替えるときは 1 箇所を 1 行直すだけで済みます。

### 2. リポジトリ内スキルへの既定 `model:` frontmatter

リポジトリ内の各スキルの `SKILL.md` frontmatter に、対応表（作業 1）の段階に合わせた `model:` を追加します。
`implement-be` は重、`ideation` は中、`japanese-tech-writing` は中です。ハーネスはスキル実行時にこの値を直接読むので、
既定は何も覚えていなくても効き、しかも上書き可能です（難しい個別ケースではセッションが上位モデルへ引き上げられます）。
これは、ドキュメントだけの指針よりも「正しい選択を自動にする」半分を提案者が明確に選んだ結果です。

### 3. `docs/ai-development.md` へのフェーズとサブエージェントの指針（日英）

スキルの frontmatter では届かない、対話的な作業と委譲した作業のために、「どのフェーズにどのモデル/エフォートか」の短い節を
`docs/ai-development.md` とその `docs/ja/` 版に追加します。

- **セッション内のフェーズ**：探索、調査、機械的な雑務では引き下げる（あるいは `/fast`）。実装と設計では引き上げる。
- **サブエージェントへの委譲**：Agent ツールでサブエージェントを起動するときは、駆動側ではなく*委譲する*タスクに合った
  `model` を渡します。広く展開する `Explore` のファンアウトや index の再生成は、駆動しているセッションより安いモデルで
  回せます。これは、ファイルを所有していないリポジトリ外のレビュー用プラグイン（`pr-review-toolkit`）に対して唯一効く
  つまみでもあります。frontmatter ではなく起動時にモデルを指定します。

`CLAUDE.md` からこの新しい節を指し、作業合意の側から辿れるようにします。

### 4. 指針にとどめ、ゲートで強制しない

セッションがどのモデルを使ったかは**差分から復元できません**。そのためゲートでは検証できませんし、固定的に縛ると、
本来は軽いタスクの難しい個別ケースで人間が引き上げる余地を奪います。本項目は文書化した既定値と frontmatter として
出荷し、意図的に CI チェックにはしません。機械的に確認する価値がある唯一の面は、各スキルの `model:` の値が構文として
妥当な既知のモデル id であること（打ち間違いを黙って握りつぶさず、ローカルで落とすこと）です。規約そのものは指針の
ままで、これはコントリビューターの作業フローの残り（BE-0069）が「手順をコマンドに」しつつゲートで強制する方針では
ないのと一貫します。

### プライム・ディレクティブとの整合

完全にコントリビューターの作業フローの範囲です。編集するのは `.claude/` のスキルのメタデータとドキュメントだけで、
製品コードや `bajutsu/` には手を触れません。決定的な `run` / CI ゲートはそのままです。*開発*セッションが何で動こうと
ゲートはモデルを呼ばないので、本項目が LLM を判定役に近づけることはなく（ディレクティブ 1）、決定性（ディレクティブ 2）
やアプリ非依存（ディレクティブ 3）とも関わりません。

## 検討した代替案

- **何もせず、各セッションの手動選択に任せる。** 却下しました。無駄が静かに出る非対称性ゆえに、既定は常に最上位へ流れ、
  明文化されていない規約は守られません。これは本項目が正そうとしている現状そのものです。
- **文書化だけで、既定 frontmatter を入れない。** 提案者が検討のうえ見送りました。ドキュメントだけの規約は、あらゆる
  セッション（人間もエージェントも）が切り替えを覚えていることに依存します。既定をスキルの frontmatter に埋めれば
  経済的な選択が自動になり、frontmatter では表せないフェーズやサブエージェントの指針はドキュメントが担います。
- **モデル選択をゲートで強制する。** 却下しました。使ったモデルは差分から復元できず、固定的な縛りは「軽い」タスクが
  難しかったときに人間が引き上げる判断と衝突します。指針と既定値で、硬直させずに節約の大半を得られます。
- **独自のモデルルーターや自動選択を作る。** 過剰設計として却下しました。ハーネスはスキルごと、サブエージェントごとの
  モデル選択と対話的なエフォート制御をすでに備えています。欠けているのはそれらのつまみが*ない*ことではなく、使って
  いないことです。
