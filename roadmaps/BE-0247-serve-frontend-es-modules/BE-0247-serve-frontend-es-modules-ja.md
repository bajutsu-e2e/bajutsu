## はじめに

BE-0202 は、それまで一体だった `bajutsu/templates/serve.js` を `serve.core.js`、
`serve.panels.js`、`serve.crawl.js`、`serve.metrics.js`、`serve.author.js` という 5 つの
セクション別ファイル（合計で約 3,200 行）に分割しました。それぞれが独立したファイルとして
読み書き・差分確認できるようにするための分割です。`bajutsu/serve/handler.py` はこれらを
固定順（`_JS_ASSETS`、`handler.py:657-663`）で連結し、1 つの `<script>` タグへ埋め込んで
います。5 つのファイルは、1 ファイルだったころと同じく、モジュールシステムを持たない単一の
グローバルスコープを共有したままです。本項目は、このセクション間の区切りに ES モジュールという
実在する言語機構を与え、あるファイルが何に依存しているかを、手作業で保守するタプル内の位置では
なく、そのファイル自身に宣言させます。

## 動機

分割後も `serve.core.js`（約 1,017 行）と `serve.author.js`（約 1,079 行）は単体で見て
大きいままです。加えて、状態はモジュールレベルの可変な `let` グローバルとして存在し、
セクション間に強制力のある境界がないため、どのファイルも他のどのファイルの状態でも読み書き
できてしまいます。分割後にも残っている例を挙げると、`serve.core.js:35-41` とその設定関連の
セクションにある `poll`・`recPoll`・`selectedRun`・`recPath`・`scnFiles`・`targets`・`sims`・
`recJobId`・`runJobId` や `keyState`・`ccTokState`・`provState`・`gitCredState`・
`projectsCache`、`serve.crawl.js` の `crawlPoll`・`crawlJobId`・`crawlRunId`・
`crawlGraphData`、`serve.metrics.js` の `metricsCache`・`metricsSort` などです。
`serve.author.js` が `serve.crawl.js` の `crawlPoll` を読みにいくことも、将来追加される
セクションが `selectedRun` を上書きしてしまうことも、言語レベルでは止められません。今 5 つの
ファイルの一貫性を支えているのは、たまたま `_JS_ASSETS` が定義を使用より先に並べる順序に
なっているという事実だけです。

これは重複したロジックの問題ではありません。各セクションが呼び出す共通のプリミティブ
（`getJSON`・`postJSON`・`streamJob`・`startJob`・`cancelJob`・`setStatus`・`esc`）は、
BE-0202 の `startJob` 統一によって、すでに `serve.core.js:48-96` に一本化済みです。残っている
負債はもっと狭い範囲、すなわちフラットなグローバル名前空間と、使う側からは見えないのに
挙動を左右している読み込み順です。本プロジェクト自身の ESLint ガードレール（BE-0129、
`eslint.config.mjs`）も、ツール側から同じ隙間を記録しています。`no-undef` を意図的に無効化
している理由は 2 つあります。主たる理由は、有効にするとブラウザおよび ES のグローバル一式を宣言
する必要があり（`globals` という npm パッケージ）、この Python リポジトリに Node のツールチェーンを
持ち込むことになる、というものです。理由書きはそのうえで副次的な点として、「ESLint はセクション別
ファイルが共有するファイル横断のグローバルが見えないため、いずれにせよ誤検知になる」と付け加えて
います。グローバルの衝突や読み込み順の入れ替えミスを検出する
lint が存在しないという事実こそ、本項目が取り除こうとしている脆さであり、UI にパネルが増える
ほど、この脆さは改善どころか悪化します。

## 詳細設計

1. **ネイティブの ES モジュールを採用します。** 各セクション別ファイルに公開面を `export`
   させ、他のファイルから必要なものを `import` させます（`serve.core.js` が
   `export function getJSON(...)` のような最初のモジュールになり、`serve.author.js` が
   エントリモジュールになります）。ビルドステップは不要です。ブラウザは
   `<script type="module">` をネイティブに実行でき、これは localhost 向けの開発ツールで
   あり、BE-0202 が保った「bundler なし」の方針をそのまま維持できます。サーバ側で実際に
   変わるのは配信方法だけです。`import` 指定子は URL に対して解決されるため、5 つのファイル
   はもう `_JS_ASSETS` のように 1 つのインライン `<script>` へ連結できません。代わりに
   `bajutsu/serve/handler.py` が各セクションをそれぞれのパスで配信し（`bajutsu/templates/
   serve.*.js` を読む数個の静的な `GET` ルートで、`serve.html.j2` がすでに各ファイル名を
   把握しているのと同じ発想であり、新しいアセットパイプラインではありません）、ページは
   エントリモジュールを `<script type="module" src="...">` で読み込みます。localhost では
   1 つのインラインの塊の代わりに小さなリクエストが 5 つに増えるだけで、コストは無視でき、
   気になる場合は `modulepreload` リンクで和らげられます。
2. **段階を踏む場合の最小構成として、明示的な名前空間を用意します。** (1) の配信方法の変更が
   一度に行うには大きすぎると判断される場合、まずこちらだけを先に着地させます。各セクションを
   IIFE で包み、トップレベルの裸の `let`・`function` 宣言の代わりに、`window.bajutsu.crawl =
   {...}` のような明示的な名前空間オブジェクトを代入します。今のとおり 1 つに連結・インライン
   化した `<script>` のままで構いません。これにより、ファイル横断の読み取りは、どのファイルの
   ものか判別できない裸の識別子ではなく、`bajutsu.crawl.crawlJobId` のような明示的なプロパティ
   参照になります。これは (1) の厳密な部分集合であり、(1) の準備が整った時点で、そのまま実際の
   `import`・`export` へ移行できます。
3. **読み込み順を、慣習ではなく強制されたものにします。** (1) と (2) のどちらを取るにせよ、
   目指すところは同じです。`_JS_ASSETS` のタプルの並び順という契約を、言語かツールのどちらかが
   検査できるものに置き換えます。(1) を採用した場合、`import` のグラフそのものが読み込み順に
   なります。`import` されていないモジュールは単に読み込まれず、依存が欠けていれば、どこか
   わからないグローバル参照の失敗ではなく、その `import` 文自体での読み込み時の
   `ReferenceError` になります。変換したファイルについては `eslint.config.mjs` の
   `languageOptions.sourceType` を `"script"` から `"module"` に変更して ESLint が
   `import`・`export` 構文を解釈できるようにし、各ファイルの入力が宣言されるようになった
   段階で、リポジトリ全体で無効化している `no-undef` をファイル単位で再度有効にできないかも
   合わせて検討します。ただしこれで取り除けるのは副次的な障害（ファイル横断のグローバル）だけです。
   `no-undef` を再度有効にすると、主たる障害である「ブラウザのグローバル（`window`、`fetch`、
   `document` など）を `globals` パッケージで宣言すること」を解決しない限り、それらの素のグローバルで
   なお誤検知します。したがって入力を明示するのは必要条件ではあっても、それだけでは十分ではありません。

**検証について。** これはコードの構造的な整理であり、決定的な `run` / CI ゲートが実行する
コードではないため（プライムディレクティブ 1 により、AI と UI のコードはその経路から外れて
います）、`make lint-js`（各ファイルへの `node --check`）と既存の serve dogfood シナリオ
（BE-0058、BE-0189）を超える自動的な合否判定はありません。確立した手順に従い、各パネル
（record、replay、triage、crawl、metrics、author）を実際に読み込み、変更の前後で挙動が同じ
であることをアプリ内ブラウザで手作業により再現・確認したうえで判断します。コードの読解だけで
判断しません。serve の JS/CSS の変更は、コードレビューだけでの判断が誤った修正を出荷したことが
過去にあるためです。UI の挙動は変わりません。本項目が変えるのは、フロントエンドのコードの
組み立て方と読み込まれ方だけです。

## 検討した代替案

- **bundler（Webpack、esbuild、Rollup など）の導入。** BE-0202 が同じ選択肢を不採用とした
  理由と同じ理由で不採用とします。ここは Python のリポジトリであり、この UI の価値は
  ツールチェーン不要の単純さにあります。ネイティブの ES モジュールなら、ビルドステップも
  `node_modules` のツリーも、ソースと同期を保つ必要のあるコンパイル済み成果物も持ち込まず
  に、依存関係を明示できるという同じ利点が得られます。
- **フラットなグローバルスコープを現状のままにする。** 本項目が取り組む現状そのものであり、
  不採用です。すでに 2 つのファイルが大きく、ファイル横断の状態には強制力のある境界がなく、
  serve に機能が追加されるたびにパネルが増える傾向を踏まえると、フラットな名前空間は時間が
  経つほど読み解きにくくなります。
- **ネイティブモジュールへは移行せず、名前空間オブジェクトだけを導入する。** *詳細設計* (2)
  で述べた段階的な着地点ではなく、恒久的な対応として検討しました。最終形としては不採用です。
  ファイル横断のアクセスは呼び出し箇所で明示されますが、読み込み順は依然として `_JS_ASSETS`
  内の強制力のない慣習のままであり、名前空間がその読み込みより先に参照されても、何も大きな
  音を立てて失敗しません。ネイティブな `import`・`export` であれば、同じだけのコード変更量で
  明示的なアクセスと強制された読み込み順の両方が手に入るため、(2) を最終形にはせず、目指す
  対象として残します。
