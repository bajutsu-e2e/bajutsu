## はじめに

クロールグラフのノードを一つずつ任意の位置へドラッグできるようにし、あわせてグラフ全体を
自動レイアウトへ戻す**再整列**ボタンを追加します。クロールレポートの画面マップは、すでに
ズーム、パン、グループの展開と折りたたみ、エッジのハイライトに対応していますが、ノード自体は
BFS の階層グリッドに固定されたままです。本提案は、ノードを自由に配置できるという自由度を一つ
加えるものです。クロールエンジンやランナーには手を入れず、変更はレポートの描画層に限られます。

## 動機

画面マップは決定的な BFS グリッドとしてレイアウトされます。BFS の深さごとに列を作り、列の
なかでユニットを縦に積みます。このレイアウトは再現性があり、小さなマップでは読みやすいものの、
実際のアプリではすぐに混み合います。エッジが交差し、関連する画面が離れて配置され、二つのノードを
並べて見比べたり、読みながら混雑した領域をほどいたりする手立てがありません。ズームとパンはビュー
全体をまとめて動かすだけで、あるノードを隣のノードに対して動かすことはできません。

ノードをドラッグできれば、マップは静的なスナップショットではなく作業の場になります。検討中の
画面どうしを引き寄せたり、重なって埋もれたノードを引き出してラベルを読んだり、分岐を自分の
納得する形に並べ替えたりできます。ただし、並べ替えたあとには元の標準的なレイアウトへ一度で
戻したくなります。これが再整列ボタンの役割で、手動の移動をすべて破棄して自動の BFS グリッドを
復元します。

この機能はプライムディレクティブと衝突しません。レポートの閲覧 UI に閉じた変更であり、LLM は
関与せず、決定的な `run`/CI ゲートにも触れず、アプリ固有の要素もありません。同じ描画があらゆる
ターゲット（iOS、Web）に使われます。むしろ「まず決定性」を補強します。自動レイアウトは標準で
再現可能な配置として残り、再整列はつねにそこへ戻すからです。

## 詳細設計

変更はすべて、レポートのグラフ描画を担う
[`bajutsu/templates/serve.js`](../../bajutsu/templates/serve.js) に入ります。レイアウト処理は
すでに `pos`（各ユニット id から算出した `{x, y}` への `Map`。id は折りたたんだグループなら
`g:<key>`、ノードやメンバーなら `n:<fingerprint>`）を計算しています。エッジ、グループの枠、
ユニットのボックスはいずれも `pos` を読んで配置されるため、あるノードの `pos` のエントリを上書き
すれば、そのノードが動くと同時に、付随する要素もすべて自動的に追従します。

**手動位置の上書き。** 既存の `expandedGroups` 集合や `gview` オブジェクトと並べて、ユニット id を
キーとするモジュールスコープの `const nodeOverrides = new Map()` を追加します。`renderGraph` では、
算出した `pos` を組み立てたあとに上書きを重ねます
（`nodeOverrides.forEach((p, id) => { if (pos.has(id)) pos.set(id, p) })`）。これをエッジ・枠・
タイルの出力より前に行うため、動かしたノードはベジエの端点もグループの枠もハイライトの配線も
一緒に持っていきます。利用側ごとの個別対応は要りません。この `Map` は（`gview` や `expandedGroups`
と同じく）モジュールスコープなので、クロールがストリーミングされるあいだ手動位置は**ポーリング
ごとの再描画をまたいで保持されます**。既存のノードはユーザーが置いた位置にとどまり、新しく
見つかったユニットだけが算出された位置に収まります。ページを再読み込みすると状態はクリアされます
（`localStorage` への永続化はしません）。これは意図的なスコープの線引きです（代替案を参照）。

**ノードのドラッグ。** ノードのドラッグはキャンバスのパンとは別のジェスチャーとして扱います。
`#crawl-graph` の既存の `mousedown` ハンドラで対象によって分岐し、押した位置が `.gnode` の内側
だった場合はパンではなく*ノードの*ドラッグを開始して、そのノードのユニット id（`data-uid`）と
開始点を記録します。それ以外は今日どおりのパンの挙動を保ちます。`mousemove` では、カーソルの
移動量を現在のズーム（`gview.k`）で割って内容座標に変換し（内側の層は CSS の `zoom` で拡大される
ため、画面ピクセルと内容ピクセルはちょうどこの係数だけ異なります）、ドラッグ中のノードの
`style.left/top` をその場で更新します。フレームごとに全体を再描画せず滑らかに動かすため、更新する
のは関係するエッジのパス（`data-a`/`data-b` がドラッグ中のユニット id に一致するもの）だけにし、
その `d` をノードの新しい位置から計算し直します。他の要素には触れません。`mouseup` で、確定した
内容座標を `nodeOverrides` に書き込みます。

**クリックとドラッグの区別。** 確立済みの `moved` のしきい値（押下点から約 3px を超えてずれたら
ドラッグとみなす）を再利用します。ずれなかったノードの押下は既存の `click` ハンドラへ落ち、
ノードをタップすればこれまでどおりライトボックスが開き、グループの展開・折りたたみのボタンも
機能します。クリックを抑止するのは本当にドラッグしたときだけです。

**再整列ボタン。** 既存のズームボタン（`#crawl-zoomin` / `#crawl-zoomout` / `#crawl-zoomreset`）
の隣にコントロール（たとえば `#crawl-realign`）を追加し、そのハンドラで
`nodeOverrides.clear(); redrawGraph()` を実行します。これで手動の移動をすべて捨て、純粋に算出した
`pos` から再描画し、グラフ全体を標準の BFS レイアウトへ戻します。`resetView`（ズームとパンだけを
戻す）とは独立しているので、ズームを失わずにノードを再整列することも、手動レイアウトを失わずに
ビューを戻すこともできます。

**グループの展開・折りたたみとの関係。** グループを展開する（`g:<key>` がメンバーごとの
`n:<fingerprint>` に変わる）か折りたたむと、ユニット id が変わります。上書きはユニット id を
キーにしているため、ユニットの同一性が変わると上書きは単に一致しなくなり、そのノードは算出した
位置にフォールバックします。これは妥当で予測しやすい挙動です。グループの手動位置をメンバーへ
引き継ぐことはしません。

**タッチ。** 本項目の核はポインター（マウス）でのドラッグです。これをタッチに対応させること
（ノード上での一本指ドラッグはノードを動かし、背景上のドラッグは従来どおりパンする）は、同じ
上書きと確定の経路を再利用する自然な拡張で、ポインターの経路が入ったあとに続けるのがよいでしょう。
コストが低ければ同じ変更で出してもよく、直後のフォローアップとして出してもかまいません。

## 検討した代替案

- **手動位置を再読み込みをまたいで永続化する（run ごとに `localStorage`）。** より重くなります。
  run id ごとの保存キー、初回描画時の読み込み経路、破棄の仕組みが要ります。見送ります。よくある
  用途は閲覧やクロールのセッション*中*の並べ替えで、これはモジュールスコープの状態ですでに
  まかなえます。上書きのモデルを変えずに、あとから積み増せます。
- **力学的・物理ベースのレイアウト（ドラッグ後にばねで戻る、自動でほどける）。** これはより大きく、
  別種の機能です。決定的な BFS グリッドを確率的なレイアウトに置き換え、物理ループかグラフ
  ライブラリ（描画は意図的にライブラリ非依存です）を持ち込むことになります。ここではスコープ外
  とします。要望は明確に*自由な*配置と、標準へ戻すボタンです。
- **ドラッグ中のグリッドへのスナップや整列ガイド。** 使い勝手の向上であって、核となる要望には
  必須ではありません。「離した位置にそのまま置く」というジェスチャーを保つため省きます。あとから
  同じドラッグ経路の上に追加できます。
- **ノードごとの「このノードだけ戻す」（全体の再整列に対して）。** 採用するスコープは全体の再整列
  一つです。右クリックで「このノードを元の位置へ戻す」は将来の改良としてあり得ます。`nodeOverrides`
  からキーを一つ削って再描画するだけです。
