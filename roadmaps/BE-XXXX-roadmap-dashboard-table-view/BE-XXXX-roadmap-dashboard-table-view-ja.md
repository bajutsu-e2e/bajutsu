## はじめに

ロードマップステータスダッシュボード（[BE-0094](../BE-0094-roadmap-status-dashboard/BE-0094-roadmap-status-dashboard-ja.md)
で公開し、[BE-0219](../BE-0219-roadmap-dashboard-search/BE-0219-roadmap-dashboard-search-ja.md)
で検索を追加した GitHub Pages のページ）に、カードグリッド表示の代替としてテーブル表示を追加します。
読者は Cards と Table を切り替えられ、Table では各項目が1行になり、ID・タイトル・トピック・状態・
作成日・更新日の6つのソート可能なカラムを持ちます。カラム見出しをクリックすると、テーブル全体が
そのカラムで昇順・降順にソートされます。選んだ表示モードはブラウザの `localStorage` に保存されて
次回訪問時にも復元され、既存の検索ボックスとステータスチップは両方の表示モードでそのまま機能します。
作成日と更新日は、これまでダッシュボードになかった2つのカラムです。それぞれは各項目の2つの言語
ファイルにまたがる最初のコミットと最新のコミットから、ビルド時に Git の履歴を読んで導出します。
手作業で保守するフィールドではなく実体から導出するという、ダッシュボードがすでに採用している方針
（トピックごとの進捗率も `状態` フィールドのみから導出しており、独自の完了率を持ちません）に
合わせています。

## 動機

カードグリッドはトピックごとに項目をグループ化し、各項目の状態をバッジで示す表示なので、1つの
カテゴリを順に眺める用途には向いています。一方で、別の問いには答えられません。最近更新された項目は
どれか、あるいはトピックを問わずID順に並べるとロードマップ全体はどう見えるか、という問いです。
カードグリッドにはソートの軸となる列が存在せず、ソートは全項目を1つのフラットなテーブルの行として
扱って初めて意味を持ちます。

貢献者の作業には、まさにこのフラットでソート可能なテーブルを必要とする場面が2つあります。最近の
変更を確認する作業では、更新日でソートして上位を見ればよく、現状のようにカテゴリを1つずつ開いて
ページに存在しない日付を目で比較する必要はなくなります。提案がどれだけ放置されているかを確認する
作業では、作成日でソートします。どちらも現状のカードレイアウトでは満たせず、ブラウザの Ctrl+F や
ダッシュボード自身の検索ボックスも解決しません。両者とも「どのカードを表示するか」を絞り込むだけで、
「どの順序で並べるか」は変えないからです。

これはドキュメント・貢献者ワークフロー上の改善にとどまり、いずれのプライムディレクティブ
（[`CLAUDE.md`](../../CLAUDE.md)）にも抵触しません。ダッシュボードは静的な docs サイトの成果物なので
大規模言語モデル（LLM）は一切関与せず、`run` や CI の合否判定の経路も、決定性に関わる面も存在せず、テーブルはメタデータが
持つ項目に対して汎用的に働くため、アプリ固有の要素も含みません。

## 詳細設計

ダッシュボードは [`scripts/build_roadmap_dashboard.py`](../../scripts/build_roadmap_dashboard.py) が
生成しています。`render_html` は `.be-cat` というトピックごとのセクションの中に項目ごとの `.be-card`
を1つずつ出力し、各セクションは `状態` のみから導出した進捗バーを持ち、`<script>` ブロックが検索
ボックスとステータスチップ（BE-0219）をプログレッシブエンハンスメントで有効化しています。テーブル表示は
この既存の構造を置き換えるのではなく、そこへ追加する形で組み込みます。作業は MECE
（互いに排他的で全体として網羅的）な次の単位に分解できます。

1. **Git 履歴からの作成日・更新日の導出。**
   [`scripts/build_roadmap_index.py`](../../scripts/build_roadmap_index.py) の `Item` に、ISO 8601
   形式の日付を2つ追加します。ビルド時に `git log --follow --format=%aI -- <path>` を、英語ファイルと
   `-ja.md` ミラーのそれぞれに対して1回ずつ実行し、2つの結果を合わせます。`created` はどちらかの
   コマンドが報告するもっとも古い行（各ファイルの最初のコミット）、`updated` はもっとも新しい行（いずれかの
   ファイルの最新のコミット）です。2つのファイルを別々に、それぞれ `--follow` 付きで実行するのは意図的な
   設計です。`--follow` は1つのパスしか受け付けませんが、CI が実際の ID を割り当てる際に全項目が経る
   リネーム（`BE-XXXX-<slug>` → `BE-NNNN-<slug>`）を越えて、そのファイルの履歴を正しく追跡できます。
   さらに両方のファイルの結果を合わせることで、たとえば日本語側だけの表現の修正のように、2つのうち
   片方だけが更新された場合にも `updated` を正しく保てます。ダッシュボードが示す事実は現状すべて
   実体から導出されており（トピックごとの進捗率も `状態` のみから導出し、独自の完了率は持ちません）、
   この設計はその原則を保ちます。将来の編集で実際の履歴からずれうる `作成日` や `更新日` のフィールドを、
   どの項目ファイルにも追加しません。ただし1つ前提が加わります。
   [`.github/workflows/docs.yml`](../../.github/workflows/docs.yml) の checkout ステップを、現状の
   浅いクローンから完全な履歴を取得する `fetch-depth: 0` に変更する必要があります。1コミットだけの
   checkout では、`git log` がすべての項目を同じ日に作成・更新されたものと報告してしまうためです。
   その代償は、以後のすべての docs ビルドで checkout にかかる時間がわずかに増え続けることであり、
   一度きりのコストではありません。項目1つにつき `git log --follow` を2回実行することにも、もう1つ
   別の継続的なコストが加わります。現在のロードマップ全体では約600回の実行になります。どちらも、
   すでに取得済みの履歴に対するローカルな読み取りでネットワーク通信を伴わないため、checkout 自体に
   比べれば無視できる範囲にとどまると見込んでいます。ただしビルドを目に見えて遅くするようであれば、
   リポジトリ全体を1回だけ `git log --name-status -M` で走査し、全項目の日付を1パスで解決する方式に
   まとめるほうが安く済みます。
2. **表示モードの切り替え。** 既存の `.be-filters` 行の隣に、`Cards` と `Table` を切り替える
   コントロールを配置します。両方の表示モードは同じレンダリング済みの項目データを読むだけで、切り替えは
   兄弟関係にある2つのコンテナ（`.be-cards-view` / `.be-table-view`）の一方を表示し他方を隠すだけです。
   読者が切り替えるたびに何かを取得し直したり再計算したりすることはありません。
3. **テーブルのマークアップ。** 項目集合ごとに `<table>` を1つ、項目ごとに `<tr>` を1つ描画し、
   ID・タイトル・トピック・状態・作成日・更新日の6つの `<th>` を持たせます。各 `<th>` には
   `data-sort-key` とクリックハンドラを付けます。各カードがすでに持つフィールド（id・タイトル・
   トピック・状態）をそのまま再利用し、2つの日付カラムを加えます。各行は対応するカードと同じ
   `data-status` / `data-topic` 属性を持つため、検索ボックスとステータスチップ（BE-0219）は、
   別のマッチングロジックを新設することなく、カードと全く同じようにテーブルの行を絞り込みます。
   Cards 表示で読者が頼りにしているトピックごとの進捗バーも失われません。既存の `_progress_bar` の
   計算をそのまま再利用し、全トピックとその積み上げバー・パーセンテージを列挙するコンパクトな帯を
   テーブルの上に描画します。
4. **クライアントサイドのソート。** 既存のフィルタスクリプト（`_SCRIPT`）にソート処理を追加します。
   `<th>` をクリックすると、そのカラムでテーブルの `<tr>` を並べ替えます（タイトル・トピック・状態は
   文字列比較、ID は数値としての比較、作成日・更新日は日時としての比較）。同じカラムを再度クリックすると
   昇順・降順が切り替わり、現在ソート中のカラムと向きは `aria-sort` 属性で示します。ソートは、すでに
   ステータス・検索フィルタを通過して表示されている行の並び順を変えるだけで、どの行を表示するかには
   一切影響しません。
5. **表示モードの永続化。** Cards / Table を切り替えるたびに、その選択を専用のキーで
   `localStorage` に書き込みます。ページ読み込み時にはそのキーを読み、最初の描画より前に対応する
   表示モードを反映します（キーが存在しない場合や `localStorage` が使えない場合は、現状のデフォルトで
   ある Cards にフォールバックします）。ダッシュボードの他のインタラクティブな要素と同様にプログレッシブ
   エンハンスメントとして働き、JavaScript が無効な環境では現状と変わらず常に Cards 表示になります。
6. **スタイリング。** テーブル・切り替えコントロール・進捗帯の CSS を `_STYLE` に追加します。既存の
   チップやカードのスタイルに合わせ、MkDocs のライトテーマ・ダークテーマの両方で読みやすくします。
7. **テスト。** [`tests/test_roadmap_dashboard.py`](../../tests/test_roadmap_dashboard.py) を拡張し、
   各項目が6カラムを持つ行を1つ描画すること、各日付カラムの値が ISO 8601 形式であること、表示モードの
   切り替えコントロールと2つのコンテナが存在すること、ソートスクリプトが各 `<th>` にクリックハンドラを
   配線していることを検証します。これが機械的に検証可能な成果物であり、ブラウザを必要とせず、ゲートを
   高速かつ LLM 抜きに保ちます。

ページが**内容として持つもの**は変わりません。2つの日付カラムが増える以外、項目集合とそのフィールドは
そのままで、同じ項目群を並べる方法がもう1つ増えるだけです。`docs/architecture.md` / `docs/ja/` の
更新は不要です。これらの文書が説明しているのはダッシュボードの目的であって表示モードの選択肢ではない
ためです。もしそこにダッシュボードの表示モードを列挙する一文があれば、同じ PR で更新します
（[BE-0113](../BE-0113-design-doc-realignment/BE-0113-design-doc-realignment-ja.md)）。

## 検討した代替案

- **切り替えではなく、カードグリッドをテーブルで完全に置き換える。** 却下しました。カードグリッドが
  すでにうまく提供しているトピックごとの進捗バーとカテゴリ単位の閲覧という使い方が失われるか、
  テーブルの行の中で作り直す必要が生じ、この項目が狙うソートという用途には何の利点もありません。
  切り替え式であれば、両方の使い方を損なわずに済みます。
- **`ideation` / `implement-be` スキルが更新のたびに手で書き換える、`作成日` / `更新日` の
  メタデータフィールドを項目ファイルに追加する。** 却下しました。既存の約300項目に対する一度きりの
  バックフィルが必要なうえ、以後のすべての編集がフィールドと実際の履歴とのずれを生む機会になります。
  これはまさに BE-0094 以来ダッシュボードが避けてきた種類の手作業フィールドです（トピックごとの
  進捗率を `状態` のみから導出しているのも同じ理由です）。両方の日付をビルド時に `git log` から
  導出すれば、ずれのリスクはなくなります。代償は、docs ワークフローの checkout が今後すべてのビルドで
  完全な履歴取得になり、その分だけわずかに時間が延び続けることです。
- **フラットなテーブルではなく、トピックごとのカードセクションの中だけでソートする。** 却下しました。
  「最近何が変更されたか」と「この提案はどれだけ放置されているか」という2つの動機はどちらもロードマップ
  全体にまたがる問いであり、ソートを1つのトピックの中に閉じ込めても、トピックをまたいで比較するには
  結局すべてのセクションを手で開く必要が残ります。フラットなテーブルであれば、どちらの問いにも1回の
  ソートで答えられます。
- **サーバーサイド・ビルド時のソート（生成したページに固定の並び順を焼き込む）。** 却下しました。読者が
  見たい並び順はそのときどきで変わり、ID 順のときも更新日順のときもあります。静的なページに焼き込める
  並び順は1つだけです。すでに描画済みの行に対するクライアントサイドのソートであれば、ビルド時の
  コストなしに読者が欲しいどの並び順にも応えられます。

## 進捗

> 開発の進行に合わせて常に最新の状態に保ってください。チェックリストは *詳細設計* の MECE な
> 作業分解（作業の単位ごとに 1 つ）に対応し、ログには変更内容と時期（古い順）を PR へのリンクと
> ともに記録します。

- [ ] `Item` への Git 履歴由来の `created` / `updated` の追加と、`docs.yml` の checkout の
      `fetch-depth: 0` への変更。
- [ ] 既存のフィルタ行の隣に表示モード切り替え（`Cards` / `Table`）を描画する。
- [ ] テーブルのマークアップ: 項目ごとに1行、6つのソート可能なカラム、テーブル上部にトピックごとの
      進捗帯を維持する。
- [ ] 各カラム見出しへのクライアントサイドソートの配線。既存の検索・ステータスフィルタと組み合わせる。
- [ ] `localStorage` による表示モードの永続化（デフォルトは Cards）。
- [ ] ライトテーマ・ダークテーマ両方のスタイリング。
- [ ] `tests/test_roadmap_dashboard.py` へのテスト追加。
