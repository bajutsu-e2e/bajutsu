## はじめに

[BE-0073](../BE-0073-serve-zip-bundle-upload/BE-0073-serve-zip-bundle-upload-ja.md) では、ブラウザの利用者が `.zip` バンドルをアップロードすると、`serve` がそれを展開し、Replay / Record / Crawl タブが実行に使う**アクティブな config** としてバインドします。アップロードされた config は信頼できない入力です。BE-0073 はそのうち一つをすでに拒否しており、ターゲットの `build` コマンドはホスト上で**決して実行しません**（バンドルはビルド済みバイナリを同梱するためです。[DESIGN §1](../../DESIGN.md)）。しかし config には、同じく `serve` ホスト上でシェルコマンドを実行するフィールドがほかにもあります。主なものは `launchServer.cmd` で、こちらは**統制されていません**。バインドしたバンドルを実行すると、`bajutsu run` のサブプロセスが、運用者の環境ごと、素のホスト上でそのコマンドを記述どおりに起動します。

本項目は、その素のホストでの実行を `sandbox` モードに置き換えます。アップロードされたコマンドを使い捨ての **Docker** コンテナの中で実行し、`serve` ホストに触れさせないモードです。あわせて、そもそもコマンドを実行する*かどうか*を決める明示的なポリシー（`deny` / `reuse` / `sandbox`）を組み合わせます。素のホストに到達するモードはもうありません。アップロードされたサーバを実行することが、マルチテナントのホストでも許せるものになります。実行されるとしても、その唯一の経路が封じ込められているからです。

## 動機

BE-0073 はアップロードされたバンドルをアクティブな config としてバインドし、それに対して決定的な `run` を実行します。config 内でシェルコマンドを指すフィールドは 3 つありますが、いずれも現に動いているわけではありません。

- `build`：`appPath` のオンデマンドビルド（`bajutsu/serve/jobs.py` の `_build_app`）。ローカル config では実行しますが、アップロードされた config では**すでに拒否しています**。
- `launchServer.cmd`（[BE-0059](../BE-0059-launch-target-server/BE-0059-launch-target-server-ja.md)）：run のために `baseUrl` の背後にあるホストを起動します。**現に実行されている**もので、`serve` ホスト上で、専用のプロセスグループとして動きます（`bajutsu/runner/launch_server.py` の `subprocess.Popen(shlex.split(ls.cmd), env={**os.environ, …}, start_new_session=True)`）。
- `mockServer.cmd`：アプリが呼び出す依存先のモックを起動するものとしてスキーマに宣言されています（`bajutsu/config.py` の `MockServer.cmd`）が、**まだ実行器につながっていません**。モックは Playwright の network collector がプロセス内で処理するため、`cmd` は現状では動いていません。実行器が用意されれば `launchServer` と同じ形で動くことになります。

つまり、現に実行されているのは `launchServer.cmd` だけです。`mockServer.cmd` はまだ動いていませんが、実行器につながる前に同じポリシーで覆っておくべきフィールドです。BE-0073 はアップロードに対して `build` の経路は塞いでいます（アクティブな config がアップロードされたバンドルのとき、`start_run` が `build=None` を強制します）が、`launchServer` は**開いたまま**にしています。そうせざるを得ませんでした。Web バンドルには `appPath` のバイナリが**なく**、`launchServer` こそがバンドルを自己完結させる唯一の手段だからです（run 時に同梱の静的アプリを配信します）。一律にブロックできないのは、まさにこの理由によります。

この穴が問題になるのは、ラップトップ上ではなく、デプロイの境界においてです。

- **Tier-A（認証済みの単一 Mac の `serve`）。** アップロードされた config の `launchServer.cmd` を実行することは、同じバンドルが同梱するアップロード済みの `.app` バイナリを実行することと、危険度が変わりません。認証済みの運用者は、自分のスイートを持ち込んで実行することをすでに信頼されているので、ここでは素のホストでの実行も許容できてきました。しかしそれは、ホスト型では受け入れられないものと同じ仕組みです。だからこそ本項目は、素のホストへの抜け道を残すのではなく、Tier-A も含めたすべてのデプロイをサンドボックス経由に通します。
- **ホスト型 / マルチテナントの `serve`**（[BE-0015](../BE-0015-web-ui-public-hosting/BE-0015-web-ui-public-hosting-ja.md)、[BE-0016](../BE-0016-web-ui-self-hosting/BE-0016-web-ui-self-hosting-ja.md)）。認証済みだが信頼できないテナントが、`launchServer.cmd: rm -rf …`（あるいは任意の内容）を config に仕込んだバンドルをアップロードすると、それは**ホスト上での任意コマンド実行**であり、リモートコード実行の経路になります。BE-0073 は単一 Mac の Tier-A に明示的に限定し、マルチテナントの隔離を BE-0015 / BE-0016 に委ねているので、これは BE-0073 のバグではありません。ホスティングを安全にする前に、それらの項目が塞ぐべき箇所です。

*拒否*か*許可*の二択しか持たないポリシーでは、ホストのケースは行き詰まります。`deny` は、アップロード経路が存在する理由そのものである自己完結 Web バンドルの正当な用途を壊し、`allow` は素のホストでの RCE になります。欠けているのは、**安全に許可する道**です。アップロードされたコマンドを実行しつつ、たとえ悪意あるコマンドであっても使い捨てコンテナの外には届かないよう封じ込める、という選択肢です。サーバの起動コマンドを Docker で仮想化すれば、まさにこれが手に入ります。しかもこれは、すでにプロジェクトの方針にも沿っています。Tier-B のセルフホストは `docker compose` の control plane を同梱しており（[`deploy/self-host/`](../../deploy/self-host/)、[docs/self-hosting.md](../../docs/self-hosting.md)）、BE-0015 / BE-0016 はジョブ単位のコンテナ隔離を自分たちの領分として挙げています。本項目は、その隔離を具体的な実行モードとして前倒しで取り込み、その背後に置く決定的なポリシーゲートを定めます。

## 詳細設計

アップロードされた config がシェルコマンドを実行するフィールド（`build`、`launchServer.cmd`、`mockServer.cmd`）をひとまとめに扱い、`serve` レベルの単一のポリシーで判定します。さらに、コマンドをコンテナの中で実行するサンドボックスモードを加えます。判定も実行もすべて決定的で、どちらにも LLM は介在しません。設定はアプリごとではなくホスト単位で、`targets.<name>` を通じて行います。いずれも prime directives に沿った形です。

- **ポリシーの設定：3 つのモード。** `serve` のオプション `--upload-exec=<deny|reuse|sandbox>`（ホスト型バックエンドには環境変数で対応）が、*アクティブな config がアップロードされたバンドルであり*、run がこれらのコマンドのいずれかを起動しようとするときの挙動を決めます。対象はアップロード由来の config だけで、ローカルや Git ソースの config（すでに運用者が信頼しているもの）には影響しません。
  - **`deny`**：コマンドは一切実行しません。それを必要とする run は明確なエラーで失敗します（後述）。
  - **`reuse`**：アップロードされた `cmd` は実行せず、`launchServer` が既に行っている `readyUrl` の確認を通じて、`baseUrl` で既に応答しているサーバ（運用者が別途用意したもの）を*利用する*ことだけを許します。
  - **`sandbox`**：アップロードされた `cmd` を、`serve` ホスト上ではなく**使い捨ての Docker コンテナの中で**実行します。これが、安全に許可するための手段であり、既定でもあります（後述）。
- **素のホスト実行はなし、既定はサンドボックス。** アップロードされた `cmd` を `serve` ホスト上で直接実行するモードは、**意図的に設けていません**。その経路こそ本項目が塞ごうとしている RCE の入口であり、オプトインとしてすら残しません。`serve` は Docker をアップロード経路の必須依存として扱うため、`sandbox` があらゆるデプロイで既定になり、Docker が無い場合に備える必要もありません。単一 Mac の Tier-A の `serve` でも扱いは同じです。Docker は常にあり、コンテナに閉じ込めても失うものはないので、Tier-A でもアップロードされた `launchServer.cmd` はコンテナの中で実行します。アップロードされたコマンドをそもそも実行させたくない運用者は、`deny` か `reuse` を選びます。3 つのモードのどれも、素のホストには届きません。
- **`sandbox` モード：サーバ起動コマンドを仮想化する。** `sandbox` が有効で、run が `launchServer.cmd`（実行器につながれば `mockServer.cmd` も）を必要とするとき、bajutsu はホスト上で `subprocess.Popen` する代わりに、新しく立てたコンテナの中でコマンドを実行します。
  - **ランタイム（`dockerImage` か `Dockerfile`）とポートは、コードではなく config で指定する。** サンドボックスの `launchServer` は、**コンテナの作り方**を 2 通りのいずれかで宣言します。`dockerImage`（既存のベースイメージ。たとえば `node:20-slim` のように、`cmd` が必要とするランタイムを備えたもの）か、`dockerfile`（バンドル内のパス。bajutsu がこれを `docker build` で使い捨てのイメージにビルドし、それを実行します）です。フィールド名を素の `image` ではなく `dockerImage` にしているのは、ここに書くのが **Docker のイメージ参照**（`[レジストリ/]リポジトリ[:タグ | @ダイジェスト]`。たとえば `node:20-slim` や `ghcr.io/acme/web-runtime:1.4.2`）であって、アプリのスクリーンショットやアセットのような「画像」ではないことを、config 上ではっきりさせるためです。2 つのうちどちらか一方が必須です。`Dockerfile` を同梱すれば、Web バンドルは自分のランタイムを自分で記述でき、レジストリ上にベースイメージが存在することに依存せず、完全に自己完結します（その代わり、ビルドの一手間がかかります）。手頃なベースイメージで足りるなら、`dockerImage` を指名するほうが軽い道です。config はあわせて `port`（サーバが待ち受けるコンテナ内のポート。bajutsu がこれをループバックのホストポートへ公開します）も指定します。バンドルの静的ファイルは `cwd` に**読み取り専用**でバインドマウントし、`cmd`、`env`、`readyTimeout` はそのまま引き継ぎます。`dockerImage` も `dockerfile` もなければ、`sandbox` は明確なエラーで失敗します。何の上で動かすのかは、バンドル自身が宣言しなければならないからです。素のホストでの実行に切り替わることは決してありません。`docker build` 自体も、素のホスト上のシェルコマンドとしてではなく、Docker デーモンのビルダーを通じて走ります。ビルド時の隔離（ビルド*中*の egress やリソース上限）は、実行時のコンテナと同じく、後述の BE-0015 / BE-0016 の堅牢化に委ねます。
  - **堅牢で使い捨てのコンテナ。** `--rm` で使い捨て、ホストのバインドマウントは読み取り専用のバンドルだけ、capability は全て落とし（`--cap-drop=ALL`）、`--security-opt=no-new-privileges`、ルートファイルシステムは読み取り専用にして作業領域だけ tmpfs、非 root ユーザ、CPU / メモリ / pids に上限、そして公開するのは前述のポート 1 つだけ。コンテナは run ごとに名前を付け、teardown で破棄します。従来 `start_launch_server` がホストのプロセスグループに対して行っていた kill を、これが置き換えます。
  - **起動確認の流れは変わりません。** run は引き続き `readyUrl`（いまは公開したループバックのポート）を**条件待ち**で確認し、固定の sleep は使いません（[DESIGN §2](../../DESIGN.md)）。`reuse` の動きもそのまま効きます。外部で `readyUrl` が既に応答していれば、コマンドの起動もコンテナの生成も行いません。
  - **VM ではなく封じ込め。** Docker は被害の範囲を使い捨てコンテナに封じ込めますが、VM 級の境界ではありません。egress の制限、テナント単位のネットワーク名前空間、VM レベルの隔離は、BE-0015 / BE-0016 の領分に残します。`sandbox` は、それらが土台とする、現時点で実現できる決定的な堅牢化です。公開するポートはループバックに限ったままなので、サンドボックスがホストの露出を広げることはありません。
- **黙って飛ばさず、失敗ははっきり示す。** どのモードでも、run が、そのモードでは禁じられている、あるいは満たせないコマンドを必要とするとき（`deny` / `reuse` で `readyUrl` に応答するサーバがない、`sandbox` で `dockerImage` も `dockerfile` もない、`docker build` が失敗する、など）、run は、問題のフィールドと理由を名指しした明確なエラーで失敗します（[DESIGN §2](../../DESIGN.md)：失敗は声高に、暗黙のフォールバックはなし）。ブロックされた `launchServer` や、サンドボックスの設定を欠いた `launchServer` が、不安定な run のように見えてはなりません。`sandbox` が、素のホストでの実行へ黙ってフォールバックすることもあってはなりません。`build` は、アップロードに対する既存の「常に拒否」の扱いをそのまま保ち、個別の特例としてではなく、このポリシーの先例として組み込みます。
- **来歴。** ポリシーの判断（拒否 / 再利用 / サンドボックスのいずれか、どのフィールドか、サンドボックスの場合は、コンテナが指名した `dockerImage` から来たのか、ビルドした `dockerfile` から来たのか、そしてそのイメージ参照またはビルドしたイメージのダイジェスト）を、BE-0073 のアップロード来歴と並べて run の `manifest.json` に記録し、「この run は何を、どこで実行し、何を抑止したのか」を後から辿れるようにします。

本項目が扱うのは、アップロード経路に対する**ポリシーの線引きと、コンテナ化した実行モード**です。すなわち、アップロードされた config のコマンドを実行するかどうかの判断と、実行する場合に使い捨てコンテナの中で実行することです。これより深いジョブ単位の隔離（egress 制御、ネットワーク名前空間、VM 級のサンドボックス化）は、BE-0015 / BE-0016 の領分に残します。本項目は、それらに対して、決定的なゲートと、拡張の土台となる、実際に動くコンテナの実装を用意します。

## 検討した代替案

- **ポリシーゲートのみ（`deny|reuse|allow`）にとどめ、サンドボックスを設けない。** 本項目の当初の形です。ホスティングには不十分なので却下します。コマンドを実行する*かどうか*は決められても、その「許可」を安全にはできないため、マルチテナントのホストは、壊れた `deny` か、素のホストでの RCE かの二択に置かれます。欠けていたのが `sandbox` モード、すなわち安全に許可する道であり、それこそマルチテナント対応の要点です。
- **素のホストの `allow` をオプトアウトとして残す（たとえば Tier-A 向けに）。** 却下します。アップロードされたコマンドを素のホストで実行することは、本項目が塞ごうとしている RCE の入口そのものです。これをオプトインとして残せば、危険な経路はフラグ一つ先にあり続け、設定を一つ誤れば既定として動き出します。Docker が常にあるなら、`sandbox` は、素のホストへの抜け道に見合うほどのコストもなく Tier-A をまかなえるので、`allow` はモードとして残さず削除します。
- **`sandbox` を唯一のモードにする（`reuse` と `deny` も捨てる）。** 却下します。運用者が用意したサーバに対する `reuse` は正当なホスト型のパターンであり（run はコンテナを一切必要としません）、`deny` はアップロードされたコマンドを丸ごと拒みたいときの正しい答えです。`sandbox` は安全な*既定*であり、コマンドを*実行する*唯一の経路ですが、`reuse` と `deny` を保つコストは小さく、それぞれに居場所があります。どちらも素のホストには届きません。
- **`build` のように `launchServer` / `mockServer` もアップロードに対して一律ブロックする。** 却下します。`launchServer` は run 時にテスト対象アプリを起動するために必要で、Web バンドルにはほかに自己完結する手段がないため、一律ブロックは、アップロード経路が存在する理由そのものである Tier-A の正当な用途を壊します。
- **何もせず、BE-0073 の Tier-A 限定に頼る。** 恒久的な答えとしては却下します。規則は Tier-A には正しいものの、暗黙的で一貫しておらず（`build` は拒否、残りは許可）、潜在的な落とし穴であり、BE-0015 / BE-0016 にとって明確な障害になります。アップロード経路が新しいうちに境界を定義するほうが、ホスティングの納期に追われて後付けするより安く済みます。
- **ビルド済みの `dockerImage` だけを受け付け、`Dockerfile` は認めない。** 却下します。既存または事前公開のベースイメージを必須にすると、Web バンドルは、自分では管理できない、あるいはバージョンを固定できないかもしれないレジストリ上の成果物に縛られ、ランタイムの記述もバンドルの外に追い出されます。バンドルが `Dockerfile` を同梱できれば、ビルドの一手間と引き換えに自己完結を保てます。しかもそのビルドは、素のホストではなく Docker デーモンのビルダーを通じて走るので、素のホストには触れないという同じ性質を引き継ぎます。両方の形を認め、どちらを使うかはバンドルが選びます。
- **特定のコマンド文字列をホワイトリスト化する。** 脆いので却下します。コマンド文字列のホワイトリストは回避が容易で、正しく作るのも難しく、少数の固定したモードをコンテナの境界の後ろに置くほうが、単純でセキュリティの筋も明快です。
- **Docker でないサンドボックス（macOS の `sandbox-exec` / seatbelt、bubblewrap、VM）。** 反対ではなく先送りします。seatbelt は macOS 専用で（Linux の control plane がありません）、bubblewrap は Linux 専用、VM ははるかに重いものです。Docker は、プロジェクトのホスティングスタックにすでに同梱され、Tier-A の Mac（Docker Desktop）でも Tier-B の Linux ノードでも使える唯一の仕組みなので、最初のサンドボックスとして現実的です。より強い VM 級の隔離は、同じポリシーゲートの上に重ねる BE-0015 / BE-0016 の選択肢として残ります。
