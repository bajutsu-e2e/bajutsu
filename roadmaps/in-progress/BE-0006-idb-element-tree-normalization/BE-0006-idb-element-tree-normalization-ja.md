## はじめに

`.searchable` 等 SwiftUI 標準要素のツリー表現が壊れていないかを実機で継続的に確認します。

## 動機

セレクタ解決は決定性の核です。セレクタはちょうど 1 要素にマッチし、0 件は「不在」、2 件以上は ambiguous エラーです。これらの判断はすべて、`IdbDriver.query()` が idb の `ui describe-all` から正規化した要素ツリーに対して下されます。したがって run ループ*全体*の正しさは、その正規化の精度以上にはなりません。標準的な SwiftUI コントロールが、私たちの想定どおりにツリーに現れない場合（識別子が違う、trait が欠ける、コンテナが子要素を呑み込んでしまう、など）、それを対象に書いたシナリオは、要素を黙って見つけられないか、さらに悪いことに別の要素にマッチします。

DESIGN §11 はすでにここで 2 つの具体的な症状を記録しています。`.searchable` 等の SwiftUI 標準要素が精度リスクとして名指しで挙げられており、また画面遷移中に idb が（画面は描画済みでも）識別子なしの 1 要素というほぼ空のツリーを返す観測例があります。後者の症状は `IdbDriver.query()` の、上限付きでゲート付きのリトライ（`_max_seen` / `_READY_MIN`）で緩和されていますが、それは*一過性*への防御であり、ある SwiftUI コントロールの*落ち着いた*ツリーがシナリオの想定どおりの形かどうかは教えてくれません。リトライの注記自体、本層では意図的にマスクしない「8 秒級の病的ケース」を指して終わっています。欠けているのは、実アプリが使う SwiftUI コントロールの定常状態の表現がずれていないことを実機で継続的に確認する仕組みです。これがあれば、idb や iOS、あるいはコントロールの作り方の回帰が、現場でのフレーキーまたは誤ったシナリオではなく、失敗するチェックとして現れます。

## 詳細設計

本提案は、シナリオが実際に依存する SwiftUI コントロールについて、期待される正規化ツリーを固定する、維持されたオンデバイス fixture です。実行は、idb をすでに検証している重めの e2e 経路（高速な PR 単位ゲートではない）で行います。

- **サンプルアプリ内のコントロールカタログ画面。** デモアプリ（`demos/`）に、対象とする標準 SwiftUI コントロール（`.searchable`、`List`、`NavigationStack`、`TextField`、トグル、セグメントコントロール等）を並べた画面を持たせ、それぞれにアプリの id 名前空間から安定した `accessibilityIdentifier` を与えます。アプリ単位の識別子はアプリ側の事柄なので、この fixture は特定の本番アプリではなく*正規化*そのものを動かします。ツールは app-agnostic のままです。
- **正規化ツリーの golden（期待値）。** 各コントロールについて、期待される `Element` の形（`Selector` セマンティクスが依存するフィールド: `identifier`、`label`、`traits`、`value`、妥当な `frame`）を、チェックインされた期待値として記録します。チェックはアプリをその画面へ駆動し、`query()` を呼び、正規化ツリーを golden と照合します。これは機械チェック可能なアサーションなので、LLM を介さず決定性ゲートに収まります。
- **落ち着いた状態と一過性を明示的に区別する。** チェックは既存の条件待機経路で画面を待ち（固定 sleep なし）、その後*落ち着いた*ツリーに対して照合します。`_max_seen` ゲート付きリトライは引き続き遷移中の空を吸収します。本提案の golden はそのリトライが役目を終えた後の定常状態を対象とするので、2 つの関心事は分離されたままで、どちらも他方をマスクしません。
- **ずれは失敗するアサーションとして表面化する。** idb や iOS リリース、あるいはコントロールの構成が正規化された形を変えたとき、その変更は、要素を見つけられなくなった現場のシナリオとして漏れ出すのではなく、具体的な差分（どのコントロールのどのフィールドが動いたか）とともに golden のアサーションが e2e 経路で失敗する形で現れます。これは BE-0005 のバージョン監視と自然に対になります。あちらは*どの* idb が入っているかを見張り、こちらは idb が*何を生み出すか*を見張ります。

ここにセレクタの厳格さを緩めるものはありません。ambiguous は依然失敗、不在は依然失敗で、golden アサーションは厳密な比較です。

### コントロールカタログ画面

このカタログは新しく作るアプリではありません。サンプルアプリ（`demos/features/app/BajutsuSample/`）がすでに備えている標準コントロールの一群を、golden の対象として明示的に据えるものです。`RootView.swift` の `MainTabView` がカタログに必要な画面をすでに配線していて、それぞれ `launchEnv`（`SAMPLE_TAB`）で単独に起動できます。そのためチェックは UI をたどらずに目的の画面へ直接駆動できます。

- **`ControlsView`（`SAMPLE_TAB=controls`）** は値コントロールの一群です。`Toggle`（`ctrl.toggle`、要素を素のスイッチにするための `.labelsHidden()`）、`Stepper`（`ctrl.stepper`）、刻みを付けた `Slider`（`ctrl.slider`）、セグメントごとの `Button` で組んだ単一選択セグメント（`ctrl.segment.one` から `ctrl.segment.three`）、`Menu`（`ctrl.menu`、その項目はシステムのポップオーバーからラベルで指定します）、そして有効と恒久的に無効の `Button` の対（`ctrl.button` と `ctrl.buttonDisabled`）が並びます。各コントロールは自身の状態を兄弟の `ctrl.<name>.value` 結果ラベルへ写します。
- **`ListsNavView`（`SAMPLE_TAB=lists`）** は動機で名指しされたコンテナ系コントロールです。`NavigationStack`、`.searchable` 相当の `TextField`（`lists.search`）、データ由来の id を持つ `NavigationLink` 行（`lists.row.<id>`）からなる `List`（`lists.list`）、そして `EditButton`（`lists.edit`）です。「コンテナが子要素を呑み込む」や「`.searchable` が誤った識別子を報告する」という症状が実際に現れるのはここです。

これらの id はアプリ自身の名前空間、すなわち既存の `ControlsView` ／ `ListsNavView` のシナリオがすでに使っている `ctrl` ／ `lists` の接頭辞に由来します。現在 [`demos/features/demo.config.yaml`](../../../demos/features/demo.config.yaml) の `idNamespaces` は `home` ／ `list` ／ `comp` などを並べていますが、`ctrl` ／ `lists` は**含みません**。したがってカタログを配線するということは、その 2 つの接頭辞をそこへ足すこと（一行の設定変更）でもあります。提案がリポジトリの現状に即すよう、ここに明記しておきます。カタログが動かすのは、idb が `Toggle` や `List` をどう `Element` へ正規化するかであって、特定のアプリが選んだ名前ではありません。将来、固定する価値のあるコントロール型が現れ、それを載せる既存画面がない場合は、安定した `accessibilityIdentifier` を与えてカタログに追加します。対象面は広がりますが、仕組みは変わりません。カタログを別の fixture アプリではなく既存のサンプルアプリ内に置くことで、e2e ジョブがすでに行うビルドと起動とインストールがそのまま使えます。

### golden の形式とアサーション

各コントロールの期待値は 1 つの `base.Element`（`bajutsu/drivers/base.py`）です。これは `query()` が生み出す形そのものなので、照合は間にアダプタを挟まない値どうしの比較になります。golden は、選択と操作を左右する per-element のフィールドを記録します。`base.matches` が参照する 4 つ（`identifier` ／ `label` ／ `traits` ／ `value`）に加え、`matches` は読まないものの `find_all` の `within`（幾何的な包含）と idb の座標タップが依存する `frame` も、その正規化形を固定する価値があるので含めます。

- `identifier` は `id` と `idMatches` のセレクタキーです。最も比重の大きいフィールドで、厳密に照合します。ここのずれが、まさに「後で要素が見つからなくなる」障害そのものです。
- `label` は `label` と `labelMatches` のキーです。厳密に照合します（カタログでは固定の英語文字列で、ローカライズされた値や実行時の値ではありません）。
- `traits` は `idb._traits` が正規化したトークン（AX 型を `_norm_type` したものに、`notEnabled` や `selected` を加えたもの）です。trait の順序で golden がフレーキーにならないよう、集合として照合します。`ctrl.buttonDisabled` が `base.Trait.NOT_ENABLED` を、選択中のセグメントが `base.Trait.SELECTED` を必ず帯びるべき、というのがここで効きます。
- `value` は `value` セレクタキーです。`ctrl.<name>.value` の写しラベルにより、画面の初期状態では固定で決定的な文字列になります。
- `frame` は `base.matches` は参照しませんが、`find_all` の `within` スコープと座標タップが依存します。寛容に照合します（後述の「実行場所と検証計画」を参照）。

golden ファイルはカタログのシナリオの隣、`demos/features/app/scenarios/` の下（コントロール別または画面別に分けた `goldens/` サブフォルダ、たとえば `goldens/controls.json`）に、チェックインした JSON として置きます。これは `parse_describe_all` が出力する `base.Element` の辞書そのものなので、golden の更新はフレーム許容を適用して記録した `query()` のスナップショットにほかなりません。アサーションは 1 本の決定的な流れです。`launchEnv` で画面へ駆動し、画面が落ち着くのを待ち（後述）、`IdbDriver.query()` を呼び、正規化ツリーをコントロール id ごとにフィールド単位で golden と照合します。比較は決定的ですが、**辞書全体の等価判定ではなくフィールドごと**です。`identifier` ／ `label` ／ `value` は厳密、`traits` は集合一致（順序非依存）、`frame` は後述の寛容な幾何チェックです。経路のどこにも LLM はありません。run と CI の判定が求める決定性の規律にそのまま収まります。

### 落ち着いた状態と一過性

チェックは固定時間の sleep を一切しません。既存の条件待機経路、すなわち目的画面の既知のアンカー（たとえば `ctrl.title` や `lists.list`）が現れるまで `query()` をポーリングする `IdbDriver.wait_for(sel, timeout)` で待ち、その後にツリーを撮って照合します。

これにより golden は、既存の 2 つの一過性への防御ときれいに分離され、どちらも他方をマスクしません。

- `query()` の `_max_seen` ゲート付きリトライ（`_is_transient_empty` と `_READY_MIN`）は、アサーションの下で引き続き idb の遷移中のほぼ空のツリーを吸収します。golden が照合するのはそのリトライが返すツリー、つまり落ち着いた定常状態です。一過性を捕まえるのは golden の役目ではありません。
- **BE-0087**（idb のアクション settle、進行中）は補完する軸です。あちらはアクションの効果が落ち着いた後にアサーションすることを保証します。BE-0006 は落ち着いたツリーがどんな形かを固定し、BE-0087 はそれを待つことを固定します。一方の回帰はもう一方の回帰を隠すことなく、それ自身のチェックで失敗します。BE-0006 の golden は静止した画面で読むので、settle はそこで検証中の変数ではありません。

### ずれの差分

idb や iOS リリース、あるいはコントロールの構成が正規化出力の形を変えたとき、golden アサーションは e2e 経路で、フィールド単位の具体的な差分とともに失敗します。たとえば「`ctrl.toggle`: traits expected `{switch}` got `{button}`」や「`lists.search`: identifier expected `lists.search` got `null`」のように、どのコントロールのどのフィールドが動いたかを名指しします。これにより、現場で静かに起きる障害の一群（シナリオが要素を黙って見つけられなくなる、あるいは別の要素にマッチする）を、修正が明白な、声を上げる位置特定済みのビルド失敗へ変えます。修正は明白です。`idb.py` の正規化器を意図的に更新し、同じレビュー済みの変更で golden を撮り直すか、本物の idb や iOS の回帰だと認めるかのいずれかです。

これは **BE-0005**（idb_companion バージョン監視、実装済み）と明快な接合面で対になります。BE-0005 は*どの* idb が入っているかを見張ってバージョン範囲を固定し、BE-0006 は*その idb が何を生み出すか*を見張ります。BE-0005 の毎週の `idb-monitor.yml` はすでに smoke シナリオを最新の companion で `parse_describe_all` に通しています。カタログ golden を同じ間隔で走らせれば、「スキーマが変わった」という曖昧な smoke の失敗を、動いた具体的なフィールドへと変えられます。

### 実行場所と検証計画

golden アサーションは重めのオンデバイス経路で走り、高速な PR 単位ゲートでは決して走りません。`make -C demos/features e2e` や [`.github/workflows/e2e.yml`](../../../.github/workflows/e2e.yml) の `smoke` ジョブと同じ macOS と Simulator の面で、これらはすでにサンプルアプリをビルドし、起動し、インストールし、実際の idb を通してシナリオを走らせています。カタログ golden はそこにシナリオとして追加します（BE-0005 の毎週の監視へ加えるのも自然です）。これにより、metered な macOS 分を消費するのはデバイスに関係する PR と所定の間隔のときだけになり、`make check` は手つかずのままです。

唯一の本当の緊張は `frame` です。正確なピクセル座標はデバイスやランタイム、スケール、セーフエリアのインセットによって変わるので、厳密に照合すると構造的にフレーキーになります。これは決定性の指針の趣旨（アサーションは本物の回帰のときだけ失敗すべき）に正面から反します。解決は**同一性は厳密に、形状は寛容に**です。`identifier`、`label`、`traits`、`value` は厳密に比較し、`frame` は妥当性だけを確認します。幅と高さが非ゼロであること、そして要素が画面の境界内に収まっていること（`within` セレクタがすでに使う `base._contains` と同じ包含判定を再利用します）です。これで、重要な障害（要素がゼロフレームに潰れる、コンテナが子要素をもう囲まない）を、デバイスの正確なピクセルに固定せずに捕まえられます。したがって検証計画はこうです。各カタログ画面へ駆動し、`query()` を撮り、同一性と状態の 4 フィールドを golden と厳密に、`frame` を境界内で照合し、不一致があれば前述のフィールド単位の差分を出します。

## 検討した代替案

- **既存のシナリオ別 e2e デモに頼る。** デモシナリオはすでに実機で idb に触れますが、アサーションの対象はアプリの挙動であって、各コントロール型の正規化ツリーではありません。たとえば `.searchable` が識別子を報告する仕方に限った回帰は、現行のどのシナリオも壊さないまま、次にそれを対象に書かれるシナリオを壊しうるのです。専用のコントロールカタログは、表現そのものをテスト対象にします。
- **`query()` を強化して奇妙なツリーを修復する。** 想定外の SwiftUI の形を「期待される」形へ正規化経路内で書き換えるヒューリスティックを足すこともできます。それはずれを報告せず隠し、決定性が要の経路にアプリ固有の推測を載せます。golden に照合し、正規化器を意図的に更新する方が、`query()` を単純に、変更をレビュー済みに保てます。
- **実アプリの全画面をスナップショットする。** 網羅性は最大ですが、脆くアプリ固有です。チェックを 1 つのアプリの進化する UI に結び付け、ツールレベルの保証にアプリ単位の事柄を再導入します。標準コントロールに絞ったカタログの方が安定し、app-agnostic を保てます。
