## はじめに

AI が自律的に画面遷移をクロールして画面マップを生成し、クラッシュと到達不能な状態を報告します。Tier 1 の `record` 機能を拡張します。

`record` が *目的指向*（1 つの自然言語ゴールに向かって AI が探索し、1 本の決定的シナリオを書き出す。[recording.md](../../../docs/ja/recording.md)）であるのに対し、`crawl` は *幅優先* です。到達できる画面をできる限り体系的に巡り、画面と画面間の遷移のグラフを構築して、見つけたものを報告します。これは探索と発見のためのツールであって、合否ゲートには決してなりません。

## 動機

現在のオーサリング経路は「テストしたいフローを既に知っている」ことを前提にしています。`record` にゴール（「ログインして設定を開く」）を渡せば、その経路のシナリオが得られます。書くべき内容が分かっている段階では正しい道具ですが、クロールが埋める 3 つの空白が残ります。

**1. オーサリング前の発見。** 新しいアプリをオンボーディングするとき（per-app の [`apps.<name>`](../../../DESIGN.md) config と [BE-0024 doctor / onboarding](../../proposals/BE-0024-doctor-onboarding/BE-0024-doctor-onboarding-ja.md)）、最も難しい問いは「そもそもどんなフローが存在し、どの画面をテストする価値があるのか」です。今はこれに答えるには手作業でアプリを読むしかありません。クロールは画面マップ（到達できる各画面の代表スクリーンショットと要素ダンプを添えた、ラベル付きの一覧）を生成し、「アプリを知らない」を「40 画面ある、どれにシナリオを書くか選べ」に変えます。これは `record` の自然な入口です。発見された各画面とそこへの最短経路が、そのままシード（種）となるゴールになります。

**2. アプリ全体のカバレッジ計測。** [DESIGN §7.2](../../../DESIGN.md) は、`doctor --app` がアクセシビリティ識別子のカバレッジを **入口画面 + 宣言済み deeplink で開ける画面** に限って計測すると明言しており、さらに `doctor --app <name> --from <runId>` モード（過去の run が残した画面ごとの `elements` ダンプを再利用してアプリ全体のカバレッジを正直に算出する）を既に想定しています。クロールは、そのダンプを生み出す run そのものです。§7.2 の「測定範囲の正直さ」という約束を実体化するものであり、推測ではなく、クロールが実際に到達した全画面に対してカバレッジを算出できるようになります。

**3. 堅牢性のスモークテスト。** 体系的なクロールは、手書きした数本のハッピーパスよりはるかに広くアプリを操作し、シナリオテストが決して訪れないクラッシュ、ハング、行き止まりを表面化させます。これが「App Explorer / Robo test」的な価値（本項目の由来である Autify VAX）です。「*この*順序で*あれ*を押すとアプリが落ちる」を、ユーザーより先に捕まえる、安価で広範でオーサリング不要のカバレッジです。

**この機能を Bajutsu で行う意義は、決定性の境界そのものにあります。** クロールは本質的に非決定的で（探索順序も AI の選択もアプリの状態も run ごとに変わる）、クロール自体が CI ゲートになることは決してありません（[prime directive #1](../../../CLAUDE.md): *AI は著者であり失敗時の調査役であって、判定者ではない*）。汎用のモンキー/エクスプローラに対して Bajutsu が加える価値は、クロールの *副産物* が一級の決定的シナリオである点です。クラッシュの再現経路と発見した各フローを、ユーザーがレビューでき、コミットすれば `run` が AI なしで Tier 2 の回帰として再実行できるプレーンな YAML として書き出します。したがってクラッシュは CI に *届きます*。フレーキーなクロールとしてではなく、コミット済みの再現シナリオとして届くのです。これはツールの他部分と同じハブモデルです（シナリオが共有の成果物であり、AI が一度だけ書き、以後は決定的ランナーが所有します）。

## 詳細設計

### コマンド面

`record` のフラグに倣った新しい Tier 1 コマンド:

```
bajutsu crawl --app <name>
    [--max-screens N] [--max-steps N] [--budget DURATION] [--max-depth N]
    [--seed <deeplink> ...]          # 起動画面以外の追加の入口
    [--guide ai|off]                 # AI で優先度付け/入力補完するか
    [--backend idb] [--udid booted] [--dismiss-alerts]
    [--out runs/<runId>]
```

`record` と同様、これは *AI ライブ* の経路であり、決定的ゲートの一部ではありません（[DESIGN §3.1](../../../DESIGN.md)）。`--guide off` ならクロールは後述の識別子駆動ヒューリスティックのみで動き、`ANTHROPIC_API_KEY` を必要としません。`--guide ai` はその上に任意のエージェントを重ねます。いずれにせよ AI が合否を決めることはありません。

### 状態モデル（グラフのノード）

クロールが訪れる各画面は **状態フィンガープリント** へ縮約され、同じ画面の再訪を再探索ではなく「認識」できるようにします。フィンガープリントは `driver.query()`（[`drivers/base.py`](../../../bajutsu/drivers/base.py)）から導きます。

- **主:** 画面上に存在する `accessibilityIdentifier` の集合をソートしてハッシュ化したもの。識別子は非ローカライズかつデータ由来（[DESIGN §7.3](../../../DESIGN.md)）なので、このフィンガープリントは locale や軽微な内容差に対して安定します。id ベースのセレクタを決定的にするのと同じ性質が、id ベースの状態同定を決定的にするのです。
- **フォールバック（id が乏しい画面）:** id を持つ要素が少なすぎる場合は、操作可能要素の `(traits, frame バケット)` に対する構造ハッシュへフォールバックします。安定性は下がるため、stability ladder の正直さの規約（[DESIGN §5](../../../DESIGN.md)）に倣って report に明示します。

フィンガープリントがグラフのノード識別子です。各ノードは代表スクリーンショット、`elements` ダンプ、id 集合を保持します。これは §7.2 が必要とする画面ごとの成果物そのものです。

### フロンティアとアクション選択（グラフのエッジ）

ある状態からの **候補エッジ** は、その操作可能要素（`button` / `link` / `textField` / `searchField` の traits を持つもの）で、`id` を持つ要素を優先します（stability ladder, [DESIGN §5](../../../DESIGN.md)）。クロールは未探索の `(状態, アクション)` 対のフロンティアを保持し、幅優先で巡回します。1 状態内の候補順序は決定的（id でソート）なので、変更のないアプリを 2 回クロールすれば、アプリ自身の非決定性が許す限り同じ順序で探索します。これは得られるマップの再現性のために重要です。

テキストフィールドには入力値が要ります。`--guide off` では trait ごとに固定の安全なプレースホルダを使い、`--guide ai` では任意のエージェントが文脈に合った入力を供給します（後述の通り、これも Tier 1 のままです）。

### その場での後戻りではなく、決定的リプレイによる巡回

アプリの遷移は通常は不可逆です（タップを確実に「元に戻す」ことはできません）。逆方向にナビゲートしようとすると（フレーキーになるため）、クロールはフロンティアのノードを、`run` が任意の状態へ到達するのと同じ方法で再訪します。すなわち **クリーン環境へ erase し、そのノードへの既知の最短経路をリプレイします**（[DESIGN §2](../../../DESIGN.md) 環境のクリーン化、[`env.py`](../../../bajutsu/env.py) の erase/boot/launch）。各エッジは記録済みの `Step` なので、入口画面から発見済みノードへの経路は *それ自体が決定的シナリオ* です。それをリプレイすれば（オーケストレータの `_do_action` / `_wait` を再利用、[`orchestrator.py`](../../../bajutsu/orchestrator.py)）ノードに着地し、クロールは次の未試行アクションを取ります。これにより巡回全体を、ツールが既に持つ決定性フレンドリなプリミティブの上に保てます。そして全ノードには、コミット可能でリプレイ可能な到達経路が最初から付随します。

アクション間は条件待機（`screenChanged` / 対象要素）で待ち、固定 sleep は使いません（[DESIGN §6.3](../../../DESIGN.md)）。

### クラッシュと行き詰まり状態の検出（決定的シグナル）

各アクションの後、クロールはアプリが健全であることを決定的に確認します。

- **クラッシュ** = アプリプロセスがもはやフォアグラウンド/生存していない（`simctl`/env の pid チェック。a11y ツリーがアプリ内容のない素のウィンドウに collapse した状態は、[`record.py`](../../../bajutsu/record.py) の `_shows_app_ui` を再利用してシステムアラートと区別します。後者は既存のアラートガードが代わりにクリアします）。クラッシュ時は既存の `result:error` 安全網（[DESIGN §9](../../../DESIGN.md)）で全証跡を取得し、リプレイした経路を **最小の再現シナリオ** として書き出します。
- **行き詰まり / 行き止まり** = 操作可能要素が無い状態、または条件待機の予算内でどの候補アクションも画面変化を起こさない状態。
- **到達不能** = *期待された* のに到達されなかった状態。期待の出どころは 2 つ: 宣言済み deeplink（開くのに失敗した `--seed` / config の deeplink）と、アプリが宣言した id 名前空間（[DESIGN §7.3](../../../DESIGN.md)）のうち、その画面がマップに一度も現れなかったもの。これらは失敗ではなく空白として報告します。

これらの分類はいずれも LLM の判断ではありません。クラッシュはプロセスの事実、「行き詰まり」は「どのエッジも状態フィンガープリントを変えなかった」こと、「到達不能」は宣言済みと観測済みの集合差です。AI は判定に関与しません。

### 任意の AI レイヤ（`--guide ai`、厳密に Tier 1）

クロールは AI が一切無くても有用です。有効にした場合、エージェント（既存の [`Agent` プロトコル](../../../bajutsu/agent.py) の拡張）は *探索を導く* だけで、結果を評価することは決してありません。

- **フロンティアの優先度付け:** 候補アクションを関心度（例:「概要」より「購入手続き」）で並べ替え、単純な幅優先の代わりに限られた予算を有効に使います。
- **現実的な入力の供給:** テキストフィールドに妥当そうなメールアドレスや検索語を入れます。
- **近重複状態の併合:** 構造フィンガープリントなら分けてしまう状態（例: 行がデータ違いだけのリスト）をまとめます。

これは `record` の基盤（`_screenshot_bytes`、アラートガード周り、`_execute_with_recovery`）を直接再利用します。境界は堅固です。エージェントは *何を探索するか* に影響しますが、*何が起きたか* は上記の決定的チェックが決めます。

### 出力

すべては既存の `manifest.json`（[DESIGN §9](../../../DESIGN.md)）と並んで `runs/<runId>/` 配下に出ます。

- **画面マップ**：`screenmap.json`（ノード + エッジ）に加え、`report.html` にレンダリングしたグラフ（Mermaid/Graphviz）。各ノードはスクリーンショットと `elements` ダンプにリンクします。
- **クラッシュレポート**：各クラッシュとその再現 `Scenario`（`dump_scenarios` で YAML として書き出し、`run` で直接実行可能）。
- **候補シナリオ**：発見したフローごとに、ユーザーが本物のテストへ昇格できる下書きシナリオ。すべての AI オーサリングと同様、これらは人間レビュー用の *提案* として出し、コミット済み YAML に黙って書き込むことはしません（[DESIGN §6.5](../../../DESIGN.md)）。
- **カバレッジ供給**：`doctor --app <name> --from <runId>` がアプリ全体のカバレッジに使う、画面ごとの `elements` ダンプ（[DESIGN §7.2](../../../DESIGN.md)）。

### 他項目との関係

- **`record`**（[recording.md](../../../docs/ja/recording.md)）：クロールは幅優先の補完であり、発見した画面が `record` のゴールのシードになります。ループ、アラート処理、証跡周りを共有します。
- **[BE-0012 アクションキャプチャ record](../../proposals/BE-0012-action-capture-record/BE-0012-action-capture-record-ja.md) / [BE-0014 record の切り分け](../../proposals/BE-0014-record-demarcation/BE-0014-record-demarcation-ja.md)**：これらは *人間駆動* のキャプチャと、AI `record` からの切り分けを扱います。クロールは第 3 の、完全に自律的なオーサリング入口であり、それらと並べて切り分けるべきです。
- **[BE-0024 doctor / onboarding](../../proposals/BE-0024-doctor-onboarding/BE-0024-doctor-onboarding-ja.md)**：クロールは doctor の §7.2 計測が必要とするアプリ全体カバレッジの入力を供給します。

### 実装状況

クロールの**エンジン**（`bajutsu/crawl.py`）と CLI（`bajutsu crawl`）は出荷済みです。決定的リプレイによる幅優先巡回、状態グラフ（`ScreenMap`：ノード／エッジ／クラッシュ／アラート／フロンティアの plan／停止理由）、クラッシュと行き詰まり状態の検出、任意の `--guide ai` レイヤ、並列プール（[BE-0064](../../implemented/BE-0064-parallel-crawl/BE-0064-parallel-crawl-ja.md) / [BE-0077](../../implemented/BE-0077-parallel-web-crawl/BE-0077-parallel-web-crawl-ja.md)）を備えます。クロールは `screenmap.json` と画面ごとの `screens/<fingerprint>.png` を書き出し、`serve` の **Crawl** タブがマップをリアルタイムに描きます。

最新のスライスは、*出力*の節が掲げる **描画済みの画面マップグラフ**を、オフライン／CLI 用に提供します。`bajutsu crawl` は JSON と並べて自己完結した `screenmap.html`（`bajutsu/crawl_report.py`、`crawl.html.j2`）も書き出すようになりました。画面を幅優先の深さ列に並べ（web UI の実証済みレイアウトを移植）、遷移を**静的なインライン SVG** で描き（OS アラートをタップして通過した遷移はアンバー色＋🛡️）、各画面をスクリーンショットへリンクし、クラッシュと閉じたアラートの経路を一覧にします。CSS は埋め込み、JavaScript も外部アセットも無しなので、run ディレクトリから直接開けます。この描画は `ScreenMap` の純粋で決定的、かつモデル非依存な関数で、決定的な探索にも判定にも一切触れません。

さらに次のスライスは、*出力*の節が掲げる**クラッシュ再現シナリオの自動生成**を提供します。`bajutsu crawl` は、忠実に再現できるクラッシュごとに `crashes/crash-NNN.yaml` を 1 件書き出すようになりました（`bajutsu/crawl_repro.py`）。各クラッシュは到達までの正確なアクション経路をすでに記録しており、`crash_scenario` がその経路を実行可能な `Scenario` に戻します（tap／type／fill はそれぞれ対応するステップに写像します）。生成物は `dump_scenario_file` で YAML として書き出され、`run` がそのまま再生できます。つまり、発見したクラッシュは人間のレビューを経て、コミットされた Tier 2 のリグレッションになります。この変換は `ScreenMap` の純粋で決定的、かつモデル非依存な関数です。正規化座標をタップする経路（`tap_point`）は指定できるセレクタを持たないため、欠損のあるシナリオを出すのではなく、何も出力しません（忠実に再現できないなら出さない）。構造化された経路を運ぶため、`Crash` は人間可読の `path` に加えて再生可能な `actions` を保持するようになりました（`screenmap.json` に直列化し、それを欠く古いマップも読み込めます）。

この先に残るもの：フォーム中心のフローに対する AI ガイドのテキスト入力供給と、発見したフローごとの候補シナリオ提案です。

## 検討した代替案

**純粋なランダム「モンキー」テスト**（Android Monkey / UIAutomation のファジングのようなランダムタップ）。作るのは容易でクラッシュのあぶり出しは得意ですが、きれいな画面マップを生まず、id の安定性を尊重せず、経路が再現不能です。そのため `record` や `doctor` に供給できず、決定的な再現も書き出せません。体系的な id 駆動の幅優先巡回を優先します。ランダムなファズモードは、識別子がほぼ無い座標専用アプリ向けのフォールバックとしてのみ残す価値があります。

**純粋な AI 自由探索**（「すべて探索して」のような曖昧なゴールで `record` をそのまま再利用）。LLM は明示的な状態グラフを持たずにターンごとにさまようため、ループし、再訪し、重複排除もカバレッジ計測もできません。遅く高価でもあります。クロールには明示的な状態モデルとフロンティアが必要で、AI はその上の *任意のガイド* であってエンジンではありません。いずれにせよ結果からは排除されます。

**クロール自体を CI / Tier 2 ゲートにする**（「クロールがクラッシュを見つけたらビルドを落とす」）。クラッシュ検出は決定的なので魅力的ですが、クロールの *経路* は非決定的です。フレーキーで変化するクロールを必須ゲートにするのは determinism-first に反します（[DESIGN §2](../../../DESIGN.md)）。解決策は次のとおりです。クロールは Tier 1 の発見にとどめ、書き出した再現シナリオが人間レビューを経て Tier 2 ゲートになる、通常の `record → run` 昇格です。クラッシュはクロールとしてではなくコミット済みシナリオとして CI に届きます。

**アプリのバイナリ/ソースの静的解析** で、実行せずに画面マップを作ります。スコープ外です。Bajutsu はアプリ非依存でビルド成果物のみを受け取りますし（[DESIGN §1](../../../DESIGN.md)）、価値の半分であるランタイムクラッシュを見つけられません。
