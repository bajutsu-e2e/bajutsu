## はじめに

idb バックエンドの flaky を、画面が安定してから操作することで減らします。現在は tap・swipe・type といった操作ステップが対象を解決した直後に発火するため、画面がまだアニメーション中に放った tap が、移動し終えた後の要素を外してしまうことがあります。本提案では、各操作の前に上限付きの条件待ち（settle）を挟み、過渡的な空ツリーへのリトライのバックオフを改善し、`wait_for` に残る潜在的な timeout のバグを直します。いずれも idb ドライバ内で完結し、決定的なコアには手を入れません。

## 動機

idb バックエンドは座標ベースです。セマンティックな tap を持たないため、`IdbDriver.tap()` は `query()` から対象の frame 中心を解決し、その座標に対して `idb ui tap x y` を発行します。したがって tap の精度は、解決した瞬間の frame の精度がそのまま効きます。SwiftUI の画面遷移中はツリーが動いており、要素は存在していても frame が最終位置へ向けてアニメーションしている途中です。この最中に解決すると移動途中の中心座標が得られ、idb が tap を届けるころには要素が動いてしまっていて、tap が外れたり隣の要素に当たったりします。これは遅い・混んでいる実行環境（とくに CI）で非決定的な失敗を生む典型的な原因です。

ドライバは遷移に伴う症状のうち一つ、すなわち idb が遷移中にほぼ空のツリーを返すことには、`query()` 内の上限付き・`_max_seen` で門番をしたリトライで既に対処しています。しかしこの防御が救えるのは「空」のときだけです。要素は揃っているが動いている画面は空ではないため、リトライはそのまま通過させてしまい、操作は動いている対象に対して発火します。さらに、遷移を引き起こす操作と、その次に新しい画面を操作するステップとの間に settle のステップはありません。オーケストレータには `wait: settled` がありますが、これはシナリオごとに明示的に書く任意指定なので、既定の経路は安定を待たずに操作してしまいます。

これに、より小さな問題が二つ重なります。

- **`wait_for(sel, timeout)` が timeout を無視している。** idb の実装は `query()` を一度だけ呼び、その時点で selector が存在するかを返すだけで、期限まで待つことをしません。現在は呼び出し元がないため実際の実行には影響しませんが、実際の待機を期待して使い始める人にとっては潜在的な落とし穴です。
- **過渡的な空へのリトライが固定バックオフを使っている。** `query()` は退化したツリーを固定で 5 回、一律 0.2 秒間隔でリトライします。短い指数バックオフなら同じ上限に達しつつ、画面が早く復帰したときはより早くポーリングし、そうでないときは間隔を広げられます。

狙いは、よくある操作経路が遷移を決定的に乗り越えられるようにすることです。固定 sleep を使わず、selector の厳密さを緩めず、実行ループに LLM を一切持ち込まずに実現します。

## 詳細設計

変更はすべて `bajutsu/drivers/idb.py`（idb バックエンド）に閉じます。web（Playwright）バックエンドは独自の auto-wait を持つため手を付けません。`Driver` インタフェースと決定的なランナーも変更しないので、ツールはアプリにもバックエンドにも依存しないままです。

- **操作前の settle（保留 — 実機で成立する設計が必要）。** 最初の試みでは `IdbDriver._settle()` を追加し、ツリーが連続 2 回の読み取りで変化しなくなるまでポーリングして各操作メソッドの冒頭で呼びました。しかし「ツリー完全一致」による素朴な settle は実機で成立しないことが分かりました（smoke E2E で検証）。`describe-all` は負荷のかかった CI シミュレータで 1 回 ~2.5 秒かかり、稼働中の画面はステータスバーや揮発的な値のため連続 2 回が完全一致することがほとんどありません。その結果 settle は**毎アクションでポーリング上限を使い切り**、1 アクションあたり ~50 秒を足して smoke run をステップの timeout 超過まで膨張させました。実用的な settle には**安価で頑健な安定シグナル**が要ります。揮発的なフィールドを無視した射影（例: identifier と frame だけ）で比較し、ポーリングを厳しく上限し、後続の解決がどのみち払う describe-all を再利用して新たな低速呼び出しを足さない、という設計です。そして着地前に実機で検証する必要があります。したがってこのスライスはその設計ができるまで保留し、下の衛生対応とバックオフのスライスは独立しているので先に出荷します。

- **固定 sleep は使わない。** settle はツリーの安定に対する条件待ちであり、prime directive（条件待ちのみ、盲目的な `sleep` は使わない）に沿います。唯一の遅延はポーリング間隔で、これも上限付きです。テストでは間隔をゼロに注入します。

- **過渡的な空へのリトライを指数バックオフに。** `query()` の一律 `_EMPTY_BACKOFF_S` のループを、短い指数バックオフ（例: 0.1 → 0.2 → 0.4 秒、上限あり）に置き換えます。全体の上限は現状を超えません。空ツリーが早く解消したときの復帰を速めつつ、「本当に疎な初回画面はそのまま返す」という既存の契約（`_max_seen` の門番）は維持します。

- **`wait_for` に timeout を守らせる。** `IdbDriver.wait_for(sel, timeout)` を、selector が解決するか timeout に達するまで `query()` をポーリングするよう再実装し、オーケストレータの待機の流儀に合わせます。現在は呼び出し元がないため、実行への影響なしに潜在的なバグを取り除けます。

- **解決の期限を引数で渡す。** `_resolve` は 3.0 秒の既定を保ちつつ、呼び出し元から期限を受け取れるようにします。操作経路の許容時間を、ハードコードではなく一箇所で見通せるようにするためです。

- **決定性は維持する。** selector の厳密さには触れません。曖昧なら即座に失敗し、存在しなければ失敗します。settle とバックオフが変えるのは画面を読む「タイミング」だけで、selector がどの要素に一致するかは変えません。run / CI のゲートに LLM を足す箇所もありません。

これは意図的に **ツリーの修復ではありません**。[BE-0006](../../proposals/BE-0006-idb-element-tree-normalization/BE-0006-idb-element-tree-normalization-ja.md) は SwiftUI コントロールの**定常状態**の正規化形をゴールデンと突き合わせて検証するもので、本提案はドライバが安定したツリーを読む「タイミング」だけを定めます。両者は別の関心事として分けます。BE-0006 のゴールデンは settle が役目を終えた後に評価され、互いに覆い隠すことはありません。

### 実装状況

三つのスライスのうち二つが出荷済みです（PR [#295](https://github.com/bajutsu-e2e/bajutsu/pull/295), [#296](https://github.com/bajutsu-e2e/bajutsu/pull/296)）。いずれも、台本どおりのツリー列を返す fake な `run` を注入する高速なユニットゲートで検証しています。

- **`wait_for` / `_resolve` の衛生対応。** `IdbDriver.wait_for(sel, timeout)` は、一度だけ確認して期限を無視するのではなく、selector が解決するか timeout に達するまで `query()` をポーリングするようになりました。`_resolve` は `timeout` を受け取り、操作経路の許容時間を一箇所で見通せるようにしています。`test_wait_for_polls_until_the_element_appears` と `test_wait_for_times_out_when_absent` で検証しています。
- **過渡的な空へのリトライの指数バックオフ。** `query()` の一律リトライは `_empty_backoff`（0.05 秒から始めて倍々で 0.2 秒の上限に達する短い指数バックオフ）になりました。全体の上限は従来の固定値を超えず、`_max_seen` の門番も保ちます。`test_empty_backoff_grows_exponentially_then_caps` で検証しています。

**操作前の settle は保留のままです**（最初の設計項目を参照）。ツリー完全一致による素朴な settle は実機で成立せず、着地の前に安価で実機検証済みの安定シグナルが必要です。これが残る唯一のスライスで、高速ゲートではなくより重い Simulator 経路を要するため、項目は **実装中** のままにします。

### 検証

- **ユニットテスト（ゲート）。** 既存の `IdbDriver` のテスト様式に倣い、台本どおりのツリー列を返す fake な `run` を注入し、ポーリング間隔をゼロに設定します。settle が安定を待ってから進むこと、上限で打ち切ること、操作が安定したツリーに対して解決すること、空リトライのバックオフ系列とその上限が成り立つこと、`wait_for` が timeout までポーリングすることを確認します。
- **実機。** より重い e2e 経路（`e2e.yml` の smoke / xcuitest）が、実際の Simulator に対してこの変更を実行します。
- **前後の flaky。** 代表的なシナリオに対して [BE-0049](../../implemented/BE-0049-determinism-flakiness-audit/BE-0049-determinism-flakiness-audit-ja.md) の repeat-and-diff 監査を回し、非決定性の低減を定量化します。

### 提供

小さく的を絞った PR で進めます。実機検証を受けて順序を見直しました。(1) `wait_for` / `_resolve` の衛生対応と (2) 空リトライの指数バックオフは出荷済みです（PR #295, #296）。操作前の settle は、安価で実機検証済みの安定シグナルができるまで保留します（上記参照）。

## 検討した代替案

- **ドライバではなくランナー層で自動 settle する。** オーケストレータで毎ステップの間に settle してもこの問題は covers できますが、動く対象の問題は idb 固有のタイミングの癖です（web バックエンドは既に auto-wait します）。settle を idb ドライバに置くことで、必要なバックエンドに修正を局所化し、不要なバックエンドを遅くせずに済みます。
- **遷移後に固定 sleep を入れる。** 単純ですが prime directive で禁じられており、そもそも筋が悪いです。速い画面には長すぎ、遅い画面には短すぎます。安定に対する条件待ちこそが、決定性を保った等価物です。
- **既存の `wait: settled` ステップに頼る。** 既に存在しますが、シナリオごとの任意指定なので、既定の操作経路は無防備なままで、著者は毎回追加し忘れないようにしなければなりません。操作を既定で settle させれば「待ち忘れ」の flaky をまとめて取り除けますし、より細かい制御のための明示ステップは残ります。
- **アサーションを含むすべての `query()` で settle する。** 安定は最大になりますが、読み取り・アサーションの経路が遅くなり、あえて速くしている存在チェックが複数回ポーリングの操作に変わってしまう恐れがあります。settle を操作メソッドに限ることで、実際の失敗モードに的を絞ります。
