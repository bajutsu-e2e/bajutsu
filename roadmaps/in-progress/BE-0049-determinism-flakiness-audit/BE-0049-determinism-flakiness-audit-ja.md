## はじめに

フレーキネスを許容するのではなく、シナリオの決定性を*証明する*読み取り専用の診断です。同一条件で繰り返し実行して非決定的な結果を検出し、セレクタと待機の安定性を静的に採点し、さらに run が system of record に貯まれば、同じシナリオの安定性をその run 履歴を通して追跡します。これは助言にとどまり、ゲートの完全に外側にあります。ばらつきは所見として報告するもので、黙って通過させることも、判定に使うこともありません。

## 動機

Bajutsu が競合に対して掲げる中心的な主張は、ゲートが吸収によってではなく**契約として**決定的である、という点です。Maestro は逆の立場をとります。「フレーキネスへの組み込み耐性」を謳い、暗黙の自動待機やタップ再試行で不安定さを呑み込み、さらに `retry` コマンドを備えます。その結果、フレーキーなテストは*修正*されるより*隠される*傾向にあります。Bajutsu はそれを拒みます（固定 `sleep` なし、曖昧なセレクタは即座に失敗）。しかし今日、その主張は**暗黙**です。あるスイートの決定性を定量化し証明する仕組みがツール内に何もなく、導入を検討する側は信用するしかありません。

本項目はこの差別化を実体のあるものにします。「私たちは決定的だ」を、チームが実行できるレポートに変えます。シナリオを K 回実行し、結果やタイミングがばらついたステップやアサーションを表面化させ、各セレクタを安定性のはしご（一意に解決する `id` が `label` ／ `traits` に勝り、それらが `index` ／生座標に勝る）に照らして静的に採点し、タイムアウトのない待機や緩すぎる条件の待機を指摘します。これは `doctor` の規約スコア（[BE-0024](../../proposals/BE-0024-doctor-onboarding/BE-0024-doctor-onboarding-ja.md)）を、「id は適切に命名されているか」から「このスイートは再現可能か」へと自然に広げるものです。

繰り返して差分を取る方法は、*ある一時点の*決定性を証明します。run が system of record（[BE-0015](../../proposals/BE-0015-web-ui-public-hosting/BE-0015-web-ui-public-hosting-ja.md) の DB 記録）に貯まる今、もう 1 つの経時的なシグナルが使えます。同じシナリオの判定の履歴です。ただし「同じシナリオ」が意味を持つのは、各 run に**安定したシナリオ同定子**と**変動要素のバージョン**がスタンプされている場合だけです。さもないと、判定が反転しても、シナリオを編集したのか、アプリ／ツールが変わったのかと区別できません。今の run は結果は記録しますが、**provenance を持ちません**。実行したシナリオの content hash も、git revision も、ツールやアプリのバージョンもありません（保存されるのは実行した `scenario.yaml` の本文そのものと、環境タブ用の `device_runtime` だけ）。そのため履歴をフレーキネス分析に使えません。各 run にその provenance をスタンプするのが、それを可能にする安価な前提です。

**境界に関する注記（重要）。** 本項目はロードマップの「取り込まない → 失敗テストの自動リトライ」の線に触れるため、区別を明示しておきます。あの項目を不採用にしたのは、通過するまでのリトライがフレーキネスを*隠す*からです。この監査はその**逆**です。繰り返し実行は、非決定性を*検出して報告するための診断*です。ばらついた結果は修正すべき所見であって、赤を緑に変えるリトライではありません。監査は判定を決して変えず、run／CI ゲートに決して供給しません。したがって、決定性優先を損なうのではなく強化します。

## 詳細設計

提案粒度です。設計上の制約は、監査が純粋に観測に徹することです。

- **繰り返して差分を取る（動的）。** `--audit` モード（または `bajutsu audit` サブコマンド）が、同一の前提条件下でシナリオを `K` 回実行し、結果を比較します。比較するのはステップごとの pass/fail、解決されたセレクタの対象、アサーション結果です。run をまたいでばらつくものは非決定的として、ばらついた証跡（要素ツリー／スクリーンショット／ネットワーク）を添えて報告します。分類は `deterministic` か `flaky` かであり、緩めた通過には決してなりません。既存の並列実行と証跡の機構を再利用し、固定 sleep は導入しません。
- **安定性スコア（静的）。** シナリオをパースし、デバイスなしで採点します。各セレクタを安定性のはしご（[selectors.md](../../../docs/ja/selectors.md)）で採点し、`timeout` のない `wait` ステップや広すぎる条件でゲートされた待機を指摘し、安定した `id` で置き換えられる生座標ジェスチャを指摘します。`doctor` の id カバレッジスコア（`bajutsu/doctor.py`）と並ぶ、シナリオごとの決定性スコアを出力します。
- **run の provenance ／バージョンスタンプ（経時分析の前提）。** 各 run に、`manifest.json` と serve の run 記録（[BE-0015](../../proposals/BE-0015-web-ui-public-hosting/BE-0015-web-ui-public-hosting-ja.md) 7c-4）の両方へ、**シナリオの content hash**（すでに本文を保存している実行済み `scenario.yaml` の安定した指紋）、**ツールのバージョン**（`bajutsu.__version__`）、そして取得できるときは**シナリオの git revision**と**アプリの build id**（どちらも任意。すべての run が git 管理下とは限らず、build が分かるとも限らない）をスタンプします。デバイスのモデル／ランタイムはすでに取得済みです。これらのスタンプがあれば、経時ビューは貯まった run を content hash で束ね、合格率の推移を示し、**content hash（できればアプリ／ツールのバージョンも）が変わらないのに判定が反転する**シナリオを指摘できます。これが真のフレーキネスで、編集されたシナリオや動いたアプリと切り分けられます。これは貯まった run 記録を再利用し、フィールドを足すだけで、判定は足しません。
- **出力。** 助言的なレポート（HTML ／ JSON）と、参考情報としての終了ステータスです。シナリオと判定に対しては読み取り専用で、テストを編集することも、CI をゲートすることもありません。

新しい `audit` コマンドとして実装するか、`doctor` ／ `run` のフラグとして実装するかは、採用時に先送りする実装上の選択です。静的な部分は `doctor` の近縁であり、単にそれを拡張する形でもよいでしょう。

### 実装状況

- **静的な安定度スコア**と**反復＆差分（動的）**は、どちらも `bajutsu audit` コマンドとして出荷済みです。`bajutsu audit <scenario>` はデバイス無しでセレクタ／wait の安定度を採点し、`bajutsu audit <scenario> --repeat K --target <app>` は K 回実行して、結果が変動したステップやアサーションを報告します。助言的かつ read-only で、CI ゲートにはなりません。
- **run の来歴・バージョンスタンプ**を出荷しました。各 `manifest.json` に任意の `provenance` ブロック（`scenarioHash`＝実行した `scenario.yaml` の `sha256:` フィンガープリント、`toolVersion`、git 配下の run なら `gitRevision`）が載ります。蓄積した run を同一性でグルーピングするための、安価な前提部分です。純粋なメタデータで、判定には一切入りません。
- **残り:** スタンプ済みの run 記録を採掘する**縦断ビュー**（内容ハッシュでグルーピングし、経時の合格率を報告）と、同じ来歴を serve の DB run 記録に刻むこと（[BE-0015](../../proposals/BE-0015-web-ui-public-hosting/BE-0015-web-ui-public-hosting-ja.md) 7c-4）です。

## 検討した代替案

* **フレーキーテストの隔離 + 自動リトライを採用する（業界の一般的な答え／Maestro の `retry`）。**
  ゲートに対しては不採用です。フレーキネスを隠すからで、それはまさに Bajutsu が対置して売る性質です。監査はそれを意図的に反転させ、フレークを暴いて修正させます。
* **run が「フレーキーに見えるか」を LLM に判定させる。** 不採用。非決定的で不要です。同一の run をまたいだばらつきは、精密に機械検出できる事実です。
* **何もしない（現状）。** 許容できますが、最も強い差別化が暗黙のままで、導入候補者に対して証明できません。証跡と並列実行の機構はすでに存在するので、実行可能な証明書を出す費用は小さく済みます。
