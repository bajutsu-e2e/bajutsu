## はじめに

record のすべてのターンにフル解像度のスクリーンショットを添付するのをやめます。現在のループは、
最大で `max_steps` 回のターンそれぞれに新しい画像を送っています。この画像はターンあたりのコストの
なかで最も大きく、システムプロンプトやツール定義とは違ってプロンプトキャッシュの対象になりません。
本項目は、視覚情報（スクリーンショット）を必要なときだけ送る方式にします。ターンごとのメッセージ
には画面のアクセシビリティ要素を常に含め、スクリーンショットは実際に情報を足すときだけ添付します。
つまり、その画面を初めて見るとき、要素ツリーが操作に足りないほど貧弱なとき、あるいはエージェントが
画面を見たいと明示的に求めたときに限ります。要素だけで操作対象を特定できる画面では、後続のターンが
テキストのみになり、視覚情報を失わず、決定性の保証も弱めないまま、キャッシュ外の支配的なトークン
コストを削れます。

## 動機

record のループ（`bajutsu/record.py:record`）は、繰り返しのたびにスクリーンショットを撮り
（`_screenshot_bytes`、`record.py`）、`ClaudeAgent` はそれが存在すれば毎ターンのメッセージに
添付します（`_user_content` と `ImagePart`、`bajutsu/claude_agent.py`）。画像は生の PNG として
送られ、リクエスト内で base64 にそのままエンコードされます（`bajutsu/ai/anthropic.py`）。

ここを削るのが妥当だと言える理由が 2 つあります。

- **ターンあたりのコストを画像が支配しています。** record のループは膨らむ会話履歴を送り直しません。
  各 `next_action` は user メッセージ 1 通だけの独立したリクエストで、状態を先へ運ぶのは直近 6 件の
  アクションの短い要約だけです（`_render`、`claude_agent.py`）。そのためターンあたりの入力量は
  セッションを通じてほぼ一定で、その大半はスクリーンショットと要素ツリーが占めます。
- **画像はキャッシュされません。** プロンプトキャッシュが効くのは静的なシステムプロンプトとツール
  定義だけで（`cache_control: ephemeral`、`bajutsu/ai/anthropic.py`）、ターンごとの観測（画像と
  要素）はその性質上キャッシュ対象になりません。つまりスクリーンショットは、フル解像度のまま、毎
  ターン、`max_steps` 回すべてで全額支払うことになります。

しかし、ほとんどのターンでスクリーンショットは情報を足していません。エージェントは要素の `id` か
`label` でしか操作しないため（存在しない id を作り出すことはありません。`docs/ja/recording.md` を参照）、
操作対象がすべて特定できる画面では、画像は判断の決め手ではなく確認材料にすぎず、要素リストだけでも
同じアクションを選びます。スクリーンショットがコストに見合うのは限られた場面だけです。画面を初めて
見るとき、アクセシビリティツリーが取りこぼす操作対象に到達するとき（`tap_point` によるフォール
バック。たとえば idb が 1 つの不透明なグループに畳み込んでしまう、ボトムタブバーの個々のタブ）、
あるいはツリーが公開しない見た目や状態を読むときです。毎ターン送るのは、エージェントが使っていない
視覚情報にお金を払っているのと同じです。

本項目は
[BE-0178](../BE-0178-record-multi-action-turn/BE-0178-record-multi-action-turn-ja.md) を補完するもので、
互いに独立しています。BE-0178 は同一画面内のアクションをまとめてモデルのターン*数*を減らします。
本項目は、エージェントが必要としない画像を送り直さないことでターン*あたりのペイロード*を減らします。
両者は組み合わさります。ターンが減り、残る各ターンも軽くなります。あわせて、姉妹提案の
`record-screenshot-downscale`（画像を載せるターンで画像を適正サイズにする）と、姉妹提案の
`record-turn-payload-diet`（要素ツリーのテキストを痩せさせる）とも補完関係にあり、これらでターン
あたりコストのすべての項に手を入れられます。

### 精度は落とさず、引き換えにもしません

本設計は、視覚情報を**取り除くのではなく後回しにする**ように組み立てています。

- どの画面も、スクリーンショット付きで**少なくとも一度は**見ます（初回観測トリガー）。
- 貧弱なツリー、つまり id による特定ができず `tap_point` だけが入口になる場合は、**必ず**画像を
  付けます（貧弱ツリートリガー）。
- それでも残る必要は、エージェント自身が画面を要求することで拾います（エスカレーショントリガー）。
  このとき当該ターンだけを画像付きで再発行します。

したがってエージェントは必要なものを常に見られます。要素リストがアクションを完全に決められるターン
で画像を渡さないだけです。本項目がトークン削減を精度低下と引き換えにせず主張できるのは、この構成に
よります。

## 詳細設計

変更は作成経路（`record.py` / `claude_agent.py` / `agent.py`）に閉じ、記録されるシナリオの成果物は
バイト単位で変わりません。したがって `run`、`codegen`、レポートは影響を受けません。
`Observation.screenshot` はすでに `bytes | None` なので、ループはターンごとに値を入れるかどうかを
決めるだけで、エージェントには画像を取り戻す手段を 1 つ追加します。

### 1. ループがターンごとに添付の有無を決める

`record()` は直前ターンの画面フィンガープリント（`crawl.fingerprint(elements)`。画面遷移の検出に
すでに使っている id と状態の射影。`bajutsu/crawl.py`）を保持します。`Observation` を組み立てる前に、
現在の要素に対して 2 つの決定的なトリガーを評価します。

- **初回観測トリガー**：現在のフィンガープリントが直前ターンと異なる（または最初のターンである）。
  エージェントはまだこの画面を見ていないので、スクリーンショットを添付します。
- **貧弱ツリートリガー**：要素リストがセレクターで操作するには薄すぎる。id を持つ要素が 1 つも
  ない、操作対象が少ない、あるいはツリーが画面全体を覆う 1 つの不透明なコンテナに畳み込まれている
  （`tap_point` が想定される経路になる、id なしのタブバー的な場合）。スクリーンショットを添付します。

どちらも発火しなければ、つまりすでに見た画面で、操作対象を十分に特定できるツリーがあれば、その
ターンは**テキストのみ**にします。要素、プラン、直近アクションの要約だけで、`ImagePart` は付けません。
撮影自体も節約するため、スクリーンショットは**遅延取得**にし、毎回ではなく、トリガー（または後述の
エスカレーション）が求めたときにだけ撮ります。

### 2. ツリーで足りないときエージェントが視覚情報へエスカレーションできる

`claude_agent.py` に `need_screenshot` ツールを追加します（Claude Code バックエンド側には同等の
構造化出力フィールドを設け、どちらのプロバイダでも同じ挙動にします）。テキストのみのターンでは、
システムプロンプトがエージェントに次のように伝えます。画面の要素は常に渡す。足りるならそれで操作
する。**画面を見ないと進めないなら**、つまり必要な操作対象がリストにない、あるいはツリーが公開しない
見た目を読む必要があるなら、`need_screenshot` を呼ぶ。`proposal_from_call` はその呼び出しを、ステップ
を持たず `need_screenshot=True` の合図を載せた `Proposal` に対応づけます。ループはそれを見て、
**同じ、変化していない画面**について `next_action` をスクリーンショット付きで再発行します（画像は
ここで遅延取得します）。エスカレーションは 1 ターンにつき最大 1 回なので、あるターンのコストは
最大でもテキストのみの往復 1 回分増えるだけで、しかもそれは 2 つの決定的トリガーで拾えず、実際に
視覚情報を要した画面に限られます。

### 3. システムプロンプトの指針

システムプロンプトを拡張し、あるターンにスクリーンショットが**あるとは限らない**こと、操作対象の
特定には要素リストが常に権威を持つこと、`need_screenshot` を呼ぶ場面を明記します。既存のルール、
すなわちリストにある要素だけを操作し、`id`/`label` を優先し、リストに本当に存在しない操作対象に
だけ `tap_point` を使う、は変えません。テキストのみになりうるターンに対して同じルールが働くだけです。

### 4. タイミング・settle・秘匿の挙動は変えない

固定の `sleep` は導入しません。ループの settle 挙動とアクションごとのライブ解決はそのままです。
秘匿処理（BE-0047）はすでに要素テキスト中の秘密をマスクしますが、画像はマスクできません。本項目は
画像を**より少なく**送ることで、マスク不能な画像面をむしろ減らします（増やしません）。プライバシー
上の小さな副次的利点であって、後退ではありません。

### 決定性・プライムディレクティブ適合・ゲート

本項目は厳密に**Tier 1（record のみ）**にとどまります。`run` や CI にモデル呼び出しは加えません。
添付するかしないかの 2 つのトリガーは、要素ツリーに対する決定的な計算（フィンガープリントの比較と
id 被覆率の確認）であって、大規模言語モデル（LLM）の判断ではありません（プライムディレクティブ 1）。
エスカレーションも作成時にエージェントが行う要求で、ゲートの外です。アクションごとの解決は変わらず、
セレクターは今もライブ画面に対して一意に解決し、曖昧なら即座に失敗します（プライムディレクティブ 2）。
トリガーは汎用の要素ツリーだけを読み、アプリ固有の設定を必要としません（プライムディレクティブ 3）。
シナリオ成果物の形は不変なので、下流全体とそのテストは影響を受けません。

### コストモデルと唯一のトレードオフ

アクセシビリティが充実したアプリ（たとえば id ベースの showcase バリアント）では、ある画面での
後続ターンの多くがテキストのみになり、大きく削れます。id なしのアプリでは、貧弱ツリートリガーが
（正しく）画像を送り続けます。ここで残るリスクは、曖昧なテキストのみのターンでエージェントが
エスカレーションの往復を 1 回要する場合です。エスカレーションを常態ではなく例外にとどめるため、
貧弱ツリートリガーは意図的に**寛容**にします（id 被覆率が低ければ先回りして添付します）。難しい
画面で不要な画像を数枚送ることと引き換えに、エスカレーションの往復が連鎖するのを避けます。総じて、
計装のよいアプリでは明確な削減、計装の乏しいアプリでも損益分岐を下回らない結果になります。

### テスト戦略（Simulator 不要、Linux の `make check` ゲートに収まる）

- **スクリプト化した偽ドライバーとエージェントによるループテスト**：ある画面の最初のターンでは
  スクリーンショットを添付し、同じフィンガープリントの後続ターンでは添付しないことを確認する。
  フィンガープリントの変化で再添付されることを確認する。貧弱な（id なしの）要素リストでは
  フィンガープリントによらず添付することを確認する。遅延取得を確認する（テキストのみのターンでは
  偽ドライバーの `screenshot` が呼ばれない）。
- **エスカレーションテスト**：テキストのみのターンで `need_screenshot` を返す偽エージェントが、
  同じ画面についてループに画像付きの再発行を最大 1 回させ、最終的なアクションは通常どおり記録される。
- **バックエンド対応テスト**：`need_screenshot` が、API エージェントでは `proposal_from_call` を
  通じてエスカレーションの合図に対応づき、Claude Code の構造化出力の同等物も同じ合図に対応づく。
- **回帰**：毎ターン新しいフィンガープリントになるような画面では、現状どおり（毎ターン画像）に
  振る舞う。

### スコープと非目標

**スコープ内**：ターンごとの添付判断（初回観測トリガーと貧弱ツリートリガー）と遅延取得、両
バックエンドでの `need_screenshot` エスカレーション、システムプロンプトの指針。

**非目標**：添付した画像の*サイズ*を減らすこと（姉妹提案 `record-screenshot-downscale`）、要素
ツリーのテキストを痩せさせること（姉妹提案 `record-turn-payload-diet`）、ターン*数*を減らすこと
（BE-0178）、`crawl` の探索ループ（`bajutsu/crawl.py`。画面ごとの視覚情報の必要性が異なるためここ
では対象外）、シナリオスキーマや `run`/再生の挙動の変更。

### 決定事項

1. **決定的なトリガーを先に置き、エージェントのエスカレーションを安全網にする。** ループは要素
   ツリーから添付・省略を決め（LLM を使わない）、エージェントはツリーが不十分なときいつでも画像を
   取り戻せます。*根拠*：共通経路を無駄な往復から守りつつ、視覚情報が使えなくなることは決してない
   と保証します。構成上、精度は保たれます。
2. **貧弱ツリートリガーは寛容にする。** id 被覆率が低ければ、エスカレーションを待たずに先回りして
   添付します。*根拠*：計装の乏しいアプリで、画像 1 枚の節約とエスカレーションの往復 1 回を引き換え
   にするのを避け、そこでも損益分岐を下回らないようにします。
3. **撮影は遅延させる。** トリガーやエスカレーションが必要とするときだけ撮り、毎回は撮りません。
   *根拠*：テキストのみのターンでも `simctl screenshot` のサブプロセスを省け、トークン削減に加えて
   実時間の小さな利得になります。

## 検討した代替案

- **スクリーンショットを完全にやめる（要素のみで record する）。** トークン削減は最大ですが、
  `tap_point` の視覚フォールバックと見た目による曖昧性解消を失い、id なしのアプリで実際に精度が
  落ちます。却下：本件の趣旨はトークンを削りつつ精度を落とさないことなので、視覚情報は使える状態を
  保つ必要があり、廃止ではなくオンデマンドにします。
- **セッション全体の最初のターンだけ画像を送る。** より単純ですが、以後に現れる新しい画面すべてで
  エージェントを視覚情報から遠ざけ、そこは最も必要とされる場面です。却下：画面ごとの初回観測での
  ゲートを選びます。
- **画面途中で視覚情報が要るかどうかを、エージェントではなくループが判断する。** ループにはエージェ
  ントが*なぜ*詰まっているのかという概念がないため、常に送る（節約なし）か推測するかになります。
  却下：視覚情報を求めるべき場所はエージェントであり、判断を要さない場合を決定的トリガーが担います。
- **ターンごとの画像を `cache_control` でキャッシュする。** 画像はほとんどのターンで変わるため
  キャッシュはめったに当たらず、キャッシュ書き込みコストが増えます。ここでの利得は画像を*送らない*
  ことから来るのであって、キャッシュすることからではありません。主機構としては却下します。
