## はじめに

AI の record ループで、いまの画面だけを見て確定できるアクションが複数あるとエージェントが判断したとき、
それらを 1 回の観測から **まとめて提案し、まとめて実行する** ようにします。典型的にはフォームの複数項目を
埋めてから送信する場面です。現在のループは 1 ターンにつき 1 アクションに固定されていて、画面を観測し、
1 手を得るためにモデルとの往復を 1 回（これが遅い）行い、それを実行して、また繰り返します。1 ターンの
画面内アクションをまとめれば、N 回の「観測 → モデル → 実行」が 1 回の「観測 → モデル → N 手を実行」に
縮まります。1 手あたりの支配的なコストであるモデルとの往復とスクリーンショットの送信を減らせて、しかも
記録されるシナリオの形も、決定性の保証も変わりません。

## 動機

record ループ（`bajutsu/record.py:record`）は、繰り返しのたびに同じ 3 つの処理を行います。`driver.query()`
で現在の画面を取得し、`agent.next_action(...)` で 1 手の `Step` を得て（毎回、エンコードし直した
スクリーンショットを載せてモデルへネットワーク往復します）、その 1 手を実行します。作者がすでに手順を
覚えているフロー（「メールアドレスを入力し、パスワードを入力し、Sign in をタップする」）でも、ログイン画面を
一目見れば 3 手とも先に確定できるのに、モデルとの往復を 3 回まるごと消費します。

各繰り返しで重いのはモデルとの往復です。トークンを消費する遅いネットワーク呼び出しであり、そのたびに
スクリーンショットをエンコードし直して送ります（`_screenshot_bytes` と `ImagePart`）。ループ冒頭の
`driver.query()` とスクリーンショット取得も、その上に 1 手ごとに乗ります。項目の多い画面では、これが
record の実時間とトークンコストの大半を占めます。しかもそのコストは、作者が項目を埋めているあいだ、
実質的に変わっていない画面を観測し直すことに費やされています。

まとめて実行しても安全性はそのまま保たれます。これがこの提案の要になる前提です。アクチュエーションは、
実行するその瞬間の生きた画面に対して毎回セレクタを解決するからです。`IdbDriver.tap(sel)` は
`_center → _settle → _resolve` を呼び、そこで新しい `query()` と `resolve_unique` が走ります。ですから
バッチの内側であっても、（変化しているかもしれない）画面にもう一致しなくなった後続アクションは、いま
たまたま一致する何かをタップするのではなく、`ElementNotFound` や `AmbiguousSelector` で明示的に失敗します。
1 手ずつの record を決定的にしている安全網が、そのままバッチを途中で安全に打ち切る仕組みになります。
この項目は、各アクションの解決を雑にする話ではなく、消費するモデル往復の回数を減らす話です。

これは [BE-0105](../BE-0105-xcuitest-single-snapshot-query/BE-0105-xcuitest-single-snapshot-query-ja.md)
とは独立で、互いを補完します。BE-0105 は XCUITest バックエンドで *1 回の* `query()` を安くしました
（1 クエリあたりの XCUITest 往復を減らす）。この項目は、record セッションが必要とするモデル往復（および
ループ冒頭の観測）の *回数* を減らします。片方は 1 回の観測のコストを縮め、もう片方は観測の回数を削ります。

## 詳細設計

記録されるシナリオは `Step` の平坦なリストのままなので、`run`・`codegen`・レポートはまったく変わりません。
まとめて実行するかどうかは *record ループがステップをどう生み出すか* だけの話で、成果物には影響しません。
変更はオーサリング経路（`agent.py`・`claude_agent.py`・Claude Code バックエンド・`record.py`）に閉じます。

### 1. `Proposal` が順序付きのバッチを持つ

`bajutsu/agent.py:Proposal` を、単一の `step: Step | None` から順序付きの `steps: list[Step]` に広げます
（現在の 1 アクション提案は長さ 1 の場合になります。`finish` は従来どおり `expect` を伴う独立したフラグの
ままです）。バッチに入るのは、先行アクションの結果を見なくても *現在の* 観測から実行できるとエージェントが
判断したアクションです。バッチの切れ目はエージェントが決めます（決定 1）。`wait_for` もバッチの一員として
正当です（現在の画面にすでに見えている要素を待つ場合）。

### 2. エージェントが 1 ターンで複数アクションを出す

API エージェント（`ClaudeAgent.next_action`）は、応答全体をすでに `MessageResponse.content`（ブロックの
リスト）として受け取っていて、いまは `first_tool_use()` だけを残しています。まとめて実行するには、その
ターンの **すべての** tool-use ブロックを、既存の `proposal_from_call` を通して順番にステップへ写します。
これには強制ツールのターンで並列ツール実行を許す必要があります（現在の `tool_choice=AnyTool()` はちょうど
1 回の呼び出しを強制します）。リクエスト側で複数呼び出しを許可し、システムプロンプトに次のことを書き加えます。
モデルは、**各アクションが直前のアクションの結果を観測せずに現在の画面から確定できるときに限り**、1 ターンで
複数のアクションを出してよい（たとえばフォームの複数項目を埋めてから送信する）。そうでなければ従来どおり
1 アクションです。`finish` ブロックはバッチを終端します。同じターンでそれより前にあるアクションブロックが
先に実行され、その後に `finish` のアサーションでループが完了します（決定 3）。

Claude Code バックエンド（`bajutsu/ai/claude_code.py`）は並列ツール呼び出しではなく単一の構造化出力
オブジェクトを返すため、そのアクションスキーマに順序付きの `actions` リスト（またはリスト形の同等の出力）を
加え、そこに `proposal_from_call` を適用します。こうして両バックエンドとも、同じアクションの形を同じステップへ
写す点は変わりません。どちらのバックエンドも、プロバイダによらず同じ `Proposal.steps` を生み出します。

### 3. record ループがバッチを実行し、画面が変わったら打ち切る

`record()` は提案のステップを順に実行します。各ステップを実行した後、画面の `crawl.fingerprint(...)`
（画面遷移の検出にすでに使っている、識別子と状態の射影）を、バッチ開始前に取ったフィンガープリントと
比較します。

- **変わっていない** → バッチの次のステップへ進みます。
- **変わっていて、まだステップが残っている** → 画面が計画の前提から動いてしまったので、**残りのバッチを
  打ち切り**、次のループ反復で新しい画面から観測し直して計画し直します（決定 2、「仕切り直し」）。バッチの
  最後のステップが正当に遷移を起こす場合（たとえば送信のタップ）は打ち切りではありません。バッチにはもう、
  無効になりうるステップが残っていないからです。
- ステップの解決に失敗した場合（既存の `_execute_with_recovery` によるアラート解消の後の
  `base.SelectorError`）も同様に、残りのバッチを打ち切って次のターンで観測し直します。

肝心なのは、**実際に実行できたステップだけが記録されるシナリオへ追加される** ことです。打ち切られた
（実行されなかった）提案の残りは捨てられ、書き込まれません。ですから記録される YAML は、アプリに対して
実際に行われたことを、起きた順に 1 手ずつ、常に正確に反映します。従来のループと同じ成果物を、より少ない
モデル往復で生み出すだけです。

### 4. タイミングと settle の挙動は変わらない

バッチは構成上つねに画面内で完結するため（画面が変わった瞬間に打ち切ります）、バッチの *内側で* 画面を
またぐ `wait` を挿入する必要はありません。既存の settle の挙動もそのままです。`finish` のアサーションの前に
記録される `_settle_step` の wait も引き続き働き、各アクション自身のドライバ側の `_settle`/`_resolve`
リトライも引き続き守ります。固定の `sleep` はどこにも導入しません。

### 決定性・プライム・ディレクティブ遵守・ゲート

これは厳密に **Tier 1（record のみ）** にとどまります。`run` や CI にモデル呼び出しを一切足さず、
打ち切りの判定は LLM の判断ではなく決定的なフィンガープリント比較です（プライム・ディレクティブ 1）。
アクションごとの解決は変わりません。各ステップは引き続き生きた画面に対して一意に解決し、曖昧なセレクタは
即座に失敗するので、まとめて実行しても「最初に一致した何か」をタップすることはありません
（プライム・ディレクティブ 2）。アプリ固有の要素はなく、新しい config も読まず、ドライバのインターフェースも
触りません（プライム・ディレクティブ 3）。シナリオ成果物の形はバイト単位で同じなので、下流全体
（`run`/codegen/レポート）とそのテストは影響を受けません。

### テスト方針（Linux の `make check` ゲートに収まり、Simulator は不要）

- **スクリプト化した fake エージェントでのループテスト**：複数ステップの `Proposal` を返させて、fake
  ドライバのフィンガープリントが一定のあいだは全ステップが順に実行され記録されることを確認します。fake
  ドライバがバッチ途中でフィンガープリントの変化を報告したら、バッチが打ち切られ、実行できた前半だけが
  記録されることを確認します。バッチ途中の `SelectorError` でも打ち切られ、前半だけが記録されることを
  確認します。長さ 1 の提案が現在のループとまったく同じに振る舞うこと（回帰の防止）も確認します。
- **バックエンドの写像テスト**：複数の tool-use ブロックを持つ `MessageResponse` が、`proposal_from_call`
  を通して順序どおり `Proposal.steps` に写ること。アクションブロックの後ろに置かれた `finish` ブロックが、
  先行アクションを残したまま `done=True` になること。Claude Code のリスト形の出力が同じ `steps` に写ること。
- **ラウンドトリップ**：出力される `Scenario` が引き続き `dump_scenario_file` → `load_scenario_file` を
  そのまま往復すること（成果物の形が不変であること）。

### スコープと非目標

**スコープに含む**：エージェントが 1 ターンで画面内の順序付きバッチを提案すること。両バックエンドが同じ
`Proposal.steps` へ写すこと。record ループが、画面変化での打ち切りと解決失敗での打ち切りを伴ってバッチを
実行し、実行できた前半だけを記録すること。バッチが適切な場面を限定するシステムプロンプトの指針。

**非目標**：画面遷移をまたぐバッチ（バッチは構成上つねに画面内で完結し、遷移でバッチは終わります）。
`crawl` 探索ループ（`bajutsu/crawl.py`）。そのステップごとの分岐判断は探索に本来備わるもので、ここでは
対象外です。シナリオスキーマや `run`/replay の挙動の変更。1 回の `query()` のコスト削減（これは BE-0105
の別の関心事です）。

### 決定事項

1. **バッチの切れ目はエージェントが決める。** モデルが現在の画面を見て、先に確定できると判断した
   アクション群を提案します。ループが機械的にバッチを推測することはしません。*理由*：どのアクションを
   安全に先取りできるかは、フローに関する意味的な判断（いますべての項目が見えているか、それとも前のタップの
   後にはじめて現れる項目か）であり、オーサリングエージェントがそれを下す立場にあります。過信があっても
   打ち切りの安全網が決定的に捕まえるので、誤ったバッチは「前半を実行し、観測し直した」に劣化するだけで、
   おかしな記録には決してなりません。
2. **バッチ途中で画面が変わったら打ち切って観測し直す（「仕切り直し」）。** バッチのステップが残った状態で
   画面フィンガープリントが変わったら、残りのステップを捨て、次の反復で観測し直します。記録されるのは実行
   できたステップだけです。*理由*：画面が変わればバッチが計画された前提は無効になり、観測し直すのはまさに
   1 手ずつのループの挙動です。これにより記録は、古い計画を無理に押し通すのではなく、実際に起きたことに
   忠実であり続けます。
3. **`finish` はバッチを終端し、先行するアクションが先に実行される。** ターンはいくつかのアクションの後に
   `finish` で終わってよく、アクションが走ってから `finish` のアサーションでループが完了します。*理由*：
   「実行してから終了する」ための特別なプロトコルなしに、エージェントが 1 ターンで行動と結論の両方を下せます。
   アサーションも従来どおり settle した最終画面に対して走ります。

## 検討した代替案

- **ループが機械的にまとめる（エージェントの判断を使わない）。** モデルに切れ目を決めさせず、フィンガー
  プリントが変わるまで候補アクションのキューをループが貪欲に実行する案です。主案としては採らないことに
  しました。ループは *意図*（このフローで、どの見えている項目を、どの順で、どの値で埋めるか）を持たないため、
  結局アクションごとにモデルが必要になる（節約にならない）か、推測する（「そこにある何か」に作用する状態が
  戻ってくる）ことになります。バッチを決めるのはエージェントが適所で、フィンガープリントの確認は計画者では
  なく安全網です。
- **順序付きアクションのリストを取る専用の `batch` ツール。** 並列 tool-use ブロックを集める代わりに、
  入力がアクションオブジェクトの配列である 1 つのツールを足す案です。実現可能で、Claude Code バックエンドが
  結局必要とする形でもあります（こちらは 1 オブジェクトを返します）。API エージェントの主案としては採り
  ませんでした。アクションごとのスキーマを配列ラッパの内側に重複させ、「複数のツール呼び出しを出す」という
  自然な形から離れるためです。並列 tool-use が使えないバックエンドで、両者の整合を取る仕組みとして残します。
- **1 ターン 1 アクションのまま、観測だけをキャッシュする。** 直前の `query()`/スクリーンショットを次の
  ターンで再利用し、ループ冒頭の観測を省く案です。採りませんでした。アクションごとにモデル往復を 1 回ずつ
  消費する点（支配的なコスト）が残り、しかもモデルが古い画面に作用する危険があります。まとめて実行すれば
  往復そのものに正面から効き、アクションごとの生きた解決も保てます。
- **バッチを 1 つの複合ステップとして記録する。** シナリオへ「バッチ」ステップを 1 つ出す案です。明確に
  却下します。シナリオスキーマと replay の意味論を変えてしまい、オーサリング成果物が平坦で個別に解決される
  ステップのリストであり、それを `run` が決定的に再生するという不変条件を壊すためです。
