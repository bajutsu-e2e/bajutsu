## はじめに

`visual` アサーション（BE-0029）を拡張し、比較の対象を画面全体ではなく**単一の要素にスコープ**
できるようにします。あわせて、**除外領域を生のピクセル矩形ではなくセレクタで指定**できるように
します。どちらも完全に決定的なままです。比較は依然として保存済みベースラインに対するピクセル差分で
あり、モデルは関与しません。

## 動機

BE-0029 は決定的な `visual` アサーションを出荷しましたが、比較の対象は**画面全体**であり、動的な
コンテンツをマスクする手段はハードコードしたピクセル矩形（`{ x, y, w, h }`）だけです。ここから 2 つの
制約が生じます。どちらも MagicPod や mabl の要素単位ビジュアルテストがまさに解こうとしている問題です。

1. **画面全体の比較は壊れやすく、精度も粗いです。** 関心が 1 つのコンポーネント（グラフ、ボタン、
   アバター）だけにあるテストでも、画面上の*ほかの何か*（無関係なバナー、行が 1 つ増えたリスト）が
   変わるたびに失敗します。作者は「*この*カードだけを回帰テストし、残りは無視する」と指定できず、
   ベースラインが頻繁に差し替わって信号がノイズに埋もれます。競合が要素単位の画像比較を提供して
   いるのは、ビジュアル回帰の多くが局所的だからです。

2. **ピクセル矩形のマスクは端末やレイアウトをまたぐと脆いです。** `exclude` 領域は絶対座標で書くため、
   レイアウトが折り返したり、端末の解像度が変わったり、ロケールによってラベルが広がったりした瞬間に、
   気付かないままずれます。作者が実際に無視したいのは、たいてい*要素*（時計、「最終更新」のタイムスタンプ、
   ランダムに差し替わる広告枠）であって、固定された箱ではありません。要素をセレクタで指定すれば、
   座標を壊すのと同じ条件のもとでも安定します。

どちらもプライムディレクティブに無理なく収まります。要素を境界矩形に解決し、ピクセル差分の前に切り出し
またはマスクするのは、純粋で決定的な機械操作だからです。`run`/CI ゲートに AI は入り込まず、同じ画面は
毎回同じ判定を返します。これは BE-0029 が「曖昧すぎる」として棚上げされずに受け入れられたのと同じ論理です。

## 詳細設計

`VisualMatch` スキーマ（`bajutsu/scenario/models/assertions.py`）への追加は 2 つで、いずれも任意であり
後方互換です。既存の `visual:` ブロックは、これまでどおり画面全体を比較する挙動を保ちます。

**1. `element:` — 比較を 1 つの要素の境界矩形にスコープする。**

```yaml
expect:
  - visual:
      element: { id: "summary-card" }   # Selector。一意に解決する
      baseline: summary-card.png
      threshold: 0.1
```

セレクタは既存の一意解決ルール（[DESIGN §5](../../DESIGN.md)）で解決します。**曖昧なセレクタは即座に
失敗し**（プライムディレクティブ 2）、「最初に一致したものを切り出す」ことは決してしません。要素のフレームはドライバの要素ツリーから読み取り、
撮影したスクリーンショットをそのフレームに切り出して、ベースラインと差分を取ります。つまりベースラインは
画面ではなく*要素*そのものになるため、無関係な変化に強くなります。

**2. セレクタによるマスク — `exclude` の要素に矩形の代わりにセレクタを指定できるようにする。**

```yaml
expect:
  - visual:
      baseline: home.png
      exclude:
        - { selector: { label: "last updated" } }   # この要素のフレームをマスク
        - { x: 0, y: 0, w: 390, h: 54 }              # 生の矩形も引き続き使える
```

`ExcludeRegion` を共用体にします。現在の `{ x, y, w, h }` 矩形**または** `{ selector: <Selector> }`
の形です。セレクタによるマスクは評価時に要素のフレームへ解決し、現在の矩形とまったく同じようにマスク
します。何にも一致しないセレクタはマスクの無操作です（画面上に隠す対象がありません）。曖昧なセレクタは
失敗し、ディレクティブ 2 と整合します。

### 作業分解（MECE）

- **スキーマ** — `VisualMatch` に `element: Selector | None` を追加し、`ExcludeRegion` をピクセル矩形と
  `{ selector: Selector }` 形式の共用体にします。どちらも任意にして、BE-0029 のシナリオを変更せずに
  保ちます。
- **フレーム解決** — `Selector` をドライバの要素ツリー経由でピクセルフレームに解決する小さなヘルパーを
  用意し、既存の一意解決経路を再利用します（曖昧なら失敗）。
- **フレーム解決** — `Selector` をドライバの要素ツリー経由でスクリーンショットのピクセルフレームに解決する
  小さなヘルパーを用意し、既存の一意解決経路を再利用します（曖昧なら失敗）。要素のフレームはポイント単位、
  スクリーンショットはデバイスピクセル単位なので、スクリーンショットとポイントの比（スクリーンショットの
  サイズをポイント空間の画面範囲で割った値）でスケールします。
- **比較エンジン**（`bajutsu/visual.py`）— 変更しません。ベースラインは*要素*の切り出し画像（下記）なので、
  評価段階で actual を要素フレームに切り出し、既存エンジンが要素サイズの 2 画像を比較します。解決済みの
  セレクタマスクは既存のマスク経路に渡す `ExcludeRegion` 矩形が増えるだけです。ピクセル差分の中核は
  そのままです。
- **評価**（`bajutsu/assertions.py` の `_eval_visual` / `VisualContext`）— `element` とセレクタマスクを
  ピクセルフレームへ解決し、actual を要素に切り出して（その切り出しをエビデンスの `actual` として記録）、
  マスクを切り出しの局所座標へ変換し、要素サイズのベースラインと比較します。
- **エビデンスとレポート** — `visual` のエビデンスに、比較が要素スコープだったか（`element_scoped`）、
  どのセレクタがマスクしたか（`masked_selectors`）を記録します。HTML レポートの baseline / actual / diff の
  並びには切り出し後の画像（記録されたパスそのもの）と由来のバッジを描画し、レビュアーが何を比較したのかを
  そのまま見られるようにします。
- **承認ワークフロー** — `bajutsu approve` は、要素スコープのベースラインを画面全体の場合と同じ手順で
  昇格させます。エビデンスの `actual` が指すものをコピーするだけで、要素スコープでは切り出し画像を指すため、
  特別扱いは不要です（確認済み）。
- **ドキュメントとスキーマリファレンス** — `element` とセレクタマスクをシナリオスキーマと
  `docs/` / `docs/ja/` に記載し、showcase のビジュアルシナリオに例を追加します。
- **テスト** — 要素スコープの成功と失敗、曖昧なセレクタの失敗、動的な要素を隠すセレクタマスク、
  変更していない画面全体の `visual:` ブロックの後方互換性を確認します。

## 検討した代替案

- **`visualElement` という別のアサーション種別。** 却下しました。`baseline` / `threshold` / `exclude` を
  重複させ、スキーマを分断してしまいます。既存の `visual` に任意の `element` フィールドを足せば、概念は
  1 つに保てます。
- **関心領域の自動検出。** ここでは却下しました。「関心のある部分」を推論するのは判断を要する作業で、
  LLM に近づきますが、LLM はゲートに入れられません（ディレクティブ 1）。作者が要素を明示的に指定する
  ことで、決定性を保ちます。
- **セレクタマスクだけ、または要素スコープだけを入れる。** どちらも同じフレーム解決ヘルパーを共有し、
  同じ脆さを両端から解きます（何を*比較する*かと、何を*無視する*か）。まとめて出荷するほうが片方ずつより
  安く、競合が提供する要素単位の機能を作者に一通り届けられます。
