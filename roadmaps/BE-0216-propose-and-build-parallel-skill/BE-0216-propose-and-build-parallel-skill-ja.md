## はじめに

Bajutsu のロードマップワークフローは、意図的に二段階に分かれています。[`ideation`](../../.claude/skills/ideation/SKILL.md)
スキルは BE（Bajutsu Evolution）項目の提案を**執筆**し、ロードマップのファイルで手を止めます。
[`implement-be`](../../.claude/skills/implement-be/SKILL.md) スキルは、番号の付いた既存の項目をその
ID から**出荷**します。この二つは構造上、必ず直列になります。恒久的な `BE-NNNN` の ID は、提案 PR が
`main` にマージされた**後**に初めて採番されるからです（[BE-0089](../BE-0089-merge-time-be-id-allocation/BE-0089-merge-time-be-id-allocation-ja.md)）。
そのため、提案がマージされるまでは、本物の ID の下で実装を始められません。

この項目は、**三つめ**のスキルとして `propose-and-build` を追加します。小さくスコープの定まった機能で、
著者が設計に十分な確信を持っている場合に、提案とその実装を同時に書き、二本の PR のスタックとして着地させる
ためのものです。着地の順序は、**BE 提案が先、実装が後**です。このスキルは `ideation` / `implement-be`
を置き換えるものではなく、BE-0089 のマージ時採番も変えません。スタックは、提案 PR がレビュー中のあいだ
だけ存在する**一時的な**構成であり、提案がマージされて ID が採番された瞬間に、二本の独立した PR へと畳み
直されます。設計をコードより先にレビューしたい通常のケースでは、これまでどおり直列の経路が既定のままです。

## 動機

### 直列の経路には実際の待ち時間があり、小さな項目ではそれが純粋なオーバーヘッドになる

`ideation` → マージ → 採番 → `implement-be` という連鎖は、既定として妥当です。コードを書く前に設計を
レビューに通すことを強制しますし、実際に出荷される項目にだけ番号を使うことで `BE-NNNN` の連番を隙間なく
保ちます（BE-0089）。しかしこの連鎖は、小さくリスクの低い機能では著者がすでに頭の中に完成した形で持って
いることの多い二つの活動を、直列にしてしまいます。そうした項目でも、著者は次の手順を踏まねばなりません。

1. `ideation` を実行し、提案 PR を開いて、マージされるのを**待つ**。
2. `roadmap-id` ワークフローが `main` 上で `BE-NNNN` を採番するのを待つ。
3. そこで初めて `implement-be BE-NNNN` を実行し、手順 1 で持っていた文脈をすべて立て直す。

(1) と (3) のあいだの待ちは、著者が今すぐ作れる変更にとっては空白の時間です。そして手順 3 では、
提案・関連箇所・周辺コードを読み直すという、文脈を取り戻すためのコストを払います。これは ideation の
段階ですでに払ったものです。プライムディレクティブのどれも、この直列化を要求してはいません。直列化は
ID 採番のタイミングから派生するものであって、正しさの必要から来るものではありません。

### しかし素朴に並行化するとマージ時採番と衝突する

「提案と並べてコードも書けばよい」という素朴な解決は、BE-0089 に真正面からぶつかります。提案がマージ
されるまで、その項目には本物の ID が**ありません**。ツリー上では `BE-0216` であり、提案 PR は慣習として
`[BE-NNNN]` タイトルプレフィックスを**持ちません**（`scripts/lint_pr.py` はここでプレフィックスの有無を
要求も禁止もしません。タイトルチェックが働くのはブランチ名が本物の ID を含む場合だけであり、
`claude/<topic>` のような BE 起票用ブランチは決して ID を含まないからです）。したがって実装は、本物の
ID に*依存する* `implement-be` の二つのステップ、すなわち項目を
`Status: Implemented` にして `Implementing PR` 行を付ける処理と、実装 PR のタイトルに `[BE-NNNN]` を
付ける処理を守れません。並行フローには、明示的で順序の定まった**引き継ぎ**が要ります。ID が判明し、
実装ブランチがそれを取り込む地点を、はっきり定めるということです。この規律がなければ、「並行」は番号を
めいめいに推測する半分同期した二つのブランチへと崩れます。これはまさに、BE-0061 と BE-0089 が防ぐために
作られた、衝突と欠番を招きやすい状態です。

### なぜドキュメントの注記ではなくスキルにするのか

引き継ぎには、正確で間違えやすい手順があります。提案がマージされたらリベースし、実装差分の中に残る古く
なった `BE-0216` 参照を採番後の `BE-NNNN` に書き換え、スタックした PR のベースを提案ブランチから `main`
へ付け替え、*そのうえで* `implement-be` の昇格ステップとゲートを走らせる、という順序です。これを、
`ideation` や `implement-be` が自分たちのフローを収めているのと同じやり方でスキルとして符号化すれば、
安全な経路がそのまま楽な経路になり、既存の二つのスキルは変更されず単一の役目のまま保たれます。

## 詳細設計

`propose-and-build` は、明示的な引き継ぎを軸にした二つの既存スキルの**合成**であって、どちらかを書き直す
ものではありません。`ideation` の執筆規則と `implement-be` の実装・昇格規則をそのまま再利用します。新しい
内容は、スタックと引き継ぎの段取り、そして*いつ*これを使うかというプライムディレクティブ由来のガード
レールだけです。

### 二本の PR のスタックとそのライフサイクル

```
                 （小さくスコープの定まった機能に著者が確信を持っている）
                                    │
      ┌─────────────────────────────┴─────────────────────────────┐
      ▼                                                             ▼
  PR #1  BE 提案                                          PR #2  実装
  branch: claude/<topic>                                 branch: claude/<topic>-impl
  base:   main                                           base:   claude/<topic>   （スタック）
  BE-0216-<slug>/ @ Status: Proposal                     BE-0216 仕様に対するコード + テスト
  素の scoped タイトル、[BE-…] プレフィックスなし          DRAFT、Status フリップなし、[BE-…] なし
      │                                                             │
      │  そのままレビューされマージ（BE-0216 のまま）               │  draft のまま、#1 の変化に追随してリベース
      ▼                                                             │
  roadmap-id が main 上で BE-NNNN を採番（BE-0089）                 │
      │                                                             │
      └──────────────────────── 引き継ぎ ───────────────────────────┘
                                    │
                                    ▼
   #2: origin/main にリベース · 差分中の BE-0216→BE-NNNN を書き換え · ベースを main へ付け替え
       · implement-be のステップ 8〜10（Status フリップ、Implementing PR 行、索引再生成、ゲート、[BE-NNNN] タイトル）
```

このスタックは**一時的**です。PR #2 のベースが提案ブランチであるのは、#1 が開いているあいだだけです。#1 が
マージされて ID が採番されると、#2 は `main` にリベースされてベースを付け替えられ、通常の `implement-be`
の形をした PR になります。この設計が BE-0089 と両立するのは、この点によります。番号を*いつ*採番するかは
何も変わらず、スキルは採番後に実装ブランチがそれをどう取り込むかを定めるだけです。

### 作業分解（MECE）

1. **SKILL.md を執筆する。** `.claude/skills/propose-and-build/SKILL.md` を作成します。フロントマター
   （`name: propose-and-build`、「提案と実装を同時に作る」「BE とコードを一緒に書く」「実装を提案の上に
   スタックする」といったトリガー句を含む `description`、BE-0103 に沿った既定の `model:`）と、以下の各
   フェーズを構造とする本文を持たせます。`ideation` と `implement-be` の内容は複製せず、リンクで参照します。
2. **フェーズ A — 提案を執筆する（`ideation` の規則に委譲）。** `claude/<topic>` を `origin/main` から
   切り、`make new-roadmap-item` で `BE-0216-<slug>/` を `Status: Proposal` として scaffold し、内容を
   埋め、日本語側を [`japanese-tech-writing`](../../.claude/skills/japanese-tech-writing/SKILL.md) スキルに
   従って整えます。PR #1 は素の scoped タイトルで、`[BE-…]` プレフィックス**なし**で開きます。`ideation`
   と同一であり、スキルは再掲せずそう述べます。
3. **フェーズ B — プレースホルダに対して実装する（`implement-be` ステップ 3〜7 に委譲）。**
   `claude/<topic>-impl` を `claude/<topic>` の**上にスタックして**作成します（あるいは分離した
   `make worktree` の作業スペースにして、二つを本当に並行して進められるようにします）。`BE-0216-<slug>/` の提案を仕様と
   して扱い、`implement-be` のコードを読み込むステップ、計画して確認するステップ、テスト付きで実装する
   ステップ、差分をレビューするステップを実行します。**ただし、まだ採番されていない ID に依存する二つの
   処理は外します。** `Status` を Implemented にフリップ**しません**し、`[BE-NNNN]` プレフィックスも付け
   **ません**。PR #2 は `base: claude/<topic>` の **draft** として開きます。
4. **トラッキング Issue はまだ存在しない。** `implement-be` のステップ 2 は項目の `roadmap-tracking`
   Issue を自己アサインしますが、その Issue は [BE-0109](../BE-0109-roadmap-tracking-issues/BE-0109-roadmap-tracking-issues-ja.md)
   の同期が、番号の付いた項目が `main` に存在してから初めて作成します。フェーズ B のあいだはアサインする
   対象がありません。スキルはこれを注記し、自己アサインを引き継ぎ（またはその後の最初の `implement-be`
   相当の実行）まで先送りします。
5. **引き継ぎ（要となる新しいステップ）。** PR #1 がマージされ、`roadmap-id` が `main` 上で `BE-NNNN` を
   採番したことが引き金になります。実装ブランチで、次の順に行います。
   - `git fetch origin && git rebase origin/main`。マージされた提案*と*採番コミットの両方を取り込みます。
     `BE-0216-<slug>/` → `BE-NNNN-<slug>/` のディレクトリ改名は git のリネームなので、衝突は機械的に解けます。
   - PR #2 の差分**そのものの中**にある `BE-0216` 参照（ドキュメントの相互参照、テストフィクスチャ、
     コメント）をすべて、採番後の `BE-NNNN` に書き換えます。提案自体のファイルはワークフローが `main` 上で
     すでに採番済みなので、このステップは*実装*が持ち込んだ参照だけを直します。
   - PR のベースを提案ブランチから `main` へ付け替えます（`gh pr edit --base main`）。GitHub はスタック
     した PR のベースブランチがマージ時に削除されると自動でベースを付け替えますが、ベースブランチが削除
     されることにフローが依存しないよう、スキルは明示的に行います。
   - `BE-NNNN` が存在するようになったので、トラッキング Issue を自己アサインし（`implement-be` ステップ 2）、
     `implement-be` のステップ 8〜10 を実行します。両言語のファイルで `Status: Implemented` にフリップして
     `Implementing PR` 行を付け、`make roadmap-index`、`make check` を走らせ、PR を ready にし、`[BE-NNNN]`
     タイトルプレフィックスを付けます。
6. **ガードレールとスコープ（プライムディレクティブ由来）。** スキルは、*いつ*これを使い、いつ使わないかを
   はっきり述べます。
   - 使うのは、小さくスコープの定まった項目で、著者が設計をレビューで大きく変えられるとは見込んでいない
     場合に限ります。レビューが提案を作り直せば、実装ブランチが手戻りを負います。スキルはこれを許容した
     トレードオフとして明記し、設計が本当に不確かなときは直列の経路に戻すよう述べます。
   - このスキルは執筆*と*実装の両方を行うので、製品コードに一切触れない `ideation` とは違い、`bajutsu/` /
     `BajutsuKit/` / テストを `implement-be` の規則の下で**変更します**。スキルは、フェーズ A が執筆のみに
     とどまり、製品コードは別の実装ブランチ上のフェーズ B でだけ現れることを明示します。
   - 三つのプライムディレクティブはすべて、`implement-be` の場合とまったく同じように効きます。`run` / CI の
     判定経路に LLM を置かないこと、決定性を最優先すること、アプリ非依存の設定に寄せることです。スキルは
     CI に自動化を足しません。人間・エージェントの手順にすぎないので、どのゲートにも新しい仕組みを持ち込み
     ません。
7. **ドキュメント。** [`docs/ai-development.md`](../../docs/ai-development.md)（と `docs/ja/` の対応版）に、
   三つのスキルの三角形、すなわち `ideation`（執筆のみ）、`implement-be`（番号付き項目を出荷）、
   `propose-and-build`（両方をスタックで）を説明し、それぞれをいつ選ぶかを述べる短い小節を追加します。
   三つの SKILL.md の `References` を相互リンクします。

### 機械的に検証できる成果

このスキルは手順なので、決定的な成果は実装ブランチが生み出したものに既存のゲートを当てたものになります。
引き継ぎ後の PR #2 で `make check` が緑であること、ロードマップ索引が `Status: Implemented` を反映して
いること、そして（引き継ぎ後の）実装ブランチが `BE-NNNN` を含むようになった時点で CI の `pr-title`
チェックが強制するとおり、そのブランチのタイトルに一致する `[BE-NNNN]` プレフィックスがあることです。
提案 PR のタイトルにプレフィックスが*ない*こと自体は CI が検証する対象ではありません
（`scripts/lint_pr.py` はブランチが本物の ID を含む場合にだけプレフィックスを検証し、BE 起票用ブランチは
そもそも ID を含まないからです）。素のタイトルはスキルが従う慣習であって、CI が強制するものではありません。
新しいチェッカーは足しません。スキルの正しさは、すでにゲートされている `implement-be` の終状態と、
SKILL.md が符号化する順序の規律に帰着します。

## 検討した代替案

- **何もせず、`ideation` → `implement-be` の厳密な直列分割を保つ。** 安全な既定であり、ほとんどの項目では
  今も正しい選択です。*唯一の*選択肢としてだけ却下します。著者が今すぐ作れる小さな項目に、避けられる待ち
  時間と文脈の取り直しを課すからです（前述の動機のとおり）。
- **恒久的なスタック PR。実装 PR を提案ブランチにベースしたままマージまで通し、スタックを順にマージする。**
  「真の」スタックとして見た目は綺麗ですが、BE-0089 に真正面からぶつかります。実装 PR は、その `Status`
  フリップとタイトルプレフィックスのために、提案がマージされる*前*に本物の ID を必要とします。これはまさに
  マージ時採番が与えることを拒むものです。これを着地させるには、マージ前採番とその欠番・衝突の問題
  （BE-0061、BE-0089）を復活させることになります。却下します。一時スタックしてから畳む設計は、採番の
  タイミングに触れずに並行性を得られます。
- **このフローのためだけに ID を早く採番する（PR オープン時または承認時に）。** 恒久的なスタックは動く
  ようになりますが、BE-0089 が閉じたまさにその決定を開き直します（項目が受理される前に番号を使う、番号が
  連番でなくなる）。リポジトリ全体の不変条件にスキル単位の例外を設けるのは、取り除く直列化よりも悪い
  ことです。却下します。
- **提案と実装を一つにまとめた巨大 PR。** ブランチ運用は最も単純ですが、二段階ロードマップが拠って立つ
  「提案をそれ自体の是非でレビューする」という分離を潰し、BE-0089 を無効化します（項目が `BE-0216` の
  期間を経ずに一度に採番・実装される）。提案 PR に製品コードを載せることにもなり、レビューを濁します。
  却下します。二本の PR のスタックは、提案を単独でレビューできる状態に保ちます。
- **引き継ぎを CI で自動化する（提案がマージされたら実装ブランチをリベース・採番するワークフロー）。**
  手動のリベースをなくせる点で魅力的ですが、貢献者の実装ブランチと製品コードに触れる、バイパス権限を持つ
  自動化を足すことになり、BE-0089 が慎重に区切った `roadmaps/**` という狭い影響範囲を大きくはみ出します。
  見送ります。スキルはまず手動の手順を定めます。ヘルパー（たとえばリベースと参照書き換えを走らせる `make`
  ターゲット）はフローが実証されてからの後続候補ですが、CI での完全な自動化はスコープ外とします。
