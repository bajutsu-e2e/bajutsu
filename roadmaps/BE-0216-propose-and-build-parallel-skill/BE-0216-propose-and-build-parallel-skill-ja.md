## はじめに

Bajutsu のロードマップワークフローは、意図的に二段階に分かれています。[`ideation`](../../.claude/skills/ideation/SKILL.md)
スキルは BE（Bajutsu Evolution）項目の提案を**執筆**し、ロードマップのファイルで手を止めます。
[`implement-be`](../../.claude/skills/implement-be/SKILL.md) スキルは、番号の付いた既存の項目をその
ID から**出荷**します。この二つは構造上、必ず直列になります。恒久的な `BE-NNNN` の ID は、提案 PR が
`main` にマージされた**後**に初めて採番されるからです（[BE-0089](../BE-0089-merge-time-be-id-allocation/BE-0089-merge-time-be-id-allocation-ja.md)）。
そのため、提案がマージされるまでは、本物の ID の下で実装を始められません。

この項目は、**三つめ**のスキルとして `propose-and-build` を追加します。小さくスコープの定まった機能で、
著者が設計に十分な確信を持っている場合に、提案とその実装を同時に書き、ロードマップ項目とコードとテストを
まとめて運ぶ**1 つの BE 作成 PR** として着地させるためのものです。このスキルは `ideation` / `implement-be`
を置き換えるものではなく、BE-0089 のマージ時採番も変えません。項目は PR が生きているあいだ `BE-XXXX` の
プレースホルダを保ち続け、PR がマージされた瞬間に、通常の提案とまったく同じように CI が本物の ID を採番
します。設計をコードより先にレビューしたい通常のケースでは、これまでどおり直列の経路が既定のままです。

## 動機

### 直列の経路には実際の待ち時間があり、小さな項目ではそれが純粋なオーバーヘッドになる

`ideation` → マージ → 採番 → `implement-be` という連鎖は、既定として妥当です。コードを書く前に設計を
レビューに通すことを強制しますし、実際に出荷される項目にだけ番号を使うことで `BE-NNNN` の連番を隙間なく
保ちます（BE-0089）。しかしこの連鎖は、小さくリスクの低い機能では著者がすでに頭の中に完成した形で持って
いることの多い二つの活動を、直列にしてしまいます。そうした項目でも、著者は次の手順を踏まねばなりません。

1. `ideation` を実行し、提案 PR を開いて、マージされるのを**待つ**。
2. `roadmap-id` ワークフローが `main` 上で `BE-NNNN` を採番するのを待つ。
3. そこで初めて `implement-be BE-NNNN` を実行し、手順 1 で持っていた文脈をすべて立て直す。

(1) と (3) のあいだの待ちは、著者が今すぐ作れる変更にとっては空白の時間です。そして手順 3 では、
提案・関連箇所・周辺コードを読み直すという、文脈を取り戻すためのコストを払います。この取り直しは ideation
の段階ですでに払ったものです。プライムディレクティブのどれも、この直列化を要求してはいません。直列化は
ID 採番のタイミングから派生するものであって、正しさの必要から来るものではありません。

### マージ時採番は 1 つの PR の形を決めるが、1 つの PR を妨げはしない

「提案と並べてコードも書き、両方まとめて着地させればよい」という素朴な解決は、PR がマージされるまで項目に
本物の ID がないため、BE-0089 とぶつかるように見えます。しかし二つの事実を分けて見れば、ぶつかりません。
第一に、`implement-be` の PR が通常持つ二つの仕上げは、どちらも番号を必要とするわけではありません。項目を
`Status: Implemented` にする処理と `Implementing PR` 行を記録する処理は、状態と PR 番号だけに依存し、ID には
依存しません。ですから項目は、`BE-XXXX` のプレースホルダを保ったまま `Status: Implemented` に到達できます。
`roadmap-id` ワークフローがマージ時にプレースホルダを採番後の `BE-NNNN` へ書き換え、項目はマージ後の手直し
なしに、採番されて Implemented の状態で着地します。番号を本当に必要とするのは `[BE-NNNN]` タイトルプレ
フィックスだけであり、BE 作成 PR はそれを持つ必要がありません。`scripts/lint_pr.py` がタイトルチェックを
働かせるのはブランチ名が本物の ID を含む場合だけであり、`claude/<topic>` のようなブランチは決して ID を含み
ません。素の scoped タイトルは、あらゆる提案 PR と同じくここでも正しい形です。

第二に、採番スクリプト（[`scripts/allocate_roadmap_ids.py`](../../scripts/allocate_roadmap_ids.py)）は、
`BE-XXXX` → `BE-NNNN` の書き換えを**項目自身のディレクトリの中だけ**で行い、リポジトリ全体を掃きません。
ですからマージ時採番がこのフローに課す唯一の制約は、`BE-XXXX` のプレースホルダが項目自身のファイル以外の
どこにも現れないことです。実装コード、テスト、コメント、あるいは別の場所のドキュメントに書かれたプレース
ホルダは、マージを越えて `main` に古い参照として残ります。マージ時採番は、ID が現れてよい場所を定めること
で 1 つの PR の*形を決める*のであって、1 つの PR を退けるのではありません。

### なぜドキュメントの注記ではなくスキルにするのか

1 つの PR のフローには、小さく正確で間違えやすい規律があります。プレースホルダの ID を項目自身のファイルに
閉じ込め、番号を待たずにプレースホルダの上で `Status: Implemented` にし、タイトルを素のまま保ち、`roadmap-id`
を実際に発火させる経路でマージする、というものです。このフローを、`ideation` や `implement-be` が自分たちの
フローを収めているのと同じやり方でスキルとして符号化すれば、安全な経路がそのまま楽な経路になり、既存の
二つのスキルは変更されず単一の役目のまま保たれます。

## 詳細設計

`propose-and-build` は、二つの既存スキルの**合成**であって、どちらかを書き直すものではありません。`ideation`
の執筆規則と `implement-be` の実装・昇格規則をそのまま再利用します。新しい内容は、1 つの PR の段取り、ID を
閉じ込める不変条件、そして*いつ*これを使うかというプライムディレクティブ由来のガードレールだけです。

### 1 つの PR とそのライフサイクル

```
                 （小さくスコープの定まった機能に著者が確信を持っている）
                                    │
                                    ▼
   1 本のブランチ: claude/<topic>   base: main
     roadmaps/BE-XXXX-<slug>/  @ Status: Implemented   （プレースホルダ ID。BE-XXXX は
     提案仕様に対するコード + テスト                       項目自身のファイルの中だけ）
     素の scoped タイトル、[BE-…] プレフィックスなし
                                    │
                                    │  人が PR を開き、Implementing PR 行の番号は
                                    │  追いの push で埋め、そのうえでマージする
                                    ▼
   roadmap-id が main 上で BE-NNNN を採番（BE-0089）
     · BE-XXXX-<slug>/ → BE-NNNN-<slug>/ にリネーム
     · 項目自身のファイルの中で BE-XXXX → BE-NNNN を書き換え
                                    │
                                    ▼
   項目は採番されて Implemented の状態で着地。マージ後の手直しなし
```

`BE-XXXX` のプレースホルダは PR のレビュー全体を通じて生きており、番号が確定する唯一の地点がマージです。
この設計が BE-0089 と両立するのは、この点によります。番号を*いつ*採番するかは何も変わらず、スキルはマージ
時の書き換えが仕上がった項目を着地させるように PR を整えるだけです。

### 作業分解（MECE）

1. **SKILL.md を執筆する。** `.claude/skills/propose-and-build/SKILL.md` を作成します。フロントマター
   （`name: propose-and-build`、「提案と実装を同時に作る」「BE とコードを一緒に書く」といったトリガー句を
   含む `description`、BE-0103 に沿った既定の `model:`）と、以下の各フェーズを構造とする本文を持たせます。
   `ideation` と `implement-be` の内容は複製せず、リンクで参照します。
2. **フェーズ A — 提案を執筆する（`ideation` の規則に委譲）。** `claude/<topic>` を `origin/main` から
   切り、`make new-roadmap-item` で `BE-XXXX-<slug>/` を `Status: Proposal` として scaffold し、内容を
   [`document-writing`](../../.claude/skills/document-writing/SKILL.md) スキルに従って埋め、日本語側を
   [`japanese-document-writing`](../../.claude/skills/japanese-document-writing/SKILL.md) に従って整えます。
   `ideation` と同一であり、スキルは再掲せずそう述べます。
3. **フェーズ B — プレースホルダに対して実装する（`implement-be` ステップ 3〜9 に委譲）。** **同じ
   ブランチ**で、`BE-XXXX-<slug>/` の提案を仕様として扱い、`implement-be` のコードを読み込むステップ、
   計画して確認するステップ、テスト付きで実装するステップ、差分をレビューするステップを実行します。そのうえ
   で昇格ステップを**この時点で**走らせます。`Status` と PR 番号は ID に依存しないからです。両言語のファイル
   で `Status: Implemented` にフリップして `Progress` のチェックを付け、`BE-XXXX` のプレースホルダを保った
   まま `Implementing PR` 行を付けます。PR 番号はフェーズ B のあいだ存在しないので、行の番号は人が PR を開いた
   あと（後述）の追いの push で埋めます。`implement-be` がステップ 8 で埋めをステップ 10 に先送りするのと同じ
   やり方です。番号かマージ後にしか存在しない Issue を
   必要とする二つの処理は先送りします。`[BE-NNNN]` タイトルプレフィックス（PR は素の scoped タイトルのまま）
   と、トラッキング Issue の自己アサイン（[BE-0109](../BE-0109-roadmap-tracking-issues/BE-0109-roadmap-tracking-issues-ja.md)
   の Issue は番号の付いた項目が `main` に存在してから作成される）です。これは ID がマージ時にのみ採番される
   BE 作成 PR なので、スキルはこの PR を自動では作成せず、ブランチを push して人が開くのに委ねます（BE-0230）。
   製品コードを含むので Draft として開きます。
4. **唯一の不変条件 — プレースホルダの ID を閉じ込める。** スキルは、`BE-XXXX` が項目自身のディレクトリ
   （`roadmaps/BE-XXXX-<slug>/`）**以外のどこにも**現れてはならないと、はっきり述べます。採番がそのディレク
   トリだけを書き換えるからです。コードでは機能を名前で参照し、ID は採番が直す唯一の場所であるロードマップ
   項目に置きます。ほかの場所に書いた参照は `main` に古いまま残り、CI はそれを捉えません（ロードマップの
   フォーマットチェックは `roadmaps/` の中しか歩かないからです）。
5. **マージは採番を発火させねばならない。** 既定の `GITHUB_TOKEN`（ネイティブ auto-merge）によるマージは
   `push` 起動のワークフローを再発火させないので、`roadmap-id` が走らず、PR はコードごと `BE-XXXX` のまま
   `main` に着地してしまいます。スキルはこれを注記します。PR は `roadmap-id` を発火させる経路でマージせねば
   なりません（手動マージ、あるいはロードマップ App トークンでの auto-merge）。これは素の `ideation` 提案が
   すでに置かれている制約と同じですが、実装が項目とともに着地するぶん、ここではより重く効きます。
6. **ガードレールとスコープ（プライムディレクティブ由来）。** スキルは、*いつ*これを使い、いつ使わないかを
   はっきり述べます。
   - 使うのは、小さくスコープの定まった項目で、著者が設計をレビューで大きく変えられるとは見込んでいない
     場合に限ります。1 つの PR は設計のチェックポイントをコードレビューと一体にするので、マージは提案と
     実装を一度に受け入れます。スキルはこれを許容したトレードオフとして明記し、設計が本当に不確かなときは
     直列の経路に戻すよう述べます。
   - このスキルは執筆*と*実装の両方を行うので、製品コードに一切触れない `ideation` とは違い、`bajutsu/` /
     `BajutsuKit/` / テストを `implement-be` の規則の下で**変更します**。スキルは、フェーズ A が執筆のみに
     とどまり、製品コードはフェーズ B でだけ現れることを明示します。
   - 三つのプライムディレクティブはすべて、`implement-be` の場合とまったく同じように効きます。`run` / CI の
     判定経路に LLM を置かないこと、決定性を最優先すること、アプリ非依存の設定に寄せることです。スキルは
     CI に自動化を足しません。人間・エージェントの手順にすぎないので、どのゲートにも新しい仕組みを持ち込み
     ません。
7. **ドキュメント。** [`docs/ai-development.md`](../../docs/ai-development.md)（と `docs/ja/` の対応版）の
   三つのスキルの三角形、すなわち `ideation`（執筆のみ）、`implement-be`（番号付き項目を出荷）、
   `propose-and-build`（両方を 1 つの PR で）と、コントリビュータ向けチュートリアルを、スキルと歩調を合わせ
   ます。三つの SKILL.md の `References` を相互リンクします。

### 機械的に検証できる成果

このスキルは手順なので、決定的な成果は PR が生み出したものに既存のゲートを当てたものになります。`make check`
が緑であること、ID が採番されたあとロードマップダッシュボードが `Status: Implemented` を反映していること、
そして CI の `pr-title` チェックのとおり、BE 作成ブランチが保つべき素の scoped タイトルであることです
（このチェックはブランチが本物の ID を含む場合にだけ働き、このフローのブランチは決して含みません）。新しい
チェッカーは足しません。スキルの正しさは、すでにゲートされている `implement-be` の終状態と、SKILL.md が符号
化する ID を閉じ込める不変条件に帰着します。

## 検討した代替案

- **何もせず、`ideation` → `implement-be` の厳密な直列分割を保つ。** 安全な既定であり、ほとんどの項目では
  今も正しい選択です。*唯一の*選択肢としてだけ却下します。著者が今すぐ作れる小さな項目に、避けられる待ち
  時間と文脈の取り直しを課すからです（前述の動機のとおり）。
- **一時的な二本の PR のスタック（この項目の元の設計）。提案 PR と、その提案ブランチにスタックした実装 PR、
  そして提案がマージされたあと実装を `main` にリベースして付け替える引き継ぎ。** これは `propose-and-build`
  が最初に出荷された形です（#848）。スタックは、実装を仕上げる前に本物の ID を取り戻し、実装 PR が `[BE-NNNN]`
  タイトルプレフィックスを取れて、固まった仕様に対してリベースできるようにするためにありました。スキル自身の
  スコープ、つまり著者が確信を持つ小さな設計では、この二つの見返りはどちらも成り立ちません。プレフィックスは
  不要であり（BE 作成 PR は素のタイトルを保ちます）、対して固め直すべき仕様もありません。その一方でスタックは、
  二本目のブランチ、提案の変化に追随して draft PR をリベースし続ける手間、ベースの付け替え、間違えやすい引き
  継ぎのリベースを要します。1 つの PR の設計（#1304）が、これらすべてを一つ加えた規則（プレースホルダの ID を
  閉じ込める）と引き換えに取り除き、この代替案に取って代わりました。
- **`BE-XXXX` の規律を持たない巨大な単一 PR。** ブランチ運用は最も単純で、この項目が今採る設計そのものですが、
  それは ID を閉じ込める不変条件が*あってこそ*です。不変条件がなければ、実装に書かれた ID が `main` に古いまま
  残ります。採番が項目自身のディレクトリしか書き換えないからです。この不変条件こそが、単一 PR を単に単純な
  ものではなく安全なものにします。スキルはそれを慣習ではなく厳格な規則として述べます。
- **このフローのためだけに ID を早く採番する（PR オープン時または承認時に）。** PR が最初から `[BE-NNNN]`
  プレフィックスを取れるようになりますが、BE-0089 が閉じたまさにその決定を開き直します（項目が受理される前に
  番号を使う、PR が却下されると番号が連番でなくなる）。リポジトリ全体の不変条件にスキル単位の例外を設けるのは、
  それが買う素のタイトルよりも悪いことです。却下します。
- **ID を閉じ込める不変条件を CI で強制する（プレースホルダ項目自身のディレクトリの外に `BE-XXXX` が現れない
  ことをチェックする）。** 唯一の規則をゲートにできる点で魅力的ですが、リポジトリ全体の `BE-XXXX` 走査は、
  正当なプレースホルダの自己参照、散文中の例、タイトルプレフィックスの例を、本当に古い参照から見分けねば
  なりません。これは、ロードマップのフォーマットチェックが `roadmaps/` の中ですでにかわしているのと同じ
  誤検出の面です。見送ります。スキルはまず不変条件を規律として述べます。範囲を絞ったチェックは、フローが実証
  されてからの後続候補です。

  スキル段階の一覧に三つのスキルの三角形を記述しました。
- [#1257](https://github.com/bajutsu-e2e/bajutsu/pull/1257) — 昇格ステップの一覧から `make roadmap-index`
  を撤去しました。ロードマップダッシュボードが、生成済みの README 索引表が担っていた役割を今では引き継いで
  いるため、そこで再生成するものがなくなったからです。
- [#1304](https://github.com/bajutsu-e2e/bajutsu/pull/1304) — スキルを一時的な二本の PR のスタックから、
  1 つの BE 作成 PR へと作り直しました。項目は `BE-XXXX` のプレースホルダを保ったまま PR のなかで
  `Status: Implemented` に到達し（状態と PR 番号は ID に依存しません）、CI がマージ時に本物の `BE-NNNN` を
  採番して、項目自身のファイルの中でプレースホルダを書き換えます。引き継ぎ、スタックしたブランチ、ベースの
  付け替えはなくなり、一つの不変条件（プレースホルダの ID は項目自身のファイル以外のどこにも現れない）と、
  マージが `roadmap-id` を発火させねばならないという注記に置き換わりました。
