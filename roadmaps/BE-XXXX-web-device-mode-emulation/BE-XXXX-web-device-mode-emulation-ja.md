## はじめに

web（Playwright）バックエンドは、常にデスクトップのブラウザを駆動します。どのコンテキストも素の
`browser.new_context(reduced_motion="reduce")` であり、既定のデスクトップ viewport、タッチ入力なし、
`is_mobile` は false です。したがってテスト対象の web アプリは、チームがテストしたい挙動がモバイルの
それであっても、常にデスクトップのレイアウトとデスクトップの入力しか目にしません。本提案は、ターゲット単位の
**デバイスモード**（viewport、`is_mobile`、`has_touch`、デバイススケール、ユーザーエージェントによる
デスクトップまたはモバイルのエミュレーション）を追加し、web ターゲットをどちらとしても駆動できるようにします。
Playwright に組み込まれたデバイスエミュレーションを使います。

## 動機

レスポンシブな web アプリはスマートフォンで異なる挙動を示します。レイアウトが変わり、上部ナビゲーションの
代わりにハンバーガーメニューが出て、タッチスクロールやタッチ専用のジェスチャーが使われ、モバイルの
ユーザーエージェントになります。Bajutsu は今も web を駆動できますが、そのデスクトップの顔だけです。
「このターゲットを iPhone としてテストする」と指定する方法がありません。そのため、しばしば主となるモバイルの
体験が、他の点では対応しているバックエンドの上で、ツールの手の届かないところに置かれています。

これは別の箇所での正しい修正も妨げます。対になる **web-swipe-scroll-fidelity** 項目は、ブラウザの入力モードで
分岐して `swipe` をスクロールさせます。デスクトップのポインターコンテキストでは wheel スクロール、タッチ
コンテキストではタッチドラッグです。この分岐は `has_touch` で決まりますが、タッチコンテキストを設定する
方法がなければ、デスクトップ経路しか実行されません。本項目がモバイル（タッチ）モードを供給することで、
タッチ経路が実際に動くようになり、両者が合わさって web の `swipe` にデスクトップとモバイルの双方で正しい
挙動を与えます。

Playwright は仕組みを既に備えています。`playwright.devices["iPhone 13"]`（ほかも同様）は、viewport、
`device_scale_factor`、`is_mobile`、`has_touch`、`user_agent` をまとめたプリセット記述子で、`new_context` に
適用します。作業は、これを決定的でアプリ非依存の設定として露出し、バックエンドのコンテキスト生成に通すこと
であり、エミュレーションを発明することではありません。

先に述べておくべきスコープの境界があります。これはデスクトップ級のブラウザ（Chromium、Firefox、WebKit）で
モバイルの viewport とタッチ入力を用いたエミュレーションであり、Chrome DevTools のデバイスツールバーが行う
のと同じものです。実機のモバイルブラウザやデバイスクラウドではありません。それらは
[DESIGN §1](../../DESIGN.md) とロードマップの「クラウドデバイスファーム」の非対象方針のとおり、スコープの
外に留まります。本物のモバイル OS が必要な場合は Android バックエンド（BE-0007）が経路であり、本項目は
あくまで web ターゲットのレンダリングと入力のモードに関するものです。

## 詳細設計

web ターゲットの設定に**デバイスモード**を追加し、Playwright バックエンドがブラウザコンテキストを生成する
ときに適用します。

**設定。** web ターゲットに、任意のデバイス設定を持たせ、既存の設定の階層化（`bajutsu/config.py`）を通して
解決します。自然な形は Playwright 自身のモデルに倣います。

- **名前付きデバイスプリセット**（例：`device: "iPhone 13"`）。`playwright.devices` に対して解決し、
  viewport / `device_scale_factor` / `is_mobile` / `has_touch` / `user_agent` に展開する。あるいは
- 任意の viewport を伴う明示的な**デスクトップ／モバイルのモード**。特定の機種を固定せずに、素の
  デスクトップや汎用的なモバイルのコンテキストが欲しいチームのため。

既定は**デスクトップ**であり、今日から変わりません。既存の web シナリオは、今とまったく同じように動き続けます。
キー名の具体や、プリセットと明示フィールドを一つの設定に同居させるかは、PR で詰める実装の細部です。制約は、
シナリオ単位やステップ単位ではなく、ターゲット単位（アプリ非依存）の設定であることです。

**コンテキスト生成。** `PlaywrightDriver._new_context`（およびスターターの初回コンテキスト）が、解決した
デバイス記述子を、既存の `reduced_motion="reduce"` と並べて `new_context(**kwargs)` の呼び出しにマージ
します。リローンチ（BE-0077）と `reset_context`（クロールのクリーンな起点）は同じ記述子を組み直すため、
モードはブラウザのライフサイクル全体を通して安定します。`reduced_motion` と選択したエンジン（BE-0076）が
既に守っているのと同じ不変条件です。

**入力との関係。** コンテキストが `has_touch` を true にして生成されれば、web-swipe-scroll-fidelity の
振り分けが自動的に `swipe` を CDP（Chrome DevTools Protocol）タッチドラッグへ通します。そこにさらなる変更は
要りません。本項目の責務は
正しく設定されたコンテキストを生み出すところまでであり、その中でジェスチャーをどう実現するかは対になる
項目のものです。

**ケイパビリティとプリフライト。** バックエンドのケイパビリティ集合とプリフライト（BE-0082）を見直し、
モバイルコンテキストが実際にできることを表明するようにします。`run`／CI の判定経路は変わりません。

作業分解（MECE）：

1. **設定の露出**：web ターゲット設定（`bajutsu/config.py`）にデバイスモードの設定を追加し、デスクトップを
   既定とし、検証する（不明なプリセットは明確なエラー）。
2. **記述子の解決**：プリセット名を `playwright.devices` に対して解決する（または明示的なデスクトップ／
   モバイルと viewport から記述子を組む）。設定の import が Playwright を import しないよう遅延で行う。
3. **コンテキストへの配線**：記述子を `_new_context` とスターターの初回コンテキストに適用し、
   `reset_context` / `relaunch` をまたいで保つ。
4. **ケイパビリティとプリフライトの見直し**：表明するケイパビリティとプリフライトがモバイルコンテキストで
   正しいことを確認する。
5. **テスト**：プリセットの解決、デスクトップ既定（挙動不変）、記述子を載せたコンテキストの kwargs、
   reset／relaunch をまたいだ安定性を確認する。
6. **ドキュメント**：`docs/configuration.md` / `docs/drivers.md`（と日本語ミラー）にデバイスモードを記す。
   モバイルモードが実機ではなくデスクトップブラウザのエミュレーションである点も含める。

以下のとおり、いずれもプライムディレクティブの範囲内です。

- **決定性。** デバイス記述子はコンテキスト生成時に適用される固定的なデータであり、そのどこもモデルには
  問い合わせません。`run`／CI ゲートには手を触れません。エミュレーションは、今日の固定デスクトップ
  コンテキストと同じく再現可能です。viewport とスケールを固定の数値に定めるため、レンダリングされる
  レイアウトも実行のたびに再現でき、ウィンドウサイズが変わりうる実機よりむしろ安定します。
- **アプリ非依存。** モードはターゲット単位の設定であり、設定が既にターゲットごとの差分をモデル化している
  場所（`targets.<name>`）そのものです。ドライバーとランナーの形は変わりません。

## 検討した代替案

- **シナリオ単位・ステップ単位の viewport。** シナリオが実行の途中でデバイスモードを切り替えられると、
  同じシナリオがステップごとに違う意味を持ち、アプリ非依存の境界が曖昧になります（デバイスは、個々の
  アクションではなく、ターゲットをどう駆動するかの性質です）。却下します。デバイスモードはターゲット単位の
  設定であり、設定が既にターゲットごとの差分をモデル化しているやり方に合致します。
- **専用の `setViewport` / `emulate` ステップ。** 環境の性質を命令的なアクションに変え、シナリオ途中の
  非決定的なリサイズを招きます。同じ理由で却下します。両方の顔が必要なチームは、シナリオを二つのターゲットの
  もとで実行します。
- **名前付きプリセットだけを露出する（明示的なデスクトップ／モバイルと viewport を出さない）。** プリセットは
  よくあるスマートフォンは網羅しますが、チーム固有のブレークポイントは網羅しません。明示的な viewport も
  対応させれば、よくあるケースは一語のままで、独自サイズを締め出しません。両方を残します。
- **本項目を web-swipe-scroll-fidelity に一つの項目としてまとめる。** 検討しましたが、両者は分けられます。
  デスクトップ経路で `swipe` を正しくスクロールさせることはそれ単独で価値があり、エミュレーションなしで
  出荷できます。一方でエミュレーションは、独自の設定とテストを伴う、より広いケイパビリティ（レイアウト、
  入力、ユーザーエージェント）です。それぞれが小さく独立してレビューできるよう分割します。
