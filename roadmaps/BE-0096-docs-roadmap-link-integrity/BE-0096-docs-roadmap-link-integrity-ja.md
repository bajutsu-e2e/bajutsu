## はじめに

ロードマップ項目が出荷されると、`make roadmap-promote` がそのディレクトリを状態フォルダ間で移動し
（`proposals/` → `in-progress/` → `implemented/`、[BE-0078](../BE-0078-roadmap-status-folders/BE-0078-roadmap-status-folders-ja.md)）、
**`roadmaps/` の中の**その項目へのクロスリンクを修復します。しかし `docs/`（や `README*` / `CLAUDE.md`）から
その項目へのリンクには手を触れず、ゲートもそれらのリンクを検証しません。そのため昇格のたびに、移動した項目を
指す `docs/` のリンクが黙って腐ります。本項目はこの穴を塞ぎます。`docs/` → ロードマップのリンク切れをゲートの
失敗にし、さらに promote のリンク修復を `docs/` まで広げて、よくあるケースを自動で直します。

## 動機

このリポジトリはすでに、ロードマップのクロスリンクを自動で正直に保つ仕組みを持っています。`roadmap-promote`
は項目が移動したとき **`roadmaps/` の中の**すべてのリンクのフォルダ部分を書き換え、コミット済みの索引がずれ
ていれば `make test` が失敗します（[BE-0069](../BE-0069-executable-contributor-guardrails/BE-0069-executable-contributor-guardrails-ja.md)、
[BE-0078](../BE-0078-roadmap-status-folders/BE-0078-roadmap-status-folders-ja.md)）。しかしその
修復は `roadmaps/` の境界で止まり、ゲートの `test_roadmap_format` も*ロードマップ項目どうし*のリンクしか検査
せず、`docs/`（やトップレベルの `README*` / `CLAUDE.md`）から項目へのリンクは決して見ません。

結果として、リンクは黙って腐ります。`docs/drivers.md → roadmaps/in-progress/BE-0041-…` とリンクするドキュメント
は、BE-0041 が `implemented/` へ昇格したあともそのパスをそのまま保つので、リンクは 404 になり、`make check` の
何もそれに気付きません。これは仮の話ではありません。一度のスイープで、すでに出荷してフォルダが移動した
**BE-0041、BE-0010、BE-0050、BE-0063、BE-0065、BE-0068、BE-0051** への `docs/` のリンク切れが見つかり、手で
修復するはめになりました。項目が出荷されるほど、腐りは増えるだけです。対策は、`roadmaps/` 側ですでに信頼している
のと同じ種類、すなわち Linux ゲートで動く、決定的で AI 不使用のガードであるべきです。

## 詳細設計

2 つの相補的な部品、すなわちゲートの**検出器**と promote の**自動修復**です。どちらも純粋・決定的で Linux
完結です（Simulator なし、LLM なし、ネットワークなし）。

### 1. docs → ロードマップのリンクがすべて解決することを確かめるゲート検査

テストを足します（`make test` が実行し、`make check` の一部になります）。`docs/` 配下のすべての Markdown と、
トップレベルの `README.md` / `README.ja.md` / `CLAUDE.md` を走査し、`roadmaps/` を指す相対リンクを取り出し、
リンク元ファイルからの相対で解決し、対象ファイルが存在することをアサートします。対象が無ければ、問題の
`ファイル:行 → 対象` を示してゲートを失敗させます。これは**両方**の失敗様式、すなわち対象を移動させた昇格と、
手書きの打ち間違いや古いパスを捕まえ、しかも後日の手作業のスイープではなく PR の時点で捕まえます。検査は
ファイルツリーの純粋関数なので、デバイスもモデルも要りません。`test_roadmap_format` がすでに `roadmaps/`
の中のリンクを守っているのを、1 つ外のディレクトリへ広げたものです。

検査の範囲は `roadmaps/` を指す相対リンクだけ（昇格で腐る面）です。汎用の外部 / HTTP リンク検査ではありません
（それは重く、ネットワークに縛られます。代替案を参照）。

### 2. `roadmap-promote` のリンク修復を `docs/` まで広げる

`roadmap-promote` が項目のディレクトリを移動するとき、移動したフォルダ部分
（`<旧状態>/BE-NNNN-<slug>` → `<新状態>/BE-NNNN-<slug>`）を `docs/` とトップレベルの `README*` / `CLAUDE.md`
のリンクでも書き換えます。`roadmaps/` の中で今やっているのとまったく同じです。これにより、よくあるケース、
すなわち項目が出荷されてその `docs/` リンクが追従すべき場合を、同じ変更の中の自動でレビュー可能な差分にします。
そのため検査（部品 1）が昇格経路で失敗することはほとんどなくなります。書き換えは項目の slug を鍵にした
フォルダ部分の単純な置換で、既存の `roadmaps/` 内修復と同一なので、同じ低リスクです。

### なぜ両方か

検出器は契約です（不変条件を強制可能にし、自動修復が知りえない手書きのミスも捕まえます）。自動修復は、検出器が
日常の昇格経路で発火しないようにする利便です。どちらか一方では弱くなります。修復のない検出器は、昇格のたびに
手作業の修正を残します。検出器のない修復は、打ち間違いや昇格以外のずれを通してしまいます。検出器を先に出すのが
価値の高い半分で、単独でも役立ちます。

### 決定性とスコープ

これは contributor のワークフロー用のツールです。docs だけで、実行時の挙動は変わらず、どこにも LLM はなく、
既存の Linux `make check` ゲートの中で完結します。ドライバにもランナーにも決定性の保証にも触れません。

## 検討した代替案

- **ゲート検査だけで自動修復なし。** より単純で、不変条件を強制可能にしますが、昇格のたびに手作業のリンク
  修正を残します。まさに本項目が無くそうとしている手間です。*先に*出す価値はありますが（それが契約です）、
  答えのすべてではありません。
- **自動修復だけでゲート検査なし。** 日常の昇格ケースは直しますが、手書きの打ち間違いや、修復が想定しなかった
  パスを黙って通します。検出器がなければ保証はなく、ベストエフォートだけです。
- **汎用の外部リンク検査（HTTP ＋ リポジトリ内）。** より多くを捕まえます（外部 URL のリンク切れ、アンカー）が、
  重く、ネットワークに依存し、レート制限でフレーキーになります。速くて密閉されたゲートには不向きです。本項目が
  狙う腐りは内部の構造的なもので、狭い内部リゾルバが決定的でオフラインの適切な解です。
- **手作業のまま（現状維持）。** 不採用です。本項目のきっかけになったスイープは、すでに 7 項目のリンクを手で
  修復する必要があり、コストは昇格のたびに増えます。
