## はじめに

ロードマップ項目は「何を」「なぜ」作るかを記録しますが、その作業が「どこまで進んだか」は記録しません。
項目の状態、つまりどのスライスが出荷済みで、何が残り、何がいつ変わったかは、ファイルの外にあります。
`実装 PR` 行や、コミット履歴や、貢献者の頭のなかです。実際に二つの項目
（[BE-0087](../BE-0087-idb-action-settle/BE-0087-idb-action-settle-ja.md)、
[BE-0052](../BE-0052-device-state-timezone-clipboard-shake/BE-0052-device-state-timezone-clipboard-shake-ja.md)）
は、この空白を埋めるために即席の `### Implementation status` サブセクションを育てましたが、その形は
それぞれ独自のもので、最新の状態に保つ仕組みもありませんでした。

この項目は、正規のテンプレート
（[BE-0074](../BE-0074-be-template-standardization/BE-0074-be-template-standardization-ja.md)）の拡張を提案します。
新しいセクション `## Progress`（`## 進捗`）を一つと、任意のメタデータ欄 `関連`・`無効化` を二つ加えます。
進捗セクションは生きたセクションです。作業分解に対応する MECE なチェックリストと、短い時系列ログを置き、
開発の進行に合わせて最新の状態に保ちます。メタデータ欄は項目間のリンクを記録します。この項目が土台にする
項目や関連する項目と、この項目を無効化する項目です。あわせて、これまで暗黙に守られてきた規範も明文化します。
`詳細設計` では作業を MECE に列挙する、という規範です。変更の全体はドキュメントとスカフォルダーと
フォーマット検査だけで、ツールの挙動は変えず、どの経路にも LLM を持ち込みません。

## 動機

ロードマップはプロジェクトの共有の計画面ですが、「これは完了したか」に答えられるのは、`状態` が表す両端だけ
です。`提案`（何もしていない）と `実装済み`（すべて済んだ）の二つです。進行中の項目には、どの部分が
出荷されたかを構造的に書く場所がありません。その影響は具体的です。

- **進捗が見えないか、独自形式です。** 読み手は、`実装中` の項目がどこまで実体を持つかを、散文を最後まで
  読み PR を突き合わせない限り判断できません。進捗を記録した二つの項目は独自の
  `### Implementation status` サブセクションを発明しましたが、検査されず一貫性もないため、次の著者が
  まねる手本になりません。
- **作業が完全な集合として並んでいないことがあります。** 詳細設計のなかには、作業の単位を列挙せず方針を
  素描するだけのものがあり、進捗を照らし合わせる基準がありません。そもそも描かれていないチェックボックスは
  埋められません。
- **項目間の関係が散文にしかありません。** 後続の項目が先行の項目を置き換える、または関連するとき、その
  リンクは（あったとしても）片側の `参考` の箇条書きに置かれるだけで、相互でも grep 可能でもありません。

[BE-0074](../BE-0074-be-template-standardization/BE-0074-be-template-standardization-ja.md) は、
「テンプレートが揺らぐ」という問題へのプロジェクトの答えをすでに確立していました。形を固定して検査する、という
答えです。あの項目は五つのセクションとメタデータ欄の集合を固定し、進捗は意図的に対象外としました。本項目は
同じ仕組みの上に、その欠けていた部分を加えます。これにより「どこまで進んだか」が、すべての項目の一級の、
検査される要素になります。

## 詳細設計

実装は四つの互いに重ならない部分です。テンプレートの面、ツール、散文、そして既存項目への遡及追記です。
**遡及追記は本項目の実装の一部であり、実装 PR で行います。この提案では行いません。**

### 1. テンプレート：新しいセクション一つと新しい欄二つ

六つ目の H2 セクション `## 進捗`（英語側は `## Progress`）を、`## 検討した代替案` と `## 参考` のあいだに
入れます。設計の論拠の後、参照リンクの前です。中身は順に、`詳細設計` の MECE な作業分解に対応するチェック
リスト（作業の単位ごとに `- [ ]` を一つ置き、着地したものから `- [x]` にする）と、何がいつ変わったか
（古い順）を PR へのリンクとともに記録する短い時系列ログです。雛形は次のとおりです。

```markdown
## 進捗

> 開発の進行に合わせて常に最新の状態に保ってください。チェックリストは *詳細設計* の MECE な
> 作業分解（作業の単位ごとに 1 つ）に対応し、ログには変更内容と時期（古い順）を PR へのリンクと
> ともに記録します。

- [ ] TBD — スコープが固まり次第、作業分解（MECE）をここに列挙します。
```

任意のメタデータ欄を二つ、`トピック` の後・`由来` の前の正規の順序に加えます。`関連`（英語側は `Related`）は
この項目が土台にする項目や関連する項目を、`無効化`（英語側は `Superseded by`）はこの項目を無効化する項目
（その後継）を指します。どちらもリンクで、どちらも任意なので、既存項目に欄を足す必要はありません。リンクは
相互に張ります。置き換える側は `関連` で相手を指し、置き換えられる側は `無効化` で後継を指します。

### 2. 規範：MECE な作業分解と、生きた進捗セクション

二つの規範を `CLAUDE.md`、`roadmaps/README.md`（と `-ja`）、`docs/ai-development.md` に書きます。

- **`詳細設計` では作業を MECE に列挙します。** 漏れなく重複なく列挙することで、チェックリストが対応する
  完全で重ならない集合を持てます。
- **`進捗` は開発の進行に合わせて最新の状態に保ちます。** 項目を進める PR は、`実装 PR` 行を埋めるのと
  まったく同じように、同じ変更のなかでボックスにチェックを入れログを足します。未着手の `提案` は
  プレースホルダーのボックスを一つだけ持ち、`実装済み` の項目は `実装 PR` を指す全完了のチェックリストを
  持ちます。

これらはレビューで守る規範であり、機械で強制するものではありません。検査はセクションの存在は確認できますが
（後述）、散文の分解が本当に網羅的か、ボックスが正直かまでは判断できません。

### 3. ツール

- **スカフォルダー**（[`scripts/new_roadmap_item.py`](../../scripts/new_roadmap_item.py)）は、素の
  `TBD` ではなく上の雛形で初期化した `進捗` セクションを出力します。
- **フォーマット検査**（[`tests/test_roadmap_format.py`](../../tests/test_roadmap_format.py)）は、
  必須で順序のある H2 見出しに `Progress` / `進捗` を加え、既知の任意メタデータ欄に `Related` / `Superseded by`
  （と日本語のラベル）を加えます。インデックス生成器はメタデータを欄名で読むため、新しい任意欄の影響を
  受けません。

このときフォーマット検査は**すべての**項目に新セクションを要求するので、検査の変更と下記の遡及追記は実装 PR で
一緒に着地させる必要があります。片方だけではゲートが緑になりません。

### 4. 遡及追記（実装 PR）

既存のすべての項目に `## 進捗` セクションを加えます。履歴を捏造せず、正直に書きます。

- **実装済み**の項目は、`実装 PR` を指す全完了の一行チェックリストを持ちます。日ごとの経過は捏造しません。
- **実装中**の項目は、実際の状態を反映した本物のチェックリストを持ちます（二つの
  `### Implementation status` サブセクションは移行します）。ボックスは、スライスが確かに出荷されたところ
  だけチェックします。
- **提案・保留**の項目は、着手時に列挙するためのプレースホルダーのボックスを一つ持ちます。

### 基本原則との整合

変更はドキュメントと、決定的なスカフォルダーおよび検査です。どの経路にも LLM は入らず、`run` / CI ゲートには
触れず、特定のアプリやバックエンドに依存するものもありません。

## 検討した代替案

- **`詳細設計` 配下の H3 `### Implementation status`**（既存の自然発生的な慣行）。標準形としては棄却しました。
  H3 はフォーマット検査から見えないため、「すべての項目が持ち、最新に保つ」を固定できず、進捗は設計に埋もれる
  より独立したセクションのほうが自然に読めるからです。
- **`Work plan` と `Progress log` の二セクション。** 棄却しました。チェックリストがそのまま作業計画なので、
  チェックリストとログを一つの `進捗` セクションに置けば、分解を重複させずに済み、遡及追記で増える見出しも
  二つではなく一つになります。
- **セクションではなくメタデータ欄としての `進捗`**（割合や状態語）。棄却しました。スカラーでは単位ごとの
  チェックリストもログも持てず、チェックリストが具体的にする見かけの精度を招きます。
- **「最新か」を機械で検査する規則。** 実現不能として棄却しました。散文の分解が網羅的か、ボックスが正直に
  チェックされているかを判定できる決定的な検査はありません。検査はセクションの存在を確認し、最新かどうかは
  docstring の品質と同じくレビューの規範とします。
- **出荷済み項目の完全な時系列の復元。** 正直さの観点から棄却しました。実装済み項目に遡及的な日ごとの履歴を
  でっち上げると記録を捏造することになります。代わりに `実装 PR` を指す正直な一行を持たせます。
