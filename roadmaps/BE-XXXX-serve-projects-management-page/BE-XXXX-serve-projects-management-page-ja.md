## はじめに

設定プロジェクトハブ（[BE-0225](../BE-0225-config-project-hub/BE-0225-config-project-hub-ja.md)）は、
`serve` を複数の名前付き設定バインディングを束ねるハブにし、それをヘッダーのスイッチャーと、
プロジェクトを一覧して切り替え・実行する **Projects** モーダルとして提示しました。モーダルが担うのは
その読み取りと切り替えと実行の範囲だけで、ハブを増やしたり減らしたりする 2 つの操作、すなわちプロジェクトの
追加と削除は `bajutsu project` CLI に委ねられています。モーダル自身のヒントもそう述べています
（「Add or remove projects with `bajutsu project add` / `rm`.」）。

本項目は、このギャップをモーダルへの操作追加ではなく、**プロジェクト管理を serve シェルのトップレベルの
ページへ昇格させること**で埋めます。Record / Replay / Crawl / Stats / Metrics と並ぶ **Projects** ビューを設け、
そこでプロジェクトを一覧し、追加・削除・切り替え・実行を行います。モーダルはこのページに引き取られて廃止します。
理由は情報構造にあります。**プロジェクトは設定の上位にある概念**であって、設定に属する機能ではありません。
だからこそ、一時的なオーバーレイの中ではなく、シェルのトップレベルに置かれるべきです。

## 動機

モーダルは、シェルの情報構造のなかでは機能に紐づくオーバーレイです。現在の文脈に対する何らかの操作のために
現れ、その文脈へ戻って閉じます（設定の閲覧、Settings、実行結果の Replay）。プロジェクトハブはこの形に
当てはまりません。プロジェクトはアクティブな設定に対して行う操作ではなく、**アクティブな設定が内側に存在する
入れ物**です。トップレベルの各ビュー（Record、Replay、Crawl、Stats）はすでに選択中のプロジェクトの内側で
動いており、それらを所有して束ねるものが、階層を反転させずにそれらから起動されるモーダルであることはできません。
BE-0225 のモーダルは手早い一覧と切り替えには使えますが、ハブの置き場所としては構造的に誤っています。
上位の概念を下位のオーバーレイとして描いているからです。

この置き違えの帰結は具体的です。

- **ライフサイクル操作の居場所がない。** プロジェクトの追加と削除はハブ規模のライフサイクル操作であり、
  閉じられるオーバーレイはその居場所として不自然です。BE-0225 がこれらを CLI に残した一因でもあります。
  ブラウザからハブを管理する利用者には、ハブに最低限必要な「増やす・減らす」を行う場所が UI にありません。
  とりわけホスティング構成（[BE-0015](../BE-0015-web-ui-public-hosting/BE-0015-web-ui-public-hosting-ja.md)）
  ではブラウザが利用者に与えられた唯一の画面であり、`bajutsu project add` を実行するシェルがありません。
- **ハブに恒久的な自分の画面がない。** 兄弟であるプロジェクト横断のメトリクスビュー
  （[BE-0226](../BE-0226-cross-project-metrics-dashboard/BE-0226-cross-project-metrics-dashboard-ja.md)）は、
  すでにハブのプロジェクトを扱うトップレベルのビューです。そのビューがランク付けするプロジェクト自体には
  トップレベルの画面がなく、オーバーレイしかないのは不自然です。ハブについて考える起点となる一覧は、
  クリックで消える一時的なものではなく、リンクでき、そこへ移動できて残り続ける場所であるべきです。

このページに必要なものは、すべてすでに存在します。エンドポイント（BE-0225 ユニット 3。`GET` / `POST` /
`DELETE /api/projects`、`POST /api/projects/<name>/activate`、`POST /api/projects/<name>/run`）、ランチャーが
使う設定ソースの入力ウィジェット（任意の認証情報を伴う Git リポジトリ欄と `.zip` アップロード、
[BE-0063](../BE-0063-git-config-source/BE-0063-git-config-source-ja.md) /
[BE-0073](../BE-0073-serve-zip-bundle-upload/BE-0073-serve-zip-bundle-upload-ja.md)）、許可リスト検査
（[BE-0108](../BE-0108-hosted-config-source-restriction/BE-0108-hosted-config-source-restriction-ja.md)）、
そして RBAC のゲートです。本項目はこれらをページへ移し、ハブに欠けていた 2 つのライフサイクル操作を加えます。

これはプライムディレクティブの範囲内に収まります。プロジェクトページは完全に決定的で、CLI が書き込むのと
同じ `ProjectRegistry` の継ぎ目を通じて名前付きの設定ソースを一覧・バインド・アンバインドするだけであり、
どの経路にも LLM は入りません。ハブそのものはアプリ非依存のままです。アプリごとの違いは各プロジェクトの設定に
置かれ、このページは設定ソースに名前を付けてバインドするだけだからです。

## 詳細設計

作業は 4 つのユニットに MECE に分かれます。ページそのもの、そのページが備える 2 つのライフサイクル操作、
そしてそれらを統べるロールと許可リストの結線です。

### 1. Projects トップレベルビュー

serve シェルの新しいトップレベルビュー（ヘッダーナビの `data-view="projects"` タブ。他のビューと同様に
独自のメイン領域へ描画する）がハブの居場所になります。ここでは登録済みの各プロジェクトを、ハブについて考えるのに
必要な情報とともに一覧します。名前、設定ソース（Git・アップロード・ファイルのバインディングを人間が読める形に
要約したもの）、アクティブかどうか、そして直近の実行結果です。BE-0225 のヘッダー**スイッチャー**は
残します（文脈セレクターのような、正当な手早い切り替えの操作だからです）。一方で **Projects モーダルは廃止**し、
その一覧、行ごとの Run、そして担っていた切り替えは、このページに移します。プロジェクトの選択・アクティブ化は
既存の `POST /api/projects/<name>/activate` を通じてライブ設定を再バインドし、モーダルと同じ挙動になります。
これにより他のすべてのビューが、再起動なしに切り替え先のプロジェクトに対して動きます。

Metrics ビュー（BE-0226）やヘッダーのスイッチャーと同様に、Projects タブはハブを見せる価値があるときに
のみ現れます。したがって単一設定の `serve` は見た目が変わりません。ただしユニット 4 で述べる 1 点の意図的な
例外（単一設定の `serve` をハブへ育てられるようにすること）を除きます。

### 2. プロジェクトの追加（ページ上で）

ページは**プロジェクト追加**フォームを備えます。名前欄と、新しいウィジェットを作らずにランチャーの既存の
ウィジェットを再利用する設定ソースの選択欄です。すなわち Git リポジトリ欄（spec と任意の認証情報、
[BE-0063](../BE-0063-git-config-source/BE-0063-git-config-source-ja.md)）と `.zip` アップロード
（[BE-0073](../BE-0073-serve-zip-bundle-upload/BE-0073-serve-zip-bundle-upload-ja.md)）です。ローカルの
ファイルシステムパスは、サーバーが許可する場合に限り提示します（ユニット 4）。送信すると `/api/projects` へ
`POST` し、一覧を更新します。`POST /api/projects` は `(org, name)` について冪等なので（BE-0225 ユニット 1・3）、
既存の名前で再送するとエラーにならず、そのソースを**再バインド**します。フォームはこれを明示的な「再バインド」
として示し、サーバー側の拒否（別 id 下での名前衝突、許可リストによる拒否、不正な Git spec）は、ランチャーの
エラー表示を再利用してインラインで示します。

### 3. プロジェクトの削除（ページ上で）

各行に**削除**操作を加えます。実行すると確認を求め（登録解除は履歴には破壊的でないものの、バインディングには
破壊的です）、`/api/projects/<name>` を `DELETE` して一覧を更新します。ページは BE-0225 の契約を明示します。
すなわち**実行履歴は保持され**、バインディングだけが取り除かれます。アクティブなプロジェクトを削除した場合は、
その状況に対してエンドポイントがすでに行う挙動に従います。ページは削除後の `GET /api/projects` の状態を反映し、
結果を推測しません。

### 4. 表示条件、RBAC、許可リスト

ページとその操作は、ハブ UI の他の部分がすでに従っている規則に従います。

- **表示条件。** Projects タブは、スイッチャーや Metrics タブが使うのと同じ「ハブが存在する」という信号に
  従います。実装時に解決する唯一の未決点は、単一設定の `serve` でもタブ（少なくともその**追加**フォーム）へ
  到達できるようにするかどうかです。ハブを最初に現す 2 つ目のプロジェクトを作るために CLI へ落ちる必要をなくし、
  UI からハブへ育てられるようにするためです。ここでは唯一の意図的な UI 上の判断として記録します。
- **RBAC。** `POST` / `DELETE /api/projects` はサーバー側で **admin** ゲートです（BE-0225 ユニット 3、
  `/api/config` と同じ扱い）。ページもこれをそのまま反映します。追加フォームと削除操作は非 admin のセッションでは
  隠すか無効にし、そのセッションには読み取り専用のプロジェクトページ（一覧・切り替え・実行）を見せ、403 になる
  操作を決して見せません。ここでは UI が他の admin 専用画面ですでに利用しているロール情報を再利用します。
- **ホスト時の許可リスト。** ソース選択欄は、サーバーが許可しない場合にファイルシステムの選択肢を隠します
  （[BE-0108](../BE-0108-hosted-config-source-restriction/BE-0108-hosted-config-source-restriction-ja.md)）。
  サーバー側の検査と揃え、ホストの利用者に API が拒否するソースを提示しないようにします。

新しいエンドポイントも、スキーマ変更も、CLI 変更もありません。これは UI の移設（モーダルからページへ）と、
2 つのライフサイクル操作、そしてそれらが必要とするわずかなクライアント状態だけです。

## 検討した代替案

- **既存の Projects モーダルに追加・削除を足す（モーダルのまま）。** 情報構造の観点から却下します。モーダルは
  機能に紐づくオーバーレイですが、プロジェクトは設定の上位にある概念であって、設定に対する操作ではありません。
  各ビューはすでに選択中のプロジェクトの内側で動くので、それらを所有して束ねる画面は、内側から起動される
  オーバーレイではなく、トップレベルのページであるべきです。これは本提案の初期の枠組みであり、ページがこれを
  置き換えます。
- **追加・削除を CLI 専用のままにする（現状維持）。** 却下します。これは本項目が解消しようとしているギャップ
  そのものであり、シェルのないホスト版ハブを自身の UI から育てられないままにします。
- **ランチャーとは別の、追加専用の設定ソース選択欄を作る。** 却下します。ランチャーにはすでに、レビュー済みで
  許可リストを意識したソース UI（Git と認証情報、`.zip` アップロード）があります。これを再利用すれば、ソース
  入力の概念と検査経路を 1 つに保て、
  [BE-0108](../BE-0108-hosted-config-source-restriction/BE-0108-hosted-config-source-restriction-ja.md) と
  歩調を合わせ続けるべき 2 つ目のウィジェットを抱えずに済みます。
- **プロジェクトページを Metrics ビュー（BE-0226）へ畳み込む。** 却下します。両者は関心が異なります。Metrics は
  プロジェクトをランク付けする（読み取り専用で助言的）ものであり、プロジェクトページはそのライフサイクルを
  管理する（登録・切り替え・実行・削除）ものです。同じプロジェクト一覧を扱う `Related` の兄弟であって、
  1 つのビューではありません。

## 進捗

> 開発の進行に合わせて常に最新の状態に保ってください。チェックリストは *詳細設計* の MECE な
> 作業分解（作業の単位ごとに 1 つ）に対応し、ログには変更内容と時期（古い順）を PR へのリンクと
> ともに記録します。

- [ ] 1 — Projects トップレベルビュー。`data-view="projects"` タブでプロジェクト（名前・ソース・アクティブ・直近結果）を一覧し、切り替え・実行する。BE-0225 のモーダルをここへ引き取って廃止し、ヘッダーのスイッチャーは残す。
- [ ] 2 — ページ上でのプロジェクト追加。ランチャーの設定ソースウィジェットを再利用したフォーム。`POST /api/projects`（再バインド対応、インラインエラー）。
- [ ] 3 — ページ上でのプロジェクト削除。行ごとの操作、確認 + `DELETE /api/projects/<name>`、履歴は保持。
- [ ] 4 — 表示条件（ハブゲート、および単一設定を「ハブへ育てる」判断）、RBAC、許可リストの反映。非 admin とホストのセッションには許可された操作だけを見せる。
