## はじめに

オンデバイスのエンドツーエンド（E2E）ジョブ `xcuitest (multi-touch)` は、シナリオの最初の tap で
不安定に失敗します。対象の要素は画面に存在しているのに、`element vanished (stale handle)` という
エラーで落ちます。この失敗は、シナリオやテスト対象アプリの欠陥ではありません。原因は、XCUITest の
しくみが要素をどう指定するかにあります。

XCUITest のしくみは、要素を探すことと実際に tap などの操作をすることを、オンデバイスの runner との
あいだで2回のやり取りに分けています。1回目のやり取りで要素を探し、その時点の画面の状態に結びついた
参照を受け取ります。2回目のやり取りは、その参照を使って操作をします。この2回のやり取りのあいだに、
画面の要素の情報は取り直される場合があります。新しく前面に出た画面が落ち着くまでのあいだ、ふつうに
起こる現象です。取り直しが起こると、1回目のやり取りで受け取った参照は、2回目のやり取りが使うころ
には古くなっています。

本項目は、同じ状況なら必ず同じ結果になるという判定のしかたを弱めずに、この2回のやり取りのあいだの
すき間を塞ぐことを提案します。具体的には、参照が古いという返事を受けたら runner に問い合わせ直し
ます。同じ条件からなお1つの要素だけが見つかる場合に限って、操作をやり直します。要素が見つからなく
なった瞬間には、今までと同じように大きく失敗させます。変更は、xcuitest のしくみのうち Python 側の
部分だけで完結します。触れる範囲は、runner とのやり取り、やり直しの組み込み方、判定のしかたの3つに
限られます。

## 動機

この不安定さは、必須の `E2E (iOS)` という関門で再現します。実行 29630685037 では、`xcuitest
(multi-touch)` ジョブの2本目のシナリオが、最初のステップで失敗しました。シナリオは
[`permission.yaml`](../../demos/showcase/scenarios/permission.yaml) です。

```
step 0 (tap): element vanished (stale handle): {'label': 'Permissions', 'traits': ['button']}
FAIL  runs/20260718-044659/manifest.json
```

このステップが tap する「Permissions」タブバーボタンは、アプリの起動画面の固定された要素です。
そのため、ボタンは消えません。古くなったのは参照であって、要素ではありません。原因は、要素の
指定のしかたが2段階に分かれていることにあります。

具体的には、XCUITest のしくみは、1回の tap を、runner が持つ要素の情報に対する2回のやり取りに
分けます。まず `_resolve_handle` が `GET /elements` を送ります。Python 側で条件から要素を1つに
絞り込み、runner がその要素に対して発行した、その時点の画面の状態に結びついた参照を受け取ります。
続いて `tap` が `POST /tap {handle}` を送ります。runner は参照のひとつひとつを、特定の画面の状態に
結びつけています。そのため、問い合わせと tap のあいだで画面の要素の情報が取り直されると、先の
参照はもう生きた要素に対応しません。runner は `status: "stale"` を返し、ドライバは要素消失の
エラーを送出します。新しく前面に出た画面は、落ち着くまでのあいだ何度も要素の情報を取り直します。
シナリオの最初の tap が、このタイミングのずれの起こる場所になるのはそのためです。

実際、失敗したステップの直前のログも、同じ結論を裏づけます。原因は runner の欠陥ではなく、起動
直後の安定化にあります。失敗したステップの前のおよそ60秒間、ドライバはあるログを記録し続けました。
ログは `runner channel GET /health failed ... [Errno 61] Connection refused` です。`await_ready`
が runner への接続をくり返し確かめていたことを示すログで、待ち時間を固定するのではなく、条件が
整うまで待つやり方によるものです。tap は、runner が応答してから初めて発火しました。つまり runner
は、`stale` という返事をした時点で、すでに生きていて応答していました。runner は自身が操作をする
前に `stale` を返すので、tap はどの要素にも届いていません。画面上で実際に消えたものは、何もあり
ませんでした。

この不安定さは、3つの意味で負担になります。1つ目は、関門への信頼です。必須の集約チェック
`E2E (iOS)` がランダムに失敗すると、メンテナは本物の不具合と見分けられません。見分けられないため
に、反射的な再実行か、無駄な調査のどちらかを招きます。2つ目は、実行の負担です。再実行のたびに、
macOS を動かす費用がかかります。最初の tap に到達するまでに、アプリをビルドし、Simulator を
起動し、runner を立ち上げる必要があるからです。3つ目は、調査を誤らせる効果です。もっとも厄介な
点でもあります。`element vanished` というメッセージは、「Permissions」要素が存在しないかのように
読めます。しかし実際に起きたのは、ずっと存在しているボタンの上でのタイミングのずれです。調査者
は、存在しない消失を追わされてしまいます。

既存のどの仕組みも、このタイミングのずれを扱っていません。もともと扱う設計でもありませんでした。
[BE-0207](../BE-0207-xcuitest-channel-transient-retry/BE-0207-xcuitest-channel-transient-retry-ja.md)
の一時的なやり直しの仕組みは、1回のやり取りの中で起こる通信のブレを吸収します。しかし `stale`
という返事は、通信の失敗ではなく、結果が出たうえでの返事です。この仕組みは、この結果をあえて
やり直しません。結果そのものをやり直すことは、
[BE-0049](../BE-0049-determinism-flakiness-audit/BE-0049-determinism-flakiness-audit-ja.md) が
退ける「吸収による不安定さ」だからです。
[BE-0218](../BE-0218-e2e-simulator-flaky-readiness-actuation/BE-0218-e2e-simulator-flaky-readiness-actuation-ja.md)
の準備完了と時間切れの作業は、runner が誤った画面で準備完了と判定されるのを止め、遅い書き込みに
長い猶予を与えました。ただし、どちらの変更も、「探してから操作する」構成が生むやり取りのあいだの
すき間には触れていません。
[BE-0287](../BE-0287-xcuitest-runner-multitouch-resilience/BE-0287-xcuitest-runner-multitouch-resilience-ja.md)
の姉妹の不安定さは、別の故障のなかまです。BE-0287 では runner が止まって応答しなくなりますが、
本項目では runner は生きていて、`stale` をただしく返します。このタイミングのずれを塞ぐには、
専用の仕組みが必要です。

## 詳細設計

作業は4つの単位に分かれます。Unit 1 はドライバの変更、Unit 2 は根本原則の下で変更を誠実に保つ
作業、Unit 3 は実機なしで変更を確かめる作業、Unit 4 は文書を実際の動きに合わせる作業です。各単位
は依存順に並びますが、別々のプルリクエストとして着地できます。

**Unit 1 — 参照が古いという返事を、探し直すきっかけとして扱う。** 変更する箇所は、
[`bajutsu/drivers/xcuitest.py`](../../bajutsu/drivers/xcuitest.py) です。対象は、参照を使って
操作をする部分です。この部分は、`_resolve_handle` という探す処理と、それを使う `_actuate` という
操作の処理から成ります。ここで `stale`（参照が古いという返事）を、すぐに要素消失のエラーとする
のではなく、探し直すきっかけとして扱います。具体的には、`_resolve_handle(sel)` をもう一度実行
します。同じ条件からなお1つの要素だけが見つかるときは、参照を取り直して操作をやり直します
（小さな決まった回数まで）。条件から0個または複数の要素が見つかるときは、`ElementNotFound` や
`AmbiguousSelector` を送出します。この場合はそれ以上試さず、その結果ですぐに失敗します。探し
直すことこそが、2つの場合を分ける決め手です。要素がなお存在するタイミングのずれと、要素が存在
しない本当の消失です。この決め手があるので、たまたま条件に合った何かを tap することはありません。
参照を使う操作、すなわち `tap`、`double_tap`、`long_press`、`pinch`、`rotate` は、すべてこの部分
を通ります。したがって、やり直しの対象になります。座標を指定する `tap_point` は、古くなる参照を
持たないので、変更しません。試す回数の上限には、stale のやり直し専用の決まった数を使います。
初期値は BE-0207 と同じです（3回、0.5秒・1.0秒の指数バックオフ）。ただし、BE-0207 の通信のやり
直しの `_MAX_ATTEMPTS` と `_BACKOFF_BASE_SECONDS` とは分けて持ちます。2つのやり直しは、上限を決める
対象が違うからです。一方は `_resolve_handle` と `_actuate` のあいだで画面が落ち着くのを待つやり
直し、もう一方は `_with_retry` の中の通信のブレへのやり直しです。片方をあとから調整しても、
たとえば遅い CI 環境向けに通信の待ち時間を緩めても、もう片方が黙って変わってはいけません。その
ため全体は、1秒未満ではなく数秒程度で収まります。本当になくなった要素を、長く試し続けることは
ありません。

この探し直しは、やり直しの場合に限って、「探し当てたまさにその要素だけに操作をする」という決まり
を緩めます。本提案は、この引き換えを隠さずに述べます。探し直して条件から1つの要素だけが見つかっ
ても、証明できるのは「条件に合う要素が存在すること」だけです。それが、1回目に見つけた要素と
「同じ」であることまでは証明しません。2回のやり取りのあいだに、画面は変わることがあります。
たとえば、モーダルが閉じるような場合です。画面が変わると、同じ条件は、別の要素だけに合うように
なります。そのとき、やり直しはその別の要素に操作をしてしまいます。それでも、この引き換えは受け
入れられます。見かけよりも狭い引き換えでもあります。理由は次のとおりです。要素を探す処理は
Python 側にとどまります。したがって、決まりの本来の目的は損なわれません。この目的とは、要素を
探す権限を runner の外に保ち、あいまいな条件のもとで runner が黙って複数の候補から1つを選ぶのを
防ぐことです。やり直しは、Python 側で1つに絞り込めた場合にのみ操作をします。複数の候補のうちの
1つに操作することは、決してありません。しかも、操作する瞬間に探し直すという動きは、idb がすでに
行っていることと同じです。idb は、要素を探すことと tap を1回の手順で行います。したがってこの
やり直しは、XCUITest の `stale` への対応を、ほかのしくみに揃えるものです。より緩い規則を作り出す
ものではありません。より厳しいやり方も考えられます。条件だけでなく、もとの要素そのものの特徴にも
合うときに限って操作をやり直す形です。このやり方は、条件だけの一致では緩すぎるとわかった場合に
備えて、`implement-be` の選択肢として残します。

**Unit 2 — 判定のしかたを保ち、操作を二重にしない。** `stale` という返事のあとで tap をやり直す
ことは、時間切れのあとでやり直すこととは違って安全です。この違いは、BE-0218 が拠り所にした区別
と同じです。runner は、自身が要素へ触れる前に `stale` を返します。
[`BajutsuKit/Sources/BajutsuRunner/Router.swift`](../../BajutsuKit/Sources/BajutsuRunner/Router.swift)
では、`store.lookup(handle:)` が `.stale` を報告します。呼び出し先は、操作をせずに戻ります。
したがって `stale` という返事は、操作が起きなかったことの確かな証拠です。もう一度送っても、操作
を二重にすることはありません。対して時間切れは、届いたかどうかが不明のままです。不明であるため
に、BE-0218 は期限を過ぎた書き込みのやり直しを拒みました。判定のしかたは、そのほかの点でも損な
われません。大規模言語モデル（LLM）は経路に入りません。決まった待ち時間も足しません。問い合わせ
直しそのものが条件であり、試すたびの待ち時間は、安定化のための待機ではなく、Unit 1 で述べた
stale のやり直し専用の待ち時間です。本当の `stale` による消失、あいまいな探し直し、使い切った
試行の回数は、いずれも今までと同じ大きな失敗のままです。

**Unit 3 — 実機なしで変更を確かめる。** この部分は、見せかけの通信相手に対するユニットテストで
覆います。Simulator のいらない高速な関門で動くテストです。場合は4つあります。`stale` という返事
が1回あってから `ok` になり、条件からなお1つの要素だけが見つかるときは、回復します。`stale` が
続いても条件からなお1つの要素だけが見つかるときは、上限を使い切って大きく失敗します。`stale` の
あとの探し直しで条件に合う要素がもう見つからないときは、追加で試さず `ElementNotFound` ですぐに
失敗します。`stale` のあとの探し直しで条件に複数の要素が合うときは、`AmbiguousSelector` で失敗
し、決して操作をしません。4つの場合は、2つのことを確かめます。1つは、タイミングのずれが回復する
ことです。もう1つは、本当の消失やあいまいさが、なお失敗することです。この2つを確かめることで、
あとの変更が、誠実な関門をこっそり吸収する関門に変えることはできなくなります。

**Unit 4 — 文書を実際の動きに合わせる（BE-0113）。**
[`bajutsu/drivers/xcuitest.py`](../../bajutsu/drivers/xcuitest.py) の冒頭にある説明は、いまの
動きをこう述べています。参照が古くなると「idb が送出するのと同じ要素消失エラーとして表面化する」
というものです。この説明を、探し直すやり直しと、参照が古くてもなお失敗する正確な条件を述べる
ものに更新します。[`DESIGN.md`](../../DESIGN.md) と [`docs/architecture.md`](../../docs/architecture.md)
が `stale` の扱いを述べているなら、説明が実際の動きに合うように更新します。

## 検討した代替案

**探し直す決め手なしに、`stale` になった操作を無条件でやり直す。** この決め手を外すと、本当の
消失（前のステップの遷移が取り除いた要素）を、静かなやり直しに吸収してしまいます。BE-0049 が
禁じる「吸収による不安定さ」です。探し直す決め手こそが、やり直しを誠実に保ちます。要素がなお
存在するときに限って、操作をやり直すからです。

**起動後、最初の tap の前に決まった長さの安定化待ちを入れる。** 状況を無視した決まった待ち時間
は、根本原則2が明確に禁じています。しかもこの待ちは、タイミングのずれの起こる幅を狭めるだけ
です。その幅をなくすことはありません。安定化が決まった待ち時間より遅ければ、それでも参照は古く
なります。探し直すやり直しは、画面が落ち着くまでどれだけかかっても、このタイミングのずれを防ぎ
ます。

**要素を探す処理を runner に移し、2回のやり取りを1回にまとめる。** runner が操作のときに条件から
要素を探せば、画面の状態の記録によるすき間は消えます。しかし、Python 側で探すことは、xcuitest
のしくみが必ず守る決まりです。要素を探す権限は Python 側にとどまり、複数の要素に合いうる runner
側の条件には移りません。`stale` のやり直しも、この権限を Python 側に保ちます。Python 側で、しか
も1つに絞り込めた場合にのみ探し直すからです（この引き換えは *詳細設計* で詳しく述べています）。
探す処理を runner に移すと、この権限を手放すことになります。手放した先には、判定のしかたが防ごう
としてきたあいまいさが生まれ、タイミングのずれとの引き換えになってしまいます。

**CI でチェックを対象から外すか、ジョブ単位でやり直す。** `xcuitest (multi-touch)` を失敗しても
よいことにします。あるいは、ジョブをやり直しで包みます。どちらの場合も、タイミングのずれに触れ
ないままチェックを緑にできます。BE-0287 が姉妹の不安定さに対して退けるのと同じ理由です。本当の
タイミングのずれを緑のチェックの裏に隠し、静かなやり直しのたびに macOS を動かす費用を払い続ける
からです。
