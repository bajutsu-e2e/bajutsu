## はじめに

record の各ターンの**テキスト**側を痩せさせ、そのトークン消費を可視化して抑えられるようにします。
スクリーンショットに次いでターンあたり最大の入力は、描画されるアクセシビリティ要素ツリーで、要素
1 件につき 1 行、しかも**要素数に上限がありません**。加えて今日は、セッションの消費を抑える手段が
ユーザーにありません（`max_steps` の上限も `with_screenshot` の切り替えもループには存在しますが、
CLI に露出していません）。トークンがどこに使われたかを見る手段もありません（使用量は 1 つの累計と
してしか報告されません）。本項目は、精度に触れずに 3 つの補完的なことを行います。要素ツリーの描画を
可逆に圧縮し、すでにあるトークン予算ノブを CLI に露出し、使用量の報告を種別ごとに分解して、あらゆる
トークン削減策（本項目とその 2 つの姉妹提案）の効果を測れるようにします。

## 動機

ターンごとの user メッセージは `_render`（`bajutsu/claude_agent.py`）が組み立てます。ゴール、
プラン、直近アクションの要約に続けて、**画面上の要素 1 件につき 1 行**
（`- id=… label=… value=… traits=[…]`）を、操作対象になりうる要素すべてについて出力します。解析
（`parse_describe_all`、`bajutsu/drivers/idb.py`）はすでに `frame` などの生フィールドを捨て、
`_render` はすでにアプリルートと属性のないノードを飛ばしていますが、描画する要素**数には上限が
ありません**。長いスクロールリストやデータテーブルのある画面では、行ごとに 1 行ずつ、上限なく出力
されるため、要素ツリーのトークンコストは画面上の要素数に比例します。これはテキスト側で最大のコスト
であり、2 つの姉妹提案のどちらも扱わない唯一の項です（両者は画像を対象にします）。

さらに、コストを*制御*しにくく、*見*にくくしている 2 つの欠落があります。

- **CLI にトークン予算の制御がありません。** `record()` は `with_screenshot: bool = True`
  （`bajutsu/record.py`）を受け取り、`max_steps=30` を既定にしますが、
  `bajutsu/cli/commands/record.py` はそのどちらも露出していません。ループが両方を支えているのに、
  ユーザーはセッションのターン数を抑えることも、画像を落とすこともできません。要素ツリーを抑える
  フラグもありません。
- **使用量が 1 つの不透明な累計です。** `usage.py` はすべての AI 呼び出しを 1 つの `TokenUsage` に
  積み上げ、CLI は 1 つの累計を表示します。スクリーンショット、要素テキスト、初回の `plan` 呼び出し、
  アラートガードの視覚呼び出しのどれにどれだけ使ったかをユーザーは見られないため、どこに労力を割くか
  も、削減が実際に効いたかの確認もできません。

本項目は 2 つの姉妹提案に対するテキストと計測の対応物です。姉妹提案 `record-vision-on-demand` は
画像を送る頻度を減らし、姉妹提案 `record-screenshot-downscale` はそのサイズを抑えます。本項目は要素
テキストを痩せさせ、そして何より、3 つすべてを測れるようにする種別ごとの可視化を加えます。
[BE-0178](../BE-0178-record-multi-action-turn/BE-0178-record-multi-action-turn-ja.md)（ターン数）とは
独立しています。

### 精度を保つ理由

要素ツリーの圧縮は**操作対象の特定について可逆**です。エージェントが操作しうる要素（`id` / `label` / `value` / `traits`
を持つもの）はすべて常に完全な形で描画されるので、今日エージェントが取れるアクションが使えなくなる
ことはありません。圧縮が取り除くのは冗長な文字だけ（情報を持たない空の `value`/`traits`）で、病的に
長い画面では、操作対象にならない残りを黙って落とさず件数として要約します。エージェントには何を省いた
かを伝え、誤解させません。CLI のノブは今日の挙動を既定にし（オプトインのみ）、使用量の分解は報告
専用です。3 つのどれも、通常の画面でエージェントの入力を判断が変わるように変えることはありません。

## 詳細設計

### 1. 可逆な要素行の圧縮

`_render` の要素ごとの行を変え、情報を持たないフィールドを省きます。空の `value` と空の `traits` は、
`value='' traits=[]` と印字せず落とします。要素を*区別する*フィールド、すなわち `id`、`label`、空で
ない `value`、空でない `traits` は常に残すので、描画は忠実で完全に特定可能なリストのままです。これは
純粋な文字の削減で、要素の集合とそれぞれを特定できることは変わりません。

### 2. 病的な画面のための決定的で安全な上限

要素数が定数の閾値を超える画面には、**構成上安全な**上限を適用します。`id` / `label` / `value` / `traits` を持つ要素
（エージェントが操作しうる、特定可能なもの）は**すべて**残し、装飾だけの残り（id も label も value も traits もない）を
末尾の要約行にまとめます。たとえば `- (+N further non-addressable elements omitted)` のようにします。
特定可能な要素は決して落とさず、画面が切り詰められたことをエージェントに明示するので、必要なら `swipe` して追加の要素を露出できます（既存の off-screen-control の経路）。
### 3. CLI のトークン予算制御

ループの既存ノブを `bajutsu/cli/commands/record.py` から通します。

- `--max-steps N`：ループの `max_steps`（今日はハードコードの既定 30）を露出し、ユーザーがターン数、
  ひいては最悪ケースの消費を先に抑えられるようにします。
- `--no-screenshot`：ループの `with_screenshot=False` を露出し、アプリが完全に計装されていると
  ユーザーが分かっていて最も安価に record したいときの、要素のみのセッションにします。（これは
  姉妹提案 `record-vision-on-demand` の自動判断の、率直で手動な版です。両者は共存します。フラグは
  強制的に無効化し、フラグが未指定なら自動経路がターンごとに判断します。）

どちらも今日の挙動を既定にするので、ユーザーがオプトインしない限り何も変わりません。

### 4. 種別ごとのトークン使用量

使用量の集計（`bajutsu/usage.py` と record CLI の報告）を拡張し、トークンを消費した呼び出し箇所に
帰属させます。少なくとも `plan` 呼び出し、ターンごとの `next_action` 呼び出し、アラートガードの視覚
呼び出しに分け、セッション末尾の要約を 1 つの累計ではなく小さな内訳として読めるようにします。これは
報告専用で（`usage.py` の既存のベストエフォート契約と一貫して、合否には決して触れません）、§1〜§3 と
2 つの姉妹提案の効果を、想定ではなく**測れる**ようにするものです。

### 決定性・プライムディレクティブ適合・ゲート

厳密に**Tier 1（record のみ）**です。`run` や CI にモデル呼び出しは加えません（プライムディレク
ティブ 1）。圧縮と上限は要素リストに対する決定的なテキスト操作で、`sleep` を含まず、セレクター解決を
変えません。曖昧なセレクターは今も即座に失敗します（プライムディレクティブ 2）。圧縮・上限の閾値・
ノブはすべて全体的で、アプリ別の分岐を持ちません（プライムディレクティブ 3）。記録されるシナリオ
成果物は変わらないので、`run`、`codegen`、レポート、それらのテストは影響を受けません。

### テスト戦略（Simulator 不要、Linux の `make check` ゲートに収まる）

- **圧縮描画テスト**：空の `value`/`traits` を持つ要素はそれらのフィールドなしで描画される。
  `id`/`label`/空でない `value`/`traits` を持つ要素はそれぞれ描画される。圧縮後の出力も同じ特定可能な
  セレクターの集合に戻る。
- **上限テスト**：閾値未満の画面はすべての要素を描画する。閾値超の画面は id/label を持つ要素をすべて
  残し、操作対象にならない残りだけを要約行にまとめ、省略件数が正しい。特定可能な要素は決して落ちない。
- **CLI ノブテスト**：`--max-steps` がループのターンを抑える。`--no-screenshot` が要素のみの観測
  （`ImagePart` なし）を生む。どちらも未指定なら今日の挙動を既定にする。
- **使用量内訳テスト**：`plan` 呼び出し、N 回の `next_action` 呼び出し、アラートガード呼び出しを持つ
  スクリプト化セッションが、トークンを正しい種別に帰属させ、種別の合計が既存の累計に一致する（二重
  計上しない）。

### スコープと非目標

**スコープ内**：可逆な要素行の圧縮、安全で決定的な要素の上限、`--max-steps` と `--no-screenshot` の
CLI フラグ、種別ごとのトークン使用量の報告。

**非目標**：画像を送るかどうかの判断（姉妹提案 `record-vision-on-demand`）、画像のリサイズ（姉妹提案
`record-screenshot-downscale`）、ターン数を減らすこと（BE-0178）、特定可能な要素を落とすことや LLM で
ツリーを要約すること（精度を危険にさらし、不必要に作成のクリティカルパスへモデルを載せます）、
`crawl` 探索器の描画（ここでは対象外）、シナリオスキーマや `run`/再生の変更。

## 検討した代替案

- **長い要素ツリーを LLM で要約する。** 本ターンの前に安価なモデルでツリーを凝縮させる。却下：決定的
  な圧縮なら精度の危険なしに達成できるテキスト削減のために、作成経路へモデル呼び出しを足すことになり、
  しかもエージェントが必要とするまさにその要素を落としかねません。
- **要素リストを N 件で強制的に切り詰める。** 最も単純な上限ですが、ゴールが必要とする特定可能な要素
  を黙って落とし、record を壊しかねません。却下：操作対象にならないノイズだけをまとめ、省略を報告する、
  id/label を保つ上限を選びます。
- **直近アクションの窓をさらに絞る。** 履歴はすでに直近 6 件だけで、各件は 1 行の要約（`_history_line`）
  であり、要素ツリーに比べて小さいものです。却下：ノブに値しません。テキストコストがあるのは要素
  ツリーです。
- **使用量を 1 つの累計のままにする。** より単純ですが、そうすると本項目や姉妹提案のどの変更も、効いた
  ことを*示せ*ません。却下：内訳は安価で（報告専用）、トークン削減の取り組み全体の計測基盤です。
