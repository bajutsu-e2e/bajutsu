## はじめに

散文の執筆規範を一つに束ねる **tech-writing** スキルを追加します。BE 項目とすべての散文
ドキュメントが、英語でも日本語でも、この一つの規範に従います。規範の中身は、Jeffrey Scott Vitter
の *Structure + Style = Communication* から言語に依存しない執筆技法を抽出し、その記事が定める
英語固有の作法と組み合わせたものです。既存の
[`japanese-tech-writing`](../../.claude/skills/japanese-tech-writing/) スキルは、この下位に位置する
日本語レイヤーになります。一つの規範が執筆全体を統べながら、各言語はその文法にしか関わらない指針
だけを保ちます。

## 動機

プロジェクトの執筆規則は存在しますが、二つの言語のあいだで偏っています。日本語の散文には、専用の
厚い規範があります。[`japanese-tech-writing`](../../.claude/skills/japanese-tech-writing/)
スキルが、冗語や重複、段落構成、論証の厳密さ、演出の抑制までを定めています。いっぽう英語の散文には、
[`docs/ai-development.md`](../../docs/ai-development.md) の *Documentation style* 節にある数項目
（造語をしない、不自然な訳語を避ける、省略しない、略語を初出で綴る）しかありません。英語について、
文をどう組み立てるか、強調をどこに置くか、主語と述語をどう近づけるか、どの冗語を削るかは、どこにも
書かれていないのです。同じ書き手が一つの BE 項目の日本語側と英語側を書くとき、日本語では詳しい
規則を相手にし、英語ではほぼ沈黙を相手にする、という不均衡が生まれています。

Vitter の記事は、この英語側の空白をちょうど埋める、簡潔で実績のある出典です。そしてその多くは言語に
依存しません。書き始める前に上から下へアウトラインを立てる価値、文の二つの強調位置、主語と述語の
近接、能動態、冗語を削る規律は、日本語の散文にも同じように効きます。これらのいくつかは
`japanese-tech-writing` に日本語版の対応物がすでにあります（冗語と重複の規則です）。だからこそ二つの
規範は、同じことを二重に書いて時とともに食い違わせるのではなく、言語に依存しない中核を一つ共有する
べきです。

具体的な受益者は BE 項目です。BE 項目はどれも二言語で書かれた論証文書であり、冒頭で主眼を述べる
「はじめに」、既知の問題から新しい結論へと進む「動機」、二つの言語でともに読みやすくあるべき
「詳細設計」を持ちます。共有の規範は、今後書かれるすべての項目の下限を引き上げ、レビューが指し示せる
拠り所を一つ与えます。docstring について BE-0065 が、項目テンプレートについて BE-0074 が果たしたのと
同じ役割です。

## 詳細設計

作業は独立した四つの単位に分かれます。

1. **新しい `tech-writing` スキル**（`.claude/skills/tech-writing/SKILL.md`）。二部構成です。
   - **言語に依存しない原則（両言語に適用）。** Vitter から抽出します。
     - **段階的推敲**：上から下へ書きます。まず論点を扱う順に並べた粗いアウトライン、次に各節の
       見出しと主眼、次に小論点と筋、最後に完成に近い散文、という順に進めます。論証のある長い節を、
       最初の一文から順に線形に書き下ろさないようにします。
     - **明快さと簡潔さ**：推敲のとき、意味を持たない語や句や文を削ります。
     - **主眼を冒頭で述べる**：BE 項目の「はじめに」と「動機」は、問題、背景、貢献、なぜ重要かを、
       詳細より先に述べます。最初の二段落で読むのをやめた読者にも、要点が残るようにします。
     - **文の強調位置は文頭と文末**：文末を最も重要な要素のために空けておき、既知から未知へ、背景
       から結果へと進めます。接続や背景を表す語（「したがって」「しかし」「前節では」）は文頭に置き、
       中身を文末に取っておきます。
     - **主語と述語を近づける**：あいだに長い節を挟むと文が読み取りにくくなるので、主語に対応する
       述語が早く現れるように書き直します。
     - **能動態と一人称複数**：能動態を選び、提案では「私」ではなく「私たち」として書きます。
     - **冗語を削る**：意味を足さない語や句を落とします（英語の "actually"、"basically"、
       "essentially"、"it is important to note that"、"note that" や、意味のない "now" など）。
       "thus" と "therefore" は、本当に結論を示す箇所だけに残します。
   - **英語固有の作法（英語の散文のみ）。** Vitter の文法・スタイル節から採ります。"a lot of" より
     "many" / "much" を使う。短縮形は綴る。記号や省略記号は数式に限り、本文では語を綴る。三つ以上の
     並列にはシリアルコンマ（オックスフォードコンマ）を打つ。制限用法の *that*（コンマなし）と
     非制限用法の *which*（コンマで区切る）を区別する。"this" / "that" を裸の名詞にせず「this X」と
     書く。性別を特定しない単数の人を "they" で受けない（複数形か役割名に言い換える）。複合形容詞に
     ハイフン、範囲に en ダッシュ、強調に em ダッシュ（前後に空白を伴う。これはこのリポジトリ独自の
     慣例であり、シカゴ方式は空白を入れない）を使う。形容詞
     として使う 10 未満の正の整数は綴り、名詞として使うときは数字にする。コロンは完全な文のあとにだけ
     置く。出典を記す短い注（HTML コメント）で Vitter を明記します。`japanese-tech-writing` が自身の
     出典を記しているのと同じ形です。
2. **`japanese-tech-writing` との階層関係。** 新しいスキルは自らを上位の規範と宣言し、
   `japanese-tech-writing` を日本語レイヤーとして名指します。日本語の散文を書くときは両方を適用し、
   重なる部分（冗語、重複、演出の抑制）では日本語スキルの具体的な記述を優先します。
   `japanese-tech-writing` の冒頭には、共有の言語非依存規範である `tech-writing` を指す一行を足します。
   どちらに辿り着いた読者も、もう一方を見つけられるようにするためです。日本語の規則は何も変わりません。
   英語固有の作法が日本語に適用されることもありません（日本語にはダッシュや約物の独自規則があり、
   それはすでにこのスキルに含まれています）。
3. **重複のない参照配線。** 中身を写し取るのではなく、既存の置き場所から新しいスキルを指すようにします。
   - [`docs/ai-development.md`](../../docs/ai-development.md) の *Documentation style* 節で、両言語の
     散文規範として `tech-writing` を名指します。現在の `japanese-tech-writing` へのポインタと並べます。
   - [`CLAUDE.md`](../../CLAUDE.md) の *Conventions* に短い形のポインタを、既存の
     `japanese-tech-writing` の行と並べて足します。
   - [`roadmaps/README.md`](../../roadmaps/README.md)（と `README-ja.md`）の BE 執筆節に、項目は
     `tech-writing` に従う旨を、既存の敬体・`japanese-tech-writing` の記述の隣に書きます。
   - BE の散文を起草するスキルは、書くその瞬間に `tech-writing` を指します。
     [`ideation`](../../.claude/skills/ideation/SKILL.md)（起草の手順）と
     [`propose-and-build`](../../.claude/skills/propose-and-build/SKILL.md) の Phase A が、既存の
     `japanese-tech-writing` ポインタの隣で、起草の前にこれを呼び出します。規範がドキュメントで
     見つかるだけでなく、執筆時に能動的なものとして働くようにするためです。
4. **適用範囲は BE 項目とすべての散文ドキュメント、両言語。** 規範は `roadmaps/BE-*/*.md` と
   `*-ja.md`、および `docs/` 以下（とその `docs/ja/` ミラー）のすべての文書に適用します。*Documentation
   style* 節が現在持つ適用範囲（「BE 項目を含む」）と揃えます。コードの docstring には及ばず、docstring
   は自身の別の規範（BE-0065）を保ちます。

**執行はレビュー時の規範であり、CI ゲートにはしません。** 散文が実際に明快に読めるか、主眼を冒頭で
述べているか、冗語を削れているかの判定には、意味の判断が要ります。それをゲートに載せれば、`run` や
CI の合否経路に LLM を置くことになり、プライムディレクティブ 1 が禁じます。したがってこの規範は、
二言語ドキュメントの規則や DESIGN.md 整合の規則（BE-0113）とまったく同じ形で成り立ちます。機械が
検証する表明ではなく、コントリビュータとレビュアーが人手で確かめる期待です。

## 検討した代替案

- **専用スキルを作らず、*Documentation style* の項目をその場で厚くする。** 却下しました。直そうと
  している不均衡がそのまま残るからです。日本語の指針は、書き手が執筆の前に呼び出すよう促される厚い
  スキルにあります。英語の技法を数項目のドキュメントに畳み込むと、書いたあとに読む受け身の参照に
  とどまります。スキルは、規範を執筆時に能動的なものにするために、このリポジトリがすでに使っている
  仕組みです。
- **上位規範を置かず、`japanese-tech-writing` と対称の独立した `english-tech-writing` スキルを作る。**
  却下しました。二つのスキルは言語に依存しない中核（先にアウトラインを立てる、冗語を削る、文の強調
  位置）を二重に書き、時とともに食い違います。共有の原則を一つの上位規範が持ち、各言語の作法はその
  言語に委ねる形にすれば、真実の出所が一つに保たれます。

  **その後採用しました（[#1157](https://github.com/bajutsu-e2e/bajutsu/pull/1157)）。** 英語固有の
  作法は後に `english-tech-writing` スキルへ分離しました。ただし上位規範は外さず残したので、却下の
  理由だった食い違いは起きません。`tech-writing` は引き続き言語非依存の中核だけを持ちます。
  `english-tech-writing` と `japanese-tech-writing` は、各言語の作法だけを持って上位規範に委ねる、
  対称なレイヤーになりました。きっかけは、`tech-writing` の下の textlint に英語の散文向けルールが
  加わったことです。英語にも機械的な検査が及ぶようになった一方で、英語の作法だけを上位規範の中に
  置いたまま（日本語には専用レイヤーがある）にするのは、解消すべき非対称でした。
- **一部を CI で執行する（冗語や "a lot of" などを検出する散文リンタ）。** プライムディレクティブに
  照らして却下しました。語彙のブロックリストは機械で検証できますが、規範の本体（明快さ、強調、論証の
  順序）は検証できません。部分的なリンタは、検証できない大多数を任意のものと見なす誘惑を生みます。
  この規範は、リポジトリの他のすべての散文規則と同じく、レビューの期待とします。
