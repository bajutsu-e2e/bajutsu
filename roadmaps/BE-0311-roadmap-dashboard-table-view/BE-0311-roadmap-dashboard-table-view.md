## Introduction

Add a table view to the roadmap status dashboard (the GitHub Pages page shipped by
[BE-0094](../BE-0094-roadmap-status-dashboard/BE-0094-roadmap-status-dashboard.md) and extended
with search by [BE-0219](../BE-0219-roadmap-dashboard-search/BE-0219-roadmap-dashboard-search.md)),
as an alternative to its card grid. A reader toggles between Cards and Table; the table renders
every item as one row with six sortable columns — ID, Title, Topic, Status, Created, and Updated —
and clicking a column header sorts the whole table by that column, ascending or descending. The
chosen view persists across visits through the browser's `localStorage`, and the existing search box
and status-filter chips keep working unchanged in both views. Created and Updated are two columns
the dashboard has never shown before: each is derived from the item's Git history at build time (the
earliest and latest commit across both of the item's language files) rather than added as a new
hand-set field, so the dashboard need not trust every future edit to keep two dates in sync with the
truth.

## Motivation

The card grid groups items by Topic and shows each item's status as a badge, which suits browsing
one category at a time. It has no way to answer a different, common question: which items changed
most recently, or how the whole roadmap looks sorted by id regardless of topic. A card grid has no
stable column to sort by — sorting becomes meaningful once every item is a row in one flat table, no
longer boxed into its topic's section.

Two contributor workflows want exactly that flat, sortable table. Reviewing what changed recently
means sorting by Updated and scanning the top of the list, instead of opening each category by hand
and comparing dates that do not exist on the page today. Reviewing how long a proposal has sat
untouched means sorting by Created. Neither is served by the existing card layout, and neither the
browser's Ctrl+F nor the dashboard's own search box narrows the problem: both only narrow *which*
cards show, never *what order* they show in.

This is a documentation/contributor-workflow improvement only, and it stays clear of every prime
directive ([`CLAUDE.md`](../../CLAUDE.md)): the dashboard is a static docs-site artifact, so no large
language model (LLM) sits anywhere near it, no `run`/CI verdict path exists here, no determinism
surface is touched, and the table is generic over whatever items the metadata produces, so nothing
about it is app-specific.

## Detailed design

The dashboard is generated by
[`scripts/build_roadmap_dashboard.py`](../../scripts/build_roadmap_dashboard.py): `render_html` emits
one `.be-card` per item inside `.be-cat` topic sections, each section carrying a progress bar derived
from `Status` alone, and a `<script>` block progressively enhances the search box and status chips
(BE-0219). The table view slots into that same structure rather than replacing it. The work is
mutually exclusive, collectively exhaustive (MECE) across these units:

1. **Git-derived Created / Updated dates.** Extend
   [`scripts/build_roadmap_index.py`](../../scripts/build_roadmap_index.py)'s `Item` with two ISO-8601
   dates, read at build time by running `git log --follow --format=%aI -- <path>` once per language
   file (the English file and its `-ja.md` mirror) and combining the two results: `created` is the
   oldest line either command reports (each file's first commit), `updated` is the newest (either
   file's most recent commit). Running the two files separately, each with `--follow`, is deliberate:
   `--follow` only accepts one path and correctly walks a file's history through the rename every
   item undergoes when CI allocates its real id (`BE-0311-<slug>` → `BE-NNNN-<slug>`), and combining
   both files' results is what keeps `updated` correct when only one of the two — say, a
   Japanese-only wording fix — changes after the other. The dashboard's one firm rule about the data
   it shows (BE-0094's "Honesty of the data") is to never invent a number with no source in the
   metadata — true of `Origin`, rendered verbatim when an item sets it, and of the per-topic progress
   percentage, derived purely from `Status`. A hand-set `Created`/`Updated` pair would not break that
   rule on day one, but unlike `Origin` — written once at authoring time and rarely revisited — a date
   pair needs a correct edit on every single future change to the item, making it exactly the kind of
   field most likely to silently drift from the truth; deriving both from `git log` avoids adding that
   field at all. The one prerequisite this adds is
   that [`.github/workflows/docs.yml`](../../.github/workflows/docs.yml)'s checkout step must fetch
   full history (`fetch-depth: 0`) instead of its current shallow clone, since `git log` on a
   single-commit checkout would report every item as created and updated on the same day; the
   trade-off is a small, recurring increase in checkout time on every docs build from then on, not a
   one-time cost. Running two `git log --follow` subprocesses per item — roughly 600 across the
   current roadmap — adds a second, related recurring cost; each is a local read over an already
   fetched history with no network round-trip, so it is expected to stay negligible next to the
   checkout itself, but if it ever measurably slows the build, collapsing it to one repository-wide
   `git log --name-status -M` walk that resolves every item's dates in a single pass is the cheaper
   alternative to reach for.
2. **View toggle.** Render a two-way control (`Cards` / `Table`) beside the existing `.be-filters`
   row. Both views read from the same rendered item data — the toggle only shows one of two sibling
   containers (`.be-cards-view` / `.be-table-view`) and hides the other; nothing is fetched or
   recomputed when a reader switches.
3. **Table markup.** Render one `<table>` per item set, one `<tr>` per item, with six `<th>` columns
   — ID, Title, Topic, Status, Created, Updated — each `<th>` carrying a `data-sort-key` and a click
   handler. Reuse each card's existing per-item fields (id, title, topic, status) and add the two new
   date columns; a row carries the same `data-status` / `data-topic` attributes as its corresponding
   card, so the search box and status chips (BE-0219) filter table rows exactly as they filter cards,
   with no separate matching logic. The per-topic progress bar a reader relies on in Cards view is not
   dropped: a compact strip listing every topic with its existing stacked bar and percentage renders
   above the table, reusing the same `_progress_bar` computation the card sections already use.
4. **Client-side sort.** Extend the existing filter script (`_SCRIPT`) with a sort step: clicking a
   `<th>` re-orders the table's `<tr>` elements by that column (string comparison for Title/Topic/
   Status, id-numeric comparison for ID, chronological comparison for Created/Updated), toggling
   ascending/descending on repeat clicks of the same column, and marking the active column and
   direction with an `aria-sort` attribute. Sorting only reorders the already-rendered rows that
   have passed through the status/search filter — it never changes which rows are visible.
5. **Persisted view choice.** On toggling Cards/Table, write the choice to `localStorage` under a
   dedicated key; on page load, read that key and show the matching view before first render (falling
   back to Cards, the current default, when the key is absent or `localStorage` is unavailable). Like
   the rest of the dashboard's interactivity this is progressive enhancement: without JavaScript the
   page shows the Cards view unconditionally, exactly as it does today.
6. **Styling.** Add the table, toggle, and progress-strip CSS to `_STYLE`, matching the existing chip
   and card styling and legible in both the light and dark MkDocs themes.
7. **Tests.** Extend
   [`tests/test_roadmap_dashboard.py`](../../tests/test_roadmap_dashboard.py) to assert: every item
   renders one table row with its six columns; each date column holds a value in ISO-8601 form; the
   view toggle and its two containers are present; and the sort script wires a click handler to every
   `<th>`. This is the machine-checkable outcome; it needs no browser and keeps the gate fast and
   LLM-free.

No change to what the page *contains* — the item set and its fields are unchanged apart from the two
new date columns — only a second way to lay the same items out. `docs/architecture.md` / `docs/ja/`
need no update, as they describe the dashboard's purpose, not its view affordances; if a sentence
there enumerates the dashboard's views, update it in the same PR
([BE-0113](../BE-0113-design-doc-realignment/BE-0113-design-doc-realignment.md)).

## Alternatives considered

- **Replace the card grid outright instead of adding a toggle.** Rejected: the per-topic progress bar
  and the browsing-by-category workflow the card grid already serves well would either disappear or
  need reinventing inside a table row, for no benefit to the sorting use case this item targets. A
  toggle keeps both workflows at full strength.
- **Hand-maintained `Created` / `Updated` metadata fields on each item, updated by the `ideation` and
  `implement-be` skills.** Rejected: it needs a one-time backfill across roughly 300 existing items,
  and — unlike `Origin`, a fact set once at authoring time that rarely goes stale — every future edit
  becomes an opportunity for a hand-set date pair to drift from the file's real history. Deriving both
  dates from `git log` at build time removes the drift risk entirely, at the cost of a small, recurring
  increase in the docs workflow's checkout time (full history instead of shallow) on every future
  build.
- **Sort within each topic's card section instead of a flat table.** Rejected: the two motivating
  workflows — "what changed most recently" and "how old is this proposal" — are both roadmap-wide
  questions, and confining sort to one topic at a time would still require opening every section by
  hand to compare across them. A flat table answers both in one sort.
- **A server-side / build-time sort (bake one fixed sort order into the generated page).** Rejected:
  a reader wants ID order sometimes and Updated order other times, and a static page can only bake in
  one order — client-side sort over already-rendered rows costs nothing at build time and serves every
  order a reader might want.
