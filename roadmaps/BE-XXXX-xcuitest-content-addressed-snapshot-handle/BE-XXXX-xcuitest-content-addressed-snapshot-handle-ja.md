## はじめに

XCUITest のしくみは、画面の要素が1つも動いていない状況でも、tap を `element vanished (stale handle)`
というエラーで失敗させることがあります。参照が古いという返事を受けたら探し直す
[BE-0289](../BE-0289-xcuitest-stale-handle-reresolve/BE-0289-xcuitest-stale-handle-reresolve-ja.md)
のやり直しでも、この失敗のなかまは塞げません。原因は、アプリやシナリオではなく、オンデバイスの
runner が参照を決めるやり方にあります。runner は、要素の情報を取り直すたびに、画面上のすべての
要素に新しい参照を発行し、以前の参照をすべて無条件に無効にします。参照が、要素そのものの内容では
なく、要素を見た「取り直しの世代」を表しているからです。そのため、余分な `GET /elements` が1回
あるだけで世代は進みます。通信のやり直しの仕組みは、読み取りが一瞬途切れたように見えると
`GET /elements` を送り直すので、余分な `GET /elements` は容易に起こります。要素を探してから操作
するまでのあいだに画面が何も変わっていなくても、送り直しによって世代が進み、これから使おうとして
いる参照が無効になります。本項目は、参照を回数を数えるカウンタからではなく、要素の内容（識別子、
ラベル、値、traits）から決定的に導くことを提案します。画面に残っている要素は、情報を何度
取り直しても同じ参照を保ち、runner が余分に情報を取り直しても参照は有効なままになります。変更は
runner の `SnapshotStore` だけで完結します。参照はどの利用者にとっても不透明な文字列のままなので、
Python 側のやり取りと runner の HTTP ルータには手を入れません。

## 動機

この不安定さは、BE-0289 を入れたあとでも、参照を使って操作する経路で再現します。`id` を使った tap
が、大きな画面遷移の直後——たとえばフルスクリーンのモーダルが出た直後——に
`element vanished (stale handle)` で失敗します。対象の要素は、そのあいだずっと画面に存在しています。
次の4つの観測は、原因が画面の変化ではなく runner 側にあることを示します。

- 要素への `wait` が成功した瞬間に取った evidence のスナップショットと、続く tap が `stale` で失敗
  したときに取った evidence のスナップショットは、対象の要素の frame が1ピクセルまで一致します（どちらも
  `[16.0, 784.0, 370.0, 40.0]`）。要素は、動いても、大きさが変わっても、画面から出てもいません。
- `dismissAlerts: false` にしても不安定さは止まりません。人工知能（AI）の経路も、画像認識の経路も
  関わっていないということです。
- 参照を持たない座標指定の tap である `tapPoint` では、この不安定さは一度も再現しません。原因が
  tap そのものではなく参照にあることを示します。
- 同じシナリオの中でも、大きな画面遷移の直後ではない tap のステップは成功します。不安定さは条件では
  なく runner の混み具合に連動します。

参照の決め方が、余分な読み取りを古い参照に変えます。
[`BajutsuKit/Sources/BajutsuRunner/SnapshotStore.swift`](../../BajutsuKit/Sources/BajutsuRunner/SnapshotStore.swift)
の `refreshSnapshot` は、呼ばれるたびに単調増加する `generation` カウンタを進め、`currentElements`
を空にし、要素ごとに `h-<generation>-<index>` を発行します。そのため、あるスナップショットの参照は、
画面が変わったかどうかに関わらず、次のスナップショットで無効になります。新しいスナップショットの
唯一のきっかけは `GET /elements` です。参照を使うステップでは、1回の `GET /elements`（`_resolve_handle`
を経由）に続いて1回の `POST /tap`（`_actuate` を経由）が送られるので、ふつうのタイミングでは、探して
から操作するまでのあいだに世代は動きません。世代を動かすのは、余分な `GET /elements` です。

余分な読み取りは、通信のやり直しの仕組みから生まれます。
[`bajutsu/drivers/xcuitest.py`](../../bajutsu/drivers/xcuitest.py) の `_is_retry_eligible` は、
すべての `GET` に対して真を返します（`return not delivered or method == "GET"`）。読み取りは何度
実行しても同じ結果になるので、BE-0207 が一瞬の通信のブレを吸収するためにこの仕組みを入れたからです。
`GET /elements` がクライアント側で時間切れに見えると、`_with_retry` は送り直します。大きな画面遷移の
直後は、runner のメインスレッドが落ち着きかけの画面の情報を取り直すのに忙しく、読み取りが遅くなって
ちょうど途切れたように見えます。2つの効果が重なって、古い参照が生まれます。

1. 1回目の `GET /elements` がクライアント側で通信の失敗と判定され、`_with_retry` が送り直します。
2. クライアントは、送り直した読み取りの応答（世代 N+1）から条件を絞り込み、runner が世代 N+1 で発行
   した参照を付けて `POST /tap` を送ります。
3. 1回目の読み取りは、忙しい runner の上で実際にはまだ動いており、送り直しの**後**で完了して
   `refreshSnapshot` を実行します。世代は N+2 に進み、`currentElements` はまるごと入れ替わります。
4. クライアントが送った世代 N+1 の参照は、もう `currentElements` のキーではありません。runner は
   `stale` を返し、しくみは要素消失のエラーを送出します。画面から一度も出ていないボタンに対してです。

BE-0289 のやり直しは、この場合を回復できません。回復できない理由が、別の直し方を選ぶ動機になります。
BE-0289 のやり直しは、条件からなお1つの要素だけが見つかるあいだ問い合わせ直して操作をやり直します。
参照が古くなった原因が、新しい問い合わせに映る画面の変化だと見なす作りです。ここでは画面は変わって
おらず、runner が忙しいだけです。しかも、やり直しが送る問い合わせ直しは、それ自体が `GET /elements`
であり、同じ送り直しと割り込みのタイミングのずれにさらされます。runner が忙しいあいだ——遷移の
直後の窓——は、BE-0289 の上限まで数えた試行のどれもが、同じ理由のなさで `stale` になり、ステップは
失敗します。2つの項目は、別々の層を扱います。BE-0289 はクライアント側で本物のタイミングのずれを
許すのに対し、本項目は参照から世代を取り除き、変わっていない要素にそもそも新しい参照を割り当てない
ようにします。

この不安定さは、BE-0289 が記したのと同じ3つの意味で負担になります。必須のオンデバイスの関門が
ランダムに失敗すると、関門への信頼が損なわれ、反射的な再実行か無駄な調査のどちらかを招きます。
再実行のたびに、アプリをビルドし Simulator を起動するジョブで、macOS を動かす費用がかかります。
そして `element vanished` というメッセージは、要素がずっと存在しているのに、要素が存在しないかの
ように読めて調査を誤らせます。

## 詳細設計

作業は4つの単位に分かれます。Unit 1 は参照の導き方を変える作業、Unit 2 は BE-0289 のやり直しが
拠り所にする `stale` と `notFound` の区別を保つ作業、Unit 3 は実機なしで変更を確かめる作業、Unit 4
は文書を実際の動きに合わせる作業です。各単位は依存順に並びますが、別々のプルリクエストとして着地
できます。

**Unit 1 — 参照を、世代のカウンタからではなく要素の内容から導く。** `SnapshotStore` で、
`h-<generation>-<index>` の方式を、要素の安定した身元——`identifier`、`label`、`value`、`traits`
——から決定的に導いた参照に置き換えます。スナップショットが持つフィールドのうち2つは、導出から
あえて外します。1つは `backingElement` です。同じ画面上の要素でも、`queryElements` を呼ぶたびに
新しいオブジェクト参照が返るので、`backingElement` を含めると、本項目が取り除こうとしているまさに
その欠陥——スナップショットのたびに参照が変わる——を招くからです。もう1つは `frame` です。遷移の
あとに画面が落ち着きかけているときこそ runner が情報を取り直すので、割り込む2つの読み取りのあいだで
frame が1ピクセル未満でもずれると、変わっていない1つの要素に別々の参照を発行し、本項目が狙う窓で
タイミングのずれが再発するからです。しかも frame は操作の精度を上げません。参照は `backingElement`
に解決され、runner はその参照を通じて操作し、記録された frame を通じて操作することはないからです。
`refreshSnapshot` は今までどおり呼ばれるたびに `currentElements` を空にして作り直すので、参照が指す
`backingElement` は常に最新の問い合わせのオブジェクト参照です。変わるのは、キーが内容から導いた参照に
なる点だけです。身元が変わらない要素は、スナップショットをまたいで同じ参照の文字列に対応します。
そのため、余分な `GET /elements` は、変わらない要素に同一の参照を再発行し、*動機* で述べた割り込みは
もう参照を無効にできません。クライアントの参照がどのスナップショット由来であっても、要素が画面に
ある限り、参照は生きたキーのままだからです。同じ画面上に、内容が完全に一致する要素が複数ありうる
ので（識別子のない「Delete」ボタンが2つある場合など）、参照を区別するために、1つのスナップショットの
**中**での衝突には、問い合わせの順に出現番号を付けます。最初の出現は素の内容の参照を保ち、あとの
出現は番号を付けた変種を取ります。

**Unit 2 — `stale` と `notFound` の区別を保つ。** BE-0289 のやり直しは、runner の返事で分岐します。
`stale` の返事は探し直しのきっかけになり、`not-found` の返事はすぐに失敗させるので、参照の方式を
変えたあとも2つを見分けられなければなりません。今この2つを見分けているのは世代のカウンタです。
`lookup` が参照を解析し、正しい形式で世代が最新でない参照は `stale`、解析できない参照は `notFound`
と読みます。内容から導いた参照には、解析すべき世代がありません。そこで `SnapshotStore` は、これまでに
発行したすべての参照の集合を記録し、`lookup` は、参照が生きたキーなら `.found`、以前に発行したが生きた
キーではない（参照が名指した要素が画面から出たか内容が変わった）なら `.stale`、一度も発行していない
（形式が不正か、でっち上げの文字列）なら `.notFound` を返します。runner の HTTP ルータは、結果に対する
今の3分岐をそのまま保ち、BE-0289 のドライバ側のやり直しもそのまま動きます。本当の消失は、以前に発行
した参照を今までどおり `.stale` に対応させるので、やり直しは探し直し、条件がもう見つからないと分かって
大きく失敗します。

**Unit 3 — 実機なしで変更を確かめる。** `SnapshotStore` を、
[`BajutsuKit/Tests/BajutsuRunnerTests/SnapshotStoreTests.swift`](../../BajutsuKit/Tests/BajutsuRunnerTests/SnapshotStoreTests.swift)
の Swift ユニットテストで覆います。Simulator のいらない runner の実機なしテスト群で動きます。中心と
なる追加は、本項目が防ごうとしている回帰です。変わらない同じスナップショットを2回以上取り直したうえで、
**最初**の取り直しの参照を `.found` に解決します。古い世代の方式が失敗する場合であり、古い方式が2回目の
取り直しでその参照を古くするからです。あわせて、1つのスナップショットの中で内容が一致する2つの要素が
別々の参照に解決することを、出現番号のタイブレークを固定する連れのテストで確かめます。既存のテストは
今の形のまま新しい方式でも通り続け、テスト群はそれに依存します。**別の**要素での取り直しは最初の要素の
参照をなお古くし（本当の消失、`.stale`）、一度も発行していない参照はなお `.notFound` を読み、発行した
参照はなお `h-` の接頭辞を持ち、並行実行テストもなお通ります。内容からの導出は、Unit 2 が保つ
`.stale`／`.notFound` の約束を壊さず、競合状態も再び持ち込まないということです。

**Unit 4 — 文書を実際の動きに合わせる（BE-0113）。** `SnapshotStore` のクラスの説明は、いま
「`refreshSnapshot` のたびに前のスナップショットを置き換え、以前の参照はすべて古くなる」と述べて
います。本項目が変えるまさにその動きなので、内容から導いた参照と、参照がなお古くなる条件（参照の
名指す要素が画面から出たか内容が変わったこと）を述べるものに書き直します。
[`DESIGN.md`](../../DESIGN.md) と [`docs/architecture.md`](../../docs/architecture.md) が runner の
参照の古さの扱いを述べているなら、説明が実際の動きに合うように更新します。

既知のフォローアップが2つあり、あえて範囲の外に置きます。1つは、これまでに発行した参照の集合が runner
プロセスの寿命のあいだ増え続けることです。とても長い run では、参照の文字列が単調に積み上がります。
増え方は小さく（これまでに見たそれぞれの要素につき短い文字列が1つ）、run はプロセスの寿命に閉じて
いますが、計測で問題だと分かれば、上限——直近のいくつかのスナップショット分の参照だけを保つ——を
あとから足せます。もう1つは、内容が一致する要素への出現番号の割り当てが、内容の同じ要素の問い合わせ
順がスナップショットをまたいで安定していることを前提とする点です。見た目が同じ要素の順がスナップ
ショットのあいだで入れ替わる画面では、参照が振り直されることがありますが、そのような画面はまれで、
その要素は条件からも見分けられないので、この隙間を塞ぐのは、実際のシナリオが必要とするまで見送ります。

## 検討した代替案

**`GET /elements` をやり直しの対象から外し、余分な読み取りを送らせない。** 読み取りを BE-0207 の
通信のやり直しの仕組みから外せば、*動機* の具体的なきっかけは消えます。しかし、その代償として、
BE-0207 が塞ぐために入れた一過性のブレの不安定さが再び開きます。読み取りに本物の一瞬の通信のブレが
起こると、ステップはまた即座に失敗します。やり直しは、余分なスナップショットの唯一のきっかけとも
限りません。将来 runner が別の理由で情報を取り直すようになれば、不安定さは再び戻ります。つまり、
やり直しを外すのは原因ではなく1つの症状に対処するだけです。内容から導いた参照は、余分なスナップ
ショットがどこから来ても無害にします。

**BE-0289 のやり直しだけに頼る。** *動機* が、やり直しがこの場合を回復できない理由を示しています。
やり直しは、新しい問い合わせに映る画面の変化を前提としますが、ここでは画面が変わっておらず、問い
合わせ直しはそれ自体が同じ送り直しのタイミングのずれにさらされるので、上限まで数えた試行のどれもが
同じ理由のなさで `stale` になりえます。2つの項目は代わりではなく補完し合います。BE-0289 はクライアント
側で本物のタイミングのずれを許し、本項目は作り出されたタイミングのずれを源で取り除きます。

**最新だけでなく、直近の複数の世代の参照を有効に保つ。** 直近のいくつかの世代の参照を生きたものと
して扱えば、方式を変えずに割り込みを覆い隠せます。しかし、それは恣意的な窓を広げるだけで、食い違いを
直しません。参照は、その要素が画面にあるあいだちょうど有効であるべきで、これは内容についての言明で
あって、いくつのスナップショットが経ったかについての言明ではありません。内容から参照を決めることは、
その不変条件を直接述べ、遅い runner が追い越しうる調整用の数値を必要としません。

**要素の frame を参照のキーに含める。** 位置もキーにすれば、内容が一致するが位置の違う2つの要素に、
出現番号なしで別々の参照を割り当てられます。しかし、本項目が閉じるまさにその race を開き直します。
遷移のあとに画面が落ち着きかけるあいだ何度も情報を取り直し、割り込む2つの読み取りのあいだで frame が
ずれると、変わっていない1つの要素に2つの参照を発行するので、余分な読み取りが、本項目の狙う窓で再び
参照を古くします。出現番号のタイブレークは、1つのスナップショットの中で内容の一致する要素をすでに
区別します。また frame は操作の精度を上げません。runner は記録された frame ではなく `backingElement`
を通じて操作するからです。frame は、設計が避けようとする代償を払うだけで、設計が必要とするものを何も
生みません。
