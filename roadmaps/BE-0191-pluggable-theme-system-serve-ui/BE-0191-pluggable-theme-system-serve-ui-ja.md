## はじめに

serve Web UI に、第一級の**プラガブルなテーマシステム**を導入します。ここでのテーマとは、二つのものを束ねた名前付きのバンドルです。一つは見た目を決めるビジュアルデザイントークンの集合（色と、背景・境界線・アクセントのパレット）、もう一つは画面遷移の定義（ビュー・モーダル・ペイン・実行ステップの進行がどうアニメーションするか）です。テーマは `<html>` の `[data-theme="<name>"]` で選択します。要点は、operator が**自作のテーマをドロップインできる**ことです。今日シナリオや設定ファイルを置くのとまったく同じ要領で、ソースには一切手を入れずに追加できます。組み込みのダーク・ライトの対（`midnight` / `daylight`）は、開かれたレジストリの最初の二要素にすぎなくなります。テーマは **UI の中で作ることもできます**。ライブプレビュー付きのガイド付きエディタで編集し、ローカルに保存するか、serve インスタンスへアップロードします。ドロップインのファイルは作成経路の一つであって、唯一の経路ではありません。

この項目の対象は、[`bajutsu serve`](../../README.md) が配信するライブのシングルページアプリケーション（SPA、`bajutsu/templates/serve.html.j2`）に限ります。独立したレポート・カバレッジ・統計・クロールの各ページは、別系統の古い `prefers-color-scheme` 機構を使っており、ここでは対象外です（「検討した代替案」を参照）。

## 動機

serve UI にはすでに動作するダーク・ライト機構がありますが、これは専用のロードマップ項目を持たないまま場当たり的に育ったもので、見た目はちょうど二種類だという前提をハードコードしています。

- **見た目を差し替えられない。** テーマは `bajutsu/templates/serve.themes.css` に、`[data-theme]` で選ばれる CSS カスタムプロパティの固定 2 ブロックとして存在し、ヘッダーのコントロールはダーク・ライトの**二択トグル**です。このファイル自身の冒頭コメントが制約をはっきり書いています。「3 つ目のテーマを足すなら、この二択スイッチをもっと広いピッカーに置き換えることになる」。ユーザーが自分の見た目を足す手段はありません。発見の仕組みも、ピッカーも、設定フックもないのです。
- **トークン契約が不完全で、テーマが「すべてを定義している」と信頼できない。** `serve.css` は、レジストリが定義しないカスタムプロパティ（`--accent`・`--muted`・`--bad`）を参照しており、一方でレジストリが定義しているのは `--acc`・`--mut`・`--ng` です。これらの規則は実際に使われている UI（Author タブのステップ一覧、enrich の採用・却下、カバレッジのサブヒント）を支えており、現状では意図したテーマ色ではなく未設定・継承色で描画されています。すべての色が定義済みトークンになるまで、ドロップインのテーマは完全にはならず、フォールバックも当てにできません。
- **トランジションはそもそも見た目の一部になっていない。** ビュー切り替えは即時の `replaceChildren` で起こり、ビュー・モーダル・レスポンシブなペイン再構成（BE-0072）・実行ステップの進行について、テーマ化できる enter/leave アニメーションという概念がありません。動き（モーション）は UI のアイデンティティの大きな部分ですが、今日それは固定で、どのテーマからも見えないままです。
- **ファイルシステムに触れずにテーマを作る手段がない。** テーマがドロップインのファイルになったとしても、一つ作るにはディスク上の生の CSS を編集し、正確なトークン名を覚えている必要があります。アクセント色を少し変えたい、ビューの遷移を少し遅くしたい、という程度のユーザーには、製品内の経路がありません。トークン契約はコメントに書かれているだけで、編集して変化を確かめられる対象としては露出していないのです。

プラガブルなテーマシステムは、見た目と操作感を**バックエンド非依存のインターフェース**に変えます。これはドライバ層が拠って立つのと同じ設計原則です。ツールがトークンとトランジションの*契約*を定め、テーマはその*実装*として自由に差し替えられます。アプリごとの違い（ある組織のブランドの見た目）は、コアではなくドロップインされたテーマと設定に置かれます（プライム原則 3）。

これらは決定論的な経路には一切触れません。テーマ化は純粋な表示層です。Tier-2 の `run` / CI の判定に入り込むことはなく（プライム原則 1）、固定の `sleep` も持ち込みません（プライム原則 2）。アニメーションは CSS が駆動し、`prefers-reduced-motion` の下では無効になるので、ドッグフードしている serve UI の回帰テスト網は条件待ちベースのまま、決定論を保ちます。

## 詳細設計

作業は MECE に 6 つの単位へ分解します。単位 1〜3 がプラガブルな*見た目*のテーマを実現し、単位 4 が差し替え可能な*トランジション*層を足し、単位 5 が変更を決定論的かつドッグフードと矛盾しない状態に保ち、単位 6 が UI 内のエディタとアップロードを足して、ファイルシステムに触れずにテーマを作れるようにします。

### 1. デザイントークン契約の完成（前提作業）

`serve.css` のすべての色・サーフェススタイルを定義済みカスタムプロパティに解決させ、テーマを完全で自己完結した契約にします。これにより、部分的なテーマも予測どおりにフォールバックします。

- 命名のずれを解消します。正規の集合を一つに定め、未定義変数を参照する規則がなくなるように、すべての消費側を更新します（今日 `--accent` / `--muted` / `--bad` は何にも解決しません）。レジストリ側で不足名をエイリアスするか、`serve.css` の消費側を書き換えるか、どちらか一方向に統一して適用します。
- `serve.css` の生のカラーリテラル（約 15 箇所、多くは色付き背景上の `#fff`）を洗い出し、それぞれをトークン経由（`--on-acc`・`--on-run` など）に通して、色がシステムを迂回しないようにします。
- トークン契約を `serve.themes.css` にインラインコメントで明文化します（必要なトークン名と、各々の意味）。このコメントこそがテーマ作成の API 面です。
- フォールバックの規則：テーマが省いたトークンは `:root` の既定ブロック（`midnight`）から継承します。これにより、不完全なドロップインテーマは壊れるのではなく穏やかに劣化します。

### 2. プラガブルなテーマの発見と登録

固定の 2 ブロックのレジストリを、serve 起動時に発見される開かれた集合に変えます。

- **テーマの供給元：** テーマ用ディレクトリを用意し、`bajutsu serve` の新しい `--themes <dir>` CLI フラグから解決します（既存の `--scenarios` / `--runs` / `--baselines` のディレクトリフラグに倣います）。初期選択のために、設定に任意の `ui.default_theme` フィールドを持たせます。組み込みテーマは今日どおりリポジトリに同梱し、発見されたテーマは組み込みを置き換えるのではなく拡張します。
- **テーマの形（プラグ）：** テーマは**宣言的なものだけ**です。トークン値の CSS ブロック（および単位 4 のトランジション定義）と、小さなマニフェスト（表示名、`dark` か `light` かの種別）からなります。テーマに**任意の JavaScript は持たせません**。これにより信頼境界をシナリオや設定と同じ水準（operator 自身がファイルを置く）に保ちつつ、攻撃面を CSS に限定します（安全性のトレードオフは「検討した代替案」を参照）。
- **登録：** serve は起動時にディレクトリを一度走査し、発見した CSS をインライン化するテーマスタイルシートに連結し、ピッカーが選択肢を描画できるようテーマのマニフェストをクライアントへ渡します。テーマはプロセスの生存期間中は静的なので、`bajutsu/serve/handler.py` の `_asset` / `_index_html` にかかる既存の `functools.lru_cache` はそのまま有効で、発見された集合はキャッシュされた描画結果に畳み込まれます（テーマのライブリロードは対象外です）。

### 3. テーマピッカー UI

レジストリのコメントが予告したとおり、ヘッダーの二択トグルをピッカーに置き換えます。

- 組み込みと発見済みのテーマを列挙するヘッダーのドロップダウン（メニュー）にし、ダーク・ライトの種別で分類・ラベル付けして、月・太陽のチェックボックスを置き換えます。
- 明示的な選択を `localStorage['bajutsu-theme']`（キーは変更なし）に永続化し、間違ったテーマのちらつきを避けるために既存のインライン `<script>` で初回描画前にシードします。OS のスキーム変更は、ユーザーが手動選択をしていない間に限り、引き続きライブで反映されます。
- BE-0058 の `data-testid` 規約に従い（例：`nav.theme-picker`）、ヘッダーのコントロール群に対して曖昧さのないセレクタを用います。
- **配置：** ピッカーは `<header>`（今日のトグルと同じ）かモーダルに置きます。`main#view-*` 要素の直接の子には**しません**。デスクトップのタイラーの `rebuild()` がそれらの要素に対して `replaceChildren` を呼ぶため、タイル再構成のたびにピッカーが消えてしまうからです。

### 4. テーマが定義する差し替え可能なトランジション

議論で挙がった 4 つの面すべてにわたって、画面遷移をテーマ化できる見た目の一部にします。4 つの面とは、**ビュー切り替え、モーダルの enter/leave、レスポンシブなペイン再構成（BE-0072）、実行ステップの進行**です。

- **関心の分離：** JavaScript は各遷移点で*意味的な状態クラス*（例：`.is-entering` / `.is-leaving` / `.view-switching`）を当てるだけにし、**その状態がどう見えるかはテーマの CSS が決めます**。`--transition-*` トークン（duration / easing）とキーフレームを介して定義します。これにより JS はテーマ非依存のまま、アニメーション自体が完全にプラガブルになります。トランジション規則を持たないテーマは即時に描画します（今日の挙動）。
- **タイラーの改修（主要なコスト）：** ビュー切り替えは今、`replaceChildren` で古いサブツリーを即座に破棄するため、leave アニメーションを見せる要素が残りません。leave アニメーションを可能にするには、遷移が続くあいだ退場するノードを保持する必要があります。`rebuild()` に二重バッファのステップを入れるか、より軽い第一版として入場するビューだけをアニメーションさせます。ここが変更のうち最もリスクの高い部分なので、後述の reduced-motion のガードの背後に置いて、今日の即時挙動へ劣化できるようにします。
- **モーション低減：** `prefers-reduced-motion: reduce` の下では、有効なテーマにかかわらずすべてのトランジションが即時に畳まれます。これはアクセシビリティ要件であり、同時に単位 5 の決定論のためのてこでもあります。

### 5. 決定論とドッグフードとの整合

- serve UI 自体がドッグフードの対象です（BE-0058、`demos/serve-ui/`）。単位 4 の reduced-motion による畳み込みが、追加したモーションが競合状態や固定 `sleep` の必要を決して生まないことの保証になります。モーションを切れば遷移は即時になり、既存の条件待ちのアサーションはそのまま成り立ちます。
- `demos/serve-ui/scenarios/theme.yaml` は現在 `nav.theme-toggle` をタップして `checked` 状態の反転をアサートしています。二択からピッカーへの変更（単位 3）はこの前提を崩すので、シナリオを同時に更新し、ピッカーを操作して `[data-theme]` をアサートするようにします。
- この経路のどこにも LLM は入りません（プライム原則 1）。テーマの選択と描画は、すべて決定論的なクライアント・サーバーのコードです。

### 6. UI 内のテーマエディタ、ライブプレビュー、アップロード / エクスポート

ファイルシステムに触れずにテーマを作成・共有できる製品内の経路を足します。単位 1 のトークン契約の上に直接構築します。

- **契約から導かれるエディタ：** 明文化した必要トークンから生成するフォームにします。色トークンごとにカラー入力、`--transition-*` トークンごとに duration / easing / 効果の入力を持ちます。契約こそが API なので（単位 1）、エディタのフィールドは手で保守するのではなく契約から導かれます。
- **クライアント側のライブプレビュー：** 編集を適用すると、稼働中の SPA の有効な `[data-theme]` のカスタムプロパティを上書きする `<style>` ブロックを注入し、サーバー往復なしで変化を即座に見せます。配置は単位 3 に倣い、モーダル（`#settingsmodal` と同様）にして、タイラーが消してしまう `main#view-*` 要素の直接の子には決してしません。
- **永続化の 2 層：**
  - *ローカル下書き* ── 編集したテーマを `localStorage` に `custom` テーマとして保存し、ピッカー（単位 3）に即座に出します。ブラウザ内のみで、サーバー面も新たな信頼境界も増やさず、見た目を試行錯誤するのに摩擦のない経路です。
  - *serve インスタンスへのアップロード* ── テーマを `--themes` ディレクトリ（単位 2）へ書き出し、発見可能なドロップインにして、そのインスタンス内でセッションをまたいで共有します。既存のアップロードの継ぎ目（BE-0073）を再利用します。ここがテーマの `lru_cache` を無効化しなければならない**唯一**の箇所です。その書き込みに限った、範囲の限られた明示的な再走査で、単位 2 が除外する汎用のライブリロードとは別物です。
- **エクスポート / インポートのラウンドトリップ：** テーマは宣言的な CSS＋マニフェストなので、「このテーマをダウンロード」はドロップイン機構とラウンドトリップするファイルを生み（UI で編集 → エクスポート → リポジトリにコミット、または共有）、「インポート」はそのようなファイルを取り込みます。これで 2 つの作成経路が閉じます。
- **信頼と決定論は不変：** アップロードするテーマも、JavaScript を持たない宣言的な CSS＋マニフェストのままです（単位 2）。アップロードするのは自分の localhost serve を操作する認証済みの operator なので、アップロードされたテーマはドロップインのファイルと同じ信頼を帯びます。残る CSS の面（例：テーマが外部 URL を参照する）は、その信頼水準で許容し、サンドボックス化するのではなく注記します。LLM は入らず、`run` 経路への影響もありません。

## 検討した代替案

- **ビルドステップや CSS フレームワークの導入（Sass、Tailwind、バンドラ）。** 採用しません。serve UI は意図的にビルド不要の素の JS/CSS で、ディスクから直接読み込みます。CSS カスタムプロパティがすでにネイティブに表現できる機能のために、その制約を破ることになります。
- **ファイルを介さず、API や `localStorage` 経由で任意の CSS を注入する。** 採用しません。アセットキャッシュを複雑にするうえ、より重要な点として信頼境界を動かします。ドロップインされたファイルは operator のシナリオや設定と同じ信頼水準にありますが、リモートや貼り付けの CSS 経路は、より広い面からの注入を招きます。ドロップインのテーマファイルが信頼境界です。
- **テーマに任意の JavaScript を持たせる**（完全にプログラム可能なテーマ）。同じ理由で採用しません。宣言的な CSS とマニフェストからなるテーマは、見た目と動きを表現するのに十分な表現力があり、しかも攻撃面を CSS に限れます。
- **UI 内エディタをクライアント専用（`localStorage`）にし、サーバーへのアップロードを持たない。** それだけを全体像とするのは採用しません。ブラウザ内のカスタムテーマは試行錯誤には優れますが、一つのブラウザに閉じ込められます。両方の層を残します。摩擦のない編集のためのローカル下書きと、serve インスタンスをまたいだ永続化・共有のための任意のアップロードです。
- **ホスト型のテーママーケットプレイス / リモートのテーマレジストリ。** 採用しません。serve インスタンスはローカルのドロップインとアップロードされたテーマを発見しますが、ネットワーク越しにテーマを取得はしません。テーマの共有は、中央サービスではなくエクスポート / インポートのファイルのラウンドトリップ（およびバージョン管理）に乗せます。
- **設定のみ、またはピッカーのみ（両方は持たない）。** 薄い 2 層を持つ方を採ります。設定（`ui.default_theme` と `--themes`）がデプロイ・ターゲットごとの既定を決め、クライアントのピッカーがユーザーにローカルな上書きを許します。これは、OS 既定のあとに手動上書きが効く現在のトグルの挙動に合っています。
- **独立したレポート・カバレッジ・統計・クロールの各ページを同じテーマシステムに統合する。** 今回は採用せず先送りします。この項目の対象は SPA に限ります。これらのページは別系統の古い `prefers-color-scheme` 機構を、異なる変数セット（`--ink` など）とトグルなしで使っており、統合には価値があるものの、別の、より大きな変更になります。関連するフォローアップとして残します。

- [x] 6. UI 内のテーマエディタ、ライブプレビュー、アップロード / エクスポート（契約から導くフォーム、クライアント側ライブプレビュー、ローカル下書きと専用の `POST /api/theme` エンドポイントによるアップロードの永続化 2 層、エクスポート / インポートのラウンドトリップ）。

### ログ

- 単位 1 — デザイントークン契約を完成させました。`--accent`/`--muted`/`--bad`/`--border` の名前のずれを、
  正規のトークン `--acc`/`--mut`/`--ng`/`--line` へ寄せて解消し、`serve.css` に残っていた生の hex リテラルを
  すべてトークン経由に置き換えました（`--canvas`、`--on-ok`、`--on-ng`、`--on-mut`、`--on-rung`、および
  `--rung-*` の凡例トークンを追加し、未定義だった `--mono` グローバルも定義しました）。トークン契約を
  `serve.themes.css` にインラインで明文化し、`:root`（midnight）をフォールバックの供給元にしたので、
  一部のトークンを欠くテーマも破綻せずに縮退します。この不変条件を守るために
  `tests/serve/test_theme_tokens.py` を追加しました。（[#826](https://github.com/bajutsu-e2e/bajutsu/pull/826)）
- 単位 2 — プラガブルなテーマの発見と登録。`bajutsu/serve/themes.py` を追加しました（宣言的なテーマとは、
  トークンを並べた `*.css` ブロックに、先頭の `/* bajutsu-theme name:/kind: */` マニフェストを添えたもので、
  JavaScript を含みません）。`bajutsu serve --themes <dir>` フラグを追加し、起動時に一度だけ走査して、
  キャッシュ済みの `_index_html` 描画へ畳み込みます。serve 専用の `ui.default_theme` キーは、`orgs:` と同じく
  コアの `Config` からは落とし、`themes.read_default_theme` で読みます。登録済みのマニフェスト（組み込み＋発見分）と
  設定された既定は、ピッカー（単位 3）向けに `window.__bajutsuThemes` / `window.__bajutsuDefaultTheme` として
  クライアントへ渡します。初回描画前のシードは、OS の配色より先に設定された既定を優先するようにしました。
  テストは `tests/serve/test_themes.py`、`tests/serve/test_http_themes.py`、およびコアの `ui:` 除去テストです。（[#837](https://github.com/bajutsu-e2e/bajutsu/pull/837)）
- 単位 3 — テーマピッカー UI。二択のヘッダートグルを、登録済みマニフェストからサーバー側でレンダリングする
  ネイティブの `<select>`（`data-testid="nav.theme-picker"`）へ置き換えました。ダークとライトの種別で
  グループ分けし、各オプションの値はテーマ id です。`serve.core.js` は適用中のテーマからウィジェットの
  選択状態をシードし、明示的な選択を `localStorage['bajutsu-theme']` に永続化し、明示的な選択が無いあいだ
  だけ OS の配色に追従します。OS の配色は、組み込みの 2 種だけでなくレジストリの種別に照らして解決します。
  初回描画前のシードは単位 2 の時点で既定優先だったため変更していません。ネイティブのコントロールなので
  キーボードとスクリーンリーダーに対応したまま、タイラー（BE-0072）と重なりません。ピッカーがテーマごとに
  1 つのオプションを描画し旧トグルが消えたことを検証するケースを `tests/serve/test_http_themes.py` に追加し、
  `docs/web-ui.md`（と日本語版）を更新し、`demos/serve-ui/scenarios/theme.yaml` のドッグフードも歩調を
  合わせて書き換えました。ドッグフードはピッカーの初期状態、すなわちレンダリングされ OS の配色を反映して
  いることだけを検証します。選択の切り替えは駆動しません。ネイティブの `<select>` を決定論的に切り替える
  アクションが DSL に無いためです（`type` の typeahead はあるヘッドレス Chromium では動きましたが、CI では
  何も起きませんでした）。決定論的な切り替えには専用の select-option ドライバアクションが必要で、これは
  単位 5 で入ります。（[#855](https://github.com/bajutsu-e2e/bajutsu/pull/855)）
- 単位 5 — 決定論とドッグフードとの整合。`selectOption` という DSL アクション（`SelectOption` モデルと
  `Step.select_option`。`orchestrator/actions/handlers/gestures.py` でディスパッチ）と、それが駆動する
  `Driver.select_option(sel, option)` プロトコルメソッドを追加しました。Playwright バックエンドは、共有の
  `resolve_unique` コアで `<select>` を一意に解決したうえで、解決した座標（`elementFromPoint`。クリックと
  同じ座標なので、マッチングは決定論コアに残ります）でその要素を特定し、option を value で設定して `change`
  を発火します。fake ドライバは解決して記録し、idb / adb / xcuitest は `UnsupportedAction` を送出します
  （`<select>` にネイティブの対応物がないため）。これにより `demos/serve-ui/scenarios/theme.yaml` の
  ドッグフードは、ピッカーを両方向に切り替えてその変化を検証できるようになり、単位 3 が先送りにした穴を
  塞ぎました。reduced-motion による条件待ちの保証は、現時点ではトランジションが存在しないため自明に成り立って
  おり、`prefers-reduced-motion` での畳み込みは単位 4 のモーションと一緒に入ります。テストは
  `tests/test_select_option.py` と、`tests/test_playwright.py` / `test_idb.py` / `test_adb.py` /
  `test_xcuitest.py` の `select_option` ケースです。ドキュメントは `docs/scenarios.md` と
  `docs/dsl-grammar.md`（および日本語版）を更新しました。（[#859](https://github.com/bajutsu-e2e/bajutsu/pull/859)）
- 単位 4 — テーマが定義する差し替え可能なトランジション。トークン契約を `--motion-*` 群（ビュー・モーダル・
  ペインの時間、共有のイージング、開始と終了のアニメーション名トークン）で拡張し、`:root`/midnight の
  フォールバックブロックに定義したうえで `serve.themes.css` に明文化しました。クライアントは意味的な状態
  クラスを付けるだけです。`showView` は表示側のビューに開始アニメーションを再生し、`MutationObserver` は
  モーダルが表示されるたびにその開始を再生し（どの表示側も変更不要）、新しい `closeModal` ヘルパーが終了を
  アニメーションしてから隠します。スマートフォン幅ではペイン切り替えで全幅になったペインをフェードさせ、
  タイラーの `rebuild()`（`serve.author.js`）は二重バッファ化しました。`render()` が生きたパネルノードを
  新しいルートへ移すため、離脱するルートを `data-testid` を剥がした深いクローン（見た目だけのゴースト）として
  ビューの上に絶対配置で重ね、再構築したルートが入ってくるあいだに離脱させます。こうすることで、どのセレクタも
  一時的に曖昧になりません。すべての動きは `prefers-reduced-motion: reduce` で即座の切り替えに畳み込まれ、
  Playwright バックエンドはすべてのコンテキストを `reduced_motion="reduce"` で開くようになりました。これが
  決定性のレバー（単位 5）です。ドッグフードと CI では遷移がすべて即座になり、ゴーストの二重バッファは元の
  即時 `replaceChildren` に縮退し、条件待ちがアニメーションと競合しません。テストは
  `tests/serve/test_theme_tokens.py` の `--motion-*` 契約と reduced-motion 畳み込み、`tests/test_playwright.py`
  の `reduced_motion` コンテキスト検証です。ドキュメントは `docs/web-ui.md`（および日本語版）を更新しました。
  （[#881](https://github.com/bajutsu-e2e/bajutsu/pull/881)）
- 単位 6（全 2 回のうち 1 回目） — UI 内テーマエディタ、ライブプレビュー、クライアント側のエクスポートと
  インポート。デザイントークン契約を JSON として公開しました。`themes.parse_theme_tokens` が
  `serve.themes.css` からすべての `--*` 宣言を抽出して分類し（色か `--motion-*` かを判定し、時間・
  イージング・キーフレームを推測します）、新しい `serve/operations/theme_editor.py` の
  `get_theme_contract` が各トークンの既定値を `:root`/midnight ブロックから埋め、`GET /api/themecontract`
  で配信します。クライアント（`serve.core.js`）はこの契約からエディタのフォームを生成します。色トークンごとに
  色入力を、モーショントークンごとにテキスト入力を、タイラーが壊さない `#thememodal` の中に並べます。編集すると
  `<style id="theme-editor-preview">` ブロックを注入するので、サーバとのやり取りなしに変更をその場でプレビュー
  できます。ローカルドラフトの保存（`localStorage`）、CSS のエクスポート（ドロップイン形式と往復できます）、
  インポートがクライアント側を締めくくります。テストは `tests/serve/test_theme_contract_parse.py`（パーサ）と
  `tests/serve/test_theme_editor.py`（契約エンドポイントが実際の同梱 CSS を読んで既定値を埋めること）です。
  2 回目に残すのは、ローカルドラフトをピッカーに出すこと、`--themes` へのサーバアップロード（BE-0073 の継ぎ目）と
  それに伴う限定的な `lru_cache` 無効化、そしてドッグフードのシナリオです。
  （[#883](https://github.com/bajutsu-e2e/bajutsu/pull/883)）
- 単位 6（全 2 回のうち 2 回目） — ローカルドラフトのピッカー表示とサーバアップロードを足し、この単位を
  完了しました。クライアント（`serve.core.js`）は、保存したローカルドラフトをヘッダーのピッカーに `custom` として
  出すようになりました。`surfaceCustomDraft` がテーマ一覧に登録し、`custom` を選んだときだけ描画される
  `<style id="theme-custom">[data-theme="custom"]{…}</style>` を注入し（省いたトークンは `:root` にフォール
  バックします）、その `<option>` を追加・更新します。「Save to Local Draft」を押すとこのテーマへ切り替わり、
  ドラフト・エクスポート・アップロードに名前を付けられるよう、エディタに名前フィールドを足しました。
  サーバアップロードは、BE-0073 の zip 継ぎ目ではなく専用の `POST /api/theme`（`{name, kind, tokens}`）として
  実装しました。テーマは小さな宣言的ファイル 1 つなので、書き込みは apikey / config の書き込みと同じ形にしています。
  `theme_editor.upload_theme` は `--themes` を要求し、名前を `[data-theme]` の id にスラグ化し（セレクタが
  ファイル名の語幹と必ず一致します）、各トークンの名前と値が規則から抜け出さないよう検査し、組み込みテーマとの
  id 衝突を拒否し、正規のマニフェストと規則を組み立て、`_index_html` の `lru_cache` を無効化します（単位 2 が
  除外したライブ再読み込みを、ここだけ解きます）。契約の GET とアップロードの POST は FastAPI バックエンド
  （`server/app.py`）にも配線し、エディタが stdlib ハンドラでしか動かなかった 1 回目の抜けを塞ぎました。
  アップロードボタンは隠した状態で出荷し、`--themes` を設定したときだけ表示します（サーバが設定する
  `window.__bajutsuThemesWritable` フラグで判定します）。テーマモーダルは、フォームを高さ上限つきの領域で
  スクロールさせ、操作ボタンをフッターとして固定するようにしたので、Save / Upload が画面外に隠れません。テストは
  `tests/serve/test_theme_upload.py`（op と stdlib ルートとキャッシュ無効化）、`test_server_app.py`（FastAPI の
  両ルート）、`demos/serve-ui/scenarios/theme-editor.yaml` のドッグフード（ローカルドラフト → `custom` の
  ピッカー表示、`--themes` なしでアップロードボタンが出ないこと）です。ドキュメントは `docs/web-ui.md`
  （および日本語版）を更新しました。（[#900](https://github.com/bajutsu-e2e/bajutsu/pull/900)）
