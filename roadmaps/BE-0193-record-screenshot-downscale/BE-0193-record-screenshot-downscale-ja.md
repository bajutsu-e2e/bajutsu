## はじめに

record が実際にスクリーンショットをモデルへ送るとき、適正サイズで送ります。現在のループは
フル解像度のデバイススクリーンショットを撮り、生の PNG バイトをそのままモデルへ渡していて、作成
経路にもアダプター経路にもリサイズの工程がありません。画像のトークンコストは**ピクセル寸法**に
比例するため、フル解像度の Simulator スクリーンショット（長辺が 2000px を優に超えることも多い）は、
モデルが使えるより遥かに多くのピクセル分を課金されます。本項目は、エンコード前にクライアント側で
スクリーンショットを、モデルがそれ以上得るもののない解像度を上限として長辺を抑えて縮小します。
これにより画像あたりのトークンコストが決定的かつプロバイダに依存しなくなり、エージェントが実際に
読み取る細部は失いません。

## 動機

`_screenshot_bytes`（`bajutsu/record.py`）は `xcrun simctl io <udid> screenshot`
（`bajutsu/drivers/idb.py` の `screenshot_cmd`）で画面を撮り、これはサイズ指定のないフル解像度の
PNG を書き出します。その PNG はリクエストへそのまま運ばれます。`ImagePart(data=screenshot)`
（`bajutsu/claude_agent.py`）が Anthropic アダプターで base64 にそのままエンコードされ
（`bajutsu/ai/anthropic.py`）、`media_type` は既定で `image/png` です（`bajutsu/ai/base.py`）。
リポジトリ全体を検索しても、AI・record・ドライバーのいずれの経路にも画像のリサイズや縮小、圧縮の
工程は**ありません**。

画像の課金には、これを無駄にする 2 つの性質があります。

- **トークンコストはファイルサイズではなくピクセル寸法に従います。** Anthropic API では、画像の
  トークンコストは幅 × 高さの関数であり、ある上限（おおむね長辺 1568px、約 1.15 メガピクセル）を
  超えるとモデルは追加の利益を得ません。フル解像度の Simulator スクリーンショットはたいていそれを
  大きく超えるので、余分なピクセルは課金されたうえで捨てられます。
- **縮小は現状では暗黙的で、プロバイダに依存します。** Anthropic API は大きすぎる画像をサーバー側で
  縮小するので、今日のコストは*あたかも*上限が効いているかのようになります。しかしそれはプロバイダ
  の挙動であって、こちらのものではありません。別のバックエンド（たとえば Amazon Bedrock、BE-0053）
  や将来のプロバイダでは保証が異なる場合があり、Anthropic API でも大きすぎるバイト列は毎ターン送信
  されます。クライアント側で縮小すれば、解像度、ひいてはトークンコストを Bajutsu が制御し、プロバイダ
  をまたいで一定に保てます。

これは小さく自己完結した変更で、2 つの姉妹提案と組み合わさります。姉妹提案
`record-vision-on-demand` は画像を送る*頻度*を減らし、本項目は送る各画像の*大きさ*を抑えます。姉妹
提案 `record-turn-payload-diet` はターンのテキスト側を痩せさせます。また
[BE-0178](../BE-0178-record-multi-action-turn/BE-0178-record-multi-action-turn-ja.md)（ターン数を減らす）
とも独立しています。

### 精度が後退しない理由

上限は、モデルが内部で縮小する解像度と同じ値に置き（それより下げません）、エージェントが読み取る
ピクセルは今日と変わりません。クライアントは、プロバイダがどのみち行う縮小を代わりに行うだけで、
それより細かくすることはありません。操作対象の特定にエージェントが権威として用いる要素ツリー
（`id`/`label`）はそのままです。`tap_point` は**正規化された [0,1] 座標**を使い
（`bajutsu/claude_agent.py`）、ピクセル寸法に依存しないため、縮小した画像でもまったく同じ位置を
タップします。したがって、適正サイズの画像がフル解像度に比べてエージェントの判断を変える画面は
ありません。

## 詳細設計

### 1. モデルへ届く前に適用する共有の縮小ヘルパー

撮影した PNG バイトを受け取り、長辺が定数 `MAX_IMAGE_LONG_EDGE`（既定はプロバイダの実用上限、
約 1568px）以下になるバイト列を返す小さなヘルパーを追加します。縮小のみで、小さい画像を拡大せず、
すでに小さい画像はそのまま通します。これは作成側（`Observation`/`ImagePart` を組み立てる場所）で
適用し、ベンダー中立なアダプターの内側では適用しません。こうすることで、どのプロバイダも同じ適正
サイズの画像を受け取り、アダプターは純粋な変換器のままです（BE-0104）。アスペクト比は保ちます。

### 2. 形式は PNG のまま（意図的に変えない）

画像は PNG のままにします。Anthropic はバイトサイズではなくピクセル寸法で課金するため、JPEG へ
再エンコードしてもトークン数は**減りません**。アップロードが小さくなるだけで、エージェントが読み取る
小さな文字や細い操作対象をぼかす恐れがあります。トークンのレバーは寸法の縮小がすべてであり、判読性を
守るため形式は触りません。（将来の読み手がトークン削減を期待して JPEG へ「最適化」しないよう、
ここで明示します。）

### 3. リサイズをどこで行うか（バックエンド別）

- **iOS（idb / Simulator）**：撮影は macOS の `simctl` スクリーンショットなので、撮影後の PNG バイト
  に対して縮小を行います。実装は軽量な画像処理を使ってよく、依存関係の論点は*検討した代替案*にあります。
- **Web（Playwright）**：バックエンドが目標サイズで直接撮影できるので、後処理ではなく撮影時に同じ
  `MAX_IMAGE_LONG_EDGE` の上限を適用します。いずれにせよ、作成経路へ渡すバイトのペイロードはすでに
  上限を守っています。

ヘルパーの契約（バイトを入れる → 長辺を抑えたバイトを返す）はバックエンドに依存しません。各バック
エンドがどうバイトを作るかはバックエンドの詳細です。

### 4. 上限はアプリ設定ではなく定数

`MAX_IMAGE_LONG_EDGE` はモジュール定数 1 つで、どのターゲットでも同一です。`targets.<name>` の設定
ではありません。適切な解像度は*モデル*の画像処理の性質であって、テスト対象アプリの性質ではないため、
アプリ固有にするのは、利益なくアプリ非依存のプライムディレクティブに反します。（将来プロバイダの実用
上限が変わることがあれば、全体設定 `ai.*` のノブを後から足せますが、ここでは対象外です。）

### 決定性・プライムディレクティブ適合・ゲート

厳密に**Tier 1（record のみ）**です。`run` や CI にモデル呼び出しは加えません（プライムディレク
ティブ 1）。リサイズは `sleep` を含まない決定的なピクセル操作で、セレクター解決には触れません
（プライムディレクティブ 2）。上限はアプリ別の分岐を持たない全体定数です（プライムディレクティブ 3）。
記録されるシナリオ成果物は変わりません。スクリーンショットは作成モデルへの入力であって、出力される
YAML の一部ではないため、`run`、`codegen`、レポート、それらのテストは影響を受けません。

### テスト戦略（Simulator 不要、Linux の `make check` ゲートに収まる）

- **縮小ヘルパーの単体テスト**：大きすぎる画像がアスペクト比を保って長辺 `MAX_IMAGE_LONG_EDGE` に
  縮小される。すでに小さい画像はバイト単位でそのまま通る（拡大しない）。正方形と横長・縦長のそれぞれ
  で正しい辺が上限に収まる。
- **作成経路での結合**：`ImagePart` に届くバイトが上限を守る（合成した大きすぎる PNG を偽ドライバー
  経由で確認。Simulator 不要）。
- **tap_point の不変性**：縮小画像のターンで記録された `tapPoint` が、フル解像度のターンのものと
  バイト単位で一致する（座標は正規化）。精度の主張を守るためのテスト。

### スコープと非目標

**スコープ内**：クライアント側の縮小ヘルパーと `MAX_IMAGE_LONG_EDGE` の上限、iOS と web の作成撮影
経路への組み込み、形式を PNG に保つこと。

**非目標**：そもそも画像を送る*かどうか*の判断（姉妹提案 `record-vision-on-demand`）、要素ツリーの
テキストを痩せさせること（姉妹提案 `record-turn-payload-diet`）、ターン*数*を減らすこと（BE-0178）、
`crawl` やアラートロケーター・triage の視覚経路の画像処理の変更（後から同じヘルパーを採用できますが、
ここでは対象外）、シナリオスキーマや `run`/再生の変更。

## 検討した代替案

- **プロバイダのサーバー側縮小に任せる（何もしない）。** これが今日の挙動で、Anthropic API では
  *コスト*がすでに上限に収まっている理由です。却下：それはプロバイダの保証であってこちらのもので
  はなく、大きすぎるバイトを毎ターン送り続け、Bedrock や将来のバックエンドで一様に成り立つとは
  限りません。解像度を自分で握れば、コストが決定的かつプロバイダ非依存になります。
- **ペイロードを縮めるため JPEG へ再エンコードする。** 却下：トークンコストは寸法基準なので JPEG では
  トークンが減らず、バイトが縮むだけで、エージェントが読むべき小さな文字をぼかす恐れがあります。
  トークンの利益がないのに精度を危険にさらします。
- **モデルの実用上限より下へ縮小する（積極的に縮める）。** 却下：モデルの内部縮小より細かくすると、
  エージェントが実際に読むピクセルを捨て始め、精度をトークンと引き換えにします。本項目の前提の逆です。
- **Simulator スクリーンショットを最初から低い解像度で撮る。** `simctl io screenshot` にはサイズ指定
  がないため、iOS 経路では結局撮影後のリサイズなしには実現できません。web 経路はすでに目標サイズで
  撮影します（詳細設計の第 3 節）。
