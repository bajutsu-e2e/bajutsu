## はじめに

`bajutsu serve` の Web UI を、スマートフォンや幅の狭いタブレットといった小さい画面でも使えるようにします。デスクトップ前提の多ペインレイアウトを、一定の画面幅を境に 1 カラムのタッチ操作向けレイアウトへ組み替え、マウス専用の操作にタッチの経路を与えます。変更はインライン展開した `serve.*` テンプレートの**フロントエンドだけ**に閉じます。サーバのルートもジョブも設定も足さず、決定的なランナーの挙動も一切変えません。`serve` は決定的な CLI を起動するだけの Tier 1 の補助機能のままなので、この提案はプライムディレクティブ（[CLAUDE.md](../../CLAUDE.md)）の内側に完全に収まります。ゲートに LLM は入らず、合否にも触れず、アプリ固有の挙動は設定に残ります。

## 動機

`serve` の UI は、`bajutsu/templates/serve.html.j2` で `<meta name="viewport" content="width=device-width, initial-scale=1">` をすでに宣言している。これはスマートフォンのブラウザに、ページを縮小したデスクトップとしてではなく実際の CSS 幅で描画せよ、と指示するものです。だが、その指示を裏で支えるはずのスタイルシートが約束を守っていません。`bajutsu/templates/serve.css` には **`@media` クエリが 1 つもなく**、すべてのレイアウト規則が広い画面向けの 1 デザインに固定されています。この食い違いは、具体的に 4 つの形で表れます。

1. **中心となるレイアウトがスマートフォンに収まらない。** Replay ビューはパネルを 5 トラックのグリッドで並べる（`main{grid-template-columns:var(--rep-left,340px) var(--gw) var(--rep-mid,340px) var(--gw) 1fr}`）。約 340 px の左右カラムにリサイズ用の溝、さらにレポートのカラムが続く構成です。これはレポートに幅を割り当てる前の時点ですでに 390 px のスマートフォンより広く、カラムは横スクロールを強いるか、読めない幅まで潰れます。crawl のサイドパネルは `flex:0 0 22rem` で幅が固定され、ヘッダー（ブランド、`.toptabs` の行、`margin-left:auto` で右端へ押し出したテーマ切り替え）は横 1 行に並ぶ前提で折り返しません。

2. **操作がすべてマウス専用で、タッチのハンドラが 1 つもない。** この UI は積み重なった 2 つのリサイズ機構とパン可能なグラフを持つが、そのどれもマウスイベント専用に結線されています。カラムの溝は `mousedown` と `e.clientX` でリサイズし（`bajutsu-splits` として保存される）、より高機能なタイリングレイアウト（パネルの `.tile-grip` をドラッグしてペインを分割し入れ替え、木構造を `bajutsu-tiles` として保存する仕組み）は `mousedown` と `elementFromPoint` で駆動し、crawl のグラフは `mousedown`/`clientX` でパンし `wheel` でズームします。`serve.js` には `touchstart` も `pointerdown` も **1 つも**登録されていません。スマートフォンでは、溝もグリップも死んだ操作部品になり（押してドラッグするポインタが存在せず、ドラッグして分割するタイリングは手のひらの画面では意味をなさない高機能）、グラフはパンもピンチズームもまったくできません。

3. **縮小したデスクトップは使えるスマートフォン UI ではない。** 広いレイアウトしか存在しないため、スマートフォンにはデスクトップがビューポートへ押し込まれた姿が表示されます。共存する余地のない左右のペイン、カーソル前提の大きさのタップ対象、ズームとパンでしか届かないコンテンツ。機能はすべてそこにあるのに、親指で届くようには並んでいません。

4. **ホスティングがスマートフォンからのアクセスを現実の入り口にする。** ホスティングの取り組みが `serve` を URL でアクセスできる共有サービスに変えていくにつれ（[BE-0015](../BE-0015-web-ui-public-hosting/BE-0015-web-ui-public-hosting.md) の公開ホスティングと [BE-0016](../BE-0016-web-ui-self-hosting/BE-0016-web-ui-self-hosting.md) のセルフホスティング）、スマートフォンから開く場面は例外ではなくなります。「テストが落ちた」という通知を受け取ったレビュアーは、手元の端末でその run のレポートを開き、結果をひと目で確かめ、必要なら再実行を始めたいものです。いまはそれが横スクロールとの格闘になります。なお、埋め込まれたレポートは*すでに*そこへ半分近づいている（`bajutsu/templates/report.css` は自前の `@media` 規則を持つ）ので、レポートの中身は画面幅に追従するのに、それを囲む `serve` の外枠は追従しません。取り残されているのは外枠のほうです。

狙いは意図して控えめで、範囲も限られています。**機能も内容も同じ UI** を、狭いビューポートでは縮小されたデスクトップではなく、読めてタップできる 1 カラムとして並べ直す、それだけです。

## 詳細設計

### 範囲と制約

変更は `bajutsu/templates/serve.css` に閉じ、唯一本当にインタラクティブな欠落（crawl グラフのタッチパン）のために `bajutsu/templates/serve.js` へ小さく手を入れます。サーバのルートもジョブモデルも設定キーも変えず、UI はこれまでどおり決定的な CLI を起動するだけです。[BE-0011](../BE-0011-local-web-ui-serve/BE-0011-local-web-ui-serve.md) からの 2 つの制約を守ります。レスポンスは CSS/JS をインライン展開した**自己完結の HTML 1 枚**のまま（`/static` ルートもアセットのパイプラインも持たない）であること、そして CSS は**ビルド工程を持たない手書き**のまま（フレームワークを使わない）であることです。

### ブレークポイントの方針

`max-width` のブレークポイントを 1 つ（必要なら 2 つ）導入します。スマートフォン向け（`≤ 640px` 程度）と、タブレット向けの調整が要るなら狭い段（`≤ 900px` 程度）です。狭い段の規則は既存のデスクトップ規則の**後ろ**に重ね、変える必要のある部分だけを上書きします。こうすれば広いレイアウトが手つかずの既定として残り、デスクトップの体験を損なう恐れがありません。正確なピクセル値は小さな未決事項であり、要となる判断は「デスクトップのレイアウトを土台とし、狭い段はその上に足す上書きである」という点にあります。

### 画面ごとの 1 カラム化

スマートフォンのブレークポイントを下回ると、多トラックのグリッドは 1 カラムに畳まれ、各ビューのペインは縦に積まれます。3 つのビューはいずれも複数のペインを抱えうるため、縦積みには「一度に 1 つのペインを全幅にする」手段を組み合わせます。

* **Replay**：5 トラックのグリッド（`--rep-left` / `--rep-mid` の左右カラム、溝、レポート）を 1 カラムにします。シミュレータと設定フォーム、ストリーミングされるログ、埋め込みレポートを縦に積み、セグメント切り替え（「フォーム、ログ、レポート」）を前面に置いて、選んだペインが全幅を使えるようにします。この段ではカスタムプロパティの幅と溝は使いません。
* **Record**：`.rec-stack` のエージェント進捗と出力の分割も、同じように積みます（「進捗、出力」）。
* **Crawl**：固定幅 `22rem` のサイドパネル（「プラン、コンソール」）とグラフを、切り替え（「グラフ、プラン、コンソール」）を前面にした縦積みのセクションにします。パン可能なグラフは、それが選ばれているときは全幅を保ちます。

### 小さい画面でマウス専用の操作モデルを置き換える

ドラッグでリサイズする溝と、ドラッグで分割し入れ替えするタイリングは、タッチに対応物がなく、スマートフォンには場所もないデスクトップ向けの高機能です。狭い段ではこれらを隠します（`.tile-divider`、`.tile-grip`、カラムの溝を `display:none` にし、保存された `bajutsu-splits` / `bajutsu-tiles` のレイアウトは単に適用しない）。その役割は 1 カラムの縦積みとビューごとの切り替えが引き継ぎます。残るタップ対象（上部タブ、ビューごとの切り替え、実行ボタンと停止ボタン、履歴の行）は、指で快適な最小寸法（おおむね 44 px）に整えます。これはほぼ CSS だけで済み、`serve.js` には狭いメディアクエリが一致するときに保存済みのデスクトップ用レイアウトを適用しないガードを足す程度です。

### crawl グラフ（唯一インタラクティブな変更）

crawl グラフは、CSS ではなく `serve.js` の作業が本当に要る唯一の箇所です。パンが `mousedown`/`clientX` に、ズームが `wheel` に結ばれていて、どちらもタッチでは発火しないからです。変更は、既存のパンとズームの計算を再利用するタッチ対応の追加にとどまります（ビューはすでに平行移動と `zoom` の組み合わせのモデルなので、対応づけは変わりません）。1 本指のパンには `touchstart`/`touchmove` を、ピンチ（2 本指の距離）は既存の `zoomBy` に対応づけます。グラフは調査の補助であってゲートの一部では決してないので、これは純粋に足すだけのインタラクティブ性です。

### ヘッダーの組み替え

狭い段ではヘッダー行（ブランド、上部タブ、テーマ切り替え）を折り返すか詰めて並べる（たとえばブランドの列を 1 行に、タブを次の行に置き、テーマ切り替えは届く位置に保つ）ことで、横スクロールを強いないようにします。

### すでに可変な部分は作り直さず活かす

スクリーンショットのモーダルはすでにビューポート基準で可変です（`.shotbox{width:min(960px,94vw);max-height:88vh}`、`.shotmain img{max-width:46vw}`）。その可変なサイズ指定は活かし、左右に並んだ画像と次手順リストだけをスマートフォン幅で縦積みに直します。埋め込みレポートはすでに `report.css` に自前の `@media` 規則を備えています。この項目はレポートの内部には触れず、それを枠取りする `serve` の外枠だけを扱います。

### 決定性、アプリ非依存性、ゲート

* **LLM を入れず、合否に触れない。** UI は決定的な `run` / `record` を起動するだけで、この変更はピクセルの並べ替えとタッチハンドラの追加です。合否はここで計算しないので、プライムディレクティブの 1 と 2 は構造上保たれます。
* **アプリ非依存。** レイアウトはビューポートに依存し、特定のアプリには依存しません。`apps.<name>` もドライバもランナーも読みません。
* **Linux で検証できる。** 検証の経路は既存の Web バックエンドで、後述のとおり Linux のゲートで動きます。シミュレータも macOS も要りません。

### テストの契約（機械判定可能）

[BE-0058](../BE-0058-dogfood-web-ui/BE-0058-dogfood-web-ui.md) はすでに、Web（Playwright）バックエンドで `serve` UI を駆動し、決定的な Tier 2 の回帰ネットにしています。レスポンシブ対応はそこで自然に機械判定できます。Playwright が `viewport` を設定し、アサーションは「見た目が良い」という LLM の判断ではなく構造的な事実にとどまるからです。

* **スマートフォン幅で横方向のあふれがない**：スマートフォンのビューポート（たとえば 390×844）で、ドキュメントのスクロール幅がクライアント幅を超えません。「何も横にはみ出さない」を決定的に言い換えたものです。
* **主要な操作部品に届く**：実行ボタン、上部タブ、ビューごとの切り替えが、スマートフォン幅で見えていて見切れていません（既存のアサーションを狭いビューポートで再実行します）。
* **デスクトップのレイアウトが変わらない**：広いビューポートでは既存の Replay/Record/Crawl の UI シナリオがこれまでどおり通り、追加した狭い段の規則が既定を損なっていないことを固定します。

このように BE-0058 のネットを広げれば、決定性ファーストの契約を保ったまま、レスポンシブの規則が知らぬ間に壊れることを防げます。

### 範囲外

リサイズや折りたたみが可能なタイリングのデスクトップ向けの広い再設計（この項目は小さい画面で隠すだけで、デスクトップ向けに作り直しはしない）、`run` が捕捉し判定する内容への変更、そしてレポートの内部（`report.css` ですでにレスポンシブ）。

## 検討した代替案

* **モバイル専用テンプレート、あるいはモバイル専用アプリ。** 不採用。`serve` の機能が増えるたびに同期を取るべき面が二重になり、BE-0011 が意図的に返している自己完結 HTML 1 枚とも噛み合いません。1 つのレスポンシブなテンプレートのほうが保守の負担は小さく、一度足した機能がどの画面でも動き続けます。
* **グリッドとブレークポイント目当てに CSS フレームワーク（Tailwind / Bootstrap）を採用する。** 不採用。`serve` は標準ライブラリのみで、CSS は手書きでインライン展開し、アセットのビルド工程を持ちません（BE-0011 の「検討した代替案」）。素の `@media` クエリだけで、フレームワークの依存やビルドのパイプラインを抱え込まずにレスポンシブ対応が得られ、新規クローンからそのまま動かせる性質も保てます。
* **デスクトップ専用のまま、ブラウザのピンチズームとパンに頼る。** 不採用。それがまさにこの項目が直そうとしている現状です。利用者がズームとパンで動き回らねばならない縮小されたデスクトップであり、タッチハンドラが存在しないためグラフはそもそもパンできません。
* **デスクトップでもペインを折りたたみとリサイズ可能にする（レイアウト全体の刷新）。** ここでは範囲外。この項目はあくまで小さい画面への組み替えに絞ります。デスクトップ向けにペインを可変かつ折りたたみ可能にする広い再設計は、望むなら別の提案とし、狭い画面の対応を止めるものにしてはなりません。
* **マウスのドラッグハンドラをポインタイベントに変えてタッチ対応を「ただで」得る。** 一見魅力的だが、ポインタモデルに統一すると、ドラッグで分割するタイリングや溝のリサイズまでタッチに持ち込むことになり、小さい画面ではぎこちなく無意味です。しかも、動作しているデスクトップの操作に対する、より大きく危うい変更でもあります。それらの操作部品を狭い段で隠し、指の下で本当に動かしたい唯一のもの（グラフ）にだけタッチの経路を足すほうが、小さく安全な変更です。
