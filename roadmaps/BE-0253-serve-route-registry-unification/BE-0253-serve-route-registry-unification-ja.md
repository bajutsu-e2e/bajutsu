## はじめに

`serve` は、1 つの API 面の背後で 2 つの HTTP バックエンドを動かしています。標準ライブラリの
`http.server` によるハンドラ（`bajutsu/serve/handler.py`）と、それと並行して動く FastAPI アプリ
（`bajutsu/serve/server/app.py`）です。すべてのエンドポイントは 2 回宣言されています。1 回目は
標準ライブラリのハンドラの `do_GET` / `do_POST` にある `match path` の一分岐として
（`handler.py:186-444`）、2 回目は `app.py:122-490` の `@app.get` / `@app.post` / `@app.delete`
ルートとして宣言されており、どちらの分岐も同じ `bajutsu.serve.operations`（`ops`）関数へ
ディスパッチします。
リクエストレベルの認証ゲート、CSRF の `Origin` チェック、`Host` allowlist、レスポンスを堅牢化する
ヘッダも、両方のバックエンドでそれぞれ独立に再実装されています
（`handler.py:61-69,121-170,273-290` と `app.py:69-118`）。本項目では、この 2 つの手動管理の
ルートテーブルと、その周囲で重複しているセキュリティ実施を、両方のバックエンドが参照する 1 つの
宣言的なレジストリに置き換えます。

## 動機

2 つのルートテーブルは、すでに気づかれないまま drift しています。次のエンドポイント群は標準
ライブラリのハンドラには存在するのに FastAPI アプリには対応するものがなく、FastAPI バックエンド
でデプロイした環境ではこれらが 404 になります。しかも、それが意図的なものかどうかを宣言している
コードはどこにもありません。

- `GET /flakiness`（`handler.py:237`）
- `GET /api/ant/login` と `POST /api/ant/login`（`handler.py:214,352`）
- `POST /api/enrich`（`handler.py:382`）
- `POST /api/codegen`（`handler.py:376`）
- `POST /api/capture/start`、`POST /api/capture/mark`、`POST /api/capture/finish`、
  `GET /api/capture/screenshot`（`handler.py:388-393,266`）
- `POST /api/jobs/{id}/respond-human`（`handler.py:414`）
- `GET /runs/{id}/archive.zip`（`handler.py:264`）。FastAPI アプリの `GET /runs/{rel:path}`
  （`app.py:128`）は run の静的な成果物をパスで返しますが、標準ライブラリのハンドラの
  `_serve_run_archive` のように `archive.zip` を特別扱いしません。そのため、このパスへの
  リクエストは zip を生成する代わりに `ops.run_file` に渡り、`archive.zip` という名前の
  ファイルそのものを探しにいってしまいます。

このうちのいくつかは、設計上ローカル限定であってもおかしくありません。`/api/capture/*` は
start / mark / finish の一連の流れの間、プロセス内に `Driver` のインスタンスを保持し続けますが、
これは標準ライブラリのハンドラの単一プロセスモデルでは成立しても、水平スケールした FastAPI
デプロイでは成立しない可能性があります。`/api/ant/login` はローカルの CLI をシェルアウトして
呼び出します。しかし、どのエンドポイントが意図的にローカル限定なのか、それとも単に実装が漏れて
いるだけなのかを、コードのどこも述べていません。このギャップも、2 つの `match` / ルートの一覧を
手作業で突き合わせて見つけたものであり、どこかの宣言を読んで判明したものではありません。今後
どちらか一方のバックエンドにだけエンドポイントを追加し、もう一方への反映を忘れても、同じように
気づかれずに失敗します。発見できるのは、300 行ほどの 2 つのファイルを手作業で突き合わせたときだけ
です。認証、CSRF、Host、ヘッダの重複した実施には、これよりも鋭いリスクがあります。両バックエンド
のセキュリティ上の振る舞いが実際に一致するためには、2 つのコピーを手作業でバイト単位まで一致
させ続ける必要があり、それが一致しているかどうかを検証するものは何もありません。これは、単なる
機能の欠落ではなく、セキュリティに関わる乖離につながりうる潜在的な経路です。この 2 つの問題は
どちらも根は同じで（1 つの振る舞いを、同期の必要な 2 つの独立した実装として持ってしまっている
こと）、直し方も同じです（重複させるのではなく、単一のソースオブトゥルースをそのまま実行可能に
することです）。

## 詳細設計

この作業は、両バックエンドが現在すでに提供しているすべてのルートについて挙動を変えません。
変わるのは各ルートテーブルのデータがどこに存在するかだけで、現在気づかれないまま起きている
drift を、grep できる明示的な宣言に変えます。作業は MECE な 5 つの部分からなります。

1. **単一の宣言的なルートレジストリ。** 1 つのテーブルを導入します。各エントリは、HTTP
   メソッド、パスパターン（`/api/jobs/{job_id}` のような変数を含むパスセグメントについて、2 つの
   バックエンドがそれぞれ異なる書き方をしていること、たとえば `handler.py` は
   `path.startswith(...)` / `endswith(...)` による手動のマッチングを使っていることも含みます）、
   ディスパッチ先の `ops.*` 呼び出し可能オブジェクト、そしてルートごとの小さなフラグの集合
   （`needs_actor`：ハンドラがリクエスト元の identity を解決して渡す必要があるかどうか、
   `off_loop`：SSE ストリーミングやファイル配信のように、統一された JSON レスポンスのヘルパーを
   経由しないルートであるかどうか、`local_only`：後述のポイント 4 を参照）を持ちます。この
   レジストリは両バックエンドが走査する単一のソースオブトゥルースであり、レジストリ自体は
   リクエストを実行しません。
2. **標準ライブラリのハンドラはレジストリからディスパッチする。** `do_GET` / `do_POST` /
   `do_DELETE`（`handler.py:181-429,431-`）は、手書きの `match path` の代わりにこのレジストリを
   走査し、パスパラメータを解決してから、現在の各ケースと同じ方法で対になる `ops` 関数を
   呼び出します。ルートごとのフラグ（`needs_actor`、`off_loop`）は、現在は各ケースに書き出されて
   いる条件分岐（`self._actor()` の呼び出しや、ストリーミングか `self._json(...)` かの切り替え）
   を、そのまま駆動します。
3. **`make_app` はレジストリから FastAPI のルートを生成する。** `bajutsu/serve/server/app.py` の
   ルート宣言（`app.py:122-490`）を、同じレジストリを走査して各エントリを `app.get` /
   `app.post` / `app.delete` として登録するループに置き換えます（`local_only` のエントリは
   ポイント 4 のとおりスキップします）。これにより、エンドポイントごとに手書きしていた
   `@app.<method>` デコレータがなくなります。
4. **`local_only` によって、気づかれない drift を宣言に変える。** *動機* で挙げた、FastAPI
   アプリに欠けているエンドポイントをすべて洗い出し、判定します。設計上本当にローカル限定で
   あるもの（`/api/capture/*` とその `Driver` のプロセス内での寿命、`/api/ant/login` のローカル
   CLI へのシェルアウトなど）は、レジストリ上で `local_only=True` と印を付けます。これにより、
   ポイント 3 の FastAPI 生成処理は意図的にそれをスキップし、読み手はレジストリを検索するだけで、
   欠けているものの一覧とその理由を確認できるようになります。そのような理由がないと判明した
   エンドポイントは（一覧のどれがどちらに分類されるかを本項目は事前に決め打ちせず、この判定
   自体を作業の一部とします）、通常の FastAPI ルートとして補い、ギャップをそのまま解消します。
5. **重複した認証、CSRF、Host、ヘッダの実施を、共有の 1 つの gate ヘルパーにまとめる。**
   `handler.py:61-69`（堅牢化ヘッダ）、`handler.py:121-170`（認証チェックの `_gate`）、
   `handler.py:273-290`（`_csrf_ok` / `_host_ok`）にあるロジックは、機能的には `app.py:69-118`
   の `gate` ミドルウェアと `_hardened` ヘルパーと同一です。このポリシーの実装を 1 つだけ切り出し
   （各バックエンドがリクエストのヘッダ、Cookie、メソッド、パスをどう読み取り、レスポンスをどう
   書き出すかをパラメータ化した形にします）、`handler.py` と `app.py` の両方がそれを呼び出すよう
   にします。これにより、セキュリティ上の振る舞いは 2 回ではなく 1 回だけ定義されるようになります。

## 検討した代替案

- **2 つのルートテーブルと 2 つのセキュリティ実施のコピーを手動管理のリストのまま残し、互いに
  drift したら失敗するテストを追加する。** 主な修正としては却下します。テストは事後的に drift を
  検出しますが、それも誰かが drift の出荷前にテストを走らせることを覚えていた場合に限られます。
  本項目自体のきっかけとなったギャップも、検出されないまま出荷済みでした。しかも、このアプローチ
  は重複を取り除くのではなく、重複を監視するだけです。単一のレジストリであれば、drift する元の
  リストが 1 つしか残らないため、2 つのバックエンドは構造的に drift しようがなくなります。
- **レジストリを構築せず、現在欠けているエンドポイントだけを（`app.py` へ手作業で）補う。**
  却下します。これは現在判明しているギャップを塞ぐだけで、誰かが次にエンドポイントを追加する
  ときには同じ手作業の重複がそのまま残るため、同じ種類の drift が再発します。また、重複した
  認証、CSRF、Host、ヘッダのロジックにも手を付けないままになり、これは *動機* で述べたとおり、
  より鋭いリスクです。

- 2026-07-15 — Part 5（共有の gate ヘルパー）を [#1098](https://github.com/bajutsu-e2e/bajutsu/pull/1098)
  で実装しました。フレームワークに依存しない新しいモジュール `bajutsu/serve/gate.py`
  （`HARDENING_HEADERS`、`allowed_hosts` / `host_allowed`、`csrf_ok`、`is_open`、`is_authorized`、
  `actor_for`）を両バックエンドが呼び出す形にし、トランスポート固有の機構は各バックエンドに残しました。
  挙動は変えていません。Part 1〜4（宣言的なルートレジストリ）は後続のスライスで実装します。
- 2026-07-15 — Part 1〜2（レジストリの定義と標準ライブラリ側のディスパッチ）を
  [#1108](https://github.com/bajutsu-e2e/bajutsu/pull/1108) で実装しました。フレームワークに
  依存しない新しいモジュール `bajutsu/serve/routes.py` に、1 つの `ROUTES` テーブル（凍結
  データクラス `Route`、バックエンド中立の `RequestCtx` プロトコル、`{name}` / `{name:path}` の
  テンプレートマッチャ）を置き、各ルートの `ops.*` 呼び出しをルートごとのアダプタ
  `handle(state, ctx)` にまとめました。`handler.py` の `do_GET` / `do_POST` / `do_DELETE` は、この
  テーブルからディスパッチします。挙動は変えていません。Part 1 の草案からの逸脱として、`needs_actor`
  フラグは実装しませんでした。ルートがアクターを解決するかどうかはアダプタのクロージャ内ですでに
  表現されており、独立したフラグを設けると同じ事実の情報源が二重になる（この項目が取り除こうと
  している食い違いそのもの）ためです。`off_loop` と `local_only` は残し（クロージャでは表現できない
  ため）、`content_type` はテキスト応答を選びます。`off_loop` のルート（SSE、ファイル配信、生の
  アップロード、OAuth、ログイン、インデックス）はここで宣言しつつ、各バックエンドの個別処理のまま
  にしています。Part 3（`app.py` をレジストリから生成する）と Part 4（`local_only` による FastAPI
  側で欠けているエンドポイントの判定）は次のスライスで実装します。
