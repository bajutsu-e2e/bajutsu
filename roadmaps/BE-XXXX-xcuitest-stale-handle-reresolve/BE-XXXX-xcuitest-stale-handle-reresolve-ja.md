## はじめに

オンデバイスのエンドツーエンド（E2E）ジョブ `xcuitest (multi-touch)` は、シナリオの最初の tap で
不安定に失敗します。対象の要素は画面に存在しているのに、`element vanished (stale handle)` という
エラーで落ちます。この失敗は、シナリオやテスト対象アプリの欠陥ではありません。原因は、XCUITest
バックエンドのアドレス指定にあります。

XCUITest バックエンドは、要素の解決とアクチュエーションを、オンデバイスの runner への2回の往復に
分けています。1回目の往復は要素を解決してスナップショット単位のハンドルを受け取り、2回目の往復は
そのハンドルを使ってアクチュエーションします。この2回の往復のあいだに、アプリのアクセシビリティ
ツリーは取り直される場合があります。新しく前面に出た画面が落ち着くまでのあいだ、ふつうに起こる
現象です。取り直しが起こると、1回目の往復が発行したハンドルは、2回目の往復が使うころには古くなって
います。

本項目は、決定論的な判定を弱めずに、この往復間のギャップを塞ぐことを提案します。具体的には、
`stale` の応答を受けたら runner に問い合わせ直します。セレクタがなお1つの要素だけに解決できる
場合は、再アクチュエーションします。セレクタが解決できなくなった瞬間には、今までと同じように
大きく失敗させます。変更は、xcuitest バックエンドの Python チャネルの内側だけで完結します。触れる
面は、runner とのやり取り、リトライの継ぎ目、決定論的な判定の3つに限られます。

## 動機

この不安定さは、必須の `E2E (iOS)` ゲートで再現します。実行 29630685037 では、`xcuitest
(multi-touch)` ジョブの2本目のシナリオが、最初のステップで失敗しました。シナリオは
[`permission.yaml`](../../demos/showcase/scenarios/permission.yaml) です。

```
step 0 (tap): element vanished (stale handle): {'label': 'Permissions', 'traits': ['button']}
FAIL  runs/20260718-044659/manifest.json
```

このステップが tap する「Permissions」タブバーボタンは、アプリの起動画面の固定された要素です。
そのため、ボタンは消えません。古くなったのはハンドルであって、要素ではありません。原因は、
2段階のアドレス指定にあります。

具体的には、XCUITest バックエンドは、1回の tap を、runner の要素スナップショットに対する2回の
往復に分けます。`_resolve_handle` はまず `GET /elements` を送ります。Python 側でセレクタを1つの
要素に解決し、runner がその要素に対して発行した不透明なスナップショット単位のハンドルを読み取り
ます。続いて `tap` が `POST /tap {handle}` を送ります。runner は各ハンドルを1つのスナップショット版に
対応づけています。そのため、問い合わせと tap のあいだにアクセシビリティツリーが取り直されると、
先のハンドルはもう生きた要素に対応しません。runner は `status: "stale"` を返し、ドライバは要素消失の
エラーを送出します。新しく前面に出た画面は、落ち着くまでのあいだ何度もスナップショットを取り直し
ます。シナリオの最初の tap が、このレースの当たる場所になるのはそのためです。

実際、失敗したステップの直前のログも、同じ結論を裏づけます。原因は runner の欠陥ではなく、起動
直後の安定化にあります。失敗したステップの前のおよそ60秒間、ドライバはあるログを記録し続けました。
ログは `runner channel GET /health failed ... [Errno 61] Connection refused` です。`await_ready`
が runner のループバックサーバをポーリングしていたことを示すログで、固定 `sleep` ではなく有界の
条件待ちによるものです。tap は、runner が応答してから初めて発火しました。つまり runner は、`stale` を報告した
時点で、すでに生きていて応答していました。runner は自身がアクチュエーションする前に `stale` を
返すので、tap はどの要素にも届いていません。画面上で実際に消えたものは、何もありませんでした。

この不安定さは、3つの意味でコストがかかります。1つ目は、ゲートへの信頼です。必須の集約チェック
`E2E (iOS)` がランダムに失敗すると、メンテナは本物の回帰と見分けられません。見分けられないために、
反射的な再実行か、無駄な調査のどちらかを招きます。2つ目は、実行コストです。再実行のたびに、
macOS ランナーの従量課金枠を消費します。最初の tap に到達するまでに、アプリをビルドし、Simulator
を起動し、runner を立ち上げる必要があるからです。3つ目は、調査を誤らせる効果です。もっとも
厄介な点でもあります。`element vanished` というメッセージは、「Permissions」要素が存在しないかの
ように読めます。しかし実際に起きたのは、ずっと存在しているボタンの上でのスナップショット版の
レースです。調査者は、存在しない消失を追わされてしまいます。

既存のどの仕組みも、この stale レースを扱っていません。もともと扱う設計でもありませんでした。
[BE-0207](../BE-0207-xcuitest-channel-transient-retry/BE-0207-xcuitest-channel-transient-retry-ja.md)
の一時的リトライの継ぎ目は、1回のチャネル呼び出しの中で起こるトランスポートのブレを吸収します。
しかし `stale` の応答は、トランスポートの失敗ではなくデコード済みの結果です。継ぎ目は、この結果を
あえて再試行しません。結果そのものを再試行することは、
[BE-0049](../BE-0049-determinism-flakiness-audit/BE-0049-determinism-flakiness-audit-ja.md) が
退ける「吸収による不安定さ」だからです。
[BE-0218](../BE-0218-e2e-simulator-flaky-readiness-actuation/BE-0218-e2e-simulator-flaky-readiness-actuation-ja.md)
の準備完了とタイムアウトの作業は、runner が誤った画面で準備完了と判定されるのを止め、遅い書き込みに
長い有界の窓を与えました。ただし、どちらの変更も、「解決してからアクチュエーションする」構成が生む
往復間のギャップには触れていません。
[BE-0287](../BE-0287-xcuitest-runner-multitouch-resilience/BE-0287-xcuitest-runner-multitouch-resilience-ja.md)
の姉妹の不安定さは、別の故障クラスです。BE-0287 では runner がクラッシュして応答を止めますが、
本項目では runner は生きていて、`stale` をただしく返します。stale レースを塞ぐには、専用の仕組みが
必要です。

## 詳細設計

作業は4つの単位に分かれます。Unit 1 はドライバの変更、Unit 2 は決定論のディレクティブの下で変更を
誠実に保つ作業、Unit 3 は実機なしで変更を証明する作業、Unit 4 はドキュメントを一致させる作業です。
各単位は依存順に並びますが、別々のプルリクエストとして着地できます。

**Unit 1 — ドライバチャネルに、stale をゲートにした再解決リトライを入れる。** 変更する箇所は、
[`bajutsu/drivers/xcuitest.py`](../../bajutsu/drivers/xcuitest.py) です。対象は、ハンドルベースの
アクチュエーション経路です。この経路は、`_resolve_handle` の継ぎ目が供給する `_actuate` から成ります。
ここで `stale` の応答を、即座の要素消失エラーとしてではなく、問い合わせ直しのきっかけとして
扱います。具体的には、`_resolve_handle(sel)` を実行し直します。セレクタがなお1つの要素に解決
できるときは、ハンドルを取り直してアクチュエーションを再発行します（小さな有界回数まで）。
セレクタが0個または複数に解決するときは、`ElementNotFound` や `AmbiguousSelector` を送出します。
この場合はそれ以上試行せず、その結果で即座に失敗します。再解決こそが、2つのケースを分ける条件
です。要素がなお存在するスナップショット版のレースと、要素が存在しない本物の消失です。この条件が
あるので、ゲートはたまたま一致した何かを tap しません。ハンドルベースのアクチュエーション、
すなわち `tap`、`double_tap`、`long_press`、`pinch`、`rotate` は、すべてこの継ぎ目を通ります。
したがってリトライを継承します。座標指定の `tap_point` は、古くなるハンドルを持ちません。そのため
変更しません。試行回数の上限には、stale リトライ専用の定数を使います。初期値は BE-0207 と同じ
（3回、0.5秒・1.0秒の指数バックオフ）でも、BE-0207 のトランスポートリトライの `_MAX_ATTEMPTS` と
`_BACKOFF_BASE_SECONDS` とは分けて持ちます。2つのループが上限を定める対象が異なるからです。一方は
`_resolve_handle` と `_actuate` のあいだで落ち着く画面、もう一方は `_with_retry` の中のトランスポートの
ブレです。一方の後からの再調整、たとえば遅い CI ランナー向けにトランスポートの予算を緩めることが、
もう一方を黙って動かしてはなりません。そのためループ全体は、1秒未満ではなく数秒程度に収まります。
本当になくなった要素を、長く再試行することはありません。

この再解決は、リトライ経路に限って、バックエンドの「解決したまさにその要素に作用する」という
保証を緩めます。本提案は、このトレードオフを隠さずに述べます。再解決でセレクタが一意に一致
しても、証明できるのは「セレクタに一致する要素が存在すること」だけです。その一致が、1回目の
往復で見つけた要素と「同じ」であることまでは証明しません。2回の往復のあいだに、画面は変わることが
あります。たとえば、モーダルが閉じるような場合です。画面が変わると、同じセレクタは、別の要素だけに
一致するようになります。そのときリトライは、その別の要素に作用してしまいます。それでも、
このトレードオフは受け入れ可能です。見かけよりも狭いトレードオフでもあります。理由は次のとおり
です。解決は Python 側にとどまります。したがって、不変条件の本来の目的は損なわれません。この
目的とは、解決の権限を runner の外に保ち、曖昧な述語が黙って複数の一致から1つを選ぶのを防ぐ
ことです。リトライは、Python 側の一意の一致にのみ作用します。複数のうちの1つに作用することは、
決してありません。しかも、作用する瞬間に解決し直すという振る舞いは、idb がすでに行っている
ことと同じです。idb は、要素の解決とタップを1手で行います。したがってこのリトライは、XCUITest の
stale 処理を、ほかのバックエンドに揃えるものです。より緩い規則を発明するものではありません。
より厳しい変種も考えられます。セレクタだけでなく、元の要素の同一性属性にも一致するときに限って
再アクチュエーションする形です。この変種は、セレクタだけの一致が緩すぎるとわかった場合に備えて、
`implement-be` の選択肢として残します。

**Unit 2 — 決定論を保ち、二重にアクチュエーションしない。** `stale` の応答の後で tap を再発行
することは、タイムアウトの後で再発行することとは違って安全です。この違いは、BE-0218 が拠り所に
した区別と同じです。runner は、自身が要素へ触れる前に `stale` を返します。
[`BajutsuKit/Sources/BajutsuRunner/Router.swift`](../../BajutsuKit/Sources/BajutsuRunner/Router.swift)
では、`store.lookup(handle:)` が `.stale` を報告します。ハンドラは、アクチュエーションせずに戻ります。
したがって `stale` の応答は、アクチュエーションが起きなかったことの確定的な証拠です。再送しても、
ジェスチャを二重に適用することはありません。対してタイムアウトは、届いたかどうかが不明のまま
です。不明であるために、BE-0218 は期限を過ぎた書き込みの再試行を拒みました。決定論は、そのほかの
点でも損なわれません。大規模言語モデル（LLM）は経路に入りません。固定 `sleep` も足しません。
問い合わせ直しの往復そのものが条件であり、試行間のバックオフは安定化のための待機ではなく、Unit 1 の
stale リトライ専用の有界バックオフです。本物の `stale` の消失、曖昧な再解決、使い切った試行の予算は、
いずれも今までと同じ大きな失敗のままです。

**Unit 3 — 実機なしで変更を証明する。** 継ぎ目は、模擬トランスポートに対するユニットテストで
覆います。Simulator のいらない高速ゲートで動くテストです。ケースは4つあります。`stale` の応答が
1回あってから `ok` になり、セレクタがなお一意に解決するときは、回復します。`stale` が続いても
セレクタがなお解決するときは、上限を使い切って大きく失敗します。`stale` の後の問い合わせ直しで
セレクタがもう解決しないときは、追加の試行を使わず `ElementNotFound` で即座に失敗します。`stale`
の後の問い合わせ直しでセレクタが曖昧に解決するときは、`AmbiguousSelector` で失敗し、決して
アクチュエーションしません。4つのケースは、2つのことを固定します。1つは、スナップショットの
レースが回復することです。もう1つは、本物の消失や曖昧さが、なお失敗することです。この2つを
固定することで、後の変更が、誠実なゲートをこっそり吸収するゲートに変えることはできなくなります。

**Unit 4 — ドキュメントを一致させる（BE-0113）。**
[`bajutsu/drivers/xcuitest.py`](../../bajutsu/drivers/xcuitest.py) のモジュール docstring は、
いまの挙動をこう述べています。stale なハンドルは「idb が送出するのと同じ要素消失エラーとして
表面化する」というものです。この docstring を、再解決リトライと、stale なハンドルがなお失敗する
正確な条件を述べるものに更新します。[`DESIGN.md`](../../DESIGN.md) と [`docs/architecture.md`](../../docs/architecture.md)
が stale のセマンティクスを述べているなら、散文の記述が挙動に合うように更新します。

## 検討した代替案

**再解決のゲートなしに、stale なアクチュエーションを無条件で再試行する。** ゲートを外すと、本物の
消失を静かな再試行に吸収してしまいます。本物の消失とは、前のステップの遷移が取り除いた要素です。
これは、BE-0049 が禁じる「吸収による不安定さ」にあたります。再解決のゲートこそが、リトライを誠実に
保ちます。要素が明らかになお存在するときに限って、再アクチュエーションするからです。

**起動後、最初の tap の前に固定の安定化待機を入れる。** 条件を無視した固定 `sleep` は、プライム
ディレクティブ 2 が明確に禁じています。しかもこの待機は、レースの窓を狭めるだけです。窓を塞ぐ
ことはありません。安定化が固定の予算より遅ければ、それでもハンドルは古くなります。再解決リトライ
は、画面が落ち着くまでどれだけかかっても、レースを塞ぎます。

**解決を runner に移し、2回の往復を1回に畳む。** runner がアクチュエーション時にセレクタを解決
すれば、スナップショットのギャップは消えます。しかし、Python 側での解決は、バックエンドの核と
なる不変条件です。解決の権限は Python 側にとどまり、複数の要素に一致しうる runner 側の述語には
移りません。stale リトライも、この権限を Python 側に保ちます。Python 側で、しかも一意の一致に
のみ再解決するからです（このトレードオフは *詳細設計* で詳しく述べています）。解決を runner に
移すと、この権限を手放すことになります。手放した先には、決定論的な核が防ごうとしてきた曖昧さが
生まれ、stale レースとの引き換えになってしまいます。

**CI でチェックを隔離するか、ジョブ単位で再試行する。** `xcuitest (multi-touch)` を失敗許容にする、
あるいはジョブを再試行で包むと、レースに触れないままチェックを緑にできます。BE-0287 が姉妹の
不安定さに対して退けるのと同じ理由です。本物のレースを緑のチェックの裏に隠し、静かな再試行の
たびに macOS の分を消費し続けるからです。
