## はじめに

ホスト版またはセルフホスト版の `serve` が蓄積する、データベース（DB）上の Run 履歴を採掘します。実際にフラッキーなシナリオ（シナリオ自体は変わっていないのに合否が揺れるもの）を洗い出し、洗い出した各シナリオについて、AI の調査役が成功 Run と失敗 Run の**両方**の証跡を読んで、Run 間で何が変わっているのかを説明し、具体的な修正を提案します。修正はシナリオ YAML の全体書き直しまで含みます。サジェスト側（どのシナリオがフラッキーか、どの程度か、順位付き）は完全に決定的で AI を必要とせず、修正側は人間がレビューして取り込む助言的な提案です。自動編集はせず、合否判定の経路にも一切入りません。

これは、すでに出荷済みの 2 つのコマンドに対する、履歴全体を対象とした縦断的な対応物です。[BE-0049](../BE-0049-determinism-flakiness-audit/BE-0049-determinism-flakiness-audit-ja.md) の `audit --history` による分類を「各シナリオの分類はこうです」から**順位付きで行動につながる提示面**へ拡張し、[BE-0021](../BE-0021-ai-triage/BE-0021-ai-triage-ja.md) と [BE-0022](../BE-0022-update-structured-fixes/BE-0022-update-structured-fixes-ja.md) のトリアージを「1 つの失敗 Run を診断する」から「同一シナリオの多数の Run にまたがる**パターン**を診断する」へ拡張します。

## 動機

Bajutsu はすでに 2 通りの方法でフラッキーさを検出します。`audit` はシナリオを `K` 回実行して結果を差分比較し（ある時点での証明）、`audit --history` は蓄積した Run を `(scenarioHash, name)` でグループ化して各シナリオを `flaky` / `deterministic` / `unproven` に分類します（BE-0049）。しかし検出は分類で止まっています。ホスト版 `serve` を数百のシナリオと増え続ける Run の DB とともに運用しているチームは、依然として次の 2 つを手作業で行う必要があります。第一に、その分類をシナリオごとに 1 つずつ探しにいくことです。第二に、フラッキーなものを見つけたら、成功 Run と失敗 Run を並べて開き、何が違ったのかを読み解く退屈な作業を手でこなすことです。

ここから 2 つのギャップが生じます。

第一のギャップは、フラッキーさが分類されても提示されていないことです。どのシナリオがどれだけフラッキーかでスイートを順位付けするものはなく、「今週のワースト 5 はこれです」をチームが行動を起こす場所に届けるものもありません。シグナルは DB に存在します。ただ提示されていないだけです。

第二のギャップは、トリアージが失敗ごとであってパターンごとではないことです。`triage` は**1 つの**失敗 Run の証跡を読んでその失敗を説明します。間欠的な失敗は別の問いです。問うべきは「この Run はなぜ失敗したか」ではなく「このシナリオは同一の content hash のもとでなぜ成功したり失敗したりするのか」です。これにうまく答えるには、複数の Run の証跡を一度に読む必要があります。成功したものと失敗したものの両方を読み、その差分について推論します（Run A では解決したセレクタが Run B では曖昧だった、5 回中 4 回はスピナーより待機が先に成立した、ネットワーク応答が Run ごとに違った、など）。この Run 横断の読み解きこそ、LLM が得意とする判断を伴わない調査作業であり、既存のトリアージの境界を自然に延長したものです。

両方を埋めることで、決定性優先の立場が証明可能（BE-0049）であるだけでなく**保守可能**になります。チームは最もフラッキーなテストを順位付きで見られ、それぞれに対して具体的でレビュー可能な修正を得られます。その間、合否判定は決定的ランナーの中に完全にとどまります。

**境界に関する注記。** BE-0049 と同様、この提案は決定性優先を損なうのではなく強化します。サジェスト側は履歴から採掘したフラッキーさを*報告する*だけで、テストを再試行して緑にすることはなく、合否を変えることもありません。修正側は人間がレビューする編集を*提案する*だけです。しかも [BE-0023](../BE-0023-self-healing-guards/BE-0023-self-healing-guards-ja.md) に従い、アサーションを弱める提案（`expect` の削除、`value` の緩和、一意性を超えたセレクタの拡張）は「テストを甘くするもの」として明示され、黙って提示されることはありません。この提案のどこにも、LLM を `run` / CI の合否判定経路に置く要素はありません。

## 詳細設計

提案レベルの粒度で述べます。2 つの半分から成り、前半は決定的で後半の前提になります。

### 前提：DB の Run レコードへの Run provenance の付与

DB を対象とした Run 横断のフラッキー検出には、「同一シナリオ」が明確に定義されるよう各 Run にスタンプが必要です。BE-0049 はすでに `manifest.json` に `provenance` ブロック（`scenarioHash`、`toolVersion`、`gitRevision`）を付与しており、同じ provenance を serve の DB の Run レコードにも載せる必要があると、フォローアップとして記しています（[BE-0015](../BE-0015-web-ui-public-hosting/BE-0015-web-ui-public-hosting-ja.md) 配下で追跡）。この項目はそのスタンプが DB レコード上に存在することに依存します。まだ出荷されていなければ、それを届けることがここでの最初の作業単位になります（列を追加し、可能な範囲で保存済みの `manifest.json` からバックフィルします）。

### 前半：フラッキーサジェスト（決定的、AI 不要）

- **DB を対象とした Run 横断のフラッキースコア。** DB の Run レコードを `(scenarioHash, name)` でグループ化し、設定可能なウィンドウ（直近 `N` 件、直近 `D` 日など）にわたってシナリオごとのフラッキー指標を計算します。指標は、content hash が変わらないままの合否反転率です。これを、シナリオが編集された場合（hash が変化）やアプリ / ツールのバージョンが動いた場合と切り分けます（BE-0049 と同じ区別を再利用します）。`bajutsu audit --history` の分類（`flaky` / `deterministic` / `unproven`）を土台にし、その上に順位付けを重ねます。
- **serve Web UI の順位付き提示面。** フラッキーさで並べ替えたシナリオ一覧を表示するパネルです。成功 / 失敗の件数、ウィンドウ、分類を示し、各行から代表的な成功 Run と失敗 Run の証跡へリンクします。これは「CLI 機能を serve Web UI に露出する」パターンを履歴ビューに適用したものです。同じ順位付けを CI やスクリプトから使えるよう、`--json` / CLI 形式も用意します。
- **合否への影響なし。** この提示面は Run 履歴に対して読み取り専用です。合否を計算せず、何もゲートしません。

### 後半：Run 横断の修正提案（AI 調査役、助言的）

- **Run 横断の TriageContext。** 既存の `assemble` ステップ（BE-0021）を「1 つの Run の最初の失敗シナリオ」から「1 つのフラッキーシナリオを、その複数 Run にまたがって見る」形へ拡張します。代表的な**失敗** Run から失敗メッセージ / 失敗ステップ / 要素ツリー / スクリーンショットを集め、**あわせて**代表的な**成功** Run から対応する証跡を集め、さらにシナリオ定義と関係するセレクタ id を加えます。この context が差分を読み解く材料になります。
- **並行プロトコルの背後でのパターン診断と修正提案。** `TriageAgent` のパターンを踏襲し、`CrossRunTriageAgent` プロトコル（その `triage_flaky` が Run 横断の context を受け取ります）を用意します。`ClaudeTriageAgent` 系の実装が Run 横断の context を読み、構造化されたツール呼び出しを通じて次の 2 つを返すよう強制されます。第一に、間欠性の根本原因カテゴリ（`selector-ambiguity` / `timing` / `network-variance` / `state-leak` / `unknown`）です。第二に、提案する修正です。修正は、的を絞った構造的編集（`label` セレクタを `id` へ昇格する、`wait` 条件を追加 / 厳格化する、曖昧さ解消のため `within` スコープを足す）から、シナリオが構造的に脆いときの**シナリオ YAML 全体の書き直し**までにわたります。
- **常にレビュー可能な提案であり、自動編集ではない。** BE-0022 および DESIGN §6.5 と整合し、出力は人間がレビューして取り込む提案差分です。コミット済みの YAML がひとりでに変わることはありません。全体書き直しも、現在のシナリオに対する差分として提示し、黙って上書きしません。
- **甘くしないためのガード。** BE-0023 に従い、テストのアサーションを弱める提案（`expect` の削除、`value` / `label` 一致の緩和、一意性を超えたセレクタの拡張、`wait` タイムアウトの除去）はいずれも「テストを甘くするもの」として明示します。これにより、レビュー担当者はトレードオフを明示的に把握でき、修正が「通すため」に静かにカバレッジを削ることを防ぎます。

### 提示面

後半を新しいサブコマンド（`bajutsu triage --flaky <scenario>` など）として出すか、既存の `audit --history` のモードとして出すか、純粋に serve Web UI のアクションとして出すかは、採用時に委ねる実装上の選択です。serve のパネル（前半）を主たる提示面とし、CLI 形式によって CI とスクリプトを第一級に保ちます。

## 検討した代替案

* **フラッキーテストの自動再試行 / 隔離（業界で一般的な答え）。** BE-0049 と同じくゲートでは却下します。再試行で通すことはフラッキーさを隠すためです。この項目は逆に、洗い出して修正します。
* **AI に書き直しを自動適用させる。** 却下します。非決定的なモデルにコミット済みのテストを黙って変えさせることになり、DESIGN §6.5（「AI の出力は常に提案差分」）と BE-0023 の甘くしないガードに反します。書き直しは常にレビュー可能な差分とします。
* **ある時点での repeat-and-diff のみ（現状の BE-0049 `audit`）。** 許容できますが不完全です。`K` 回実行することは 1 つのシナリオのフラッキーさをその場で証明しますが、チームの実際の Run 履歴がすでに含んでいるフラッキーさを採掘せず、スイートを順位付けせず、蓄積された成功 / 失敗のペアをまたいで推論もしません。
* **失敗ごとのトリアージのみ（現状の BE-0021）。** 単一の失敗はうまく診断しますが、間欠性（同一シナリオが 1 つの content hash のもとで成功したり失敗したりすること）を見ることはできません。それを露わにできるのは Run 横断のビューだけです。
* **DB の Run レコードではなく専用のフラッキー用ストアを設ける。** 却下します。Run 履歴はすでに DB に蓄積されており、並行するストアを足すと記録の源が二重になります。この項目は既存レコードに対するクエリと提示面を足すだけです（加えて BE-0049 がすでにスコープした provenance のスタンプを使います）。
* **2 つのアイテムに分割する（前半：フラッキーサジェスト提示面、後半：Run 横断修正提案）。** 同じトピックのトリアージ関連作業が BE-0021、BE-0022、BE-0023 の 3 アイテムに分かれて出荷された事実を踏まえると、検討に値する選択肢です。それでも 1 アイテムにまとめたのは、前半単独では価値が限定的だからです。フラッキーなシナリオの順位付き一覧を提示しても、ユーザーがすぐに行動できなければ意味が薄く、その行動（Run 横断修正提案のレビュー）こそが後半です。また両者は同じ Run 横断証跡組み立てのステップ（前提となる DB への provenance スタンプと `TriageContext` の拡張）を共有しており、分割すると同じ前提を 2 つの提案にまたがって調整する必要が生じます。実装上、後半の AI エージェント設計が前半より著しく成熟に時間がかかることが判明した場合は、その時点で分割するのが適切です。Progress のチェックリストはすでに前半のみを先に完了できる構造になっています。
