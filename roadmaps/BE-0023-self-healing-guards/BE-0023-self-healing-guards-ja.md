## はじめに

自己修復が合否基準を緩めるリスクに対応します。fix は**必ず人間が diff レビュー＋ `--write` で明示適用**（自動適用しない）、断片不一致は安全 no-op です。

## 動機

自己修復にはよく知られた失敗モードがあります。テストをグリーンに保とうとするあまり、こっそりとテストを弱くしてしまうのです。リネームされた id は画面上にあるものへ「修正」され、タイムアウトは実際の遅延を覆い隠すまでじりじり伸び、アサーションは常に通るまで緩められます。一つひとつは無害に見えますが、合わさるとスイートが空洞化し、回帰を捕まえられなくなります。Bajutsu の前提全体が「決定的ゲートは信頼できる」ことにある以上、防御のない fix 機構は、このツールが守るべき唯一のものを掘り崩します。本提案は、自己修復が fix を提案できる一方で、テストが何を検査するかをこっそり緩めることを決してできないようにする防止策の集合です。

## 詳細設計

防止策は助言的ではなく構造的です。

* **自動適用は一切しない。** fix は必ずまず人間が読む unified diff として提示されます。ファイルへの書き込みには明示的な `--write` が必要で、既定は何も変えないドライランです。検証はモデルの言い分ではなく決定的な `run`（`--rerun`）です。
* **制約された fix 種別。** fix は `renameId`／`addIndex`／`raiseTimeout` のいずれかで、シナリオ source に対する `find` -> `replace` として表現されます。モデルは任意の書き換えを出せず、トークンのリネーム、セレクタの一意化、wait の延長しかできません。しかも `find` は source の正確な部分文字列でなければなりません。
* **断片不一致は安全な no-op。** `find` 断片が source に一致しなくなった場合（ファイルが先に進んだ、あるいは fix が古い）、`apply_fix` は置換回数 0 を返し、テキストを変更しません。CLI は no-op を報告します。黙って中途半端に適用されることはありません。
* **AI は調査役側に留まる。** モデルは提案するだけで、合否を決めません。判定は決定的な `run` の `expect` アサーションだけから生まれます。ここのどの防止策もアサーションを緩めません。そもそもトリアージはアサーションを緩める fix を出さないからです。`raiseTimeout` は wait を延ばしますが、何をアサートするかは変えません。そして「期待値を弱める」fix 種別は存在しません。

これらを合わせると、fix にできる最悪のことは、レビューされて却下されることだけです。人間が diff を読んで opt-in しない限り、コミット済み YAML には到達できません。

## 検討した代替案

* **信頼度しきい値つきの自動適用。** 魅力的ですが、どれほど確信があっても自動書き込みはレビューなしに条件を緩めうるものであり、まさに本提案が防ごうとする失敗モードです。人間を介した diff が防止策であり、しきい値は線を動かすだけで保てません。
* **より強い警告つきでアサーション弱化 fix を許す。** 却下しました。期待値を緩めることはテストを甘くする最も直接的な方法なので、どの fix 種別もそれを生みません。アサーション失敗は人間向けの提案を得るだけで、適用可能な fix にはなりません。
* **レビューだけを信頼し no-op の安全策をやめる。** 断片不一致の no-op がなければ、古い fix が部分適用されたり、思わぬ形でファイルを壊したりしかねません。不一致を目に見える no-op にすることで、却下された fix や古い fix でも害を及ぼさないようにします。
