## はじめに

Bajutsu が増やしつつある著述の入り口、すなわち AI record ループ、操作キャプチャ（BE-0012）、シナリオ GUI エディタ（BE-0013）の役割分担を定義し、シナリオがそれらの間をどう移動するかを明文化します。*意図* からの著述、*実演* からの著述、既存シナリオの *編集* は、それぞれ別の問いに対する別の答えですが、いずれも単一の `Scenario` を生成し消費しなければなりません。そうしてはじめて、著者は同じファイルに対して 3 つを自由に組み合わせられます。本提案は、どの入り口をいつ選ぶかを定め、仕組みを要する唯一の変換、すなわち構造的にキャプチャした、または手で編集したシナリオに、意図推論だけが供給できるアサーションを補強する変換を規定します。

## 動機

「人が検証してほしいこと」からシナリオへ至る経路は、これまで AI record ループだけでした。その後ろに 2 つが続きます。操作キャプチャ（BE-0012）は実演からフローを著述し、オフラインかつ API キー無しで動きます。GUI エディタ（BE-0013）は既存シナリオを、スクリーンショット駆動の要素ピッカーで編集します。3 つの著述の入り口が同時に到着することは、仮想ではなく現実のリスクです。境界を明示しなければ、乖離したシナリオ形が増え、同じセレクタと保存の配管を 3 度重複させ、どれを開けばよいか分からない著者を残します。3 つは実際に異なる問題を解くため、答えは統合ではありません。境界を明示的に引き、ファイルを 3 つの共有物にすることです。

境界は著述の *入力* で引きます。

- **AI ループ** は *意図*、つまり自然言語のゴールから著述します。著者が正確なステップをまだ知らないとき、フローに探索が要るとき、ゴールを実行するより述べる方が容易なときに最適です。ステップと検証アサーションの両方を推論します。`ANTHROPIC_API_KEY` が必要で、LLM（大規模言語モデル）の往復を消費します。
- **操作キャプチャ** は *実演*、つまり起動済みデバイス上の実操作から著述し、キャプチャ時に安定セレクタへ解決します。著者がすでにフローを知っているとき、オフラインで高速にしたいとき、記述より実演が容易なときに最適です。API キー不要で、解決は純粋に構造的であり、出力はステップのみで、アサーションは推論しません。
- **GUI エディタ** は既存シナリオを *編集* します。スクリーンショット上で要素をクリックすると、最も安定したセレクタを `doctor` スコア付きで得られ、ステップとアサーション DSL（ドメイン固有言語）をフィールド単位で調整できます。どの入り口が生成した下書きであっても、生の YAML を手で書かずに磨いたり、脆いセレクタを直したりするのに最適です。

この 3 つがもたらすリスク、すなわち乖離した形式、3 重化した配管、混乱する著者を、本提案が先回りして防ぎます。

## 詳細設計

### 単一のシナリオ、3 つの入力、単一の保存経路

境界は出力ではなく、つねに著述の *入力* で引きます。3 つの入り口はいずれも同一の `Scenario`（steps + `expect`）を生成し消費し、同じ決定的な `run`、`codegen`、レポートに供給します。下流はどの入り口が生成したかを知る必要がなく、3 つのいずれも Tier 2 / CI ゲートには入りません。

3 つを、互いに置換可能にし、ばらばらな道具にしないものは、**3 つとも単一の著者所有の保存経路を通って書く** ことです。その経路が `bajutsu/serve/scenarios.py:ScenarioScope.save()` / `.authored()` です。AI ループはすでにこの経路で書きます（`record` → `authored()`）。キャプチャは解決した各ステップを追記してこの経路で再保存します（BE-0012）。エディタは同じ `*.yaml` をこの経路で読み書きします（BE-0013）。保存経路が共有で形式が単一であるため、`$EDITOR` での手編集、キャプチャしたステップ、AI が著述した下書き、ピッカーでの編集は、すべて同じ成果物です。PR でレビュー可能であり、決して黙って書き換えられません。書き込みはスコープが所有するので、この性質はローカルのオンディスクストアでも、サーバ側のストレージストア（BE-0015）でも、どの入り口も違いを知らないまま同一に成り立ちます。

役割分担を一度だけ表に示します。著者がどれを開けばよいか分かるためのものです。

| 著述の入力 | 入り口 | 出力 | キーの要否 | 選ぶ場面 |
|---|---|---|---|---|
| *意図*（ゴール） | AI record ループ（`record.py`） | ステップ **と** アサーション | 必要 | まだ知らないアプリを探索する。実行するよりゴールを述べる |
| *実演*（操作） | 操作キャプチャ（BE-0012） | ステップのみ | 不要 | 知っているフローをオフラインで高速に再現する。記述より実演する |
| *編集*（既存ファイル） | GUI エディタ（BE-0013） | ステップ / アサーションの編集 | 不要 | 下書きを磨く。YAML を手で書かずに脆いセレクタを直す |

### キャプチャ / 編集 → アサーション補強（明文化すべき唯一の変換）

入り口どうしの変換は非対称であり、仕組みを要するのは一方向だけです。

3 つとも同じ `Scenario` を生むため、**キャプチャした、または編集したシナリオはすでに第一級のシナリオ** です。追加ステップ無しに手編集、実行、`codegen` への投入ができます。「キャプチャ → シナリオ」変換は恒等であり、「エディタ → シナリオ」は保存そのものです。逆向き、つまり AI が著述したシナリオを手でキャプチャし直す、または編集し直す操作は、通常の編集または再記録に過ぎず、特別な仕組みは要りません。

設計を要する唯一の向きは、**構造的なシナリオをアサーションで補強する** ことです。キャプチャとエディタは *ステップ* を生みますが、`wait` / `expect` が検証すべき *意図* は推論できません。そのため、キャプチャした、または編集したばかりのシナリオは、ステップ列は完全でもアサーションが薄い、あるいは無いことがあります。それを *検証済み* のシナリオに変えるのは意図推論であり、AI ループの仕事です。受け渡しは次のとおりです。

1. 著者は、ステップはあってもアサーションが薄いシナリオを持っています。キャプチャ由来、エディタ由来、または手書きのいずれかです。
2. 著者は、その既存のステップ列に対して AI ループを **補強モード** で起動します。ループはステップを再生または読み取り、各ステップが何を確立しようとしているかを推論し、アサーションを欠くステップに対して検証アサーション（`expect` の項目、settle `wait`）を提案します。
3. 提案は、著者が承認または却下する **レビュー可能な差分** として届きます。決して黙って書き換えず、「AI 出力は常に提案差分」と整合します。承認すると同じ `ScenarioScope.save()` を通って書き戻され、補強済みシナリオは同じ成果物のまま、検証済みになります。

これにより、入り口どうしが端から端まで組み合わせ可能になります。キャプチャでフローを実演し（オフライン、キー不要）、エディタでセレクタを整え、それから一度だけキーを使って AI ループにアサーションを足してもらえます。最初から著述し直す必要はありません。逆向きの境界は固いままです。**キャプチャは決して意図を推論しません。** ステップが何を検証すべきかの推論は AI の著述・調査の役割であり、キャプチャは純粋に構造的であり続けるので、オフラインで API キー不要という保証は無条件です。AI はアサーションを提案しますが、pass/fail を決めることは決してありません。

### serve における統合された著述面

3 つの入り口は、`serve` に取り付けた 3 つの別々の道具ではありません。**1 つの開いているシナリオ** に対する 3 つのモードです。`serve` UI はすでにシナリオをホストします（BE-0011：シナリオビュー、生の YAML テキストエリア、画面ごとのスクリーンショットと要素ツリーを含む実行レポート）。残りの 2 つは、その同じビューの拡張であり、新しいページではありません。

- **エディタ**（BE-0013）は、シナリオビューを、スクリーンショットを真実の源とし要素ピッカーがセレクタを解決する、フィールド単位の構造化編集に変えます。
- **キャプチャ**（BE-0012）は、同じビューを「実演」モードにしたものです。著者がライブのスクリーンショット上でアクションをマークすると、Bajutsu が各点をセレクタへ解決して代理実行し、開いているシナリオへステップを流し込みます。
- **補強** は、開いているシナリオに対するアクション（「アサーションを提案」）であり、AI ループの差分を同じビュー内にインラインで提示します。

つまり著者は 1 つのシナリオを開き、その上で著述モードを切り替えます。実演でステップを足し、ピッカーでセレクタを直し、提案でアサーションを足す形で、最初に道具を選ぶのではありません。ピッカーと `doctor` スコアのコンポーネントはキャプチャとエディタで共有され（BE-0012 と BE-0013 の双方がこれを明記しています）、すべてのモードが単一の `ScenarioScope` を往復するので、モードを切り替えてもファイルの形は変わりません。キャプチャ唯一のアーキテクチャ上の逸脱、すなわち起動済みターゲット向けにリクエストをまたいで保持するライブの `Driver` は、キャプチャモードのセッション内に閉じ、純粋な resolver / emitter コアの外に留まります（BE-0012）。エディタと生のテキストエリアは、実行が取得済みの成果物に対してステートレスなままです。

### prime directive、Tier 分け、ゲート

この構えはすべての prime directive を保ちます。

- **AI は著述し調査するが、決して判定しない。** AI ループの関与は、Tier 1 の著述（意図 → ステップ + アサーション）と、補強差分の *提案* に閉じます。キャプチャとエディタはモデルを呼びません。どの入り口も決定的な `run` / CI ゲートに LLM 呼び出しを入れず、pass/fail は機械のみのままです。
- **まず決定性。** どの入り口も安定 `id` で選択し（同じ安定性ラダーを下り）、固定 sleep ではなく条件待機を使い、曖昧なセレクタは推測せず提示します。同じ `resolve_unique` のルールを、キャプチャとエディタは著述時に、ランナーは実行時に適用します。
- **app-agnostic。** 単一の `Scenario` 形式と単一の保存経路がツールをターゲット非依存に保ちます。3 つとも `targets.<name>` を読み、アプリのシナリオディレクトリへ同じスコープを通して同じ方法で書くので、アプリ固有のものが入り口へ漏れません。

Tier 分けは鍵となる境界で明快に割れます。**AI の部分はキーを要し**（`record` ループと補強モード）、**構造的な部分は要しません**（キャプチャ、エディタ、結果の実行）。いずれも Tier 2 の run / CI ゲートへの経路ではありません。3 つとも、決定的なランナーがその後で単独で判定するシナリオを、著述または編集するだけです。

## 検討した代替案

* **各入り口を、別々のシナリオ形を持つ別々のコマンドに分ける。** 不採用です。乖離した形式は `run` / `codegen` / レポートを断片化し、「人間が所有する単一の YAML」モデルを壊します。共有の単一 `Scenario` と単一の `ScenarioScope.save()` 経路こそが、入り口を競合ではなく相補にし、ばらばらではなく組み合わせ可能にします。
* **キャプチャ（またはエディタ）自身にもアサーションを推論させる（ヒューリスティックに、またはインラインの AI 呼び出しで）。** 不採用です。意図の推論は AI の仕事であり、キャプチャした、または編集したファイルに黙って焼き込むのではなく、レビュー可能な提案として届かねばなりません。キャプチャとエディタを純粋に構造的に保つことが、オフラインで API キー不要という保証を守り、アサーション推論をすべて AI ループの補強モードに通すことが、「AI は差分を提案し、人間が承認する」という約束を 1 か所にまとめます。
* **1 つのシナリオに対するモードではなく、`serve` 内の 3 つの別々の道具にする。** 不採用です。別々の道具はピッカー / doctor スコアのコンポーネントと保存の配管を重複させ、どの入力を持っているか分かる前に道具を選ばせてしまいます。1 つの開いているシナリオに対するモードなら、コンポーネントを共有し、作業の都合に応じて切り替えられます。
* **棲み分けを文書化せず、運用に任せて自然に落ち着かせる。** 不採用です。3 つの著述の入り口が近接して登場する以上、境界を明示しないことこそが、重複コードとユーザーの混乱を生みます。本提案が先回りして防ごうとしているコストそのものです。
