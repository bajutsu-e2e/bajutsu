## はじめに

[`implement-be`](../../.claude/skills/implement-be/SKILL.md) スキルを拡張し、実装が決定的なゲートを
通過したあとの一連の後処理をスキル自身が進めるようにします。ドラフトの Pull Request を開き、セッションを
compact して文脈を解放し、その後 [`pr-followup`](../../.claude/skills/pr-followup/SKILL.md) スキルを
一定間隔でループさせ、PR が静かになりゲートが緑になるまで対応を続けます。現在の `implement-be` は「ブランチへ
push したら人間が PR を開く」で止まっており、CI ログの確認、レビューへの返信、コンフリクトの解消といった後続の
一巡は、そのたびに手で起動されています。この項目は、こうした手動の受け渡しを、それが純粋な手間にしかならない
一つのスキル、すなわち出力が常に自己完結しゲートも緑な `implement-be` に限って取り除きます。

これは**スキルと working agreement だけの変更**です。プロダクトコード（`bajutsu/`、`BajutsuKit/`、runner、
driver）には触れません。変更はすべて author と investigation の経路にとどまり、`run` や CI の判定には
関与しません。

## 動機

`implement-be` は、すでに明確な状態で終わります。ブランチが push され、`make check` が緑になり、ロードマップ
項目が `Implemented` に変わった状態です。ここから先に残る作業は、機械的で反復的なものです。

1. 人間がドラフト PR を開く。
2. CI が走る。落ちたら、誰かがログを読み直して修正を push する。
3. レビュアーがコメントする。誰かが答え、修正し、スレッドを解消する。
4. `main` が進む。コンフリクトが発生したら、誰かが rebase する。

1〜3 は `pr-followup` が自動化できる作業ですが、その起動は注意を奪う手動ポーリングです。作業者は
CI が終わったか、レビューが届いたかを確認し続けなければなりません。4（rebase）は、この提案でも
人間が行う手動ステップのままです。bajutsu 固有の `pr-followup` スキルは、rebase や force-push を
明示的に行わない設計だからです。コンフリクトが検出されたら即座にエスカレートし、人間が rebase して
ループを再起動します。このポーリングは手間であるうえに、長く続く
巨大な文脈を抱えたセッションの使い方としても効率が悪いものです。implement セッションは設計の会話をまるごと
抱えているため、手動の後続ターンはそのたびに高価な文脈を読み直します。

次の二つの変更が、この手間を取り除きます。

- **auto-PR と自動 followup** は、受け渡しを一続きの流れにします。implement → ゲート → ドラフト PR → followup
  ループとつながり、人間が呼ばれるのは*判断*が要るときです。設計変更を求めるコメントへの対応、コンフリクト発生時の
  rebase、および最終的な Ready 化がそれにあたります。
- **ループ前の compact** は、この流れのトークンコストを下げます。implement フェーズの文脈（設計のやり取りや
  ファイルの読み込み）は、PR ができてしまえば不要な重荷です。followup ループに入る前に compact しておけば、
  ポーリングの各ターンは implement のトランスクリプト全体ではなく、痩せた文脈に対して走ります。長いセッションの
  トークン経済は、このプロジェクトで継続的に気にかけている点です。

適用範囲は意図して狭く、**`implement-be` に限ります**。BE を作成するスキル
（[`ideation`](../../.claude/skills/ideation/SKILL.md) と
[`propose-and-build`](../../.claude/skills/propose-and-build/SKILL.md) の提案フェーズ）は auto-PR しては
いけません。提案 PR は人間のレビューの分岐点であり、その BE id は人間がマージしたあとに初めて採番されるからです
（[BE-0089](../BE-0089-merge-time-be-id-allocation/BE-0089-merge-time-be-id-allocation.md)）。この分岐を
明示するため、[`CLAUDE.md`](../../CLAUDE.md) の working agreement も更新します。

## 詳細設計

作業はドキュメントと working agreement の編集であり、実行されるプロダクトコードはありません。以下の単位は
MECE です。

### Unit 1 — implement-be の step 10 でゲート通過後にドラフト PR を自動で開く

現在の「頼まれたときだけ PR」の step を、**auto-PR** の step に置き換えます。step 9 の `make check` が緑に
なりブランチが push されたら、次を行います。

- `gh pr create --draft` でドラフト PR を開きます。タイトルと本文は既存の規約に従う英語です（`[BE-NNNN]`
  接頭辞、テンプレートに沿った十分な本文、`make check` の検証行）。
- 両言語の BE ファイルの `Implementing PR:` 行に実際の番号を入れ、その追従を push します（既存の step 10 の
  要件を、無条件で行うようにしたものです）。
- 「ユーザーが頼まない限り PR を開かない」という但し書きは、**このスキルに限り**取り除きます。`CLAUDE.md` の
  ドラフトで作成する規則と、緑になるまで ready にしない規則はそのまま生きます。PR はドラフトとして作られ、
  ready にするのは人間（またはあとの明示的な step）だけで、CI が赤いあいだに自動で ready になることはありません。
  例外：`implement-be` の成果物がドキュメントのみ（スキル・`CLAUDE.md`・ロードマップの文章など、プロダクト
  コードなし）の場合は、`CLAUDE.md` 既存の「ドキュメントのみの PR は Ready for review で開く」規則が優先します。
  この場合は Draft ではなく Ready で開き、`steering-committee` をレビュアーに指定します。この項目自体の実装は
  まさにこの例外が当てはまるケースです。

### Unit 2 — followup ループを subagent に隔離して軽く保つ

followup の作業に implement フェーズの文脈（設計の会話やファイルの読み込み）は不要で、引き継ぐべき状態は PR と
そのブランチ、BE ファイルだけです。トークンを多く使う followup の作業を、周回のたびに implement のトランスク
リプト全体の上で走らせることは、長寿命のセッションでこのプロジェクトのトークン経済が気にかけているコストそのもの
です。

元の提案が記録した意図は「手渡しの時点で一度 compact する」で、具体的な呼び出し方法は未解決のままでした。実装は
この未解決の問いを `/compact` では解けないと結論づけました。**スキルは実行途中にスラッシュコマンドを発行でき
ない**ため（[claude-code#68629](https://github.com/anthropics/claude-code/issues/68629)）、ループが自分自身を
compact することはできません（外部の SDK ドライバであれば `/compact` を入力として送れますが、これは *Alternatives
considered* に記すとおり、この項目が意図して選んだセッションローカルな `/loop` とは別の、より重い実行モデルです）。
同じトークン経済の利得は、代わりに**構造で**得ます。各周回の `pr-followup` の作業を**新しい subagent の文脈**
（Unit 3）で走らせ、CI ログや差分、レビューコメントの読み込みや修正といった重い調査が、implement のトランスク
リプトや直前の周回のログを一切引きずらないようにします。セッションに残るループ層は1周回あたりほとんど何もせず
（`gh` 呼び出しが一度と、subagent の短い要約を読むだけ）、自身の周回も軽いままです。将来の編集者がこの隔離を
「最適化」で消してしまわないよう、その理由をその場に書き残します。

### Unit 3 — 一定間隔の pr-followup ループと、その停止条件

`implement-be` はセッションに、followup のペース配分を組み込みの **`/loop` スキル**で行い、**各周回の作業を
subagent に委譲する**よう指示します（Unit 2）。具体的には、「`/loop run pr-followup for #NNN` を実行してください」
とセッションに伝えます。ペース配分は `/loop` スキルが `ScheduleWakeup`（`/loop` dynamic mode のハーネスの
self-pacing 機構）で担います。各周回でループ層は `mergeable` を確認し、subagent（`Agent` ツール）を起動して
その prompt に `pr-followup` の手順を直接渡し（`Skill` ツールはメインの会話でしか動かないため、subagent は
`/pr-followup` を自分では起動できません）、その subagent の短い要約を読んで停止条件を評価します。間隔は
標準のキャッシュウィンドウの指針に従い、CI が動いているあいだ（実行の完了を待つあいだ）は短く、人間のレビューを
待つあいだは長くします。

ループが**停止する**のは、次の**すべて**が成り立つときだけです。

1. **CI が緑**。必須チェックがすべて通っています。人間しか満たせない必須チェック（たとえば必要な承認数。BE 提案
   に対する「2件の承認」ゲートなど）はループでは緑にできません。それだけが赤で残り、かつレビュー面が静かな場合、
   ループはその PR を「承認待ちで静かかつ緑」とみなして停止し、まだ人間を待っているものを報告します。CI 待ち上限
   まで空回りさせることはしません。
2. **「変更を要求」レビューが残っていない**。`reviewDecision` が `CHANGES_REQUESTED` でないことを確認します。
   レビュアーはインラインコメントを付けずにトップレベルの「変更を要求」レビューを提出できます。その立場での
   拒否が残っている間は、2回静止条件が満たされても、ループを停止してはなりません。注記：`pr-followup` は
   インラインコメント（`position != null` フィルタ）しか読まず、`/pulls/{pr}/reviews` は参照しないため、
   トップレベル body だけの異議は `pr-followup` には見えません。ループ層は履歴で判別します。`CHANGES_REQUESTED`
   かつアクティブなインラインスレッドがゼロという同じ観測状態が、二つの異なる状況で現れるためです。**元々
   解決すべきインラインスレッドが無かった**場合は、マージコンフリクトと同様に**即時エスカレート**します
   （`pr-followup` が対応できるものが何も無いためです）。**インラインスレッドがあり**、それがすべて解決された
   のに決定が残っている場合は、代わりに**催促**します。GitHub の `CHANGES_REQUESTED` 状態は**同じレビュアー**
   が再レビューするかメンテナが stale レビューを却下しない限り解除されず、インラインスレッドを resolve しても
   `reviewDecision` は変わらないためです。ループはレビュアーへ再レビューを依頼する PR コメント（`gh pr comment`）
   を投稿しますが、この催促は stale レビュー一件につき一度だけとします（直近の PR コメントがすでにこの催促で
   あれば再投稿しません）。こうすることで、数時間離席しているだけのレビュアーに 20〜30 分おきの周回のたびに
   同じコメントで通知が飛ぶことを防ぎます。
3. **レビューの新規コメントがない周回が2回連続**。レビュー面が静かになった状態です（1回の空振りでは足りず、
   2回目で静止を確かめます）。

停止条件が成り立ってループが終わっても、**Draft → Ready への遷移は人間が行います**。ループは PR が
静かになりゲートが緑になったことを報告しますが、`gh pr ready` は自身では呼びません。会話の内容を確認し、
見落とした懸念がないかを判断したうえで準備完了にするのは人間の最終確認です。「hands-free」が指すのは機械的な
後処理（CI 修正、コメントへの返信）であり、rebase とマージへの判断は人間に残します。

さらにループは、次の二つの場合にも**停止して人間へエスカレートします**。

- `pr-followup` が設計や仕様の変更を要するコメントに当たったとき。これは `pr-followup` の既存の
  エスカレーション規則そのままで、上の停止条件よりも優先します（設計の判断はループではなく人間のものだからです）。
- **マージコンフリクトが検出された**とき。このチェックは **`implement-be` のループ層**が担います。
  `pr-followup` 自体は変更しないため（Unit 5）、今の `pr-followup` は `mergeable` を参照していません。
  各周回の先頭でループ層が `gh pr view --json mergeable` を照会し、`CONFLICTING` であれば subagent
  を起動せずに即エスカレートします。エスカレートするのは `CONFLICTING` のときだけです。GitHub は
  マージ可否を非同期に計算し、算出中（たとえば push の直後）は `UNKNOWN` を返すため、ループは
  `UNKNOWN` を「まだコンフリクトなし。`pr-followup` へ進む」とみなし、次の周回で再確認して、GitHub が
  確定したあとに本当のコンフリクトを捉えます。bajutsu 固有の `pr-followup` スキルは rebase や
  force-push を行わず、人間が rebase してコンフリクトを解消したあとにループを再起動できます。

無限ループを防ぐ歯止めは二つあります。

- **レビュー待ち上限：20周回**（推奨 20〜30 分ケイデンスで約 7〜10 時間、`ScheduleWakeup` 上限の 3600 秒でも
  最大 20 時間）。CI 待ちのポーリングはこのカウントに含めません。
- **CI 待ち上限：30周回**（CI が緑にならないまま連続する場合）。不安定なインフラ、無関係な外部要因、効かない
  修正の繰り返しなど、`pr-followup` 単独では解消できない理由で CI が赤のまま続くケースに対応します。
  `pr-followup` 自身のエスカレーション規則は設計・仕様変更コメントにのみ反応するため、この独立した CI 待ち
  上限がなければ CI 待ちフェーズが際限なく続くおそれがあります。レビュー待ちと CI 待ちの周回は別々に計上します。
  **分類ルール**：いずれかの必須チェックが緑でない周回は CI 待ちとして計上します（レビュー活動の有無に関わらず）。
  必須チェックがすべて緑の周回はレビュー待ちとして計上します。つまり PR を開いた直後（CI 実行中・レビューなし）
  という典型的な状態は CI 待ちとして数えます。

いずれかの上限に達したらループを続けるのではなく、現在の状態（CI 状態、未解決コメント数）を報告して終了します。
セッションを止めれば人間はいつでもループを中断・再起動できます。セッションローカルの `/loop` をスケジュール型
クラウドエージェントより選んだのは、設定が簡単で、いつでも割り込みが可能なためです。

Prime directive の確認として、ここには `run` や CI の判定に LLM を載せる箇所はありません。`pr-followup` の
修正は依然として `make check` と CI が判定し、ループはその決定的なチェックを*予定に組む*のとレビュアーへ答える
だけです。エスカレーション規則により、本当の判断はすべて人間に残ります。

### Unit 4 — 新しい分岐に合わせて CLAUDE.md を整合させる

working agreement の PR 規則を更新し、二つの経路を明示的かつ一貫して述べます。

- **BE 作成の作業**（`ideation` や `propose-and-build` の提案フェーズによる提案 PR）：これまでどおり PR を
  自動作成せず、push して人間を待ちます。その理由（提案は人間のレビューの分岐点であり、id はマージ時に採番される）も
  書き添えます。
- **BE 作成以外の実装**（`implement-be`）：Unit 1〜3 のとおり、**ドラフト PR を自動作成し、そのあと
  pr-followup ループを走らせます**。

既存の「Claude Code が作る PR は必ずドラフトで始まる」規則と「赤いあいだは ready にしない」規則は保ち、相互に
参照させます。新しい auto-PR の経路もこれらを引き継ぎます。

### Unit 5 — スキル同士の相互参照に変更を反映する

`implement-be` の References と `pr-followup` スキルの位置づけに、両者が `implement-be` において自動化された
後処理（implement → PR → followup ループ）へと組み合わさる旨の短い注記を加えます。どちらのスキルの読者も、
この流れに気付けるようにします。ほかのスキルの挙動は変わりません。

## 検討した代替案

- **すべてのスキルで auto-PR する（`ideation` や `propose-and-build` を含む）**。却下します。提案 PR は
  意図された人間のレビューの分岐点であり、その id は人間がマージしたときに初めて採番されます（BE-0089）。自動で開くと、
  この分岐点が損なわれます。`implement-be` に絞ることで、出力が明確に準備できている場所に自動化を置きます。
- **受け渡しの時点で一度 `/compact` する（元の意図）**。実現できません。スキルは実行途中にスラッシュコマンドを
  発行できないため（[claude-code#68629](https://github.com/anthropics/claude-code/issues/68629)）、ループが自分
  自身を compact することはできません。Unit 2 はトークン経済の狙いを保ちつつ、それを構造で達成します。各周回の
  `pr-followup` を新しい subagent の文脈で走らせ、implement のトランスクリプトを重い作業が読み直さないようにします。
- **外部の SDK プログラムでループを駆動し、周回のあいだに `/compact` を送る**。Claude Agent SDK は入力として
  送られた `/compact` を実際に受け付ける（`query({prompt: "/compact", options: {continue: true}})` を送ると
  `compact_boundary` システムメッセージが返る）ため、これは動きます。ただし下の CI ボット案と同じ理由で却下します。
  独自のスケジューリング context を持つ外部オーケストレータという、別で重い実行モデルになるためです。この項目は
  意図して、より単純で中断可能なセッションローカルの `/loop` を選びました。将来ループをセッションの外へ移す項目が
  出たときに再検討する道として記録します。
- **隔離を省いて、セッション内でそのままループする**。トークン経済の理由で却下します。implement のトランスクリプト
  は大きく、followup には無関係なので、周回ごとの高価な調査をその上で走らせると毎回不要文脈を読み直すことになり
  ます。subagent への隔離は、安価で大きな削減です。
- **最初の静かな周回で止める（CI 緑・コンフリクトなし・新規コメント0が1回で停止）**。性急すぎるため却下します。
  1回の空振りは、コメントを書いている最中のレビュアーと競合しえます。2回連続の静かな周回を求めるのが、合意した
  やや保守的な停止です。
- **単一の固定間隔でポーリングする**。却下します。CI 待ちとレビュー待ちは自然な周期が大きく異なります。適応的な
  間隔（CI が動くあいだは短く、レビュー待ちのあいだは長く）はハーネスのキャッシュウィンドウの指針を尊重し、
  無駄な高速ポーリングと反応の鈍さの両方を避けます。
- **スキルではなくプロダクトコードや CI ボットとして作る**。範囲外であり、必要以上に重いため却下します。この
  流れはエージェントの working agreement であり、`pr-followup` はすでに存在します。スキルの中に置けば新しい
  サービスは要らず、あらゆる修正を既存のゲートが判定する形を保てます。
