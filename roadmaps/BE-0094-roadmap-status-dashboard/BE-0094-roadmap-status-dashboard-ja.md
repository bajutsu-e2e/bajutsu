## はじめに

既存の API リファレンスと並べて GitHub Pages へ公開する、ロードマップのダッシュボードページです。すべての BE 項目を、ライフサイクルの区分（実装済み、実装中、提案、保留）とトピックでまとめたカードとして表示します。ページはドキュメントをビルドするたびに各項目のメタデータから生成するので、コミットされたロードマップを常に反映し、古くなることがありません。

## 動機

ロードマップにはすでに唯一の出典があります。`roadmaps/<category>/BE-NNNN-<slug>/` の各項目が持つメタデータブロックです。そして [BE-0043](../BE-0043-conflict-resistant-file-flow/BE-0043-conflict-resistant-file-flow-ja.md) の生成器が、そのメタデータから `roadmaps/README.md` の索引表を再生成します。この索引は網羅的ですが密度が高く、上から順に読み下す平坦な表の集まりです。コントリビュータが最もよく抱く問い、すなわち「いま何が進行中か」と「作業はトピックごとにどう分かれているか」に、この索引は答えるのが遅いのです。今は複数の表を走査し、自分で数える必要があります。

ダッシュボードはこの二つの問いに一目で答えます。上部に区分ごとの件数を置き、続いてライフサイクルとトピックでまとめたカードを並べ、各カードが提案本体へのリンクになります。これは索引がすでに管理しているのと同じデータに対する、もう一つの見せ方です。新しい出典を増やすのではなく、既存の出典を「ロードマップの全体像を眺める」用途に合わせて描き直すだけです。

これを安全にしている制約は、索引が従っているものと同じです。すなわち、ページは生成物であって、手で書くものではありません。手で保守する状況ボードは、項目が出荷された瞬間に誰も更新せず、たちまち実態とずれます。ビルドのたびに項目のメタデータからページを生成すれば、この破綻はそもそも起こりません。ダッシュボードを変える唯一の方法は項目のメタデータを変えることであり、変更はまさにそこに属するからです。

## 詳細設計

**出典。** 生成器は索引生成器のローダ（`scripts/build_roadmap_index.py` の `load_items` と `BUCKETS` と `TOPICS`）を再利用します。そのためダッシュボードと索引はロードマップを同一に解釈し、項目の区分やトピックについて食い違うことがありません。項目のライフサイクル区分は [BE-0078](../BE-0078-roadmap-status-folders/BE-0078-roadmap-status-folders-ja.md) のとおり `Status` から導出し、手で設定するフィールドには依りません。

**ビルド生成物であり、コミットしません。** `scripts/build_roadmap_dashboard.py` は `docs/api/roadmap.md` を書き出します。これはインラインスタイルとデータを埋め込んだ自己完結の HTML ページで、生成される API リファレンスの `site/` と同じく gitignore します。この選択が設計の要です。

- **古くなりません。** `make docs` と `make docs-serve` は先にこのページを再生成し、`docs` ワークフローも公開前に再生成するので、公開されるページは常にコミットされたロードマップと一致します。
- **BE ID の採番と切り離せます。** プレースホルダの項目は、CI がプルリクエストで本番の番号を割り当てる（[BE-0061](../BE-0061-be-id-allocation-hardening/BE-0061-be-id-allocation-hardening-ja.md)）まで、文字どおりの `BE-0094` という ID を持ちます。ディレクトリ名の正規表現（`^BE-\d{4}-`）は `BE-0094` を弾くので、索引と同様にプレースホルダはダッシュボードからも除外されます。さらにページはコミットしないため、採番が後から `BE-0094` を `BE-NNNN` に書き換えても、再描画すべきものは何もありません。

**描画。** カードはカテゴリ（トピック）を主としてまとめ、並び順は索引のトピック順に従います。各カードは自分の状態を色とバッジ（実装済み、実装中、提案、保留）で示し、メタデータが `Origin` を持つ項目にはその注記を添え、項目の英語ファイルへ GitHub 上でリンクします。各カテゴリの見出しには、進捗の数値と状態構成の積み上げバーを置きます（次項を参照）。項目がすべて実装済みのカテゴリは、ほかと分けて下部の「Completed」グループにまとめるので、主な表示は進行中の作業になります。ページは MkDocs の `nav` に追加するので、新しいホスティングを立てずに既存の Pages サイトに収まります。

**カテゴリごとの進捗。** 各カテゴリには、その項目のうち実装済みが占める割合、すなわち `Implemented` の件数をカテゴリの総数で割った値をパーセントで示し、横に状態別の件数を表す積み上げバーを添えます。進捗を数値で表すのはこの一つだけで、ロードマップがすでに管理している `Status` フィールドだけから導きます。したがって出典があります。これは項目集合についての事実であって、見積もりではありません。割合の隣には内訳（`N/M implemented`）を出すので、数値の根拠が見えます。

**操作性（プログレッシブエンハンスメント）。** このページにある唯一の JavaScript である小さなインラインスクリプトが、操作を加えます。読み込み時にすべてのカテゴリを畳んでコンパクトな概観にするので、最初に表示されるのはロードマップ全体の姿を一目で捉えた状態です。そこから先は次のように操作します。見出しを押すとそのカテゴリのカードが開きます。上部の件数サマリは状態のチェックボックスを兼ねていて、各状態は独立したオン・オフのチェックボックスです（初期状態はすべてチェック済み）。一つのチェックを外すとその状態のカードをすべての場所から取り除き、空になったカテゴリやグループを隠します。すべての状態にチェックを戻すと、畳んだ概観に戻ります。畳んだ状態はスクリプトが付けるものであって、マークアップには埋め込みません。そのためスクリプトを切ってあればすべての状態がオン、すべてのカテゴリが開いたまま、すべてのカードが表示され、JavaScript なしでもページは問題なく読めます。取得や計算はせず、すでに描画済みのマークアップを表示したり隠したりするだけです。

**データの誠実さ。** 表示するのはメタデータが持つ事実だけです。状態、トピック、Origin、タイトル、件数、そしてそれらから導いたカテゴリごとの進捗です。越えない一線は、項目ごとの完了度の数値（「BE-0041 は 85% 完了」など）です。その種の数値はどの項目のメタデータにも存在しないので、描画時に作り出せば、出典のない数字を公式なページに載せることになります。カテゴリごとの割合は種類が違います。これはカテゴリ内の `Implemented` を総数で割った値であり、状態の件数についての事実であって、見積もりではありません。より細かい進捗を知りたい読者はカードから項目へたどります。そこでは `Implementing PR` の行と本文が実際の経緯を語ります。

**プライムディレクティブとの整合。** これはドキュメントのためのツールです。ドライバにも、ランナーにも、run と CI のゲートにも触れず、どこにも LLM を持ち込みません。ロードマップのメタデータに対する純粋で決定的な関数です。

### Pages の有効化

`docs` ワークフローはすでに `main` への push のたびにサイトをビルドし、手動実行で公開します。ダッシュボードと API リファレンスを配信するには、リポジトリで一度だけ GitHub Pages を有効にする必要があります（Settings → Pages → source を "GitHub Actions" に設定）。この一度きりの設定は本変更の範囲外です。

## 検討した代替案

- **索引と同様に、コミットしてドリフト検査するページ。** 退けました。描画したページをコミットすると、BE ID の採番機構と結びついてしまいます。CI が `BE-0094` を `BE-NNNN` に書き換えるとき、ページも再描画して再コミットしなければ、ドリフト検査が失敗します。ビルド生成物として生成すれば、この結びつきと「再生成を忘れる」たぐいの破綻をまとめて取り除けます。
- **独立したダッシュボードのサイトや URL。** 退けました。二つ目の Pages の対象は、利点のないまま基盤を増やします。既存の MkDocs サイトに 1 ページ足せば、すでにあるワークフローを再利用でき、ロードマップを API リファレンスから 1 クリックの距離に保てます。
- **手で保守する静的な HTML スナップショット。** きっぱり退けました。どの項目でも出荷や状態変更があった瞬間に古くなり、常に最新であるダッシュボードという目的そのものを損ないます。
- **項目ごとの完了度のパーセント。** 退けました。その数値はメタデータに存在しないので、描画時に作り出すことになり、出典のない数字を公開ページに載せてしまいます。実際に表示するカテゴリごとの進捗はこれと正反対で、カテゴリの総数に対する `Implemented` の件数そのものであり、すべて `Status` フィールドから導きます。
