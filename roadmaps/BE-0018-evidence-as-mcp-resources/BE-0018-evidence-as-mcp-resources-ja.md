## はじめに

`manifest.json` / `report.html` 等の実行結果を、エージェントが読める MCP（Model Context Protocol）リソースとして公開します。

## 動機

MCP（Model Context Protocol）サーバはすでに `bajutsu_run` と `bajutsu_doctor` をツールとして公開して
おり、エージェントは run を*開始*できます。しかし run の価値はその出力にあります。すなわち、各ステップが
何をしたかの唯一の真実である `manifest.json` と、人間向けの要約である `report.html` です。現状ではエージェントはツールの
応答から manifest のファイルパスを知るだけで、そのファイルをプロトコル外で読みにいく必要があります。これは
プロトコルの境界を壊します。エージェントは直接ディスクにアクセスでき、run のレイアウト
（`runs/<runId>/manifest.json`）を知っている必要があり、しかも MCP サーバを動かすホストとエージェントが
そもそも同じファイルシステムを共有しているとは限りません。

証跡を MCP リソースとして公開すれば、この隔たりは埋まります。リソースは、ドキュメントをエージェントに渡す
ためのプロトコル本来の手段です。URI でアドレス指定でき、他のすべてと同じ経路で取得します。run を起動した
ばかりのエージェントは、プロトコルを離れずにその manifest を読み戻せます。これにより、「AI で書き、AI で
調べる」ループが、ローカルシェル上だけでなく MCP 越しにも機能します。

## 詳細設計

run の証跡を `bajutsu://runs/...` という URI スキームの MCP リソースとして公開します。ツールを公開するのと
同じ `bajutsu mcp` サーバが配信します。

- `bajutsu://runs/{run_id}/manifest.json`：指定した run の構造化 manifest です。
- `bajutsu://runs/{run_id}/report.html`：その run の自己完結 HTML レポートです。
- `bajutsu://runs/latest/manifest.json`：直近の run の manifest です。`bajutsu_run` を呼んだばかりの
  エージェントが、応答から run id を取り出すことなく結果を読めるようにするためです。

各ハンドラは、設定された `runs/` ディレクトリに対して run id を解決し、読み取り前に**パストラバーサルを
拒否**します。リクエストは常にそのディレクトリ内のファイルにのみ解決され、上位には決して抜けません。存在
しない run やアーティファクトに対しては空のドキュメントではなくプレーンなエラーを返すので、エージェントは
「該当する run がない」と「run が manifest を生成しなかった」を区別できます。

これはプライムディレクティブの内側に収まります。リソースは**読み取り専用**です。決定的なランナーがすでに
書き出した証跡を見せるだけで、LLM が合否に影響を与えうる経路は一切増えません。エージェントが読む manifest
は、run/CI ゲートが判定したのとまったく同じ `manifest.json` です。リソースは配信チャネルであって、第二の
真実ではありません。リソース配信は run ループと直交するため、ゲートは完全に LLM フリーのまま保たれます。

`runs/` ディレクトリはサーバ単位の設定（サーバに指定したディレクトリ）であり、アプリ単位ではありません。
どのアプリの証跡も同じ run ツリー配下に出力されるためです。

## 検討した代替案

**`bajutsu_run` から manifest をインラインで返す。** ツールが manifest 全体をテキスト応答に埋め込む案です。
却下しました。ステップごとの証跡を含む manifest は大きく、run とともに肥大化します。エージェントが詳細を
必要とするかどうかにかかわらずすべてのツール応答を膨らませますし、*過去の* run の証跡を取得する手段が
ありません。取れるのは実行したばかりのものだけです。リソースはアドレス指定でき、必要なときに取得できます。

**エージェントにファイルパスを読ませる。** 最も単純な選択肢です。ツールはすでに manifest のパスを報告して
います。却下しました。ファイルシステムの共有を前提とし、エージェントをプロトコルの外へ押し出すからです。
MCP サーフェスの眼目は、エージェントが Bajutsu のディスク上のレイアウトを知らずに済むことにあります。
