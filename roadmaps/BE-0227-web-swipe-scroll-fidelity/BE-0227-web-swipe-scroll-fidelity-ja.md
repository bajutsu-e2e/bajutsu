## はじめに

web（Playwright）バックエンドでは、`swipe` ステップがページをスクロールしません。このジェスチャーは
本来スクロールを意図したものです。方向指定形式の `swipe: { on, direction }` は画面外のコンテンツを
表示させるためのクロスプラットフォーム共通の手段であり、record の認可エージェントにも「スクロールして
要素を表示させるには swipe を使う」と指示しています。それでも web では、このステップは何も動かしません。
本提案は、ジェスチャーを実際にスクロールを起こす入力プリミティブへ振り分けることで、web バックエンドの
`swipe` に本物のスクロールを実行させます。あわせて、その振り分けをブラウザの入力モード（ポインターか
タッチか）で分岐する構造にし、モバイルエミュレーションのモードが用意されたあと（対になる
**web-device-mode-emulation** 項目）も正しく動くようにします。

## 動機

`swipe` はシナリオスキーマ（`bajutsu/scenario/models/actions.py`）の 1 アクションであり、すべての
バックエンドが共有します。方向指定形式は、要素を解決し、バックエンド非依存のジェスチャー計算
（`bajutsu/orchestrator/actions/handlers/gestures.py`）で `(from, to)` の移動量を求め、
`driver.swipe(from, to)` を呼びます。idb バックエンドはこれを `idb ui swipe … --duration 0.2` に変換し、
OS がこれを本物のドラッグとして認識するため、リストやスクロールビューがスクロールします。一方 Playwright
バックエンド（`bajutsu/drivers/playwright.py`）は、`swipe` を素のマウスドラッグ
（`mouse.move → down → move → up`）として実装しています。

マウスドラッグは通常の web ページをスクロールしません。テキストを選択したり、カスタムのドラッグ可能要素を
動かしたりはしますが、ブラウザのスクロールコンテナはこれに反応しません。結果として web では次のことが
起きます。

- **record が意図どおりに動かないシナリオを生成する。** エージェントは「スクロールには swipe を使う」という
  指示に従い、画面外のコントロールを表示させるために `swipe` を発行します。しかし記録されたステップは何も
  スクロールしないため、後続のタップが誤った要素に当たったり、要素を解決できずに失敗したりします。
- **`run` が同じ無効操作を再生する。** record と `run` が唯一の `swipe` ハンドラを共有しているため、iOS
  ではスクロールするシナリオが、web では黙ってスクロールに失敗します。ツールが約束する決定性が、
  ジェスチャーを実行できないバックエンドによって損なわれます。
- **`codegen` が同じ欠陥を出力する。** `bajutsu/codegen_playwright.py` の `_emit_swipe_direction` は
  同一のマウスドラッグ列を生成するため、生成された Playwright テストも無効操作を引き継ぎます。

この挙動は直接検証しました。オーケストレーターが計算するのと同じ `(from, to)` で `PlaywrightDriver.swipe()`
を、高さ 3000px のページに対して実ヘッドレス Chromium 上で叩いたところ、`window.scrollY` は `0` のまま
でした。一方、同じ移動量の `page.mouse.wheel()` は期待どおりページをスクロールさせました。

このギャップが問題なのは、スクロールが自明でないページの前提条件だからです。長いフォーム、フィード、設定の
一覧など、折り返しの下に届く必要があるシナリオは、今の web では静かに壊れています。

## 詳細設計

修正の方針は、web バックエンドの `swipe` をスクロールを起こす入力プリミティブへ振り分け、その振り分けを
デスクトップのポインターブラウザとタッチ（モバイル）ブラウザの双方で正しくなるよう**入力モードで分岐**
させることです。正しいプリミティブはモードによって異なります。

- **デスクトップ（ポインター）モード**：`page.mouse.wheel(delta_x, delta_y)` を使います。wheel イベントは
  ブラウザの本物のスクロールコンテナを駆動し（前述のとおり検証済み）、ユーザーがホイールやトラックパッドで
  スクロールする操作に対応します。`swipe` の方向は wheel の移動量に対応づけます。`up` の swipe（サーフェスを
  上へ押す操作でコンテンツが上へ動く）はページを**下へ**スクロールさせ（`delta_y` が正）、物理的なジェス
  チャーと向きが一致します。移動量は、ジェスチャー計算が既に求めている画面相対の距離から決めます。
- **タッチ（モバイル）モード**：CDP（Chrome DevTools Protocol）のタッチドラッグを使います。これは `pinch` と `rotate` が既に使って
  いる経路（`_touch_drag` → `Input.dispatchTouchEvent`）とまったく同じです。タッチページをスクロールさせる
  のは本物のタッチドラッグであり、合成 DOM イベントと違ってページのタッチ／ジェスチャーリスナーを発火
  させます。`swipe` は 2 本指を 1 本指に畳んだうえで `_touch_drag` を再利用します。

**モードの選択。** 分岐は、コンテキストを生成したときの `has_touch` の値で決めます。Playwright の Python 版
`BrowserContext` はこの値の公開 getter を持たない（`new_context()` の生成時オプションにすぎない）ため、
ドライバーはコンテキストから読み戻すのではなく、生成時にタッチモードを（`self._context` と並べて）覚えておき
ます。なお `self._context` は 2 つの独立した箇所で生成されます。`_start_browser` のスタータークロージャ
（初回の起動と `relaunch()`）と、`_new_context()`（`reset_context()` と録画用の差し替え）です。したがって対に
なる項目は、`has_touch` の設定とキャッシュしたタッチモードの更新を**両方**で行う必要があります。さもないと、
（`_new_context()` ではなくスターターを通る）`relaunch()` の後に、キャッシュしたモードが新しく生成された
コンテキストと食い違うおそれがあります。今日の web バックエンドは常に素の
デスクトップコンテキストを起動する（`has_touch` は false）ため、本提案では**デスクトップ（wheel）**の経路を
実挙動として出荷し、タッチ経路はコンテキストをタッチ用に設定できるようになった時点で自動的に有効になるよう
分岐を配線します。その設定（viewport、`is_mobile`、`has_touch` を備えたデスクトップ／モバイルのデバイス
モード）を用意するのは、対になる **web-device-mode-emulation** 項目です。本項目はそれを導入しませんが、
のちの書き直しなしに両者が組み合わさるよう、振り分けを受け入れ可能な状態にしておきます。

**座標形式は変えません。** `swipe: { from, to }` は、ドキュメント（`docs/scenarios.md`）どおり素のドラッグの
最終手段のままとします。カスタムの canvas やマップのパン、ドラッグハンドルなど、文字どおりのポインター
ドラッグ自体が意図である場合のためのものであり、他のバックエンドで座標形式が占める位置と同じです
（[BE-0025](../BE-0025-coordinate-swipe-generation/BE-0025-coordinate-swipe-generation-ja.md) を参照）。
変わるのは、意味が「スクロール」である方向指定形式 `{ on, direction }` だけです。

**要素アンカーのポインタドラッグのための `drag` アクションを新設。** 方向指定 `swipe` をスクロールに
再定義したことで、実在の出荷済みユースケースが行き場を失いました。すなわち、要素を**セレクタで**掴んで
方向へドラッグする操作（リサイズ用の仕切り、スライダーのつまみなど）で、web では従来の方向指定 swipe の
マウスドラッグに依存していました（serve-UI の `panel-resize` dogfood がまさにこれです）。座標形式では
代替できません（生ピクセルのみでセレクタ解決を持たないため）。そこで実装では第一級の
`drag: { on, direction, amount? }` アクションを追加しました。方向指定 `swipe` と同じ端点計算を共有しつつ、
`driver.scroll` ではなく `driver.swipe`（本物のポインタドラッグ）へ振り分けます。web ではこれが掴んだ
要素を動かす `page.mouse` のドラッグになり、iOS / Android では OS の本物のドラッグが既にスクロールと
ハンドル移動を兼ねるので、`drag` と方向指定 `swipe` は一致します。codegen は各バックエンドで出力します
（web は `page.mouse` のドラッグ、iOS / Android は方向指定 swipe が出す `swipeX()` / `swipe(Direction, …)`）。

**codegen の整合。** `bajutsu/codegen_playwright.py` の `_emit_swipe_direction` を更新し、方向指定形式に
対して対応する wheel スクロール（例：`page.mouse.wheel(...)`）を出力させます。これにより、生成された
Playwright テストは `run` とまったく同じようにスクロールします。構造的な「シナリオからテストへ」の写像を
忠実に保つ方針であり、他のエミッターについて BE-0025 や BE-0085 が守っているのと同じ原則です。

作業分解（MECE）：

1. **`PlaywrightDriver.swipe` の入力モード分岐**：コンテキストのタッチモードで分岐します。デスクトップは
   `mouse.wheel`、タッチは `_touch_drag`（1 本指）です。今日の実挙動はデスクトップ経路です。
2. **方向から wheel 移動量への対応づけ**：ジェスチャー計算の `(from, to)`、方向、`amount` を、物理的に
   正しい向きと距離でスクロールする wheel の `(delta_x, delta_y)` に変換します。
3. **codegen の整合**：`bajutsu/codegen_playwright.py` の `_emit_swipe_direction` を更新し、方向指定形式に
   wheel スクロールを出力します。座標形式はドラッグのままです。
4. **テスト**：`tests/test_playwright.py::test_swipe` のマウス呼び出し列のアサーションを、本物のスクロール
   効果（`window.scrollY` や対象の `scrollTop` の変化）を確認するものに置き換え、方向から移動量への符号の
   対応づけも検証します。`tests/test_codegen_playwright.py` の codegen テストも追加・調整します。
5. **ドキュメント**：`docs/scenarios.md`（と日本語ミラー）および `docs/drivers.md` に、web バックエンドが
   方向指定の `swipe` をどう実現するか（デスクトップでは wheel スクロール、タッチコンテキストでは
   タッチドラッグ）を記します。座標形式の記述はそのまま保ちます。

以下のとおり、いずれもプライムディレクティブの範囲内です。

- **決定性。** 振り分けは、解決済みのジェスチャーから wheel／タッチ入力への固定的な写像であり、モデルには
  問い合わせません。`run`／CI ゲートには手を触れません。wheel スクロールは、置き換え対象のマウスドラッグと
  同じく再現可能です。
- **アプリ非依存。** アプリごとの設定はありません。写像はすべての web ターゲットで同一です。デスクトップ／
  モバイルの**モード**は、アプリごとのジェスチャーロジックではなく、ターゲット単位のブラウザ設定（対になる
  項目）です。
- **バックエンド非依存のコア。** シナリオスキーマと共有のジェスチャー計算は変わりません。変わるのは、web
  ドライバーによるジェスチャーの実現と、その codegen エミッターだけであり、アーキテクチャがこのために
  用意しているバックエンドごとの継ぎ目そのものです。

## 検討した代替案

- **常に CDP タッチドラッグを使う（モード分岐をやめる）。** 単一のタッチドラッグ経路はページのタッチ
  ハンドラを発火させますが、デスクトップのポインターコンテキストでは、合成したタッチドラッグは
  デスクトップページが期待する wheel スクロールの挙動（慣性、オーバースクロール、wheel 専用リスナー）に
  対応しません。また、ターゲットがデスクトップとして扱うデバイス上で、エミュレートした入力をタッチだと
  偽って伝えてしまいます。却下します。入力モードはデスクトップとモバイルで実際に異なり、それこそが分岐の
  存在理由だからです。
- **`page.evaluate` 経由の `element.scrollIntoView()` / `scrollBy()`。** 注入した JS でスクロールするのは
  確実ですが、ブラウザの本物の入力経路を迂回するため、ページのスクロール／wheel／タッチのリスナーが一度も
  発火しません。これは、ドライバーが `pinch` / `rotate` について合成 DOM イベントを退ける際に既に記して
  いる反対理由と同じです。ユーザーが実際にスクロールする挙動からも乖離します。忠実さを利便性と引き換えに
  するため、却下します。
- **マウスドラッグを残し、web の `swipe` はスクロールしないと明記する。** クロスプラットフォームのアクション
  が、着地済みのバックエンドで静かに壊れたままになり、record も実態と食い違うシナリオを出し続けます。
  却下します。
