## はじめに

Bajutsu の **backend-agnostic**（バックエンド非依存）の思想——*プラットフォームは 1 つのインターフェースの裏側にあるバックエンドである*——を、`email` ステップ（BE-0046）がメールを読み取る箇所にも適用します。すなわち **メールボックスも 1 つのインターフェースの裏側にあるバックエンドである** と捉えます。`email` ステップは、正規化した `MailboxMessage` を返す単一の `MailboxReader` という seam（接合点）を通してのみ受信箱とやり取りしており、その実装は現在ちょうど 1 つ、汎用的な HTTP-JSON リーダーだけです。本項目は、この単一のハードコードされた経路を、**トランスポートのプロトコル（`http`、後続で `imap`）をキーとするプロバイダレジストリ**へ置き換え、config でターゲットごとに選べるようにします。これにより、別種のメールボックスを足す作業が「runner を書き換える」ことではなく「アダプタを登録する」ことになります。BE-0104 が AI プロバイダに対して行ったのと同じ一手です。

レジストリは HTTP のリファレンスアダプタ（既存のリーダーをレジストリの裏側へ移したもの）だけを同梱し、それ以外は含みません。IMAP をはじめとする他のトランスポートは、本項目が定義する seam に差し込む後続項目へ意図的に先送りします。

## 動機

BE-0046 が汎用的な HTTP-JSON メールボックスを選んだのは、ベンダー SDK を持ち込まずにプロバイダ中立でいるためでした。その `messages` / `fields` マッピングは、*JSON を返すプロバイダ間* のキー名の違いをすでに吸収しています。しかしそれが扱えないのは、**別のトランスポート** を話すメールボックスです。とりわけ IMAP は、実在のメールサービスが共通して話す唯一の標準プロトコルであり、BE-0046 は「需要が出れば同じ `email` ステップの契約の裏側に」追加できるものとして明示的に先送りしていました。現状で IMAP の受信箱に届くようにするには、取得元で分岐するよう `bajutsu/runner/mailbox.py` を書き換えることになり、これはまさに BE-0104 が AI プロバイダについてレジストリへ置き換えた、あのファクトリの `if` 連鎖そのものです。

いま registry が適切な形である理由は 2 つあります。

- **seam はすでに存在する。** `MailboxReader`（`fetch(timeout) -> list[MailboxMessage]` の protocol）と中立型の `MailboxMessage` は、BE-0104 の `AiBackend` protocol とその中立メッセージ型に対応するメールボックス版です。レジストリの 3 層のうち 2 層はすでに実装済みで、欠けているのは name → adapter の対応表だけです。
- **決定論的なコアにトランスポートの `if` 連鎖を生やしてはならない。** ポーリングループ、開始後（after-start）のベースライン、照合と抽出、そしてタイムアウトは、純粋な `bajutsu/mailbox.py` と run loop にあり、取得元が増えても変わってはいけません。レジストリは、トランスポート固有の詳細をすべて周辺に、コアがすでに依存している唯一のインターフェースの裏側に閉じ込めます。

これは Tier-1 と config だけに関わる話で、どのプライムディレクティブにも触れません。合否の判定は引き続き決定論的な runner が下します。`email` ステップの決定論的な保証（スキューに強い開始後の選択、上限つきポーリング、正規表現による抽出）はコアの性質であり、本項目はコアを変更しません。

## 詳細設計

変更はすべて周辺に閉じます。シナリオに面する `email` の契約と、決定論的なコア（`bajutsu/mailbox.py`、`_do_email` のポーリングループ）は変わりません。

### レジストリ（`bajutsu/ai/registry.py` に倣う）

- **キーはトランスポートのプロトコルで切り、ベンダーでは切らない。** レジストリは `kind`（`http`、後続で `imap`）を、解決済みのメールボックス設定から `MailboxReader` を組み立てるアダプタへ対応づけます。`mailosaur` / `mailslurp` / `gmail` をキーには **しません**。これらの違いは JSON のフィールド名だけで、既存の `fields` マッピングが 1 段下ですでに吸収するからです。ベンダーをキーにすると、BE-0046 が退けたプロバイダ別クライアントを再び生むことになります。
- **`register(kind, adapter)` と fail-closed な解決。** 未知の `kind` は、run がメールボックスを解決する最初のタイミングで fail-closed に失敗します（明確な設定エラー）。BE-0104 の `_provider_name` が未登録のプロバイダで例外を投げるのと同じで、暗黙のフォールバックは決してしません。
- **組み込みの HTTP リファレンスアダプタ。** 既存の `build_mailbox_reader` の HTTP-JSON リーダーを、組み込みの `http` アダプタとして置き直します。同梱するアダプタはこれ 1 つのままです。

### config — 選択はシナリオではなく config に置く（`mailbox.kind`）

プロバイダは config でターゲットごとに選びます。これによりシナリオは app-agnostic かつ認証情報を持たないまま（BE-0046 の原則）に保たれ、BE-0104 が AI プロバイダを config（`defaults.ai` / `targets.<name>.ai`）で選び、呼び出し箇所ごとには選ばないのとも一貫します。

```yaml
targets:
  myapp:
    mailbox:
      kind: http            # 省略時の既定 → 既存のリーダー。したがって従来の config はそのまま動く
      url: "${secrets.MAILBOX_URL}"
      headers: { Authorization: "Bearer ${secrets.MAILBOX_TOKEN}" }
      messages: "items"
      fields: { body: "text" }
```

- **`kind` は任意で、既定は `http`。** 既存の `mailbox:` ブロックはすべて 1 バイトも変えずに動き続けます（後方互換。BE-0104 が旧来の `anthropic` というプロバイダ名を解決し続けたのと同様）。
- **認証情報とエンドポイントは config に留まり**、`${secrets.*}` として参照します。プロバイダに関する情報はシナリオファイルへ一切漏れません。

### 任意: ステップから名前で参照するメールボックス（`email.from`）

1 つのターゲットが正当に複数の受信箱を持つ場合（たとえば HTTP のテスト用メールボックスと IMAP の受信箱が 1 つずつ）、シナリオは *どの論理的な受信箱* を読むかを名前で指定します。config のエントリを名前で参照するだけで、プロバイダや認証情報を **埋め込みません**。したがってシナリオは移植可能なままです。

```yaml
targets:
  myapp:
    mailboxes:                     # 名前つきの定義（単一の `mailbox:` の上位互換）
      primary: { kind: http, url: "${secrets.MAILBOX_URL}" }
      otp:     { kind: imap, host: "${secrets.IMAP_HOST}" }
```

```yaml
- email: { from: otp, match: { subjectMatches: "code" }, extract: { var: code, bodyMatches: "[0-9]{6}" }, timeout: 30 }
```

- **`from` は任意** で、既定は唯一の、または `primary` のメールボックスです。したがって単一メールボックスのシナリオは変わりません。`from` が選ぶのは *論理的な受信箱の名前* であって、トランスポートやベンダーではありません。
- この層は **完全性のために設計へ含めますが、レジストリより後に出荷してもかまいません**。レジストリだけで「実装を分け、config で選ぶ」はすでに満たせます。名前つきメールボックスが要るのは、1 つのターゲットが複数の受信箱を持つようになってからです。

### スコープの境界

- **含む:** レジストリと fail-closed な解決、`http` のリファレンスアダプタ（既存リーダーの置き直し）、`http` を既定とし後方互換を保つ `mailbox.kind` 設定フィールド、レジストリ / 解決 / 既定に対するユニットテスト。
- **後続へ先送り:** IMAP アダプタ（クライアント、認証情報の扱い、MIME 本文のパース）と、名前つきメールボックスの `email.from` 層。いずれも本項目が定義する seam に差し込み、コアは変更しません。

### プライムディレクティブの遵守

- **run 経路に LLM を置かない。** レジストリは決定論的な name → adapter の対応表で、メールボックスの解決や取得に LLM は関わりません。判定は引き続き機械検証可能なアサーションだけから下されます。
- **決定論。** `email` のコア（開始後のベースライン、上限つきポーリング、正規表現による抽出）は手つかずのままです。設定ミスや未知の `kind` は、ステップまたは設定のクリーンな失敗であって、静かに誤った値を返すことはありません。
- **app-agnostic。** プロバイダ・エンドポイント・認証情報は `targets.<name>` の config に留まります。シナリオは同じ `email` の契約を保ち、指定するとしても論理的な受信箱の名前までです。

## 検討した代替案

- **レジストリをメールのベンダー（`mailosaur` / `mailslurp` / `gmail`）でキーづけする。** 却下。同一トランスポート内でこれらが違うのは JSON のフィールド名だけで、既存の `fields` マッピングがすでに吸収します。したがってベンダーをキーにしたアダプタは、ベンダーごとにほぼ同一のリーダーが増えるだけで、BE-0046 がすでに退けたプロバイダ別クライアントになります。トランスポート（`http` / `imap`）をキーにすれば、アダプタ数はベンダーではなくプロトコルに比例します。
- **プロバイダをシナリオの YAML で選ぶ。** 却下。シナリオをプロバイダに縛りつけ、認証情報がシナリオへ漏れる恐れがあり、BE-0046 の app-agnostic かつ認証情報を持たない原則を壊し、AI プロバイダを config で選ぶ BE-0104 とも乖離します。シナリオが指定してよいのは、config のエントリを参照する *論理的な受信箱*（`email.from`）までで、プロバイダそのものではありません。
- **単一の HTTP リーダーのまま、必要になったら IMAP 用の `if` を足す。** 却下。それは BE-0104 が取り除いたファクトリの `if` 連鎖を呼び戻し、2 つ目のトランスポートの関心事を runner に押し込みます。レジストリはいま小さな層を 1 つ足すだけで、以降のあらゆるトランスポートを差し込み式にします。
- **本項目で IMAP アダプタまで出荷する。** 却下ではなく先送り。BE-0104 の先例は「seam とリファレンスアダプタ 1 つ、2 つ目のバックエンドは後続で」でした。すでに使っているプロバイダで seam を実証することで、変更を小さく低リスクに保てます。IMAP（クライアント + 認証情報 + MIME パース）はそれ自体が独立した面であり、本項目が定義する seam の裏側で実装します。
