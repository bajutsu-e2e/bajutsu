## はじめに

出荷も保留もされていない、`状態` が `提案` または `実装中` のすべての項目、つまり**オープンな**
ロードマップ項目それぞれに GitHub Issue を起票します。以後は、その Issue が備える標準機能である
担当者（Assignees）を、「誰かがこれに取り組んでいるか、取り組んでいるなら誰か」を示す一次情報
として扱います。提案として存在した時点で Issue が起票されるので、担当者が付いていない Issue が、
今のロードマップに欠けているシグナルそのものになります。すなわち、まだ誰も拾っていない項目です。
Issue の起票とクローズは、項目の `状態` と、トラッキング用の Issue がすでにあるかどうかだけから
機械的に決まり、専用フィールドやダッシュボードを新設して人手で運用する必要はありません。

## 動機

CLAUDE.md の「並行作業で壊し合わないために」には、複数の人間とエージェントが同時にこのリポジトリ
で作業するという前提があり、「自分の担当範囲にとどまる」という指針もあります。しかし、オープンな
項目のうちどれにすでに誰かが取り組んでいて、どれがまだ手つかずなのかを一目で確認する手段は、今の
ところありません。次にどの項目に取り組むかを選ぶ開発者は、それが人間であれエージェントであれ、
重複作業を避けるために開いている PR を検索したり周囲に確認したりする必要があり、「何が未着手か」
に答える一箇所が存在しません。

ロードマップの `提案者` 欄も、この問いには答えません。この欄が記録するのは項目を最初に提案した
人物であり、実際に実装する人物とは別であることが多く、一度設定されると変わりません。項目が
`実装中` のまま、あるいは提案が拾い手を待ったまま、誰の担当かわからない状態で置かれることも
あります。

GitHub にはこの用途にちょうど合う機能がすでに備わっています。Issue の担当者です。`assignee:` や
`no:assignee` で絞り込め、Issue 一覧や通知にも表示され、`gh issue list --assignee <user>` でも参照
できます。オープンな項目それぞれの背後に Issue があれば、Issue 一覧そのものがロードマップに必要
なボードになります。担当者の付いた Issue は誰が何に取り組んでいるかを示し、`no:assignee` の
フィルタが未着手の作業のバックログになります。専用の「担当者」欄やステータスダッシュボードを新たに
考案する必要はなく、実際に欠けているのは、BE 項目と Issue を結び付ける仕組みだけです。

## 詳細設計

この設計は、追跡する二つの事実、すなわち誰が項目を担当しているか（Issue の担当者）と、トラッキング
用の Issue がすでに存在するか（その項目の BE ID をタイトルに持つ、ラベル付きの Issue）の両方に
ついて、**GitHub 自身を一次情報**とします。リポジトリ側には何も書き戻さないので、この同期処理は、
`main` へのコミットも、`roadmap-id` の main への push が必要とする bypass 用 App も要りません。
デフォルトトークンの `issues: write` だけで済みます。

1. **Issue の存在判定は、リポジトリ側の欄ではなく GitHub をキーにします。** `BE-NNNN` の
   トラッキング Issue は、`roadmap-tracking` ラベルを持ち、タイトルに `BE-NNNN` を含むオープンな
   Issue として識別します。同期処理は、BE ファイルから欄を読むのではなく、その Issue が存在するか
   を GitHub に問い合わせます（`gh issue list --label roadmap-tracking --state open --search
   "BE-NNNN in:title"`）。これがリポジトリへの書き戻しを不要にする鍵であり、最も頻繁に変わる状態である
   担当者が、GitHub がすでに管理している場所だけに存在することを意味します。

2. **ライフサイクル規則。項目の現在の `状態` だけを見る関数です。** 番号の付いた各項目について、
   - `状態` が `提案` または `実装中`（オープンな項目）で、対応するオープンな Issue が存在しなければ、
     起票する。
   - 対応するオープンな Issue が存在するのに、項目の `状態` が `実装済み` または `提案（保留）`
     （出荷済み、または保留）であれば、クローズする。保留の解除（`提案（保留）` → `提案`）は
     項目を再びオープンにするので、次の実行で Issue が再作成される。
   存在を PR の差分ではなく現在の `状態` だけから導くことで、同期処理は BE-0043 や BE-0061 で
   確立された自己修復的かつ冪等な設計に揃います。すでに整合した集合に対して二度実行しても結果は
   変わらず、一つの項目に対してトラッキング Issue が二つ生まれるマージ競合も起きません。二度目の
   実行は、一度目が作った Issue を見つけるからです。

3. **`main` で、ID が確定してから実行します。`BE-0109` はスキップします。** 新しい提案は、PR の
   あいだは `BE-0109` プレースホルダのままで、その実番号 `BE-NNNN` はマージ後に `roadmap-id` が
   `main` で割り当てます（BE-0089）。したがって実番号をタイトルに持つ Issue は PR 上では作成でき
   ません。そこで同期処理は `push: main`（パス `roadmaps/**`）で実行し、`BE-0109` の項目は単純に
   **スキップ**します。プレースホルダを改名する割り当てコミット自体が `main` 上の `roadmaps/**`
   への変更であり、このワークフローを再び起動するので、番号の付いた項目はその二度目の実行で拾われ
   ます。`roadmap-id` との実行順の依存も、改名すべきプレースホルダ Issue も生じません。

4. **新規スクリプト `scripts/sync_roadmap_tracking_issues.py`。** `promote_roadmap_items.py` と
   同じ形で、四つのフォルダすべての番号付き項目を走査し（同スクリプトの `read_status` を再利用）、
   `gh` コマンド（`gh issue create` / `gh issue close`）を介して上記の規則を適用します。`--check`
   モードは、`gh issue list` による読み取りだけを使って、Issue が欠けているオープンな項目や、
   クローズすべき Issue といったずれを報告し、何も変更しません。メンテナやダッシュボードが実行
   できる監査です。

5. **Issue の形式。** タイトルは `[BE-NNNN] <項目のタイトル>`、本文は項目の英語ファイルへの
   リンクとその `Introduction` の引用とします。ラベルは新設する `roadmap-tracking` とし、`bug`
   や `enhancement` とは別に絞り込めるようにするとともに、存在判定を一意にします。起票時点では
   担当者を割り当てません。着手する人が、他の GitHub Issue と同じように自分自身をアサインします。
   こうして Issue 一覧がロードマップの求めるボードになります。`label:roadmap-tracking
   assignee:<user>` は一人の担当分、`label:roadmap-tracking no:assignee` は未着手のバックログ
   です。

6. **新規ワークフロー `roadmap-tracking-issues.yml`。** `push: main`（パス `roadmaps/**`）で、
   `issues: write`（およびチェックアウト用の `contents: read`）を持つ一つのジョブが、チェック
   アウトしたツリーに対して同期スクリプトを実行します。ブランチへの push し返しも、`main` への
   コミットも、bypass 用 App もありません。`roadmap-promote` や `roadmap-id` がすでに行っている
   ことの厳密な部分集合です。集合がすでに整合していれば、何もしないジョブとしてグリーンのまま
   です。

7. **一度きりのバックフィル。** ロードマップにはすでに、実番号を持つオープンな提案と実装中の項目
   のバックログがあります。同期処理の最初の実行（または手動の `gh workflow run`）が、それらの
   Issue を一度に起票します。ライフサイクル規則は冪等なので、特別扱いは要りません。

8. **`make check` には含めません。** このスクリプトはネットワーク越しに `gh` を呼び出すため、
   決定的なゲートの内側では実行しません。`--check` モードは読み取りだけでずれを報告する、任意
   実行のコマンドにとどめます。`gh auth` とネットワークアクセスを要求する時点で、ゲートが意図的
   に要求しないものだからです。

9. **ドキュメント。** `docs/ai-development.md` に、既存の `roadmap-id` の説明と並べる形で、この
   ライフサイクル規則と二つのフィルタを短く追記します。あわせて、項目に着手する前にトラッキング
   Issue の担当者を確認し、拾うときには自分をアサインするよう、開発者向けの案内を添えます。

## 検討した代替案

- **トラッキング Issue を `実装中` の項目に限定する案。** この提案の初期の形は、作業が始まって
  はじめて Issue を起票するものでした。提案の大半は着手されず、提案ごとの Issue はノイズだと
  考えたためです。しかしこれは本来の目的を逆にしています。狙いは*どの提案が未着手か*を見えるように
  することであり、Issue が一つも無い項目は「着手可能」ではなく、そもそも見えません。オープンな
  項目（提案でも実装中でも）すべてに Issue を起票すれば、未着手のバックログは、開発者が自力で
  再構成するものではなく、絞り込める一級の一覧（`no:assignee`）になります。ノイズの懸念には、
  Issue を出さないことではなく、`roadmap-tracking` ラベルと担当者フィルタで答えます。

- **Issue の代わりに、BE のメタデータブロックに「担当者」欄を設ける案。** 採用しません。担当者は、
  項目の提案文やトピックといった他のメタデータに比べてはるかに頻繁に変わります。担当を引き継ぐ
  たびにファイルを編集することになれば、レビューや差分、マージ競合の種が増えます。GitHub がすでに
  無償で提供している情報のためにです。担当者と Issue の存在の両方を GitHub にキーづけることが、
  同期処理がリポジトリに一切触れずに済む理由になっています。

- **BE のメタデータに「トラッキング Issue」の逆参照欄を持つ案。** 便利さ（ファイルを読む人が
  Issue へのリンクを見られる）としては魅力的ですが、解決済みの Issue 番号を書き戻すには保護された
  `main` へのコミット、ひいては bypass 用 App が必要になり、GitHub を一次情報とする設計がまさに
  取り除いた依存を呼び戻します。この逆リンクは向きを逆にして、Issue の本文からファイルへ張るほうが
  適切です。のちに BE-0139 が、この難点を避ける「トラッキング Issue」行を追加しました。その値は
  項目の不変の ID から組み立てる Issue の検索 URL なので、そのつどの取得を必要とせず、Issue が
  起票されても、担当者が付いても、クローズされても変わりません。生成されるダッシュボード（BE-0094）が
  カードごとに出すのも、この同じ検索 URL であり、ここでかつて見込んでいた読み取り専用の
  `gh issue list` は使いません。

- **Issue の代わりに GitHub Projects のボードを使う案。** プロジェクトボードを使えば、かんばん
  風の一覧が手に入ります。ただし、項目とプロジェクトを同期する仕組みを別途持つ必要があり、
  Issues の REST API や `gh` コマンドに比べて重い Projects v2 の GraphQL API が必要になります。
  この提案を検討する過程で確認したところ、今の環境には `read:project` スコープがなく、
  `gh project list` の実行がその不足で失敗しました。Issue はすでにこのリポジトリで使われており
  （`bug_report` / `feature_request` テンプレート）、今回の目的には単純さの面でも十分です。
  将来 Projects のボードを重ねること自体は妨げず、今回の設計を変更せずに上乗せできます。

- **同期処理を `main` ではなく PR（`roadmap-promote`）で実行する案。** 採用しません。新しい提案
  は PR のあいだ `BE-0109` プレースホルダのままなので、実番号を持つ Issue をそこで作成できません。
  割り当て後の `main` で実行し、`BE-0109` をスキップすることが、すべての Issue を不変の ID で
  タイトル付けし続ける方法です。

> 作業分解（作業の単位ごとに 1 つ）に対応し、ログには変更内容と時期（古い順）を PR へのリンクと
> ともに記録します。

  ライフサイクル、`BE-XXXX` プレースホルダのスキップ、`--check`）。
- [x] 2. `roadmap-tracking` ラベル（同期処理が必要時に作成）、および Issue のタイトル・本文の形式。

- [x] 4. 既存のオープンなバックログの一度きりのバックフィル（ワークフローが `main` に載ったあと、
  最初の `roadmaps/**` への push、または手動の `gh workflow run roadmap-tracking-issues.yml` で実行）。
