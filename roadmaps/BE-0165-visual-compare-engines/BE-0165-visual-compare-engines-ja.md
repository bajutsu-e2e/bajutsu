## はじめに

`visual` アサーション（[BE-0029](../BE-0029-visual-regression-assertions/BE-0029-visual-regression-assertions-ja.md)）は、スクリーンショットをベースライン画像と比較しますが、その比較方法は単一の固定されたもの、つまりピクセル完全一致の差分に限られています。この提案は、比較エンジンを**選択可能**にします。現在の完全一致比較をデフォルトとして残したまま、サブピクセルのレンダリング差を許容する知覚的な `pixelmatch` エンジンを追加し、将来の構造的なエンジン（`ssim`）を差し込むための拡張点を用意します。どのエンジンも純粋で決定的な機械判定なので、Tier-2 のゲートの内側にとどまります。

## 動機

ピクセル完全一致の比較は false positive を生み、ビジュアルゲートへの信頼を損ないます。現在のエンジンは、いずれかの色チャンネルがわずかでも異なれば、そのピクセルを差分として数えます（`bajutsu/visual.py` は差分をグレースケールに落とし、`px > 0` のピクセルをすべて数えます）。既存の `threshold` が制約するのは、変化したピクセルの*面積*だけで、1 ピクセルが*どれだけ*変化したかは一切許容しません。次の 2 つのレンダリングの現実が、この仕組みを日常的に踏み抜きます。

- **サブピクセルの色のばらつき。** アンチエイリアスとフォントヒンティングは、グリフやアイコンの縁で RGB の値を 1〜3 ずらします。どれも人間には見えませんが、それぞれが完全に「異なる」ピクセルとして数えられ、その面積が積み上がって threshold を超えます。
- **1 ピクセルの縁のずれ。** サブピクセル単位のレイアウトの丸めが、境界線やグリフの縁を 1 ピクセル横に動かします。純粋な色の許容ではこれを吸収できません（ずれた縁は、その座標では実際に異なる色だからです）。しかし周囲を調べる知覚的なエンジンなら、これをアンチエイリアスと認識して差分から除外できます。

競合ツール（MagicPod、Autify）は、この種のノイズで失敗しないビジュアルテストを提供しており、業界標準の解決策は知覚的なピクセル単位の比較（`pixelmatch` アルゴリズム。Playwright の `toHaveScreenshot` などが採用しています）です。知覚的な差分は、モデルに委ねる判断ではありません。2 枚の画像とエンジンの選択が固定されていれば、モデルを介さず毎回同じ合否を返します。これこそが、この機能を AI 経路ではなく決定的なゲートに置ける理由です。

この設計は、全面的な置き換えではなく、あえて**選択式**にしています。完全一致エンジンは可能なかぎり厳格な判定であり、ピクセル完全一致を求めるターゲットには今も適切なので、これをデフォルトとして残し、`pixelmatch` は opt-in にします。エンジンを config レベルとアサーション単位の選択にすることは、app-agnostic の原則にもかないます。アンチエイリアスの多いアプリは `targets.<name>` でデフォルトのエンジンを設定でき、ツール・ドライバ・ランナーは変更されないままです。

## 詳細設計

`visual` アサーションの `compare` セレクタでエンジンを選び、エンジン固有の許容値をそのそばに置きます。デフォルトは `exact`（現在の挙動で、完全に後方互換）です。

```yaml
expect:
  - visual:
      baseline: home.png
      compare: pixelmatch      # exact（デフォルト）| pixelmatch
      threshold: 0.1           # 差分ピクセルの面積 % の許容（既存、意味は不変）
      colorTolerance: 0.1      # ピクセル単位の知覚的な色の許容（pixelmatch、0〜1）
      antialiasing: true       # アンチエイリアスのピクセルを差分から除外（pixelmatch）
      exclude:                 # 既存のマスク領域
        - { x: 0, y: 0, w: 390, h: 54 }
```

作業分解（MECE）は次のとおりです。

1. **スキーマ。** `bajutsu/scenario/models/assertions.py` の `VisualMatch` に `compare: Literal["exact", "pixelmatch"]`（デフォルト `"exact"`）と pixelmatch の許容値（`colorTolerance`、`antialiasing`）を追加します。`threshold` は、どのエンジンでも現在の面積 % の意味を保ちます。エンジン固有のフィールドが、そのエンジンでのみ意味を持つことを検証します。
2. **config のデフォルト。** `targets.<name>` の下にターゲット単位のデフォルト比較エンジンを持てるようにし、アプリごとのレンダリング差を config に閉じ込めます。アサーション単位の `compare` はこれを上書きします。実効エンジンは評価の前に解決します（`bajutsu/config.py` と解決箇所）。
3. **エンジンの拡張点。** `bajutsu/visual.py` をリファクタリングし、`compare_images` がエンジンで分岐するようにします。完全一致の経路はそのまま切り出し、`pixelmatch` エンジンを追加し（`colorTolerance` を用いたピクセル単位の YIQ 知覚的色距離と、`antialiasing` が有効なときの周囲を見たアンチエイリアス検出）、将来の `ssim` エンジンが書き換えではなく追加で済むよう拡張点を開けておきます。
4. **性能のための短絡評価。** まず高速な完全一致（バイト単位）の事前チェックを行い、画像が同一なら知覚的な処理を経ずに即座に合格させます。合否は常に選択されたエンジンのものです。これは速度の最適化に限られ、完全一致するものは pixelmatch でも必ず合格する（単調性が成り立つ）ため、短絡評価が結果を変えることはありません。
5. **依存欠如の明示的なエラー。** 選択されたエンジンに必要な依存（たとえば知覚的な計算）が無い場合、より緩いエンジンに黙って下げるのではなく、明確なメッセージでアサーションを失敗させます。黙って下げると合否が実行環境に依存し、CI と手元のマシンで判定が食い違って決定性が崩れます。
6. **差分出力と配線。** 差分画像は引き続き書き出します。`pixelmatch` では、除外されずに残った差分ピクセルを強調し、差分画像が実際に失敗した内容を反映するようにします。解決済みのエンジンと許容値を `bajutsu/assertions.py` の `_eval_visual` / `VisualContext` を通して渡し、使用したエンジンをアサーションの証跡に記録します。
7. **ドキュメントとテスト。** エンジンごとに決定的なユニットテスト（同一・サブピクセルのノイズ・1 ピクセルの縁のずれ・threshold 超過のケース）を、テスト内で生成した画像で書きます（fixture は使いません）。`docs/evidence.md`（および `docs/ja/` の対訳）とシナリオスキーマのリファレンスを更新します。

`ssim` は、あえてこの項目のスコープ外とします。この項目は具体的な知覚的エンジンとして `pixelmatch` を出荷し、構造的なエンジンのための拡張点を用意するにとどめます。構造的なエンジンは threshold の意味（類似度スコアか面積 % か）が異なり、独自の提案に値するからです。

## 検討した代替案

- **完全一致を知覚的比較で全面的に置き換える。** 見送りました。完全一致は最も厳格な判定で、ピクセル完全一致を求めるターゲットには正しいものです。これをデフォルトに残すことで後方互換を保ち、緩さへの切り替えを明示的な opt-in にできます。
- **ピクセル単位の色の許容のみ（アンチエイリアス検出なし）。** より小さな変更ですが、1 ピクセルの縁のずれを吸収できません（ずれた縁はその座標で実際に異なる色だからです）。`pixelmatch` は単純な色の許容を包含したうえで縁のずれも扱うので、目標としてはこちらが優れます。単純な許容は `pixelmatch` の一部分にすぎません。
- **救済フォールバック（失敗したらより緩いエンジンで再試行し、どれかが合格すれば合格とする）。** 見送りました。determinism first に反します。失敗時により緩いエンジンへ上げると、実効的な判定基準が最も緩いエンジンになり、ゲートを黙って甘くします。厳格なエンジンは、生き残る失敗を一度も出さなくなります。ゲートは黙って緩めるのではなく失敗させるべきです。採用するフォールバックは、合否を変えない上記の性能のための短絡評価だけです。
- **実行環境に応じた自動フォールバック（依存が無いときに格下げする）。** 見送りました。合否がマシンに依存し、CI と手元の実行で判定が食い違います。代わりに明示的に失敗させます。
- **SSIM をデフォルト/主要エンジンにする。** 先送りしました。その類似度スコアの意味は既存の面積 % の `threshold` と非互換で、独自の互換設計を要し、目先の false positive の問題を直すには重すぎます。ここで用意する拡張点により、SSIM はきれいな後続作業になります。
