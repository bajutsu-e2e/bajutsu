## はじめに

この項目は、読み取り専用の **AI 使用量ダッシュボード**を serve の Web UI に追加します。属性付きの使用量台帳（`ai-usage-cost-ledger` 項目が定義します）を読み、Bajutsu の AI のトークンとドルがどこに向かっているかを、プロバイダ・モデル・コマンド・シナリオごとに分解し、時系列の傾向として示すページです。これは *AI usage and cost observability* トピックの可視化の半分にあたります。記録の半分がデータを生み出し、この項目がそれを行動に移せるほど読みやすくします。

## 動機

台帳は増え続ける JSONL ファイルです。永続的で完全ですが、チームが手で読むものではありません。属性付きの使用量を記録することの意義は、それを*見る*ことにあります。あるシナリオが crawl の予算を占めていることに気付く、あるモデルが返す価値に比べて不釣り合いに高いと気付く、あるコマンドのプロバイダを切り替えれば結果を損なわずにコストを下げられると気付く、そのためです。生のログはこれらのどれにも一目で答えませんが、ダッシュボードは答えます。

Bajutsu にはすでにこのためのパターンがあります。BE-0102 は読み取り専用の実行統計ダッシュボード（テスト実行履歴の合否の集計）を serve に導入し、BE-0169 は `/metrics` の可観測性エンドポイントを追加しました。AI 使用量ダッシュボードは、同じ形を新しいデータソースに適用したものです。そして現在、`bajutsu/usage.py` は CLI の標準出力に出力するだけで、`serve.js` はトークンやコストについて何も表示しないため、これはまったく存在しない唯一の AI 使用量サーフェスです。

肝心なのは、これが可観測性にとどまることです。ダッシュボードはコストを*報告し*、人がより良いプロバイダやモデルの判断を下せるようにします。実行時に何かを決めることはなく、`run` や CI の合否判定に大規模言語モデル (LLM) を持ち込むこともありません。

## 詳細設計

作業は、次の相互排他かつ網羅的（MECE）な単位からなります。

1. **集計済み使用量の読み取り API。** 台帳を読み、要求された時間範囲について、台帳のディメンション（プロバイダ、モデル、コマンド、シナリオ）ごとに集計した値と合計を返す serve エンドポイント（例：`GET /api/usage`）。集計はサーバ側で行い、ブラウザは生のイベントストリームではなくコンパクトな要約を受け取ります。実行統計ダッシュボード（BE-0102）が使う既存の serve API の慣例に従います。

2. **使用量ダッシュボードのビュー。** `bajutsu/templates/serve.js` に新しいビューを追加します（デスクトップタイラーのレイアウト規則に沿い、保持されるペインの中に描画します）。表示するのは、見出しの合計（期間のトークンとコスト）、プロバイダ別とモデル別の内訳のテーブル/チャート、コマンド別とシナリオ別の内訳、そして時系列の傾向です。`cost = null` のサブスクリプションプロバイダは、捏造したドルの値ではなく、トークン数と「トークン単価なし」の明示的な印とともに表示します。

3. **プロバイダ/モデルの比較ビュー。** 最適化の目標に直接応えるビューです。呼び出しあたりのコストとシナリオあたりのコストをプロバイダとモデルで比較し、ある種類のタスクにどのプロバイダ/モデルが最も効率的かをチームが読み取れるようにします。これは記録されたデータの提示だけであり、それが助ける判断を下すのは人（またはのちには決定的な設定）であって、ダッシュボードではありません。

4. **台帳が空、または存在しない場合の扱い。** 台帳がまだ存在しない場合（AI 経路が一度も動いていない、または永続化を無効にしている）、エラーにせず、使用量の記録を有効にする方法を説明する明快な空状態を描画します。doctor/readiness パネルが使うのと同じ、緩やかな縮退です。

5. **テスト。** 集計エンドポイント（グループ化、時間範囲のフィルタ、`cost = null` 経路、空台帳のケース）を、フィクスチャの台帳に対してユニットテストします。純粋で Simulator を必要とせず、標準のゲートで動きます。JS のビューは、ほかの `serve.js` のビューと同じやり方で検証します。

## 検討した代替案

- **新しいビューではなく BE-0102 の実行統計ダッシュボードを流用する。** 採用しません。実行統計はテストの合否履歴を集計するもので、データソースも問いも異なります。AI のコストをそこに畳み込むと 1 つのページが過積載になります。同じ serve のダッシュボードパターンの下の姉妹ビューのほうが素直です。
- **使用量を BE-0169 の `/metrics` だけで公開し、Grafana は各自で用意してもらう。** すでに Prometheus を運用しているチームには妥当で、小さなフォローアップとしてやる価値があります。主たるサーフェスとしては採用しません。外部のツールを要し、実行統計ダッシュボードが前例を作った「serve を開けば自分の AI コストが見える」というセットアップ不要の体験を提供しないからです。両者は共存できます。スクレイプには `/metrics`、人が見るにはこのダッシュボードです。
- **Web UI のビューではなく CLI サブコマンド（`bajutsu usage`）。** より安価ですが、トピックの親の動機は*可視化*のサーフェスであり、Bajutsu が集計履歴をすでに提示しているのは serve です。CLI のレポータは、同じ台帳と読み取り API の上に、のちの追加としてありえます。
