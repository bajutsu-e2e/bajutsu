## はじめに

項目のトラッキング Issue（BE-0109）、担当者（Assignees）や議論が実際に存在する場所へのリンクを、
読み手が項目を目にする二つの場所の両方に追加します。一つは
[ロードマップ状況ダッシュボード](https://bajutsu-e2e.github.io/bajutsu/api/roadmap.html)
（BE-0094）の各カードに追加するもう一本小さなリンク、もう一つは項目自身のメタデータブロック
（`roadmaps/` 配下の `.md` ファイル）に追加する新しい行です。どちらも同じ、項目の ID だけから
機械的に組み立てる GitHub の検索 URL であり、特定の Issue 番号をそのつど取得するものではありません。
したがって、ダッシュボードのビルドも項目のファイルそのものも、ネットワーク通信、GitHub トークン、
ファイル作成後に変わり得る状態のいずれにも依存しません。

## 動機

BE-0094 はすでに各カードから 1 本のリンクを出しています。GitHub 上にある、項目の Markdown 版の
提案本体へのリンクです。一方 BE-0109 は、`状態` が `提案` または `実装中` であった（あるいは今も
そうである）項目ごとに `[BE-NNNN] タイトル` という形式の Issue を起票し、その担当者を「すでに
誰かが取り組んでいるか」の一次情報として扱います。ところが今は、項目からその Issue へ繋がる場所
がどこにもありません。読み手は一度離れて、リポジトリの Issues タブを開き、`roadmap-tracking`
ラベルと `in:title BE-NNNN` という検索の慣習をあらかじめ知っているか、その場で組み立てるほかありません。

この隙間が問題になるのは、読み手が続けて抱く二つの問い、「この項目は何か」（提案本体）と「すでに
誰かが取り組んでいるか」（Issue、その担当者）のうち、今は前者しか、読み手がいるその場では答えられ
ないからです。しかも隙間は一箇所だけではありません。読み手は項目自身のファイルにも、ダッシュボード
と同じくらいの頻度で直接たどり着きます。別のドキュメントからのリンク、PR の本文、GitHub 上の
コード検索、あるいはプレーンテキストの [`roadmaps/README.md`](../README-ja.md) 索引からです。
これらの経路はどれもダッシュボードを経由しません。リンクをダッシュボードという派生した表示だけに
置けば、こうした経路のすべて、そして項目のファイルそのものは、リンクを持たないままになります。
隙間を出典である項目のファイル自身で埋めれば、たどり着き方によらずすべての読み手に届きます。
ダッシュボードのリンクは、その同じ事実を、すでに表示している提案本体へのリンクの隣にもう一度
出すだけになります。

## 詳細設計

1. **機械的に組み立てる検索 URL であり、Issue のそのつどの取得ではありません。両方の表示先で
   共有します。** BE-0109 が起票する Issue のタイトルは常に `[BE-NNNN] 項目のタイトル` で、常に
   `roadmap-tracking` ラベルを持ちます。したがって項目の ID だけから、その Issue 番号を知らなく
   ても見つけられる GitHub の検索 URL を組み立てられます。
   `https://github.com/bajutsu-e2e/bajutsu/issues?q=is%3Aissue+label%3Aroadmap-tracking+in%3Atitle+"BE-NNNN"`
   このクエリには `is:open` を含めないので、Issue がまだオープンでも、後でクローズされていても
   見つかります（`提案` や `実装中` を経て出荷された項目は、クローズ済みの Issue を持ち続けます）。
   URL の組み立てに必要なのは ID の文字列だけで、`gh` の呼び出しも、GitHub トークンも、ビルド時
   や起草時のネットワーク通信も要りません。
2. **ダッシュボード側はカードの変更だけです。** `scripts/build_roadmap_dashboard.py` の
   `_card()` に、既存の提案本体へのリンクと並べて、もう一本小さなリンク（たとえば「Issue」と
   書いた小さなピル状のリンク）を追加し、上記の検索 URL へつなぎます。カードの主たるクリック対象
   である提案本体へのリンクは変えません。新しいリンクは追加するだけです。
3. **項目ファイル側は、新しい「トラッキング Issue」メタデータ行です。** 番号を持つすべての項目
   の `.md` ファイルに、フェンスで囲んだメタデータブロックの新しい行「トラッキング Issue」を
   追加し、値は同じ検索 URL へのリンクにします（`| トラッキング Issue | [検索](…) |`）。PR が
   出荷されて初めて手で埋める「実装 PR」とは違い、この項目には著者の判断がまったく必要ありません。
   値はファイル自身が持つ `BE-NNNN` の ID だけから決まる純粋な関数であり、冒頭の言語切り替え
   リンクや `# BE-NNNN — …` というタイトルと同じくらい機械的です。これは BE-0109 がすでに退けた
   「Issue 番号を書き戻す」という発想（後述の *検討した代替案* を参照）ではありません。値は Issue
   が後で起票されようと、担当者が付こうと、クローズされようと一度書いたきり変わらないので、保守
   も、Issue のライフサイクルに合わせた書き戻しも必要ありません。
4. **正規のスキーマ、生成時のひな形、既存項目への反映。** 「トラッキング Issue」は
   `tests/test_roadmap_format.py` が固定する正規のフィールド順（`ORDER_EN` / `ORDER_JA`）に加わります。
   位置は `状態` の直後、`実装 PR` の手前です。そして `提案` / `提案者` / `状態` / `トピック`
   と並ぶ必須フィールドになります。`scripts/new_roadmap_item.py` の生成内容にもこの行を追加し、
   新規に起草するすべての項目が最初から持つようにします。生成時点ではプレースホルダー `BE-0139`
   から値を計算するので、採番前は `…in:title+"BE-0139"…` のままです。これは冒頭の言語切り替え
   リンクや `提案` 行がすでにそうしているのと同じです。既存の番号を持つ項目には、一度きりの機械的な
   反映用スクリプトで、両言語のファイルに行を追加します。各値はそのファイル自身がすでに持っている
   ID だけから計算できるので、ここでも著者の判断は要らず、スクリプトを一度走らせるだけで済み、
   およそ 120 件を手で編集する必要はありません。CLAUDE.md の正規フィールド順を説明する一文も、
   新しいフィールド名を含むように更新します。
5. **`BE-0139` プレースホルダーは、追加の仕組みなしに解決します。** この行の値は、採番されるまで
   `BE-0139` を含む文字どおりのテキストなので、`scripts/allocate_roadmap_ids.py` がすでに行っている
   ファイル全体の `text.replace(old_token, new_token)`（BE-0089）が、言語切り替えリンクや
   タイトルを書き換えるのと同じ一回の処理で、実際の `BE-NNNN` への書き換えも済ませます。採番処理
   側に新しいロジックは要りません。
6. **検証。** 既存のフィールド順・必須フィールドのチェックに加えて、`tests/test_roadmap_format.py`
   に、番号を持つすべての項目の「トラッキング Issue」の値が、その項目自身の ID から導かれる検索
   URL とちょうど一致するかを確かめるチェックを追加します。機械的なだけのフィールドが崩れ得る
   唯一の失敗、すなわち別項目のファイルからのコピー貼り付けで古い値が残ってしまう事態を捕まえる
   ためです。
7. **BE-0109 の同期処理やラベル、ワークフローは変更しません。** 検索 URL は、ID と、BE-0109 が
   すでに保証しているタイトル・ラベルの慣習だけから導けるので、GitHub から何かを読み戻す必要が
   ありません。`sync_roadmap_tracking_issues.py` と `roadmap-tracking-issues.yml` はそのままです。
8. **このリンクは、検索結果が 0 件になり得ます。** BE-0109 が Issue を起票するのは、同期処理が
   走った時点でその項目が `提案` または `実装中` だった場合だけです。追加された時点ですでに
   `実装済み` として出荷された項目、いわば生まれながらに実装済みの項目は、オープンな状態を一度も
   経ないので Issue を持たず、検索リンクは 0 件になります。ダッシュボードも項目ファイルも、これを
   事前に知る手立てを持ちません。GitHub に問い合わせれば分かりますが、それは 1. で避けたネットワーク
   依存を呼び戻すことになります。そこで両方のリンクとも「Issue そのもの」ではなく「Issue
   の検索」だと明示し、0 件のページに着地した読み手が理由を理解できるようにします。
9. **ドキュメント。** 本項目のファイル自身に加えて、BE-0109 の `参考` 節へ短い相互参照を追加します。
   BE-0109 の検討した代替案にある「ダッシュボード（BE-0094）は、読み取り専用の `gh issue
   list` から項目と Issue のリンクを描画できる」という記述は、本項目が採る設計（後述のとおり
   `gh issue list` は使いません）に置き換わるものとして扱います。4. で触れた CLAUDE.md の更新も
   ここに含みます。

## 検討した代替案

- **`gh issue list` によるそのつどの取得。** ダッシュボードのビルド時であれ、「トラッキング
  Issue」行を起草する時点であれ、検索 URL の代わりに実際の Issue 番号を解決する方法です。これは
  ダッシュボードについては BE-0109 の `参考` 節がすでに見込んでいた方向です（「ダッシュボード
  （BE-0094）は、読み取り専用の `gh issue list` から項目と Issue のリンクを描画できる」）。両方の
  表示先について退けました。今の `make docs` と `make docs-serve` は、ダッシュボードを stdlib
  だけで再生成しており、ネットワーク通信も `gh auth` も要りません。`docs` ワークフローに至っては
  `issues` 権限を一切持たずにビルドしています。そのつどの取得に切り替えると、ローカルでの `gh`
  セッションと、CI 側の新しい `issues: read` 権限の両方が必要になります。項目ファイルの行を
  そのつどの取得で埋める方法はさらに分が悪く、新規の提案では実際の Issue 番号がまだ存在しません
  （BE-0109 は、ID が `main` で採番されて初めて Issue を起票します）。つまり起草の時点ではそもそも
  埋められません。検索 URL ならどちらの依存も増やさず、プレースホルダー ID が存在した瞬間から
  埋められます。
- **ダッシュボードだけに留め、項目自身のファイルには追加しない方法。** 本項目が当初持っていた
  範囲です。退けました。項目のファイルには、ダッシュボードと同じくらいの頻度で直接たどり着きます。
  別のドキュメントからのリンク、PR の本文、コード検索、プレーンテキストの索引などからで、
  そのどれもダッシュボードを経由しません。リンクを派生した HTML の表示だけに置けば、出典である
  ファイル自身と、そこへ直接たどり着くすべての読み手が、リンクを持たないままになります。
- **Issue の実際の「番号」を項目のメタデータブロックに書き戻す方法**（値が実際に解決された Issue
  へのリンクである「トラッキング Issue」行）。これは BE-0109 自身がすでに一度退けています。どの
  Issue が開いているか、誰が担当しているかという、項目のメタデータの他の部分よりずっと頻繁に
  変わる状態のために、保護された `main` への、bypass 用 App を介したコミットを強いることになり、
  「GitHub を唯一の一次情報にする」という BE-0109 の設計を崩すからです。本項目の「トラッキング
  Issue」行は見た目こそ似ていますが、中身は違います。不変の `BE-NNNN` の ID から一度だけ計算する
  URL で、どの読み手にとっても同一であり、その先の Issue に何が起きようと二度と書き換わりません。
  したがって、そのつどの取得も、Issue のライフサイクルに合わせた書き戻しも要らず、BE-0109 がすでに
  決着させた論点を蒸し返すものでもありません。
- **既存のリンクや行を Issue へのリンクに置き換える方法。** ダッシュボードなら提案本体へのリンク
  を、項目ファイルなら既存のいずれかの行を置き換える案です。退けました。提案本体、そして項目の
  既存のメタデータは、読み手がまず欲しいものです。Issue は担当状況と議論への薄い副次的な導線に
  すぎません。追加のリンク・行として足すことで、既存のクリック対象や行はどれも変わらずに済み
  ます。
