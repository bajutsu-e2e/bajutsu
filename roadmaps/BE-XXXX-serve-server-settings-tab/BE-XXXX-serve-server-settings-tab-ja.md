## はじめに

`serve` の Settings モーダルに、読み取り専用の **Server** タブを追加します。このタブは、起動中の
サーバがどう構成されているか（デプロイ形態、束縛している config とその由来、実行できる
[バックエンド](../../docs/ja/glossary.md#driver-backend-actuator-platform)、実行結果の保存先や
保持期間、同時実行数の設定）を表示し、そして最も有用な情報として、このサーバが同梱の iOS XCUITest
テストランナーを備えているかどうかを示します。表示する値はすべて、起動時に解決済みのサーバ状態と
ファイルシステムの探索 1 回から得られるため、このタブは Tier-1 かつ読み取り専用で、AI を一切
使いません。表示するものは `run` の判定に一切関与しません。

## 動機

`serve` は起動時に、コマンドラインフラグや環境変数、束縛した config から多くの設定を解決します。
しかしブラウザ側はそのほとんどを見せていません。UI を見ているだけでは「このサーバは結局どんな
設定で動いているのか」に、起動コマンドを見直さないかぎり答えられません。Settings モーダルは、
すでに公開している 2 つの運用者向けの関心事（AI プロバイダとシナリオシークレット）をまとめて
いますが、サーバ自身の設定にはそこに置き場所がありません。

この欠落が最も鋭く現れるのが、iOS テストランナーです。
[BE-0292](../BE-0292-xcuitest-bundled-runner/BE-0292-xcuitest-bundled-runner-ja.md) 以降、
`xcuitest.testRunner` を指定しない iOS [シミュレータ](../../docs/ja/glossary.md#target-app-device)
実行は、ウィールにパッケージデータとして同梱される汎用 XCUITest ランナーへフォールバックします。
ただしそれは、ウィールが実際にランナーを同梱している場合にかぎられ、素のソースチェックアウトや
macOS 以外向けのウィールは同梱しません。XCUITest が iOS の唯一のバックエンドになった今
（[BE-0290](../BE-0290-xcuitest-default-ios-backend/BE-0290-xcuitest-default-ios-backend-ja.md)）、
このランナーが配置されているかどうかが、iOS シミュレータ実行を開始できるかどうかそのものを
左右します。今のところ、ランナーが配置済みかを知る唯一の手段は、実行を始めて結果を見ることです。
「iOS テストランナー: 配置済み（Xcode 16.2 / SDK 18.2 でビルド）」、あるいは「未配置」という
1 行があれば、その混乱を未然に防げます。これはまさに、Settings モーダルが答えるべき、読み取り
専用でサーバ全体にかかわる事実です。

## 詳細設計

構造上、Tier-1 の読み取り専用で、AI に依存しません。この機能は、解決済みの `ServeState` を読み、
同梱ランナーの有無をウィール上で探索するだけです。表示するものは実行ごとのシグナルではないため、
決定的な `run` / CI ゲートには一切触れません（プライムディレクティブ 1）。

作業は MECE な 4 つの単位に分かれます。

**単位 1 — 読み取り専用のサーバ情報エンドポイント。** `GET /api/server` → `ops.server_settings(state)`
を追加します（`bajutsu/serve/operations/config.py` の `config_info` の隣に新設し、両バックエンドが
ディスパッチできるよう共有のルートテーブルに登録します。[BE-0253](../BE-0253-serve-route-registry-unification/BE-0253-serve-route-registry-unification-ja.md)）。起動中サーバの解決済み設定を
JSON として組み立てます。含めるのは、デプロイ形態（ローカルかホスト型か）、束縛した config と
その取得元および provenance（`config_info` や `config_content` がすでに算出しているものを再利用）、この
サーバが実行できるバックエンド、実行結果と baseline の保存ディレクトリ、ソフト削除の保持期間、
同時実行数の上限（全体、ユーザ単位、org 単位）、そして起動中のバージョン文字列（BE-0272 の
ヘッダバッジ向けに `server_version` がすでに返している公開値）です。このエンドポイントの認可の
姿勢は `config_info` と同じで、どの閲覧者にも開かれています。したがって、その姿勢がすでに許す
ものだけを露出しなければなりません。`server_checkout` が返す admin 限定のコミットとブランチは、
ここではあえて出しません。ブランチ名は進行中の作業の主題を漏らしうるため、`server_checkout` は
まさにその理由で admin 呼び出しの背後に置いているからです。自分のホストパスには、BE-0108 の
論拠を適用します。ホスト型の利用者はホストとファイルシステム上の関係を持たないので、ホストの
絶対パスは無意味な情報であり、不要な露出です（[BE-0108](../BE-0108-hosted-config-source-restriction/BE-0108-hosted-config-source-restriction-ja.md)）。
そこで、config のパスと、実行結果および baseline の保存ディレクトリを `state.hosted` のとき
伏せます。これは `config_info` よりあえて厳しくしています。`config_info` は自身の `config` パスを
無条件で返しますが、より広いサーバ情報の面では、それをそのまま真似るのではなく、ホスト型では
ホストパスを伏せます。パス以外の項目（デプロイ形態、バックエンド、保持期間、同時実行数の上限、
バージョン）はどちらでも表示します。公開するのは有無と設定内容だけで、シークレットの平文は決して
含めません。

**単位 2 — ペイロードへの同梱 iOS テストランナーの配置状態。** 単位 1 のペイロードに、同梱ランナー
の探索（`bundled_products_dir()` と `bundled_runner_build_info()`）から得る `iosRunner` ブロックを
追加します。含めるのは、このビルドが同梱の汎用 XCUITest ランナーを備えているか
（`bundled_products_dir()` が `None` でないか）、備えている場合にそれがどのツールチェインでビルド
されたか（`bundled_runner_build_info()` から得る Xcode と シミュレータ SDK のバージョン）、そして
束縛した config が `xcuitest.testRunner` でランナーを明示的に上書きしているか、の 3 点です。これらの
探索はプライベートなサブモジュール `bajutsu.platform_lifecycle.environments._bundled_runner` にあり、
その `environments` パッケージの `__init__` は、プラットフォーム個別のモジュールへ手を伸ばすのでは
なくパッケージルートから import するよう求めています。`server_settings` は別のパッケージにあるため、
この単位ではまず 2 つの探索を `environments` パッケージルートで再エクスポートし、そこから import
することで、オペレーション層をプライベートなサブモジュールから切り離します。これらを合わせ
れば、「ランナーを指定しない iOS シミュレータ実行はランナーを見つけられるのか、そしてそれは何を
対象にビルドされたのか」に、実行を始めずに答えられます。同梱ランナーは設計上アプリ非依存なので
（BE-0292）、この情報はアプリ単位ではなくサーバ全体にかかわるものにとどまります。ここで iOS に
だけ専用ブロックを設けるのは、その不在がバックエンド 1 つ丸ごとを黙って塞いでしまう、サーバ全体に
かかわる唯一の readiness の事実だからです。同じ形は、2 つめの一回きりのフィールドを継ぎ足すのでは
なく、後からバックエンド単位の readiness マップ（Playwright のブラウザが入っているか、`adb` に
到達できるか）へ一般化できます。ただしそれらの事実は本項目が埋める欠落ではないので、ここでは
iOS ランナーだけを挙げ、その一般化はフォローアップに委ねます。

**単位 3 — Settings モーダルの Server タブ。** `bajutsu/templates/serve.html.j2` の AI タブと
Secrets タブの隣に 3 つめのタブ（`data-settab="server"`、パネル `#setpanel-server`）を追加し、
既存の 2 つとまったく同じように `serve.core.mjs` の `showSettingsTab` に組み込みます。
`loadServerInfo()` が、モーダルを開いたときに `/api/server` を取得し、各値をラベル付きの読み取り
専用の行として描画します。iOS ランナーは、配置済みなら理由とビルド時のツールチェインを添えた
「配置済み / 未配置」の 1 行で示します。このタブに入力欄はありません。報告するだけで、何も変更
しません。

**単位 4 — テスト。** `server_settings` のオペレーション単位のテストとして、ローカルとホスト型での
ホストパスの伏せ方、同梱の有無によるランナー分岐（products ディレクトリの注入と不在）、
`xcuitest.testRunner` の上書きがペイロードに反映されることを確かめます。あわせて、`GET /api/server`
が両バックエンドでディスパッチされることを確かめるルート登録のテストを置きます。いずれも決定的で
デバイス不要です。エンドポイントは状態を読みファイルシステムを探索するだけなので、シミュレータは
要らず、コアのゲートは Linux でも動きます。

## 検討した代替案

- **doctor パネル（BE-0148）に統合する。** 却下しました。doctor は *ターゲット単位* の問い、
  すなわち、このターゲットは実行可能か、その画面はアドレス可能か、に答えるもので、起動済みデバイス
  とライブ画面のクエリを必要とします。サーバ設定ビューは *サーバ全体* にかかわり、静的で、デバイス
  不要です。問いも入力も異なり、同居させれば両方が曖昧になります。
- **バージョンバッジ（BE-0272）を拡張する。** 却下しました。バッジは起動中コミットを示すヘッダ上の
  単一の要素です。複数の値を持つ読み取り専用の設定面は、ヘッダに詰め込むのではなく、他の運用者向け
  の関心事の隣、Settings モーダルに置くべきです。
- **Settings タブではなく新しいトップレベルビューにする。** 却下しました。これは運用者の設定であり、
  Settings モーダルがすでにまとめている関心事（プロバイダ、シークレット）そのものです。別のトップ
  レベルビューにすると「どう設定されているか」を見る面が 2 か所に分裂します。
- **iOS ランナーの配置状態だけを表示する。** 検討のうえ、サーバ設定を網羅的に見せるビューを選んで
  見送りました。ランナーは、ブラウザが起動中サーバの設定をほとんど見せていないという一般的な欠落の
  最も鋭い一例なので、タブはその欠落全体に対処しつつ、ランナーの配置状態を最も目立つ行に据えます。
