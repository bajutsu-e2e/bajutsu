## はじめに

`run` が出力した一つの実行ディレクトリを、`report.html` とその `manifest.json`、`junit.xml`、実行した
`scenario.yaml`、そして証跡（スクリーンショット、動画、`network.json`、`device.log` など）すべてを含めて、
一つの `.zip` として取り出せるようにします。提供面は二つで、いずれも共通の生成ロジック（以下、アーカイバ）を
呼びます。一つは `bajutsu serve` の Web UI に置く run ごとのダウンロードボタン（History の一覧と Replay の
結果画面）、もう一つは CLI（インラインで固める `bajutsu run --zip` と、既存の run を後から固める
`bajutsu export <run>`）です。zip は `runs/<id>/` ディレクトリ全体を、内部の最上位に `<id>/` フォルダを置く形で
梱包します。こうすると `report.html` が証跡を指す**相対**リンクがオフラインで解決し、サーバなしでも（ファイルを
ダブルクリックするだけで）レポートが開きます。本項目は Tier-1 の利便機能です。どこにも LLM を足さず、`run` の
合否には一切触れず、run がすでに適用した機密のスクラブをそのまま引き継ぎます。

## 動機

`run` の出力は、それだけで完結し、自己記述的です。決定的なランナーが `report.html`、`manifest.json`、
`junit.xml`、実行したシナリオ、そして証跡ツリーを書き出します。ところがその一式は**出力したマシンの中に
閉じ込められた**ままで、完全なコピーを外へ持ち出す手立てが用意されていません。この欠落は、具体的に四つの場面で
効いてきます。

1. **`report.html` だけでは持ち運べません。** レポートは証跡（スクリーンショット、区間の動画、`network.json`）を
   **相対**リンクで参照します。ローカル Web UI がその証跡を配信してリンクを解決しているのは、まさにこのためです
   （[BE-0011](../BE-0011-local-web-ui-serve/BE-0011-local-web-ui-serve-ja.md)）。HTML だけを取り
   出すと、画像は壊れ、動画も開けないレポートになります。動くレポートを動かすには、ディレクトリを丸ごと、配置を
   保ったまま移すしかありません。
2. **失敗の共有は日常的な要求です。** 不具合チケットに run を添付する、Mac も Simulator も持たない同僚に渡す、
   リリース時の run をオフラインの記録として残す。どれも、持ち運べる一つのファイルを必要とします。今は手作業で
   `zip -r runs/<id>` を打つしかなく、これは間違えやすく（誤ったルートから固めれば相対リンクが壊れます）、その
   シェルの前にいない人には使えません。
3. **ホスト型やリモートの `serve` では、ブラウザからファイルシステムに触れません。** `serve` をリモートの worker や
   共有ホストで動かす場合（[BE-0015](../BE-0015-web-ui-public-hosting/BE-0015-web-ui-public-hosting-ja.md)、
   [BE-0016](../BE-0016-web-ui-self-hosting/BE-0016-web-ui-self-hosting-ja.md)）、`runs/` ディレクトリは
   そのマシン上にあり、利用者の手元にはありません。ブラウザの利用者には、完全な run を取り出す手段がありません。
   埋め込まれたレポートを見ることはできても、持ち帰れないのです。これを塞げるのは、ダウンロード用のエンドポイント
   だけです。
4. **CI はレポートを一つのアーティファクトとして扱いたいものです。** zip があれば、CI から実行結果を上げる作業は一つの
   ファイルに対する一度の `upload-artifact` で済みます。ディレクトリツリーのまま送って相対配置が保たれるかを気に
   する必要がありません。`bajutsu run --zip` は、合否を出したのと同じコマンドの中で「アーティファクトを作る」
   ところまでを担います。

アーカイブを扱うコードは、現在のコードベースには存在しません（`zipfile` / `tarfile` / `make_archive` の利用は
どこにもありません）。これはレポートサブシステムに欠けている**エクスポート**側の半分であり、しかも純粋な梱包の
関心事なので、決定性優先の原則とは構造上ぶつかりません。

## 詳細設計

### 一つのアーカイバ、二つの提供面

`run_dir` を渡すとその中身の zip を作るアーカイバを一つだけ用意し、CLI と `serve` の両方がそれを呼びます。こう
すれば、「run の zip」に何が入るかが二つの提供面で食い違うことはありません。置き場所は既存のレポート書き出し
（`bajutsu/report/`、`manifest.py` や `html.py` の隣）が自然です。実装は stdlib の `zipfile` に乗せ、新しい依存を
足しません。`serve` が stdlib のみで動くことや、ゲートを依存の軽いまま保つ方針と揃えます。

### 梱包範囲は run ディレクトリ全体

zip には `runs/<id>/` の下にあるものをすべて入れます。`report.html`、`manifest.json`、`junit.xml`、実行した
`scenario.yaml`、そしてシナリオやステップごとの証跡ファイルです。エントリ名は**run ディレクトリからの相対**とし、
最上位に一つの `<id>/` フォルダを置きます。展開すると `<id>/report.html` が現れ、証跡は `serve` が配信するときと
同じ配置に並びます。そのため `report.html` の相対リンクはそのまま解決します。これが、単体の HTML ではなく
ディレクトリを丸ごと固める理由です。ディレクトリの走査はソート順で行い（ファイルの mtime を固定値に揃えても
よい）、同じ run ディレクトリからは再現可能な zip が得られるようにします。ただしバイト単位の一致は、証跡
（[DESIGN §2](../../DESIGN.md)）と違ってここでは契約ではなく、あれば望ましい程度のものです。

### CLI の提供面

CLI には二つの形を用意し、いずれも同じアーカイバを共有します。

* **`bajutsu run … --zip`**：run が終わってレポートを書き出したあと、続けて `runs/<id>.zip` を出力します（その
  場所を表示します）。これは CI に向いた形で、一つのコマンドで合否とアップロード用のアーティファクトの両方が
  得られます。フラグの処理は決定的な合否が決まった**あと**に走るので、合否に影響を与えることはできません。
* **`bajutsu export <run-id|path> [-o out.zip]`**：すでにある run を固める単独のコマンドです。共有アーカイバの
  自然な置き場所であり、「前にやった run の zip がほしい」に答えます。出力先の既定は `runs/<id>.zip`（run
  ディレクトリと同じ階層）で、`-o/--output` で変更でき、既存ファイルがあれば `--force` なしでは上書きを拒みます。
  `record` が黙って上書きしないのと同じ作法です。

コマンド名の細部は小さな未決事項です。本質は、両方が同じアーカイバを呼ぶことにあります。

### serve の提供面

新しいエンドポイント `GET /runs/<id>/archive.zip` が、`Content-Type: application/zip` と
`Content-Disposition: attachment; filename="<id>.zip"` を付けて zip を返します。配信は**既存の `ArtifactStore`
抽象**（`bajutsu/serve/artifacts.py`）を通します。これはすでに run ファイルへのアクセスを閉じ込め、パス
トラバーサルを防いでいます。インタフェースに `archive(run_id)` メソッドを足し、`LocalArtifactStore` は閉じ込めた
run ディレクトリを走査して実装します。将来のホスト型ストアは自分の側で実装します（ファイルを持つ worker が
そこで固めます）。この境界を再利用することで、ハンドラがファイルシステムに直接触れずに済みます。

UI では、ダウンロードボタンを History の一覧と Replay の結果画面の、埋め込みレポート
（[BE-0011](../BE-0011-local-web-ui-serve/BE-0011-local-web-ui-serve-ja.md)）の脇に置きます。一度
押せば `<id>.zip` がダウンロードされます。run が大きい場合（動画と多数のスクリーンショット）には、ストアが zip を
一時ファイルに作ってストリーミングし、zip 全体を RAM に抱え込まずメモリを抑えます。

### 決定性、ゲート、機密

* **LLM なし、合否への影響なし。** これは、決定的な `run` がすでに出力した成果物を、事後に梱包するだけです。
  `run` の合否は zip より前に、かつ zip とは独立に決まります。`--zip` は合否のあとに梱包の一手を足すだけです。
  prime directive の 1 と 2（[CLAUDE.md](../../CLAUDE.md)）は構造上保たれます。
* **Linux でテストできます。** ディレクトリの zip 化に Simulator は要りません。アーカイバとエンドポイントは、
  フィクスチャの run ディレクトリを相手に既存の Linux ゲートで単体テストできます。
* **機密はスクラブされたままです。** 機密の値は run のあとに証跡から redact 済みです
  （[BE-0032](../BE-0032-secret-variables/BE-0032-secret-variables-ja.md)）。zip はディスク上のものを
  梱包するだけなので、そのスクラブを引き継ぎ、新たに何かを混入させません。アーカイバは**run ディレクトリだけ**を
  固め、`.env` や設定、その上位のものには決して手を伸ばしません。

### テストの契約（機械チェック可能）

アーカイバとエンドポイントは、Simulator を要しないテストで固定します。(a) zip が、単一の `<id>/` ルートの下に run
ディレクトリのツリーをちょうど含みます。(b) `report.html` と、それが参照する資産がすべて存在し、相対リンクが解決
します。(c) run ディレクトリの外のものが混入しません。(d) 既知の機密の値が、zip のどのバイトにも現れません。(e)
`serve` のエンドポイントが `application/zip` と `attachment` を付け、`ArtifactStore` を介して解決します（パス
トラバーサルが起きえないまま）。

### スコープ外

複数 run の一括梱包、軽量な「レポートと参照資産だけ」の部分集合（「検討した代替案」を参照）、クラウドへの
アップロードや共有リンク（ホスティングの関心事、
[BE-0015](../BE-0015-web-ui-public-hosting/BE-0015-web-ui-public-hosting-ja.md) /
[BE-0016](../BE-0016-web-ui-self-hosting/BE-0016-web-ui-self-hosting-ja.md)）、古い run の保持や掃除は
対象外です。

## 検討した代替案

* **`report.html` だけを zip する（自己完結した単一ファイル）。** 既定としては却下します。レポートは証跡を相対
  パスで参照するため（[BE-0011](../BE-0011-local-web-ui-serve/BE-0011-local-web-ui-serve-ja.md)）、
  HTML 単体ではスクリーンショットも動画も network ログも失われます。レポートを実際に持ち運べるようにするのは、
  run 全体の zip です。サイズが気になる場面に向けて「レポートと、それが参照する資産だけ」の軽量版を後から足す
  ことはできますが、それは上に乗せる最適化であって、安全な既定ではありません。
* **利用者に `zip -r runs/<id>` を促す。** ローカルの Mac のシェルでは動きますが、(a) ファイルシステムに触れない
  ホスト型やリモートの `serve` の**ブラウザ**利用者には何も提供できず、(b) 手作業で間違えやすく（誤ったルートから
  固めればレポートの相対リンクが壊れます）、(c)「run の zip とは何か」について UI と CI が共有できる安定した契約が
  残りません。内部配置の決まった組み込みのアーカイバは、この三つをまとめて解決します。
* **別のアーカイブ形式や、外部の梱包用依存。** 却下します。stdlib の `zipfile` はどこでも使え、どの OS でも
  ダブルクリックで開け、依存を足しません。`serve` が stdlib のみであることや、ゲートを依存の軽いまま保つ方針と
  揃います。tarball は、共有されたレポートが届きうる Windows の利用者には扱いにくいでしょう。
* **常に `run` の中で zip を作る（フラグなし）。** 却下します。大半の run はそれを必要とせず、すべての CI の run に
  I/O とディスクを足すことになります。オプトイン（`--zip`）か、必要なときだけ（`export` や serve のボタン）に
  留めます。
* **「実行レポートのエクスポート / 共有」という専用トピックの新設。** 見送ります。単一の項目のためにトピックを
  切らないという [BE-0055](../BE-0055-operational-logging/BE-0055-operational-logging-ja.md) の先例に
  ならい、当面は「オーサリング体験」の下に置きます。兄弟となる項目（クラウドアップロード、共有リンク、別の
  エクスポート形式）が現れたら、そのとき専用トピックを切り出せます。
