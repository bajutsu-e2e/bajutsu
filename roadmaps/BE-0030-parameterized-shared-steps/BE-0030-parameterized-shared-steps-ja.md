## はじめに

`use` ステップで引数付きの再利用部品（component）を定義して呼び出し、`${params.*}` を展開します（`expand_components`）。`setup` プレリュード（引数なし）と併用できます。ログイン等の共通手順の重複を排除できます。

## 動機

多くのシナリオは、ログイン、オンボーディングの承認、特定画面への遷移といった共通の前半部分を共有しています。これを各シナリオにコピーすると、同じ手順が何十回も繰り返され、フローが変わると同期が崩れ、ファイルも肥大化します。既存の `setup` プレリュード（固定の共通フローをシナリオの前に差し込む仕組み）は引数なしのケースは扱えますが、呼び出しごとに値を変えられません。シナリオごとに異なるユーザーでログインしたい、あるいは異なるタブへ遷移したいといった場合に、引数を渡す手段がないのです。1 つの定義を多くの呼び出し箇所で使い回し、値を呼び出しごとに埋められるよう、引数を取る再利用可能なステップ単位が必要です。

## 詳細設計

*コンポーネント* は、再利用可能でパラメータ化されたステップ列で、単一マッピングとして独立したファイルに定義します。呼び出し側が渡すべき引数名を並べた `params` リストと、それらを `${params.<name>}` として参照する `steps` から成ります（`bajutsu/scenario/` の `Component`）。シナリオは `use` ステップ（`use: { component: <file>, with: { <name>: <value>, … } }`）で呼び出し、宣言された各 param に `with` で値を渡します。

展開はランタイムのアクションではなく、純粋なコンパイル時マクロです。run の前に `expand_components` が全シナリオを走査し、各 `use` ステップをその場で参照先コンポーネントの steps に置き換え、`with` の値を共通の `${...}` 補間プリミティブ（`interp.interpolate`、キーは `params.<name>`）で置換します。コンポーネントはそれ自体がさらに別のコンポーネントを `use` できるため、展開は再帰します。参照の循環、不足または未知の param、宣言されていない param を指す未解決の `${params.*}` トークンはいずれもハードエラーで、`max_depth` を超えるネストも同様です。展開後は `use` ステップが一切残らないため、決定的なランナーは常に展開済みのプレーンなステップだけを見ることになり、決定性は損なわれません。`use` は引数なしの `setup` プレリュードと併用できます。プレリュードは前置（`apply_setups`）され、コンポーネントは同じロード時のパスで展開されます。

## 検討した代替案

**ランタイムでの call/return（コンポーネントをランナーが入り込むサブルーチンとして扱う）。** 不採用。挙動上の利点がないのに、コールスタックと実行中フレームの状態を run ループに持ち込み、オーサリングの糖衣構文と実行の境界を曖昧にします。コンパイル時展開なら、ランナーはフラットなステップだけを見るため、実行されるシナリオはレポートに表示されるものと完全に一致します。

**インライン限定の再利用（YAML アンカー／ファイルへのテンプレートエンジン適用）。** 不採用。アンカーは 1 ファイル内のテキストは複製できても、ファイルをまたげず、名前付き引数もきれいには扱えません。汎用テンプレートエンジンはシナリオファイルに任意のロジックを招き入れます。明示的な `params` 契約を持つ名前付きコンポーネントのほうが範囲が狭く、検証され（未知または不足の param は失敗する）、入力を自己文書化します。
