## はじめに

継続的インテグレーション（CI）の `smoke (idb)` レーンが、シナリオの最初のステップで断続的に失敗します。showcase の smoke シナリオは冒頭で最初のリスト行（`stable.row.1`）を `wait` しますが、この待機が操作を一つも行わないうちに 10 秒でタイムアウトします。Simulator は boot し、アプリは起動し、起動後の準備完了判定も通過しているにもかかわらず起こります。この失敗は毎回バイト単位で同一で、再実行すれば解消します。PR #936 では 3 回続けて失敗し、4 回目でようやく緑になりました。

このアイテムは、その最初の待機の**根本原因を突き止めて硬化する**ことを提案します。手動の再実行なしにレーンが 1 回目で緑になる状態を目指します。失敗したレーンを自動で再実行する仕組みは、あえて提案しません。ロードマップはそれを「取り込まない」に挙げています。フラッキーさを隠してしまい、決定性優先の方針と緊張関係にあるからです。ここでの修正は、cold な CI Simulator の settling に対して待機そのものを堅牢にすることであって、再実行でタイムアウトを覆い隠すことではありません。

## 動機

`smoke (idb)` レーンの再実行は 1 回あたり Simulator の boot、ビルド、実行でおよそ 7 分を要します。それ以上に、ゲートへの信頼を損ないます。3 回再実行してようやく緑になるレーンは、赤いチェックを反射的に再実行する習慣を貢献者に植え付けます。この習慣こそ、決定性優先の方針が避けようとしているものです。4 回試してやっと得た緑は、頼れる信号ではありません。

この flake は環境由来であり、プロダクトの欠陥ではありません。PR #936 で検証対象だった変更は純粋に読み取り専用の Python 集約で、ランナー、ドライバ、待機ロジック、デバイス経路のいずれにも触れません。そして同じ idb バックエンドを同じ Simulator で走らせる姉妹レーン `conformance (idb + xcuitest)` は、その間ずっと緑でした。したがって原因は、検証対象のコードとは独立に、cold boot した CI Simulator 上での smoke レーン自身の起動タイミングにあります。

過去の作業は隣接する層を硬化しましたが、この層には及んでいません。

- **BE-0218** は `_await_ready` を名前空間に基づく判定にしました。これにより SpringBoard の名前空間外アイコンが準備完了判定を満たさなくなりました。これは準備完了判定の誤発火を修正したものです。
- **BE-0087** は idb ドライバの操作経路に、投影に基づく settle と一時的な空ツリーへのバックオフを加えました。
- **BE-0207** は XCUITest チャネルに一時的失敗のリトライを、**BE-0088** は Simulator の boot をクリティカルパスから外す並行化を、それぞれ加えました。

このアイテムが対象とする隙間は、準備完了判定とシナリオの最初の `wait` の受け渡しにあります。`_await_ready` はアプリを準備完了と判定できます（名前空間内の要素が現れた時点で）。ところがその判定は、シナリオが直後に待機する特定の要素（最初のリスト行）が、遅い cold boot の Simulator 上でレンダリングされて settle するより一瞬早く成立しうるのです。10 秒の最初の待機はそのレンダリングと競合し、ときおり負けます。対策は条件ベースにとどめ（固定の `sleep` を使わない。プライムディレクティブ 2）、アプリ非依存に保つ必要があります（ターゲットごとの差異は設定に置く。プライムディレクティブ 3）。

## 詳細設計

作業は、まず決定的な診断、次に的を絞った硬化、最後に機械検証可能な受け入れ条件、の順で進めます。各ユニットは独立して出荷できます。

1. **失敗を再実行ではなく成果物から診断できるようにします。** 最初の待機がタイムアウトしたとき、候補となる原因のどれが発火したかを判別できるだけの状態を採取します。タイムアウト時点の要素ツリー、準備完了判定が通過していたか、どの信号で通過したか、そして待機対象の要素がいつクエリ可能になったか（なった場合）です。これに既存の BE-0049 の Run provenance（シナリオハッシュ、ツールバージョン、git リビジョン）を刻み、緑になるまでの再実行が証拠を捨ててしまう状況を解消します。このユニットは、後述の 3 つの仮説をデータから判別し、推測に頼りません。
2. **準備完了判定から最初の待機への受け渡しを締めます。** シナリオが待機する内容が現れる前に準備完了判定が成立していると診断で示された場合、その隙間を閉じます。showcase ターゲットの `readyWhen` をシナリオが必要とする最初の内容を持つ要素に向けるか、名前空間の判定を強めて、名前空間内の任意のノードではなく内容を持つ要素を要求します。変更は可能なかぎり設定を優先し（ターゲットの `readyWhen`）、隙間が一般的なものであるときにのみ共有の準備完了ロジックに触れます。
3. **最初の `wait` を遷移途中の空ツリーに対して堅牢にします。** シナリオレベルの `wait_for` が、一時的な空ツリーや遷移途中のクエリを、BE-0087 が操作経路で許容したのと同じように許容することを確認します。起動遷移の最中で最初のポーリングが空ツリーに当たったとき、待機の予算を静かに消費してはなりません。この経路がまだ一時的な空ツリーへのバックオフで覆われていなければ、拡張します。
4. **最初の待機の予算を cold な CI Simulator に見合う大きさにします。** cold boot した CI Simulator では（warm なローカルと比べて）レンダリングが本当に 10 秒を超えると診断で示された場合、待機の予算を固定の定数ではなくターゲットや環境ごとに設定可能にします。条件ベースは保つので、速い環境は要素が現れ次第すぐ返り、遅い環境だけが大きな予算を使います。固定の `sleep` は導入しません。
5. **緑が続くことを証明します。** 受け入れ条件は機械検証可能にします。`smoke (idb)` レーンが、有界の回数だけ連続する CI 実行を通じて、手動の再実行なしに 1 回目で通過することです。これはゲート自身に対する決定的なチェックであり、LLM の判断ではありません。

## 検討した代替案

- **失敗したレーンの自動再実行（reboot を伴う）。** このアイテムの背骨としては採用しません。ロードマップはすでに「失敗テストの自動リトライ」を「取り込まない」に挙げています。フラッキーさを隠し、決定性優先の方針と緊張関係にあるからです。仮にそうした仕組みを採用するとしても、quarantine 用途に限り、必ず loud であり（アサーション失敗を静かに覆い隠さない）、provenance を刻んで、隠れた flake が BE-0220 の Run 横断トリアージから見えるようにする必要があります。それは別に切り出して慎重に範囲を定めるアイテムであって、このアイテムではありません。
- **グローバルな待機タイムアウトを闇雲に引き上げる。** 採用しません。根本原因を覆い隠し、あらゆる環境のすべての実行を遅くします。ユニット 4 は、cold boot の真の分散が診断で示された箇所に限り、条件ベースかつ環境ごとの上限として予算を引き上げます。
- **boot 後に固定の `sleep` でウォームアップする。** 端的に採用しません。プライムディレクティブ 2 が固定の sleep を禁じており、待機は条件ベースのみです。

ログ（古い順）:

- [#952](https://github.com/bajutsu-e2e/bajutsu/pull/952) — ユニット 2〜4。`showcase-swiftui` ターゲットの `readyWhen` を、smoke シナリオ冒頭の
  `wait` が必要とする要素、すなわち最初の Stable 行（`stable.row.1`）に向けます。これにより
  `_await_ready` が名前空間内の別のノードで早期に成立し、cold boot した CI Simulator 上で最初のステップが
  まだレンダリングされていない行とレースする状況をなくします（ユニット 2）。あわせて、シナリオの `for`
  待機が空の初回ポーリングを「まだ現れていない、ポーリングを続ける」と扱い、待機の予算を消費しないことを
  テストで固定します（ユニット 3）。ユニット 2 だけでは十分ではありませんでした。本 PR 自身の
  `smoke (idb)` 実行で、同一の `wait timeout: for stable.row.1 (10.0s)` としてフレークが再現したためです。
  ゲートの 10 秒とシナリオの 10 秒を合わせた約 20 秒を、レンダリングが本当に超えていたことになります。
  そこでユニット 4 として、`smoke (idb)` レーンの待機の下限を `BAJUTSU_MIN_WAIT_TIMEOUT=20` で引き上げます
  （Android e2e レーンが既に opt-in している既存の設定項目です）。これにより最初の行には起動から約 30 秒の
  予算が与えられます。条件ベースなので、速いレンダリングは即座に返り、固定の `sleep` は導入しません。設定を
  優先し（プライムディレクティブ 3）、条件ベースを保ち（プライムディレクティブ 2）、判定経路に LLM を
  置きません（プライムディレクティブ 1）。ユニット 5（レーンが 1 回目で緑を保つこと）は、本 PR が
  マージされた後に `smoke (idb)` レーンで観測します。
