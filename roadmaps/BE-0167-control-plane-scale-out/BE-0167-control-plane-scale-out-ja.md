## はじめに

セルフホストの制御プレーン（[BE-0016](../BE-0016-web-ui-self-hosting/BE-0016-web-ui-self-hosting-ja.md)）は、
現在 `deploy/self-host/` の compose スタックで**単一**の `bajutsu serve --asgi --backend=server`
アプリコンテナとして動いています。これは軽量な HTTP 処理であり、水平にスケールできます。この提案は、
プルベースのジョブ分配を変えずに、**ロードバランサ配下で N 個のアプリレプリカ**を動かし、制御プレーンが
レプリカ障害に耐え、より多くの同時利用者をさばけるようにすることを定めます。BE-0016 の「単一ノードを
プールへ育てる」作業から切り出したものです。

## 動機

Mac プールを増やしても、制御プレーンは一つのコンテナのままです。その単一のアプリプロセスは容量の上限
（すべてのブラウザセッション、ワーカーのリース呼び出し、実行履歴の読み取りがそこを通ります）であり、
可用性のリスク（再起動すると UI 全体が短時間使えなくなります）でもあります。制御プレーンは共有の
データベースとオブジェクトストアの上に載るステートレスな HTTP なので、これを複製することが両方の制約を
取り除く自然な方法です。作業は、その複製を正しくする（スティッキーセッションをなくし、長寿命の接続を
失わない）ことと、この構成が必要とするロードバランサの設定を文書化することです。

## 詳細設計

この設計は、スタックについてすでに成り立っている二つの事実に立脚します。セッションは **Postgres** に
存在し（BE-0106 がセッションストアをデータベースへ畳み込みました）、ワーカーモデルは**完了後**であるため、
実行中のログストリームをレプリカ間へ分配する必要がありません。作業分解は次のとおりです。

1. **ステートレスなレプリカ、スティッキーセッションなし。** どのレプリカもどの要求も処理できなければ
   なりません。認証は署名付き Cookie で、セッションは Postgres に存在するため、あるレプリカで認証された
   要求は、スティッキーなルーティングなしに別のレプリカでも受理されます。これは手動で検証します。あるレプリカで
   ログインし、別のレプリカで実行履歴を読みます。
2. **ロードバランサの方針。** レプリカの前段には Caddy か HAProxy を置き、ラウンドロビンではなく
   **最小接続数**を使います。Server-Sent Events（SSE）の接続は長寿命であり、ラウンドロビンだと開いた
   ストリームを多く抱えたレプリカへ負荷が偏るためです。TLS はロードバランサ（または既存の `caddy`
   プロファイル）で終端します。
3. **レプリカをまたぐワーカーの経路。** ワーカーモデルは完了後であるため、`bajutsu worker` はどのレプリカ
   からでも `POST /api/worker/lease` でき、結果全体をどのレプリカの `/api/worker/result` へでも送れます。
   ワーカーとリースを処理したレプリカのあいだに、実行ごとの結び付きはありません。ブラウザは完了した実行を
   共有のデータベースとオブジェクトストアから読むため、こちらにも結び付きは不要です。
4. **N レプリカのための compose 配線。** `deploy/self-host/` を拡張し、ロードバランサ配下で N 個の
   アプリレプリカを動かします（スケール係数とロードバランサ設定）。ステートフルなサービス（Postgres、
   MinIO）は変えず共有します。運用者の手順を文書化します。

**検証。** このティアはデプロイと運用の作業であり、Linux だけの `make check` ゲートでは動かせません
（Docker も複数コンテナの構成もないため）。デプロイ上で手動で検証します。二つのアプリレプリカを立ち上げ、
一方のレプリカでのログインが他方のレプリカが処理する要求で受理されること、一方のレプリカからリースした
実行が完了し、その結果が他方を通じて読めることを確認します。`bajutsu/serve/` に入るステートレス性の
小さな補助（たとえばプロセス内のセッション状態を持たないことの表明）には単体テストを残しますが、構成
そのものにはゲートの契約がありません。

## 検討した代替案

- **スティッキーセッション（ロードバランサでのセッションアフィニティ）。** 却下します。レプリカごとの
  状態の結び付きを復活させ、レプリカ再起動時に負荷を偏らせ、セッションが Postgres に存在すれば不要です。
  ステートレスなレプリカのほうが単純で堅牢です。
- **ラウンドロビンの負荷分散。** 最小接続数を採り、却下します。SSE のストリームは長寿命なので、
  ラウンドロビンでは開いた接続が一部のレプリカへ徐々に集中します。最小接続数は実際の負荷を追跡します。
- **単一レプリカのまま垂直方向にだけスケールする。** 出発点としては受け入れられますが、最終形としては
  却下します。一つのアプリコンテナは可用性の単一障害点であり容量の上限でもあり、水平方向のレプリカが
  それを取り除きます。
