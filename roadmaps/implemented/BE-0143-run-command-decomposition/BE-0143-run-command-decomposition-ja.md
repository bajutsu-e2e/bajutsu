## はじめに

すべてのシナリオの合否を決める `run` コマンドは、418 行の 1 関数として実装されています。本項目
では、そのオプション宣言とオーケストレーションの各ステップを、同じ CLI 面の裏側でまとまりのある
補助関数へ切り出し、決定的コアのうち最も重みのあるこのコマンドを、挙動を変えずに読みやすく、
独立してテストできる形にします。

## 動機

`bajutsu/cli/commands/run.py:186` から始まる `run` 関数は、`bajutsu/cli/commands/run.py:603`
まで戻り値を返さず（`register` は 604 行目から始まります）、1 つの関数本体が 418 行あります。
そのうち約 118 行（186〜303 行目）はこのコマンドの広大なフラグ面に対応する `typer.Option` 宣言
で、残りの約 300 行（304〜601 行目）がオーケストレーションです。effective 設定の解決と必要に
応じたアプリのビルド、`--headed` / `--browser` / `--browsers` の解決、`--tag` / `--exclude` に
よるシナリオの読み込みと絞り込み、actuator の選択と `--browsers` マトリクスの整合性検証、
デバイスレーンの解決、alert-guard ファクトリの構築、baseline・schema・golden ディレクトリの
解決、開始 webhook の発火、launch server の起動、単一エンジンとクロスブラウザマトリクスの
どちらの実行経路を取るかのディスパッチ、そして最後に判定結果・終了 webhook・任意の `--zip`
アーティファクト・AI 使用量サマリの出力までを、この 1 関数がすべて担っています。この規模の
関数は最初から最後まで読み通すのが難しく、「コマンド全体を実行する」より細かい粒度でテストする
ことができず、変更のリスクも高くなります。本体のどこか一箇所を編集するだけで、それまでの
300 行で積み上げてきた状態に触れてしまうためです。規模は M です。各ステージの境界はコード自身の
コメント（「Validate the backend before touching the Simulator CLIs」「Mocks ride the network
channel」など）にすでに暗黙のうちに現れているため、切り出しは大部分が再設計ではなく命名と
移動の作業になります。

## 詳細設計

この作業は挙動を変えずに行います。`run` の CLI 面（フラグ、デフォルト値、終了コード、標準出力・
標準エラー出力）は変更せず、まず cli-command-coverage 項目のカバレッジが先に着地していれば、
この分解にはそれを検証する安全網があります。作業は独立した 2 つの軸に分解します。

- **オプション宣言のグルーピング。** 約 118 行の `typer.Option(...)` 宣言
  （`bajutsu/cli/commands/run.py:186`〜`303`）は、並び順にすでに現れているまとまり — target/
  scenario の選択、backend/device の選択、browser/engine の選択、baseline・schema・golden
  ディレクトリの上書き、`--progress` や `--zip`、`--runs-dir` といったレポート・出力系フラグ —
  にグルーピングできます。これらは別関数にする必要はありません（`typer.Option` の呼び出しは
  シグネチャの中に置く必要があります）が、シグネチャ自体をより明確なセクションコメントで
  整理するか、`run` が再展開する少数の `dataclass` ベースのオプションバンドルにまとめる
  ことができます。いずれの手法を取るにせよ、`typer` の introspection を壊さず、フラグ面を
  二重管理しないことが条件です（この二重管理そのものは兄弟項目の serve-cli-flag-mirror-drift
  が扱う裏返しの問題であり、`serve` 側は現在この同じ面を手作業で複製しています）。
- **オーケストレーションステップの切り出し。** 約 300 行の本体は、このファイルに既存の
  `_resolve_*` / `_scenario_files` / `_expand_file` といった補助関数と並ぶ形で、それぞれが
  プレーンなデータを受け取り返す補助関数へと分解し、Simulator なしでユニットテストできる
  ようにします。
  - 設定・ビルドの解決（`_load_effective_with_source` の呼び出しから `build_if_missing` まで）
  - browser/engine の解決（`--headed`、`_resolve_browser`、`_parse_browsers`） — 一部はすでに
    切り出し済み
  - シナリオの読み込みと `--tag` / `--exclude` による絞り込み（`--erase` の上書きを含む）
  - backend/actuator の選択と、web 以外での `--browsers` マトリクス検証
  - デバイスレーンの解決と `--dismiss-alerts` の上書き
  - alert-guard ファクトリの構築（`_guard_for` クロージャとその credential-gap チェック）
  - baseline・schema・golden ディレクトリの解決（すでに切り出し済み。新しいステップへ配線する）
  - 単一エンジンとクロスブラウザマトリクスのディスパッチ（`run_pass` /
    `device_pool` / `run_and_report` / `run_matrix_and_report` の分岐）
  - 実行後のレポーティング（判定結果の出力、終了 webhook、`--zip` パッケージング、
    AI 使用量サマリ）

  こうすると `run` 自体はこれらの補助関数を順に呼び出す薄いシーケンスになり、それぞれを
  cli-command-coverage 項目が追加する高速テストで独立して検証できます。

## 検討した代替案

- **`run` を 1 関数のまま残し、コメントで各ステージを区切るにとどめる。** 却下します。
  この関数にはすでに各ステージの境界に説明的なコメントが付いています（前述のとおり）。これは
  境界がすでに認識されていながらコードとしては表現されていないことを示しており、コメントだけ
  ではステージを独立してテストしたり、単独で安全に変更したりできるようにはなりません。
- **オーケストレーションを共有状態上のメソッド群として持つ `Runner` クラスを導入する。** 今回の
  作業としては却下します。これは大きな面を（`self` を介した暗黙の共有状態を伴う形で）メソッド群
  に散らすだけで、本項目が志向するプレーンなデータを扱う補助関数ほど単体でのテストや理解が
  容易にはなりません。切り出した補助関数が関数引数として渡すには収まりが悪いほどの共有コンテキスト
  を必要とすると判明した場合にのみ、改めて検討する価値があります。
- **切り出しとカバレッジ追加を 1 つの PR で行う。** 却下します。既存のテスト網がない状態で先に
  リファクタリングすると、切り出しが挙動を保ったかどうかを判別しづらくなります。
  cli-command-coverage 項目を先に進めることで、この作業を検証する安全網を確保します。
