# BE-0022 — `update`（最小差分提案＝構造化 fix の適用）

* 提案: [BE-0022](BE-0022-update-structured-fixes-ja.md)
* Author: [@0x0c](https://github.com/0x0c)
* 状態: **実装済み**
* 実装 PR: PR 単位の履歴より前（初期インポートにスカッシュ済み・単一 PR なし）
* トラック: [可決済み](../../README-ja.md#可決済み)
* トピック: 自己修復トリアージ（M4）

## はじめに

壊れたシナリオを全体再記録せず最小差分で更新します。triage が構造化 fix（`renameId`/`addIndex`/`raiseTimeout`）を提案し、`--apply`（dry-run diff）/`--write` で source に適用、`--rerun` で再実行検証を行います。実機で rename・addIndex の閉ループは実証済みです。

## 動機

ちょっとした UI 変更——識別子のリネーム、新たに重複した部品、少し読み込みが遅くなった画面——は、シナリオを無効にすることなく壊してしまいます。1 行を直すためにシナリオ全体を再記録するのは無駄であり、さらに悪いことに、それ以降に人間が YAML へ加えた手編集を捨ててしまいます。本当に欲しいのは、シナリオを再び決定的にする最小の差分を、他のコード変更と同じようにレビュー用に提示することです。BE-0021 のトリアージは既に原因を言い当てています。本提案はその診断を機械的に適用可能な編集へ変換し、fix のループを新規記録ではなくレビュー可能なパッチで終えられるようにします。

## 詳細設計

トリアージは診断に構造化された `Fix` を付与できます。シナリオ source に対する `find` -> `replace` で、種別は次の 3 つです。

* **`renameId`** はセレクタ id をトークン単位で置換します。否定先読み・後読みにより `nav.setting` が `nav.settings` の内側に一致するのを防ぐため、出現箇所すべてに安全に適用できます。リネームされた識別子に対する古典的な自己修復です。
* **`addIndex`** は失敗ステップの正確なセレクタ断片を書き換えて `index:`（または `within:`）を追加し、複数要素に一致したセレクタを一意化します。
* **`raiseTimeout`** は失敗した wait の正確な `timeout: N` 断片をより大きな値へ書き換えます。

`bajutsu triage --apply <file>` はその source ファイルに fix を適用します。`apply_fix` は純粋関数で、パッチ後のテキストと置換回数を返します。CLI は unified diff（`diff_fix`）を表示します。既定では**ドライラン**で、diff を出力するだけで何も書き込みません。実際にファイルを書くには `--write` が必要で、`--rerun`（`--write` の後、`--app` とデバイスを伴う）はパッチ済みシナリオを通常の決定的な `run` で再実行し、fix が成立することを確認します。`--apply` が与えられたときは、トリアージは run の正規化済みダンプではなくファイル自身のテキストに対して診断します。これにより断片 fix の `find` がパッチの編集対象のバイト列に一致します。

ここでは何ひとつ合否に触れません。fix は人間がレビューするシナリオ source への編集であり、検証は変わらず決定的な `run` です。rename と addIndex の閉ループ（破損 → トリアージ → 適用 → 再実行 → グリーン）は実機で実証済みです。

## 検討した代替案

* **破損時にシナリオ全体を再記録する。** 当初の M4 のフォールバックですが、人間の編集を捨て、1 行の原因に対して大きくノイズの多い差分を生みます。最小の構造化 fix はパッチを小さくレビュー可能に保ちます。
* **実行中に fix を自動適用する。** 一部の競合が行っていること（BE-0039 参照）であり、却下します。実行の途中で黙ってシナリオを編集すると、誰の目にも触れずに条件が緩む恐れがあります。ここではすべての fix がまずドライラン diff となり、明示的な `--write` のときだけ書き込まれます（防止策は BE-0023）。
* **モデルからの自由形式テキストパッチ。** 任意の編集を許せばモデルが何でも書き換えられます。fix を 3 つの正確な `find`/`replace` 種別に制約し、`find` を source の部分文字列に限ることで、fix はきれいに適用されるか、目に見える no-op になるかのどちらかになり、黙った広範な書き換えにはなりません。
