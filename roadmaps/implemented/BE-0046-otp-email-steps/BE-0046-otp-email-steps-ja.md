## はじめに

[BE-0036](../../implemented/BE-0036-utility-steps/BE-0036-utility-steps-ja.md) の `http` スライスが
出荷されたあとに切り出した、側方チャネル（side channel）のユーティリティステップです。`totp`
（時刻ベースのワンタイムパスワードをローカルで生成）と `email`（テスト用メールボックスをポーリング
して受信コードを抽出）を扱います。これらは帯域外の認証値を run に取り込み、実際のログインフローを
自動化できるようにします。

## 動機

実アプリのログインフローは、パスワードを入力するだけで済むことはまれです。多くの場合、UI 自身では
供給できない側方チャネルが必要です。認証アプリからのワンタイムパスワード（OTP）、メールや SMS で
届く二要素認証（2FA）コードなどです。`http`（BE-0036）はすでに HTTP エンドポイントから取得する値を
カバーしますが、一般的な 2FA のケースは認証アプリや実際の受信箱から来ます。これらの値を取り込む
手段がなければ、最も重要なフロー（サインイン、アカウント検証、パスワードリセット）を決定的に自動化
できません。著者は、テストビルドで 2FA を無効化する（すると実際の経路を検証しなくなります）か、値を
手で貼り付ける（再現できません）かを強いられます。

**競合の文脈（Maestro）。** Maestro は同じニーズを、著者を JavaScript の逃げ道へ送り込むことで満たし
ます。コードを計算する、メールボックスをポーリングするといった処理を `runScript` / `evalScript` で
行うものです。機能はしますが、ロジックをレビュー可能な YAML の外、汎用コードへ移し、そのフローへの
影響はレポートからは不透明になります（制御フローについて
[BE-0033](../../implemented/BE-0033-scenario-variables-control-flow/BE-0033-scenario-variables-control-flow-ja.md)
が回避したのと同じトレードオフです）。Bajutsu の賭けは、少数の目的特化で監査可能なステップ（`totp` /
`email`）です。これらは値を決定的に `${vars.*}` へ生成し、シナリオとレポートの中で見えたままにします。
違いは、スクリプトの逃げ道なしに、LLM もなしにログインフローの能力を提供することです。これにより
ゲートは決定的なまま、フローはレビュー可能なままになります。

## 詳細設計

既存の `${vars.*}` 名前空間（[scenarios](../../../docs/ja/scenarios.md#ランタイム変数-vars)）へ値を
生成するステップを 2 つ追加します。これにより、後続の `type` / `assert` がその値を決定的に消費でき
ます。

```yaml
- totp: { secret: "${secrets.TOTP_SEED}", into: { var: code } }   # RFC 6238 の時刻ベース OTP
- type: { text: "${vars.code}", into: { id: auth.code } }
```

```yaml
- email: { match: { to: "test@example.com", subjectMatches: "code" }, extract: { var: code, bodyMatches: "[0-9]{6}" }, timeout: 30 }
- type: { text: "${vars.code}", into: { id: auth.otp } }
```

マッピングと契約：

- **`totp`** はシード（種）から時刻ベースのワンタイムパスワードをローカルで計算します（RFC 6238）。
  純粋な計算で、シードとクロックが与えられれば完全に決定的、ネットワーク不要です。
- **`email`** は、一致するメッセージが届くまでテスト用メールボックス API をポーリングし、コードを
  抽出します。この待機は **`timeout` 必須の条件待機**です（他のすべての待機と同じルールで、固定
  sleep も無限ポーリングもありません）。タイムアウトはステップ失敗です。

シークレット（TOTP シード、メールボックスの認証情報、API のベース URL）は `${secrets.X}` として参照
し、config で宣言します。これによりシナリオファイルはトークンを保持し値は保持せず、証跡では自動的
にマスクされます。複数シナリオで共有するエンドポイント設定は `targets.<name>` に置きます。

prime directive の保持：

- **run パス上に LLM を置かない。** 各ステップは決定的な計算、または決定的な一致です。`totp` は
  算術、`email` は正規表現による抽出です。合否判定は引き続き機械チェック可能なアサーションのみ
  から得られ、run／CI ゲートは AI 非依存のままです。
- **決定性。** `email` は上限のある timeout 付きの条件で待ち、sleep は使いません。`totp` は即時
  です。外部メールボックスが不安定な場合は、黙った誤った値ではなく、きれいなステップ失敗として
  表面化します。
- **アプリ非依存。** ステップとその配線はアプリをまたいで同一で、異なるのはエンドポイントと
  シークレットだけです。
- **codegen。** これらはアプリではなく bajutsu ランナーで動き、XCUITest 等価物がないため、codegen
  はラベル付き `// TODO` を出力します
  （[BE-0026](../BE-0026-shrink-unsupported-syntax/BE-0026-shrink-unsupported-syntax-ja.md) に従います）。

### email ステップのメールボックス面

`totp` スライスが先送りにした未決の設計点は、`email` が*どのメールボックス*と話すか、でした。決定は、
プロバイダ固有のクライアントではなく **汎用 HTTP ポーリングメールボックス**です。ステップは設定した
エンドポイントを `GET` し、受信箱を JSON で受け取り、待っているメッセージを絞り込み、正規表現でコードを
抽出します。これは `http` ステップの HTTP 配管（BE-0036）を再利用し、ツールをプロバイダ非依存に保ち
（HTTP API で読める受信箱ならどれでも動きます。Mailosaur、MailSlurp、チーム自前のテスト用
SMTP-to-HTTP ブリッジなど）、プロバイダ SDK への依存を増やしません。

**設定（`targets.<name>.mailbox`）。** エンドポイントと、そのレスポンスの読み方を config に置きます。これで
シナリオはアプリ非依存で、認証情報も持ちません。

```yaml
apps:
  myapp:
    mailbox:
      url: "${secrets.MAILBOX_URL}"        # 受信箱エンドポイント（GET）。クエリや box id を含んでよい
      headers: { Authorization: "Bearer ${secrets.MAILBOX_TOKEN}" }
      # このプロバイダの JSON の読み方（レスポンスへのドット区切りパス）。プロバイダ固有コードは不要
      messages: "items"                     # メッセージ配列へのパス（既定：ルートの配列）
      fields: { to: "to", subject: "subject", body: "text", receivedAt: "receivedAt" }
```

既定値は、よくある「`to` / `subject` / `body` / `receivedAt` を持つメッセージの配列」という形に合わせ
ます。したがって、その形に従う API では `messages` / `fields` のマッピングは一切不要です。マッピングは、
名前の異なるレスポンスをコード変更なしに吸収するためだけに存在します。

**ステップの契約。**

```yaml
- email:
    match: { to: "test@example.com", subjectMatches: "verification" }   # どのメッセージを待つか
    extract: { var: code, bodyMatches: "[0-9]{6}" }                     # 値を vars.code へ取り出す
    timeout: 30
```

- **`match`** はメッセージを選びます。`to`（完全一致）、`subject`（完全一致）／`subjectMatches`
  （正規表現）を AND で組み合わせます。少なくとも 1 つの条件が必要です（何にでも一致する `email` は、
  空のセレクタと同様、読み込み時に却下します）。
- **`extract`** は一致したメッセージの本文から値を `${vars.<var>}` へ取り出します。`bodyMatches` は
  正規表現で、その **最初のキャプチャグループ**（無ければマッチ全体）が値になります。値は実行時の
  `vars.*` にのみ存在し、記録されるシナリオは（`totp` / `http` の `saveBody` と同様）`${vars.*}`
  トークンを保持するので、manifest／レポートには入りません。
- **`timeout`**（必須、秒）はポーリングの上限です。他のすべての待機と同じ条件待機のルールで、固定
  sleep も無限ポーリングもありません。

**決定性。重要な点。**

- **ステップ開始*後*に届いたメールだけを数えます。** ステップは開始時点に存在した**メッセージ id** の
  集合を記録し、その id を無視します。これにより、以前の run が受信箱に残した古いコードに一致することは
  ありません。プロバイダの `receivedAt` をローカルの run クロックと比較するのではなく id で判定するので、
  メールボックスのクロックと runner のクロックが一致している必要がなく、**スキューに強い**設計です。
  プロバイダ id を持たないメッセージには内容ハッシュの安定 id を与えるので、基準集合でも区別できます。
- **待っているメッセージは一意。** 開始後という境界があれば、想定するのは新着の一致メッセージがちょうど
  1 件です。複数が一致した場合は **`receivedAt` が最新のもの**を採用します（同時刻はメッセージ id で
  安定にタイブレークします）。これにより結果が到着順の競合に左右されません。`receivedAt` はこの順序付けに
  のみ使い、開始後の判定には使いません。
- **上限つきポーリング。** 一致するか `timeout` の期限に達するまで、fetch と判定を繰り返します。固定
  sleep ではなく、他の待機と同じ条件待機です。タイムアウト、`extract` の正規表現が本文に当たらない一致
  メッセージ、2xx 以外の fetch は、いずれも**きれいなステップ失敗**で、黙った誤った値にはなりません。

**ゲート検証可能性。** fetch は注入可能な HTTP 関数（`http` ステップのクライアント／`RunFn` 風の継ぎ目）の
背後に置きます。これにより、ポーリングループ、開始後フィルタ、match／extract、最新優先のタイブレーク、
タイムアウトのすべてを、ネットワークなしで、捏造したメールボックスレスポンスに対してユニットテスト
できます。モックする面はライブの HTTP 呼び出しだけで、これは許可された「外部 API 呼び出し」の例外で
あり、`http` のテストのしかたと同じです。

### 実装状況

外部依存のない **`totp`** スライスを出荷します。`totp: { secret, into: { var } }` は、base32 の
`secret`（通常は `${secrets.*}`。ステップ実行前に補間されます）と現在時刻から RFC 6238 のコードを
計算し、後続の `type` / `assert` 用に `${vars.<var>}` へ書き込みます。アルゴリズムは純粋でゲート検証済みの
関数（`bajutsu/totp.py`。RFC 6238 のテストベクタで検証）で、ステップはそれを実時刻で呼ぶ薄いハンドラです
（`bajutsu/orchestrator/actions/handlers/totp.py`）。値を `vars.*` に入れる点で `http`/`saveBody` の前例に
倣い、デバイスには触れず、codegen はラベル付き `// TODO` を出力します。

**`email`** スライスを出荷し、本項目を完了します。`email: { match, extract, timeout }`
（`bajutsu/scenario/models/actions.py`）は、設定した `targets.<name>.mailbox`（`bajutsu/config.py`）を、
ステップ開始*後*に届いたメッセージが `match` を満たすまでポーリングし、その本文から `bodyMatches` で値を
`${vars.<var>}` に取り出します。match／extract／レスポンス形の読み取り／開始後の選択は純粋でゲート検証済みの
関数です（`bajutsu/mailbox.py`）。上限つきポーリングは run ループ（`_do_email`、`orchestrator/loop.py`）で、
**注入された** `MailboxReader`（`orchestrator/types.py`）越しに動きます。これは config から `${secrets.*}` を
解決して run ごとに 1 度だけ構築され（`runner/mailbox.py`）、ネットワークに触れるのはライブの HTTP GET だけで、
ロジックはネットワークなしでテストできます。開始後の境界は**メッセージ id**（ステップ開始時点に存在した集合）で
判定するのでクロックスキューに強く、プロバイダのタイムスタンプをローカルクロックと比較しません。`receivedAt` は
新着の一致同士で最新を選ぶ順序付けにのみ使います。タイムアウト、本文に正規表現が当たらない一致メッセージ、
到達不能 / 2xx 以外のメールボックスは、きれいなステップ失敗です。生成した値は実行時の `vars.*` にのみ存在し、
記録されるシナリオは（`totp` / `http` と同様）`${vars.*}` トークンを保持するので、manifest／レポートには
入りません。codegen はラベル付き `// TODO` を出力します。デバイスも LLM も使いません。

## 検討した代替案

- **テストビルドで 2FA／OTP を無効化し、側方チャネルを丸ごと省く。** 最も単純で、スモークテストには
  十分ですが、実際の認証経路を検証しなくなります。それはまさに最もカバーする価値のあるフローです。
  唯一の選択肢としては却下します。ビルドがテスト用バイパスを提供することはあってよいですが、ツール
  がそれを要求すべきではありません。
- **`mocks` で OTP／メールプロバイダをモックする。** `mocks` はアプリ*自身*の送信トラフィックを
  スタブするので、アプリ自身が発行して観測するリクエストでコードが届く場合には機能します。値が帯域外の
  チャネル（認証アプリ、実際の受信箱）から来る場合には役立たず、それが一般的な 2FA のケースです。
  置き換えではなく補完として維持します。
- **すべてを `http`（BE-0036）で済ませる。** `http` はエンドポイントから値を取得するもので、
  バックエンド発行のトークンはカバーしますが、ローカル計算の TOTP やポーリングする受信箱はカバー
  しません。HTTP のケースには引き続き使い、`totp` / `email` が残りをカバーします。（`email` は内部で
  `http` の HTTP 配管を*再利用*しますが、一回限りの `http` 取得では表現できないポーリング、match／extract、
  開始後の決定性境界を加えます。）
- **プロバイダ固有のメールボックスクライアント（Mailosaur／MailSlurp の SDK）。** 1 つのプロバイダに
  対しては最速で組めますが、ツールをそのベンダの SDK と API に縛り、2 つ目のプロバイダには 2 つ目の
  クライアントが要ります。フィールドパスを設定できる汎用 HTTP の面を採り、却下します。汎用の面なら、
  どのプロバイダの JSON 受信箱も依存を増やさず読めます。HTTP 以外の API しか持たないプロバイダも、
  小さなブリッジを前段に置けば扱えます。
- **生の IMAP メールボックスをポーリングする。** テストメール用プロバイダなしに実際の受信箱へ当たれ
  ますが、IMAP クライアントと認証情報の扱いを抱え込み、本文を得るための MIME パースもそれ自体が一つの
  面になります。見送ります。HTTP メールボックスが、動機となる 2FA フローの使うテストプロバイダのケースを
  カバーします。IMAP ソースは、需要が出れば同じ `email` ステップの契約の背後に後から追加できます。
