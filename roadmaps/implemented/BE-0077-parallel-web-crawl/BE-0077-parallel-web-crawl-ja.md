## はじめに

[BE-0066 の Web クロール](../../implemented/BE-0066-web-crawl/BE-0066-web-crawl-ja.md)を、**複数のブラウザで同時に**走らせ、独立したフロンティアの作業を重ね合わせて、1 つの画面マップを実時間ベースで大幅に短縮して構築します。これは [BE-0064（複数シミュレータでの並列クロール）](../BE-0064-parallel-crawl/BE-0064-parallel-crawl-ja.md)の Web 版です —— BE-0064 も BE-0066 も、後続項目として既にこの交点を名指ししています（「並列 Web クロールは、この項目と並列クロール軸との交点であり、両者が揃えば後続できる」）。iOS と同じく、クロールはあくまで Tier 1 の探索・発見ツール（CI ゲートには決してならない）のままで、並列化されるのは**スケジューリングだけ**です。画面同一性・遷移・クラッシュの判定は従来とまったく同じに行われます。

各ワーカーは、1 つの共有ブラウザ内の `BrowserContext` レーンではなく、それぞれ自分の**ブラウザプロセス**（起動した Playwright の `Browser`）を持ちます。ここでの狙いは**高速化** —— 1 つの共有画面マップをより速く構築すること —— であって、クロスエンジンの網羅ではありません。同じアプリを異なるエンジン（Chromium / Firefox / WebKit）で走らせてエンジン固有の差異を洗い出すのは別機能であり、後述のとおりスコープ外とします。

## 動機

現在の Web クロールは逐次実行です。1 つのブラウザで 1 画面ずつ探索します。1 画面あたりのコストは、マシンを遊ばせてしまう 2 つのレイテンシ律速の待ちが支配的です —— BE-0064 が iOS で挙げた 2 つと同じものの、Web 版です。

1. **AI ガイドの往復.** `--guide ai` では、新しく発見した画面ごとに 1 回以上モデル呼び出し（操作プロポーザ）が走ります。これらはネットワーク往復です。
2. **ブラウザとページの操作.** 未探索画面へ到達するには、クリーンな初期状態へ `reset` し、記録済みパスを再生し（途中の JS ダイアログを処理しながら）、操作を実行して `observe` します。各遷移・操作・観測はブラウザとネットワークを待ち、再生コストは画面の深さとともに増えます。

どちらも独立したブラウザ間できれいに重なるので、AI のレート上限やコーディネータの競合が支配的になるまで、実時間はおおよそワーカー数に反比例して減ります。

**並列クロールを最も低摩擦で活かせるのが Web です。** Mac もエミュレータも要らず、既存の Linux の `make check` / CI ゲート上で動くので（[BE-0041](../BE-0041-web-playwright-backend/BE-0041-web-playwright-backend-ja.md)）、速いクロールを CI 内の発見ステップとしてそのまま回せます。ブラウザプロセスは端末起動なしに数秒で立ち上がるので、N レーンを用意するコストも小さいものです。`run` は既にレーンで並列化されており —— [BE-0054](../BE-0054-web-backend-completion/BE-0054-web-backend-completion-ja.md) がプールの Web 分岐を N レーンへ一般化します —— クロールだけが、いまだ 1 つのブラウザに固定された唯一の Tier-1 経路です。これが、`record` の入口として、また全画面のカバレッジ計測の実行として（[DESIGN §7.2](../../../DESIGN.md)、[BE-0038](../../in-progress/BE-0038-autonomous-crawl-exploration/BE-0038-autonomous-crawl-exploration-ja.md) 動機 #2）クロールを使いにくくしています。

## 詳細設計

### コーディネータ + ワーカー

**コーディネータ**がロック下で共有の画面マップとフロンティアを保持します —— `nodes`・`edges`・`path_to`（各画面への再生可能なパス）・`pending`（画面ごとの未試行操作）・`visited`、および予算カウンタ —— BE-0064 とまったく同じです。**N ワーカー**はそれぞれ対象アプリを読み込んだブラウザプロセスを 1 つ持ちます。

各ワーカーのループ:

1. **ロック** → 同じ決定的規則（`path_to` が最短、次に fingerprint）で最安のフロンティアエントリを選び、操作を 1 つ取り出して実行中印を付ける → **アンロック**。
2. 自分のブラウザ上で `reset`、`path_to[fp]` を再生（途中の JS ダイアログを決定的なダイアログ自動ハンドラで処理しながら）、操作を実行、`observe`。
3. **ロック** → エッジ / クラッシュ / ダイアログを記録。遷移先が新規ならノード + `path_to` + `pending` を追加し、その画面にいる間に当該ブラウザでガイドを実行 → **アンロック**。

これにより、ガイドの AI 呼び出しがワーカー間で並行に走ります —— これが主たる短縮要因です。**フォワードウォーク**の最適化はワーカーごとに保たれます。ワーカーは未試行操作が無くなるまで今いる画面を操作し続け、別のフロンティアエントリへ到達するためにのみバックトラック（reset + 再生）します。これは現在の 1 ブラウザ戦略そのもので、各ワーカー上で動くだけです。

### レーンはブラウザプロセス、リセットごとのクリーン状態は新しいコンテキスト

ワーカーの永続的なレーンは 1 つの**ブラウザプロセス**です —— 障害時にコーディネータが強制終了して再起動する単位がこれです（後述）。一方、反復ごとの `reset` は**ブラウザの再起動ではありません**。それはワーカーのブラウザ内に新しい `BrowserContext` を作ることで、`erase` に相当し、ほぼ無コストです。これは BE-0066 の既存の Web リセット継ぎ目をそのまま再利用します（`reset` は既に、新しいコンテキストへの `driver.navigate()` にすぎず、[`runner/pool.py`](../../../bajutsu/runner/pool.py) の `_web_relauncher` と同じです）。つまり「ブラウザプロセスを分ける」コストは、画面ごとではなく**ワーカーごと**（および障害ごと）のブラウザ起動 1 回です。

### 決定性の境界（要点）

画面の*同一性*（fingerprint）・遷移判定・クラッシュ判定・画面マップの内容は、要素ツリーの純粋な決定的関数のまま変わりません。Web のクラッシュ判定は BE-0066 の決定的シグナルを保ちます（`page.on("pageerror")` による未捕捉 JS 例外、メインフレームの HTTP 4xx/5xx ステータス、または空の DOM）。いずれも LLM の判断ではありません。AI は依然「何を試すか」だけを選び、判定は下しません（[prime directive #1](../../../CLAUDE.md)）。

並列化で緩むのは**探索順序**と**記録される正準 `path_to`** です。どのワーカーが画面に先に到達するかはスケジュール依存なので、アプリ自体に非決定性があると、記録されるパス（およびタイブレークされた発見順）は実行ごとに変わりえます。決定的なアプリでは、発見されるノードとエッジの*集合*は不変で、順序やパスのメタデータだけが変わります。これが許容されるのは、まさにクロールが **Tier 1 であり、合否ではなく発見成果物を出力する**からです —— BE-0038 と BE-0064 が既に挙げているのと同じ理由です。決定的な*副産物*はその保証を保ちます。記録されたクラッシュ再現 / フローのパスは、Web の `run` で AI なしの Tier 2 リグレッションとして再生できます。

### インターフェース

* **CLI:** `bajutsu crawl --backend web --workers N`（既定 `1` = 現在の逐次 Web クロール）。各ワーカーは自分のブラウザプロセスを起動します。Web には端末が無いので、BE-0064 の `--udid a,b,c` プールとは違い、ワーカー数だけでレーン集合の大きさが決まります —— BE-0054 の単一レーン Web プール（ダミー udid 1 つ、`workers = 1`）を N へ一般化したものです。すべてのワーカーは設定された 1 つのエンジンを使います（クロスエンジンのファンアウトはスコープ外。検討した代替案を参照）。
* **WebUI（serve）:** Crawl タブの Web ターゲットにワーカー数の指定（数値入力。Web にはシミュレータ選択のような端末の複数選択が無いため）を加えます。ライブの画面マップは既に共有マップから配信されるので、描画側の変更は不要です —— BE-0064 と同じです。
* **予算と停止:** `--max-screens` / `--max-steps` はロック下で確認する共有カウンタにします。フロンティアが空になるか予算に達したら停止し、同じ `stop_reason` を報告します。
* **障害の隔離:** ブラウザが固まったワーカー —— タイムアウトする遷移、レンダラのクラッシュ、固まったページ、解決しなくなった再生 —— は現在のフロンティアエントリを破棄し、自分のブラウザプロセスを破棄して再起動してから続行します。これにより 1 つの不調がクロール全体を沈めません。プロセス単位のレーンがコストに見合うのはここです。コーディネータは他に触れずに 1 ワーカーのブラウザだけを強制終了でき、これは BE-0064 のシミュレータ単位の隔離の Web 版です。

### Web プールの再利用（BE-0054）

BE-0054 が N レーンへ一般化する [`runner/pool.py`](../../../bajutsu/runner/pool.py) の Web 分岐が、本項目の土台となる継ぎ目です。唯一の違いはレーンの隔離単位です。本項目のクロールレーンは（クロールはわざとクラッシュや固まりを誘発するので、障害隔離のために）1 つのブラウザプロセスであり、BE-0054 の `run` レーンは `BrowserContext`（より安価で、独立シナリオには十分）です。どちらも「Web プールの N レーン」であり、単位は用途ごとに選びます。

## 検討した代替案

* **別プロセスではなく `BrowserContext` レーン（BE-0054 の `run` モデル）.** クロールでは却下：N 個のコンテキストは 1 つのブラウザプロセスを共有するため、あるレーンでの固まったページ・暴走スクリプト・ブラウザプロセスのクラッシュが、全レーンを止めるか巻き込んで落とします。クロールの仕事がまさにクラッシュや固まりの誘発である以上、これは許容できません。別プロセスなら OS レベルの障害隔離と、レーン単位の強制終了 / 再起動が得られ、BE-0064 のシミュレータ単位の隔離に倣えます。（リセットごとの安価な `erase` には、各ワーカーのブラウザ*内*の新しいコンテキストを引き続き使います。）
* **クロスエンジンのファンアウト**（同じアプリを Chromium / Firefox / WebKit で同時にクロール）。別機能 —— 高速化ではなくクロスブラウザの互換性発見であり、共有フロンティアではなく差分を取るための N 個の*独立*マップを欲します。本項目は設定された 1 エンジンの N プロセスで 1 つのマップを速く作ります。将来のクロスブラウザクロール項目が本項目と組み合わさりうる（エンジンごとに本並列クロールを走らせ、マップを差分する）が、ここに畳み込むと共有フロンティアモデルと決定性の筋の両方が濁ります。
* **サイトの静的分割**（各ワーカーに URL サブツリーを割り当てる）。BE-0064 と同じく却下：グラフは動的に発見されるため、事前に分割を計算できず、ワーカーが遊ぶか冗長に探索してしまいます。
* **独立クロールを後で統合**（各ワーカーが入口から個別にクロールし、最後に重複排除）。BE-0064 と同じく却下：共有画面の冗長な再探索が膨大になります —— 共有フロンティアこそが作業を重複なく分担させる仕組み。
* **1 つのブラウザでの非同期化**（ブラウザを増やさず AI 呼び出しだけ重ねる）。AI レイテンシには効くがブラウザ / ページの操作には効かず、障害隔離も得られず、1 ページでは 2 画面を同時に操作できません。ブラウザプールが本質的な手段。後で組み合わせる価値はありますが代替ではありません。
* **本項目を BE-0064 に畳み込む.** 却下：BE-0064 は iOS シミュレータ（simctl の reset / 再生、起動済み端末プール）を前提に書かれています。Web はブラウザプロセスのレーン、コンテキストによるリセット、BE-0066 の Web クラッシュ / ダイアログ意味論を使う —— 別の設計面です。これは BE-0066 が BE-0038 に対して持つ「同じ軸・別プラットフォーム → 別項目」の関係と同じで、BE-0066 自身が並列 Web クロールを後続の交点として名指ししています。
