## はじめに

`${secrets.X}` を環境変数から解決して入力に使用し、その実値を証跡で自動的にマスクします（既存の `redact` を入力値まで拡張）。config の `secrets:` で宣言します。

## 動機

実際のフローには本物の認証情報が必要です。API トークン、パスワード、ワンタイムのログインなどです。これらをシナリオファイルに直書きすると、リポジトリにアクセスできる誰もがそのシークレットをレビューでき、コミット履歴に漏れます。さらに悪いことに、同じ値が証跡（ログ、要素ツリー、ネットワーク交信、manifest）にも現れ、ディスクにコピーされレポートで共有されてしまいます。値をシナリオに書くことなく、また取得した証跡に残すことなく、ランタイムでステップにシークレットを供給する手段が必要です。

## 詳細設計

シークレット変数名は config の `secrets:` 下に宣言します（環境変数名のリスト。`bajutsu/config.py`）。run の開始時、CLI は宣言された各名 `X` をプロセス環境から解決し、束縛 `secrets.X → <値>` を作ります（`bajutsu/cli.py`）。シナリオファイルが持つのは常にトークン `${secrets.X}` だけで、値ではありません。

ロード時に展開される `${params.*}` / `${row.*}` とは異なり、`${secrets.X}` はアクション時に run ループが解決します。`_interp_step`（`bajutsu/orchestrator/`）が、アクション実行の直前にシークレット束縛を置換したステップのコピーを作るため、実値を見るのはライブのドライバだけです。元のステップは manifest とレポートのために保持されるので、そこにはトークンが記録されます。トークンを含まないステップではコピーを省く高速なサブストリング／トークンチェックが入ります。

マスキングは既存の `redact` の仕組みを再利用して拡張します（`bajutsu/redaction.py`）。`Redactor` はシークレットの実値（`values=secret_values`）で構築され、それらが現れる箇所すべてをマスクして `[REDACTED]` に置き換えます。対象はフリーテキスト（device log や app trace）、要素ツリー、ネットワーク交信です。長い値から先にマスクするため、ある値が別の値の部分文字列であっても部分的な漏れは起きません。これは値ベースのマスキングなので、アプリがログやレスポンスに反映し返したシークレットも捕捉でき、キー名ベースのマスキングだけでは見逃すケースを拾います。最後の安全網として、レポート書き出し後に `_scrub_secret_values`（`bajutsu/runner/`）が全シークレットの実値を run レベルの成果物（`manifest.json`、`junit.xml`、`report.html`、`scenario.yaml`）に対して走査し、アサーションの期待値／実値といった結果テキストに到達したシークレットを捕捉します。画像（スクリーンショット／動画）はテキストマスクできないためそのまま残るので、画面に表示されるシークレットを撮らないことは作成者の責任です。これらすべてに LLM は関与しません。解決もマスキングも決定的であり、シークレットが run/CI の判定を変えることはありません。

## 検討した代替案

**シナリオにシークレットの値を直書きする（あるいはリテラルの別ファイルを置く）。** 不採用。値は依然としてコミットされ、レビュー可能で、証跡にもコピーされます。環境から解決すれば、値はバージョン管理から完全に外れ、ファイルにはトークンだけが残ります。

**キー名ベースのマスキングのみ（設定したヘッダ／フィールド／ラベルでマスク）。** 単独では不採用。予期しないキーの下にアプリが出すシークレットや、ログ行のより大きな文字列に埋め込まれたものを捕捉できません。実値が現れる箇所すべてをマスクすればその隙間を塞げます。一般的な（シークレットでない）ケースにはキー名ベースのマスキングが引き続き適用されます。

**暗号化されたシークレットストア／vault 連携。** 範囲外として不採用。環境は認証情報の慣例的で CI ネイティブな経路であり、追加の依存やデーモンを必要としません。vault は後から同じ環境変数を埋める形で組み込め、シナリオの表層を変えずに済みます。
