## はじめに

ロードマップの各項目は、英語の `BE-NNNN-<slug>.md` と日本語の `BE-NNNN-<slug>-ja.md` という2つの
ファイルの組であり、両者は決まった形を共有する。すなわち、対訳のヘッダリンク、H1 のタイトル、メタデータ
ブロック、そして Swift-Evolution 形式の5つの節（Introduction / Motivation / Detailed design /
Alternatives considered / References）である。この形は [`CLAUDE.md`](../../../CLAUDE.md)、
[`roadmaps/README.md`](../../README-ja.md)、[`ideation`](../../../.claude/skills/ideation/SKILL.md)
スキルの散文で説明されており、索引テーブルのほうはすでに各項目のメタデータから生成され、ゲートで検査
されている（[`scripts/build_roadmap_index.py`](../../../scripts/build_roadmap_index.py)、
`tests/test_roadmap_index.py`）。まだ固定されていないのは各項目ファイルの本体、つまりメタデータの
ラベル、見出しの文言、どの節を必須とするかである。著者は人間も AI も、その都度近くのファイルから形を
組み立て直しており、そのために細かなばらつきが入り込んでいる。

本項目はこの形を固定する。言語ごとに正規のスケルトンを一つ書き下し、規範となるフィールド集合と節見出しを
定め、隣にある索引検査と同じく `make check` で走る決定的な検査を提案する。この検査は、`BE-NNNN-*` の
ファイル組がスケルトンから逸脱したときに失敗する。対象はすべて文書とテストであり、ツールの挙動は変えず、
どの経路にも LLM を持ち込まない。

## 動機

両言語69項目を走査すると、形は「ほぼ」一様である。ヘッダリンク、タイトル行、メタデータの並び順（`Proposal`
→ `Author` → `Status` → 任意の `Implementing PR` → `Track` → `Topic` → 任意の `Origin`、`Origin`
は常に末尾）、そして Status の語彙は、どこでも一致している。逸脱は、現状どの生成器もテストも見張って
いない、ごく一部の箇所に限られる。

- **ある日本語見出しが2通りに訳されている。** `## 検討した代替案` が66ファイルにあるのに対し、同じ語を
  並べ替えた `## 代替案の検討` が2ファイル（`BE-0064-ja`、`BE-0066-ja`）にある。
- **ある日本語ファイルだけ見出しが未翻訳のままである。** `BE-0044-ja` は、兄弟ファイルが使う日本語見出し
  ではなく、英語の `## Introduction` / `## Motivation` /（以下同様）を今も持つ。
- **ある項目が必須の節を落としている。** `BE-0017`（両言語）には `## Motivation` / `## 動機` がない。
- **ある項目がメタデータのフィールドを落としている。** `BE-0064`（両言語）には Author（提案者）の行が
  ない。
- **あるメタデータのラベルが68ファイルで単数、1ファイルで複数になっている。** `* Implementing PR:` が
  どこでも使われるなか、`BE-0051` だけは2つの PR を並べるために `* Implementing PRs:` と書く。
- **提案者のラベルは、日本語側で唯一英語のまま残るフィールドである。** 他のメタデータラベルは日本語側で
  すべて訳されている（`提案`、`状態`、`トラック`、`トピック`、`実装 PR`、`由来`）のに、提案者の行だけは
  `* Author:` のままである。今は揃っているが文書化されておらず、決定の結果ではなく見落としのように読める。

どれも単独では小さい。まとめて見れば、「隣をコピーして目視で揃える」やり方が生む当然の帰結である。散文の
指針は逸脱を防がない。コミット時にその散文を読み直す者がいないからだ。衝突に強いファイルフロー
（[BE-0043](../../implemented/BE-0043-conflict-resistant-file-flow/BE-0043-conflict-resistant-file-flow-ja.md)）
は、この種の問題に対する本プロジェクトの答えをすでに確立している。不変条件を機械検査にかけ、「ローカルで
緑なら CI も緑」を成り立たせることだ。ロードマップの索引はそのように守られているが、項目の本体は守られて
いない。本項目は同じ扱いを本体へ広げる。

## 詳細設計

### 正規のメタデータ

メタデータブロックは、`| field | value |` の行を、マーカーの対 `<!-- BE-METADATA -->` …
`<!-- /BE-METADATA -->` で囲んだものである。これは索引の `<!-- GENERATED:* -->` 領域に倣う。`Field` /
`Value` のような見出し行は置かない。メタデータの場所を示すのは見出しではなく囲みであり、見出しは雑音に
なるだけだからだ。要となるのはこの囲みである。パーサがこれらの行だけを読み、本文の同形の行を決して読まずに
済むのは囲みのおかげであり、箇条書き（`* Key: value`）には境界がなくこれを与えられなかった。フィールド
ごとに一行という形は、自由な箇条書きより形を保ちやすくもある。

ブロックは以下のフィールドをこの順で持つ。`Proposal`、`Author`、`Status`、`Track`、`Topic` は必須、
`Implementing PR` の行は項目が実装されてから現れ、`Origin` の行は競合調査やドッグフード調査に由来する
項目だけに現れ、常に末尾に置く。

| 順 | 英語フィールド | 日本語フィールド | 必須 |
|---|---|---|---|
| 1 | `Proposal` | `提案` | 常に |
| 2 | `Author` | `提案者` | 常に |
| 3 | `Status` | `状態` | 常に |
| 4 | `Implementing PR` | `実装 PR` | 実装後 |
| 5 | `Track` | `トラック` | 常に |
| 6 | `Topic` | `トピック` | 常に |
| 7 | `Origin` | `由来` | 調査由来のみ |

フィールドに関する2つの決定をここで確定する。`Implementing PR` は **PR の数によらず単数** とする。値の
セルはリンクのコンマ区切りであり、フィールド名は複数形にしない（`BE-0051` の解消）。日本語の提案者
フィールドは **`提案者`** とする。英語のまま残っていた唯一のフィールドを訳すものであり、`提案`（提案
リンク）と衝突せず、その隣で自然に読める。

`Status` は固定された集合から一つの値を取り、言語間で対にする。

| 英語 | 日本語 |
|---|---|
| `**Implemented**` | `**実装済み**` |
| `**Accepted, in progress**` | `**可決・実装中**` |
| `**Proposal**` | `**提案**` |
| `**Proposal (deferred)**` | `**提案（保留）**` |

### 正規の節

どの項目も、同じ5つの H2 節を同じ順で持ち、見出しは次のとおりとする。

| 英語 | 日本語 |
|---|---|
| `## Introduction` | `## はじめに` |
| `## Motivation` | `## 動機` |
| `## Detailed design` | `## 詳細設計` |
| `## Alternatives considered` | `## 検討した代替案` |
| `## References` | `## 参考` |

`## 検討した代替案` を日本語の正規見出しとする（66ファイルの多数派）。`## 代替案の検討` は使わない。5節は
すべて必須であり、まだ書くことのない節も省略せず、一行の `TBD` を置く（`BE-0017` の欠けた `Motivation`
の解消）。

### 正規のスケルトン

英語ファイル。

```markdown
**English** · [日本語](BE-NNNN-<slug>-ja.md)

# BE-NNNN — <Title>

<!-- BE-METADATA -->
| Field | Value |
|---|---|
| Proposal | [BE-NNNN](BE-NNNN-<slug>.md) |
| 提案者 | [@handle](https://github.com/handle) |
| Status | **Proposal** |
| Track | [Proposals](../../README.md#proposals) |
| Topic | <one of the index topics> |
<!-- /BE-METADATA -->

## Introduction
...
```

日本語ファイルは同じ形で、ヘッダリンクを日本語向け（`[English](BE-NNNN-<slug>.md) · **日本語**`）に、
フィールド名（`提案` / `提案者` / `状態` / `トラック` / `トピック`）と Status の値を日本語に、節見出しを
日本語にする。タイトル行は `BE-NNNN — <タイトル>` の形を保つ（ここでのダッシュは、地の文ではなく固定された
タイトル書式の一部である）。

H1 のタイトル行は両ファイルとも `# BE-NNNN — <タイトル>` とする。ID、空白、前後を空白で挟んだ em ダッシュ
（`—`、U+2014）、続いてタイトルである。番号が未確定の新規項目は、`ideation` スキルと
[`README.md`](../../README-ja.md) が説明するプレースホルダのディレクトリ名で起草し、CI
（[`scripts/allocate_roadmap_ids.py`](../../../scripts/allocate_roadmap_ids.py)）が実番号へ書き換える。
本テンプレートはファイルのそれ以外をすべて律する。

### メタデータのパーサ

メタデータブロックは、索引生成器である
[`scripts/build_roadmap_index.py`](../../../scripts/build_roadmap_index.py) の `parse_metadata` と、
`Status` フィールドについては
[`scripts/promote_roadmap_items.py`](../../../scripts/promote_roadmap_items.py) の `read_status` が
読む。両者の規約は次のとおりである。

1. **囲みに限定する。** `<!-- BE-METADATA -->` と `<!-- /BE-METADATA -->` の間のテキストだけを読む。
   これがマーカーを置く理由である。マーカーがなければパーサは、本文にも現れる形（`* Key:` 行や
   `| a | b |` 行）を手掛かりにするため、誤った行に食いつきうる。囲みはメタデータ領域を明示し（索引
   テーブルを `<!-- GENERATED:* -->` で囲うのと同じやり方）、見出し行を不要にする。
2. **行ごとに一フィールドを読む。** 囲みの中の各 `| field | value |` 行を、`field` から `value` への
   対応として読み、値の `**` 強調は除く。読み飛ばす見出し行はない。
3. **未移行の項目には代替の読み方を用いる。** 囲みのないファイルは、既存の `* Field: value` 箇条書き規則で
   読む。これにより、まだ変換していない項目も移行のあいだ読めつづける。

両パーサはこの規約に従う（囲みの行を読み、箇条書き形式もフォールバックとして動く）。これにより、手で編集
した項目が箇条書きへ戻ってしまっても、ビルドを壊さず読めつづける。

### 決定的な検査

新しいテスト `tests/test_roadmap_format.py` を、既存の索引テストの隣で `make test` の下に走らせる。
`roadmaps/{implemented,proposals}/BE-*/` の各項目について、言語ごとに次を検証する。

1. 先頭行が、そのファイルの言語と slug に対する正確な対訳ヘッダリンクであること。
2. H1 が em ダッシュつきの `# BE-NNNN — …` であること。
3. メタデータブロックが `<!-- BE-METADATA -->` … `<!-- /BE-METADATA -->` で囲まれ、その言語の必須
   `| field | value |` 行を正規の順で並べ、見出し行も未知のフィールド行も含まないこと。
4. `Status` が4つの正規値のいずれかであり、英語と日本語の値が組内で一致すること。
5. 5つの H2 見出しが、正規の文言で、順に存在すること。

この検査は**整形器ではなくゲート**である。違反したファイルと行を報告して失敗し、修正は著者に委ねる。書き
換えるのではなく報告する `format-check` / `lint` に倣う。組の2ファイルは一緒に検証し、英日の Status 不一致
を捉える。

### 実装したこと

本項目は一つの変更として入った。すべてのロードマップ項目を囲みメタデータブロックへ移し、両パーサをそれを
軸に再定義し（手で編集した項目が壊れないよう、旧来の `* Field: value` 形式はフォールバックとして残す）、
`tests/test_roadmap_format.py` を `make test` に組み込んだ。「動機」に挙げた逸脱も同じ変更で直した。2つの
`## 代替案の検討` の見出し、`BE-0044-ja` の未翻訳見出し、`BE-0017` の欠けた `## 動機`、`BE-0064` の欠けた
提案者フィールド、`BE-0051` の複数形 `Implementing PRs`、そして日本語側で英語のまま残っていた著者フィールド
（現在は `提案者`）である。`CLAUDE.md` と `roadmaps/README.md`（および日本語版）の散文も、囲みブロックを
説明する形に更新した。ツリーは新しい検査の下で緑である。

## 検討した代替案

- **CLAUDE.md / README の散文に委ねる。** これは現状であり、先の逸脱一覧がその結果である。コミット時に誰も
  読み直さない指針は形を保てない。索引を手作業ではなく生成にした理由と同じく、却下する。
- **失敗する検査ではなく、ファイルを形に書き換える整形器**（`make format` の方式）。最初の一歩としては却下
  する。散文を含むファイルの書き換えは内容に触れる危険があり、逸脱は手で直せる程度に小さい。検査ができた
  あとの利便として、自動修正は後続にできる。
- **[BE-0069](../BE-0069-executable-contributor-guardrails/BE-0069-executable-contributor-guardrails-ja.md)
  （実行可能なコントリビュータガードレール）に統合する。** 近縁ではある。どちらもコントリビュータの手続きを
  機械検査にかける。ただし対象が異なる。BE-0069 は複数手順の**手続き**をコマンド化するもので、本項目は一つの
  **ファイル書式**とその検証器を固定する。別項目として保ち、相互参照する。
- **日本語の提案者フィールドを `提案者` ではなく `著者` に訳す。** 却下する。`提案者`（提案した人）は
  `提案`（提案そのもの）の隣で自然に読め、項目の枠組みにも合う。一方 `著者`（書いた人）は、記録としての
  著者を表すフィールドにはやや緩い。
- **メタデータを `* Field: value` の箇条書きのまま保つ**（現状の形であり、`CLAUDE.md` / `README.md` が
  「Swift-Evolution の提案形式」として示す字義どおりの形）。本項目はそこから意図して外れる。箇条書きの行に
  は境界がないため、それに限定したパーサは本文の箇条書きへ逸れうるし、箇条書きは行ごとの形より自由に
  ばらつく。囲みブロックはこの両方をふさぎ、索引自身の `<!-- GENERATED:* -->` の慣習にも合う。この逸脱は
  小さく、ここに文書化する。`CLAUDE.md` / `README.md` の散文は実装フェーズで囲みブロックへ更新する。
