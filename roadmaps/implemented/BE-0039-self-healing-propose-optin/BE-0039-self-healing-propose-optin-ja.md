## はじめに

両社は実行中にセレクタを自動補正します。Bajutsu は自己修復トリアージの方針に従い、最小差分を提案し、人間が diff をレビューしたうえで `--write` で明示的に適用します。実行中の暗黙補正は行いません。これにより、テスト条件が意図せず緩和されること（「テストを甘くする」問題、[DESIGN §11](../../../DESIGN.md) 参照）を防ぎます。

## 動機

競合調査は当然の問いを投げかけました。MagicPod も Autify もテスト実行中にセレクタを自動補正します。では Bajutsu もそうすべきか。魅力は本物です。見た目だけの UI 変化による赤い run が減ります。しかし実行中の自動補正は、テストが何を対象とするかを誰のレビューもなしにこっそり変えうることを意味し、それはまさに Bajutsu が容認しないと定める「テストを甘くする」リスク（DESIGN §11）です。本項目は意図的な決定を記録します。Bajutsu は自己修復を採用しますが、それを*提案＋opt-in 適用*に限定し、価値（提案された最小 fix）は残しつつ、コスト（黙った実行中のドリフト）を拒否します。

## 詳細設計

機構は既に構築済みの自己修復トリアージ（BE-0021／BE-0022／BE-0023）であり、ここでは競合の手法に対する意図的なスコープ選択として位置づけます。

* **提案するが、実行中に自動補正しない。** トリアージは失敗の*後*、保存済みの証跡に対して動きます。決定的な `run` の中ではありません。失敗した run は失敗のままで、実行中に黙ってターゲットが付け替えられることはありません。
* **opt-in 適用。** 提案された fix は unified diff として表示されます。明示的な `--write` のときだけシナリオ source に到達し、通常の決定的な `run`（`--rerun`）の再実行で検証されます。コミット済み YAML は人間が diff を読まない限り変わりません。
* **制約された fix。** 提案は `renameId`／`addIndex`／`raiseTimeout` のいずれかで、`find` が source の正確な部分文字列でなければならない `find` -> `replace` です。不一致は安全な no-op です。

MagicPod／Autify との対比こそが要点です。両社はその場で補正して赤い run を決して見せないことを最適化します。Bajutsu はすべての補正を可視化しレビューさせることで、信頼できるゲートを最適化します。

## 検討した代替案

* **実行中の暗黙的なセレクタ補正（MagicPod／Autify の手法）。** 却下しました。人間をループから外し、レビューなしにテストのターゲットを緩めうるため、決定的ゲートを損ないます。
* **run の後に提案 fix を自動適用する。** 実行中補正よりは危険でないものの、レビュアーなしにコミット済みシナリオへ書き込むため、誤った fix や甘い fix が気づかれず取り込まれる恐れがあります。opt-in の `--write` が人間をゲートに保ちます。
* **自己修復をまったく行わない。** 安全ですが、競合機能の本当の価値、すなわち見た目だけの破損に対する具体的な最小 fix を捨ててしまいます。提案＋opt-in 適用はその価値を残しつつ、黙ったドリフトのコストを拒否します。
