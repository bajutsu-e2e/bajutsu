## はじめに

シナリオの `tags` を `--tag`/`--exclude` でサブセット実行します（include/exclude、exclude 優先、`select_scenarios`）。CI の段階的な実行に利用できます。

## 動機

スイートが大きくなると、毎回すべてのシナリオを実行すべきではなくなります。マージ前のゲートは高速なスモークのサブセットを、ナイトリーは全件を、機能ブランチはその機能に関わるシナリオだけを実行したいのです。フレーキーな、あるいはプラットフォーム固有のシナリオを、ある run から除外したいこともあります。選択の仕組みがないと、つまみは「この 1 ファイルを実行」か「全部実行」しかなく、不格好なファイル構成を強い、段階的な CI パイプラインを不可能にします。シナリオファイルを分割したり再編成したりすることなく、シナリオにラベルを付け、ランタイムにラベルでサブセットを選べるようにしたいのです。

## 詳細設計

各シナリオは任意の `tags` リストを持ちます（`bajutsu/scenario/` の `Scenario`）。`smoke` や `checkout` のような自由形式のラベルです。CLI は `--tag` と `--exclude` を提供し、いずれもカンマ区切りのリストで、シナリオ全件をロードして展開し終えた後に適用されます（そのため選択はデータ駆動展開とコンポーネント展開を経たすべてのシナリオを見ます）。`select_scenarios` は、シナリオが少なくとも 1 つの `--tag` を持つ（または `--tag` 未指定）、かつそのタグのいずれも `--exclude` に現れない場合にそのシナリオを残します。`--exclude` が `--tag` より優先されます。フィルタリングは純粋なメタデータ選択で、宣言順を保ち、シナリオを変更しません。フィルタが何も選ばなかった場合、CLI は空のスイートを実行せず、明確なメッセージとともにクリーンに終了します。選択は決定的な run ループの完全に前で行われるため、選ばれたシナリオの実行のしかたや合否には影響しません。タグは選択以外の意味を持ちません。

## 検討した代替案

**サブセットごとに 1 ファイル（ファイル構成で選択を表現する）。** 不採用。シナリオの選択をその置き場所に結び付け、シナリオが複数のサブセットに属するとき（例: `smoke` と `checkout` の両方）に重複を強い、サブセットの組み合わせが増えると破綻します。タグなら、シナリオを自然なファイルに置いたまま、複数のサブセットに同時に属させられます。

**include だけの選択（`--exclude` なし）。** 不採用。小さな集合の除外（「遅いものを除く全部」）に、欲しいタグをすべて列挙させることになります。両方向をサポートし `--exclude` を優先させれば、段階的なパイプライン（広く include し、既知のいくつかを引く形）を自然にカバーできます。

**タグに対するクエリ／式言語（ブール結合）。** 必要に対して過剰として不採用。明確な優先規則を持つカンマ区切りの include/exclude リストで、段階的 CI のユースケースは、完全な式文法のパース面と曖昧さなしにカバーできます。具体的な必要が現れたら、より豊かな文法を後から重ねられます。
