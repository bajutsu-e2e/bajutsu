## はじめに

クロール周辺のコードは、現在の画面をベストエフォートでスクリーンショットに撮る処理を複数箇所に
持っており、そのうち二つのキャプチャ用ヘルパーは完全に同一の実装へと重複したうえ、自身の失敗を
隠してしまっています。本項目はこれらを一つのヘルパーに統合し、キャプチャの失敗を握りつぶさずに
表面化させます。エラーを黙って飲み込むのではなく大きく失敗させるという、決定性優先の原則に
沿った対応です。

## 動機

`bajutsu/record.py:52` は `_screenshot_bytes` を定義し、`bajutsu/alerts.py:48` は
`_screenshot_png` を定義していますが、両者の本体は完全に同一です。一時ファイルへ PNG を書き出し、
それをバイト列として読み戻し、一時ファイルを削除し、例外が起きた場合は `None` を返します。
`bajutsu/crawl_guide.py:82` と `bajutsu/enrich.py:92` はいずれも `record.py` の
`_screenshot_bytes` をインポートして自前のベストエフォートキャプチャに使っており、同じロジックが
すでに二つの別々に保守されているコピーを通じて四箇所の呼び出し元から依存されている状態です。
重複そのものに加えて、両方のコピーはキャプチャ処理全体を素の `except Exception: return None` で
囲んでおり、実際の失敗原因（古くなったシミュレータ、権限エラー、ディスクの空き容量不足など）を
握りつぶし、それを「そもそもキャプチャすべきものが何もなかった」場合とまったく同じ形で報告して
しまいます。`None` を受け取ってスクリーンショットの利用をスキップする呼び出し元、つまりアラート
ロケーターのプロンプトやクロールガイドの注釈、エンリッチメントの記録は、実際の失敗と想定内の
不在を区別できません。これは、ツールがシグナルとして扱うものについて決定性優先の原則が排除
しようとしている、まさにその種の曖昧さです。これはサイズ M/S の作業で、自己完結した 10 行ほどの
ヘルパーを統合し、その四つの呼び出し元に反映させる規模の変更です。

## 詳細設計

修正は重複解消とエラー処理という二つの独立した部分から成り、ヘルパーを統合する機会に合わせて
失敗時の挙動もあわせて直します。

- **二つのヘルパーを一つに統合する。** 実装を一つに絞り（呼び出し面がすでに広い
  `bajutsu/record.py` の `_screenshot_bytes` を残します）、`bajutsu/alerts.py` は独自の
  `_screenshot_png` を定義する代わりにこれをインポートして使うようにし、重複を削除します。
  `screenshot_bytes` という名前は、PNG オブジェクトではなくバイト列を返す関数の名前として
  `_screenshot_png` より実態に即しているため、呼び出し元で意味上の調整は不要です。
- **エラーを黙って握りつぶすのをやめる。** 素の `except Exception: return None` を、例外を
  そのまま伝播させる形（キャプチャの失敗が呼び出し元の処理全体を中断させても構わない場合）か、
  `None` を返す前に例外をログに記録する形（呼び出し元が本当にキャプチャの失敗をベストエフォート
  として扱い処理を続ける必要がある場合）のいずれかに置き換えます。どちらを選ぶかは、
  `bajutsu/record.py:249`、`bajutsu/alerts.py:86`、`bajutsu/crawl_guide.py:82`、
  `bajutsu/enrich.py:92` の各呼び出し元を実際に調べたうえで決めます。要点は、実際の失敗を
  空のキャプチャと区別できるようにすることであり、両者を一つに混ぜないことです。
- **四つの呼び出し元とそのインポートを更新する。** `bajutsu/alerts.py` は独自の
  `_screenshot_png` から統合後のヘルパーのインポートへ切り替えます。`bajutsu/crawl_guide.py` と
  `bajutsu/enrich.py` はすでに `record.py` からインポートしているためインポート自体の変更は
  不要ですが、（名前が変わる可能性のある）統合後の関数に対して動き続けるようにします。
- **統合後のヘルパーに対するユニットテストを追加または拡充する。** 成功パスと失敗パス
  （`screenshot()` が例外を送出するドライバ）の両方をカバーし、新しいエラー表面化の挙動を
  固定して、黙って握りつぶす挙動へ後戻りしないようにします。

## 検討した代替案

- **両方のヘルパーを残しつつ、一方をもう一方に委譲する。** 却下しました。これでも一つの挙動に
  二つの名前が残り、重複をそのまま削除するのに比べて余計な間接層が増えるだけで得るものが
  ありません。
- **重複解消はせず、両方のコピーでその場のまま握りつぶしだけを直す。** 却下しました。決定性の
  懸念には対処できますが、実際の重複そのものは残り、次に挙動を変更する際にも二つのファイルへ
  反映する必要が生じます。これはより根深いリスクです（BE-0064、BE-0066、BE-0092 はいずれも
  周辺のクロールコードに手を入れていますが、誰もこの重複には気づいていませんでした）。
- **重複は解消するが、素の `except Exception: return None` はそのまま残す。** 却下しました。
  保守負担は解消できますが、今回の指摘がまさに問題視しているサイレント障害はそのまま残ります。
  スクリーンショットの失敗は、`None` の中に消えるのではなく大きく失敗するべき性質のものです。
- **統合後のヘルパーを、`record.py` に残さず新しい共有モジュール（例えば `screenshots.py`）へ
  移す。** 検討しましたが、着手する人の実装判断に委ねます。`record.py` はすでにこの関数を
  所有しており、四つの呼び出し元のうち三つは既存のインポート経路のままで問題ないため、新しい
  モジュールを作る価値があるのは、実際に結合度を下げる効果が確認できた場合に限られます。
