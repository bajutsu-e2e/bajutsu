## はじめに

serve Web UI に、第一級の**プラガブルなテーマシステム**を導入します。ここでのテーマとは、二つの
ものを束ねた名前付きのバンドルです。一つは見た目を決めるビジュアルデザイントークンの集合（色と、
背景・境界線・アクセントのパレット）、もう一つは画面遷移の定義（ビュー・モーダル・ペイン・実行
ステップの進行がどうアニメーションするか）です。テーマは `<html>` の `[data-theme="<name>"]` で
選択します。要点は、operator が**自作のテーマをドロップインできる**ことです。今日シナリオや設定
ファイルを置くのとまったく同じ要領で、ソースには一切手を入れずに追加できます。組み込みのダーク・
ライトの対（`midnight` / `daylight`）は、開かれたレジストリの最初の二要素にすぎなくなります。テーマは
**UI の中で作ることもできます**。ライブプレビュー付きのガイド付きエディタで編集し、ローカルに保存する
か、serve インスタンスへアップロードします。ドロップインのファイルは作成経路の一つであって、唯一の
経路ではありません。

この項目の対象は、[`bajutsu serve`](../../README.md) が配信するライブのシングルページアプリケー
ション（SPA、`bajutsu/templates/serve.html.j2`）に限ります。独立したレポート・カバレッジ・統計・
クロールの各ページは、別系統の古い `prefers-color-scheme` 機構を使っており、ここでは対象外です
（「検討した代替案」を参照）。

## 動機

serve UI にはすでに動作するダーク・ライト機構がありますが、これは専用のロードマップ項目を持たない
まま場当たり的に育ったもので、見た目はちょうど二種類だという前提をハードコードしています。

- **見た目を差し替えられない。** テーマは `bajutsu/templates/serve.themes.css` に、`[data-theme]`
  で選ばれる CSS カスタムプロパティの固定 2 ブロックとして存在し、ヘッダーのコントロールはダーク・
  ライトの**二択トグル**です。このファイル自身の冒頭コメントが制約をはっきり書いています。「3 つ目の
  テーマを足すなら、この二択スイッチをもっと広いピッカーに置き換えることになる」。ユーザーが自分の
  見た目を足す手段はありません。発見の仕組みも、ピッカーも、設定フックもないのです。
- **トークン契約が不完全で、テーマが「すべてを定義している」と信頼できない。** `serve.css` は、
  レジストリが定義しないカスタムプロパティ（`--accent`・`--muted`・`--bad`）を参照しており、一方で
  レジストリが定義しているのは `--acc`・`--mut`・`--ng` です。これらの規則は実際に使われている UI
  （Author タブのステップ一覧、enrich の採用・却下、カバレッジのサブヒント）を支えており、現状では
  意図したテーマ色ではなく未設定・継承色で描画されています。すべての色が定義済みトークンになるまで、
  ドロップインのテーマは完全にはならず、フォールバックも当てにできません。
- **トランジションはそもそも見た目の一部になっていない。** ビュー切り替えは即時の
  `replaceChildren` で起こり、ビュー・モーダル・レスポンシブなペイン再構成（BE-0072）・実行ステップの
  進行について、テーマ化できる enter/leave アニメーションという概念がありません。動き（モーション）は
  UI のアイデンティティの大きな部分ですが、今日それは固定で、どのテーマからも見えないままです。
- **ファイルシステムに触れずにテーマを作る手段がない。** テーマがドロップインのファイルになったとして
  も、一つ作るにはディスク上の生の CSS を編集し、正確なトークン名を覚えている必要があります。アクセント
  色を少し変えたい、ビューの遷移を少し遅くしたい、という程度のユーザーには、製品内の経路がありません。
  トークン契約はコメントに書かれているだけで、編集して変化を確かめられる対象としては露出していないので
  す。

プラガブルなテーマシステムは、見た目と操作感を**バックエンド非依存のインターフェース**に変えます。
これはドライバ層が拠って立つのと同じ設計原則です。ツールがトークンとトランジションの*契約*を定め、
テーマはその*実装*として自由に差し替えられます。アプリごとの違い（ある組織のブランドの見た目）は、
コアではなくドロップインされたテーマと設定に置かれます（プライム原則 3）。

これらは決定論的な経路には一切触れません。テーマ化は純粋な表示層です。Tier-2 の `run` / CI の判定に
入り込むことはなく（プライム原則 1）、固定の `sleep` も持ち込みません（プライム原則 2）。アニメーション
は CSS が駆動し、`prefers-reduced-motion` の下では無効になるので、ドッグフードしている serve UI の
回帰テスト網は条件待ちベースのまま、決定論を保ちます。

## 詳細設計

作業は MECE に 6 つの単位へ分解します。単位 1〜3 がプラガブルな*見た目*のテーマを実現し、単位 4 が
差し替え可能な*トランジション*層を足し、単位 5 が変更を決定論的かつドッグフードと矛盾しない状態に
保ち、単位 6 が UI 内のエディタとアップロードを足して、ファイルシステムに触れずにテーマを作れるように
します。

### 1. デザイントークン契約の完成（前提作業）

`serve.css` のすべての色・サーフェススタイルを定義済みカスタムプロパティに解決させ、テーマを完全で
自己完結した契約にします。これにより、部分的なテーマも予測どおりにフォールバックします。

- 命名のずれを解消します。正規の集合を一つに定め、未定義変数を参照する規則がなくなるように、すべての
  消費側を更新します（今日 `--accent` / `--muted` / `--bad` は何にも解決しません）。レジストリ側で
  不足名をエイリアスするか、`serve.css` の消費側を書き換えるか、どちらか一方向に統一して適用します。
- `serve.css` の生のカラーリテラル（約 15 箇所、多くは色付き背景上の `#fff`）を洗い出し、それぞれを
  トークン経由（`--on-acc`・`--on-run` など）に通して、色がシステムを迂回しないようにします。
- トークン契約を `serve.themes.css` にインラインコメントで明文化します（必要なトークン名と、各々の
  意味）。このコメントこそがテーマ作成の API 面です。
- フォールバックの規則：テーマが省いたトークンは `:root` の既定ブロック（`midnight`）から継承します。
  これにより、不完全なドロップインテーマは壊れるのではなく穏やかに劣化します。

### 2. プラガブルなテーマの発見と登録

固定の 2 ブロックのレジストリを、serve 起動時に発見される開かれた集合に変えます。

- **テーマの供給元：** テーマ用ディレクトリを用意し、`bajutsu serve` の新しい `--themes <dir>` CLI
  フラグから解決します（既存の `--scenarios` / `--runs` / `--baselines` のディレクトリフラグに倣い
  ます）。初期選択のために、設定に任意の `ui.default_theme` フィールドを持たせます。組み込みテーマは
  今日どおりリポジトリに同梱し、発見されたテーマは組み込みを置き換えるのではなく拡張します。
- **テーマの形（プラグ）：** テーマは**宣言的なものだけ**です。トークン値の CSS ブロック（および
  単位 4 のトランジション定義）と、小さなマニフェスト（表示名、`dark` か `light` かの種別）から
  なります。テーマに**任意の JavaScript は持たせません**。これにより信頼境界をシナリオや設定と同じ
  水準（operator 自身がファイルを置く）に保ちつつ、攻撃面を CSS に限定します（安全性のトレードオフは
  「検討した代替案」を参照）。
- **登録：** serve は起動時にディレクトリを一度走査し、発見した CSS をインライン化するテーマ
  スタイルシートに連結し、ピッカーが選択肢を描画できるようテーマのマニフェストをクライアントへ渡し
  ます。テーマはプロセスの生存期間中は静的なので、`bajutsu/serve/handler.py` の `_asset` /
  `_index_html` にかかる既存の `functools.lru_cache` はそのまま有効で、発見された集合はキャッシュ
  された描画結果に畳み込まれます（テーマのライブリロードは対象外です）。

### 3. テーマピッカー UI

レジストリのコメントが予告したとおり、ヘッダーの二択トグルをピッカーに置き換えます。

- 組み込みと発見済みのテーマを列挙するヘッダーのドロップダウン（メニュー）にし、ダーク・ライトの
  種別で分類・ラベル付けして、月・太陽のチェックボックスを置き換えます。
- 明示的な選択を `localStorage['bajutsu-theme']`（キーは変更なし）に永続化し、間違ったテーマの
  ちらつきを避けるために既存のインライン `<script>` で初回描画前にシードします。OS のスキーム変更は、
  ユーザーが手動選択をしていない間に限り、引き続きライブで反映されます。
- BE-0058 の `data-testid` 規約に従い（例：`nav.theme-picker`）、ヘッダーのコントロール群に対して
  曖昧さのないセレクタを用います。
- **配置：** ピッカーは `<header>`（今日のトグルと同じ）かモーダルに置きます。`main#view-*` 要素の
  直接の子には**しません**。デスクトップのタイラーの `rebuild()` がそれらの要素に対して
  `replaceChildren` を呼ぶため、タイル再構成のたびにピッカーが消えてしまうからです。

### 4. テーマが定義する差し替え可能なトランジション

議論で挙がった 4 つの面すべてにわたって、画面遷移をテーマ化できる見た目の一部にします。4 つの面とは、
**ビュー切り替え、モーダルの enter/leave、レスポンシブなペイン再構成（BE-0072）、実行ステップの進行**
です。

- **関心の分離：** JavaScript は各遷移点で*意味的な状態クラス*（例：`.is-entering` /
  `.is-leaving` / `.view-switching`）を当てるだけにし、**その状態がどう見えるかはテーマの CSS が
  決めます**。`--transition-*` トークン（duration / easing）とキーフレームを介して定義します。これに
  より JS はテーマ非依存のまま、アニメーション自体が完全にプラガブルになります。トランジション規則を
  持たないテーマは即時に描画します（今日の挙動）。
- **タイラーの改修（主要なコスト）：** ビュー切り替えは今、`replaceChildren` で古いサブツリーを
  即座に破棄するため、leave アニメーションを見せる要素が残りません。leave アニメーションを可能にする
  には、遷移が続くあいだ退場するノードを保持する必要があります。`rebuild()` に二重バッファのステップを
  入れるか、より軽い第一版として入場するビューだけをアニメーションさせます。ここが変更のうち最も
  リスクの高い部分なので、後述の reduced-motion のガードの背後に置いて、今日の即時挙動へ劣化できる
  ようにします。
- **モーション低減：** `prefers-reduced-motion: reduce` の下では、有効なテーマにかかわらずすべての
  トランジションが即時に畳まれます。これはアクセシビリティ要件であり、同時に単位 5 の決定論のための
  てこでもあります。

### 5. 決定論とドッグフードとの整合

- serve UI 自体がドッグフードの対象です（BE-0058、`demos/serve-ui/`）。単位 4 の reduced-motion に
  よる畳み込みが、追加したモーションが競合状態や固定 `sleep` の必要を決して生まないことの保証になり
  ます。モーションを切れば遷移は即時になり、既存の条件待ちのアサーションはそのまま成り立ちます。
- `demos/serve-ui/scenarios/theme.yaml` は現在 `nav.theme-toggle` をタップして `checked` 状態の反転を
  アサートしています。二択からピッカーへの変更（単位 3）はこの前提を崩すので、シナリオを同時に更新し、
  ピッカーを操作して `[data-theme]` をアサートするようにします。
- この経路のどこにも LLM は入りません（プライム原則 1）。テーマの選択と描画は、すべて決定論的な
  クライアント・サーバーのコードです。

### 6. UI 内のテーマエディタ、ライブプレビュー、アップロード / エクスポート

ファイルシステムに触れずにテーマを作成・共有できる製品内の経路を足します。単位 1 のトークン契約の上に
直接構築します。

- **契約から導かれるエディタ：** 明文化した必要トークンから生成するフォームにします。色トークンごとに
  カラー入力、`--transition-*` トークンごとに duration / easing / 効果の入力を持ちます。契約こそが API
  なので（単位 1）、エディタのフィールドは手で保守するのではなく契約から導かれます。
- **クライアント側のライブプレビュー：** 編集を適用すると、稼働中の SPA の有効な `[data-theme]` の
  カスタムプロパティを上書きする `<style>` ブロックを注入し、サーバー往復なしで変化を即座に見せます。
  配置は単位 3 に倣い、モーダル（`#settingsmodal` と同様）にして、タイラーが消してしまう `main#view-*`
  要素の直接の子には決してしません。
- **永続化の 2 層：**
  - *ローカル下書き* ── 編集したテーマを `localStorage` に `custom` テーマとして保存し、ピッカー
    （単位 3）に即座に出します。ブラウザ内のみで、サーバー面も新たな信頼境界も増やさず、見た目を試行
    錯誤するのに摩擦のない経路です。
  - *serve インスタンスへのアップロード* ── テーマを `--themes` ディレクトリ（単位 2）へ書き出し、
    発見可能なドロップインにして、そのインスタンス内でセッションをまたいで共有します。既存のアップ
    ロードの継ぎ目（BE-0073）を再利用します。ここがテーマの `lru_cache` を無効化しなければならない
    **唯一**の箇所です。その書き込みに限った、範囲の限られた明示的な再走査で、単位 2 が除外する汎用の
    ライブリロードとは別物です。
- **エクスポート / インポートのラウンドトリップ：** テーマは宣言的な CSS＋マニフェストなので、「この
  テーマをダウンロード」はドロップイン機構とラウンドトリップするファイルを生み（UI で編集 → エクス
  ポート → リポジトリにコミット、または共有）、「インポート」はそのようなファイルを取り込みます。これ
  で 2 つの作成経路が閉じます。
- **信頼と決定論は不変：** アップロードするテーマも、JavaScript を持たない宣言的な CSS＋マニフェスト
  のままです（単位 2）。アップロードするのは自分の localhost serve を操作する認証済みの operator なので、
  アップロードされたテーマはドロップインのファイルと同じ信頼を帯びます。残る CSS の面（例：テーマが
  外部 URL を参照する）は、その信頼水準で許容し、サンドボックス化するのではなく注記します。LLM は入らず、
  `run` 経路への影響もありません。

## 検討した代替案

- **ビルドステップや CSS フレームワークの導入（Sass、Tailwind、バンドラ）。** 採用しません。serve UI
  は意図的にビルド不要の素の JS/CSS で、ディスクから直接読み込みます。CSS カスタムプロパティがすでに
  ネイティブに表現できる機能のために、その制約を破ることになります。
- **ファイルを介さず、API や `localStorage` 経由で任意の CSS を注入する。** 採用しません。アセット
  キャッシュを複雑にするうえ、より重要な点として信頼境界を動かします。ドロップインされたファイルは
  operator のシナリオや設定と同じ信頼水準にありますが、リモートや貼り付けの CSS 経路は、より広い面から
  の注入を招きます。ドロップインのテーマファイルが信頼境界です。
- **テーマに任意の JavaScript を持たせる**（完全にプログラム可能なテーマ）。同じ理由で採用しません。
  宣言的な CSS とマニフェストからなるテーマは、見た目と動きを表現するのに十分な表現力があり、しかも
  攻撃面を CSS に限れます。
- **UI 内エディタをクライアント専用（`localStorage`）にし、サーバーへのアップロードを持たない。**
  それだけを全体像とするのは採用しません。ブラウザ内のカスタムテーマは試行錯誤には優れますが、一つの
  ブラウザに閉じ込められます。両方の層を残します。摩擦のない編集のためのローカル下書きと、serve
  インスタンスをまたいだ永続化・共有のための任意のアップロードです。
- **ホスト型のテーママーケットプレイス / リモートのテーマレジストリ。** 採用しません。serve インスタンス
  はローカルのドロップインとアップロードされたテーマを発見しますが、ネットワーク越しにテーマを取得は
  しません。テーマの共有は、中央サービスではなくエクスポート / インポートのファイルのラウンドトリップ
  （およびバージョン管理）に乗せます。
- **設定のみ、またはピッカーのみ（両方は持たない）。** 薄い 2 層を持つ方を採ります。設定
  （`ui.default_theme` と `--themes`）がデプロイ・ターゲットごとの既定を決め、クライアントのピッカーが
  ユーザーにローカルな上書きを許します。これは、OS 既定のあとに手動上書きが効く現在のトグルの挙動に
  合っています。
- **独立したレポート・カバレッジ・統計・クロールの各ページを同じテーマシステムに統合する。** 今回は
  採用せず先送りします。この項目の対象は SPA に限ります。これらのページは別系統の古い
  `prefers-color-scheme` 機構を、異なる変数セット（`--ink` など）とトグルなしで使っており、統合には
  価値があるものの、別の、より大きな変更になります。関連するフォローアップとして残します。

- [ ] 4. テーマが定義する差し替え可能なトランジション（4 つの面での意味的状態クラス＋`--transition-*` トークン、タイラー `rebuild()` の改修、`prefers-reduced-motion` での畳み込み）。
- [ ] 5. 決定論とドッグフードとの整合（reduced-motion が条件待ちの安全性を保証、`demos/serve-ui/scenarios/theme.yaml` をピッカー向けに更新）。
- [ ] 6. UI 内のテーマエディタ、ライブプレビュー、アップロード / エクスポート（契約から導くフォーム、クライアント側ライブプレビュー、BE-0073 のアップロード継ぎ目を再利用するローカル下書き＋アップロードの永続化 2 層、エクスポート / インポートのラウンドトリップ）。
