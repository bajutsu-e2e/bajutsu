## はじめに

record における human-in-the-loop の二つのパターン、すなわち人による値入力
（`record-human-value-prompt`）と人による操作の引き取り（`record-human-takeover-step`）が
共有する土台です。本項目は、AI 駆動の `record` ループが途中で一時停止し、人に制御を引き渡し、
その入力や操作を受け取ってから再開する仕組みを定めます。具体的には、一時停止の契機、要求と応答の
契約、CLI と `serve` の両インターフェース、そしてそれらすべてを縛る不変条件です。不変条件とは、
ハンドオフが記録する成果物は、実行経路に人を置かずに再び決定論的に実行できなければならない、という
ものです。二つのパターンはこの土台の上に乗り、記録される成果物の「形」を決めます。本項目が担うのは
「仕組み」と「境界」です。

## 動機

`record` は AI エージェントを 1 ステップずつ駆動します。画面を観測し、次の操作を提案し、それを
実行する、という流れです（[`bajutsu/record.py`](../../../bajutsu/record.py) の `record()`
ループ）。エージェントに有効な手が無いとき、`record` は単に停止します。エージェントが操作を提案
しないか、提案されたターゲットが実際の画面で解決できないか（`could not resolve that target on the
live screen; stopping`）のいずれかです。人に助けを求める経路はありません。

このため、AI が供給できない何かで塞がれた flow は、端から端まで記録できません。エージェントが
入力欄は見えているのに値を知り得ないワンタイムパスワード（OTP）や二要素認証（2FA）のコード、
エージェントが解けない CAPTCHA や生体認証のプロンプト、繰り返し解決に失敗するジェスチャやターゲット
などです。現状では、著者はその手前で記録を諦め、残りのシナリオを手で書くしかありません。これは
`record` が取り除くために存在する、まさにその手作業です。

塞がれた箇所ごとに個別の抜け道を継ぎ足すのではなく、ツールには一つのプリミティブが必要です。
ループを一時停止し、人に明確な要求を提示し、応答を受け取り、実際の画面から再開する、という
プリミティブです。二つの具体的なケース、すなわち値を供給する場合と操作を行う場合は、何を求め何を
記録するかが違うだけで、ループがどう引き渡すかは変わりません。

この仕組みは prime directive の内側に収まります。全体が **Tier 1**（`record` とオーサリング）に
とどまるからです。人はオーサリングの最中にループへ入るのであって、決定論的な `run` や CI のゲート
には決して入りません。土台の中心的な責務は、この境界を守ることにあります。あらゆるハンドオフは
再実行可能な成果物へ解決されなければならず、そうであれば人の助けを借りて作った記録も、人が居ない
まま再生できます。ハンドオフが人だけに依存する要素を実行経路に焼き込めてしまう設計は、directive 1
と 2 に反します。それを防ぐことが本項目の役割です。

## 詳細設計

作業の対象は仕組みとその保証です。パターンごとの挙動は二つの子項目が受け持ちます。

**一時停止の契機。** ハンドオフは、静かな停止ではなく明示的な要求として、次の二つの状況で立ち上がり
ます。(a) エージェント自身が先へ進めないと表明したとき（既存の `done` や「操作なし」とは区別される、
新しい「人が必要」というターン結果）、(b) 著者が任意のターンで自ら引き取りを要求したとき、です。
特定の詰まりに対する検出のヒューリスティック（OTP らしき入力欄、解決できないターゲット）は子項目に
属します。本項目が定めるのは、その結果と、ループがそれにどう反応するかだけです。

**要求と応答の契約。** 転送方式に依存しないハンドオフの「要求」（何を、なぜ求めるか、そして現在の
画面。要素の要約とスクリーンショット）と、「応答」（供給された値、または「デバイスを操作したので
観測し直せ」）を定義します。一度だけ定めることで、CLI と `serve` の両インターフェースが同じ
プロトコルを実装し、二つの子パターンがそれをそのまま再利用できます。

**CLI インターフェース。** 端末から駆動する `record` は、要求で待機し、対話的なプロンプト（標準
入力）から応答を読み取ります。待機は上限つきで中断可能とし、際限なくハングすることはありません。
応答者の居ない非対話的な実行や CI での実行は、永久に待つのではなく、後述のとおり明確に失敗します。

**`serve` インターフェース。** `serve` の Web UI から駆動する `record` は、同じ要求を record ビュー
の UI 要素として立ち上げます（統合オーサリングサーフェス、BE-0098）。人はブラウザ上で応答し、ループ
が再開します。UI は serve のレイアウト規約に従い、保持されるペインの内側に追加します。

**再開の意味論。** 応答のあと、ループは実際の画面を問い合わせ直し、観測した状態から続行します。
観測して提案して実行する、という同じサイクルです。人の寄与（入力した値、手で行った操作）は新しい
起点として吸収されるので、下流で特別扱いする必要はありません。

**決定論的な出力という不変条件。** 土台は、ハンドオフが再実行可能な成果物へ解決されるという契約を
定義し、それを強制します。成果物の形（値のプレースホルダか、明示的な手動ステップか）を土台自身が
選ぶことはしません。それは子項目の判断です。土台が保証するのは、いかなるハンドオフも、人だけに
依存する要素を実行経路へ静かに記録できないことです。

**非対話的な実行と CI での挙動。** 人の応答者が居ない環境（CI、ヘッドレス）で `record` を走らせた
とき、立ち上がったハンドオフは明確でラベル付きの失敗になります（「この flow は人によるハンドオフが
必要です。対話的に記録し直してください」）。際限のないハングにも、AI による推測にもなりません。
これにより、自動化のもとでツール自体の決定性が保たれます。

## 検討した代替案

- **塞がれた箇所ごとに個別のコード経路で解決し、共有の土台を持たない。** 却下します。値のケースと操作
  のケースがそれぞれ一時停止と再開の配管を作り直し、しかも両者が CLI と `serve` の双方でそれを重複
  させます。仕組みを一度だけ切り出すことが、二つの子項目を小さく一貫させます。
- **エージェントに値や操作を推測（ないし hallucinate）させて先へ進める。** 却下します。これは
  determinism first に反します。誤った値や誤ったタップは記録を静かに壊しますし、それこそ本作業が
  取り除く失敗の形です。AI が解決できない詰まりは、覆い隠すのではなく表に出さなければなりません。
- **実行経路に「人を待つ」ステップを置く（毎回の `run` が止まる手動ステップ）。** 土台のレベルで
  却下します。それは決定論的な `run` と CI のゲートに人を置くことであり、directive 1 と 2 に反します。
  土台は、人を record の時点にとどめ、決定論的な出力を強制するためにこそ存在します（手動の目印が避け
  られない、明示的に CI 非対象の狭いケースは操作引き取りの子項目で論じますが、そこでも CI で静かに
  合格することは決してありません）。
