## はじめに

ステップループ（`bajutsu/orchestrator/loop.py`）は、必要以上の回数でアクセシビリティツリーを
`driver.query()` によって取得しうるものでした。BE-0234（Implemented、#987）がこの隙間の大半を
すでに解消しています。ループのステップ後の読み取りは今や遅延実行になっており、`screenChanged`
キャプチャ、付随する `extract`、または wait タイムアウトの診断が実際にツリーを必要とするときだけ
行われ、さらに前ステップの `after` を次ステップの `before` として使い回します（`loop.py` の
`BE-0234 Unit 2` コメント）。そのため、それらが付随しない素の `assert` は、もう二重に query しません。
本項目が対象とする残りの隙間はより狭いものです。画面を変化させないステップ（`assert`、あるいはそれに
付随する `extract`）が、そのステップ後の読み取りを実際に発火させる場合、`_run_step_body` はステップ
評価のためにすでにツリーを解決・query しており、その間に画面は何も変化していないのに、ステップ後の
読み取りがもう一度取得します。本項目は、この確定済みスナップショットを再取得せずに受け渡し、同じ発想を
1 層下、つまり単一の `tap` がステップループの下でさらに重複した取得を積み重ねている箇所にも適用します。

## 動機

`bajutsu/orchestrator/waits.py:64` は、この件が実機バックエンドでなぜ重要かを記しています。
idb ドライバの `query()` が実行する `describe-all`（idb の describe-all コマンド）は、
Simulator へのサブプロセス経由の呼び出しであり、インプロセスの参照ではないため、1 回あたり
おおよそ 100〜300ms かかります。`assert` ステップで状態を確認するシナリオ、つまり作用する前に
状態を読むという通常の書き方をしたシナリオは、ステップ後の読み取りが発火する場合（`assert` に
`screenChanged` キャプチャや `extract` が付く場合）に、追加の情報を何も得ないままこのコストを
2 回払うことになります。`_run_step_body` は `assertions.evaluate` を実行した
時点ですでに解決済みの要素一覧を保持しており、その呼び出しとループ自身のステップ後の query の
あいだで画面を変化させるものは何もありません。

同じ構造は、単一の `tap` の内部でも積み重なります。idb ドライバ（`bajutsu/drivers/idb.py:317`）
では、`_center` が `_settle()`（query を 1 回、ツリーの identifier と frame の組が安定していなければ
さらに複数回）を呼び、続けて `_resolve`（`idb.py:296`）を呼びます。`_resolve` は、セレクタが
最初の試行で見つからなければ、3 秒のデッドラインまで自前のポーリングで再度 query を実行します。
`tap` は `base.wait_until`（`base.py:359`）というデッドラインとポーリングのループの内側から呼ばれる
ことが多く、その結果、不安定な 1 回の tap に対して 2 つの独立したデッドライン管理がそれぞれ
自分のスケジュールで query を重ねる形になります。adb ドライバでは、`_scroll_into_view`
（`bajutsu/drivers/adb.py:369`）がスクロールの試行ごとに `_settle()` をもう 1 回追加するため
（`adb.py:320`、`326`）、画面外の要素までスクロールが必要な tap は、概念としては 1 つの操作で
あるにもかかわらず、十数回のツリー全体取得のサブプロセスを発生させることがあります。ここで
述べているのはシナリオが何をアサートするかを変える話ではなく、`assert` を多用するシナリオと
スクロールを伴うすべての tap に生じる、避けられる遅延の話です。

## 詳細設計

これは**挙動を保存する**リファクタリングです。ステップのアサーションと extract が見る要素は
これまでと**同じ**要素であり、その取得回数だけが減ります。セレクタの解決、アサーションの意味、
wait/poll の条件、ドライバのインターフェースには変更を加えません。作業は互いに独立した次の
3 単位に MECE に分かれます。

- **画面を変化させないステップについて、`_run_step_body` の query 結果をループの `after`
  スナップショットとして使い回します。** `_run_step_body`（`bajutsu/orchestrator/loop.py` の
  `_run_step_body` は 141 行目、`assert_` の query は 172 行目）を変更し、`assert_`（および query を
  実行するなら `wait`）について、既存の `(ok, reason, assertion_results)` に加えて、query した要素
  一覧を返すようにします。`exec_steps` では、ステップの種類が画面を変化させえないものであるとき、
  （BE-0234 で遅延化された）ステップ後の読み取りがもう一度 `query()` を呼ぶ代わりに、この返された
  要素一覧を `after` として使います。
  `tap`、`type`、`swipe` などのアクションステップは、実行後に画面が実際に変わっている
  可能性があるため、引き続き実行後に新しく query します。本単位が取り除くのは、「前」と「後」が
  同一の query であると証明できるステップに限った、余分な**2 回目**の query だけです。
  `extract` 修飾子（`bajutsu/scenario/models/steps.py:134`）はこの変更の恩恵をそのまま受けます。
  画面を変化させないステップに付いている場合は、使い回されたスナップショットを読みます。
  画面を変化させるステップに付いている場合は、これまでどおり実行後の新しい query を読み続けます。

- **tap の実行過程で、各層ごとに再 query する代わりに、確定した 1 つのスナップショットを
  受け渡します。** idb ドライバと adb ドライバでは、`_center`（`idb.py:317`）が `_settle()` を
  呼んだあと `_resolve`（`idb.py:296`）を呼びますが、`_resolve` はキャッシュミス時に自前で
  再度 query します。`_resolve` を変更し、`_settle()` がすでに生成したツリーを（すでにある
  `initial_tree` 引数を通じて。`idb.py:296`）自分の再試行の全体を通じて優先的に使うようにし、
  `_center` は最初に見つからなかった時点で `_resolve` にもう一度 query させるのではなく、確定した
  ツリーを渡すようにします。外側の `base.wait_until` のデッドラインループ（`base.py:359`）は
  変更しません。本単位が取り除くのは、隣接する層がすでに手元に持っているのと同じデータのために、
  下位の層が直後にもう一度実行していた query だけです。

- **query の回数が減ったことを確認する回帰テストを追加します。** `FakeDriver`（あるいは
  `query()` の呼び出し回数を数える計測用のラッパー）を使い、`assert` ステップ 1 つだけから
  なるシナリオでは、`query()` の呼び出しが現在の 2 回から 1 回に減ることをテストで確認します。
  もう 1 つのテストでは `tap` ステップを対象にし、アクションステップが影響を受けないこと
  （画面が変わっている可能性があるため、実行後の query は引き続き発生すること）を確認します。

## 検討した代替案

- **`query()` の結果を、ポーリングの反復をまたいで壁時計時間ベースでキャッシュする
  （たとえば短い TTL を設ける）。** 退けます。時間ベースのキャッシュは、画面が実際に変化した
  あとでも古いスナップショットを返しうるため、`wait` や `assert` ステップがすでに実機を
  反映していないデータを読んでしまう可能性があります。これは、プライムディレクティブ 2 が
  守ろうとしている決定性そのものを、気づかれないまま弱めることになります。本提案の使い回しは
  もっと狭い範囲にとどまり、構造的に安全です。あるステップがすでに実行した query と**証明可能に
  同一**の query の結果を差し替えるだけであり、そのステップ 1 つの生存期間に限られ、ポーリング
  やステップをまたぐことはありません。
- **ループを現状のまま残し、`waits.py:64` に明記されたコストを許容できる代償として受け入れる。**
  退けます。このコストはよく理解されていますが、意味を変えずに避けられるものであり、`assert` を
  多用するシナリオや、スクロールを伴うすべての tap で目に見えて積み重なります。これはまさに、
  本リポジトリが他の場面でも低く保とうとしているコストです（`docs/ai-development.md` の
  right-sizing の考え方は計算資源について同じ原則を述べており、本項目はそれを壁時計時間に
  適用するものです）。
- **ドライバの query インターフェースを、無効化を追跡する永続的なツリーキャッシュへ再設計する。**
  スコープ外として退けます。これは `Driver` インターフェース全体（idb、adb、Playwright の
  すべてのバックエンド）に対するはるかに大きな変更であり、無効化の正しさそのものに固有の
  リスクを抱えます。それに対して、本提案の狭い範囲の純粋な追加による使い回しは、よくあるケースでは
  すでに十分な効果を得ており、そのほうが見合っています。将来、汎用的なキャッシュを求める提案が出るのであれば、
  本項目が追加する query 回数の回帰テストの上に構築するべきです。
