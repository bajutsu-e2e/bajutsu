## はじめに

現状、文字入力を扱うステップは `type`（`bajutsu/orchestrator/actions/handlers/gestures.py`）だけです。
どのバックエンドでも `driver.type_text()` を通じて文字を追記するだけで、フィールドをクリアする、
一部だけ削除する、テキストを選択する、選択内容をクリップボードへコピーするといった操作は表現できません。
この項目では `select`・`clear`・`delete`・`copy` という 4 つの新しいステップを追加し、既存の決定性モデルの
枠内でこの隙間を埋めます。

## 動機

実際のフォーム操作は一度入力して終わりではありません。フィールドをクリアして入力し直す、打ち間違えを
バックスペースで直す、値を選択して別の場所へコピーする、既存の値の一部だけを置き換える、といった操作が
日常的に発生します。これらは現状のシナリオでは表現できず、唯一の代替手段は `tap` や `longPress` を
成否の定義もないまま無理やり組み合わせることです。これはドライバ抽象がそもそも排除しようとしている
非決定性そのものです。

ここで陥りやすい設計上の罠は、「選択」を「いま何文字が選択されているか」という状態として扱い、ステップで
その状態を設定し、後続のアサーションで検証できるようにすることです。しかしどのバックエンドも選択範囲を
問い合わせ可能な状態としては公開していません。`Element`（`DESIGN.md` §5）は `value` を持ちますが選択範囲は
持たず、idb・adb・XCUITest のいずれも選択状態を取得する手段がありません。「選択」をこの形でモデル化すると、
`query()` では一切検証できない、バックエンド固有の新しい状態が必要になり、決定性優先の方針（プライム
ディレクティブ 2）や、既存のセレクタ解決が守っている「曖昧なら推測せず失敗する」という契約と衝突します。

そこで `select`（および `copy`）は**状態ではなく操作**として位置づけます。フィールドの内容とステップの
パラメータが同じであれば操作の結果は常に同じになるため決定的であり、かつ**操作それ自体の結果を
アサーション対象にはしません**。検証は常に「その後に何が起きたか」に委ねます。選択後に `type` で
置き換えれば既存の `value` アサーションで検証でき、選択後に `copy` すれば BE-0052 がすでに実装した
クリップボードの読み戻し（`expect: - clipboard: { equals | matches }`）で検証できます。`clear` と `delete`
には新しいアサーションすら不要です。どちらも `value` に対するバックスペース相当の変更であり、`value` は
すでに `Element` の一フィールドとして、また既存の `Assertion` の一種として（`bajutsu/scenario/models/assertions.py`）
アサーション可能だからです。

## 詳細設計

既存の `type` と同じ形（`into` で対象フィールドを指定してから、ステップ自身の効果が実行される）に沿って、
4 つの新しいステップを追加します。

```yaml
- select: { into: form.note, mode: all }   # フィールドの現在の内容を全選択する
- copy: {}                                  # 選択中の内容をクリップボードへコピーする
- clear: { into: form.note }                # フィールドの内容を全消去する
- delete: { into: form.note, count: 3 }     # 末尾から 3 文字削除する
```

- **`select`** — この項目では `mode: all` のみをスコープとします（iOS Simulator ならハードウェアキーの
  ⌘A に相当するキーイベント、Android なら長押し ▸「すべて選択」、web なら `Ctrl+A` やトリプルクリックといった、
  プラットフォームの「全選択」操作です）。単語単位・範囲単位の選択（`mode: word` やオフセット指定の範囲）は
  意図的にスコープ外とします。どのバックエンドもドラッグハンドルやカーソル位置を操作するプリミティブを
  公開しておらず、しかも「単語」の境界の扱いはロケールや OS に依存するため、決定性を保ちにくい大きな
  作業になります。具体的な要望が出た時点で改めてスコープ化すれば十分です。
- **`copy`** — 現在選択されている内容に対してプラットフォームの「コピー」操作を実行します（`select` が
  作った選択状態に作用するため、自身は対象セレクタを取りません）。結果の検証はこのステップが新しく導入する
  状態にではなく、既存のクリップボード読み戻しにだけ委ねます。**何も選択されていない場合**（先行する
  `select` がない、あるいは選択を解除するステップが挟まった場合）、`copy` は黙って何もコピーしなかったり
  クリップボードの以前の内容を残したりせず、そのステップを決定的に失敗させます。選択のない「コピー」は
  シナリオ記述上の誤りであり、プライムディレクティブ 2 に従えば、後続のクリップボードアサーションが成功と
  誤読しかねない無言の no-op ではなく、失敗として表面化させる必要があるからです。この判定は各バックエンドの
  ネイティブなコピーが空の選択に対して何をするかに委ねるのではなく、Bajutsu 側で行います（前回の選択解除以降に
  選択が確立されたか）。これによりバックエンドをまたいで同じ失敗になります。
- **`clear`** — 指定したフィールドの内容を全消去します。`query()` がすでに返している `Element.value` から
  現在の文字列長がわかるため、決め打ちの回数ではなく、その文字列長ぶんだけ文字を除去します（バックスペース
  相当）。フィールドが何を保持していても対応できます。
- **`delete`** — フィールドの現在の値の末尾から `count` 文字を削除します。`clear` と同じバックスペース相当の
  仕組みを、より小さい回数に限定したものです。スクリプト化した打ち間違いの修正や、フィールド全体を
  クリアしない部分編集に使えます。

**バックエンドごとの actuation** — 具体的な実現方法は、実装時に確定させるバックエンド固有の詳細とします
（BE-0052 でも 4 つのプリミティブのうち 2 つが、実装時のトリアージで初めて実現可能な手段が見つかるという
経緯をたどっており、それに倣うものです）。検討すべき方向性は次のとおりです。

- **idb** — `driver.type_text()`（`bajutsu/drivers/idb.py`）はすでに `idb_companion` のテキスト gRPC 呼び出し
  経由でフォーカス中のフィールドに届いていますが、制御文字（`\b` など）がハードウェアのバックスペースキーと
  同じ意味で解釈されるかどうかは実装時の検証が必要です。解釈されない場合は、テキスト注入とは別の HID
  キーイベント呼び出しを使う代替が必要になります。
- **adb** — 既存の `KEYCODE_BACK` 呼び出し（`bajutsu/adb.py`）と同じ経路で `input keyevent KEYCODE_DEL`（67）を
  `count` 回実行します。`select` の「全選択」に対応する単一のキーイベントはなく、長押しのコンテキストメニュー
  経由になる見込みです。
- **XCUITest** — `type` が使っている HTTP トランスポート（`/type`）と同じ形で、`/select` と `/copy`（または
  汎用の `/keyEvent`）に相当するエンドポイントをランナー側（`BajutsuKit`）に追加できるか検討します。
  `XCUIElement` はテキストの置き換えとキーボードのキーイベントの両方をネイティブに公開しています。
- **Playwright** — 最も素直に実現できるバックエンドです。`clear` は `page.fill(selector, "")`、`delete` は
  `keyboard.press("Backspace")` の繰り返し、`select` は `selectText()` やトリプルクリック、`copy` は
  クリップボード権限つきの `navigator.clipboard`（または `Ctrl+C` のキーイベント）で実現できます。

プライムディレクティブとの整合は次のとおりです。

- **決定性。** どのステップもバックスペース相当の回数、あるいはプラットフォームの全選択・コピー操作という
  固定の変更であり、結果は既存の `value` / `clipboard` アサーションで機械的に検証できます。新たに検証不能な
  状態を持ち込まず、`sleep` も LLM も使いません。
- **アプリ非依存。** アプリ固有のコードは不要です。`into` は他のセレクタベースのステップと同じ方法で対象を
  指定します。
- **codegen。** これらは XCUITest・Espresso・Playwright のネイティブ操作に直接対応するものがあるため、
  codegen は `// TODO` のスタブではなく実際の呼び出しを出力できます（`simctl` にしか対応先がない BE-0052 の
  プリミティブとは異なる点です）。

## 検討した代替案

- **`select` を問い合わせ可能な状態（`Element` 上の選択範囲）としてモデル化する。** 却下しました。どの
  バックエンドも要素・クエリの表面から選択状態を公開しておらず、それに対するアサーションは検証不能に
  なります。これは実装上の抜けではなく、決定性優先そのものへの違反です。
- **`select` と `copy` を 1 つの `copySelection` ステップにまとめる。** 検討しました。この項目が出荷する
  フローは「全選択してからコピーする」だけだからです。それでも 2 つのステップに分けたのは、`select` は
  他に選択内容を使うアクション（選択中に `type` すれば置き換わる、など）が加わったときにそれ単体で意味を
  持つためであり、`copy` を分けておくことは `type` の「`into` + 効果」という既存の一ステップ一操作の慣習に
  沿うものだからです。この組み合わせだけを特別扱いする理由にはなりません。
- **新しい `clear` ステップではなく、`type` に `mode: replace` や `clear: true` のような修飾子を足す。**
  却下しました。`type` は `text` を必須とするため、「入力し直さずにクリアだけしたい」という意図を空文字列に
  仮託することになります。「フォーカスする・任意でクリアする・任意で入力する」を 1 つのステップの修飾子に
  詰め込むより、単一の役割を持つ小さなステップを順に並べるほうがシナリオとして読みやすくなります。
