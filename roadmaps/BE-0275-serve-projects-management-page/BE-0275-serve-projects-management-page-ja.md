## はじめに

設定プロジェクトハブ（[BE-0225](../BE-0225-config-project-hub/BE-0225-config-project-hub-ja.md)）は、
`serve` を複数の名前付き設定バインディングを束ねるハブにし、それをヘッダーのスイッチャーと、
プロジェクトを一覧して切り替え・実行する **Projects** モーダルとして提示しました。モーダルが担うのは
その読み取りと切り替えと実行の範囲だけで、ハブを増やしたり減らしたりする 2 つの操作、すなわちプロジェクトの
追加と削除は `bajutsu project` CLI に委ねられています。モーダル自身のヒントもそう述べています
（「Add or remove projects with `bajutsu project add` / `rm`.」）。

本項目は、このギャップをモーダルへの操作追加ではなく、**プロジェクト管理を serve シェルのトップレベルの
ページへ昇格させること**で埋めます。Record / Replay / Crawl / Stats / Metrics と並ぶ **Projects** ビューを設け、
そこでプロジェクトを一覧し、追加・削除・切り替え・実行を行います。モーダルはこのページに引き取られて廃止します。
理由は情報構造にあります。**プロジェクトは設定の上位にある概念**であって、設定に属する機能ではありません。
だからこそ、一時的なオーバーレイの中ではなく、シェルのトップレベルに置かれるべきです。

## 動機

モーダルは、シェルの情報構造のなかでは機能に紐づくオーバーレイです。現在の文脈に対する何らかの操作のために
現れ、その文脈へ戻って閉じます（設定の閲覧、Settings、実行結果の Replay）。プロジェクトハブはこの形に
当てはまりません。プロジェクトはアクティブな設定に対して行う操作ではなく、**アクティブな設定が内側に存在する
入れ物**です。トップレベルの各ビュー（Record、Replay、Crawl、Stats）はすでに選択中のプロジェクトの内側で
動いており、それらを所有して束ねるものが、階層を反転させずにそれらから起動されるモーダルであることはできません。
BE-0225 のモーダルは手早い一覧と切り替えには使えますが、ハブの置き場所としては構造的に誤っています。
上位の概念を下位のオーバーレイとして描いているからです。

この置き違えの帰結は具体的です。

- **ライフサイクル操作の居場所がない。** プロジェクトの追加と削除はハブ規模のライフサイクル操作であり、
  閉じられるオーバーレイはその居場所として不自然です。BE-0225 がこれらを CLI に残した一因でもあります。
  ブラウザからハブを管理する利用者には、ハブに最低限必要な「増やす・減らす」を行う場所が UI にありません。
  とりわけホスティング構成（[BE-0015](../BE-0015-web-ui-public-hosting/BE-0015-web-ui-public-hosting-ja.md)）
  ではブラウザが利用者に与えられた唯一の画面であり、`bajutsu project add` を実行するシェルがありません。
- **ハブに恒久的な自分の画面がない。** 兄弟であるプロジェクト横断のメトリクスビュー
  （[BE-0226](../BE-0226-cross-project-metrics-dashboard/BE-0226-cross-project-metrics-dashboard-ja.md)）は、
  すでにハブのプロジェクトを扱うトップレベルのビューです。そのビューがランク付けするプロジェクト自体には
  トップレベルの画面がなく、オーバーレイしかないのは不自然です。ハブについて考える起点となる一覧は、
  クリックで消える一時的なものではなく、リンクでき、そこへ移動できて残り続ける場所であるべきです。

このページに必要なものは、すべてすでに存在します。エンドポイント（BE-0225 ユニット 3。`GET` / `POST` /
`DELETE /api/projects`、`POST /api/projects/<name>/activate`、`POST /api/projects/<name>/run`）、CLI の
`bajutsu project add --config` が受け取るのと同じ設定ソース（任意の非公開リポジトリ用認証情報を伴う Git spec、
[BE-0063](../BE-0063-git-config-source/BE-0063-git-config-source-ja.md)。またはローカルパス）、許可リスト検査
（[BE-0108](../BE-0108-hosted-config-source-restriction/BE-0108-hosted-config-source-restriction-ja.md)）、
そして RBAC のゲートです。本項目はこれらをページへ移し、ハブに欠けていた 2 つのライフサイクル操作を加えます。

これはプライムディレクティブの範囲内に収まります。プロジェクトページは完全に決定的で、CLI が書き込むのと
同じ `ProjectRegistry` の継ぎ目を通じて名前付きの設定ソースを一覧・バインド・アンバインドするだけであり、
どの経路にも LLM は入りません。ハブそのものはアプリ非依存のままです。アプリごとの違いは各プロジェクトの設定に
置かれ、このページは設定ソースに名前を付けてバインドするだけだからです。

## 詳細設計

作業は 4 つのユニットに MECE に分かれます。ページそのもの、そのページが備える 2 つのライフサイクル操作、
そしてそれらを統べるロールと許可リストの結線です。

### 1. Projects トップレベルビュー

serve シェルの新しいトップレベルビュー（ヘッダーナビの `data-view="projects"` タブ。他のビューと同様に
独自のメイン領域へ描画する）がハブの居場所になります。ここでは登録済みの各プロジェクトを、ハブについて考えるのに
必要な情報とともに一覧します。名前、設定ソース（Git・アップロード・ファイルのバインディングを人間が読める形に
要約したもの）、アクティブかどうか、そして直近の実行結果です。BE-0225 のヘッダー**スイッチャー**は
残します（文脈セレクターのような、正当な手早い切り替えの操作だからです）。一方で **Projects モーダルは廃止**し、
その一覧、行ごとの Run、そして担っていた切り替えは、このページに移します。プロジェクトの選択・アクティブ化は
既存の `POST /api/projects/<name>/activate` を通じてライブ設定を再バインドし、モーダルと同じ挙動になります。
これにより他のすべてのビューが、再起動なしに切り替え先のプロジェクトに対して動きます。

Metrics ビュー（BE-0226）とヘッダーのスイッチャーは、いずれも本物のハブ（2 つ以上のプロジェクト）があるときだけ
現れます。Projects タブはこれと異なり、プロジェクトの数によらず、単一設定の `serve` でも現れます。これはユニット 4 が
記録する意図的な表示条件の判断です。追加フォームは、スイッチャーを最初に現す 2 つ目のプロジェクトを作る唯一の
UI 上の手段であり、ホスティング構成にはそれを作る CLI がないため、ハブが存在する前からページへ到達できる必要が
あるからです。

### 2. プロジェクトの追加（ページ上で）

ページは**プロジェクト追加**フォームを備えます。このフォームは、本トピックが取り込む CLI の相方を写したものです。
`bajutsu project add --config <ローカルパス | Git spec>` は単一の設定ソース文字列を受け取るので、フォームも
そうします。名前欄と、1 つの**設定ソース**入力欄（Git spec `github:owner/repo[@ref][:path]`、または
サーバーが `fs` ソースを許可する場合はローカルパス、ユニット 4）、そして任意の非公開リポジトリ用認証情報
（[BE-0063](../BE-0063-git-config-source/BE-0063-git-config-source-ja.md)）です。認証情報はランチャーの既存の
書き込み一度きりの経路で保存します。送信すると `{name, sourceSpec}` を `POST /api/projects` へ送り、サーバーは
その文字列を、CLI と同じ唯一の正規パーサー `source_from_config` を通じて、保存用の `{kind, locator}` レコードへ
正規化します。ブラウザは Git spec の解析を作り直さず、ソースが文字列で届いてもレコードで届いても同じ
`_validate_source` の許可リスト検査を通ります。エンドポイントとそのルート、レジストリのスキーマは変わりません。
`sourceSpec` は既存の登録エンドポイントへ追加した任意の要求フィールドであり、`/api/config` がすでに `git` の
spec 文字列を受け取るのと同じ形です。`POST /api/projects` は `(org, name)` について冪等なので
（BE-0225 ユニット 1・3）、既存の名前で再送するとエラーにならず、そのソースを**再バインド**します。フォームは
これを明示的な「再バインド」として示し、サーバー側の拒否（許可リストによる拒否、不正なレコード）はインラインで
示します。

以前の枠組みが挙げた `.zip` アップロードは、追加のソースには**しません**。`bajutsu project add` にアップロードの
選択肢はなく（受け取るのは `--config` 文字列です）、`POST /api/projects/<name>/activate` はコンテンツアドレスの
オブジェクトストアなしにアップロード済みバンドルを往復できません（それがなければ 409 を返します。BE-0243 /
BE-0268）。そのため、アップロードに裏打ちされたプロジェクトは登録はできても、よくあるローカル構成では切り替え
不能になります。バンドルのアップロードは引き続き、*設定ランチャー*がアクティブな設定をバインドする手段です。
アップロードを第一級のプロジェクトとして登録することは、実際の必要が現れたときの後続項目に委ねます。

### 3. プロジェクトの削除（ページ上で）

各行に**削除**操作を加えます。実行すると確認を求め（登録解除は履歴には破壊的でないものの、バインディングには
破壊的です）、`/api/projects/<name>` を `DELETE` して一覧を更新します。ページは BE-0225 の契約を明示します。
すなわち**実行履歴は保持され**、バインディングだけが取り除かれます。アクティブなプロジェクトを削除した場合は、
その状況に対してエンドポイントがすでに行う挙動に従います。ページは削除後の `GET /api/projects` の状態を反映し、
結果を推測しません。

### 4. 表示条件、RBAC、許可リスト

ページとその操作は、ハブ UI の他の部分がすでに従っている規則に従います。

- **表示条件（決定済み）。** スイッチャーと Metrics タブは「ハブが存在する」（2 つ以上のプロジェクト）を追い、
  単一設定の `serve` では隠れたままです。Projects タブはそうしません。管理すべきプロジェクトが 1 つでもあれば
  現れ、レジストリが結線されていないときだけ隠れます。これは提案が挙げた意図的な判断です。追加フォームは、
  スイッチャーを最初に現す 2 つ目のプロジェクトを作る唯一の UI 上の手段であり、ホスティング構成にはその CLI が
  ないため、ハブが存在する前からページへ到達できる必要があるからです。
- **RBAC（サーバーが強制。クライアントは操作を見せ、403 をインライン表示）。** `POST` / `DELETE /api/projects` は
  サーバー側で **admin** ゲートであり（`authz.required_role`、BE-0225 ユニット 3、`/api/config` と同じ扱い）、
  実際の強制はこのゲートが担います。ページはハブ UI の他の部分がすでに従う流儀に合わせます。追加・削除の操作を
  ロールで隠しはしません。現在の serve UI にはクライアント側のロール情報がなく、admin 専用の画面（設定バインド、
  アップロード、compose、Settings）はいずれも全員に見せてサーバーが 403 を返すからです。拒否された書き込みは、
  それらと同じくインラインで示します。非 admin のセッションに真に読み取り専用のプロジェクトページを見せるには、
  UI がまだ公開していないクライアント側のロール情報が要ります。それはここで場当たりに作らず、**別の BE 項目に
  委ねます**（他の admin 専用画面も一様に制御するロール情報です）。
- **ホスト時の許可リスト。** 追加フォームは、サーバーが `fs` ソースを許可しない場合、ヒントから「ローカルパス」の
  案内を落とします
  （[BE-0108](../BE-0108-hosted-config-source-restriction/BE-0108-hosted-config-source-restriction-ja.md)）。
  手で組み立てたローカルパスもサーバー側（`_validate_source`）で拒否するので、検査は見た目だけのものではありません。

新しいエンドポイントも、レジストリのスキーマ変更もありません。これは UI の移設（モーダルからページへ）と、
2 つのライフサイクル操作、そのわずかなクライアント状態、そして既存の登録エンドポイントに追加した 1 つの
`sourceSpec` 要求フィールド（CLI 自身の `source_from_config` を通じて正規化します）だけです。

## 検討した代替案

- **既存の Projects モーダルに追加・削除を足す（モーダルのまま）。** 情報構造の観点から却下します。モーダルは
  機能に紐づくオーバーレイですが、プロジェクトは設定の上位にある概念であって、設定に対する操作ではありません。
  各ビューはすでに選択中のプロジェクトの内側で動くので、それらを所有して束ねる画面は、内側から起動される
  オーバーレイではなく、トップレベルのページであるべきです。これは本提案の初期の枠組みであり、ページがこれを
  置き換えます。
- **追加・削除を CLI 専用のままにする（現状維持）。** 却下します。これは本項目が解消しようとしているギャップ
  そのものであり、シェルのないホスト版ハブを自身の UI から育てられないままにします。
- **追加フォームにランチャー相当のフルなソース選択欄を置く（Git と認証情報、`.zip` アップロード、fs ブラウズ）。**
  過剰として却下します。本トピックが取り込む CLI の相方 `bajutsu project add` は単一の `--config` 文字列
  （Git spec かローカルパス）を取り、アップロードの選択肢を持ちません。またアップロードに裏打ちされたプロジェクトは
  オブジェクトストアなしに切り替えられません（409）。フォームは CLI を写して、1 つの設定ソース文字列をサーバー側で
  同じ `source_from_config` により正規化します。これにより 1 つのパーサーと 1 つの許可リスト経路
  （[BE-0108](../BE-0108-hosted-config-source-restriction/BE-0108-hosted-config-source-restriction-ja.md)）を保て、
  歩調を合わせ続けるべき 2 つ目のウィジェットを抱えずに済みます。アップロードは引き続き、設定ランチャーが
  アクティブな設定をバインドする手段です。
- **追加・削除の操作を、いまクライアント側でロールにより制御する。** 本項目では却下します。serve UI には
  クライアント側のロール情報がなく、既存の admin 専用画面はいずれも全員に見せてサーバーが 403 を返します。
  このページのためだけにロール情報を作ると、共有すべき概念を重複させるか、他の admin 画面との一貫性を欠くことに
  なります。サーバーの RBAC ゲートがすでに強制しており、クライアント側のロール情報（と、それが可能にする全 admin
  画面の読み取り専用表示）は、それ自身の BE 項目に委ねます。
- **プロジェクトページを Metrics ビュー（BE-0226）へ畳み込む。** 却下します。両者は関心が異なります。Metrics は
  プロジェクトをランク付けする（読み取り専用で助言的）ものであり、プロジェクトページはそのライフサイクルを
  管理する（登録・切り替え・実行・削除）ものです。同じプロジェクト一覧を扱う `Related` の兄弟であって、
  1 つのビューではありません。
