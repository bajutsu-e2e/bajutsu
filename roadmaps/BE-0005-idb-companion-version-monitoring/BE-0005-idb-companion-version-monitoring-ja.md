## はじめに

idb 自体のメンテナンス頻度と、最新ランタイムとの互換に追従します。CI でバージョンを固定し、監視します。

## 動機

現状、実機で動く backend は idb のみ（`bajutsu/backends.py`）なので、実機経路の全体が単一の外部依存（`idb` Python クライアントと `idb_companion` バイナリ）の挙動の上に乗っています。どちらも Bajutsu とは独立に、それぞれのメンテナンス頻度で保守されています。ここから 2 種類の破綻が生じます。第一に、新しい iOS や Xcode のリリースが、古い `idb_companion` では駆動できないランタイムを載せてくることがあり、Bajutsu 側は何も変えていないのに、昨日まで green だった CI が今日は「no available actuator」になります。第二に、`idb_companion` のアップグレードが `ui describe-all` の JSON スキーマを静かに変えてしまうことがあります。`parse_describe_all` が依存しているキー名（`AXLabel` / `AXValue` / `AXUniqueId` など）が変われば、要素正規化は気づかぬうちに劣化します。

DESIGN §11 はすでに「idb 自体のメンテナンス頻度と最新ランタイムとの互換」を未解決のリスクとして挙げており、`docs/drivers.md` も、describe-all のキー名は特定の fb-idb バージョンに対して実機検証されており、idb が変わったら再確認すべきだと注記しています。今日その再確認は、手作業で忘れやすい手順です。本提案の狙いは、バージョンを「たまたまそのマシンに入っているもの」ではなく追跡可能で観測可能な量にすることにあります。これにより、互換性の破綻を、後になって紛らわしい run 失敗として発見するのではなく、それが起きた当日に CI で捕捉できます。

## 詳細設計

idb を実行する相手のバージョンを、暗黙の前提ではなく、run の明示的で記録される入力にします。

- **バージョンを manifest に記録する。** 各 run は `idb_companion` のバージョン（および `idb` クライアントのバージョン）を取得し、既存の run メタデータと並べて `manifest.json` に書き込みます。これにより、任意の成果物セットがどの idb で生成されたかを正確に示します。これは来歴であってゲートではなく、合否には一切影響しません。Tier-2 の run/CI ゲートは決定的かつ LLM フリーのままです。
- **環境ごとに期待バージョンを宣言する。** 検証対象として固定したい既知の good なバージョン範囲を config に置きます。これはアプリ単位ではなく環境単位の事柄なので、`apps.<name>` の下ではなく他の backend 設定の隣に置きます。どのアプリを対象にしても pin は同じであり、app-agnostic の原則を保ちます。`doctor` がこれを読み、インストール済みバージョンを期待値と照合して報告するので、既存の事前チェック面に「idb_companion 1.2.3 がインストール済み、期待は >= 1.2.x」のような明確な行が加わり、下流での失敗にはなりません。
- **CI 専用の監視ジョブ。** （高速かつ Simulator 不要を保つべき PR 単位のゲートとは別の）スケジュール実行ジョブが、利用可能な最新の `idb_companion` と現行の Simulator ランタイムに対して実機スモーク経路を走らせ、スキーマや挙動がずれたら大きく失敗します。これは `docs/drivers.md` が述べる手作業の再確認を自動化した形で、必要なバージョン更新を、その場限りの発見ではなく、こちらが制御する頻度で CI シグナルとして表面化させます。
- **バージョン文字列だけでなくスキーマ検証も。** スモーク経路は、`describe-all` が `Element` 正規化に必要なフィールドを依然として返すことを検証するので、バージョン互換の体裁は保ったまま JSON の形だけを変えるアップグレードも捕捉できます。これにより、保証の対象を単なる数値ではなく挙動に保ちます。

ここには固定 sleep も非決定的な選択も持ち込みません。監視ジョブは既存の条件待機済みスモークシナリオを再利用し、バージョン pin は静的な比較です。

## 検討した代替案

- **idb を 1 つの厳密なバージョンに固定して動かさない。** 再現性はありますが、Bajutsu を iOS/Xcode のリリース列車の後ろに凍結します。新しい Simulator ランタイムを、それを駆動できる新しい `idb_companion` でしか扱えないなら、到達不能になります。私たちはこの頻度から降りるのではなく追跡したいので、ハードな凍結より監視付きの範囲を選びます。
- **PR 単位の e2e ワークフローだけに頼る。** 既存の実機 e2e run はすでに idb を動かしますが、それは runner にたまたま入っている `idb_companion` で、しかも PR がその経路に触れたときだけ走ります。上流の idb やランタイムのリリースだけに起因するずれは、誰かがそうした PR を開くまで気づかれません。スケジュール監視ジョブは、このチェックを PR 活動から切り離します。
- **スキーマ破綻を実行時に検出して適応する。** `parse_describe_all` で複数の describe-all スキーマを許容するのは、問題を表面化させるどころか隠すことになり、決定性が要のこの正規化経路に分岐を増やします。変更を CI で捕捉してパーサを意図的に更新する方が、このコードを単一目的かつレビュー済みに保てます。
