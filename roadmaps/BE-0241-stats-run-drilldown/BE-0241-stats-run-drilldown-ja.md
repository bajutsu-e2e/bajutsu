## はじめに

Stats ダッシュボード（BE-0102）は、日付・バックエンド別の合格率や、失敗の多いシナリオ・ステップ・
アサーションの集計をひと目で見せてくれます。ただし表示されている数値はすべて集計値であり、個々の
run への手がかりを持ちません。「2026-07-10、idb、4 件失敗」というセルや「`LoginScenario > tap` が
12 回失敗」というホットスポットの行を見ても、それがどの run で起きたことなのかを確かめる手段が
ありません。この手がかりを得るには、今日は Stats ページを離れて `serve` の SPA が持つ run 履歴
一覧（`#panel-history`）を開き、フィルタも並び替えもできないフラットな一覧を目で追って、それらしい
run を探すしかありません。本項目は、Stats ダッシュボードの日付・バックエンド・シナリオ・ステップ／
アサーションの各軸をクリック可能にし、該当する run へ、そこから既存の run 単位のレポート
（BE-0068）へと直接たどり着けるようにするものです。

## 動機

- BE-0102 は読み取り専用の設計を保ったまま拡張できます。集計時にすでに計算済みのデータへの
  ナビゲーションを追加するだけであり、新しい判定や新しい永続データを持ち込むものではありません。
- レポートビューアはすでに存在します。BE-0068 の `/runs/<id>/report.html` は保存済みマニフェスト
  から都度描画されます。run 履歴一覧もすでに存在します。`bajutsu/templates/serve.html.j2` の
  `#panel-history` を `bajutsu/templates/serve.panels.js` の `loadHistory()` が描画しており、各行は
  クリックするとすでにレポートを開きます。「具体的なシナリオを確認したい」というニーズに対して
  新規に作るものはほとんどなく、欠けているのは、Stats の集計処理がシナリオ・ステップ・アサーション
  のホットスポットを数え終えた時点で run id を捨ててしまっていることと、履歴一覧にフィルタ機能が
  なく run id が分かってもそこへ絞り込む先がないことの二点だけです。
- 現状のフローは手作業に頼っていて間違いも起きやすいものです。「`LoginScenario` の `tap` が 12 回
  失敗」と読んでも、フィルタも並び替えもされていない履歴一覧の中から該当する 12 件を、場合によって
  は数百件の run の中から目視で探すことになります。日付・バックエンド単位のセルであれば `by_run`
  がすでに `day` と `backend` を各点に持っているため単なる絞り込みで済みますが、シナリオ・ステップ
  ・アサーションのホットスポットについては、生のマニフェストを手で読み返さない限り現状では特定
  できません。

## 詳細設計

この設計は、既存のデータを既存の UI へつなぐだけのものです。新しい永続データも、新しいエンド
ポイントも、新しいレポートビューアも必要としません。

1. **ホットスポット集計に run id を残します**（`bajutsu/stats.py`）。`_failing_scenarios` /
   `_failing_steps` / `_failing_assertions` は、すでに各 run のマニフェストを走査してシナリオ／
   `scenario > action`／アサーション種別ごとに `Counter` を積み上げていますが、数え終えた時点で
   マニフェストの `runId` を捨てています。`Hotspot` に `run_ids: tuple[str, ...]` フィールドを追加し
   （決定性のためソート済みにします）、`_hotspots()` がすでに理由の集計を落とし込んでいるのと同じ
   形で、各アグリゲータに該当する run id も合わせて集めさせます。`by_day` と `by_backend` は変更
   不要です。`Stats.by_run` は各点にすでに `run_id` と `day` と `backend` を持っているため、日付や
   バックエンドでのクリックはページがすでに持つデータへのクライアント側の絞り込みだけで答えられます。
2. **各セルをサーバー側でディープリンクとして描画します**（`bajutsu/templates/stats.html.j2`）。
   `stats.html.j2` は現状 `<script>` タグを一つも持たない純粋なサーバーレンダリングの Jinja
   ページであり、本項目もその状態を保ちます。対象となるセル（`by_day` の各行、`by_backend` の
   各行、`failing_scenarios` / `failing_steps` / `failing_assertions` の各行）を、クエリ文字列を
   テンプレートが直接組み立てた `<a href="/?tab=history&runs=id1,id2,…">` のリンクにします（日付・
   バックエンドの行はテンプレート内で `stats.by_run` を絞り込んで id を導出し、ホットスポットの
   行は新しい `Hotspot.run_ids` を使います）。Stats ページには新しい inline script を持ち込まず、
   追加の API 呼び出しもしません。クライアント側の振る舞いはすべて SPA 側（手順 3）に置き、
   BE-0102 が確立した「1 回の集計、1 枚のページ」という形を崩しません。
3. **既存の履歴一覧に絞り込んで着地させます。** `/stats` と `serve` の SPA は現状別ルートなので、
   クリックすると対象の run id を URL に載せた形で SPA へ遷移します（例えば
   `/?tab=history&runs=id1,id2,…`）。読み込み時に `serve.js` / `serve.panels.js` がこのクエリを
   読み取り、History タブへ切り替えたうえで、`loadHistory()` の描画に新しい絞り込みを適用します
   （履歴一覧は現状フィルタを持たないため、これが本項目で履歴一覧に加える唯一の新しい振る舞いです）。
   一覧の上部には「絞り込み：<ラベル>（N 件）・解除」という表示を添え、全件表示へ戻せるようにし
   ます。絞り込まれた行をクリックしたときの挙動は今日と変わらず、その run の
   `/runs/<id>/report.html`（BE-0068）を開きます。
4. **ステップ・アサーション単位の粒度は run までにとどめます。** スコープを詰めた議論のとおり、
   ステップやアサーションのホットスポットをクリックすると、それが失敗した run まで履歴一覧を
   絞り込み、既存の run 単位のレポートを開きます。`report.html` 内の該当ステップにアンカーを張った
   りハイライトしたりはしません。より細かい単位へのジャンプは自然な発展形ですが、本項目のスコープ
   には含めません。

## 検討した代替案

- **Stats ページ自体に専用の絞り込み一覧やモーダルを作る案。** 却下しました。`serve` の SPA に
  すでにある run 履歴一覧のレンダリングとレポートへのクリック遷移を重複して作ることになり、この
  リポジトリが重んじる「新しい面を作るより既存を再利用する」という流儀に反します（BE-0102 自身が
  `report.html` や `audit` への合流を退けたときの論拠と同じです）。既存の履歴一覧へ絞り込んで
  遷移させれば、レンダリング経路を一つに保ったまま同じ結果が得られます。
- **オンデマンドで計算する新規の `/api/runs?matching=…` エンドポイント。** 却下しました。集計処理
  は、今日 `/stats` が描画している件数と同じ 1 回のパスの中で、各ホットスポットの該当 run id も
  すでに計算しています。新しいエンドポイントを設けると、ページがすでに持っているデータを再計算
  するか別途キャッシュすることになり、単一の自己完結したレンダリングが、ページ本体と追加の
  フェッチの 2 段構えに変わってしまいます。id を既存の `/stats` の HTML にそのまま埋め込むほうが、
  BE-0102 が確立した「1 回の集計、1 枚のページ」という形を保てます。
- **ドリルダウン時に `report.html` 内の該当ステップにアンカーを張ってハイライトする案。** 却下
  ではなく保留です。あれば有用ですが、`report.html` には現状ステップ単位のアンカーや id が存在
  しないため、本項目が届ける run 単位のドリルダウンを妨げるものではなく、切り離して後続作業に
  できます。
