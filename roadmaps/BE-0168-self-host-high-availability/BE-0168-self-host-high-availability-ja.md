## はじめに

セルフホスト向けのサーバーバックエンド（[BE-0016](../BE-0016-web-ui-self-hosting/BE-0016-web-ui-self-hosting-ja.md)）は、
現在**単一**の Linux ノードで動いています。Redis を取り除いたことで
（[BE-0106](../BE-0106-post-completion-worker-model/BE-0106-post-completion-worker-model-ja.md)）、
**Postgres** が唯一のステートフルな単一障害点（SPOF）になりました。Postgres はジョブキュー（`jobs`
テーブル）、セッション、クォータの状態、実行メタデータを一度に抱えています。この提案は、その SPOF を
取り除く**高可用性の構成**、すなわちレプリカを伴う Postgres プライマリ、冗長化したロードバランサ、
バックアップ済みのオブジェクトストアを定めます。あわせて、単一ノードのデプロイがいつ受け入れられる、
はっきり明示されたトレードオフであるかも明らかにします。BE-0016 の「単一ノードをプールへ育てる」作業から
切り出したものです。

## 動機

単一ノードは小規模なセルフホストには十分ですが、Postgres・ロードバランサ・MinIO ノードのいずれかが
故障すればサービス全体が止まることを意味します。しかも Postgres はキュー・セッション・クォータ・実行履歴を
まとめて抱えているため、これを失うと四つすべてを一度に失います。リリースゲートとして Web UI に依存する
チームは、一つのディスクや一つのホストが原因の際限のない停止を受け入れられません。必要なのは、運用者が
各 SPOF を意図的に取り除けるよう、文書化され検証された堅牢な構成です。同時に、単一ノードは運用者が
それを SPOF として明示的に受け入れ、検証済みのバックアップを持っている**場合に限り**受け入れられる、という
明確な記述も必要です。

## 詳細設計

このティアはすべてデプロイと運用の作業であり、Python 側の面を持たないため、`make check` ゲートでは
まったく動かせません。価値は文書化され検証された構成にあります。SPOF を一つずつ扱う作業分解は次のとおりです。

1. **Postgres の高可用性。** 自動フェイルオーバーを伴う**プライマリ + レプリカ**（たとえば Patroni）、
   または少なくとも検証済みのポイントインタイムバックアップとリハーサル済みのリストア。Postgres は
   キュー・セッション・クォータ・実行メタデータを抱えているため、最初に取り除く価値が最も高い SPOF です。
2. **オブジェクトストアの冗長化。** 冗長化した MinIO（分散配置や消去符号による構成）、または少なくとも
   バックアップ済みのバケット。単一のディスク故障で成果物が失われないようにします。動画による証跡は
   容量が大きいため、バックアップ先はそれに見合う容量にします。
3. **冗長化したロードバランサ。** 浮動アドレスを持つ二台目のロードバランサ（keepalived や Virtual
   Router Redundancy Protocol（VRRP）、または DNS ベースのフェイルオーバー）を置き、入口そのものが
   SPOF にならないようにします。これは N レプリカによる制御プレーンのスケールアウト（兄弟項目の
   「制御プレーンのスケールアウト」）と組み合わさりますが、分離できます。単一のアプリレプリカでも
   入口を堅牢にします。
4. **単一ノードのトレードオフの明示。** 単一ノードは、運用者がそれを SPOF として明示的に示し、検証済みの
   バックアップとリストアを持っている**場合に限り**受け入れられるセルフホストであることを文書化します。
   多くのセルフホストが単一ノードであるという正直な既定を保ちつつ、それを耐障害性があるかのように装いません。

```
[DNS] → [HAProxy ×2 (VRRP)] → [FastAPI ×N] ─┬─ Postgres (primary+replica)  ← queue / sessions / quotas
                                            └─ MinIO (redundant / backed-up)
```

**検証。** デプロイ上で手動で検証します。ゲートによる担保はありません。フェイルオーバーの訓練が本当の
テストです。Postgres プライマリを落とし、キューとセッションを保ったままレプリカが引き継ぐことを確認し、
ロードバランサを落として浮動アドレスが移ることを確認し、オブジェクトストアをバックアップからリストアして
成果物が読めることを確認します。成果物には、これらの訓練の手順書を含めます。

## 検討した代替案

- **単一ノードを恒久的なものとして受け入れる。** 最終形としては却下します（ただし明示付きの選択肢としては
  残します）。趣味のデプロイには十分ですが、チームのリリースゲートは単一ディスクの際限のない停止に
  耐えられません。折衷は、単一ノードのトレードオフを高可用性であるかのように装うのではなく*文書化*する
  ことです。
- **セルフホストの高可用性の代わりにマネージドクラウドの Postgres・オブジェクトストアを使う。** ここでは
  対象外です。それは完全なマネージド公開クラウドであり、このセルフホスト項目ではなく
  [BE-0015](../BE-0015-web-ui-public-hosting/BE-0015-web-ui-public-hosting-ja.md) の担当です。
- **Postgres の高可用性の代わりにアプリケーション層でキューを複製する。** 却下します。Postgres が
  すでに十分に備えている永続的な複製を作り直すことになります。Postgres の高可用性に寄せれば、キュー・
  セッション・メタデータを複数ではなく一つの仕組みで一貫させられます。
