## はじめに

ロードマップ（BE）項目を `Status` で絞り込んで一覧する Claude Code スキルを追加します。これにより、
AI セッションは `roadmaps/README.md` を全文読まなくても、「`Proposal` の項目すべて」（あるいは
`In progress`、`Implemented`、`Proposal (deferred)`）を見渡せます。スキルは状態を引数に取り、`ID`、
タイトル、`Topic`、項目ファイルのパスを 1 つの表で出力します。Claude はその表を見て、実際に必要な項目
だけを開けます。

## 動機

`roadmaps/README.md` はすべての BE 項目の索引ですが、4 つの状態のバケットと多くのトピックにまたがって
100 を超える項目を抱え、700 行を超える規模に育ちました。たとえば未着手の提案だけを知りたいセッションに
とって、今の選択肢はどちらも不十分です。

- **索引を全文読む** ： 生成された表が数百行分そのまま文脈に載ります。その大半はいまの作業と無関係で、
  トークンを消費し、本当に必要な数行を埋もれさせます。
- **README を手作業で `grep` する** ： 機能はしますが、セッションごとにコマンドを組み直すことになり、
  README のバケット別・トピック別の表構成のせいで、「提案だけをファイルパス付きで」という射影を `grep`
  だけで整えるのは面倒です。

索引は[`scripts/build_roadmap_index.py`](../../scripts/build_roadmap_index.py)が各項目自身のメタデータ
（`Status`、`Topic`、タイトル）から生成しています（BE-0043、BE-0078）。状態フィルタが必要とするのも、
その同じメタデータ、すなわち機械可読で権威のある情報源です。したがって AI セッションが欲しい射影
（「`Proposal` の行をパス付きで」）は、リポジトリがすでに解析しているデータに対する小さな決定論的な
クエリであって、新たな情報源ではありません。

引数で状態を指定する的の絞れたスキルがあれば、どのセッションもこの問いを一貫した低トークンの方法で投げ、
必要な行と、次に読むべきファイルパスをそのまま得られます。

## 詳細設計

作業は互いに排他で網羅的な（MECE）3 つの単位に分かれます。

### 1. 決定論的な状態フィルタクエリ

新たな解析器を足すのではなく、索引生成器がすでに持つメタデータ解析を再利用します。
`scripts/build_roadmap_index.py` は（`roadmap_ids.iter_item_dirs` と `metadata_fields` を通じて）
すべての `roadmaps/BE-*/` 項目の `| Field | Value |` ブロックから `Status`、`Topic`、H1 タイトルを
すでに読み取っています。これをスキルが呼べる小さなクエリとして公開します。次のいずれかの形を取ります。

- 共通の解析処理を `build_roadmap_index.py` から取り出してインポート可能なヘルパーにしたうえで、それを
  呼ぶ薄い CLI エントリポイント（例 ： `python scripts/roadmap_query.py --status Proposal`）。
- それを包む新しい `make` ターゲット。

クエリの仕様は次のとおりです。

- `Status` の値を 1 つ取ります（`Proposal`、`In progress`、`Implemented`、`Proposal (deferred)`）。
  大文字小文字を区別せずに照合し、既知の集合に対して検証します（未知の状態のときは空の表ではなく、有効な
  値を表示して非ゼロで終了します）。
- `ID`、`Item`（タイトル）、`Topic`、`Path`（項目の英語版 `.md` ファイルへの相対パス）を列に持つ
  Markdown の表を出力します。出力を安定させるため `Topic`、次いで `ID` の順に整列します。
- 純粋かつオフラインで動きます。`gh` もネットワークも LLM も使わず、`roadmaps/` 以下のファイルを読むだけ
  です。

プレースホルダ（`BE-0162`）の項目も他と同じく `Status` と `Topic` を読み取るので、進行中の提案も
フィルタに現れます（`ID` の欄にはプレースホルダが表示されます）。

### 2. スキルの定義

既存の `ideation`、`implement-be`、`japanese-tech-writing` スキルと並べて、`.claude/skills/roadmap-status/`
（名称は未定）を追加します。このスキルは次の性質を持ちます。

- **読み取り専用で Claude 向け** です。セッションのためにロードマップを見渡すだけで、項目を起草・実装・
  変更することはありません。ファイルを編集しません。
- 状態を引数に取り、単位 1 のクエリを呼び出し、その表をセッションに返します。Claude はそれを見て、全文を
  開く項目を選びます。
- 有効な状態の一覧と、出力される `Path` 列が提案全文を読むために次に `Read` する対象であることを、
  フロントマターと本文に記します。

### 3. テストとドキュメント

- クエリに対するテスト ： 各状態で絞り込むと、その状態をメタデータに持つ項目がちょうど返ること、および
  未知の状態がきれいに失敗することを確認します（既存のロードマップ形式テスト・索引テストと同じ形で、
  `tests/test_roadmap_*` が自然な置き場所です）。
- [`docs/ai-development.md`](../../docs/ai-development.md)への短い記述 ： セッションがこのスキルの存在を
  知れるよう、モデルと reasoning effort のティア分けの記述（BE-0103）の隣に置きます。

**プライムディレクティブとの整合。** これはドキュメントに対するコントリビュータ向けツールです。Tier-2 の
`run` / CI ゲートに LLM は入りません（第 1 原則）。クエリは決定論的でオフラインです（第 2 原則）。
`roadmaps/` のメタデータだけに触れるので、アプリ非依存です（第 3 原則）。

## 検討した代替案

- **セッションに README を `grep` させるだけにする。** 新しいコードは不要ですが、セッションごとに
  コマンドを導き直し、README のバケット別の表から欲しい射影を組み立て直すことになり、一貫せず誤りも
  生じやすいです。メタデータに対する名前付きスキルこそ、再利用でき低トークンな形です。
- **項目メタデータではなく生成済みの索引表を対象にする。** README の表は情報源ではなくレンダリング結果
  であり、フィルタが索引の再生成に依存してしまいます。各項目自身のメタデータを読む（生成器と同じ方法）
  ほうが権威のある経路で、編集の途中でも正しさを保ちます。
- **代わりに GitHub Pages のロードマップダッシュボード（BE-0094）を拡張する。** あのダッシュボードは
  ブラウザ上の人間に向けたもので、ネットワーク取得なしに行を文脈に載せたい AI セッションの役には立ち
  ません。読み手も面も異なり、重複ではなく補完の関係にあります。
- **トピックやテキストなどでも絞れる汎用のロードマップ検索スキル。** より広い機能ですが、`Status` で
  絞ることに的を絞ったこの項目の範囲外です。豊富なフィルタは、これが有用だと分かってから別項目として
  続けられます。
