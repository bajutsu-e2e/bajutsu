## はじめに

実行レポートを、**run ディレクトリにすでに保存されているデータだけから描画する純粋な処理**にします。こうすると、終わった run からいつでもレポートを再生成でき、シナリオを実行し直す必要がなくなります。今の `report.html` は `run` の最中に、メモリ上の実行結果から一度だけ焼き込まれます。そのため、レポートのテンプレートを改良しても（タブの追加、レイアウトの改善、描画の不具合の修正、すでに取得済みの証跡から計算する新しいビューの追加）、過去の run は古いレポートのまま固定され、内容を新しくする手立ては run をやり直すことだけになります。本項目は、**版を持つレポートデータ契約**（描画器が読み込む、保存済みのモデル）を導入し、二つの提供面を一つの描画器に通します。一つは、既存の run から `report.html` を描き直し `junit.xml` も出し直す、オフラインの `bajutsu report <run>` コマンドです。もう一つは、保存済みモデルから現在のテンプレートで毎回描画する `serve` の Web UI で、ブラウザで見るレポートは常に最新になります。どこにも LLM を足さず、アサーションを再評価せず、合否を書き換えません（描画は記録済みの結果を提示し直すだけです）。したがって決定性優先の原則の内側に収まります。

## 動機

終わった run は、それだけで自己記述的です。決定的なランナーが `manifest.json`（直列化した実行結果。各ステップ、アサーション、所要時間、デバイス、成果物のパス）、実行した `scenario.yaml`、そして証跡ツリー（`screenshot` / `elements.json` / `network.json` / 動画 / device log）を書き出します。レポートは、このデータに対する一つの**ビュー**です。ところが、そのビューが作られるのは実行の瞬間に限られます。`bajutsu/runner/pipeline.py` が `write_report` を呼び、`bajutsu/templates/report.html.j2` を一度だけ描画して `runs/<id>/report.html` に書き出します。以後、その HTML は固定された成果物です。分離できるはずのデータと提示が結合しており、その結合は具体的にいくつもの形で効いてきます。

1. **レポートの改良が過去の run に届きません。** レポートへのどの変更も（証跡タブの追加、ステップ表の改善、描画の不具合の修正、run がすでに取得したデータから計算する要約）、変更の**あと**に実行した run にしか効きません。二週間前の失敗レポートは、古い提示のままです。ツール全体の中心となる成果物（[DESIGN §9](../../DESIGN.md)：`manifest.json` は「レポートと CI の単一の真実」）を、提示し直せません。
2. **「やり直して新しくする」は無駄が多く、しばしば不可能です。** テンプレートの変更を取り込むためだけにシナリオを実行し直すのは、すでにディスク上にある結果を計算し直すために、Simulator と数分、そして（AI 経路では）トークンを費やすことです。さらに、それが**選べない**場面は珍しくありません。失敗が一度きりだった、アプリのビルドやバックエンドの状態が先に進んでしまった、あるいはその run が変えてはならないリリースの記録だった、といった場合です。提示を新しくするのに再実行を要する状態は、避けたいところです。
3. **ホスト型や共有の `serve` は古いレポートを見せます。** `serve`（[BE-0011](../BE-0011-local-web-ui-serve/BE-0011-local-web-ui-serve-ja.md)）を更新しても、配信されるレポートはどれも、各シナリオを実行したときのバージョンが焼き込んだ HTML のままです。二つの run を見比べるブラウザの利用者は、実行した**時期**が違うというだけの理由で、二つの異なるレイアウトのレポートを目にします。

今は、保存した run を描画器へ読み戻す経路がありません。実行結果を書き出す `manifest_dict` はありますが、それを読み込む逆向きの処理はなく、`html_report` は保存済みの一つのモデルではなく、メモリ上の `RunResult` と補助の入力を受け取ります。この欠落を塞ぐこと（描画の**完全な**入力となる、版を持つモデルと、それを読み込むローダ）は、レポートサブシステムに欠けている**再描画**側の半分であり、しかも純粋な提示の関心事なので、決定性優先の原則とは構造上ぶつかりません。

## 詳細設計

### 一つの描画器、保存済みモデル、二つの提供面

要となる変更は、描画器を**保存済みモデルの純関数**にすることです。今の `html_report(run_id, results, run_dir, definitions, sources, …)` は、メモリ上の `RunResult` と補助の入力（メモリ上のシナリオに由来する `definitions` / `sources`）を受け取ります。再設計では、描画器の入力を **run ディレクトリへ完全に書き出される一つの直列化可能なモデル**として定義し、最初の焼き込みと以後のすべての再描画が**その**モデルを読むようにします。こうすれば、レポートが何を示すかが二つの提供面で食い違うことはありません。

モデルの自然な置き場所は、既存の **`manifest.json`** です。これを要約から、版（**`schemaVersion`**）を持つ、欠落のない描画入力へと格上げします。守るべき不変条件は、**描画器が要する入力がすべて run ディレクトリから復元できる**ことです。シナリオやステップごとの結果、実行したシナリオ（構造化定義と YAML の原文）、デバイスのメタデータ、成果物のパス、network のやり取り、そして退けたアラートです。作業は、`html_report` の入力を保存済みのものと突き合わせて欠落を一つずつ塞ぎ、`manifest_dict` の逆向き、すなわち `manifest.json`（および `scenario.yaml` と `run_dir` 配下の証跡）から描画モデルを組み立て直すローダを足すことです。モデル全体を `manifest.json` に置くか、隣に `report.json` を置くかは小さな未決事項です。本質は、描画入力が**完全に保存され、版を持つ**こと、そして一つの描画器がすべての提供面に供することにあります。

### オフラインの提供面（`bajutsu report <run>`）

新しいコマンドが、終わった run を保存済みの成果物から、**現在の**テンプレートで描き直します。`bajutsu report <run-id|path>` は `runs/<id>/report.html` をその場で書き直し（`junit.xml` も出し直し）、`bajutsu report --all`（またはグロブ）は `runs/` 配下のすべての run を焼き直します。デバイスを起動せず、モデルも呼びません。保存済みモデルを読んで描画するだけです。これは、CI が上げ、[BE-0060](../BE-0060-run-report-zip-export/BE-0060-run-report-zip-export-ja.md) が zip に固める、持ち運べる自己完結した `report.html` を新しくするための経路です。コマンド名の細部は小さな未決事項で、本質は `serve` および `run` と同じ描画器を共有することにあります。

### serve の提供面（閲覧のたびに描画する）

`serve` は、既存の `ArtifactStore` 境界（`bajutsu/serve/artifacts.py`。すでに run ファイルへのアクセスを閉じ込めています）を通して、保存済みモデルから現在のテンプレートで**閲覧のたびに動的に**レポートを描画します。焼き込まれた `report.html` は、正典のビューではなく、再生成できるキャッシュ兼エクスポートの成果物になります。そのため `serve` を更新すれば、配信するすべてのレポートがただちに新しくなり、run ごとに焼き直す手順は要らず、実行した時期の違いで二つの run のレポートが違って見える理由もなくなります。

### 再描画で反映できる範囲（正直な境界）

再描画が反映するのは、**提示と、すでに取得済みのデータから導けるビュー**です。レイアウト、スタイル、描画の不具合の修正、run がすでに持つ証跡から計算するタブや要約の追加です。一方で、**過去の run が取得しなかったデータ**（新しい証跡の種類や、あとからモデルに足したフィールド）を、さかのぼって埋めることはできません。そうした run については、描画器が古い `schemaVersion` を読み取り、欠けている部分を「この run では未取得」と示します。失敗させたり、値を捏造したりはしません。これにより約束の範囲が正確に保たれます。再描画が反映するのは、run がすでに取得したデータから導けるテンプレートや機能の変更であり、本当に新しい取得は、やはり新しい run を要します。欠けているものを明示するのは、ランナーが省いた証跡についてすでに実践している正直さ（[DESIGN §10](../../DESIGN.md)）と同じです。

### 決定性、ゲート、合否

* **LLM なし、合否への影響なし。** 描画は、決定的な `run` がすでに記録した結果を提示し直すだけで、アサーションを再評価しません。run の合否は保存済みモデルから読み取り、計算し直さないので、再描画がそれを変えることはできません。prime directive の 1 と 2（[CLAUDE.md](../../CLAUDE.md)）は構造上保たれ、結果として決定性は強まります。レポートが、保存済み証跡の再現可能な関数になるからです（[DESIGN §2](../../DESIGN.md)）。
* **アプリ非依存。** 描画は run のモデルに依存し、どのアプリにも依存しません。`apps.<name>` 設定、ドライバ、ランナーのいずれにも触れません。
* **Linux でテストできます。** 再描画に Simulator は要りません。ファイルを読んで描画するだけです。ローダ、描画器、コマンドは、フィクスチャの run ディレクトリを相手に既存の Linux ゲートで単体テストできます。

### テストの契約（機械チェック可能）

Simulator を要しないテストで固定します。(a) **ラウンドトリップ**：保存済みモデルを読み戻すと、描画器の入力が欠落なく復元されます（`load(persist(result))` がレポートの使う全フィールドを保つ）。(b) **純粋性**：フィクスチャの run を二度再描画するとバイト単位で一致し、新規 `run` が焼き込むレポートは、同じ run と同じテンプレートに対して `bajutsu report` が生むレポートと一致します（描画は保存済みデータだけに依存し、時刻はその中の値から取り、実時計からは取らない）。(c) **古い版も描画できる**：古い `schemaVersion` で書かれたフィクスチャがエラーなく描画され、新しい版だけにある部分は「未取得」と示されます。(d) **合否のぶれなし**：再描画したレポートの合否が、すべてのシナリオで保存済みモデルと一致します。

### スコープ外

新しい証跡の種類の追加や、run 中の取得を増やすこと（証跡サブシステム、[DESIGN §9](../../DESIGN.md)）、run をまたぐレポートの差分表示や複数 run のダッシュボード（データ契約が可能にする自然な後続ですが、別項目です）、そして `run` が決めることや取得することへの変更は、対象外です。

## 検討した代替案

* **焼き込みだけを続け、新しくするには run をやり直す（現状）。** これが問題そのものです。すでにディスク上にある結果を計算し直すために Simulator と数分（AI 経路ではトークンも）を費やし、しかも失敗が一度きりだった、アプリやバックエンドの状態が先に進んだ、あるいは run が変更不可なリリース記録である、といった場合には実行すらできません。
* **serve は動的に描画するが、オフラインのコマンドは設けない。** ブラウザのビューは新しくなりますが、持ち運べる `report.html`（CI が上げ、[BE-0060](../BE-0060-run-report-zip-export/BE-0060-run-report-zip-export-ja.md) が zip にする成果物）は焼き込まれた版のまま固定されます。半分の解として却下します。共有され保管されたレポートを新しく保つのは、オフラインの再描画です。
* **オフラインのコマンドは設けるが、serve は焼き込み済みのファイルを配信し続ける。** 必要に応じて成果物は新しくできますが、Web UI は各 run を焼き直すまで古い HTML を見せ続けます。反対側から見た同じ理由で却下します。両方の提供面が必要で、だからこそ一つの描画器を共有します。
* **`manifest.json` を格上げせず、別に `report.json` を設ける。** `manifest.json` を軽い要約のまま保てますが、ほぼ重複する二つめの成果物が増え、両者がずれる余地も生まれます。単一の版付きの源として `manifest.json` を再利用するほうを採ります。別ファイルは、manifest を要約のまま保つ必要がある場合の次善策です。いずれにせよモデルは完全に保存され、`schemaVersion` を持ちます。
* **生の JSON とクライアント側の JS アプリを配り、ブラウザで描画する。** レポートは、ダブルクリックで開け zip にもきれいに収まるよう、インライン資産と相対リンクを備えた自己完結の静的 HTML として作っています（[BE-0011](../BE-0011-local-web-ui-serve/BE-0011-local-web-ui-serve-ja.md)、[BE-0060](../BE-0060-run-report-zip-export/BE-0060-run-report-zip-export-ja.md)）。JSON と SPA はその可搬性を壊します。描画はサーバ側とオフラインに留め、自己完結した HTML を出し続けます。
* **テンプレートの版が食い違ったら自動で焼き直す。** 魅力的ですが、読み取りのたびに過去の run の成果物を黙って書き換えるのは意外性が大きく、run を記録として扱う姿勢と相容れません。明示的な `bajutsu report` コマンドと serve の閲覧時描画があれば、利用者の知らぬ間に過去の run を変えることなく必要を満たせます。オプトインの自動更新は、あとで再検討できます。
* **「レポート」専用のロードマップトピックの新設。** 見送ります。[BE-0060](../BE-0060-run-report-zip-export/BE-0060-run-report-zip-export-ja.md) と並べて「オーサリング体験」の下に置き、BE-0060 自身が示した先例（兄弟が増えるまでは一つのトピックに置く）にならいます。本項目と BE-0060 は、レポートサブシステムの最初の二つの提案です。さらに増えたら、そのとき「レポート」トピックを切り出せます。
