## はじめに

リポジトリのうち人手で維持している部分を、実際に出荷された内容と毎日照合する GitHub Actions
ワークフローを **2 本**追加します。差分（ドリフト）が見つかったときは、それぞれが**自分の**
**ドラフト**プルリクエストを開き、人間がレビューしてマージします。

- BE 項目（`Status` / `Progress` / `Implementing PR`）を最新に保つ **roadmap-refresh** ワークフロー
- プロースのドキュメント（`docs/` と `DESIGN.md`）を挙動と揃える **docs-refresh** ワークフロー

2 本は 1 つの設計（資格情報の型、契約ファイルの型、ジョブ内ゲート、更新用ドラフト PR）を共有し
ますが、実行も出荷も独立しています。どちらも更新の起草には、既存の自動レビューと同じ AI プロバイダ
（[BE-0203](../BE-0203-claude-code-pr-review/BE-0203-claude-code-pr-review-ja.md)）経由で Claude Code
アクションを使います。マージするのは人間だけです。LLM は `run`/CI の合否には一切関与せず、生成された
各 PR もほかの PR と同じく決定論的な `make check` で検証されます。

## 動機

このリポジトリは、文書のうち*機械的に導ける*部分についてはすでに正しさを保っています。`make
roadmap-index` が索引を再生成し、コミット済みの表がずれれば `make test` が落ちます。
[BE-0096](../BE-0096-docs-roadmap-link-integrity/BE-0096-docs-roadmap-link-integrity-ja.md) は
`docs/` からロードマップへのリンク切れをゲートの失敗にします。`roadmap-drift-check`
（[BE-0149](../BE-0149-roadmap-placeholder-format-guardrail/BE-0149-roadmap-placeholder-format-guardrail-ja.md)）
はオープン中の PR をテンプレートと再照合します。しかし、これらのいずれもカバーできないのが、
最新に保つのに*意味的な判断*を要し、そのために静かに陳腐化していく内容です。

- **BE 項目の `Status` / `Progress` / `Implementing PR`。** 作業合意では、項目を前進させた PR は
  同じ変更のなかで `Progress` のチェックを付け、`Implementing PR` を埋め、作業の開始や出荷に
  合わせて `Status` を切り替えることになっています。しかし実際には、コードだけがマージされて項目が
  更新されないことがあり、コードがマージ済みなのに項目が `Proposal` のまま残ったり、`Progress` の
  チェックリストが実態に遅れたりします。これはフォーマット検査ではなく文章と状態の照合なので、
  何も検知できません。
- **`docs/architecture.md#implementation-status`。** 「すでに何が存在するか」の拠り所とされている
  節ですが、機能が出荷されたときにコントリビュータが更新を忘れなければ、という前提でしか正確さを
  保てません。
- **`DESIGN.md` / `docs/architecture.md` を挙動と揃えること
  （[BE-0113](../BE-0113-design-doc-realignment/BE-0113-design-doc-realignment-ja.md)）。** この規則は
  あえて CI ゲートではなくレビュー時の規範にとどめています。文章の一段落がコードと合っているかを
  確かめるには意味的な判断が要り、それをゲートにすれば LLM を `run`/CI の合否経路に載せてしまう
  （プライムディレクティブ 1）からです。そのため注意で担保されており、注意は途切れます。

共通するのは、これらがまさに、決定論的なゲートではプライムディレクティブ 1 を破らずには塞げない
穴だという点です。だからこそ、人間が気づく瞬間と瞬間のあいだに溜まっていきます。照合の下書きを
定期的に作り、マージの判断は人間に委ねるエージェントは、その一線を越えずに遅延を縮めます。これは
自動レビューに対する起草側の対応物です。BE-0203 がマージを止めない AI *レビュアー*を足したのに
対し、本項目はマージしない AI *起草者*を足します。

**なぜ 1 本ではなく 2 本か。** 2 種類のドリフトは性質の異なる作業であり、分けてレビューするのが
最適です。ロードマップの照合は小さくほぼ機械的で（`Status` を切り替え、`Progress` にチェックを付け、
`Implementing PR` の行を足し、索引を再生成する）、レビュアーはマージ済みの PR と突き合わせれば
すぐに確認できます。ドキュメントの照合はより重く、2 言語にわたって挙動に合わせて文章を書き直すため、
レビュアーは実際に段落を読む必要があります。両方を 1 本の PR にまとめると、速く低リスクな
ロードマップの修正が、遅く慎重なドキュメントのレビューの後ろで待たされ（逆もまた同じ）、レビュアーが
本来分けて判断したい 2 つの差分が混ざります。分けることで失敗とノイズも切り離せます。ドキュメント
だけがドリフトした日は、ロードマップ側が空の混在 PR ではなく、ドキュメントに絞った PR が 1 本
生まれ、片方の不調がもう片方を止めることもありません。将来は異なる周期（例: ロードマップは毎日、
ドキュメントは毎週）で走らせることも、互いを乱さずにできます。

## 詳細設計

`.github/workflows/roadmap-refresh.yml` と `.github/workflows/docs-refresh.yml` という 2 本の定期
ワークフローを、1 つの**共有された型**の上に構築します。両者が違うのは、*何を*照合するかと、*どの*
ファイルに触れてよいか、だけです。AI が起草した変更はすべて、人間がレビューしゲートで検証される PR
を通ります。`main` に直接載せることはありません。

**共有された型は、この文書の上だけでなくコード上でも共有し続けます。** 以下の共通部分は設計上の
約束ですが、独立した 2 本の YAML ファイルは、それぞれが編集されるうちにドリフトしえます（2 資格情報
ゲートへの後の微修正が一方にだけ入る、など）。したがって実装では、共有された型——休止ゲート、
concurrency/timeout、ジョブ内 `make check`、固定ブランチの上書き防止ガード、PR の作成・更新——を
1 つの再利用可能な単位（`workflow_call` の再利用ワークフロー、またはコンポジットアクション）に
くくり出し、両方の薄いワークフローファイルが、それぞれ異なるパラメータ（ブランチ、契約ファイルの
パス、許可リスト）だけを渡して呼び出す形にします。そうすれば、両方を同一に編集し忘れないよう気を
つけるのではなく、構造として歩調が揃います。

### 共有された型（両ワークフロー）

1. **トリガ・concurrency・timeout。** `schedule`（毎日の cron）と `workflow_dispatch`（オンデマンド）。
   2 本の cron 時刻を少しずらし、ランナーを取り合わないようにします。各ジョブには `concurrency`
   グループ（`cancel-in-progress: false`）を付け、手動起動がそのワークフロー自身の夜間実行と競合し
   ないようにします。また `timeout-minutes` の早期打ち切り上限を設け、プロバイダ呼び出しがハングした
   ときにランナーを占有し続けずに打ち切ります。いずれも `claude-review.yml` の `concurrency:` /
   `timeout-minutes:` の前例に倣います。

2. **2 つの資格情報、全部揃うか休止か。** 各ジョブは独立した資格情報が 2 つ必要で、その両方が
   揃っていない限り緑の no-op でなければなりません。これは `roadmap-id.yml` が App トークンに用いて
   いる単独資格情報の休止ゲートと、`claude-review.yml` が AI プロバイダの資格情報に用いているそれを
   組み合わせたものです（どちらの前例も 2 つとも必須にはしておらず、両方を求めるのは本提案が初めて
   です）。設定が中途半端なリポジトリを決して赤にしないためです。
   - **AI プロバイダ。** `claude-review.yml` とまったく同じ方式で選びます（BE-0203、
     [BE-0104](../BE-0104-vendor-neutral-ai-backend/BE-0104-vendor-neutral-ai-backend-ja.md)）。Claude
     Code サブスクリプション（`CLAUDE_CODE_OAUTH_TOKEN`）があればそれを、なければ OIDC 経由の Amazon
     Bedrock（`AWS_BEDROCK_ROLE_ARN` + `BEDROCK_MODEL_ID`）を使い、どちらもなければ休止します。これが
     更新を起草します。
   - **自動化 App トークン**（`AUTOMATION_BOT_APP_ID` / `AUTOMATION_BOT_PRIVATE_KEY`、
     `actions/create-github-app-token` 経由）。`roadmap-id.yml` や `roadmap-drift-check.yml` と同じ
     使い方です。既定の `GITHUB_TOKEN` で開いた PR はほかのワークフローを起動しないため、更新 PR
     自身の `check` CI が走りません。App のインストールトークンにはその制約がなく、App の身元で PR を
     開くことこそが、AI の出力に決定論的なゲートを実際に走らせる鍵です。

3. **追跡される契約ファイル**（`.github/*-refresh-prompt.md`、`.github/claude-review-prompt.md` に
   倣ったもの）が、アクションの実行する正典の指示です。各ワークフローが自分のものを持ち、PR を開く
   前に差分へ適用する**パス許可リスト**、**保守性の規則**（ドリフトの具体的な証拠——マージ済みの PR、
   出荷された機能——がある箇所だけを更新案とし、確信が持てなければファイルを変えずに残し、推測する
   のではなく PR 本文に不確かさを記す）、そして該当する場合のバイリンガルの規則（書き起こす日本語は
   `japanese-tech-writing` スキルと敬体に従い、英日を同時に更新する）を定めます。

4. **PR を開く前のジョブ内 `make check`。** エージェントがファイルを編集したあと、ワークフローは
   ジョブ内で決定論的なゲートを実行し、AI の出力を人間の変更と同じ基準に保ちます。ゲートが赤でも PR
   を開くこと自体は止めません（いずれにせよ**ドラフト**で開きます）が、結果を PR 本文に載せます。この
   PR は通常の `check` CI の対象にもなります（だからこそ App トークンが必要です）。

5. **ワークフローごとに 1 本の、冪等・常にドラフト・人間マージの更新用 PR。** 照合しても差分が出
   なければ、ジョブは PR を開いたり触ったりせずに終了します（静かな週に毎日ノイズを出しません）。
   差分があれば、そのワークフローの固定ブランチに push し、オープン中の PR があればそれを再利用します。
   毎日新しい PR を開くことはしません。PR は `--draft` で開き、人間がレビューし、ready にしてマージ
   するのは人間**だけ**です。どちらのブランチにも自動マージはありません。

   **固定ブランチ上の人間の作業を決して上書きしません。** 日次実行がブランチを書き換えるのは、その
   時点の tip が自動化自身（App/bot の身元）が最後に作ったコミットである場合だけです。前回実行以降に
   人間がブランチへ push している場合——レビュアーの fixup コミット、レビュー中の編集——は、その上を
   force-update **しません**。ブランチをそのまま残し、スキップしたことを（ジョブ内の `::warning::` と
   PR 上の注記で）表明します。こうして人間の作業が上書きされることはなく、スキップは黙って起きるので
   はなく声高に示されます。これは `scripts/check_stale_roadmap_prs.py`（`open_or_update_fix_pr`）の
   ガードに倣ったものです。同スクリプトは、古いスナップショットから計算した内容で新しいコミットを
   潰さないよう、ブランチの生きた tip ではなく修正を計算した正確な SHA を基点にします（177〜180 行の
   コメント）。新しいブランチを作るのは、固定ブランチがまだ存在しないときだけです。

### 何が違うか

| | roadmap-refresh | docs-refresh |
|---|---|---|
| ブランチ / PR | `chore/roadmap-refresh` | `chore/docs-refresh` |
| 契約 | `.github/roadmap-refresh-prompt.md` | `.github/docs-refresh-prompt.md` |
| 照合対象 | マージ済み PR に対する BE 項目の `Status` / `Progress` / `Implementing PR`。索引に影響する項目を変えたときは `make roadmap-index` を実行 | `docs/architecture.md#implementation-status`、`DESIGN.md` / `docs/architecture.md` の文章と挙動（BE-0113） |
| パス許可リスト | `roadmaps/**` のみ | `docs/**` と `DESIGN.md` のみ |

どちらのワークフローもプロダクトコード（`bajutsu/`、`BajutsuKit/`、テスト、設定、デモ）には決して
触れません。許可リストがそれを禁じ、ideation スキルの「起草のみ、実装はしない」という境界に倣います。

docs-refresh の許可リストは、トップレベルの `README*` と `CLAUDE.md` を意図的に**除外**します。これらは
挙動を追う文章ではなくプロジェクトの契約面であり、とりわけ `CLAUDE.md` は AI 起草者自身が縛られる
プライムディレクティブを定めています。エージェントが自分の動作契約を自動で書き換える下書きを作るのは、
ドラフト PR を挟んだとしても越えたくない境界です。そこへの編集は人間が書くものとして残し、ワークフロー
の担当は挙動に対してずれる文章、すなわち `docs/` と `DESIGN.md` に限ります。（`README*` / `CLAUDE.md`
の `docs/` → ロードマップリンクの決定論的な修復は、LLM を使わない別の関心事で、すでに
[BE-0096](../BE-0096-docs-roadmap-link-integrity/BE-0096-docs-roadmap-link-integrity-ja.md) が担っています。）

### プライムディレクティブとの整合

LLM は純粋に*起草*の経路でのみ使います。`record` がシナリオを起草するのとまったく同様に更新を
下書きし、その出力は人間がレビューしてマージしなければならないドラフト PR です。`run` や
`required_status_checks` に LLM 呼び出しは一切足しません。決定論的な `check` が唯一のマージ裁定者で
あり続け、判断する人間が唯一の裁定者です。各ワークフローのパス許可リストにより、ドライバ・ランナー・
テストには触れないため、決定論やアプリ非依存の中核に影響を与えることはできません。

## 検討した代替案

- **ロードマップとドキュメントを 1 本の PR で更新する統合ワークフロー。** 本提案の当初の形です。分割を
  選んで却下しました。速くほぼ機械的なロードマップの修正が、遅く慎重なドキュメントのレビューの後ろで
  待たされ、レビュアーが本来分けて判断したい 2 つの差分が混ざり、2 つの失敗モードと周期が結合するため
  です（*なぜ 2 本か*を参照）。共有設計はここ（本項目）の 1 か所にまとまるので、2 本にする重複コストは
  小さく済みます。
- **決定論のみの更新（LLM なし）。** 索引の再生成、リンクの再検査、トラッキング Issue の URL 補正を
  定期実行する案です。*全体の*答えとしては却下します。これらの部分はすでに `make check` /
  `make roadmap-index` / BE-0096 でカバーされており、そもそも陳腐化する箇所ではありません。本項目が
  対象とするドリフト（`Status`/`Progress` の遅れ、文章と挙動のずれ）は本質的に意味的で、判断なしには
  照合できません。
- **更新を `main` に直接コミットする。** 却下します。人間をループから外し（プライムディレクティブ 1
  ——ここでは人間が裁定者）、AI が起草した文章のレビューを飛ばします。ドラフト PR にすることこそが
  眼目です。
- **新規ワークフローではなく自動レビュー（BE-0203）を拡張する。** 形が違います。BE-0203 は既存の PR
  を*レビュー*してコメントするのに対し、本項目は定期的に PR をゼロから*起草*します。プロバイダ選択の
  ロジックを再利用するのは適切な共有ですが、ジョブを統合すると無関係なトリガと範囲を 1 本に詰め込み
  すぎます。
- **オンデマンドのみ（cron なしの `workflow_dispatch`）。** 追加のトリガとしては残しますが、価値は
  無人の毎日の周期——「常に最新」——にあり、オンデマンドのボタンだけではそれを実現できません。

## 進捗

> 開発の進行に合わせて常に最新の状態に保ってください。チェックリストは *詳細設計* の MECE な
> 作業分解（作業の単位ごとに 1 つ）に対応し、ログには変更内容と時期（古い順）を PR へのリンクと
> ともに記録します。

- [x] `roadmap-refresh.yml` —— BE の `Status` / `Progress` / `Implementing PR` を照合する日次ワークフロー。専用の契約・ブランチ・更新用ドラフト PR を持つ
- [x] `docs-refresh.yml` —— `docs/` / `DESIGN.md` の文章と挙動を照合する日次ワークフロー。専用の契約・ブランチ・更新用ドラフト PR を持つ
- [x] 両者に組み込む共有された型: 2 つの資格情報による休止ゲート、concurrency + timeout、PR 本文に反映するジョブ内 `make check`、差分ゼロなら開かない冪等な終了
- [x] 共有された型を、両方の薄いワークフローが呼ぶ再利用可能な `workflow_call`（またはコンポジットアクション）にくくり出し、ドリフトしないようにする
- [x] 固定ブランチの上書き防止ガード: 自動化自身の tip の上でのみ force-update し、人間が push していたら声高にスキップ
- [x] ワークフローごとのパス許可リスト（roadmap のみ / docs のみ。`README*` / `CLAUDE.md` は除外）で、プロダクトコードにも契約面にも触れないことを強制
