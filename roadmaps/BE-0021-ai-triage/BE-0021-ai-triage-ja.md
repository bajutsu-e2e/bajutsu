## はじめに

失敗証跡を AI が読み、原因要約と修正提案を出します（人間レビュー前提）。`bajutsu triage`（ルールベース）＋ `--ai`（Claude、失敗スクショ込み）の 2 段構成です。決定的な `trace` コマンドはその下の層になります。

## 動機

決定的な run が失敗したとき、生の証跡（失敗メッセージ、失敗ステップ、アクセシビリティ要素ツリー、スクリーンショット、ログ）はそれ自体は正しいものの、読み解くのは手間がかかります。これを原因と次の具体的な編集へ変換する作業は、まさに判定を伴わない読解作業であり、LLM が得意とするところです。そしてこれは境界の調査役側にはっきり位置します。失敗を説明するだけで、失敗を判定することはありません。この層がなければ、すべての失敗でシナリオ修正に取りかかる前に、人間が証跡を手作業で掘り起こすコストを払うことになります。狙いは、合否判定を完全に決定的ランナーに残したまま、このループを短くすることです。

## 詳細設計

トリアージは**助言的**であり、`run`／CI ゲートの完全に外側にあります。ここでは何ひとつ合否を決めません。流れは 3 つの部品からなります。

* **`assemble`（純粋、AI なし）** は保存済み run の `manifest.json` を読み、最初に失敗したシナリオの `TriageContext` を再構成します。失敗メッセージ、失敗ステップ、失敗した期待値、失敗箇所に最も近いアクセシビリティ要素ツリー、最も近いスクリーンショット、シナリオ定義、そして失敗ステップが操作したセレクタ id です。これは決定的なファイル読み取りであり、API キーなしのオフラインでも動作します。
* **`TriageAgent` プロトコルと 2 つの実装。** 既定は `HeuristicTriageAgent` で、ルールベースの決定的なエージェント（AI なし）です。失敗を `selector`／`timing`／`assertion`／`unknown` に分類し、target id が見当たらないが似た id が取得画面上にあるときは「もしかして」のヒントを出します。これが古典的な id リネームの自己修復です。`ClaudeTriageAgent`（`--ai`）は同じプロトコルの背後で Claude に問い合わせ、同じ証跡（失敗スクリーンショット込み）を読み、同じ構造化された `Triage` を返す単一の `diagnose` ツールの呼び出しを強制されます。どちらも要約、カテゴリ、最小限の提案を出し、AI 版は手書きルールなしで推論する点だけが異なります。
* **`render`** は人間が読んで対処するための診断を出力します。

決定的な `trace` コマンドはトリアージの下の層です。証跡を解釈せず並べます。トリアージはその証跡を解釈します。LLM が呼ばれるのはここ、失敗後に人間が実行する調査役の経路だけであり、`run` の中では決して呼ばれません。

## 検討した代替案

* **run を読んで直接判定する LLM。** 即座に却下しました。非決定的なモデルを合否経路に置くことになり、不変の方針に反します。トリアージの本質は、決定的ランナーが既に宣言した失敗を説明することにあります。
* **ルールベースのみで AI 選択肢なし。** ヒューリスティクスはよくある形（id リネーム、曖昧なセレクタ、タイムアウトした wait）を決定的に捉えますが、スクリーンショットを読んだり未知の失敗を推論したりはできません。両者を 1 つのプロトコルの背後に置くことで、ルールベースをコストゼロのオフライン既定として残しつつ、より深い読解が必要な失敗には `--ai` を使えるようにしています。
* **構造化されたツール呼び出しの強制ではなく自由形式の LLM 文章。** 単一の `diagnose` ツールを強制することで、出力を `Triage` データクラスへ、そして BE-0022 が適用する構造化 fix へきれいに対応づけます。文章から構造を復元するパーサに頼らずに済みます。
