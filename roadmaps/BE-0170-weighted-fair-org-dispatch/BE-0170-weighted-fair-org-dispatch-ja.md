## はじめに

セルフホスト向けのサーバーバックエンド（[BE-0016](../BE-0016-web-ui-self-hosting/BE-0016-web-ui-self-hosting-ja.md)）は、
Postgres の `jobs` テーブルを介してテストジョブを Mac ワーカーのプールへ分配します。同時実行数は
すでに**全体上限**、**ユーザー単位の上限**、**組織単位の上限**（`max_concurrent_per_org`、
[#367](https://github.com/bajutsu-e2e/bajutsu/pull/367) で出荷済み）で抑えていますが、どの上限も、
超過したジョブを HTTP 429 で**拒否する**ことで守っています。この提案は、その拒否を**保留**へ置き換えます。
組織単位の待機キューと、組織間をラウンドロビンで巡回するディスパッチャを導入し、逼迫時にも希少な
Mac プールの取り分が公平に保たれるようにします。BE-0016 の「単一ノードをプールへ育てる」作業のうち、
機械的に検証できる契約を持つ唯一の部分を切り出したものです。

## 動機

Mac プールは希少で弾力性のない資源です。ステートレスな Web コンテナのように、需要に応じて
Simulator を増やすことはできません。純粋な先入れ先出し（FIFO）の分配では、ジョブをまとめて投入した
一つの組織が空きスロットをすべて占有し、その実行が終わるまで手放しません。ほかの組織は自分の上限に
達していなくても、キューの後ろで待たされます。組織単位の上限（BE-0016）は、一つの組織が*同時に占有できる*
スロット数を制限しますが、上限に達したジョブはキューに積まれるのではなく単に**拒否**（429）されます。
そのため呼び出し側は再送を迫られ、プールが空くよりわずかに速くジョブを投入する行儀のよい組織でも、
不必要な失敗を見ることになります。

足りないのは**公平なスケジューリング**です。複数の組織に待機中の作業があるとき、プールは投入順に
使い果たされるのではなく、組織間で分け合われるべきです。そのためには、まだ受け入れられないジョブを
制御プレーンが*保留*し、到着順ではなく公平さに基づいて次のジョブを選ぶ必要があります。

## 詳細設計

この設計は、新しいサブシステムを足すのではなく、既存の受け入れ判定の継ぎ目を拡張します。現在は
`_register_and_dispatch`（`bajutsu/serve/operations.py`）の末尾が `try_register`（`bajutsu/serve/jobs.py`）を
呼び、ロックの下で実行中のジョブ数を全体上限・ユーザー単位上限・組織単位上限とアトミックに突き合わせ、
いずれかの上限に達すると **HTTP 429 で拒否**します。作業分解は次のとおりです。

1. **組織単位の待機キュー。** 「上限で拒否」する末尾を**保留**へ置き換えます。すぐに受け入れられない
   ジョブ（組織が上限に達している、または全体上限が埋まっている）は、拒否する代わりに組織単位の
   待機キューへ積みます。このキューは `try_register` が守るのと同じ状態の中に置くため、受け入れ判定は
   アトミックなまま保たれます。
2. **ラウンドロビンのディスパッチャ。** スロットが空くたび（実行が完了したとき、または新しい投入の
   たび）に、ディスパッチャは待機中の作業を持つ組織を**ラウンドロビン**の順に巡り、上限に達していない
   最初の組織から次のジョブを受け入れます。上限に達している組織はその周回では飛ばします。どの組織にも
   プールを独占させないのは、このラウンドロビンのカーソルです。
3. **優先度ティアを重みとして扱う。** 優先度のティアは同じラウンドロビンに乗せます。優先度の高い
   ティアを別のスケジューラで扱うのではなく、より高い頻度で巡回する（重み付きラウンドロビン）ことで
   表現します。同じティアの中では、ジョブは投入順を保ちます。
4. **後方互換性。** 単一テナント（`orgs:` ブロックがなく、既定の組織が一つ）の場合、ディスパッチャは
   ただの FIFO に縮退するため、単一テナントのデプロイは従来どおりです。組織単位上限の既定値 `0`（無制限）も
   同様に、運用者が設定するまで既存の挙動を変えません。

**検証。** これは残るプール化作業のうち、機械的に検証できる契約を持つ唯一の部分です。そのため（組織単位
上限のテストと同じく）**Simulator を使わず** `ServeState` に対して単体テストします。二つの組織が競合する
状況で受け入れられるジョブが公平に交互になること、どの組織も自分の上限を超えないこと、単一テナントの
デプロイが FIFO のままであること、優先度ティアが重みに比例して多く受け入れられることを固定します。これらの
不変条件はすべて Python の制御プレーン内にあり、Linux の `make check` ゲートで動きます。

**調整メモ。** この変更は `bajutsu/serve/operations.py` と `jobs.py` に入ります。進行中のほかの serve 作業が
触る面と同じであるため、そのファイルを編集中の未マージ PR がマージされたあとに入れるか、コンフリクトを
避けるよう調整するのがよいでしょう。

## 検討した代替案

- **429 で拒否し続け、クライアントに再送させる。** 却下します。スケジューリングの問題を呼び出し側すべてに
  押し付け、行儀のよい組織にも不必要な失敗を生み、公平さを表現できません（一つの組織からの再送の集中は
  やはり競争に勝ってしまいます）。ジョブをサーバー側で保留してこそ、制御プレーンは公平な取り分を
  強制できます。
- **全組織を通じた純粋な FIFO。** 却下します。これは組織単位上限がすでに回避策を講じなければならなかった
  現状そのものであり、FIFO では一つの組織の集中投入が希少な Mac プールを独占できます。組織単位の待機キューと
  上限を尊重するラウンドロビンが、その解決策です。
- **公平さのラウンドロビンとは別に優先度スケジューラを設ける。** 不要として却下します。優先度を重みとして
  ラウンドロビンに畳み込めば、相互作用する二つの経路ではなく単一の分配経路で済み、推論もテストも容易に
  なります。
