## はじめに

Bajutsu は実行の証跡（スクリーンショット、アクセシビリティ要素のダンプ、`network.json`、シナリオのコピー）を、プロセスのデフォルト umask のまま作成した実行ディレクトリ配下に書き込みます。一般的な `022` umask の環境では、ディレクトリは `0755`、ファイルは `0644` となり、同一ホスト上の他アカウントから読める状態になります。

## 動機

`bajutsu/evidence.py` は実行ディレクトリのツリーを素の `Path.mkdir` 呼び出し（`step_dir.mkdir(parents=True, exist_ok=True)` や `scenario_dir.mkdir(parents=True, exist_ok=True)` など）で作成し、スクリーンショットや要素ダンプ、インターバル録画も通常のファイル書き込みで配置しています。`bajutsu/network.py` が `network.json` へ捕捉した通信を書き出す処理も同様です。いずれの呼び出しも明示的な `mode` を渡していないため、実際のパーミッションはプロセスの umask 任せになり、多くの環境ではデフォルトで他ユーザーから読める状態です。

これはすでに悪用された脆弱性ではなく、ハードニングの抜け穴です。実行の証跡には Bajutsu が他の箇所で意図的に保護している類の機微な情報が含まれ得ます。`network.json` にはリクエスト・レスポンスの本文やヘッダーが載ることがあり（ルールに基づいてマスクはしていますが、マスキングはあらゆる将来のヘッダーやボディの形を網羅する保証ではなく、あくまでベストエフォートの denylist です）、スクリーンショットやシナリオ自体の証跡にも、撮影時点で画面に表示されていたマスク済みまたは未マスクのテキストが写り得ます。深刻度は中程度です。実行ディレクトリを読むには同一ホスト上の別アカウントが必要ですが、共有 CI ランナーはまさにその隣接関係が起こり得る環境であり、BE-0032 がログに対してすでに防いでいるのと同じ脅威モデルです。

## 詳細設計

書き込む内容や合否判定の仕組みには手を加えず、ランナーが証跡を作成する際のパーミッションだけを絞ります。

- 実行ディレクトリのトップレベルを最初に作成する時点で `0700`（所有者のみ読み書き実行可）にし、配下のすべてのパスが世界読み取り不可な親ディレクトリを継承するようにします。
- 機微な内容を含み得るファイル（`network.json`、コピーしたシナリオ、スクリーンショット）は、書き込み時点で `0600`（所有者のみ読み書き可）にします。ディレクトリのパーミッションだけでアクセスを制限するのではなく、ファイル単位でも制限します。
- バックエンド間で統一して適用します。idb と Playwright はどちらも同じ `evidence.py` / `network.py` の経路で書き込むため、修正はドライバごとではなく共有のランナーコードに一度だけ入れます。
- 移植性を保ちます。`Path.mkdir(mode=...)` や明示的なモード付きの `os.open`（あるいは作成直後の `os.chmod`）は macOS と Linux で同じように動作し、決定的ゲートが必要とする範囲はこれで足ります。

## 検討した代替案

- **コードで強制する代わりに umask の要件をドキュメント化する。** 却下しました。すべての運用者・CI イメージが適切に厳しい umask を設定していることに依存してしまい、これはまさに Bajutsu が他の箇所で避けている「ホストの周辺設定に依存する」種類の前提です。作成時にモードを強制すれば、この保証は無条件に成り立ちます。
- **証跡を保存時に暗号化する。** 過剰な対策として却下しました。ここでの脅威は世界読み取り可能なファイルを別のローカルアカウントが読むことであり、ディスクの盗難ではありません。ファイルモードの制限であれば、鍵管理などの複雑さを増やさずにこの脅威に直接対処できます。
- **パーミッションを制限する代わりにマスキングをより積極的に行う。** 代替ではなく補完だと考え却下しました。マスキングは漏えいした証跡が明かす内容を減らせますが、あらゆる機微な値を網羅できる保証はなく、証跡を読める人を制限する必要性そのものはなくなりません。
