# BE-0049 — 決定性／フレーキネス監査

* 提案: [BE-0049](BE-0049-determinism-flakiness-audit-ja.md)
* 状態: **提案**
* トラック: [提案](../../README-ja.md#提案)
* トピック: 競合調査（Maestro）由来の候補
* 由来: Maestro

## はじめに

フレーキネスを許容するのではなく、シナリオの決定性を*証明する*読み取り専用の診断です。同一条件で
繰り返し実行して非決定的な結果を検出し、セレクタ／待機の安定性を静的に採点します。これは助言的で
あり、完全にゲートの外にあります。ばらつきは所見として報告され、黙って通過させることも、判定に使う
こともありません。

## 動機

Bajutsu の競合に対する中心的な主張は、ゲートが吸収によってではなく**契約として**決定的である、と
いう点です。Maestro は逆の立場を掲げています。「フレーキネスへの組み込み耐性」を謳い、暗黙の自動
待機やタップ再試行で不安定さを呑み込み、加えて `retry` コマンドを備えます。その結果、フレーキーな
テストは*修正*されるより*隠される*傾向にあります。Bajutsu はそれを拒みます（固定 `sleep` なし、
曖昧なセレクタは即座に失敗）。しかし今日、その主張は**暗黙**です。ツール内に、あるスイートの決定性
を定量化・証明するものが何もなく、導入を検討する側は信用するしかありません。

本項目はこの差別化を実体のあるものにします。「私たちは決定的だ」を、チームが実行できるレポートに
変えます。シナリオを K 回実行し、結果やタイミングがばらついたステップ・アサーションを表面化させ、
各セレクタを安定性のはしご（一意に解決する `id` が `label` ／ `traits` に勝り、それらが `index` ／
生座標に勝る）に照らして静的に採点し、タイムアウトのない待機や緩すぎる条件の待機を指摘します。これ
は `doctor` の規約スコア（[BE-0024](../../proposals/BE-0024-doctor-onboarding/BE-0024-doctor-onboarding-ja.md)）
を、「id は適切に命名されているか」から「このスイートは再現可能か」へと自然に拡張するものです。

**境界に関する注記（重要）。** 本項目はロードマップの「取り込まない → 失敗テストの自動リトライ」の
線に触れるため、区別を明示しておきます。あの項目が不採用なのは、通過するまでのリトライがフレーキ
ネスを*隠す*からです。この監査はその**逆**です。繰り返し実行は、非決定性を*検出して報告するための
診断*です。ばらついた結果は修正すべき所見であり、赤を緑に変えるリトライではありません。監査は判定を
決して変えず、run／CI ゲートに決して供給しません。したがって、決定性優先を損なうのではなく強化します。

## 詳細設計

提案粒度です。設計上の制約は、監査が純粋に観測的であることです。

- **繰り返して差分を取る（動的）。** `--audit` モード（または `bajutsu audit` サブコマンド）が、
  同一の前提条件下でシナリオを `K` 回実行し、結果を比較します。ステップごとの pass/fail、解決された
  セレクタの対象、アサーション結果です。run をまたいでばらつくものは非決定的として、ばらついた証跡
  （要素ツリー／スクリーンショット／ネットワーク）を添えて報告します。分類は `deterministic` か
  `flaky` かであり、緩めた通過には決してなりません。既存の並列実行と証跡の機構を再利用し、固定
  sleep は導入しません。
- **安定性スコア（静的）。** シナリオをパースし、デバイスなしで採点します。各セレクタを安定性の
  はしご（[selectors.md](../../../docs/ja/selectors.md)）で採点し、`timeout` のない `wait` ステップ
  や広すぎる条件でゲートされた待機を指摘し、安定した `id` で置き換えられる生座標ジェスチャを指摘
  します。`doctor` の id カバレッジスコア（`bajutsu/doctor.py`）と並ぶ、シナリオごとの決定性スコア
  を出力します。
- **出力。** 助言的なレポート（HTML ／ JSON）と参考情報としての終了ステータス。シナリオと判定に
  対して読み取り専用です。テストを編集することも、CI をゲートすることもありません。

新しい `audit` コマンドとして実装するか、`doctor` ／ `run` のフラグとして実装するかは、採用時に
先送りする実装上の選択です。静的な部分は `doctor` の近縁であり、単にそれを拡張する形でもよいでしょう。

## 検討した代替案

* **フレーキーテストの隔離 + 自動リトライを採用する（業界の一般的な答え／Maestro の `retry`）。**
  ゲートに対しては不採用です。フレーキネスを隠すからで、それはまさに Bajutsu が対置して売る性質です。
  監査はそれを意図的に反転させ、フレークを暴いて修正させます。
* **run が「フレーキーに見えるか」を LLM に判定させる。** 不採用。非決定的で不要です。同一の run を
  またいだばらつきは、精密に機械検出できる事実です。
* **何もしない（現状）。** 許容できますが、最も強い差別化が暗黙のままで、導入候補者に対して証明
  できません。証跡と並列実行の機構はすでに存在するので、実行可能な証明書を出す費用は小さいです。
