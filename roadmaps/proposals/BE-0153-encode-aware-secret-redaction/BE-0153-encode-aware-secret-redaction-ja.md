## はじめに

`Redactor` は既知の秘密情報の値を、完全一致の文字列置換によってマスクしています。そのため、
URL エンコード、Basic 認証の base64、HTML・JSON エスケープ、あるいは複数チャンクに分割された
値など、同じ値のエンコードされた別形式はマスクされずに素通りします。本項目は redaction を
エンコード考慮型にし、既知の秘密情報の値をそのバイト列だけでなく、一般的なエンコード形式でも
照合できるようにします。

## 動機

`Redactor.redact_text`（`bajutsu/redaction.py:63`）は、既知の秘密情報の値を
`text.replace(value, PLACEHOLDER)` という単純な部分文字列置換でマスクしています。テキスト中に
値がそのまま完全一致で現れる場合は捕捉できますが、それ以外は一切捕捉できません。同じ秘密情報の
よくある変換形式のいくつかがすり抜けます。

1. **パーセントエンコーディング。** URL のクエリパラメータやフォームエンコードされたボディとして
   渡された秘密情報（`token=p%40ssw0rd`）は、`p@ssw0rd` という値をそのまま含んでいません。
   エンコードされた形式は別の文字列であり、完全一致では決してマッチしません。
2. **`Basic base64(user:pass)`。** HTTP の Basic 認証は
   `Authorization: Basic <base64(username:password)>` を送信します。パスワードの生の値はヘッダーのテキストに
   一切現れず、base64 でエンコードされコロンで結合された形式だけが現れるため、完全一致のパスは
   一切これに触れません。捕捉された `network.json` の exchange には、容易にデコードできる
   認証情報がそのまま残ります。
3. **HTML・JSON エスケープ。** HTML でエスケープされた属性（`&quot;`、`&amp;`）や JSON で
   エスケープされた文字列（`\"`、`\/`、`\uXXXX`）に埋め込まれた秘密情報は、生の値とバイト単位で
   一致しないため、これもマッチしません。
4. **複数チャンクに分割された値。** 複数回の読み取りやチャンクから組み立てられた証跡
   （ストリーミングされたボディや、逐次的に構築されるテキストなど）では、秘密情報の値のバイト列が
   チャンクの境界をまたいでしまい、置換処理が一つながりの文字列として認識できないことがあります。

これらはいずれも、`${secrets.*}`（BE-0032）でバインドされた本物の秘密情報（パスワード、トークン、
API キー）が、エンコードの形式が違うだけでほぼそのままの形で証跡に残ってしまう現実的な経路です。
その証跡はディスクに書き出され、AI による authoring・調査の経路では設定済みの AI プロバイダーにも
送られます（BE-0047）。`secrets:` を正しくバインドし、値が証跡から除かれることを当然期待している
著者は、たまたまツールが文字列一致で捕捉できるエンコード形式に対してしか保護されていません。

## 詳細設計

1. **既知の各秘密情報の値について、リテラルな文字列だけでなく一般的なエンコード形式も照合する。**
   `Redactor.redact_text` と `redact_exchange` の両方で以下を扱います。
   - パーセントエンコーディング（値を `urllib.parse.quote` した形式。エンコーダーによって
     エスケープしない安全文字の扱いが異なるため、いくつかの一般的な設定でチェックします）。
   - Basic 認証の base64：Basic 認証のパスワードになりうる各秘密情報の値について（両方が
     既知の秘密情報である場合は `user:pass` の全体についても）、`Authorization: Basic ` の
     後に現れる `base64(...)` 形式もマスクします。
   - すでにリテラルなテキスト検索を行っている箇所すべてで、値の HTML エスケープ形式・JSON
     エスケープ形式（`html.escape`、引用符を除いた `json.dumps`）も対象にします。
   - これはあくまで*既知の*秘密情報の値に適用する、固定で列挙可能な変換の集合にとどめます
     （あらゆるものを総当たりでデコードして走査するわけではありません）。これによりコストと
     誤検知の範囲を抑えます。
2. **キーに基づくパターンマッチングは弱めない。** ヘッダー・フィールドのキーに基づくパターン
   （`_patterns`、`bajutsu/redaction.py:21`）はそのまま維持します。本項目が変更するのは、
   キーや Basic 認証の文脈がおおよその位置を特定したあと、*値*そのものをどう認識するかだけです。
3. **チャンク境界での分割対策はベストエフォートであり、保証ではないと認識する。** redaction が
   実行される前に証跡が組み立てられている場合（ライブストリームではなく完全なボディ文字列である
   場合など）は、redaction はすでに完全なテキストを見ているため問題になりません。実際に断片化した
   証跡が存在する場合は、完全なカバレッジを主張せず、その制約を明記します。
4. **テスト。** URL・クエリでパーセントエンコードされた秘密情報の値、`Basic base64(...)`
   ヘッダーに含まれる値、ボディ中で HTML・JSON エスケープされた値が、それぞれ正しく
   マスクされることを検証します。
5. **ドキュメント。** エンコードを考慮したマッチングと、その既知の限界（実際に断片化した
   ストリーミング証跡など）について、redaction に関するドキュメント（`docs/` と `docs/ja/`
   の両方）に記載します。

「そもそもどのヘッダーを redact 対象にするか」という既定マスキングの欠落は、別の兄弟項目
（default network secret redaction）で扱います。本項目のスコープは、どのヘッダー・フィールドが
検索のきっかけになったかにかかわらず、既知の秘密情報の*値*をどう照合するかに限られます。
本項目は決定的な `run`・CI ゲートには一切触れず、LLM も導入しません。証跡書き込み経路の内部で
完結する、純粋なテキスト変換の変更です。

## 検討した代替案

- **証跡中に見つかったすべての文字列をデコードし、秘密情報の値と比較する。** 却下しました。
  範囲が広すぎ、遅すぎます。任意のテキストをあらゆるエンコードに対してデコードして比較するのは
  高コストであり、たまたま何かにデコードできてしまうゴミデータによる誤検知も生みます。
  逆方向（秘密情報の値をエンコードして、その結果を検索する）で既知の値の既知のエンコード形式を
  照合するほうが、安価で精度も高くなります。
- **著者にはヘッダー・フィールド名のみによる redaction を求め、値に基づくマッチングを
  完全に廃止する。** 却下しました。値に基づくマッチング（BE-0032 の `values` リスト）は、
  著者が想定していなかった場所（ログの一行、レスポンスボディなど）にアプリが秘密情報を
  エコーしてしまうケースを捕捉するために存在しており、これを取り除くとエンコードの欠落を
  修正するどころか、その保護自体を後退させてしまいます。
- **マッチャーを修正する代わりに、AI data sovereignty の警告（screenshot-secret-capture-warning）に
  任せる。** 却下しました。その項目は画像の証跡を対象としており、画像は本当にピクセル単位で
  redact できません。一方でテキストの証跡は正しく照合できるため、修正はマッチャー側で
  行うべきであり、開示に頼るべきではありません。
