# BE-0036 — ユーティリティステップ

* 提案: [BE-0036](BE-0036-utility-steps-ja.md)
* 状態: **提案**
* トラック: [提案](../../README-ja.md#提案)
* トピック: 競合調査（MagicPod / Autify）由来の候補
* 由来: MagicPod

## はじめに

HTTP リクエスト発行 / OTP（ワンタイムパスワード）・2FA（二要素認証）コード生成 / メール受信を API で検証。実アプリのログインフロー自動化に必要。

## 動機

実アプリのログインフローは、パスワードを入力するだけで済むことはまれです。多くの場合、UI 自身
では供給できない側方チャネル（side channel）が必要です。認証アプリからのワンタイムパスワード
（OTP）、メールや SMS で届く二要素認証（2FA）コード、あるいはバックエンドへの HTTP 呼び出しで
得るトークンなどです。これらの値を run に取り込む手段がなければ、最も重要なフロー（サインイン、
アカウント検証、パスワードリセット）を決定的に自動化できません。著者は、テストビルドで 2FA を
無効化する（すると実際の経路を検証しなくなります）か、値を手で貼り付ける（再現できません）か、
どちらかを強いられます。ユーティリティステップは、シナリオが側方チャネルの値を取得し、後続の
ステップへランタイム変数として渡せるようにして、この欠落を埋めます。

## 詳細設計

既存の `${vars.*}` 名前空間（[scenarios](../../../docs/ja/scenarios.md#ランタイム変数-vars)）へ値を
生成する小さなステップ群を追加します。これにより、後続の `type` / `assert` がその値を決定的に
消費できます。

```yaml
- http: { method: GET, url: "${secrets.MAIL_API}/latest", into: { var: code, jsonPath: "$.otp" } }
- type: { text: "${vars.code}", into: { id: auth.otp } }
```

```yaml
- totp: { secret: "${secrets.TOTP_SEED}", into: { var: code } }   # RFC 6238 の時刻ベース OTP
- type: { text: "${vars.code}", into: { id: auth.code } }
```

```yaml
- email: { match: { to: "test@example.com", subjectMatches: "code" }, extract: { var: code, bodyMatches: "[0-9]{6}" }, timeout: 30 }
```

マッピングと契約：

- **`http`** はランナープロセスからリクエストを発行し、フィールド（本文に対する JSON パスまたは
  正規表現）を `vars` へ抽出します。これは*取得ユーティリティ*であり、アプリ自身のトラフィックを
  観測するアプリ内の `request` アサーション／`mocks` とは別物です。
- **`totp`** はシード（種）から時刻ベースのワンタイムパスワードをローカルで計算します（RFC 6238）。
  純粋な計算で、シードとクロックが与えられれば完全に決定的、ネットワーク不要です。
- **`email`** は、一致するメッセージが届くまでテスト用メールボックス API をポーリングし、コードを
  抽出します。この待機は **`timeout` 必須の条件待機**です（他のすべての待機と同じルールで、固定
  sleep も無限ポーリングもありません）。タイムアウトはステップ失敗です。

シークレット（API のベース URL、TOTP シード、メールボックスの認証情報）は `${secrets.X}` として
参照し、config で宣言します。これによりシナリオファイルはトークンを保持し値は保持せず、証跡では
自動的にマスクされます。複数シナリオで共有するエンドポイント設定は `apps.<name>` に置きます。

prime directive の保持：

- **run パス上に LLM を置かない。** 各ユーティリティステップは決定的な計算、または決定的な
  リクエスト／一致です。`totp` は算術、`http`／`email` は JSON パスや正規表現による抽出です。
  合否判定は引き続き機械チェック可能なアサーションのみから得られ、run／CI ゲートは AI 非依存の
  ままです。
- **決定性。** `email` は上限のある timeout 付きの条件で待ち、sleep は使いません。`http` と
  `totp` は即時です。外部メールボックスが不安定な場合は、黙った誤った値ではなく、きれいなステップ
  失敗として表面化します。
- **アプリ非依存。** ステップとその配線はアプリをまたいで同一で、異なるのはエンドポイントと
  シークレット（config／`${secrets.*}`）だけです。
- **codegen。** これらはアプリではなく bajutsu ランナーで動き、XCUITest 等価物がないため、codegen
  はラベル付き `// TODO` を出力します
  （[BE-0026](../../proposals/BE-0026-shrink-unsupported-syntax/BE-0026-shrink-unsupported-syntax-ja.md) に
  従います）。

## 検討した代替案

- **テストビルドで 2FA／OTP を無効化し、側方チャネルを丸ごと省く。** 最も単純で、スモークテスト
  には十分ですが、実際の認証経路を検証しなくなります。それはまさに最もカバーする価値のある
  フローです。唯一の選択肢としては却下します。ビルドがテスト用バイパスを提供することはあって
  よいですが、ツールがそれを要求すべきではありません。
- **`mocks` で OTP／メールプロバイダをモックする。** `mocks` はアプリ*自身*の送信トラフィックを
  スタブするので、コードがアプリの発行・観測するリクエストで届く場合には機能します。値が帯域外の
  チャネル（認証アプリ、実際の受信箱）から来る場合には役立たず、それが一般的な 2FA のケースです。
  置き換えではなく補完として維持します。
- **任意のコマンドを実行して値を生成する汎用 `shell`／`exec` ステップ。** 柔軟性は最大ですが、
  シナリオを非可搬かつレビュー不能にし（何でも実行できる）、テストを CI ホストへ結びつけます。
  少数の目的特化で監査可能なステップを優先し、却下します。
