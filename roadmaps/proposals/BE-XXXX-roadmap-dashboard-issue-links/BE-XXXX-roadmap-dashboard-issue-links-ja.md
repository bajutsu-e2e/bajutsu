## はじめに

[ロードマップ状況ダッシュボード](https://bajutsu-e2e.github.io/bajutsu/api/roadmap.html)
（BE-0094）の各カードに、もう一本小さなリンクを追加します。リンク先は、その項目のトラッキング
Issue（BE-0109）です。担当者（Assignees）や議論が実際に存在する場所です。このリンクは、特定の
Issue 番号をそのつど取得するのではなく、項目の ID から機械的に組み立てる GitHub の検索 URL です。
そのためダッシュボードのビルドは、今のとおりネットワーク通信も GitHub トークンも必要としません。

## 動機

BE-0094 はすでに各カードから 1 本のリンクを出しています。GitHub 上にある、項目の Markdown 版の
提案本体へのリンクです。一方 BE-0109 は、`状態` が `提案` または `実装中` であった（あるいは今も
そうである）項目ごとに `[BE-NNNN] タイトル` という形式の Issue を起票し、その担当者を「すでに
誰かが取り組んでいるか」の一次情報として扱います。ところがダッシュボードのどのカードも、この
トラッキング Issue へは繋がっていません。読み手は一度ダッシュボードを離れ、リポジトリの Issues
タブを開き、`roadmap-tracking` ラベルと `in:title BE-NNNN` という検索の慣習をあらかじめ知って
いるか、その場で組み立てるほかありません。

この隙間が問題になるのは、読み手が続けて抱く二つの問い、「この項目は何か」（カード、提案本体）と
「すでに誰かが取り組んでいるか」（Issue、その担当者）のうち、今は前者にしか一クリックで答えられ
ないからです。二つ目の問いへの導線を足せば、ダッシュボードは両方の出発点になります。新しい一次
情報を増やすわけではありません。カードは引き続き提案本体へリンクし、追加するリンクは GitHub が
すでに管理している状態へ経路を通すだけです。

## 詳細設計

1. **機械的に組み立てる検索 URL であり、Issue のそのつどの取得ではありません。** BE-0109 が
   起票する Issue のタイトルは常に `[BE-NNNN] 項目のタイトル` で、常に `roadmap-tracking` ラベル
   を持ちます。したがって項目の ID だけから、その Issue 番号を知らなくても見つけられる GitHub の
   検索 URL を組み立てられます。
   `https://github.com/bajutsu-e2e/bajutsu/issues?q=is%3Aissue+label%3Aroadmap-tracking+in%3Atitle+"BE-NNNN"`
   このクエリには `is:open` を含めないので、Issue がまだオープンでも、後でクローズされていても
   見つかります（`提案` や `実装中` を経て出荷された項目は、クローズ済みの Issue を持ち続けます）。
   URL の組み立てに必要なのは ID の文字列だけで、`gh` の呼び出しも、GitHub トークンも、ビルド時の
   ネットワーク通信も要りません。
2. **変更するのはカードだけです。** `scripts/build_roadmap_dashboard.py` の `_card()` に、既存の
   提案本体へのリンクと並べて、もう一本小さなリンク（たとえば「Issue」と書いた小さなピル状の
   リンク）を追加し、上記の検索 URL へつなぎます。カードの主たるクリック対象である提案本体への
   リンクは変えません。新しいリンクは追加するだけです。
3. **BE-0109 の同期処理やラベル、ワークフローは変更しません。** 検索 URL は、ID と、BE-0109 が
   すでに保証しているタイトル・ラベルの慣習だけから導けるので、GitHub から何かを読み戻す必要が
   ありません。`sync_roadmap_tracking_issues.py` と `roadmap-tracking-issues.yml` はそのままです。
4. **このリンクは、検索結果が 0 件になり得ます。** BE-0109 が Issue を起票するのは、同期処理が
   走った時点でその項目が `提案` または `実装中` だった場合だけです。追加された時点ですでに
   `実装済み` として出荷された項目、いわば生まれながらに実装済みの項目は、オープンな状態を一度も
   経ないので Issue を持たず、検索リンクは 0 件になります。ダッシュボードはこれを事前に知る手立て
   を持ちません。GitHub に問い合わせれば分かりますが、それは 1. で避けたネットワーク依存を呼び
   戻すことになります。そこでこのリンクは「Issue そのもの」ではなく「Issue の検索」だと明示し、
   0 件のページに着地した読み手が理由を理解できるようにします。
5. **ドキュメント。** 本項目のファイルに加えて、BE-0109 の `参考` 節へ短い相互参照を追加します。
   BE-0109 の検討した代替案にある「ダッシュボード（BE-0094）は、読み取り専用の `gh issue list`
   から項目と Issue のリンクを描画できる」という記述は、本項目が採る設計（後述のとおり
   `gh issue list` は使いません）に置き換わるものとして扱います。

## 検討した代替案

- **ダッシュボードのビルド時に `gh issue list` でそのつど Issue を取得し、実際の Issue 番号を
  解決する方法。** これは BE-0109 の `参考` 節がすでに見込んでいた方向です（「ダッシュボード
  （BE-0094）は、読み取り専用の `gh issue list` から項目と Issue のリンクを描画できる」）。
  退けました。今の `make docs` と `make docs-serve` は、ダッシュボードを stdlib だけで再生成して
  おり、ネットワーク通信も `gh auth` も要りません。`docs` ワークフローに至っては、`issues` 権限を
  一切持たずにビルドしています。そのつどの取得に切り替えると、ローカルでの `gh` セッションと、CI
  側の新しい `issues: read` 権限の両方が必要になり、「コミットされたメタデータだけから導く」と
  いう BE-0094 の前提そのものに、ビルド時の外部状態への依存を持ち込むことになります。検索 URL
  なら、どちらの依存も増やさずに同じ行き先へ導けます。
- **Issue 番号を項目のメタデータブロックに書き戻す方法（たとえば「トラッキング Issue」の行を
  追加する）。** これは BE-0109 自身がすでに退けています。どの Issue が開いているか、誰が担当し
  ているかという、項目のメタデータの他の部分よりずっと頻繁に変わる状態のために、保護された
  `main` への、bypass 用 App を介したコミットを強いることになり、「GitHub を唯一の一次情報にする」
  という BE-0109 の設計を崩すからです。本項目はこの論点を蒸し返しません。描画時に導出するリンク
  を足すだけなので、書き戻しは要りません。
- **提案本体へのリンクを Issue へのリンクに置き換える方法。** 退けました。読み手がまず欲しいのは
  実際の仕様である提案本体であり、Issue は担当状況と議論への薄い副次的な導線にすぎません。両方を
  別々のリンクとして残すことで、カードの主たるクリック対象を変えずに済みます。
