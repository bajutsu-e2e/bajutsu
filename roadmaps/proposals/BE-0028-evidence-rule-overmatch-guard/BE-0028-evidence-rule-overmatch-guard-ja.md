# BE-0028 — 証跡ルールの過剰マッチ対策

* 提案: [BE-0028](BE-0028-evidence-rule-overmatch-guard-ja.md)
* 状態: **提案**
* トラック: [提案](../../README-ja.md#提案)
* トピック: その他・保留

## はじめに

capturePolicy の過剰マッチによって成果物が肥大するのを防ぎます（`--explain` ドライランおよび既定ポリシーの軽量化）。

## 動機

`capturePolicy` のルールは、そのトリガーがマッチする「たびに」発火します（DESIGN §9 A、`docs/evidence.md`）。それこそが狙い——反復可能で決定的な証跡——ですが、同時に、広すぎるルールを、run が終わるまで見えない形で高コストにもします。`{ action: tap, idMatches: "*.submit" }` のようなトリガーは「送信ボタン」と読めますが、緩い glob や、忙しいフローでの `event: screenChanged` ルールは、著者の意図よりはるかに多くのステップにマッチしえます。そうしたルールに重い取得（`video`、`network`）を付ければ、run は静かに数ギガバイトの成果物を生み、遅くなり、§9 が警告する観測者効果のリスクを招きます。

DESIGN §11 はこれを未解決のリスクとして直接挙げています——「証跡ルールの過剰マッチによる成果物肥大 → `--explain` ドライランと既定ポリシーの軽量化」。DESIGN §9 のコスト管理の注記も両面を補強します。ルールはマッチ条件を絞るべきで、既定は `screenshot` + `elements` + `actionLog` の軽量 3 点とし、`video` / `network` はオプトインにすべきです。今日の問題は、過剰マッチを run の*前に*可視化するものが何もなく、重い取得が静かに実効的な既定になるのを止めるものが何もないことです。本提案は両方に対処します——発火をプレビューする手段と、明示しない限り安価な既定です。

## 詳細設計

互いを補う 2 つのガードで、どちらも合否には触れません。

- **run 前の `--explain` ドライラン。** あるコマンド（§11 が名付けるとおり例えば `bajutsu trace --explain <scenario>`）が、シナリオを `capturePolicy` に対して静的に辿り、各ルールが何回・どのステップで発火するかを報告し、広くマッチするルールに付いた重い取得（`video` / `network`）に印を付けます。これは既存の事後的 `trace` の*run 前*の対応物です。あちらは完了した run の manifest を読みますが、こちらは run が*何を取得するか*を事前に示すので、著者はコストを払う前に glob を絞れます。read-only かつ決定的で——run ループと同じ仕方でセレクタを解決しトリガーをマッチさせるので、その回数は実際の回数であり、LLM は関与しません。
- **軽量な既定ポリシー。** 暗黙の既定取得は軽量 3 点——`screenshot` + `elements` + `actionLog`——のままで、加えて、ステップが実際に失敗したときだけ最大限を取得する `result: error` の安全網を保ちます。区間の重い種別（`video`、`deviceLog`、`network`）は厳密にオプトインのままで、インラインの `capture:` か狭くスコープしたルールで意図的に付けます。こうして高コストの取得は著者が求めた所でのみ走り、一般的なケースは構造上安価です。
- **決定性と app-agnostic を保つ。** どちらのガードも run のアサーションを変えません。`--explain` は助言的な出力で、既定ポリシーの選択はどの証跡を集めるかに影響するだけで、合否には決して影響しません。固定 sleep はなく、セレクタの厳格さも不変です。アプリ単位の取得の好みは、他の per-app 設定と並べて `apps.<name>` の下に置くので、ツールと runner は app-agnostic のままで、Tier-2 ゲートは LLM フリーのままです。

## 検討した代替案

- **成果物のサイズや件数にハードな上限を設ける。** バイト/ファイルの予算を超えたら run を中断または切り詰めるのは被害を抑えますが、証跡を非決定的に捨てます（同じシナリオがタイミング次第で異なる成果物を取得しうる）——これは決定性第一に反します。発火回数をプレビューし、安価な取得を既定にする方が、肥大に反応するのではなく未然に防ぎます。
- **ルールが「頻発しすぎ」たら run 時に警告する。** run 時のヒューリスティックは反応的で——しきい値が引っかかる頃には成果物は既に書かれています——「頻発しすぎ」という魔法の数値はシナリオ間で恣意的です。run 前のドライランは、何のコストも払う前に著者へ正確な回数を渡し、より早くより正確です。
- **すべての取得をオプトインにする（既定の証跡をなくす）。** 最大限に安価ですが、DESIGN §10 が要求する最低限の安全網——失敗時は必ずスクショと要素ダンプを残す——を取り除きます。軽量 3 点と `result: error` の網を既定として保つことで、その保証を維持しつつ重い種別をオプトインに保てます。
