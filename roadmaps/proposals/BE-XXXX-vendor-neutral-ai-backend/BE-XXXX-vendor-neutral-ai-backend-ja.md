## はじめに

bajutsu の**バックエンド非依存**の思想、すなわち「プラットフォームは単一のインターフェースの背後に置かれた
バックエンドである」という考え方を、AI の経路にも適用します。言い換えると「**AI プロバイダも単一の
インターフェースの背後に置かれたバックエンドである**」とします。bajutsu がモデルに実際に求めているもの
（ツールを使う対話ターン、画像入力、単純な補完）だけを記述する**ベンダー中立なシーム**を導入し、
Claude を使う機能を、単に別の Claude エンドポイントではなく**別のモデルファミリ**へ、呼び出し側に
手を入れることなく差し替えられるようにします。

本項目の範囲は、**設計と参照アダプタ 1 つ（既存の Anthropic 経路）まで**です。2 つ目のベンダーは
出しません。別のベンダー（OpenAI 互換エンドポイント、Google Gemini、ローカルで動く OSS モデル）を
足すことは、本項目が定義するシームに差し込む後続項目に意図的に委ねます。ここでの目的は「AI を差し替える」
ことを*可能かつ安全*にし、そのシームをすでに使っているプロバイダで実証することです。

本項目は Prime directive に抵触しません。シームは Tier-1 のオーサリングと調査の経路にのみ存在し、
決定論的な `run` / CI ゲートには決して置きません。

## 動機

[BE-0047](../../implemented/BE-0047-ai-data-sovereignty/BE-0047-ai-data-sovereignty-ja.md) と
[BE-0053](../../implemented/BE-0053-bedrock-ai-provider/BE-0053-bedrock-ai-provider-ja.md) によって、
*プロバイダ*はすでに差し替え可能です。AI を使うすべての経路は `anthropic_client.py` の単一のファクトリを
通してモデルに到達し、設定によって Anthropic API、Amazon Bedrock、自前のゲートウェイのいずれかを
選べます。しかしそのファクトリが返すのは `Anthropic | AnthropicBedrock` であり、**どちらも Anthropic の
SDK クライアント**です。そして呼び出し側（`claude_agent.py`、`claude_triage.py`、`alerts.py`、
`claude_enrich_agent.py`、`crawl_guide.py`、`crawl_tabs.py`）は Anthropic の Messages API を直接
話しています。`tool_use` / `tool_result` のコンテンツブロック、画像入力のための Anthropic 固有の
ブロック形状、system プロンプトの流儀です。したがって現在の「差し替え」は*どの Claude エンドポイントか*
であって、*どのモデルファミリか*ではありません。

より深いシームを求める理由は、特定の行き先に結び付いているわけではありません。価値は、複数の軸で同時に
選択肢を開いたままにしておけることそのものにあります。

1. **ベンダーロックインの回避**：抽象化の見返りは、モデルを差し替えられること自体が機能である、という
   点にあります。すべての AI 呼び出し箇所に入り込む単一ベンダーへの結合は、そのベンダーがどれほど
   優れていても負債です。アーキテクチャが一つのベンダーを永久に前提にすべきではありません。
2. **オンプレミスとデータ主権**：BE-0047 は「あなたの AI、あなたのキー、あなたのデータ」を強制される
   約束にしましたが、それは Anthropic 形状のエンドポイントの範囲にとどまります。スクリーンショットや
   要素ツリーを環境の外へ一切出さないために、**ローカルまたは自前の OSS モデル**（Ollama や vLLM 経由
   など）を動かす必要がある利用者がいます。これは現在のシームでは到達できないモデルファミリです。
3. **コストと軽量モデル**：経路ごとに要求はまったく異なります（素早いアラート消去のロケータと、複数
   ターンにわたる record エージェントを比べてください）。中立なシームがあれば、運用者は一つのベンダーの
   ラインナップに固定されず、経路ごと・ターゲットごとに、より安価あるいは小型のモデルへ振り分けられます。
4. **特定ベンダーの能力**：あるモデルが、特定の経路について画像認識の精度・レイテンシ・ツール呼び出しの
   挙動で優れていることがあります。それを（その経路に限って）選べることは、2 つ目のアダプタを出す前から
   抽象化に見合う価値があります。

この 4 つをいずれも今日阻んでいるのは同じ事情です。中立な形状が存在しないことです。呼び出し側は
Anthropic の型に対して書かれているため、「OpenAI を足す」「ローカルモデルを足す」はアダプタを登録する
ことではなく、AI のすべての経路を書き換えることを意味します。本項目はその形状を作り、Anthropic で実証
します。これにより、上記の 4 つの動機は横断的な書き換えではなく、後続のアダプタとして扱えるようになります。

構造上のもう一つの見返りは、**リダクションがシームの性質になる**ことです。現在は
[BE-0047](../../implemented/BE-0047-ai-data-sovereignty/BE-0047-ai-data-sovereignty-ja.md) /
[BE-0097](../../implemented/BE-0097-crawl-ai-data-sovereignty/BE-0097-crawl-ai-data-sovereignty-ja.md)
が、入力を Anthropic クライアントに渡す前に `redaction.py` を通しています。すべてのプロバイダが単一の
中立インターフェースを必ず通るなら、そのインターフェースがデータ主権の保証を強制する唯一の関門になります。
その結果、将来のどのアダプタも構造上その保証を受け継ぎ、ベンダー統合ごとにリダクションを忘れないよう
気を配る必要がなくなります。

## 詳細設計

提案レベルの粒度です。このシームは汎用の LLM フレームワークではなく、薄い能力層です。bajutsu の既存の AI
経路がすでに使っているものだけを記述します。今日あるシーム（`anthropic_client.make_client` /
`resolve_model` / `credential_gap`、経路ごとの `MODEL` 定数、`AiConfig` / `AiSettings` の設定ブロック）の
上に組み立てます。作業は以下の 7 つの単位に沿って MECE に分割します。

### 1. 能力の棚卸し（最小の中立サーフェス）

各 AI 経路がモデルに求めるものを、それ以上でも以下でもなく列挙し、インターフェースを狭く保ちます。

- **ツールを使う対話ターン**：`record`（`claude_agent.py`）、`triage --ai`（`claude_triage.py`）、
  MCP の enrich（`claude_enrich_agent.py`）、crawl のガイド（`crawl_guide.py`）は複数ターンのループを
  回します。メッセージとツール定義を送り、テキストまたはツール使用の要求を受け取り、ツールの結果を
  返します。
- **画像入力**：`run --dismiss-alerts`（`alerts.py`）、`triage --ai`、`record` はスクリーンショットを
  画像コンテンツとして添付します。
- **単純または短い構造化補完**：`crawl_tabs.py`（タブの識別）は、実質的に単発の分類です。

この棚卸しは、中立インターフェースが表現しなければならない能力の確定的な一覧を生みます。同じくらい
重要なのは、*使われていない*ため中立インターフェースが表現する必要のない Anthropic 固有の挙動を、
一覧として切り分けることです。

### 2. ベンダー中立なインターフェース

棚卸しのサーフェスをちょうど覆う、小さなプロバイダ中立のインターフェース（`Protocol`）と、正規化した
リクエスト / レスポンスの型を定義します。

- **リクエスト**：system プロンプト、正規化したメッセージの列（テキスト、画像、ツール結果の各要素）、
  ツール定義（名前 / 説明 / 入力スキーマ）、モデル ID、`max_tokens`。
- **レスポンス**：正規化したコンテンツブロック（テキストとツール使用要求（名前と引数））と、ツール
  ループを駆動するのに十分な停止理由。
- **メソッド**：ループが反復できる `create_message(...)` 相当の呼び出しを 1 つ。ストリーミングは持ち
  ません（現状どの経路も必要としません）。サーフェスは最小のままにします。

これらの型は bajutsu 自身のものであり、Anthropic の SDK 型の再エクスポートには意図的にしません。どの呼び出し
側もベンダーの形状に依存しないようにするためです。

### 3. プロバイダレジストリ（拡張点）

現在の 2 値の `provider`（`anthropic` / `bedrock`）を、**プロバイダ名をキーとし、アダプタのファクトリを
値とするレジストリ**へ一般化します。プロバイダの追加は、ファクトリの `if` 連鎖を編集することではなく、
「アダプタを登録する」ことになります。本項目はレジストリを出し、新しいアダプタが満たすべき契約
（第 2 項のインターフェースを実装すること、自身の `credential_gap` を宣言すること）を文書化します。
2 つ目のベンダーは登録**しません**。

### 4. Anthropic 参照アダプタ

既存の挙動を、第 2 項のインターフェースの背後にある最初のアダプタとして再実装します。今日の
`make_client` をラップします（そのため Anthropic API と Amazon Bedrock の**両方**が引き続き覆われます。
Bedrock は Anthropic SDK の一変種であり、自然にこの 1 つのアダプタの中に収まります）。このアダプタは
中立リクエストを Anthropic Messages API の呼び出しへ翻訳し、Anthropic のレスポンスを中立コンテンツ
ブロックへ翻訳して返します。挙動は変わりません。BE-0047 / BE-0053 のモデル ID と `credential_gap` の
意味も保たれます。

### 5. 呼び出し側の移行

各 AI 経路（`claude_agent.py`、`claude_triage.py`、`alerts.py`、`claude_enrich_agent.py`、
`crawl_guide.py`、`crawl_tabs.py`）を、`anthropic` クライアントとそのメッセージ / ツール / 画像の型では
なく、中立インターフェースを構築して使うよう書き換えます。これは挙動を保つリファクタリングであり
（各経路の観測可能な出力は変わりません）、ベンダーへの結合を実際に取り除く工程です。

### 6. 設定スキーマ（開放型のプロバイダ）

`AiSettings` / `AiConfig` はそのまま保ちますが、`provider` を閉じた `anthropic | bedrock` の対では
なく、**レジストリで検証される開放型**の値にします。未知のプロバイダは明確なエラーで fail closed に
します（BE-0047 が確立したのと同じ fail closed の規律です）。`provider` 名が登録済みアダプタへどう
対応づくかを文書化します。本項目のために新しい設定*フィールド*は必要ありません。

### 7. 契約テスト

**プロセス内のフェイクアダプタ**を使って中立インターフェースの契約テストを追加します（ネットワークも
デバイスも不要で、高速な Linux ゲートで走ります）。ツール使用ループ、画像入力、補完の各経路がインター
フェースを正しく駆動すること、そしてどのアダプタに到達する前にもリダクションが走ることを検証します。
Anthropic アダプタが今日と同じリクエストを生成し続けること（第 5 項の挙動不変の保証）も検証します。

### Prime directive との整合

このシームを使うのは、すでにモデルを呼んでいる Tier-1 のオーサリングと調査の経路**だけ**です。それらの
呼び出しに名前を与えて置き直すのであって、呼び出しを新たに増やすことはありません。`run` と CI ゲートは
モデルを持たないまま（決定論的で、機械のみが合否を決める）で、中立インターフェースはその経路では決して
import されません（[BE-0101](../../implemented/BE-0101-ai-free-zero-config/BE-0101-ai-free-zero-config-ja.md) の
Claude を使わない経路のゼロ設定保証と整合します）。プロバイダの選択は設定（`targets.<name>.ai`）の
ままであり、ツール / ドライバ / ランナーはアプリ非依存を保ちます。決定性にも手を触れません。どの経路も
`sleep` や曖昧なセレクタを持つことはありません。

## 検討した代替案

- **サードパーティの LLM 抽象（LiteLLM、LangChain など）を採用する**：不採用としました。依存と挙動の
  面積が大きく、そのルーティングは、bajutsu が保たねばならない決定性とリダクションの保証にとって不透明です。
  また bajutsu が必要とするのは狭い能力サーフェス（第 1 項の棚卸し）だけです。小さく自前で持つインターフェースの
  ほうが、汎用フレームワークよりも見通しがよく、リダクションの関門を正直に保ちやすいです。
- **`base_url` による Anthropic 互換ゲートウェイだけに頼る**：`base_url` の上書き（BE-0053 の頃）は、
  一部のプロキシを Anthropic API の前段に立てることをすでに可能にしています。不採用としました。これは
  Anthropic の Messages スキーマを話すエンドポイントにしか到達できず、メッセージ / ツール / 画像の形状が
  異なる OpenAI、Gemini、ほとんどのローカルランタイムには対応できません。これは「別のエンドポイント」を
  解決するのであって、実際の要望である「別のモデルファミリ」を解決しません。
- **OpenAI 互換のシムだけを支える**：多くのプロバイダやローカルランタイムが OpenAI 互換 API を公開して
  いるため、OpenAI 形状のアダプタ 1 つでかなりの範囲を覆えます。*インターフェース*の選択としては不採用と
  しました。中立なシームを OpenAI の形状に結合することは、一つのベンダーロックインを別のものに置き換える
  だけで、Anthropic や Gemini のネイティブな形状をきれいに表現できません。OpenAI 互換アダプタは、中立
  レジストリに差し込む*後続*としては妥当ですが、シームそのものはベンダー中立でなければなりません。
- **今すぐ 2 つ目のベンダーを出す（範囲を広げる）**：本項目としては選択の上で不採用としました。すでに
  使っているプロバイダでシームを実証することで、変更を挙動不変かつレビュー可能に保てます。そして実際の
  各ベンダーは、その後に独立してテストできる、焦点の絞られたアダプタとして着地させられます。
- **何もしない（Anthropic のみを保つ）**：不採用としました。4 つの動機（ロックイン、オンプレ、コスト、
  能力）をいずれも横断的な書き換えの向こうに閉じ込めたままになり、リダクションも単一のシームではなく
  呼び出し箇所ごとの強制のままになります。
