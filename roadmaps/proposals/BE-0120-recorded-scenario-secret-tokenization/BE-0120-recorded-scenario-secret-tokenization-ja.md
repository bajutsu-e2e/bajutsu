## はじめに

`bajutsu record` は、著者した scenario をそのまま書き出します。そのため探索中に authoring
agent が入力したシークレットは、scenario ファイルにリテラルな平文として残ってしまいます。本
提案は、記録されたシークレットをリテラル値のまま書く代わりに `${secrets.*}` へトークン化します。

## 動機

`record`（`bajutsu/cli/commands/record.py:203`）は、完成した scenario を
`out_path.write_text(dump_scenarios([scenario]), encoding="utf-8")` でそのまま書き出します。
これは record ループ（`bajutsu/record.py`）が積み上げた `Step` オブジェクトそのものであり、
途中に redaction や置換の処理は挟まりません。record ループは、エージェントが入力したリテラルな
テキストを保持する `type` ステップを追加します（`_describe_step`、`bajutsu/record.py:46` は
`step.type.text` を読みます）。この中には、ログインや API キーのフローを実演する際に、パスワード
欄やトークン欄へ入力した値もそのまま含まれます。`${secrets.*}`（BE-0032）は、*実行時*の著者に
対してはすでにトークンだけを持つ scenario を提供しており、実際の値は環境変数から解決され、
エビデンスからも redaction されます。しかし `record` はこの仕組みよりも上流に位置しています。
シークレットという概念が登場する前の段階で、人間または記録用エージェントが与えた生の入力を
そのまま捕捉してしまうのです。したがって、実際のログインに対して記録した scenario は、
パスワードや API キーを YAML に平文で書き込みます。この値はリポジトリへのアクセス権を持つ
誰からも参照でき、コミットすればコミット履歴にも残り、後にその scenario ファイルを含む
エビデンスや成果物へも redaction されないままコピーされます。

深刻度は Medium です。本レポートの他の項目（TOTP のシード漏えいやネットワークのシークレット
漏えい）とは異なり、これは実際の資格情報を扱う scenario にとって*主たる著作経路*にあたります。
`record` はそもそもほとんどの scenario が作られる方法であるため、この露出はまれな端例ではなく、
むしろ通常のケースです。

## 詳細設計

1. **recorder が捕捉した入力の中から、束縛済みのシークレット値を検出する。** `type`（または
   テキストを保持する他の）ステップを追加する前に、その入力のリテラル値をターゲットに宣言
   された `secrets:` の解決済みの値と比較します。これは `run` がすでに `${secrets.X}` に対して
   行っている環境変数の解決（`bajutsu/cli/commands/run.py:335-338`）と同じ仕組みで、record も
   同じ config を参照できるため、新たなシークレット宣言の仕組みを追加せずに同じ束縛を解決
   できます。
2. **一致した場合はリテラル値ではなくトークンを書く。** 捕捉した値が解決済みのシークレット
   束縛と一致した場合、その値を `scenario.steps` へ追加する前に `${secrets.X}` へ置き換えます。
   こうすることで、`record` が書き出すファイルは最初から BE-0032 が想定するトークン形式を
   持つことになり、完成した YAML に対する別個の後処理は不要になります。
3. **一致しない値には手を加えない。** 宣言されたどのシークレットにも一致しない入力値（タップ
   や、シークレットではない大半のテキスト入力）は、これまでどおり記録します。本提案が変更
   するのは `secrets:` に宣言済みの束縛に対応する入力に限られるため、通常のケースに新たな
   手作業は増えません。
4. **置換をユーザーに提示する。** `record` が既に持つ進捗の説明（`bajutsu/cli/commands/
   record.py` の `say` コールバック）で、あるステップのテキストがトークン化されたことを
   報告します。これにより、実行結果を確認する著者は、どのフィールドが黙って書き換えられた
   のではなく `${secrets.X}` へ置き換わったことを把握できます。

## 検討した代替案

- **書き出した YAML に対する事後の scrub パスとして、記録済みシークレットを redaction する。**
  却下しました。この対象には適さない形だからです。BE-0032 自身が持つ書き出し後の scrub
  （`_scrub_secret_values`）は、run がすでに実際の値を使ったあとで*結果*のテキスト
  （アサーションの actual・expected）へ漏れ出たシークレット値を捕まえるためのものです。一方で
  scenario ファイルの `type.text` は、*次回*の run でそのステップが何をするかを決める正本
  そのものです。事後に scrub すると、検出できなければリテラルがそのまま残るか、プレース
  ホルダーに置き換えると再生が壊れる（redaction された scenario はもう有効な資格情報を入力
  できない）かのどちらかになります。動作する `${secrets.X}` トークンとして書き換えない限り
  この問題は解決せず、それはまさに書き出し時点でのトークン化が直接行うことです。
- **記録した scenario を著者が手動で編集し、`${secrets.*}` に置き換えるよう求める。** 修正
  すべき現状そのものなので却下しました。記録した資格情報のたびに、すべての著者がこの編集を
  忘れずに行うことが前提になり、編集を一度でも見落とせば、現状とまったく同じように平文の
  シークレットがファイルに残ってしまいます。
- **捕捉した値が宣言済みのシークレットと一致するステップの記録自体を拒否し、著者にプレース
  ホルダーで撮り直しを強制する。** 却下しました。記録フロー自体が壊れてしまいます。エージェント
  は、実演しようとしている認証後の画面へ実際に到達するために本物の資格情報を必要とするから
  です。しかも、もともと必要だった値を透過的にトークン化するほうが、著者にとっては明らかに
  望ましい結果です。
