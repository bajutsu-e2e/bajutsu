# BE-XXXX — serve Web UI のレスポンシブ対応（小さい画面・タッチ操作）

* Proposal: [BE-XXXX](BE-XXXX-responsive-web-ui-ja.md)
* Author: [@0x0c](https://github.com/0x0c)
* Status: **Proposal**
* Track: [Proposals](../../README-ja.md#proposals)
* Topic: オーサリング体験（record / GUI エディタ）

## Introduction

`bajutsu serve` の Web UI（BE-0011）を、スマートフォンや幅の狭いタブレットといった小さい画面でも使えるようにする。デスクトップ前提の多ペインレイアウトを、一定の画面幅を境に 1 カラムのタッチ操作向けレイアウトへ組み替える。変更はインライン展開している `serve.*` テンプレートのフロントエンドだけに閉じ、バックエンドの挙動は増やさない。`serve` が Tier 1 の補助機能であることも変わらないため、プライムディレクティブのどれにも触れない。

## Motivation

`serve` の UI は `<meta name="viewport" content="width=device-width, initial-scale=1">` をすでに宣言している。これはスマートフォンのブラウザに「実際の CSS 幅で描画せよ」と指示するものだが、その指示を受け取るスタイルシートのほうが広い画面しか想定していない。`bajutsu/templates/serve.css` には **`@media` クエリが 1 つもなく**、すべてのレイアウト規則が固定の 1 デザインになっている。

* **Replay** はパネルを 5 トラックのグリッドで並べる（`main{grid-template-columns:var(--rep-left,340px) var(--gw) var(--rep-mid,340px) var(--gw) 1fr}`）。約 340 px の左右カラムにリサイズ用の溝、さらにレポートが続く構成だ。幅 390 px のスマートフォンではこのグリッドは収まらず、カラムが横にあふれるか、読めない幅まで潰れる。
* **Record** と crawl は `.tile-split` を入れ子にした flex ペインを `.tile-divider` のドラッグハンドルで仕切る作りで、その `.tile-grip` はマウス前提の操作部品だ（`cursor:grab`、ポインタによるリサイズ）。crawl のサイドパネルは `flex:0 0 22rem` で幅が固定される。これらはタッチ画面ではどれも成り立たない。ホバーするポインタはなく、グリップは小さく、ペインを左右に並べる余地もない。
* ヘッダー（`.toptabs` と、`margin-left:auto` で右端に寄せたテーマ切り替え）は横 1 行に収まる前提で、折り返さない。

結果として、ノート PC ではオーサリングも Replay もできるのに、スマートフォンからは実行結果をざっと確認することも、実行を開始することも、レポートを読むこともできない。一方で `serve` は、ホスティングの取り組み（BE-0015 の公開ホスティング、BE-0016 のセルフホスティング）によって URL でアクセスできる共有サービスになりつつあり、スマートフォンから開きたい場面はむしろ増えていく。「テストが落ちた」という通知を受け取ったレビュアーは、手元の端末でそのままレポートを開きたい。いまはそれが、横スクロールとズーム・パンを強いられる作業になっている。

狙いは控えめで範囲も限られている。機能も内容も同じ UI を、狭い画面では「縮小されたデスクトップ」ではなく「読めてタップできる 1 カラム」として並べ直す、それだけだ。

## Detailed design

変更は `bajutsu/templates/serve.css` に閉じる（タッチ操作のために `serve.js` へ小さな手を入れる程度は伴う）。サーバのルートもジョブモデルも設定も変えない。UI はこれまでどおり決定的な CLI を起動するだけなので、決定性もアプリ非依存性もそのまま保たれる。

**ブレークポイントの方針。** `max-width` のブレークポイントを 1 つ（必要なら 2 つ）導入する。スマートフォン向け（たとえば `≤ 640px`）と、必要なら幅の狭いタブレット向け（たとえば `≤ 900px`）だ。これらは既存のデスクトップ規則の**後ろ**に重ね、広い画面のレイアウトを既定のまま残して、狭い画面で変える必要のある部分だけを上書きする。手書きでビルド工程を持たない CSS のやり方は維持し、BE-0011 の「標準ライブラリのみ・テンプレートはインライン展開」という制約に合わせる。

**多ペインを 1 カラムへ。** ブレークポイントを下回ったら、左右に並んだペインを縦積みにし、デスクトップのリサイズ用ディバイダに代わるペインの切り替え手段をスマートフォン向けに用意する。

* **Replay**: 5 トラックのグリッドを 1 カラムにする。シミュレータ・設定フォーム、ストリーミングされるログ、埋め込みレポートを縦に積む（あるいは「フォーム・ログ・レポート」のセグメント切り替えの背後に置き、選んだものが全幅を使えるようにする）。`--rep-left` / `--rep-mid` のカスタムプロパティ幅とリサイズ用の溝は、この段では使わない。
* **Record**: `.rec-stack` のエージェント進捗と出力の分割も、同じやり方で縦積みか切り替えにする。
* **Crawl**: 固定幅 `22rem` のサイドパネルとグラフを、縦積みのセクションかタブ（「グラフ・プラン・コンソール」）にする。パン・ズームできるグラフは、表示中は全幅を保つ。

**タッチ操作。** マウスなしでは意味をなさないリサイズ用ディバイダは、狭い段では `.tile-divider` / `.tile-grip` を隠す（縦積みや切り替えがその役割を引き継ぐ）。タップ対象（タブ、実行・停止ボタン、履歴の行）はおおむね 44 px 以上を確保する。crawl のグラフについては、パンがタッチイベントから効くことを確かめ、ピンチによるズームも検討する（いまはポインタイベントでドラッグ・パンしている）。ここだけは純粋な CSS ではなく `serve.js` の作業が要るかもしれない、唯一の箇所だ。

**ヘッダーの組み替え。** ヘッダー行（ブランド、上部タブ、テーマ切り替え）は、狭い段では折り返すか詰めて並べ、横スクロールを強いないようにする。

**すでに可変な部分は残す。** スクリーンショットのモーダル（`.shotbox{width:min(960px,94vw);max-height:88vh}`、`.shotmain img{max-width:46vw}`）はすでにビューポート基準で可変だ。スマートフォン幅で見直し（画像と次手順リストを左右に並べる部分はおそらく縦積みにする必要がある）、可変なサイズ指定そのものは活かす。

**検証・ドッグフーディング。** BE-0058 はすでに、Web（Playwright）バックエンドで `serve` UI を駆動し、決定的な Tier 2 の回帰ネットにしている。レスポンシブ対応は、ここで自然に機械判定できる。既存の UI シナリオをスマートフォンのビューポート（Playwright が `viewport` を設定する）で実行し、主要な操作部品が存在して到達できること（実行ボタンが見えていて見切れていない、横方向のあふれがない、など）をアサートすればよい。合否は依然として機械判定可能なアサーションで決まり、LLM の判断は入らないので「決定性ファースト」の約束は保たれる。同時にドッグフーディングの網が小さい画面のレイアウトまで広がり、レスポンシブの規則が知らぬ間に壊れることを防げる。

## Alternatives considered

* **モバイル専用テンプレート／モバイル専用アプリを別に持つ。** 不採用。`serve` の機能が増えるたびに同期を取るべき面が二重になり、BE-0011 が意図的に返している 1 つの自己完結 HTML レスポンスとも噛み合わない。1 つのレスポンシブなテンプレートのほうが保守の負担は小さい。
* **グリッドとブレークポイント目当てに CSS フレームワーク（Tailwind / Bootstrap）を採用する。** 不採用。`serve` は標準ライブラリのみで、CSS は手書きでインライン展開し、アセットのビルド工程を持たない（BE-0011 の Alternatives）。素の `@media` クエリだけで、フレームワークやビルドのパイプラインを抱え込まずにレスポンシブ対応が得られる。
* **デスクトップ専用のまま、ブラウザのピンチズームとパンに頼る。** 不採用。それが現状であり、まさにこの項目が直そうとしている使いものにならない体験そのものだ。
* **デスクトップでもペインを折りたためるようにする（レイアウト全体の刷新）。** ここでは範囲外。この項目はあくまで小さい画面への組み替えに絞る。デスクトップ向けにペインを可変・折りたたみ可能にする広い再設計は、望むなら別の提案とし、狭い画面の対応を止めるものにはしない。
