# BE-0006 — idb 要素ツリー正規化の精度

* 提案: [BE-0006](BE-0006-idb-element-tree-normalization-ja.md)
* 状態: **提案**
* トラック: [提案](../../README-ja.md#提案)
* トピック: 実機検証（M1 クローズアウト）

## はじめに

`.searchable` 等 SwiftUI 標準要素のツリー表現が壊れていないかを実機で継続的に確認します。

## 動機

セレクタ解決は決定性の核です。セレクタはちょうど 1 要素にマッチし、0 件は「不在」、2 件以上は ambiguous エラーです。これらの判断はすべて、`IdbDriver.query()` が idb の `ui describe-all` から正規化した要素ツリーに対して下されます。したがって run ループ*全体*の正しさは、その正規化の精度以上にはなりません。標準的な SwiftUI コントロールが、私たちの想定どおりにツリーに現れない場合——識別子が違う、trait が欠ける、コンテナが子要素を呑み込んでしまう——それを対象に書いたシナリオは、要素を黙って見つけられないか、さらに悪いことに別の要素にマッチします。

DESIGN §11 はすでにここで 2 つの具体的な症状を記録しています。`.searchable` 等の SwiftUI 標準要素が精度リスクとして名指しで挙げられており、また画面遷移中に idb が——画面は描画済みでも——識別子なしの 1 要素というほぼ空のツリーを返す観測例があります。後者の症状は `IdbDriver.query()` の上限付き・ゲート付きリトライ（`_max_seen` / `_READY_MIN`）で緩和されていますが、それは*一過性*への防御であり、ある SwiftUI コントロールの*落ち着いた*ツリーがシナリオの想定どおりの形かどうかは教えてくれません。リトライの注記自体、本層では意図的にマスクしない「8 秒級の病的ケース」を指して終わっています。欠けているのは、実アプリが使う SwiftUI コントロールの定常状態の表現がずれていないことを実機で継続的に確認する仕組みです——idb・iOS・コントロールの作り方の回帰が、現場でのフレーキーまたは誤ったシナリオではなく、失敗するチェックとして現れるように。

## 詳細設計

本提案は、シナリオが実際に依存する SwiftUI コントロールについて、期待される正規化ツリーを固定する、維持されたオンデバイス fixture です。実行は、idb をすでに検証している重めの e2e 経路（高速な PR 単位ゲートではない）で行います。

- **サンプルアプリ内のコントロールカタログ画面。** デモアプリ（`demos/`）に、対象とする標準 SwiftUI コントロール——`.searchable`、`List`、`NavigationStack`、`TextField`、トグル、セグメントコントロール等——を並べた画面を持たせ、それぞれにアプリの id 名前空間から安定した `accessibilityIdentifier` を与えます。アプリ単位の識別子はアプリ側の事柄なので、この fixture は特定の本番アプリではなく*正規化*そのものを動かします。ツールは app-agnostic のままです。
- **正規化ツリーの golden（期待値）。** 各コントロールについて、期待される `Element` の形（`Selector` セマンティクスが依存するフィールド: `identifier`、`label`、`traits`、`value`、妥当な `frame`）を、チェックインされた期待値として記録します。チェックはアプリをその画面へ駆動し、`query()` を呼び、正規化ツリーを golden と照合します——機械チェック可能なアサーションなので、LLM を介さず決定性ゲートに収まります。
- **落ち着いた状態と一過性を明示的に区別する。** チェックは既存の条件待機経路で画面を待ち（固定 sleep なし）、その後*落ち着いた*ツリーに対して照合します。`_max_seen` ゲート付きリトライは引き続き遷移中の空を吸収します。本提案の golden はそのリトライが役目を終えた後の定常状態を対象とするので、2 つの関心事は分離されたままで、どちらも他方をマスクしません。
- **ずれは失敗するアサーションとして表面化する。** idb・iOS リリース・コントロールの構成が正規化された形を変えたとき、golden のアサーションは具体的な差分（どのコントロールのどのフィールドが動いたか）とともに e2e 経路で失敗します——その変更が、要素を見つけられなくなった現場のシナリオとして漏れ出すのではなく。これは BE-0005 のバージョン監視と自然に対になります。あちらは*どの* idb が入っているかを見張り、こちらは idb が*何を生み出すか*を見張ります。

ここにセレクタの厳格さを緩めるものはありません。ambiguous は依然失敗、不在は依然失敗で、golden アサーションは厳密な比較です。

## 検討した代替案

- **既存のシナリオ別 e2e デモに頼る。** デモシナリオはすでに実機で idb に触れますが、アサーションの対象はアプリの挙動であって、各コントロール型の正規化ツリーではありません。たとえば `.searchable` が識別子を報告する仕方に限った回帰は、現行のどのシナリオも壊さないまま、次にそれを対象に書かれるシナリオを壊しうるのです。専用のコントロールカタログは、表現そのものをテスト対象にします。
- **`query()` を強化して奇妙なツリーを修復する。** 想定外の SwiftUI の形を「期待される」形へ正規化経路内で書き換えるヒューリスティックを足すこともできます。それはずれを報告せず隠し、決定性が要の経路にアプリ固有の推測を載せます。golden に照合し、正規化器を意図的に更新する方が、`query()` を単純に、変更をレビュー済みに保てます。
- **実アプリの全画面をスナップショットする。** 網羅性は最大ですが、脆くアプリ固有です。チェックを 1 つのアプリの進化する UI に結び付け、ツールレベルの保証にアプリ単位の事柄を再導入します。標準コントロールに絞ったカタログの方が安定し、app-agnostic を保てます。
