## はじめに

`bajutsu trace --explain <シナリオ>` と `bajutsu audit <シナリオ>` は、どちらも共通ヘルパー
`load_expanded_scenarios`（`bajutsu/cli/_shared.py`）でシナリオを読み込み、その呼び出しを
`except (OSError, ValueError)` で囲んで、不正な入力を分かりやすいメッセージ付きの `exit 2` に変換
します。しかしこのガードは不完全です。**YAML の構文エラー**は `yaml.YAMLError`（`ParserError` や
`ScannerError` など）を送出しますが、これは `ValueError` のサブクラスではないため、ガードをすり抜け、
`exit 1` で Python のトレースバックがそのまま表に出てしまいます。本項目は、壊れたシナリオファイルが、
構造的に不正なシナリオと同じように綺麗に失敗するようにします。

## 動機

この 2 つのコマンドは「入力が見つからない・読み取れない、またはシナリオが不正なら、メッセージ付きで
`exit 2` する」という契約を掲げており、隣り合うコードパスはそれを守っています。構造的に不正なシナリオ
（YAML としては妥当だが、内容がスキーマ検証に通らないもの）は pydantic の `ValidationError` を送出し、
これは `ValueError` のサブクラスなので捕捉され、`failed to load scenario: …` として `exit 2` で表示
されます。ところが、オーサリングで最もよくある間違い — ファイルがそもそも YAML として解析できなく
なるタイプミス（閉じ忘れた波括弧、崩れたインデント）— は、別の、より悪いパスをたどります。生の
`yaml` 例外がそのまま伝播し、利用者は 1 行のメッセージではなくスタックトレースを見せられ、終了コードも
契約上の `2` ではなく `1` になります。

これは小さな正しさの欠落ですが、利用者への影響があります。人が最も間違えやすい入力に対してこそ、
ツールはトレースバックではなく、決定的で読みやすいエラーを返すべきです。この欠落は
[BE-0117](../BE-0117-coverage-floor-ratchet/BE-0117-coverage-floor-ratchet-ja.md) の CLI 層テストを
書いている最中に見つかりました。BE-0117 のテストは意図的に到達可能な `ValueError` 分岐を使い、この
欠落は独立した修正に委ねています。欠落を塞ぐには（新しい例外を捕捉するという）挙動の変更が必要で、
テストの追加だけでは済まないからです。

## 詳細設計

作業はローダーとその 2 つの呼び出し元にわたって MECE（漏れなく重複なく）に分解できます。

- **境界をどこに置くか決める。** `load_expanded_scenarios` は `Raises: OSError` / `ValueError`
  （「ファイルは解析できるが、内容が不正」）を掲げています。YAML の*構文*エラーはファイルが
  **解析できない**ことを意味するので、今の契約の外だとも言えます。筋の通った選択肢は次の 2 つです
  （どちらか 1 つを選び、一貫して適用します）。
  1. **ローダーで正規化する。** `load_expanded_scenarios` 内の
     `load_scenario_file(path.read_text(...))` の解析を囲み、`yaml.YAMLError` を `ValueError` として
     （`from e` で連鎖させて）送出し直します。これはヘルパーが既に掲げる `ValueError` の契約に沿い、
     **すべての**呼び出し元（`trace --explain` と `audit`、および将来の呼び出し元）が綺麗な挙動を
     そのまま得られます。解析できない内容も `ValueError` を送出すると docstring を更新します。
  2. **各呼び出し元で捕捉を広げる。** 2 か所の `except (OSError, ValueError)` に `yaml.YAMLError`
     も加えます。局所的ですが、コマンドモジュールに `yaml` の import が漏れ出し、将来の呼び出し元
     ごとに繰り返す必要があります。

  選択肢 1 を推奨します（1 か所で済み、掲げた契約に一致し、呼び出し元ごとの繰り返しがない）。
- **選んだ修正を実装する。** 選択肢 1 なら `bajutsu/cli/_shared.py` に入れます。`yaml.YAMLError` が
  生じるのは解析の 1 か所だけだからです。エラー文言は行動につながる内容（ファイル名と解析の問題）を
  保ちます。
- **リグレッションテストを追加する。** 本当に壊れた YAML シナリオ（閉じ忘れたフロー写像など）で
  両方の入口をテストし、トレースバックではなく `exit 2` と `failed to load scenario` のメッセージが
  出ることを確認します。既存の `ValueError`（構造的に不正）のテストは残します。両方の分岐を
  カバーし続けるためです。
- **どの呼び出し元も退行していないことを確認する。** `run` は独自のローダーを使い
  `load_expanded_scenarios` は使わないので対象外です。`trace` / `audit` の正常系と他のエラー分岐が
  変わっていないこと、そしてゲートが緑のままであることを確認します。

## 検討した代替案

- **現状のままにする。** 却下します。掲げた契約はこれらの入力を「メッセージ付きで `exit 2`」と
  しており、最もよくあるオーサリングのタイプミス 1 つでトレースバックが出るのは、小さくとも実在する
  正しさのバグです。
- **呼び出し元で広い `Exception` を捕捉する。** 却下します。無関係なバグ（プログラミングエラー）まで
  「failed to load scenario」の陰に飲み込み、本当の失敗を隠してしまい、プロジェクトの「失敗は大きく
  鳴らす」姿勢と逆行します。修正は解析エラーの系統だけを正確に捕捉すべきです。
- **読み込み前に YAML を別途検証する。** 冗長なので却下します。ローダーは既に一度解析しています。
  事前解析をもう 1 回行うのは無駄な処理であり、真実の源も二重になります。既存の解析 1 回から出る
  例外を正規化するほうが簡潔です。
