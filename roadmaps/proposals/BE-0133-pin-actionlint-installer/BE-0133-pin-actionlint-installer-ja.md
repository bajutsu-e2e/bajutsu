## はじめに

[`.github/workflows/ci.yml`](../../../.github/workflows/ci.yml) の `check` ジョブは、`uv sync`
の対象外である唯一のゲートステップである `actionlint` を、可変なタグから取得したスクリプトを
`bash` にそのまま流し込んでインストールしています。このリポジトリの CI にあるほかのサードパー
ティ依存はすべてコミット SHA で固定されています。本提案は、`actionlint` のインストーラをその
方針に合わせます。

## 動機

`ci.yml:66` は次のとおりです。

```yaml
- name: Install actionlint
  run: bash <(curl -fsSL https://raw.githubusercontent.com/rhysd/actionlint/v1.7.12/scripts/download-actionlint.bash) 1.7.12
```

これは `rhysd/actionlint` の `v1.7.12` という参照から `scripts/download-actionlint.bash` を取得
し、そのまま実行します。git タグは可変です。アップストリームのメンテナが付け替えることもあれば、
アカウントが乗っ取られた場合には攻撃者が付け替えることもあり、コミット SHA のような不変性はあり
ません。スクリプト自体は、該当リリースに合わせて `actionlint` バイナリをダウンロードして展開しま
す。取得したスクリプトが CI ランナー上でそのまま実行される以上、`v1.7.12` が付け替えられれば
`check` ジョブの権限（`ci.yml` トップレベルの `permissions:` ブロックにある `contents: read`）の
範囲で実行されることになります。範囲は限定されているとはいえ、レビューを経ないコード実行経路で
あることに変わりはありません。

対照的に、`.github/` 内で参照されているほかのサードパーティアクション（`actions/checkout`、
`astral-sh/setup-uv`、`actions/cache`、`actions/create-github-app-token`、
`actions/upload-pages-artifact`、`actions/deploy-pages`、`maxim-lobanov/setup-xcode`、
`actions/upload-artifact`）は、いずれもバージョンをコメントで添えたフルコミット SHA で固定されて
います（例：`actions/checkout@9c091bb21b7c1c1d1991bb908d89e4e9dddfe3e0  # v7.0.0`）。`actionlint`
のインストーラは、この既存の方針から外れた唯一の例外であり、しかも `uses:` によるアクション参照
ではなく `curl | bash` という形なので、レビューのしやすさという点でもアクション参照より劣ります。

深刻度についてはこの指摘は**未確認**です。インシデントやアップストリームの侵害が実際に確認され
ているわけではなく、ジョブの権限もすでに読み取り専用に絞られているため、影響範囲には上限があり
ます。それでも取り上げるのは、このリポジトリがほかのあらゆる箇所ですでに保持しているセキュリティ
方針との間に、機械的に埋められる差分が残っているためです。

## 詳細設計

修正は `ci.yml:66` に閉じた、互いに独立な 2 つの作業からなります。

1. **インストーラスクリプト自体をコミット SHA で固定する。** `raw.githubusercontent.com` の URL
   にある `v1.7.12` タグを、そのタグが現在指しているコミット SHA に置き換えます。可読性のために、
   `uses:` の固定ですでに使われている慣例（例：`@<sha>  # v1.7.12`）と同じ形式で、末尾にタグ名の
   コメントを添えます。これにより、タグが後で付け替えられても、取得されるスクリプトの内容は不変
   になります。
2. **可能であれば、ダウンロードしたバイナリ自体も検証する。** `download-actionlint.bash` が取得
   するバイナリのチェックサムをすでに検証しているかを確認します。していない場合、あるいは
   `actionlint` の `1.7.12` リリースで公開されているチェックサムを別途固定できる場合は、その検証
   を追加し、インストーラスクリプトだけでなくバイナリ自体も改ざん検知できるようにします。これは
   小さく独立した追加作業であり、項目 1（スクリプトの固定）だけを先に単独でマージできます。

どちらの変更も `ci.yml` の 1 行に閉じており、`run`/CI の合否判定ロジックには触れません（`check`
ジョブは修正後も `actionlint` 自身の検出結果だけで合否を決めます）。プロダクトコードの対象範囲は
なく、CI のサプライチェーン強化のみです。

## 検討した代替案

- **インストーラスクリプトの代わりに、`uses:` ベースの actionlint アクションに切り替える。**
  `actionlint` バイナリの取得をラップしたコミュニティ製アクションはいくつか存在し、それを使えば
  この固定もほかのサードパーティステップと同じ SHA 固定の `uses:` パターンに揃えられます。ただし
  これは、既存のものを単に固定するのではなく、1 つのサードパーティの信頼対象（rhysd のインストー
  ラスクリプト）を別のもの（コミュニティ製アクション）に置き換えるだけであるため、本提案では見送
  ります。既存のインストーラスクリプトの固定が難しくなった場合にのみ、改めて検討する価値がありま
  す。
- **`actionlint` バイナリをリポジトリに同梱するか、CI 内でソースからビルドする。** 実行時の取得を
  完全になくせますが、同梱バイナリの更新や Go ツールチェーンのステップといった、実際の保守コスト
  が発生します。SHA 固定という 1 行の変更で解決できる問題に対しては、見合わないコストです。
- **ジョブの権限がすでに読み取り専用であることを理由に、何もしない。** 影響範囲には上限があります
  が、このリポジトリがすでに持っている SHA 固定という方針は、そもそも将来のあらゆるワークフロー
  変更でその上限に頼らずに済むようにするためのものです。この 1 つの例外を残すことは、目先のジョブ
  そのものよりも、その方針の一貫性を損ないます。
