## はじめに

scenario の `totp` ステップは `secret` フィールドに TOTP のシード値をそのまま持つことがあり、
その scenario ファイルは run 成果物へそのままコピーされます。本提案は、シードの生の値がディスクへ
書き出され、アーカイブとして配布されるものに含まれないようにします。

## 動機

`Totp`（`bajutsu/scenario/models/actions.py:127-136`）は `totp` ステップを `secret: str`
（「共有の base32 鍵（多くは `${secrets.*}`）」）としてモデル化しています。このステップは
決定的かつローカルで完結しており（`bajutsu/totp.py`）、シードとクロックから RFC 6238 のコードを
ネットワーク呼び出しなしに計算します。まさにそのために scenario 側に生のシードが必要になります。
run が実行した scenario ファイルは run のエビデンスの一部として保存され、run 成果物
（`bajutsu/evidence.py`、`report.html` やアーカイブ一式）はこのリポジトリがすでに他の箇所で
機微情報として扱っている面です。ネットワークのデフォルトシークレット redaction やスクリーン
ショットのシークレット警告に関する姉妹提案も、同種の漏えいに対するものです。`secret` が
YAML 内にリテラル文字列として書かれる場合（`${secrets.*}` 経由にしない場合。`${secrets.*}` は
BE-0032 が scenario ファイル自体に資格情報を含めないための仕組みです）、そのリテラルなシードは
redaction されないまま run の成果物一式にコピーされます。したがってアーカイブへアクセスできる
者（共有された CI 成果物、ダウンロードしたレポートなど）は、使い捨てのコードではなく、恒久的な
TOTP シードを抽出できてしまいます。

深刻度は Low です。`${secrets.*}`（BE-0032）はすでに scenario にリテラルなシードを書かずに済む
手段を著者に提供していますし、scenario ファイルにシードをリテラルでコミットすること自体、成果物
へのコピーとは独立に、それ以前からのアンチパターンです。しかしツール側はそれが起きても警告も
redaction も行わないため、執筆中の利便性からリテラルな `secret` で書かれた scenario は、以後
すべての run の成果物へ黙ってシードを持ち込み続けます。

## 詳細設計

1. **scenario ファイルが `run_dir` へコピーされる際に `totp.secret` を redaction する。**
   実行済み scenario のスナップショットを書き出す成果物ライタ（`network.json` / `report.html`
   と並ぶエビデンス側の処理）で、コピーを書き出す前に `totp.secret` フィールドのリテラル値を
   固定のプレースホルダ（例えば `"<redacted>"`）へ置き換えます。これは `serve` の API キー
   プレビューを `mask_secret` がすでに redaction している方法と同じ考え方です。
2. **解決済みの値のケースにも対応する。** scenario が `totp.secret` に `${secrets.*}` を使って
   いる場合、補間・解決された後の値も同様に成果物へ書き出してはいけません。この作業では、
   既存のシークレット変数 redaction（BE-0032）が他の `${secrets.*}` 利用と同様に解決済みの
   `totp.secret` もすでにカバーしているかを確認し、カバーしていなければその隙間を塞ぎます。
3. **回帰テストを追加する。** リテラルで書かれた場合と `${secrets.*}` から解決された場合の
   どちらでも、`run_dir` 内の scenario スナップショットに TOTP シードの生の値が含まれないことを
   検証します。

## 検討した代替案

- **scenario の検証時点で `totp` ステップのリテラル `secret` を禁止し、`${secrets.*}` を必須に
  する。** より厳格な補完策として検討しましたが、本項目の一次的な修正としては採用しません。
  成果物へのコピーによる漏えいを塞ぐだけでなく、受理される scenario の構文自体を変えることに
  なり、リテラルなシードを持つ既存の scenario には移行が必要になるためです。本提案の
  コピー時 redaction であれば、破壊的なスキーマ変更なしに既存の執筆スタイルを保護できます。
- **プロジェクト全体を汎用のシークレットスキャナで `Totp.secret` らしき文字列すべてに redaction
  をかける。** 必要以上に広い対策として却下しました。対象フィールドはモデルレベルで正確に
  わかっている（`Totp.secret`）ため、その 1 フィールドに絞った redaction のほうが単純で、
  無関係な文字列を誤検知するリスクもありません。
