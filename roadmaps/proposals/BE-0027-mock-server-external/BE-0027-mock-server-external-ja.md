## はじめに

config スキーマのみ存在します。宣言的な in-protocol `mocks`（実装済み）に置き換えられており、外部サーバ方式が必要かどうかは未決定です。

## 動機

決定的な E2E run には、ネットワークを制御下に置くことが欠かせません。同じリクエストには毎回同じレスポンスが
返らなければ、合否が再現可能でなくなるからです。config はすでにその一手段として `mockServer` スキーマ
（`cmd` / `port` / `stubs`）を備えています。run の傍らで外部モックサーバのプロセスを起動し、アプリをそこへ
向けるという方式です。アプリを本物のバックエンドではなく `localhost:<port>` に接続させ、サーバが定型の
レスポンスを返す、という想定でした。

その後、同じ要求は別の方法で満たされました。シナリオレベルの `mocks`（Bajutsu 自身のネットワーク経路の
内側で横取りする宣言的な in-protocol スタブ）が実装され、実機で検証済みです。これは、起動、ヘルスチェック、
後始末を要する第二のプロセスなしに、オフラインで決定的なレスポンスを提供します。そのため本項目で未決なのは
「外部サーバをどう作るか」ではなく「そもそも必要か」であり、これが本項目をスケジュール済みではなく**保留**と
している理由です。

## 詳細設計

採用が決まった時点で具体化します。今日存在するのはスキーマ（`config.py` の `MockServer`:
`cmd` / `port` / `stubs`）だけで、ランタイム上の効果はありません。

本項目は**保留**のため、設計は意図的に詰めていません。仮に採用するなら、in-protocol `mocks` がすでに満たして
いる基準をクリアする必要があります。

- **決定性。** 外部サーバは、シナリオ開始前に既知の準備完了状態まで立ち上げ、終了後に確実に後始末されたことを
  検証しなければなりません。固定 `sleep` は使わず、準備完了／ヘルス条件で待つことで、「条件待ちのみ」に
  従います。アプリが最初に呼ぶ時点でまだ待ち受けていないことがあるプロセスは、プロジェクトが禁じるフレーク
  性をまさに再導入してしまいます。
- **アプリ非依存の config。** 配置は現状のまま（`apps.<name>.mockServer`）とし、アプリ単位のモック配線は
  config に置き、ランナーはアプリ間で不変に保ちます。
- **LLM フリーのゲート。** モックサーバは決定的なプロセスなので、「LLM を run/CI ゲートに入れない」という
  プライムディレクティブには触れません。

**採用される**のは、in-protocol `mocks` では賄えない具体的な必要が現れた場合に限ります。たとえば、宣言的な
スタブとして表現するには状態やプロトコルが重すぎるバックエンド（本物の gRPC や WebSocket サーバ、アプリが
多数のリクエストにまたがって駆動するステートフルなセッション）で、in-protocol スタブの記述言語を拡張するより
既存のモックサーバ実装を再利用するほうが本当に単純な場合です。そうした必要がない限り、スキーマは文書化された
未配線のプレースホルダのまま残ります。

## 検討した代替案

**いま外部サーバを実装する。** 今回は却下しました。管理すべきプロセス（起動、準備完了、後始末）と、ネット
ワーク挙動の第二の源を増やすことになりますが、その必要は実装済みの in-protocol `mocks` がすでに満たして
います。動機となるユースケースなしに先回りして作るのは、複雑さだけを抱え込むことになります。

**`mockServer` スキーマを完全に削除する。** 検討しましたが、残しました。このスキーマは実在する設計上の代替案
を文書化しており、未配線である限りコストはかかりません。削除すればその選択肢を捨て、将来の担い手に形を再発見
させることになります。`architecture.md` が「スキーマのみ、未配線」と明示した状態で保留のまま保持するのが、
誠実な落としどころです。
