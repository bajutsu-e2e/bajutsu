## はじめに

**Claude を使う**機能と使わない機能の境界を、**ツールの一級の性質として目に見えるかたち**にし、
Claude を使わない側のすべてが**設定ゼロで動く**ことを保証します。資格情報も `.env` も、いかなる
事前準備も不要です。利用者はリポジトリをクローンした直後から `run` / `doctor` / `codegen` / `trace`
/ `lint` / `schema`（および `serve` の決定論的な部分）をそのまま実行でき、Claude を使う経路
（`record` / `crawl` / `triage --ai` / `run --dismiss-alerts`）は「モデルに到達するもの」として明示
されます。

軸は、特定の資格情報があるかどうかではなく、**そもそも Claude を呼ぶかどうか**です。Claude へは
複数のプロバイダと複数のエージェントバックエンドから到達できます。Anthropic API
（`ANTHROPIC_API_KEY`）、Amazon Bedrock（AWS 資格情報。[BE-0053](../../implemented/BE-0053-bedrock-ai-provider/BE-0053-bedrock-ai-provider-ja.md)）、
あるいはサブスクリプションでログインした Claude Code CLI（`--agent claude-code`）です。「API キー」は
そのうちの一つの手段にすぎず、見えるようにすべきはモデル呼び出しそのものです。したがって
「Claude を使わない」とは Claude をまったく使わないことであり、AI の資格情報もログインもランタイムも
一切不要であることを意味します。

これは、設計上すでに存在している境界の上に重ねる、発見しやすさと開発体験のための層です。
プライムディレクティブは何も変えません。ここで可視化する Claude を使う側と使わない側の分離は
Tier 1 と Tier 2 の分離そのものであり、`run` / CI ゲートにモデルを持ち込む変更は一切含みません。

## 動機

Bajutsu のアーキテクチャは、すでに明確な線を引いています。決定論的な `run` / CI ゲートはモデルを
呼ばず、Claude に到達するのは Tier 1 の作成・調査の経路だけです。[BE-0047](../../implemented/BE-0047-ai-data-sovereignty/BE-0047-ai-data-sovereignty-ja.md)
は AI 側を「強制された約束」に変えました。解決されたプロバイダに使える資格情報が無ければ AI の入口は
fail-closed で停止し（`anthropic_client.credential_gap()`）、[BE-0053](../../implemented/BE-0053-bedrock-ai-provider/BE-0053-bedrock-ai-provider-ja.md)
はプロバイダを差し替え可能にしました。つまり配管と強制の仕組みは揃っています。

足りないのは、**境界が利用者から見えること**と、**Claude を使わない側が何も要らないという明示的な
保証**です。具体的なギャップが 3 つ残っています。

1. **分離が発見できない**。どのコマンドが Claude を呼び、どれが呼ばないのかを示す単一の場所が、
   ドキュメントにも機械可読な定義にも存在しません。事実は `CLAUDE.md` の散文と、あちこちに散らばった
   呼び出し箇所に埋もれており、初めて触れる人は `run` が Claude を使わず `record` が使うと一目で
   判断できません。Bajutsu をベンダのクラウド型ツールから最も際立たせている性質（モデルに一切触れず
   に決定論的なコアを採用できること）が、最初の接触点で見えないままです。

2. **「API キー」は単位として誤りで、本当の単位は「Claude を使うか」である**。線を「キーが要る」と
   枠取るのは狭すぎ、すでに不正確です。Bedrock プロバイダはキーではなく AWS 資格情報で認証し、
   `--agent claude-code` バックエンドは Claude Code CLI 自身のサブスクリプションログインで Claude に
   到達します。誠実で安定した軸は、経路が*そもそも* Claude を呼ぶかどうかであり、資格情報の手段
   （Anthropic キー、AWS 資格情報、CLI ログイン）は「どう」呼ぶかの細部であって「呼ぶか」の問題では
   ありません。分類も、その上に作るすべての表示面も、「Claude を使う」という軸で表現しなければなり
   ません。そうすればプロバイダやエージェントバックエンドが増えても正しいままです。

3. **「ゼロ設定」は事実だが保証されておらず、`serve` も優雅に縮退しない**。Claude を使わない経路は
   今日 AI の準備無しで動きますが、それをテストで留めるものがありません。将来のモジュール先頭での
   SDK の import、AI 用の環境変数を読む設定の既定値、利用可否を探りにいく `serve` の起動経路などが、
   決定論的な経路にひそかに設定を要求するようにしても、それを捕まえるゲートがありません。さらに
   Web UI は Claude に到達できるかどうかにかかわらず Claude の作成画面（`record` / `crawl`）と
   AI トリアージを表示します。到達できないと、利用者はタブを押して失敗にぶつかって初めて要件に
   気付き、「これは Claude が要る」と前もって示す手当てがありません。

これらを塞ぐと、「Bajutsu は Claude 無しで使える」がアーキテクチャの偶然から、明文化されテストされ
目に見える性質へと変わります。同じ線の反対側で BE-0047 が掲げる「あなたの AI、あなたのキー、あなたの
データ」の自然な相方になります。

## 詳細設計

提案の高さで述べます。既存のシーム（`anthropic_client.credential_gap()`、`agents.py` の
エージェントバックエンドのレジストリ、コマンドごとの CLI 構造、`serve` の API キー用エンドポイント）の
上に積む 4 つの部品で構成し、作業はこの 4 部品で MECE に分かれます。

### 1. 「Claude を使う」軸での単一の機能分類

すべてのコマンドと機能を **Claude を使う** か **Claude を使わない** に分類する、唯一の正本を導入
します。一度だけ定義し、あらゆる場所がそれを参照します。

- CLI のコマンドレジストリと同じ場所に置く機械可読な定義。各コマンドがフラグ（たとえば
  `uses_claude: bool`）を持ち、分類が散文の重複ではなくデータになります。Claude を使う集合はちょうど
  `record`、`crawl`、`triage --ai`、`run --dismiss-alerts` の 4 つで、Claude を使わない集合はそれ以外
  （`run`、`doctor`、`codegen`、`trace`、`lint`、`schema`、`approve`、`mcp`、`worker`、および
  `serve` の決定論的な部分）です。フラグはコマンド名ではなく*経路*の粒度です。`--ai` を付けない
  `triage` は Claude を使わず（ルールベースの `HeuristicTriageAgent`）、`triage --ai` は Claude を
  使い、1 つのフラグがそれを切り替えます。
- 分類は**プロバイダとエージェントバックエンドに依存しません**。あるコマンドが*いずれかの*プロバイダ
  （Anthropic、Bedrock）で*いずれかの*バックエンド（SDK エージェントまたは Claude Code CLI）を通じて
  モデルを呼ぶなら、それは Claude を使うコマンドです。どの手段で認証するかは*分類*の対象外で（設定の
  問題であり、BE-0047 / BE-0053 に従います）、分類は「これは Claude に到達するか」だけに答えます。
- この分離を人間向けに描くドキュメントページ（`docs/` と `docs/ja/` の二言語）。「何が Claude を使い、
  何が使わないか」の正本となる参照であり、はじめに読む資料からの短い導線を置いて、初めて触れる人が
  最初に目にするようにします。
- 同じデータが下記の CLI ヘルプと `doctor` の表示にも供給されます。正本がただ 1 つになり、3 つの
  表示面が食い違うことはありません。

### 2. プロバイダとバックエンドを横断する「Claude に到達できるか」の統一判定

`credential_gap()` は **SDK** 経路（Anthropic キー、または Bedrock のモデル id）の利用可否に答え
ますが、`--agent claude-code` バックエンドは CLI を通じて Claude に到達し、利用可否の条件が異なります
（`claude` バイナリが存在し、そのサブスクリプションにログイン済みであること）。解決された
エージェントバックエンドとプロバイダを受け取り、統一された答え（「到達できる」、または具体的で対処可能
な不足。キーが無い、Bedrock モデルが無い、CLI が無い／未ログイン）を返す単一のリゾルバ
`ai_availability(effective)` を導入します。下記の serve と doctor の表示面はこの 1 つのヘルパーで制御
するので、ターゲットがどのバックエンドやプロバイダを選んでも正しいままです。これは既存のシームの薄い
一般化であり、新しいサブシステムではありません。

### 3. Claude を使わない経路に対する、テストされたゼロ設定の保証

「Claude を使わない側は何も要らない」という約束を、BE-0047 が AI 側に与えたのと同じ厳密さで留めます。
すべて高速な Linux ゲートの中で行い、実機も実 API も使いません。

- **AI の準備が一切無い状態で Claude を使わないコマンドを実行する回帰テスト**を置きます
  （`ANTHROPIC_API_KEY` 無し、AWS 資格情報無し、`.env` 無し、`claude` CLI 無し）。資格情報の不在で
  エラーにならず、エージェントを構築もしないことを確かめます。部品 1 の分類を起点にするので、テストは
  構成上 Claude を使わない集合を網羅し、新しく追加されたコマンドも自動的に対象に入ります。
- **import 時のガード**。決定論的な経路のモジュールを import しても、モジュール先頭で AI の SDK を
  import したり AI 用の環境変数を読んだりしないことを表明します。モデルクライアントは Claude の入口の
  内側でだけ遅延構築されます。ゼロ設定がひそかに壊れる最もありそうな経路（不用意なモジュール先頭の
  `import anthropic` や `os.environ["ANTHROPIC_API_KEY"]`）を、これが捕まえます。
- これらのテストは保証を実行可能にします。「ローカルで緑」が「Claude を使わない経路はゼロ設定である」を
  含むようになり、それを壊す変更はゲートで落ちます。

### 4. `serve` の優雅な縮退と、CLI ヘルプ・`doctor` での分離の可視化

利用者が境界に出会う 3 つの地点で、それを目に見えるようにします。

- **`serve`** は実行時に失敗するのではなく縮退します。Claude に到達できないとき（部品 2 の
  `ai_availability` を解決済みのバックエンドとプロバイダに対して評価します）、Claude のタブと操作
  （`record`、`crawl`、AI トリアージ）を**無効化**して表示し、何が不足しているかをその場で明確に説明
  します（「Claude が必要です。API キーを設定するか、Bedrock を構成するか、Claude Code CLI にサイン
  インしてください」）。あわせて既存の UI 内キー入力欄（`set_api_key`）へ誘導します。残りの UI
  （run、レポート、エビデンス、視覚ベースライン、GUI エディタ、`doctor`）は完全に操作可能なままです。
  小さなステータス項目（`api_key_info` を拡張）が Claude の利用可否を返し、フロントエンドがデータに
  基づいて制御し、Claude に到達できるようになるとその場で切り替わります。
- **CLI ヘルプ**は、部品 1 の `uses_claude` データに基づいて、コマンドを *Claude を使わない* と
  *Claude を使う* に分類（または各コマンドに注記）します。`bajutsu --help` 自体が境界を示します。
- **`doctor`** は、**Claude の準備状況を独立した、明らかに任意の節として**報告します。すでに採点して
  いる端末や規約の準備状況とは切り離します。AI の準備が無い利用者には、決定論的な経路について環境が
  `Ready` と採点されたうえで、Claude は別立ての「未設定（任意）」の行として示され、阻害要因と混同
  されることはありません。これは [BE-0024](../../implemented/BE-0024-doctor-onboarding/BE-0024-doctor-onboarding-ja.md)
  のオンボーディング表示面を拡張するもので、決定論的で LLM を使いません（`ai_availability` を見るだけ
  です）。

### プライムディレクティブとの整合

ここにあるものはすべて Tier 1 側に置かれるか、Tier 1 側を記述します。分類は Claude の経路に*名前を
付ける*だけで、経路を*増やしません*。`run` と CI ゲートは変わらず、モデルは無く、合否は機械判定の
ままです。`serve` と `doctor` の変更は `ai_availability` を読み、それに基づいて UI と表示を制御する
だけで、決定論的であり LLM 呼び出しを足しません。アプリ固有の差（どのプロバイダか、どのバックエンドか、
どの資格情報か）は引き続き設定（BE-0047 の `ai` ブロック）から来るので、ツールはアプリに依存しない
ままです。

## 検討した代替案

- **暗黙のまま残す（現状維持）**。却下します。BE-0047 が AI 側の保証を暗黙のまま残すことを却下したのと
  同じ理由です。Bajutsu を最も際立たせている性質（Claude 無しで採用できること）が、見込みの利用者に
  見えず、退行からも守られません。たまたま成り立つ性質は、指し示せる機能ではありません。

- **線を「API キーが要る」と枠取る**。却下します。狭すぎ、すでに誤りです。同じモデルへ至る Bedrock
  （AWS 資格情報）と Claude Code CLI（サブスクリプションログイン）の経路を無視します。「Claude を使う」
  という軸こそ、プロバイダとバックエンドが増えても正しいままであり、資格情報の手段はその下の細部です。

- **BE-0024（doctor / オンボーディング）に畳み込む**。`doctor` の Claude 準備状況の部分は確かにその
  表示面に属し、そこに着地します。しかし分類マップ、統一された利用可否判定、ゼロ設定のテスト保証、
  `serve` の縮退は横断的で、BE-0024 が対象とする「小さなオンボーディングの手直し」より大きく、独自の
  設計を持つ専用項目に値します。

- **Claude に到達できないときは Claude コマンドを完全に隠す**。却下します。隠すとツールの全体能力が
  *より*発見しにくくなり、目的に反します。狙いは、Claude 機能を見せたうえで明示する見える境界であって、
  小さく見えるツールではありません。

- **同梱の便宜的な資格情報を配って「すべてそのまま動く」ようにする**。却下します。BE-0047 が避ける
  ために存在する、ベンダクラウドへのデータ流出の懸念をそっくり再現し、本項目が見えるようにしようと
  している境界そのものを溶かしてしまいます。
