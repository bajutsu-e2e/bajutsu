## はじめに

`serve` の UDID 検証用正規表現は先頭のハイフンを許可しており、クライアントから渡される `udid` が
`-` で始まりうる状態です。本提案はこのパターンを厳格化し、UDID がサブプロセスへのフラグと誤認され
ないようにします。

## 動機

`bajutsu/serve/helpers.py:27` は `_UDID_RE = re.compile(r"^[A-Za-z0-9-]+$")` を定義しており、
`serve` のリクエストボディから来た `--udid` トークンを `bajutsu` のサブプロセス呼び出し
（`run` / `record` / `crawl` / `doctor`）に渡す前の検証（`helpers.py:481`）に使われています。
この文字クラスはハイフンを先頭を含むどの位置にも許可するため、`-rf` や `--config` のような値も
正当な UDID として通ってしまいます。`serve` は検証済みトークンをシェルを介さず単一の argv 要素と
して渡すため、古典的なシェルインジェクションは成立しません。しかし多くの CLI パーサ、とりわけ
`idb` や `xcrun` は、先頭がハイフンの引数を位置引数ではなくオプションとして扱うため、細工した
`udid` がサブプロセスの呼び出し先ツールにフラグとして解釈される余地があります。

深刻度は Low です。攻撃者は `serve` の `/api/run`（あるいは `/api/record` / `/api/crawl`）に
到達するリクエストをまず RBAC・トークンのゲート（BE-0051）を通す必要があり、また問題の引数は
シェルではなく固定されたサブプロセス argv に渡るため、被害範囲は任意コマンド実行ではなく呼び出し
先ツールでの引数の誤認に留まります。

## 詳細設計

1. **`_UDID_RE` を厳格化する。** `^[A-Za-z0-9-]+$` から `^[A-Za-z0-9][A-Za-z0-9-]*$` に変更し、
   先頭文字を英数字に限定します。実在する UDID の形状（`simctl` のハイフン区切り UUID、物理
   デバイスの 40 文字の識別子、リテラルの `booted`）はいずれも `-` で始まらないため、この変更で
   正当な値が弾かれることはありません。
2. **既存の検証呼び出し箇所はそのまま使う。** `helpers.py:481` とその呼び出し元（`/api/run`、
   `/api/record`、`/api/crawl`、`/api/doctor` の各リクエストパス）は変更しません。修正は正規表現
   自体に閉じており、値がパターンに一致しない場合は既存のバリデータ（`_valid_udid` 相当）が
   すでに値全体を拒否しているためです。
3. **単体テストを追加する。** 先頭がハイフンの値（`-rf`、`--help` など）が拒否され、実在する
   UDID の形状と `booted` は引き続き通ることを確認します。

## 検討した代替案

- **標準的な UUID の位置以外にハイフンを含む UDID をすべて拒否する。** 却下しました。8-4-4-4-12
  の UUID 形状をバリデータへハードコードすることになり、将来 UDID の形式が変わった場合に前方
  互換性を損ないます。先頭ハイフンの禁止を超えるインジェクション対策上の利点もありません。
- **先頭のハイフンを拒否ではなく除去して正規化する。** 却下しました。クライアントが指定した値を
  黙って書き換えると、意図しない別デバイスへサブプロセスをルーティングしかねません。リクエスト
  自体を失敗させるほうが安全で、デバッグもしやすくなります。
