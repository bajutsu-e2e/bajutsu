## はじめに

シナリオ YAML にはバージョンを示すマーカーがなく、その pydantic モデルは未知のフィールドを一律で拒否します。そのため、古い `bajutsu` が新しいシナリオを読み込むと、わかりやすいメッセージではなく不透明なバリデーションエラーで失敗します。本提案では、シナリオに明示的な `schema` フィールドを追加し、レポートマニフェストの `schemaVersion` がすでに確立している前例に倣って、わかりやすいエラーを出すバージョンチェックを導入します。

## 動機

`bajutsu/scenario/models/_base.py:49` は、すべてのシナリオモデルが継承する共有ベースを定義しています。`model_config = ConfigDict(populate_by_name=True, extra="forbid")` です。`extra="forbid"` は、シナリオ作成時のタイプミスや未対応フィールドを検出するうえで正しい選択です（シナリオ内のタイプミスが黙って無視されるより、大きな声で拒否されるほうがまだましです）。しかし同時に、新しいフィールドをトップレベルやステップレベルに追加した、より新しい `bajutsu` 向けに書かれたシナリオは、古い `bajutsu` では一切パースできないことも意味します。失敗の形は汎用的な pydantic の `ValidationError`（「Extra inputs are not permitted」）であり、実際の問題がバージョンの不一致であることも、どう対処すればよいかも、ユーザーには伝わりません。

この隙間は、シナリオファイルとそれを読む `bajutsu` がほぼ常に同じチェックアウトに由来する現在は、さほど問題になりません。問題になるのは、シナリオの読み込みが日常的にバージョンをまたぐようになったときです。BE-0063（Git 設定ソース）は、ローカルの `bajutsu` インストールより古い、あるいは新しいコミットに固定された Git の ref から config とそのシナリオツリーを取得します。これにより、ツールとシナリオツリーの間のスキーマ不一致は、エッジケースではなく通常の出来事になります。

この修正は今のうちなら安価ですが、バージョンマーカーを持たないシナリオファイルが存在し続けるほど、後から手当てするコストは上がります（これまでに書かれたバージョンなしのシナリオはすべて、曖昧な「バージョン 0」として移行ロジックが特別扱いする対象になってしまいます）。レポートマニフェストはすでに自身のスキーマで同じ問題を解決しています。`bajutsu/report/manifest.py:29` は `SCHEMA_VERSION = 4` を定義しており、これはすべてのマニフェストに書き込まれ（`manifest.py:111`）、`bajutsu/report/load.py:8` は本提案がシナリオにも求めたい読み込み側の規律を明文化しています。「欠けているフィールド（古い `schemaVersion`）はデフォルト値にフォールバックし、未知の新しいフィールドは無視する。こうすることで、古い実行結果もそのまま描画できる」。深刻度は中です。現時点で発生しているバグではありませんが、BE-0063 によって日常的に起こりうることが予見でき、その通信量が生まれる前に直しておくのが最も安上がりです。

## 詳細設計

1. **シナリオファイルのトップレベルに `schema` フィールドを追加する。** マニフェストの `SCHEMA_VERSION` に倣い、`ScenarioFile`（`bajutsu/scenario/models/scenario.py`）に `schema: int`（デフォルト `1`）を追加します。`schema` キーを持たないシナリオファイルはスキーマ `1` として扱います（現存するすべてのシナリオファイルは暗黙にバージョン 1 であり、現行のコーパスに対する移行作業は不要です）。
2. **ドキュメント全体を検証する前にバージョンをチェックする。** `bajutsu/scenario/load.py` の `load_scenario_file` において、パース済みの生の YAML から `schema` を読み取り、`ScenarioFile.model_validate` にドキュメント全体を渡す *前に*、現在の `bajutsu` が理解しているシナリオスキーマの定数と比較します。この `bajutsu` が理解できるより新しい `schema` の場合、`extra="forbid"` による分かりにくいエラーで生のフィールド検証を先に失敗させるのではなく、明快で対処可能な `ValueError`（例えば「このシナリオファイルはスキーマ 2 を使用していますが、この bajutsu が理解できるのはスキーマ 1 までです。bajutsu をアップグレードするか、より古いシナリオ／config のバージョンに固定してください」）を送出します。
3. **同一バージョンのドキュメントに対しては `extra="forbid"` を維持する。** バージョンチェックが変えるのは、バージョン不一致のときの失敗の *仕方* だけです。この `bajutsu` が対応しているスキーマを宣言しているシナリオに対して `extra="forbid"` を緩めることはしません。サポート範囲内のシナリオにおける本物のタイプミスは、今までどおり大きな声で失敗し続けなければなりません。
4. **バージョン更新の運用ルールを文書化する。** `schema` をいつ上げるべきかを短く記す注記を（`_base.py` の `_Model` の近く、あるいは `docs/` に）追加します。以前必須だったフィールドの意味を取り除く変更や、古い `bajutsu` が単に拒否するのではなく誤解してしまうような変更は定数を上げます。純粋に追加的なオプションフィールドは、古いツール側が新しい振る舞いを単に持たないだけであれば、必ずしも版上げを必要としません（マニフェストの `SCHEMA_VERSION` が `report/load.py` の運用ルールに従って読み込みを壊す変更のときだけ上げられているのと同様に、個別に判断します）。

これはパース部分（ディレクティブ 2 の「大きな声で失敗する」という関心事）に閉じた変更です。バージョン不一致は、古い前提のまま実行される（あるいは静かに誤動作する）のではなく、即座に、かつ明確に失敗するようになります。ここでの変更は `run` の合否判定ロジックには一切触れず、AI への依存も持ち込まないため、ディレクティブ 1 と 3 には影響しません。

## 検討した代替案

- **BE-0063 が出荷され、バージョンをまたぐ読み込みが実際に目に見える失敗を起こすまで何もしない。** 却下します。それまでの間に、`schema` フィールドを持たないシナリオファイルが未知の数だけ生まれてしまい、本提案が確立したい「バージョンなしはすべて暗黙のバージョン 1」という規約を事後的に正当化するのが難しくなります（それらのファイルの一部は、すでに新しい未文書化の振る舞いを前提にしているかもしれません）。通信量が生まれる前にバージョン付けするコストはフィールド 1 つとチェック 1 つですが、後からでは現存するすべてのシナリオファイルの監査が必要になります。
- **ファイルを書いた `bajutsu` のバージョンからスキーマバージョンを推定する（スキーマバージョンではなくツールバージョンを埋め込む）。** 却下します。ツールバージョンは、スキーマとは無関係なリリース（シナリオのフィールドに一切触れないリリースも多い）とスキーマの変更を混同させてしまい、すべての読み込み側にバージョン範囲とスキーマ形状の対応表を保守させることになります。スキーマに関係する変更のときだけインクリメントする明示的な整数の `schema` フィールドは、マニフェストがすでにうまく使っているのと同じ手法です。
- **`extra="forbid"` を `extra="ignore"` に緩め、新しいフィールドをエラーにせず黙って落とす。** それ単体では却下します。バージョンをまたぐ読み込みは直りますが、`extra="forbid"` が存在する理由であるタイプミス検出を犠牲にしてしまいます（タイプミスしたフィールドが黙って消え去るのは、「大きな声で失敗する」の正反対です）。同一バージョンのドキュメントには `extra="forbid"` を維持したまま、実際の不一致にだけ明確に失敗するバージョンチェックであれば、両方の性質を両立できます。
