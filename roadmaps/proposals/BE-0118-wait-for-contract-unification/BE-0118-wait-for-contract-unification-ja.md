## はじめに

`Driver.wait_for` は（少なくとも idb 実装が示すとおり）`timeout` までポーリングして待機する契約ですが、
Playwright バックエンドの実装は 1 回だけ即座にチェックし、引数の `timeout` を黙って無視します。これは決定性に関わる実バグです。
待機を要求した呼び出し元は、Web 上では要素がまだ現れていない瞬間に偽の `TimeoutError` を受け取ります。
本提案は、すべてのバックエンドが同じ振る舞いをするよう契約を統一します。 

## 動機

`bajutsu/drivers/base.py:114` は共有の `Driver` Protocol 上に `def wait_for(self, sel: Selector, timeout: float) -> bool: ...` を宣言しており、Protocol 自体にはドキュメント文字列がありませんが、idb の実装が実質的な契約を確立しています。`bajutsu/drivers/idb.py:306-319` は `timeout` から作った単調増加のデッドラインに対してループでポーリングし、チェックの合間に `poll` 秒だけスリープします。そのドキュメント文字列も明快です。「`sel` に一致する要素が現れるか `timeout` が経過するまでポーリングする」とあり、その理由として「要素は呼び出しの直後より少し遅れて描画されることがある」ことを挙げています。

Playwright バックエンドはこの契約を守っていません。`bajutsu/drivers/playwright.py:546-547` は次のとおりです。

```python
def wait_for(self, sel: base.Selector, timeout: float) -> bool:
    return len(base.find_all(self.query(), sel)) >= 1
```

これは 1 回だけチェックして即座に返ります。`timeout` は使われないパラメータです。`bajutsu/golden.py:190` は `driver.wait_for(anchor, timeout)` を直接呼び出しており（`golden_assert` の「drive-wait-query-compare」フロー）、戻り値が `False` のとき `TimeoutError(f"anchor {anchor!r} did not appear within {timeout}s")` を送出します。idb 上ではタイムアウトの全時間を待ち切ってから失敗しますが、Playwright 上では最初のチェックの時点で失敗します。つまり、ページの描画がまだ終わっていない状態と競合する `golden` アサーションは、同じ実世界の遅延であっても iOS では通っていたはずのものが Web では偽のタイムアウトとして報告されます。

深刻度は高く、これは設計上の隙間ではなく正真正銘のバグです。プライムディレクティブ 2（「決定性優先」）を直接損なっています。フレーキーな偽の失敗は、まさに決定性優先が防ごうとしている種類の問題であり、しかもバックエンド固有の挙動なので、「プラットフォームはバックエンドの一つにすぎない」という前提そのものへの信頼を損ないます（同じシナリオが、対象アプリとは無関係な理由でバックエンドごとに異なる振る舞いをしてしまいます）。

## 詳細設計

修正の方向性は、各バックエンドに独自のデッドラインループを実装させるのをやめ、`wait_for` を契約上「1 回だけチェックするプリミティブ」と定め、ポーリングは共有のヘルパー 1 つに任せることです。

1. **`Driver.wait_for` の契約を単発チェックとして定義し直す。** `bajutsu/drivers/base.py:114` のドキュメント文字列・契約を、`wait_for` は *1 回だけ* チェックして即座に返るものだと明記するように変更します。`timeout` はシグネチャから外すか（あるいはインターフェースの安定性のために残す場合は、未使用であり予約済みだと明示します）。これにより、現在の Playwright の振る舞いはバグではなく、（新しい）契約の正しい実装になります。
2. **共有のポーリングヘルパーを追加する。** `Driver`・`Selector`・`timeout`・ポーリング間隔 `poll` を受け取り、単調増加のデッドラインに対して `driver.wait_for(sel, ...)`（または `orchestrator/waits.py` の既存の `_wait` パターンを踏襲して `driver.query()` を直接）をポーリングする小さな関数（例えば `find_all` と並べて `bajutsu/drivers/base.py` に置く `wait_until`）を導入します。これは idb の現行の `wait_for` がすでに実装しているのと同じループを、ドライバの外に引き上げたものです。
3. **idb のポーリングループを共有ヘルパーへ移す。** `bajutsu/drivers/idb.py:306-319` を（新しい契約に合わせて）単発チェックに簡略化し、独自のデッドラインループを削除します。共有ヘルパーがすべてのバックエンドに対して一律にそれを提供するようになるためです。
4. **直接の呼び出し元を共有ヘルパーに向ける。** 今回の調査で判明した `driver.wait_for` の唯一の外部からの直接呼び出し元である `bajutsu/golden.py:190` を、`driver.wait_for` を直接呼ぶのではなく新しい共有ヘルパーを呼ぶように更新します。これにより `golden_assert` のタイムアウトは idb と Playwright で同一に守られます。
5. **リグレッションテストを追加する。** `wait_for` が最初は `False` を返し、しばらくしてから `True` を返すフェイク／スタブドライバ（要素が遅れて描画される状況を模したもの）で共有ヘルパーを検証します。このテストがあれば、今回のバグは検出できていたはずです。既存のドライバ／golden のテスト群に追加します。

これは `run`/CI の決定性を完全に保ったまま（ディレクティブ 1 には触れません。このパスに AI は一切入りません）、ディレクティブ 2 に直接資するものです。修正後は、待機に与えた `timeout` が、どのバックエンドが駆動していても同じ実時間の秒数を意味するようになります。

## 検討した代替案

- **共有ヘルパーを導入せず、Playwright の `wait_for` の内部だけでポーリングさせる。** 目先のバグは直りますが、今後追加されるすべてのバックエンド（Android）が、idb がすでに持っているのと同じデッドラインループを再実装する羽目になり、誰かがループの実装を忘れた日に同じ種類のバグが再発します。共有ヘルパーはこの重複を恒久的に取り除きます。
- **ポーリングの代わりに Playwright ネイティブの自動待機（`page.wait_for_selector`）を使う。** Playwright バックエンド単体としては魅力的ですが、`wait_for` はバックエンド横断の契約です。ポーリングモデルをバックエンド間で揃えておくほうが、1 つのバックエンドの実装方式だけを特別扱いするより、推論もテストもしやすくなります。将来的な最適化として、単発チェックの *内部で* Playwright がネイティブ待機を使う余地は残せます。
- **`timeout` を `wait_for` のシグネチャに残したまま、契約の不整合には何もしない。** 却下します。「単発チェックなのに `timeout` パラメータが存在し、しかも無視される」という形そのものが今回のバグの正体です。シグネチャは `timeout` を守る（ポーリングする）か、外す（単発チェックにしてポーリングは別の場所で行う）かのどちらかであるべきで、黙って無視する状態を許してはいけません。
