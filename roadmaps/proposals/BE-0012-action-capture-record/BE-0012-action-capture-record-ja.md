# BE-0012 — 操作キャプチャ record

* 提案: [BE-0012](BE-0012-action-capture-record-ja.md)
* Author: [@0x0c](https://github.com/0x0c)
* 状態: **提案**
* トラック: [提案](../../README-ja.md#提案)
* トピック: オーサリング体験（record / GUI エディタ）

## はじめに

Simulator 上でユーザーが行った実操作（tap / type / swipe）を記録し、AI を使わずにシナリオへ変換します。idb のイベント取得またはアクセシビリティイベント監視が必要です。

## 動機

現在、「人が行う操作」からシナリオへ至る道は AI record ループ（`record.py`）だけです。自然言語のゴールがエージェントを駆動し、エージェントがアプリを探索してステップを提案します。意図からの著述には強力ですが、著者が常に払えるとは限らないコストがあります。`ANTHROPIC_API_KEY` が必要で、LLM（大規模言語モデル）の往復を消費し、著者がすでに体で覚えているフローでは、それをゴールとして言語化してエージェントを待つより、ただ *やってみせる* 方が速いのです。さらに、記述するより実演する方が容易なフロー（精密なスワイプ、複数フィールドのフォーム、正確なタップ順）もあります。Simulator 上の実操作を直接シナリオステップへ変換するキャプチャモードは、著者に高速・オフライン・決定的な入口を与え、AI ループを置き換えるのではなく補完します。

## 詳細設計

キャプチャは、起動中の Simulator 上で人が行う操作（tap / type / swipe）を観測し、AI ループが生成するのと同じ `Scenario`（steps + `expect`）を出力します。したがって下流（`run`・`codegen`・レポート）は一切変わりません。難しいのは決定性です。生のイベントストリームは *座標* の列であり、シナリオは座標ではなく安定した `accessibilityIdentifier` で選択する、という prime directive があります。そこで各キャプチャアクションは、その操作の瞬間に取得した新鮮な `query()` に対して解決します。タップ点を、その点を frame に含む要素へ対応づけ、その要素を `id` セレクタとして書き出します（安定度ラダーに沿って `label` / `traits` へフォールバックし、最後の手段として座標タップに落ちる場合は degradation として明示します）。点が 2 要素に曖昧に解決した場合は、推測せず曖昧さを著者に提示します —— ランナーが強制する「曖昧なら最初にマッチしたものをタップせず失敗させる」と同じ規則です。

イベントストリームの取得元は 2 つが候補です。idb 自身のイベント取得か、アクセシビリティイベント監視です。本提案はどちらかには中立で、driver が backend を正規化するのと同様に、resolver と emitter を小さなインターフェースの背後に置き、どちらからでも供給できるようにします。タイミングは **固定 sleep ではなく条件待機** として記録します。あるアクションが直前と異なる画面に着地したとき、次のアクションが対象とする最初の要素への `wait` を挿入します —— AI ループが `_settle_step` で得るのと同じ自己完結性です —— これにより再実行が実時間のタイミングに依存しません。

キャプチャは app-agnostic を保ちます。既存コマンドと全く同様に、要素ツリーと config（対象アプリの `apps.<name>`、そのシナリオディレクトリ、redaction）を読み、同じ方法で YAML を書き出します。位置づけは厳密に Tier 1 です。キャプチャはシナリオを著述するもので、AI ループと同じく、その一部たりとも決定的な `run` / CI ゲートには入りません。AI ループと違い **API キーは不要** です。解決は純粋に構造的（`query()` に対する点-in-frame）であり、完全オフラインの著述経路を保ちます。

## 検討した代替案

* **生の座標タップを記録し、そのまま再生する。** 即座に不採用です。座標再生はレイアウト・デバイスサイズ・翻訳のいずれが変わっても壊れ、選択による決定性に反します。各タップをキャプチャ時に安定 `id` へ解決することこそが要点です。
* **タップ点を最前面 / 最初にマッチした要素へ黙って解決する。** 不採用です。「最初にマッチしたものをタップする」を再導入してしまいます。点が曖昧なときは、`resolve_unique` と整合するよう、解消のために著者へ提示しなければなりません。
* **キャプチャを既存の AI `record` ループの入力モードとして取り込む。** ありえますが BE-0014 へ先送りします。同提案が両方式の役割分担と相互変換を定義します。本提案はキャプチャの仕組みそのものだけを範囲とします。
