## はじめに

serve の Web UI で選んだ AI プロバイダー、model、推論の effort（reasoning effort）は、serve
プロセスの環境変数上にしか存在しません。保存時に `os.environ` へ書き込まれ、そこから読み戻され
るだけで、ディスクには書かれません（BE-0175 はプロバイダー選択について、この性質を「このセッ
ション限り、ディスクには書かない」と明記しています）。そのため serve の再起動、たとえば再デプ
ロイやクラッシュ、ローカルプロセスの単純な再起動が起きるたびに、これらの値は起動時の環境変数
（または設定ファイルの `ai:` ブロック）が与える値にリセットされ、運用担当者は手で入力し直すこと
になります。本提案は、保存した設定が再起動後も残るような永続化の仕組みを導入するものです。
Claude API キーがすでに write-once（設定・上書きはできるが、平文は返さない方式）の secret
store（BE-0136）を通じてこれを実現しているのと同じ考え方です。

## 動機

Settings 画面でプロバイダー・model・effort を設定した運用担当者は、その設定が保持され続けるこ
とを期待します。これはまさに BE-0136 がすでに API キーに対して提供している性質であり、API
キーは組織ごとに暗号化されて保存され、再起動をまたいで残ります。プロバイダー・model・effort は
機密情報ではないため BE-0136 の write-once・非公開読み出しという形式までは必要ありませんが、
ホスティングされたマルチテナント環境（BE-0015）では、API キーと同じ組織単位の永続性がこれらの
値にも必要です。デプロイのたびにプロバイダーと model を入力し直す手間は、これらの値に機密性が
ない以上、セキュリティ上のどんな利点も伴わない、単なる摩擦です。

本提案は、姉妹項目「serve の Web UI で AI プロバイダーごとの設定を持てるようにする」（未採番）
が提案するプロバイダーごとの設定構造を前提とします。永続化が扱うのは、その構造が持つ値が起動時
にどこから来て、保存時にどこへ行くかという点であり、構造そのものの形は対象にしません。

## 詳細設計

1. **保存先。** ローカルで動く serve は、プロバイダーごとの設定一式を、serve が保持する他の
   ローカル状態と並ぶ小さなファイルに永続化します。ホスティングされたマルチテナント環境では、
   secret store やジョブ記録がすでに使っている、組織ごとに永続化される DB ベースのストレージ
   （BE-0015 / BE-0136 と同じ方式）を通じて組織単位で永続化します。どちらの経路でも secret
   store そのものは再利用しません（理由は「検討した代替案」を参照）。
2. **起動時の読み込み。** 起動時に、永続化済みの設定一式が存在すればそれを読み込み、存在しな
   ければ起動時の環境変数やバインド済み設定の `ai:` ブロックが与える値にフォールバックします。
   これにより、解決されるプロバイダー・model・effort は、リセットされるのではなく最後に保存
   した内容と一致します。
3. **ゼロコンフィグとの両立。** Settings 画面を一度も開いたことがない運用担当者には変化があり
   ません。何も永続化されていない場合は現状どおり環境変数由来のデフォルトにフォールバックする
   ため、AI を使わないゼロコンフィグの経路（BE-0101）はそのまま維持されます。
4. **アクセス方法。** API キーとは異なり、これらの値は読み戻して編集画面に表示する対象です。
   BE-0136 のような write-once・非公開読み出しという制約を持ち越す必要はありません。

## 検討した代替案

- **運用担当者の `bajutsu.config.yaml` に書き戻して永続化する案。** 却下しました。Settings
  画面が、運用担当者自身が所有し Git 管理している設定ファイルを、本人の知らないところで書き換
  えるべきではないためです。Git 由来やアップロードされた設定はすでに外部からの、基本的に読み
  取り専用の入力として扱われており（たとえば BE-0121 のビルドコマンドに関する信頼境界）、
  serve ローカルの好み設定をそこに書き戻すことは、別のストアを用意すれば得られる以上の利点な
  しにこの境界を曖昧にしてしまいます。
- **既存の write-once secret store（BE-0136）をこれらの値にも流用する案。** 却下しました。
  この store は意図的に write-once・非公開読み出しという形式になっており、これは認証情報には
  適していますが、UI が読み戻して運用担当者に見せ、編集させる必要がある値には合いません。ここ
  では、普通に読み書きできるストアの方が適切です。

  の seam と、ファイルベースの `LocalProviderSettingsStore`（`bajutsu/serve/provider_store.py`）を
  追加し、ローカル serve の構築経路に組み込みました。保存したプロバイダー・model・effort は保存時
